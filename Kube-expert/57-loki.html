Loki

A log aggregation system that indexes only labels, not full text, keeping cost and operational complexity dramatically lower than traditional log search.
← Fluent Bit Course Index Distributed Tracing →

Explanation

Loki is a horizontally scalable log aggregation system built by Grafana Labs, designed explicitly to be "Prometheus, but for logs." Its defining architectural decision is that it does not index the full text of every log line the way Elasticsearch does. Instead, Loki indexes only a small set of labels (e.g. namespace, pod, app) — the same kind of low-cardinality label set Prometheus uses for metrics — and stores the actual log content compressed, in chunks, unindexed. Queries then combine a fast label-based lookup (which chunks to even look at) with a grep-style scan of only the matching chunks.

This trade-off is the whole point: full-text indexing (Elasticsearch's approach) is powerful for arbitrary text search but extremely expensive in storage and operational overhead at scale — the index is often larger than the raw logs themselves. Loki accepts slightly slower ad-hoc full-text search in exchange for an order-of-magnitude cheaper storage footprint and much simpler operations, which is the right trade for the common case: engineers overwhelmingly filter logs by which pod/namespace/deployment first, then read/grep within that already-narrow set — exactly what Loki is optimized for.

Kubernetes needs Loki (or an equivalent) because, as established in the Logging topic, logs must be shipped off-node to survive pod/node churn and be searchable. Loki fits Kubernetes' label-heavy metadata model unusually well: every log line arriving from Fluent Bit or Promtail is already tagged with namespace, pod, and custom labels pulled straight from Kubernetes object metadata — the exact same labels Prometheus already uses for metrics — so a team standardized on label-based Prometheus queries can reuse that mental model immediately for logs via LogQL, Loki's query language, which looks deliberately like PromQL: a label selector ({namespace="checkout", app="payments"}) followed by optional line filters (|= "error") and aggregation functions (rate(...)).

Loki's own agent, Promtail, is purpose-built for shipping to Loki (discovering Kubernetes pods, reading their log files, attaching labels), though Fluent Bit and the OpenTelemetry Collector can also ship to Loki via their respective Loki output plugins — teams often choose whichever agent they've already standardized on for other reasons (Fluent Bit's plugin breadth, or OTel's unification with traces/metrics) rather than being locked into Promtail specifically.

In the ecosystem, Loki occupies the same slot Prometheus/VictoriaMetrics occupy for metrics: Grafana queries it directly as a native datasource, dashboards mix log panels next to metric panels, and derived-field configuration (covered in the Grafana topic) lets an engineer click from a log line straight into a corresponding trace in Tempo.

Internal Working / Interactions

  • Distributor — receives incoming log streams (from Promtail/Fluent Bit/OTel Collector), validates and hands them off to ingesters, sharding by label set hash.
  • Ingester — batches incoming logs per stream into compressed chunks in memory, periodically flushing them to long-term object storage (S3/GCS/local disk).
  • Index — a much smaller structure than the logs themselves, mapping label sets to chunk locations (stored in a key-value store or, in newer versions, embedded via TSDB-style index files alongside chunks).
  • Querier — parses LogQL, looks up relevant chunks via the index, and scans only those chunks for the actual line filter, merging results across ingesters/storage.
  • Promtail / Fluent Bit / OTel Collector — agents that discover Kubernetes pods, tail their logs, attach labels, and push to Loki's distributor via HTTP.

Code Example

# Loki config (single-binary / monolithic mode) — simplest production-ready deployment,
# storing chunks and index locally or in object storage.
auth_enabled: false                      # disable multi-tenant auth for a single-team cluster

server:
  http_listen_port: 3100

ingester:
  chunk_idle_period: 5m                  # flush a stream's chunk after 5m of inactivity
  chunk_retain_period: 30s
  lifecycler:
    ring:
      kvstore: { store: inmemory }       # fine for single-binary mode; use consul/etcd for HA clusters

schema_config:
  configs:
    - from: 2024-01-01
      store: tsdb                        # modern index type: index embedded alongside chunks
      object_store: filesystem           # swap for s3/gcs in production for durability + scale
      schema: v13
      index:
        prefix: index_
        period: 24h

storage_config:
  tsdb_shipper:
    active_index_directory: /loki/index
    cache_location: /loki/index_cache
  filesystem:
    directory: /loki/chunks

limits_config:
  retention_period: 720h                 # 30 days; enforce via the compactor below
  ingestion_rate_mb: 16                   # per-tenant ingestion cap, protects Loki from a noisy source

compactor:
  retention_enabled: true                 # actually deletes chunks past retention_period
  delete_request_store: filesystem
---
# Promtail scrape config — discovers Kubernetes pods and ships their logs to Loki, labeling
# each stream with namespace/pod/container, the same labels used in LogQL queries.
scrape_configs:
  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod                        # same SD mechanism Prometheus uses
    pipeline_stages:
      - cri: {}                          # parse containerd's CRI log format
    relabel_configs:
      - source_labels: [__meta_kubernetes_namespace]
        target_label: namespace
      - source_labels: [__meta_kubernetes_pod_name]
        target_label: pod
      - source_labels: [__meta_kubernetes_pod_container_name]
        target_label: container

kubectl / CLI Commands

kubectl logs -n logging deploy/promtail --tail=50               # check log-shipping agent for push errors

kubectl get pvc -n logging -l app=loki                          # check storage volumes if using persistent disk
kubectl logs -n logging deploy/loki --tail=100 | grep -i compactor   # verify retention/compaction is running

Best Practices & Common Pitfalls

Best Practices

  • Keep label cardinality low and deliberate — labels should be things you filter by (namespace, app), not high-cardinality fields like request IDs.
  • Use object storage (S3/GCS) for chunks in production rather than local disk, for durability and easy scaling.
  • Enable the compactor with a defined retention period so old chunks are actually deleted, not just aged out logically.
  • Standardize label names across Promtail/Fluent Bit configs so LogQL queries are consistent cluster-wide.
  • Use line filters (|=, !=) generously — they're cheap because they run only on already-narrowed chunks.

Pitfalls

  • Putting high-cardinality fields (user ID, request ID, trace ID) into labels instead of the log line — this defeats Loki's entire cost model and can crash the index.
  • Expecting Elasticsearch-grade arbitrary full-text search performance — Loki trades that off deliberately for cost.
  • Running ingesters without a durable ring store (etcd/consul) in multi-replica production setups, causing chunk loss on restart.
  • Not setting ingestion_rate_mb limits, letting one noisy namespace starve ingestion for everyone else.

50 Interview Questions

1. What is Loki's defining architectural decision compared to Elasticsearch?
Short answer: Loki does not index the full text of every log line; it only indexes labels.
Detail: Log content is stored compressed in chunks, unindexed, while only a small low-cardinality label set (namespace, pod, app) is indexed for lookup.
Production example: Querying {namespace="checkout"} is a fast label lookup, followed by a scan of only the matching chunks.
Common mistake: Expecting Loki to support arbitrary full-text search as fast as Elasticsearch — it deliberately trades that off for cost.
2. Why is Loki described as "Prometheus, but for logs"?
Short answer: It reuses the same low-cardinality label model Prometheus uses for metrics, applied to logs instead.
Detail: Teams already using label-based PromQL queries can reuse that mental model immediately via LogQL, Loki's PromQL-like query language.
Production example: The same namespace/app labels used in Prometheus alerts are reused directly in LogQL queries against Loki.
Common mistake: Treating Loki's label design as arbitrary rather than intentionally mirroring Prometheus's cardinality discipline.
3. What is the core cost/performance trade-off Loki makes versus full-text indexing?
Short answer: Slightly slower ad-hoc full-text search in exchange for dramatically cheaper storage and simpler operations.
Detail: Full-text indexes (like Elasticsearch's) are often larger than the raw logs themselves; Loki avoids that entirely by not indexing content.
Production example: A team filtering logs by pod/namespace first, then grepping within that narrow set, sees no practical downside from Loki's approach.
Common mistake: Choosing Loki for a workload that primarily needs arbitrary cross-cluster full-text search without label narrowing.
4. What is LogQL and how does it structure a query?
Short answer: Loki's PromQL-like query language: a label selector followed by optional line filters and aggregation functions.
Detail: A query like {namespace="checkout", app="payments"} |= "error" first narrows by labels, then filters lines, optionally wrapped in functions like rate(...).
Production example: Alerting on error rate per service using rate({app="payments"} |= "error" [5m])-style LogQL.
Common mistake: Writing a line filter without a sufficiently narrow label selector first, forcing an expensive scan across too many chunks.
5. What are the four main internal components of Loki, and what does each do?
Short answer: Distributor, Ingester, Index, and Querier.
Detail: Distributor receives and shards incoming streams by label hash; Ingester batches logs into compressed chunks and flushes to object storage; Index maps label sets to chunk locations; Querier parses LogQL, uses the index to find chunks, and scans them.
Production example: A push from Promtail hits the distributor, is sharded to an ingester, batched into a chunk, flushed to S3, and later located via the index when a Grafana query runs through the querier.
Common mistake: Assuming Loki is a single monolithic process rather than a set of distinct, independently scalable components (even in single-binary mode).
6. Why is keeping label cardinality low so critical in Loki, unlike in Elasticsearch?
Short answer: High-cardinality labels defeat Loki's entire cost model and can crash the index.
Detail: Labels are meant to be things you filter by (namespace, app) — putting per-request identifiers like user ID or trace ID into labels creates enormous numbers of unique streams, each needing index entries.
Production example: Putting request_id as a label instead of leaving it in the log line causes an explosion of unique label sets and index bloat.
Common mistake: Adding high-cardinality fields (user ID, request ID, trace ID) as labels instead of keeping them in the unindexed log line — an explicitly called-out pitfall.
7. What is the role of the compactor, and why must it be explicitly enabled?
Short answer: The compactor enforces retention by actually deleting chunks past the configured retention period.
Detail: Without retention_enabled: true on the compactor, chunks past retention_period are only logically aged out, not physically deleted.
Production example: Setting retention_period: 720h (30 days) with the compactor enabled ensures old chunks are actually removed from object storage.
Common mistake: Setting a retention_period without enabling the compactor and assuming storage is automatically reclaimed.
8. Why does the ring store (kvstore) matter for the ingester's lifecycler in production?
Short answer: Running ingesters without a durable ring store like etcd/consul in multi-replica setups causes chunk loss on restart.
Detail: The in-memory kvstore is fine for single-binary mode, but production HA clusters need a durable store so ingester membership/state survives restarts.
Production example: A 3-replica ingester deployment backed by etcd for the ring avoids losing in-flight chunk data when one ingester restarts.
Common mistake: Leaving kvstore: { store: inmemory } in a multi-replica production deployment — an explicit pitfall in the file.
9. What does ingestion_rate_mb protect against?
Short answer: A per-tenant ingestion cap that protects Loki from a noisy source overwhelming ingestion.
Detail: Without this limit, one high-volume or misbehaving namespace/application could starve ingestion capacity for every other tenant/namespace.
Production example: A runaway debug-logging deployment is rate-limited rather than degrading log ingestion cluster-wide.
Common mistake: Not setting ingestion_rate_mb, letting one noisy namespace starve ingestion for everyone else — an explicit pitfall.
10. Why does Loki use object storage (S3/GCS) rather than local disk for chunks in production?
Short answer: For durability and easy scaling, versus the fragility and scaling limits of local disk.
Detail: Local filesystem storage is fine for a single-binary demo, but production deployments need chunks to survive node/pod loss and scale independent of any one node's disk.
Production example: Swapping object_store: filesystem for S3/GCS in the schema_config to move from a demo setup to a durable production one.
Common mistake: Running production Loki on local disk storage, risking data loss if the underlying node/volume fails.
