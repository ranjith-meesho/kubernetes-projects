AKS (Azure Kubernetes Service)

Azure's managed Kubernetes: node pools, Azure AD-integrated RBAC, and the Azure CNI vs kubenet networking tradeoff.
← EKS Course Index CKA Exam Preparation →

Explanation

AKS is unique among the big three managed offerings in that Azure does not charge for the control plane itself (you pay only for node VMs and any add-ons/uptime SLA tier), which changes cost calculus for small clusters. Like GKE and EKS, Azure fully manages etcd, the API server, and control-plane upgrades; your operational surface is node pools, networking mode, and identity integration.

Node pools in AKS work similarly to the other providers: a system node pool is mandatory and runs critical system Pods (CoreDNS, metrics-server, tunnelfront/konnectivity), while one or more user node pools run your actual application workloads — Azure explicitly recommends never scaling the system pool to zero or scheduling arbitrary user workloads onto it, since starving it of resources can take down cluster-critical add-ons. User node pools can be scaled independently, mix VM sizes (including GPU-enabled SKUs), use Spot VMs for fault-tolerant workloads, and each can have its own taints/labels for workload isolation, exactly like node-pool patterns on GKE/EKS.

Azure AD integration is AKS's identity story at two distinct layers. At the cluster-access layer, Azure AD-integrated RBAC lets you authenticate kubectl users against Azure AD identities/groups instead of managing a separate certificate or token system, and map Azure AD groups directly to Kubernetes RBAC Role/ClusterRole bindings — so offboarding an employee in Azure AD immediately revokes their cluster access. At the workload-identity layer, Azure AD Workload Identity (the successor to the older pod-identity add-on) federates a Kubernetes ServiceAccount with an Azure AD application/managed identity via OIDC, letting a Pod exchange its projected service-account token for a short-lived Azure AD token scoped to exactly the resources that identity is granted — the same no-static-credentials pattern as IRSA on EKS and Workload Identity on GKE.

Networking is where AKS has the starkest decision point: kubenet is the simpler, older option where Pods get IPs from a private overlay network that is NAT'd at the node for anything leaving the VNet — cheaper on VNet IP consumption, but with real limitations (User-Defined Routes overhead, no native Azure Network Policy support for some features, harder to peer/inspect Pod traffic from other VNet resources). Azure CNI assigns each Pod a real, routable IP directly from the VNet subnet, exactly like the EKS VPC CNI model — Pods are natively reachable by other VNet resources without NAT, security tools and Network Security Groups can see real Pod IPs, but the VNet subnet must be sized with real per-Pod IP headroom (Azure CNI Overlay, a newer mode, decouples Pod IP space from the VNet to solve exactly this exhaustion problem while keeping most Azure CNI benefits). For new production clusters, Azure now generally recommends Azure CNI (Overlay or standard) over kubenet, mirroring the industry's general move away from NAT-heavy overlay networking.

Practical defaults: keep a small, dedicated system node pool with a taint reserving it for system components; add purpose-built user node pools per workload shape; bind Azure AD groups to Kubernetes RBAC rather than distributing static kubeconfigs; and default to Azure CNI (Overlay for large clusters worried about VNet IP exhaustion) over kubenet for new clusters.

Internal Working / Interactions

  • Control plane: fully managed by Azure, free of charge on the Free tier (paid Standard/Premium tiers add SLA guarantees and more scale headroom).
  • System vs user node pools: the system pool runs core add-ons; the scheduler is guided by taints (CriticalAddonsOnly=true:NoSchedule) to keep arbitrary workloads off it.
  • Azure AD-integrated kubectl auth: kubelogin/az aks get-credentials configures kubeconfig to perform an Azure AD device-code or browser login, exchanging it for a token the API server validates against Azure AD.
  • Azure AD Workload Identity: AKS's OIDC issuer + a federated identity credential on the Azure AD app let a Pod's projected token be exchanged for an Azure AD access token via MSAL, without any secret material ever stored in the cluster.
  • Azure CNI: the CNI plugin requests a VNet IP for each Pod from the subnet's IP pool at Pod creation; Azure CNI Overlay instead assigns Pod IPs from a separate, non-VNet-routed range, translated at the node.

Code Example

#############################################
# 1. az CLI - create an AKS cluster with Azure AD RBAC + Azure CNI
#############################################
az aks create \
  --resource-group prod-rg \
  --name prod-aks \
  --network-plugin azure \                       # Azure CNI: real VNet IPs per Pod
  --network-plugin-mode overlay \                # Overlay mode: avoids VNet IP exhaustion at scale
  --enable-aad \                                  # Azure AD-integrated cluster authentication
  --enable-azure-rbac \                           # enforce RBAC decisions via Azure, not just in-cluster
  --node-count 3 \
  --nodepool-name system \
  --node-taints CriticalAddonsOnly=true:NoSchedule

# Add a dedicated user node pool for application workloads
az aks nodepool add \
  --resource-group prod-rg --cluster-name prod-aks \
  --name userpool --node-count 3 \
  --min-count 3 --max-count 10 --enable-cluster-autoscaler \
  --mode User                                     # explicitly a user pool, not system

---
# 2. Azure AD Workload Identity binding
apiVersion: v1
kind: ServiceAccount
metadata:
  name: orders-sa
  namespace: orders
  annotations:
    azure.workload.identity/client-id: "11111111-2222-3333-4444-555555555555"   # Azure AD app client ID
  labels:
    azure.workload.identity/use: "true"           # opts the SA into the webhook injecting federated creds
---
apiVersion: v1
kind: Pod
metadata:
  name: orders-worker
  namespace: orders
  labels:
    azure.workload.identity/use: "true"
spec:
  serviceAccountName: orders-sa
  containers:
    - name: worker
      image: prodacr.azurecr.io/orders-worker:v1.4
# Federated credential linking (one-time az command, not YAML):
# az identity federated-credential create --name orders-fic \
#   --identity-name orders-identity --resource-group prod-rg \
#   --issuer  --subject system:serviceaccount:orders:orders-sa

kubectl / CLI Commands

# Fetch kubeconfig with Azure AD auth wired in
az aks get-credentials --resource-group prod-rg --name prod-aks

# List node pools and their mode (System/User)
az aks nodepool list --resource-group prod-rg --cluster-name prod-aks -o table

# Confirm Azure AD group -> Kubernetes RBAC binding
kubectl get rolebindings,clusterrolebindings -A -o wide | grep AAD_GROUP_OBJECT_ID

# Check the AKS OIDC issuer URL (needed for Workload Identity federation)
az aks show --resource-group prod-rg --name prod-aks --query "oidcIssuerProfile.issuerUrl" -o tsv

# Verify Azure CNI Pod IP allocation per node
kubectl get pods -o wide -n orders

# Confirm system pool taint is respected (no stray app pods on system pool)
kubectl get pods -A -o wide --field-selector spec.nodeName=

# Trigger a node pool upgrade
az aks nodepool upgrade --resource-group prod-rg --cluster-name prod-aks --name userpool --kubernetes-version 1.29.2

Best Practices & Common Pitfalls

Best Practices

  • Keep a dedicated, tainted system node pool separate from user workloads.
  • Bind Azure AD groups to Kubernetes RBAC instead of distributing static kubeconfigs.
  • Use Azure AD Workload Identity for all Pod-level Azure resource access.
  • Default to Azure CNI (Overlay for large clusters) over kubenet for new production clusters.
  • Use separate user node pools per workload shape (GPU, memory-optimized, spot/batch).

Common Pitfalls

  • Scheduling application workloads onto the system node pool, starving critical add-ons.
  • Choosing plain Azure CNI on a large VNet without Overlay and running out of subnet IPs.
  • Skipping Azure AD RBAC and relying on shared static kubeconfigs for cluster access.
  • Storing Azure credentials as Kubernetes Secrets instead of using Workload Identity federation.
  • Forgetting the system node pool still needs its own min/max autoscaler bounds.

50 Interview Questions

1. Why is AKS's control-plane billing model distinctive among the big three cloud providers?
Short answer: Azure does not charge for the control plane itself (Free tier) — you pay only for node VMs and any paid uptime-SLA add-ons, unlike EKS's flat per-cluster hourly control-plane fee.
Detail: This changes cost calculus specifically for small clusters or many-cluster setups, since running additional clusters doesn't automatically add flat control-plane cost the way it does on EKS.
Production example: A team running several small dev/test AKS clusters incurs no extra control-plane charge per cluster, only the underlying node VM cost.
Common mistake: Assuming all three major clouds charge for the control plane identically when comparing AKS/EKS/GKE costs.
2. Why must the system node pool never be scaled to zero or share space with user workloads?
Short answer: The system node pool runs critical system Pods (CoreDNS, metrics-server, tunnelfront/konnectivity); starving it of resources by scheduling arbitrary workloads there can take down cluster-critical add-ons.
Detail: Azure enforces this via the `CriticalAddonsOnly=true:NoSchedule` taint pattern, actively repelling non-system Pods so the system pool's capacity stays reserved for what keeps the cluster itself functioning.
Production example: A cluster created with `--node-taints CriticalAddonsOnly=true:NoSchedule` on the system pool ensures application Deployments can never accidentally land there and crowd out CoreDNS.
Common mistake: Scheduling application workloads onto the system node pool "since it has spare capacity," starving critical add-ons under load.
3. What are the two distinct layers of Azure AD integration in AKS?
Short answer: Cluster-access layer (Azure AD-integrated RBAC for kubectl users/groups) and workload-identity layer (Azure AD Workload Identity for Pod-level Azure resource access) — solving different problems with the same identity provider.
Detail: The access layer maps Azure AD groups directly to Kubernetes RBAC bindings so offboarding in Azure AD immediately revokes cluster access; the workload layer federates a ServiceAccount with an Azure AD app/managed identity via OIDC for API calls.
Production example: An engineer's Azure AD group membership controls their `kubectl` RBAC permissions, while a completely separate `orders-sa` ServiceAccount federation controls what Azure resources the `orders-worker` Pod can call.
Common mistake: Conflating "we use Azure AD for kubectl login" with "our Pods use Workload Identity" — they are two independent integrations that must each be configured.
4. How does Azure AD Workload Identity let a Pod get an Azure AD token without stored credentials?
Short answer: It federates a Kubernetes ServiceAccount with an Azure AD application/managed identity via OIDC; a Pod exchanges its projected service-account token for a short-lived Azure AD token scoped to exactly the resources that identity is granted.
Detail: This is the successor to the older pod-identity add-on, and mirrors IRSA on EKS and Workload Identity on GKE — the same no-static-credentials pattern implemented via AKS's OIDC issuer and a federated identity credential.
Production example: A ServiceAccount labeled `azure.workload.identity/use: "true"` and annotated with a client-id gets a webhook-injected federated credential, with the actual linkage created via `az identity federated-credential create`.
Common mistake: Storing Azure credentials as Kubernetes Secrets instead of configuring Workload Identity federation, leaving long-lived credentials sitting in the cluster.
5. What's the core tradeoff between kubenet and Azure CNI?
Short answer: kubenet uses a NAT'd overlay network (cheaper on VNet IP consumption but with routing/inspection limitations); Azure CNI assigns each Pod a real, routable VNet IP (natively reachable, but requires real per-Pod IP headroom in the subnet).
Detail: kubenet has real limitations — User-Defined Routes overhead, no native Azure Network Policy support for some features, and difficulty peering/inspecting Pod traffic from other VNet resources — which is why Azure now generally recommends Azure CNI for new production clusters.
Production example: A cluster needing other VNet resources (a database VM, a security appliance) to see real Pod IPs directly chooses Azure CNI over kubenet.
Common mistake: Choosing kubenet by default for "IP savings" without weighing its routing and network-policy limitations against actual production needs.
6. What problem does Azure CNI Overlay mode solve compared to standard Azure CNI?
Short answer: It decouples Pod IP space from the VNet (assigning Pod IPs from a separate, non-VNet-routed range) to avoid VNet subnet exhaustion, while keeping most Azure CNI benefits.
Detail: Standard Azure CNI consumes a real VNet IP per Pod, which at scale can exhaust even generously sized subnets; Overlay mode keeps the "real reachability" benefits largely intact while removing the direct 1:1 VNet IP consumption per Pod.
Production example: `az aks create --network-plugin azure --network-plugin-mode overlay` is used specifically for large clusters worried about VNet IP exhaustion.
Common mistake: Choosing standard (non-overlay) Azure CNI for a large-scale cluster and later hitting the same VNet IP exhaustion problem overlay mode was designed to avoid.
7. Why does Azure recommend binding Azure AD groups to Kubernetes RBAC instead of distributing static kubeconfigs?
Short answer: Static kubeconfigs persist access independent of employment status; Azure AD group bindings mean offboarding an employee in Azure AD immediately revokes their cluster access.
Detail: This ties cluster authorization to a single, centrally managed identity source rather than distributing long-lived credential files that must be manually revoked/rotated per person.
Production example: An employee leaving the company has their Azure AD account disabled, which alone (without any Kubernetes-side action) removes their `kubectl` access.
Common mistake: Skipping Azure AD RBAC setup and relying on shared static kubeconfig files for cluster access, which don't get revoked when someone leaves.
8. What's `kubelogin`'s role in AKS's Azure AD authentication flow?
Short answer: It (alongside `az aks get-credentials`) configures kubeconfig to perform an Azure AD device-code or browser login, exchanging that login for a token the API server validates against Azure AD.
Detail: This is the mechanical piece that makes Azure AD-integrated RBAC actually work at the `kubectl` command level — without it, kubeconfig would have no way to obtain an Azure AD-backed token for API server authentication.
Production example: A developer running `az aks get-credentials --resource-group prod-rg --name prod-aks` gets a kubeconfig wired to prompt an Azure AD login on first use.
Common mistake: Assuming a standard kubeconfig (like on a non-AAD cluster) works unmodified against an AAD-enabled AKS cluster without the AAD-aware auth plugin configured.
9. Why does the system node pool need its own min/max autoscaler bounds separate from user pools?
Short answer: The system pool has its own resource needs driven by cluster-critical add-ons, independent of application workload scaling patterns governed by user pool autoscaling.
Detail: Forgetting to configure autoscaler bounds specifically on the system pool (assuming user-pool autoscaling "covers everything") can leave system-critical capacity under-provisioned as add-on resource needs grow.
Production example: A cluster carefully autoscales its user pool for application load but leaves the system pool fixed at its initial node count, missing added system-pool pressure from newly enabled add-ons.
Common mistake: Forgetting the system node pool still needs its own explicit min/max autoscaler bounds, treating it as a "set once and forget" fixed pool.
10. What are this topic's practical default recommendations for a new production AKS cluster?
Short answer: A small, dedicated tainted system node pool; purpose-built user node pools per workload shape; Azure AD groups bound to RBAC instead of static kubeconfigs; and Azure CNI (Overlay for large clusters) over kubenet.
Detail: These map directly to the topic's core risk areas: starving system add-ons (solved by tainting the system pool), unrevoked access (solved by Azure AD RBAC), and VNet IP exhaustion (solved by CNI Overlay mode).
Production example: A new cluster is created with `--enable-aad --enable-azure-rbac --network-plugin azure --network-plugin-mode overlay` and a tainted system pool, plus a separate autoscaling user pool.
Common mistake: Treating these as later hardening steps to retrofit, rather than defaults established at cluster creation before workloads onboard.
