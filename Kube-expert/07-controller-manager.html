Controller Manager

The reconciliation-loop pattern, the built-in controllers it runs, and how leader election keeps exactly one instance active.
← Scheduler Course Index kubelet →

Explanation

The kube-controller-manager is a single binary that bundles together dozens of independent controllers, each responsible for one narrow slice of cluster behavior: the Deployment controller manages ReplicaSets, the ReplicaSet controller manages Pods, the Node controller monitors node health, the Job controller runs batch workloads to completion, the ServiceAccount controller provisions default tokens, and so on. Each controller runs its own independent watch loop against the API Server — they don't know about each other, and they don't need to.

Kubernetes needs this pattern because "declarative" only means something if some process is actively working to close the gap between desired and actual state. A reconciliation loop is the mechanism: watch for changes to a resource, compare desired (spec) against observed (status), and take whatever action shrinks the difference — then repeat, forever, reacting both to new events and to periodic full resyncs (so a missed event never permanently desyncs the system). This "level-triggered" design (react to full current state, not just the last delta) is what makes Kubernetes resilient to dropped messages and controller restarts.

Bundling all these controllers into one process is a pragmatic choice rather than an architectural necessity — it means one leader-election lease, one binary to deploy, and shared caching infrastructure (informers), at the cost of them all sharing fate if the process crashes. Custom controllers and operators you write yourself follow the exact same pattern but run as separate processes/Deployments outside this binary — because the built-in controller manager only manages built-in resource types.

Some example built-in controllers: the Deployment controller orchestrates rolling updates by creating new ReplicaSets and scaling old ones down; the Node controller watches heartbeats and marks Nodes NotReady, eventually evicting their Pods; the Endpoint/EndpointSlice controller keeps Service endpoints in sync with matching Pod readiness; the Namespace controller handles cascading deletion when a namespace is deleted; the ServiceAccount/Token controllers provision credentials.

Like the Scheduler, the Controller Manager runs with multiple replicas for HA but uses leader election (a Lease object) so only one replica is ever actively reconciling — preventing two instances from racing to make conflicting decisions about the same resource.

Internal Working / Interactions

  • Controller Manager ↔ API Server: every individual controller inside it watches its resource type(s) via informers and writes changes back through the same API Server.
  • Deployment controller → ReplicaSet controller: a chain — Deployment creates/scales ReplicaSets, ReplicaSet controller creates/deletes Pods to match replicas.
  • Node controller ↔ kubelet heartbeats: monitors Node status conditions; applies NoExecute taints and triggers eviction after the configured timeout.
  • Leader election: all controllers inside the process share one Lease (kube-controller-manager in kube-system); only the leader replica runs any reconciliation.
  • Custom controllers/operators: follow the identical watch-reconcile pattern but run as separate Deployments, extending the system without modifying this binary.

Code Example

# kube-controller-manager static pod — the relevant flags controlling
# reconciliation behavior and leader election.
apiVersion: v1
kind: Pod
metadata:
  name: kube-controller-manager
  namespace: kube-system
spec:
  containers:
    - name: kube-controller-manager
      image: registry.k8s.io/kube-controller-manager:v1.29.0
      command:
        - kube-controller-manager
        - --leader-elect=true                       # enable HA leader election
        - --leader-elect-lease-duration=15s
        - --leader-elect-renew-deadline=10s
        - --node-monitor-period=5s                   # how often Node controller checks heartbeats
        - --node-monitor-grace-period=40s             # heartbeat silence before NotReady
        - --pod-eviction-timeout=5m0s                 # grace period before evicting Pods off a NotReady node
        - --concurrent-deployment-syncs=5             # parallelism for the Deployment controller's workqueue
        - --controllers=*,-bootstrapsigner,-tokencleaner  # enable all controllers except these two

kubectl Commands

# Confirm controller-manager is running and check the leader lease
kubectl get pods -n kube-system -l component=kube-controller-manager
kubectl get lease -n kube-system kube-controller-manager -o yaml

# Watch a Deployment's controller chain in action
kubectl scale deployment web --replicas=5
kubectl get replicasets -l app=web -w

# Inspect owner references to see the controller chain explicitly
kubectl get pod web-7d9f8c-abcde -o jsonpath='{.metadata.ownerReferences}'

# Check Node controller behavior via conditions/taints
kubectl describe node worker-3 | grep -A5 Taints
kubectl get events --field-selector reason=NodeNotReady -A

# View controller-manager logs directly (kubeadm cluster)
kubectl logs kube-controller-manager-controlplane-1 -n kube-system

Best Practices & Common Pitfalls

Best Practices

  • Tune node-monitor-grace-period and pod-eviction-timeout deliberately for your failure-tolerance needs.
  • Monitor leader-election lease flapping — it silently halts all reconciliation cluster-wide.
  • Understand owner-reference chains (Deployment → ReplicaSet → Pod) before manually editing any of them.
  • Write custom controllers to follow the same level-triggered reconcile pattern, not ad-hoc scripts.

Pitfalls

  • Manually editing a ReplicaSet's Pods directly instead of through the Deployment — the controller will just recreate them.
  • Assuming multiple controller-manager replicas parallelize work — only the leader is ever active.
  • Setting eviction timeouts too aggressively, causing unnecessary reschedule storms on transient network blips.
  • Forgetting that disabling a specific controller (via --controllers) silently breaks features that depend on it (e.g. disabling serviceaccount-token breaks new pod credential provisioning).

50 Interview Questions

1. What is a reconciliation loop and why is it "level-triggered" rather than "edge-triggered"?
Short answer: A continuous process comparing full desired state against full observed state and acting to shrink the gap — level-triggered means it reacts to current state, not just the last event.
Detailed explanation: Edge-triggered systems act only on the delta of the last change and can permanently desync if an event is dropped; level-triggered controllers periodically resync against full current state regardless of events, so a missed watch event never causes permanent drift.
Production example: The ReplicaSet controller doesn't just react to "a Pod died" events — it periodically re-compares desired replica count against actual running Pods regardless.
Common mistake: Writing a custom controller that only reacts to specific event deltas without a periodic full resync, making it fragile to missed watch events.
2. Why does Controller Manager bundle dozens of controllers into a single binary?
Short answer: A pragmatic choice for one leader-election lease, one deployable artifact, and shared informer caching infrastructure.
Detailed explanation: This isn't an architectural necessity — it's operational convenience. The tradeoff is shared fate: if the process crashes, every built-in controller (Deployment, Node, Job, etc.) stops simultaneously, unlike if each ran as a separate process.
Production example: A crash-looping kube-controller-manager pod halts Deployment rollouts, Node health monitoring, and Job completion tracking all at once.
Common mistake: Assuming individual controllers can be independently restarted without affecting the others — they all share one process's lifecycle.
3. Trace the controller chain when you run kubectl scale deployment web --replicas=5.
Short answer: API Server persists the new replica count; the Deployment controller updates the owned ReplicaSet's replica count; the ReplicaSet controller then creates/deletes Pods to match.
Detailed explanation: This is a chain of independent watch-reconcile loops, not one atomic operation — the Deployment controller watches Deployments and writes to ReplicaSets; the ReplicaSet controller separately watches ReplicaSets and writes to Pods. Each layer only knows about the layer directly below it via owner references.
Production example: kubectl get pod ... -o jsonpath='{.metadata.ownerReferences}' reveals this exact chain explicitly on a running Pod.
Common mistake: Believing scaling a Deployment directly creates Pods — it actually goes through an intermediate ReplicaSet update first.
4. What does the Node controller do, and what are the relevant timing parameters?
Short answer: Monitors node heartbeats and marks Nodes NotReady, eventually evicting their Pods, governed by node-monitor-grace-period and pod-eviction-timeout.
Detailed explanation: node-monitor-period controls how often it checks heartbeats (default 5s); node-monitor-grace-period is how long without a heartbeat before NotReady (default 40s); pod-eviction-timeout is the additional grace period before Pods are actually evicted (default 5m) — this staged approach avoids overreacting to brief network blips.
Production example: A transient 30-second network blip causes no Pod disruption since it's under the grace period; a genuine node failure triggers full eviction after ~5 minutes.
Common mistake: Setting these timeouts too aggressively without considering the tradeoff between fast failure response and unnecessary reschedule storms from transient issues.
5. Why is manually editing a controller-managed Pod pointless?
Short answer: The owning controller (e.g. ReplicaSet) will simply recreate or revert it to match the desired state on its next reconcile pass.
Detailed explanation: Because reconciliation is continuous and level-triggered, any drift you manually introduce into a controller-owned resource is treated exactly like drift from a crash — the controller corrects it, often within seconds.
Production example: Manually deleting a ReplicaSet-owned Pod for "testing" just results in the ReplicaSet controller creating a brand-new replacement Pod almost immediately.
Common mistake: Trying to permanently change a controller-managed object's Pods directly instead of updating the owning resource's spec/template.
6. How do custom controllers/operators relate to the built-in Controller Manager?
Short answer: They follow the identical watch-reconcile pattern but run as separate processes/Deployments, extending Kubernetes without modifying the core binary.
Detailed explanation: The built-in Controller Manager only manages built-in resource types; custom controllers (often built with client-go/controller-runtime) watch CRDs or built-in types and reconcile them the same way, just outside this specific binary.
Production example: An Argo CD application controller reconciles Application CRDs toward their desired Git state, using the same watch/informer/reconcile pattern as the built-in Deployment controller.
Common mistake: Assuming you must modify or fork the core Controller Manager to add new reconciliation behavior — writing an independent controller/operator is the standard extension mechanism.
7. Why does leader election matter specifically for Controller Manager, and what's shared across all its controllers?
Short answer: All controllers inside the process share one Lease; only the leader replica runs any reconciliation, preventing conflicting decisions from multiple active replicas.
Detailed explanation: Because it's one binary with one lease (kube-controller-manager in kube-system), leadership is all-or-nothing — you can't have one controller inside the binary active on replica A while another is active on replica B.
Production example: kubectl get lease -n kube-system kube-controller-manager -o yaml shows a single holder identity governing every built-in controller simultaneously.
Common mistake: Expecting different controllers within the binary to have independent leadership/failover — they all fail over together as one unit.
8. What's the effect of disabling a specific controller via --controllers?
Short answer: It silently breaks any feature that depends on that controller, often in non-obvious ways.
Detailed explanation: For example, disabling serviceaccount-token breaks new pod credential provisioning; the failure won't be an explicit error about the disabled controller — it'll manifest as some downstream feature just not working as expected.
Production example: A hardening pass disables several controllers to reduce attack surface, unaware one of them silently broke automatic ServiceAccount token rotation.
Common mistake: Disabling controllers without a clear understanding of every downstream feature that implicitly depends on them.
9. What does the EndpointSlice controller do and why does it matter for Services?
Short answer: It keeps a Service's EndpointSlice objects in sync with matching Pods' readiness state.
Detailed explanation: As Pods pass or fail readiness probes, this controller adds/removes their IPs from the EndpointSlice(s) backing a Service; kube-proxy then watches those EndpointSlices to program actual routing rules — this controller is the missing link between "Pod is Ready" and "traffic actually reaches it."
Production example: A Pod failing its readiness probe during a bad deploy gets removed from the EndpointSlice within seconds, stopping new traffic from routing to it.
Common mistake: Debugging "Service not routing to my Pod" by looking only at the Pod's status instead of checking whether its IP actually appears in the EndpointSlice.
10–50. Additional rapid-fire questions
10. What controller manages ReplicaSets on behalf of a Deployment? — The Deployment controller.
11. What controller creates/deletes Pods to match a desired replica count? — The ReplicaSet controller.
12. What object does leader election use under the hood? — A Lease object in coordination.k8s.io/v1.
13. What flag enables leader election on kube-controller-manager? — --leader-elect=true.
14. What controls how often the Node controller checks heartbeats? — --node-monitor-period.
15. What controls the grace period before eviction after a node is NotReady? — --pod-eviction-timeout.
16. What's the risk of setting --concurrent-deployment-syncs too high? — Increased load on the API Server and etcd from many simultaneous reconciliation operations.
17. Why does the Namespace controller matter during deletion? — It handles cascading deletion of all resources within a namespace when the namespace itself is deleted.
18. What's an owner reference and why does it matter for garbage collection? — A pointer from a child object (Pod) to its owner (ReplicaSet); when the owner is deleted, Kubernetes' garbage collector cleans up owned children automatically.
19. Why do controllers use informers instead of direct API calls on every reconcile? — Informers maintain a local cache populated via list+watch, avoiding redundant API Server load on every reconciliation pass.
20. What's the Job controller responsible for? — Running batch workloads to completion, tracking success/failure counts against the Job's spec.
21. Why might disabling bootstrapsigner/tokencleaner be acceptable in some clusters? — They're related to legacy bootstrap token cleanup, often unnecessary in clusters not using that bootstrap mechanism.
22. What's the practical effect of controller-manager leader-election flapping? — Reconciliation halts and restarts repeatedly, potentially causing inconsistent partial state changes cluster-wide.
23. How would you confirm which replica is the active controller-manager leader? — kubectl get lease -n kube-system kube-controller-manager -o yaml.
24. Why is reconciliation described as "eventually consistent" rather than instantaneous? — There's inherent latency between an event occurring, the watch firing, and the controller completing its reconcile action.
25. What's a real symptom of the ReplicaSet controller not functioning? — Deleted Pods under a Deployment don't get recreated at all, staying below the desired replica count indefinitely.
26. Why does the ServiceAccount/Token controller matter for new Pods? — It provisions the default credentials a Pod needs to authenticate to the API Server if no explicit ServiceAccount token is specified.
27. What's the relationship between reconciliation loops and idempotency? — Reapplying the same desired state repeatedly should always converge to the same result — reconciliation depends on actions being safely repeatable.
28. Why is "shared fate" a real operational risk of bundling controllers into one binary? — A bug or resource exhaustion in one controller's logic can crash the process, halting every other bundled controller simultaneously.
29. What's a common cause of Deployment rollouts appearing "stuck"? — The Deployment controller's leader (Controller Manager) losing its lease, or PodDisruptionBudgets/resource constraints blocking the ReplicaSet controller's Pod creation.
30. Why must custom controllers respect the same eventual-consistency model as built-in ones? — Kubernetes' entire object model assumes asynchronous reconciliation; a controller expecting synchronous immediate effects will misbehave.
31. What's the danger of writing a controller without a periodic full resync? — It becomes vulnerable to permanent desync if a watch event is ever dropped or missed during a restart/reconnect.
32. Why does the Controller Manager need the same version-skew discipline as other control-plane components? — Its internal API client code must stay compatible with the API Server's served API versions/behavior.
33. What's the effect of a Node controller misconfiguration causing overly aggressive eviction? — Unnecessary reschedule storms triggered by transient network blips rather than genuine node failures.
34. How do you verify the controller chain (Deployment → ReplicaSet → Pod) explicitly? — Inspect ownerReferences on the Pod and ReplicaSet objects via kubectl get ... -o yaml.
35. Why is the reconciliation pattern considered "self-healing"? — Because the loop runs continuously and reacts to any divergence from desired state, including divergence caused by crashes, manual edits, or infrastructure failure.
36. What's the risk of setting node-monitor-grace-period too low? — Nodes get marked NotReady (and Pods potentially evicted) due to brief, harmless network hiccups.
37. What's the risk of setting it too high? — Genuinely failed nodes take longer to be detected, delaying Pod rescheduling and prolonging any resulting service degradation.
38. Why does the Deployment controller create a *new* ReplicaSet during a rolling update rather than editing the existing one? — This is what enables rollback — the old ReplicaSet (with the previous Pod template) is scaled down but kept around, not destroyed.
39. What's a good first diagnostic step when a specific resource type (e.g. Jobs) stops reconciling but everything else works? — Check that specific controller's logs and confirm it wasn't disabled via --controllers.
40. Why is watching controller-manager CPU/memory usage important? — Resource starvation there can cause leader-election lease loss, halting reconciliation cluster-wide, not just for one resource type.
41. What's the difference between a controller reconciling a resource and a controller "owning" it? — Reconciling means acting to converge state; owning specifically means the resource has an ownerReference pointing to this controller's managed object, enabling cascading garbage collection.
42. Why shouldn't you rely on Deployment replica count alone for availability guarantees? — Without topology spread/anti-affinity, the ReplicaSet controller may satisfy the count without spreading Pods across failure domains.
43. What's an operator, in relation to the Controller Manager pattern? — A custom controller (often domain-specific, e.g. a database operator) that encodes operational knowledge into the same reconcile-loop pattern, extending it to custom resources.
44. How does the Controller Manager detect a Deployment's rollout is complete? — By comparing the new ReplicaSet's ready replica count against the Deployment's desired replica count and rollout strategy parameters.
45. Why is "one leader at a time" specifically important for the Node controller? — Two active instances could both decide independently to evict Pods off the same node, causing duplicate or conflicting eviction actions.
46. What's a good way to test a custom controller's resync behavior? — Deliberately drop or delay a watch event in a test environment and confirm the periodic resync still converges state correctly.
47. Why does disabling controllers require careful documentation in a hardened cluster? — Future engineers troubleshooting "missing" behavior have no obvious signal that a controller was intentionally turned off.
48. What's the relationship between PodDisruptionBudgets and the Node controller's eviction behavior? — The Node controller (and voluntary disruption paths) should respect PDBs to avoid dropping below the minimum available replica count during evictions where possible.
49. Why is understanding the full controller chain valuable during an incident? — It tells you exactly which layer to inspect first — Deployment spec, ReplicaSet status, or individual Pod events — instead of guessing.
50. What's the very first thing to check when "nothing self-heals" across the whole cluster? — Whether kube-controller-manager is running and holds its leader-election lease — a stuck lease halts every built-in controller at once.
