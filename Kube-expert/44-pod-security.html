Pod Security

Pod Security Standards and Pod Security Admission — the successor to the removed PodSecurityPolicy.
← Network Policies Course Index Helm →

Explanation

A Pod's spec can request powerful, dangerous capabilities on its node: running as root, mounting the host filesystem, running privileged containers with full kernel access, adding Linux capabilities like SYS_ADMIN. Left unchecked, any user who can create Pods can effectively achieve root on the underlying node — turning "I have permission to deploy an app" into "I have permission to compromise the whole cluster." Pod security controls exist to close this gap by restricting which of these settings a Pod is allowed to request.

Kubernetes' original answer was PodSecurityPolicy (PSP) — a cluster-scoped admission-controller object, removed in Kubernetes 1.25 after years of being deprecated, largely because its API design (indirect binding via RBAC to PSP objects) was notoriously confusing and easy to misconfigure. Its replacement is the Pod Security Standards (PSS), a set of three predefined policy levels, enforced via a built-in admission controller called Pod Security Admission (PSA) that requires no separate object at all — you just label the namespace.

The three PSS levels are: Privileged (unrestricted — for trusted system/infra namespaces like kube-system), Baseline (blocks known privilege escalations — no privileged containers, no host namespaces, no dangerous capabilities — but stays broadly compatible with common workloads), and Restricted (the hardened tier — enforces running as non-root, drops all Linux capabilities except what's explicitly re-added, requires seccompProfile: RuntimeDefault, disallows privilege escalation). Most application namespaces in a mature production cluster should target Restricted.

PSA is applied per-namespace via three independent labels — pod-security.kubernetes.io/enforce, ...warn, and ...audit — each set to one of the three levels. enforce actually rejects non-compliant Pods at admission time; warn and audit let you roll out a stricter level gradually by surfacing violations (as kubectl warnings, or audit log entries) before you flip enforcement on — this staged rollout is the main reason PSA's design is considered a major usability improvement over PSP.

PSA is intentionally coarse — it enforces a fixed menu of security-relevant fields, not arbitrary custom policy. For anything beyond the three standard levels (e.g. "images must come from our internal registry," "all Pods must have resource limits"), production clusters layer a policy engine like Kyverno or OPA Gatekeeper on top via validating/mutating admission webhooks.

Internal Working / Interactions

  • API Server (built-in admission plugin): Pod Security Admission runs as a built-in (not webhook-based) admission controller, checking each Pod's spec against the level(s) configured for its namespace.
  • Namespace object: the source of truth for which PSS level applies — read via labels at admission time, no separate cluster-scoped policy object needed.
  • kubelet / container runtime: ultimately enforce the resulting security context (seccomp, capabilities, non-root UID) when actually starting the container via the CRI.
  • External policy engines (Kyverno/OPA): run as validating/mutating admission webhooks alongside PSA for rules beyond the fixed PSS menu.

Code Example

# 1. Namespace enforcing the Restricted Pod Security Standard,
#    warning/auditing against it too so violations are visible even before enforce is tightened further
apiVersion: v1
kind: Namespace
metadata:
  name: payments
  labels:
    pod-security.kubernetes.io/enforce: restricted   # Rejects non-compliant Pods.
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted        # Surfaces warnings for anything still borderline.
    pod-security.kubernetes.io/audit: restricted        # Logs violations to the audit log.
---
# 2. A Pod spec that actually satisfies the Restricted standard
apiVersion: v1
kind: Pod
metadata:
  name: checkout-worker
  namespace: payments
spec:
  securityContext:
    runAsNonRoot: true            # Restricted requires this — no running as UID 0.
    runAsUser: 10001
    seccompProfile:
      type: RuntimeDefault        # Restricted requires an explicit seccomp profile.
  containers:
    - name: worker
      image: registry.internal/checkout-worker:3.1.0
      securityContext:
        allowPrivilegeEscalation: false   # Blocks setuid-style privilege escalation.
        capabilities:
          drop: ["ALL"]                    # Restricted requires dropping all capabilities...
          add: ["NET_BIND_SERVICE"]        # ...then only re-adding what's explicitly needed.
        readOnlyRootFilesystem: true

kubectl Commands

kubectl label ns payments pod-security.kubernetes.io/enforce=restricted   # Apply enforcement to a namespace
kubectl label ns payments pod-security.kubernetes.io/warn=restricted --overwrite  # Add warn-only staging first
kubectl apply --dry-run=server -f pod.yaml                                  # See PSA warnings before real apply
kubectl get ns --show-labels | grep pod-security                            # Audit which namespaces enforce what
kubectl get events -n payments --field-selector reason=FailedCreate          # See PSA rejections as events
kubectl auth can-i create pods --as=system:serviceaccount:payments:default -n payments  # Confirm identity separately from PSA

Best Practices & Common Pitfalls

Best Practices

  • Default every new application namespace to at least Baseline, target Restricted for anything handling sensitive data.
  • Roll out stricter levels via warn/audit first, then flip enforce once violations are cleared.
  • Reserve Privileged strictly for trusted system namespaces (CNI, storage drivers) that genuinely need host access.
  • Layer Kyverno/OPA Gatekeeper on top of PSA for org-specific rules PSA doesn't cover.

Pitfalls

  • Assuming PodSecurityPolicy still exists — it was removed in 1.25; any cluster still referencing it will break on upgrade.
  • Enforcing Restricted cluster-wide in one shot without a warn/audit staging period, breaking legitimate workloads.
  • Forgetting PSA is namespace-scoped only — it can't express per-workload exceptions within the same namespace.
  • Confusing PSA (built-in, fixed levels) with a full policy engine — PSA won't enforce custom business rules.

50 Interview Questions

1. Why was PodSecurityPolicy (PSP) removed in Kubernetes 1.25?
Short answer: Its API design — indirect binding via RBAC to cluster-scoped PSP objects — was notoriously confusing and easy to misconfigure.
Detail: Determining which PSP actually applied to a given Pod required tracing through RBAC bindings rather than a direct, readable relationship, making audits and troubleshooting difficult.
Production example: Clusters upgrading to 1.25+ that still referenced PSP objects broke outright and had to migrate to Pod Security Admission beforehand.
Common mistake: Assuming PSP still exists in a modern cluster and referencing it in new tooling or docs.
2. What are the three Pod Security Standards (PSS) levels and what does each restrict?
Short answer: Privileged (unrestricted), Baseline (blocks known privilege escalations), Restricted (hardened — non-root, dropped capabilities, seccomp required).
Detail: Privileged is meant for trusted system namespaces; Baseline stays broadly compatible while blocking dangerous settings; Restricted is the tier most application namespaces handling sensitive data should target.
Production example: kube-system stays Privileged for CNI/storage driver Pods, while a payments namespace enforces Restricted.
Common mistake: Applying Restricted cluster-wide in one shot without checking which existing workloads would break.
3. How is Pod Security Admission (PSA) configured — via an object or something else?
Short answer: Via three labels on the Namespace object itself — no separate policy object needed at all.
Detail: pod-security.kubernetes.io/enforce, ...warn, and ...audit, each independently set to one of the three levels.
Production example: kubectl label ns payments pod-security.kubernetes.io/enforce=restricted applies enforcement with a single command.
Common mistake: Looking for a PSA CRD or cluster-scoped object to create — PSA is intentionally label-driven, not object-driven.
4. What's the difference between the enforce, warn, and audit PSA labels?
Short answer: enforce actually rejects non-compliant Pods at admission; warn surfaces a kubectl warning without blocking; audit logs a violation without blocking or warning interactively.
Detail: This three-tier design lets teams roll out a stricter level gradually — turning on warn/audit first to observe violations, then flipping enforce once things are clean.
Production example: Setting warn=restricted and audit=restricted while enforce stays at baseline, to see what would break before tightening.
Common mistake: Setting only enforce to a strict level with no staged warn/audit period first, causing surprise Pod rejections.
5. What specific fields does the Restricted PSS level require on a Pod?
Short answer: runAsNonRoot: true, seccompProfile.type: RuntimeDefault, allowPrivilegeEscalation: false, and capabilities.drop: ["ALL"] (with only explicit re-adds allowed).
Detail: These map directly onto the securityContext at both Pod and container level; missing any of them causes admission rejection under enforce=restricted.
Production example: A checkout-worker Pod explicitly setting runAsUser: 10001, dropping ALL capabilities, then re-adding only NET_BIND_SERVICE.
Common mistake: Dropping ALL capabilities but forgetting to re-add one the app actually needs (like NET_BIND_SERVICE for binding to low ports), causing runtime failures instead of admission failures.
6. What kind of admission controller is Pod Security Admission — webhook-based or built-in?
Short answer: Built-in — it runs as part of the API Server binary itself, not as an external webhook call.
Detail: This makes it faster and more reliable than webhook-based checks (no network round-trip, no webhook availability dependency) but also fixed in scope to the three predefined PSS levels.
Production example: PSA continues functioning even if all external admission webhooks (Kyverno, OPA) are temporarily down, since it's compiled into the API Server.
Common mistake: Assuming PSA has the same extensibility as a webhook-based policy engine — it doesn't; it's fixed-menu by design.
7. Why do production clusters still layer Kyverno or OPA Gatekeeper on top of PSA?
Short answer: PSA only enforces a fixed set of security-relevant fields; it can't express arbitrary custom business rules.
Detail: Rules like "images must come from our internal registry" or "all Pods must declare resource limits" fall outside PSS's scope entirely and need a general-purpose policy engine.
Production example: Kyverno enforcing "no :latest image tags" alongside PSA enforcing Restricted-level security context fields.
Common mistake: Expecting PSA alone to cover organizational compliance requirements beyond core Pod security settings.
8. Is PSA scoped per-namespace or can it express per-workload exceptions within a namespace?
Short answer: Namespace-scoped only — it cannot carve out per-workload exceptions inside a single namespace.
Detail: If one workload in a namespace genuinely needs a Baseline-level exception while everything else is Restricted, you need a separate namespace for it, since PSA has no per-Pod override mechanism.
Production example: Splitting a legacy privileged sidecar into its own namespace rather than weakening the whole namespace's enforce level for it.
Common mistake: Trying to annotate a single Pod to bypass its namespace's PSA enforce level — there's no such per-Pod escape hatch in PSA itself.
9. What ultimately enforces the security context fields PSA validates?
Short answer: kubelet and the container runtime (via the CRI), which actually apply seccomp, capabilities, and non-root UID when starting the container.
Detail: PSA's job stops at admission-time rejection of non-compliant specs; the runtime enforcement of "actually run as this UID, actually apply this seccomp profile" happens later, at container start.
Production example: A Pod passing PSA admission still relies on containerd/CRI-O correctly applying the requested seccompProfile at runtime.
Common mistake: Believing PSA itself sandboxes the container — it only validates the declared intent; the runtime does the actual enforcement.
10. How would you preview PSA violations before actually applying a Pod manifest?
Short answer: Use kubectl apply --dry-run=server -f pod.yaml, which runs the request through real admission (including PSA) without persisting it.
Detail: Server-side dry-run exercises the full admission chain, so PSA warnings/rejections show up exactly as they would on a real apply.
Production example: CI pipelines running dry-run applies against a staging cluster with the target namespace's real PSA labels before merging a PR.
Common mistake: Using client-side dry-run (`--dry-run=client`), which never contacts the API Server and so skips admission checks entirely.
11–50. Additional rapid-fire questions
11. What replaced PodSecurityPolicy? — Pod Security Standards enforced via Pod Security Admission.
12. Which PSS level is meant for kube-system-style trusted namespaces? — Privileged.
13. Which PSS level should most application namespaces handling sensitive data target? — Restricted.
14. How is a namespace opted into PSA enforcement? — By labeling it with pod-security.kubernetes.io/enforce=<level>.
15. What does the enforce-version label control? — Which PSS spec version's rule-set to apply (e.g. "latest" tracks the newest release's definitions).
16. Does PSA require restarting the API Server to take effect on a namespace? — No, it's evaluated live on every Pod admission based on current namespace labels.
17. What field disallows setuid-style privilege escalation? — allowPrivilegeEscalation: false.
18. What capabilities does Restricted require dropping? — ALL, with only explicitly needed ones re-added.
19. What seccomp profile type does Restricted require? — RuntimeDefault.
20. Can a Pod run as root under Baseline? — Baseline doesn't require non-root, only Restricted does.
21. What kubectl command shows PSA rejection events? — kubectl get events --field-selector reason=FailedCreate.
22. How do you audit which namespaces enforce which PSS level? — kubectl get ns --show-labels | grep pod-security.
23. Is PSA a webhook or compiled into the API Server? — Compiled in as a built-in admission plugin.
24. Can PSA express "images must come from this registry"? — No, that's outside PSS's fixed scope — needs Kyverno/OPA.
25. What happens to a Pod that violates enforce level? — It's rejected at admission and never created.
26. What happens to a Pod that only violates warn level? — It's still admitted; kubectl shows a warning message.
27. Can a single namespace have different enforce and warn levels simultaneously? — Yes, they're independent labels — e.g. enforce=baseline, warn=restricted.
28. What's the risk of jumping straight to enforce=restricted cluster-wide? — Breaking legitimate workloads that weren't already compliant, without any staged visibility first.
29. Does PSA replace the need for NetworkPolicy? — No — PSA governs container/Pod privilege settings, NetworkPolicy governs network traffic; they're orthogonal.
30. What does readOnlyRootFilesystem: true accomplish alongside Restricted? — Prevents writes to the container's root filesystem, reducing tamper/persistence risk, though not itself a PSS-mandated field.
31. Is PSA per-cluster or per-namespace configurable? — Per-namespace via labels (a cluster-wide default can also be set via AdmissionConfiguration).
32. What replaced the old PSP-to-RBAC binding indirection? — Direct namespace labeling — no binding object required at all.
33. Can PSA enforce resource requests/limits presence? — No, that's outside its scope — needs LimitRanger admission or a policy engine.
34. Why is Restricted considered the hardened tier? — It closes off nearly all privilege-escalation vectors: root, capabilities, privilege escalation, and unconfined syscalls.
35. What's a safe way to validate a Restricted rollout without breaking prod? — Apply warn/audit=restricted first, review logs/warnings across a full deploy cycle, then flip enforce.
36. Does removing the pod-security labels from a namespace disable enforcement? — Yes, an unlabeled namespace has no PSA enforcement (defaults apply per cluster admission config, often Privileged).
37. Can a Baseline-compliant Pod also pass Restricted? — Not necessarily — Restricted adds stricter requirements (non-root, dropped capabilities) Baseline doesn't require.
38. What's the relationship between PSA and the container runtime's CRI? — PSA validates the declared spec at admission; the CRI/runtime applies the actual OS-level enforcement when starting the container.
39. Why can't PSA alone stop a malicious image from being pulled? — It only checks security-context fields, not image provenance or content — that's a policy-engine or admission-webhook concern.
40. What's the audit label's use case if enforce already blocks violations? — Auditing a stricter level than what's currently enforced, to gather compliance evidence before eventually raising enforce.
41. Is there a way to see PSA warnings without applying the Pod? — Yes, via server-side dry-run (`--dry-run=server`).
42. Can PSA be bypassed by a cluster-admin creating Pods directly via etcd access? — Technically yes if bypassing the API Server entirely, but that's outside normal operational access and considered a severe security violation itself.
43. What's a common mistake when re-adding capabilities under Restricted? — Forgetting the app actually needs a specific capability (e.g. NET_BIND_SERVICE) and only discovering it via a runtime crash, not an admission error.
44. Does Pod Security Admission validate init containers too? — Yes, all containers in the Pod spec (including init and ephemeral containers) are checked against the enforced level.
45. How does PSA interact with existing running Pods when a namespace's enforce label is tightened? — Already-running Pods aren't retroactively evicted; enforcement only applies to new admission requests (create/update).
46. What's the benefit of PSA being label-driven over object-driven (like PSP)? — Directly visible on `kubectl get ns --show-labels` with no indirection through bindings to trace.
47. Can third-party policy engines fully replace PSA? — Yes technically, but most clusters use PSA for the baseline built-in guarantee and layer engines for anything extra, rather than replacing it.
48. What's the risk of using enforce-version: latest in a long-lived namespace? — Behavior can change on cluster upgrades as "latest" tracks new Kubernetes releases' PSS definitions, potentially causing new rejections after an upgrade.
49. How would you migrate a legacy PSP-based cluster to PSA? — Map each PSP's allowed settings to the closest PSS level, apply as warn/audit first, verify via events/logs, then enforce.
50. What's the first thing to check when a Pod mysteriously fails to create with no application-level error? — Whether PSA's enforce level for that namespace rejected the Pod spec — check events for a FailedCreate reason.
