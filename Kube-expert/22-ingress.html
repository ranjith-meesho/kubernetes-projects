Ingress

L7 HTTP(S) routing into the cluster — host/path rules, TLS termination, and the Controller that actually implements them.
← Networking Course Index Storage →

Explanation

A Service of type ClusterIP or NodePort gives you L4 (TCP/UDP) access, but it has no idea about HTTP hostnames, URL paths, or TLS certificates. Once you have more than a handful of HTTP services, exposing each one via its own LoadBalancer Service becomes expensive and unmanageable — every Service gets its own cloud load balancer, its own IP, its own cost line item. Ingress solves this by giving you a single entry point that fans out to many backend Services based on HTTP-level rules: host, path, and headers.

Kubernetes deliberately splits this into two objects. The Ingress resource itself is just a declarative set of routing rules — "requests for shop.example.com/api go to the api-svc Service on port 80." It is inert on its own. The actual work — running a reverse proxy, watching Ingress objects, reloading config, terminating TLS — is done by an Ingress Controller, a separate workload you deploy (NGINX Ingress Controller, Traefik, HAProxy, cloud-provider ALB/GCLB controllers, Contour, Istio Gateway, etc.). Kubernetes ships the API shape but not the implementation, which is why "installing Ingress" always means installing a controller first.

IngressClass is the glue between the two. When multiple controllers run in the same cluster (common during a migration, or when different teams need different proxies), ingressClassName on the Ingress object tells Kubernetes which controller should honor that particular Ingress. Without it, controllers either process everything (if they claim to be default) or nothing.

TLS termination is one of Ingress's most valuable jobs: the Ingress spec references a Secret of type kubernetes.io/tls containing a certificate and key, and the controller terminates HTTPS at the edge so your backend Pods can stay plain HTTP. Combined with cert-manager, this becomes fully automated (Let's Encrypt certificates requested, renewed, and mounted without human involvement).

In production, Ingress is the boundary where you also enforce things that don't belong inside application code: rate limiting, WAF rules, canary/traffic-splitting via annotations, and centralized access logging. Because it sits at the edge, a misconfigured or overloaded Ingress Controller is a single point of failure for every HTTP service behind it — which is why controllers are run as a Deployment with multiple replicas and their own resource requests/limits, monitoring, and often a dedicated node pool.

Internal Working / Interactions

  • API Server / etcd — the Ingress object is stored like any other resource; the Ingress Controller runs a watch/list against the API Server for Ingress, Service, Endpoints/EndpointSlice, and Secret objects.
  • Ingress Controller — reconciles Ingress specs into live proxy configuration (e.g. NGINX config + reload, or Envoy xDS updates with no reload needed).
  • kube-proxy / Service — the controller ultimately forwards traffic to a backend Service's ClusterIP (or directly to Pod IPs via EndpointSlice, bypassing kube-proxy, depending on the controller).
  • cert-manager (optional) — watches Ingress annotations/Certificate objects and populates the TLS Secret the Ingress references.
  • External LoadBalancer / cloud controller manager — typically the Ingress Controller itself is exposed via one LoadBalancer Service, so only one cloud LB is provisioned for the whole cluster regardless of how many Ingress objects exist.

Code Example

# IngressClass — declares which controller implements it and whether it's the cluster default
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"   # used when Ingress omits ingressClassName
spec:
  controller: k8s.io/ingress-nginx                        # must match the controller's own identifier

---
# Ingress — host/path routing + TLS termination
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: shop-ingress
  namespace: shop
  annotations:
    nginx.ingress.controller/rewrite-target: /             # controller-specific behavior tweak
    cert-manager.io/cluster-issuer: letsencrypt-prod        # cert-manager auto-requests the TLS cert
spec:
  ingressClassName: nginx                                  # binds this Ingress to the nginx controller
  tls:
  - hosts:
    - shop.example.com
    secretName: shop-tls                                    # Secret (type kubernetes.io/tls) holding cert+key
  rules:
  - host: shop.example.com                                  # only matches requests for this Host header
    http:
      paths:
      - path: /api                                          # path-based routing
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 80
      - path: /                                             # catch-all -> frontend
        pathType: Prefix
        backend:
          service:
            name: web-svc
            port:
              number: 80

Best Practices & Common Pitfalls

Do
  • Always set ingressClassName explicitly rather than relying on a "default" class.
  • Automate certificates with cert-manager instead of hand-rotating TLS secrets.
  • Run the Ingress Controller with multiple replicas and PodDisruptionBudgets.
  • Use path pathType: Prefix/Exact deliberately — don't leave it ambiguous.
Pitfalls
  • Forgetting the Ingress Controller is a separate install — the Ingress object alone does nothing.
  • Overloading one Ingress Controller for every namespace with no resource limits, causing a shared outage.
  • Annotation drift: controller-specific annotations (NGINX vs ALB vs Traefik) are not portable.
  • Mismatched host rules silently falling through to a default backend (404s that look like bugs).

50 Interview Questions

1. Why does Kubernetes split Ingress into an Ingress object and a separate Ingress Controller?
Short answer: Kubernetes ships only the API shape (routing rules); the actual reverse-proxy implementation is left to a pluggable controller you install separately.
Detailed explanation: The Ingress resource is inert declarative YAML — host/path rules and a TLS secret reference. Nothing watches or acts on it until a controller (NGINX, Traefik, ALB, Contour, Istio Gateway, etc.) is deployed to reconcile it into live proxy config.
Production example: A cluster with no Ingress Controller installed will show Ingress objects as Pending forever with no external IP — the object alone does nothing.
Common mistake: Assuming applying an Ingress YAML is sufficient without first installing a controller.
2. What is the purpose of IngressClass and ingressClassName?
Short answer: IngressClass tells Kubernetes which controller should implement a given Ingress when multiple controllers run in the same cluster.
Detailed explanation: Each IngressClass maps to a controller identifier (e.g. k8s.io/ingress-nginx). Setting ingressClassName on an Ingress binds it to exactly one controller; omitting it relies on whichever IngressClass is marked default.
Production example: During a migration from NGINX to a cloud ALB controller, both run side by side and ingressClassName decides which Ingress objects each one processes.
Common mistake: Leaving ingressClassName unset and assuming a specific controller will pick it up — behavior depends entirely on which class is marked default.
3. How does TLS termination work with Ingress?
Short answer: The Ingress spec's tls block references a Secret of type kubernetes.io/tls, and the controller terminates HTTPS at the edge using that cert/key.
Detailed explanation: Backend Pods can remain plain HTTP since the controller handles the TLS handshake itself, decrypting traffic before proxying it internally.
Production example: Combining this with cert-manager automates requesting and renewing Let's Encrypt certificates without any human touching the Secret.
Common mistake: Forgetting the referenced Secret must exist in the same namespace as the Ingress, or hand-rotating certs and letting them silently expire.
4. Why is one misbehaving Ingress Controller a single point of failure for the whole cluster's HTTP traffic?
Short answer: Every Ingress typically routes through the same small set of controller Pods, so overload or crash there affects every HTTP service behind it.
Detailed explanation: Because the controller is exposed via one shared LoadBalancer Service, all inbound HTTP traffic funnels through it — there's no per-Ingress isolation unless you run multiple controller instances/classes.
Production example: A traffic spike to one heavily-used Ingress can exhaust the controller's resources and degrade completely unrelated services behind the same controller.
Common mistake: Running a single controller replica with no resource requests/limits or PodDisruptionBudget.
5. How does traffic actually reach a Pod after passing through the Ingress Controller?
Short answer: The controller matches the request's host/path against Ingress rules, then forwards it to the backend Service's ClusterIP or directly to Pod IPs via EndpointSlice.
Detailed explanation: Depending on the controller, it may go through kube-proxy (via ClusterIP) or bypass it entirely by reading EndpointSlice objects and load-balancing directly to Pod IPs for lower latency.
Production example: NGINX Ingress Controller by default proxies to Pod IPs directly rather than routing through the Service's ClusterIP, for better load distribution.
Common mistake: Assuming all Ingress Controllers route the same way — behavior varies by implementation.
6. What happens when multiple Ingress objects define conflicting rules for the same host and path?
Short answer: Behavior is controller-specific and often ambiguous — Kubernetes' Ingress spec doesn't define strict conflict-resolution semantics.
Detailed explanation: Most controllers merge rules across all Ingress objects they watch; conflicting paths can result in unpredictable precedence (e.g. by object creation time or alphabetical Ingress name) rather than a clean error.
Production example: Two teams both defining shop.example.com/api in separate namespaces can cause routing that depends on the controller's internal tie-breaking, not intent.
Common mistake: Assuming Kubernetes validates and prevents host/path overlap across Ingress objects — it generally doesn't.
7. Why do controller-specific annotations (like NGINX rewrite-target) create portability problems?
Short answer: Annotations are the escape hatch for behavior the core Ingress spec doesn't standardize, so they're tied to one controller's implementation.
Detailed explanation: Features like rewrite rules, rate limiting, and canary weighting aren't part of the portable Ingress API — each controller invents its own annotation namespace for them.
Production example: Migrating from NGINX Ingress to an ALB Ingress Controller requires rewriting every nginx.ingress.kubernetes.io/* annotation to its ALB equivalent, or losing that behavior.
Common mistake: Writing critical routing/security behavior purely via annotations without documenting the controller dependency.
8. How does cert-manager integrate with the Ingress TLS flow?
Short answer: cert-manager watches Ingress annotations (or Certificate objects) and populates the TLS Secret the Ingress references, automating issuance and renewal.
Detailed explanation: An annotation like cert-manager.io/cluster-issuer: letsencrypt-prod triggers cert-manager to run the ACME challenge and write the resulting cert/key into the named Secret before the controller needs it.
Production example: A new Ingress for shop.example.com gets a valid Let's Encrypt cert within minutes with zero manual cert handling.
Common mistake: Creating the TLS Secret manually and cert-manager expecting to own/manage it, causing conflicting updates or renewal failures.
9. Why does a mismatched host rule silently fall through to a default backend instead of erroring?
Short answer: If no Ingress rule matches the request's Host header, the controller falls back to its configured default backend (often a 404 page), not an explicit error.
Detailed explanation: This is by design so the controller always has somewhere to send unmatched traffic, but it means typos in host: fields produce confusing generic 404s rather than obvious configuration errors.
Production example: A typo like shop.exmaple.com in the Ingress silently routes real traffic for shop.example.com to the default backend instead of the intended Service.
Common mistake: Debugging a "the Service isn't working" ticket without first checking kubectl describe ingress for the exact host string.
10. Why is exposing every HTTP Service via its own LoadBalancer Service considered expensive and unmanageable?
Short answer: Each LoadBalancer Service typically provisions its own cloud load balancer and IP, multiplying cost and operational overhead per service.
Detailed explanation: Ingress consolidates many HTTP services behind a single L7 entry point and a single cloud LB, fanning traffic out internally based on host/path instead of provisioning infrastructure per service.
Production example: A cluster running 40 HTTP microservices needs only one Ingress Controller LoadBalancer instead of 40 separate cloud load balancers.
Common mistake: Defaulting every new HTTP Service to type LoadBalancer instead of routing it through the shared Ingress.
11–50. Additional rapid-fire questions
11. What API group does the Ingress resource belong to? — networking.k8s.io/v1.
12. Can an Ingress route to a Service in a different namespace? — No, an Ingress can only reference Services in its own namespace.
13. What's the difference between pathType: Prefix and pathType: Exact? — Prefix matches any path starting with the given segment; Exact requires a literal full match.
14. What does the is-default-class annotation on IngressClass do? — Marks that controller as the fallback for any Ingress that omits ingressClassName.
15. Why does an Ingress Controller need to watch Service and EndpointSlice objects, not just Ingress? — It must resolve backend Service names into actual routable Pod IPs/ports.
16. How would you find the external IP of your Ingress Controller? — kubectl get svc -n ingress-nginx to see its LoadBalancer Service's external IP.
17. What object type does the Ingress's tls.secretName field expect? — A Secret of type kubernetes.io/tls containing tls.crt and tls.key.
18. Can one Ingress object handle multiple hostnames? — Yes, via multiple entries under spec.rules, each with its own host.
19. Why might Ingress reconciliation involve a config reload for NGINX but not for Envoy-based controllers? — NGINX reloads its worker config file on change; Envoy-based controllers (e.g. Contour, Istio) push updates via xDS without a reload.
20. What's a practical reason to run a dedicated node pool for Ingress Controller Pods? — Isolates edge traffic load from application workloads and protects it from noisy-neighbor resource contention.
21. How does canary/traffic-splitting typically get implemented at the Ingress layer? — Via controller-specific annotations that weight traffic between two Ingress objects or backend Services.
22. What happens to in-flight requests when Ingress Controller Pods are rolling-updated? — Depends on graceful shutdown/PreStop hooks and readiness gates; without them, requests can be dropped mid-update.
23. Why is rate limiting usually enforced at the Ingress layer rather than in application code? — Centralizes protection at the single edge entry point instead of duplicating logic across every backend service.
24. What's the risk of a WAF rule configured incorrectly on the Ingress Controller? — It can block legitimate production traffic cluster-wide since it sits in front of every routed Service.
25. How do you check which controller actually implements an Ingress? — kubectl get ingress -o yaml and inspect spec.ingressClassName, then kubectl get ingressclass to see its controller field.
26. Why does Ingress not support raw TCP/UDP routing? — It's an L7 (HTTP-aware) abstraction; non-HTTP protocols need a Service of type LoadBalancer/NodePort or a controller-specific TCP/UDP ConfigMap extension.
27. What's a common cause of 502/504 errors from an Ingress Controller? — Backend Pods not ready, wrong Service port mapping, or backend timeouts exceeding the controller's proxy timeout settings.
28. How can you test Ingress routing without waiting for DNS propagation? — Use curl --resolve host:port:IP or set a local /etc/hosts entry pointing to the controller's external IP.
29. Why might a cloud-provider Ingress Controller (like ALB) not need a separate LoadBalancer Service? — It provisions and manages the cloud load balancer directly via annotations, without an intermediate Kubernetes Service.
30. What's the purpose of centralized access logging at the Ingress layer? — Gives a single place to observe all HTTP traffic entering the cluster instead of instrumenting every service individually.
31. Can Ingress enforce mutual TLS (mTLS) to clients? — Yes, many controllers support client-certificate verification via annotations/configuration at the edge.
32. Why must the Ingress Controller itself have proper resource requests/limits set? — Without them it can be starved or OOM-killed under load, taking down routing for every backend Service.
33. What does kubectl describe ingress show that kubectl get ingress doesn't? — Detailed rule breakdown, backend resolution status, and sync/admission events.
34. Why is it risky to rely on an "is-default-class" IngressClass in a multi-team cluster? — A new Ingress that forgets ingressClassName could silently land on the wrong controller.
35. How does an Ingress interact with NetworkPolicies? — NetworkPolicies still apply to traffic from the controller Pods to backend Pods, so overly strict policies can block legitimate Ingress-routed traffic.
36. What's the operational value of running the Ingress Controller as a Deployment with multiple replicas? — High availability — one replica crashing doesn't take down edge routing entirely.
37. Why would you prefer path-based routing over separate subdomains for microservices? — Fewer DNS records and certificates to manage, and a single Ingress can express the whole routing table.
38. What's a security reason to avoid wildcard TLS certs referenced by many Ingress objects? — A single compromised private key exposes every hostname covered by that wildcard, unlike scoped per-host certs.
39. How would a rolling DNS-based blue/green switch differ from Ingress-based traffic splitting? — DNS switching is coarse and slow to propagate/rollback; Ingress-layer splitting gives instant, fine-grained control without DNS TTL delays.
40. What field would you check first if an Ingress shows no address after creation? — Ingress Controller Pod health and its LoadBalancer Service's status.loadBalancer field.
41. Why does annotation-driven configuration make GitOps auditing of Ingress behavior harder? — Meaningful routing/security behavior lives in opaque annotation strings rather than structured, versioned spec fields.
42. What is the effect of setting an overly aggressive proxy-read-timeout annotation? — Long-running backend requests get killed early, causing spurious client-facing timeouts.
43. Why might two teams' Ingress objects in different namespaces need coordination even though Ingress is namespaced? — They can still collide on the same hostname served by the same shared controller.
44. What's the benefit of the aggregation of TLS termination at Ingress versus per-Pod TLS? — Centralizes certificate management and reduces CPU overhead of TLS handshakes duplicated across every Pod.
45. How do sticky sessions typically get implemented at the Ingress layer? — Via controller-specific cookie-based affinity annotations that pin a client to the same backend Pod.
46. Why is it good practice to keep the Ingress Controller's version in step with the Kubernetes API version? — Newer Ingress API fields/behaviors may require a controller version that understands them, per the compatibility matrix.
47. What's a quick way to confirm an Ingress Controller is actually watching a given Ingress object? — Check the controller's logs/events for a reconciliation entry referencing that Ingress's name and namespace.
48. Why would you choose Contour/Envoy over NGINX Ingress for very high-churn Ingress environments? — Envoy-based xDS updates avoid full config reloads, reducing latency spikes when Ingress rules change frequently.
49. What's the risk of not setting a PodDisruptionBudget on the Ingress Controller Deployment? — A voluntary disruption (node drain, cluster upgrade) could evict all replicas simultaneously, causing an outage.
50. How would you debug an Ingress that routes correctly for HTTP but fails for HTTPS? — Check the TLS Secret exists and is valid, matches the requested host (SAN), and that the controller's TLS listener/annotations are correctly configured.
