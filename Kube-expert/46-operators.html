Operators

Custom controllers that encode a human operator's knowledge of running a specific piece of software.
← Helm Course Index CRDs →

Explanation

Kubernetes' built-in controllers (Deployment, ReplicaSet, Job) know how to run generic workloads — start N copies of a container, restart it if it dies. But operationally complex, stateful software — a Postgres cluster, Kafka, Elasticsearch — needs far more than that: initializing a cluster, electing a primary, performing safe version upgrades one replica at a time, taking backups, handling failover, resizing storage. A human "operator" (the job title) traditionally did this by hand or via runbooks. The Operator pattern encodes that same operational knowledge into software: a custom controller that watches a Custom Resource representing the desired state of that application and continuously drives the real system towards it.

Kubernetes needs this because its built-in primitives are intentionally generic — they can't know that "upgrading Postgres safely" means promoting a replica before touching the primary, or that "scaling Kafka" means rebalancing partitions rather than just adding empty brokers. Operators close this gap without polluting the Kubernetes core with application-specific logic — every Operator is just an ordinary custom controller plus a CRD, running the exact same watch/reconcile loop as core Kubernetes controllers, deployed as a regular Pod.

The core building block is the reconcile loop: the Operator watches its Custom Resource (e.g. a PostgresCluster object) via the API Server, and on every change (or periodic resync) runs a Reconcile(request) function that reads the CR's desired spec, inspects the actual state of the cluster (by querying the database, checking StatefulSet/Pod status, etc.), and takes whatever action closes the gap — creating a Pod, promoting a replica, updating a ConfigMap. Crucially this function is expected to be idempotent and level-triggered: it must produce the correct result whether called once or a hundred times for the same state, exactly like Kubernetes' own core controllers.

Most production Operators are built with Operator SDK or kubebuilder — scaffolding frameworks built on top of controller-runtime, the same Go library the Kubernetes project's own core controllers are built with. These frameworks generate the boilerplate (CRD schema, RBAC manifests, the watch/informer setup) so the author focuses purely on the reconcile logic. Famous production Operators include the Postgres Operator (Zalando/CrunchyData), the Prometheus Operator, and the Strimzi Kafka Operator — each is effectively a domain expert's runbook turned into always-on software.

The Operator "maturity model" ranges from Level 1 (basic install/config) to Level 5 (full auto-pilot: automated upgrades, auto-scaling, auto-tuning, self-healing from complex failure modes) — most real-world Operators sit around Level 2-3, automating day-2 operations like backups and minor-version upgrades but still requiring human judgment for major version jumps or capacity planning.

Internal Working / Interactions

  • API Server: serves the Operator's CRD like any other resource type; the Operator authenticates via its own ServiceAccount and RBAC, same as any client.
  • Custom Resource (CR): the desired-state object the Operator watches (e.g. kind: PostgresCluster) — analogous to a Deployment's spec, but application-specific.
  • Operator Pod (controller-runtime): runs the watch/informer against the CR and any owned resources (Pods, StatefulSets, Secrets), invoking Reconcile on changes.
  • Owned native resources: the Operator typically creates and owns standard objects (StatefulSet, Service, PVC, Secret) under the hood, setting ownerReferences so garbage collection cleans them up if the CR is deleted.
  • External systems: many Operators also talk directly to the application itself (e.g. a SQL connection to Postgres to run ALTER SYSTEM) — this is where Operators go beyond what core Kubernetes controllers ever do.

Code Example

# Custom Resource an Operator watches — declarative desired state for a Postgres cluster
apiVersion: postgres-operator.example.com/v1
kind: PostgresCluster
metadata:
  name: orders-db
  namespace: payments
spec:
  version: "15"                 # Operator handles version-specific upgrade logic internally.
  instances: 3                  # Operator creates 1 primary + 2 replicas, wires up streaming replication.
  storage:
    size: 100Gi
    storageClassName: gp3
  backup:
    schedule: "0 2 * * *"        # Operator runs this via its own reconcile logic + a backup tool (e.g. pgBackRest).
    retention: "14d"
  resources:
    requests: { cpu: "1", memory: 4Gi }
    limits:   { cpu: "2", memory: 8Gi }
status:                          # Written ONLY by the Operator, never by the user.
  currentPrimary: orders-db-0
  replicationLag: "120ms"
  phase: Running

kubectl get postgresclusters -n payments                  # List custom resources managed by this Operator

Best Practices & Common Pitfalls

Best Practices

  • Only build/adopt an Operator when the operational complexity genuinely justifies it — not every app needs one.
  • Ensure the reconcile function is idempotent and safe to run repeatedly against the same state.
  • Scope the Operator's RBAC tightly to the namespaces/resources it actually manages.
  • Prefer well-maintained community Operators (Strimzi, Prometheus Operator, CloudNativePG) over building bespoke ones unless truly necessary.

Pitfalls

  • Writing non-idempotent reconcile logic that behaves differently depending on how many times it's called.
  • Granting an Operator cluster-admin "to be safe" instead of least-privilege RBAC scoped to its CRD and owned resources.
  • Ignoring the Operator's own health/version — an outdated Operator can silently fail to support a newer application version.
  • Expecting an Operator to replace all human judgment — most sit at maturity Level 2-3, not full autopilot.

50 Interview Questions

1. What problem does the Operator pattern solve that Deployments/ReplicaSets don't?
Short answer: Built-in controllers only know how to run generic workloads; Operators encode application-specific operational knowledge (failover, backups, safe upgrades) for complex stateful software.
Detail: A Deployment can restart a dead container, but it has no concept of "promote a replica before touching the primary" or "rebalance partitions when scaling" — that domain expertise has to live somewhere, and an Operator is where it lives as code.
Production example: A Postgres Operator handling primary election and safe one-replica-at-a-time version upgrades, which a plain StatefulSet has no way to express.
Common mistake: Building an Operator for an application simple enough that a Deployment/StatefulSet already handles its needs.
2. What are the two things every Operator is composed of?
Short answer: A CustomResourceDefinition (the schema for desired state) and a custom controller (the reconcile loop that acts on it).
Detail: The CRD alone is inert structured data; the controller is what actually watches it and drives real-world state towards what it declares — together they form the full extension pattern.
Production example: The PostgresCluster CRD paired with the Operator Pod running its reconcile loop against instances of that CRD.
Common mistake: Shipping a CRD without a controller watching it, leaving custom resources as unenforced data.
3. Why must an Operator's Reconcile function be idempotent and level-triggered?
Short answer: It may be called any number of times for the same state (due to resyncs, restarts, retries), and must always produce the correct result regardless of call count.
Detail: Level-triggered means it compares full desired vs. actual state each time rather than relying on remembering "what changed since last time," making it resilient to missed or duplicate events.
Production example: Re-running Reconcile after an Operator Pod restart doesn't re-trigger a backup that already completed — it checks actual state first.
Common mistake: Writing reconcile logic that assumes it's only ever called exactly once per actual change (edge-triggered), which breaks under restarts or missed events.
4. How does an Operator typically manage native Kubernetes resources like StatefulSets under the hood?
Short answer: It creates and owns them directly, setting ownerReferences so garbage collection cleans them up automatically if the custom resource is deleted.
Detail: This mirrors how a Deployment owns its ReplicaSets — the Operator's CR sits at the top of an ownership chain, with native resources (StatefulSet, Service, PVC, Secret) beneath it.
Production example: Deleting a PostgresCluster CR cascades to delete its owned StatefulSet, Services, and Secrets via Kubernetes' garbage collector.
Common mistake: An Operator creating resources without ownerReferences, leaving orphaned StatefulSets/PVCs behind after the CR is deleted.
5. What's unique about how an Operator can go "beyond" what core Kubernetes controllers do?
Short answer: Many Operators talk directly to the application itself (e.g. a SQL connection to run administrative commands), not just to the Kubernetes API.
Detail: Core controllers like the Deployment controller never reach into the application layer — they only manage Kubernetes objects. An Operator often bridges both worlds, checking replication lag via SQL or triggering an application-level failover command.
Production example: A Postgres Operator connecting directly to the database to run ALTER SYSTEM or check streaming replication lag before deciding to promote a replica.
Common mistake: Assuming an Operator only ever manipulates Kubernetes objects like any other controller — many need direct application-protocol access to do their job.
6. What frameworks are most production Operators built with, and what do they provide?
Short answer: Operator SDK or kubebuilder, both scaffolding tools built on controller-runtime — the same Go library core Kubernetes controllers use.
Detail: They generate boilerplate (CRD schema generation, RBAC manifest generation, watch/informer setup) so the author can focus purely on writing the Reconcile function's business logic.
Production example: Scaffolding a new Operator project with kubebuilder, which generates the CRD types, controller skeleton, and RBAC markers automatically.
Common mistake: Hand-rolling informer/watch plumbing from scratch instead of using the well-tested controller-runtime primitives these frameworks provide.
7. What does the Operator "maturity model" describe, and where do most real Operators sit?
Short answer: A Level 1 (basic install/config) to Level 5 (full auto-pilot) scale; most production Operators sit around Level 2-3.
Detail: Level 2-3 typically automates day-2 operations like backups and minor-version upgrades, but still relies on human judgment for major version jumps or capacity planning decisions.
Production example: A Kafka Operator automating routine broker restarts and config rollouts, but requiring a human decision for a major Kafka version migration.
Common mistake: Expecting any installed Operator to be full autopilot (Level 5) by default — most are not, and treating them as such risks unattended failures during complex scenarios.
8. Why should an Operator's RBAC be scoped tightly rather than granted cluster-admin?
Short answer: An Operator with cluster-admin can affect every namespace/resource in the cluster, far beyond what it actually needs to manage its own CR and owned resources.
Detail: Least-privilege RBAC scoped to the Operator's specific CRD group and the native resource types it creates limits the blast radius if the Operator itself is compromised or buggy.
Production example: Scoping a Postgres Operator's RBAC to only its own CRD, StatefulSets, Services, PVCs, and Secrets within the namespaces it's meant to manage.
Common mistake: Granting cluster-admin "to be safe" instead of doing the RBAC scoping work up front.
9. What's the difference between an Operator's spec and status fields on its Custom Resource?
Short answer: spec is the user-declared desired state; status is written only by the Operator itself, reflecting observed real-world state.
Detail: Users should never write to status directly — fields like currentPrimary or replicationLag are the Operator's own observations, updated as part of its reconcile loop.
Production example: A PostgresCluster's status.phase field showing "Running" only once the Operator has verified the cluster is actually healthy.
Common mistake: A user manually editing status fields expecting it to influence Operator behavior — the Operator will simply overwrite it on the next reconcile.
10. When should you adopt a community Operator versus building a bespoke one?
Short answer: Prefer well-maintained community Operators (Strimzi, Prometheus Operator, CloudNativePG) unless your operational needs are genuinely unique.
Detail: Building a bespoke Operator means owning its reconcile correctness, upgrade path, and RBAC scoping forever — a substantial ongoing engineering investment that's rarely worth it if a mature open-source option already covers the need.
Production example: Adopting the Prometheus Operator rather than writing a custom controller to manage ServiceMonitor-driven scrape config generation.
Common mistake: Building an in-house Operator for a widely-solved problem instead of evaluating existing community options first.

42. What's a risk of an Operator directly running SQL administrative commands? — It needs carefully scoped database credentials and error handling — mistakes here can affect live production data, not just Kubernetes objects.
