Topic 59 of 70

Troubleshooting

A general methodology for diagnosing Kubernetes failures fast: describe, logs, exec, events, node conditions — in that order.
← Distributed Tracing Course Index Kubernetes Internals →

Explanation

Kubernetes troubleshooting is hard mainly because failures can originate at any layer of the stack — application code, container image, Pod spec, scheduler decision, node health, network policy, or the control plane itself — and the symptom (a Pod stuck Pending, or CrashLoopBackOff, or a Service not routing traffic) rarely names its own root cause. What separates an engineer who resolves an incident in five minutes from one who takes an hour is not more knowledge of individual failure modes, but a consistent, layered methodology applied every time, narrowing the search space systematically instead of guessing.

The methodology that works reliably in production is: start with kubectl get to see current state at a glance, then kubectl describe on the failing object (this alone resolves the majority of issues — its Events section is a chronological, component-by-component trace of everything that happened to the object), then kubectl logs (and --previous if it's restarting) to see what the application itself reported, then kubectl exec to interactively probe a running container's environment if logs aren't enough, and finally check node-level conditions (kubectl describe node, disk/memory pressure, taints) if nothing at the Pod level explains the symptom. This order matters because each step is progressively more expensive and Pod-specific information is cheap to gather and usually sufficient.

Kubernetes' declarative, event-driven architecture (see Topic 1) is precisely why this methodology works: every action taken by every component (scheduler binding a node, kubelet pulling an image, a probe failing) is recorded as an Event object, visible via describe or get events. Troubleshooting Kubernetes is largely the skill of reading this event trail and mapping each event to the specific component that emitted it — a skill this whole course has been building toward, since it requires genuinely understanding what each component (scheduler, kubelet, kube-proxy, CNI, CSI) is responsible for.

Most production failures fall into a handful of categories: scheduling failures (insufficient resources, taints/affinity mismatches — Pod stuck Pending), image/registry failures (ImagePullBackOff — wrong image name, missing pull secret, registry unreachable), application crash loops (CrashLoopBackOff — bad config, missing dependency, unhandled exception on startup), readiness/liveness probe failures (app running but not marked healthy, or killed by an overly aggressive liveness probe), networking issues (Service not routing, NetworkPolicy blocking traffic, DNS failures via CoreDNS), and node-level issues (disk pressure, memory pressure, kubelet not reporting, causing evictions).

This topic sits downstream of everything else in the course and upstream of nothing — it's the practical synthesis of architecture, controllers, networking, storage, and observability knowledge into a repeatable incident-response skill, and it's precisely where metrics (Prometheus/VictoriaMetrics), logs (Loki), and traces (Tempo) covered in recent topics get used together in anger: metrics tell you when and roughly where, logs tell you what the application said, and traces tell you which downstream call was actually slow or failing.

Internal Working / Interactions

  • kubectl describe pulls the object plus its associated Events from the API server/etcd — the single richest troubleshooting signal for most Pod issues.
  • kubectl logs reads from the kubelet on the node hosting the Pod, which serves the locally-rotated container log file (see Topic 55).
  • kubectl exec opens an interactive stream through the API server to the kubelet to the container runtime, letting you run commands inside a live container.
  • kubectl get events --sort-by=.lastTimestamp shows the full cluster-wide event stream, useful when you don't yet know which object is the actual root cause.
  • Node conditions (Ready, MemoryPressure, DiskPressure, PIDPressure) are reported by kubelet and drive eviction/scheduling decisions — check these when the Pod-level story doesn't add up.
  • Observability stack correlation — once Pod-level tools are exhausted, pivot to Grafana/Loki/Tempo for historical trends and cross-service context that live `kubectl` commands can't show.

Code Example

# Common failure-mode manifests you'll intentionally break to practice this methodology.

# 1. Scheduling failure: requests exceed any node's allocatable resources -> stuck Pending
apiVersion: v1
kind: Pod
metadata:
  name: oversized-pod
spec:
  containers:
    - name: app
      image: nginx:1.25
      resources:
        requests:
          cpu: "32"          # deliberately larger than any real node -> "Insufficient cpu" event
          memory: "128Gi"
---
# 2. Image pull failure: typo'd image tag -> ImagePullBackOff
apiVersion: v1
kind: Pod
metadata:
  name: bad-image-pod
spec:
  containers:
    - name: app
      image: nginx:this-tag-does-not-exist   # kubectl describe will show "ErrImagePull" / "ImagePullBackOff"
---
# 3. Liveness probe misconfigured too aggressively -> CrashLoopBackOff from repeated kills, not app bugs
apiVersion: v1
kind: Pod
metadata:
  name: flaky-liveness-pod
spec:
  containers:
    - name: app
      image: nginx:1.25
      livenessProbe:
        httpGet: { path: /, port: 80 }
        initialDelaySeconds: 0     # too short - kills the container before nginx finishes starting
        periodSeconds: 2
        failureThreshold: 1        # one slow response = restart; too strict for real workloads

kubectl / CLI Commands

kubectl get pods -A -o wide                                  # cluster-wide quick health scan
kubectl describe pod <pod> -n <ns>                            # #1 tool: full Events trace for one object
kubectl logs <pod> -n <ns> --previous --tail=200               # last crashed instance's final output
kubectl exec -it <pod> -n <ns> -- sh                           # interactive shell to probe live state
kubectl get events -A --sort-by=.lastTimestamp | tail -30       # cluster-wide recent event stream
kubectl describe node <node>                                   # conditions, taints, allocatable vs allocated
kubectl top pod -n <ns> ; kubectl top node                     # quick resource pressure check
kubectl get pod <pod> -n <ns> -o yaml | less                  # full spec+status when describe isn't enough

Best Practices & Common Pitfalls

Best Practices

  • Always run describe before logs — Events often explain the failure without needing app-level logs at all.
  • Check --previous logs on any CrashLoopBackOff — the current attempt's logs may be empty if it just restarted.
  • Correlate the failure timestamp against a Grafana dashboard before assuming it's isolated to one Pod.
  • Reproduce systematically (same manifest change, same failure) rather than randomly changing config until it works.
  • Document root causes in a runbook so recurring failure classes get faster next time.

Pitfalls

  • Jumping straight to kubectl exec or restarting Pods before reading Events — losing the diagnostic trail.
  • Assuming CrashLoopBackOff is always an app bug — probe misconfiguration is a very common false-positive cause.
  • Ignoring node conditions and only ever looking at the Pod — some failures (DiskPressure evictions) are node-wide.
  • Not checking resource requests/limits when a Pod is OOMKilled or throttled — the symptom looks like an app bug but is a sizing issue.

50 Interview Questions

1. What is the recommended order for the Kubernetes troubleshooting methodology described in this topic?
Short answer: kubectl get for overall state, then describe, then logs (with --previous if restarting), then exec, then node-level conditions.
Detail: The order matters because each step is progressively more expensive to gather, and Pod-specific information is cheap and usually sufficient — so you exhaust cheap, high-signal steps before moving to more expensive ones.
Production example: The architecture diagram in this topic models this exact flow as a flowchart, only pivoting to Grafana/Loki/Tempo after describe, logs, exec, and node checks are all inconclusive.
Common mistake: Jumping straight to kubectl exec or restarting Pods before reading Events — this loses the diagnostic trail that describe would have surfaced instantly.
2. Why does kubectl describe resolve the majority of issues on its own?
Short answer: Its Events section is a chronological, component-by-component trace of everything that happened to the object.
Detail: Because Kubernetes' architecture is event-driven, every action taken by every component — scheduler binding a node, kubelet pulling an image, a probe failing — is recorded as an Event object, all visible in one place via describe.
Production example: Describing a Pod stuck Pending immediately shows an "Insufficient cpu" event from the scheduler, explaining the root cause without needing logs at all.
Common mistake: Skipping describe and going directly to logs, missing scheduling or image-pull failures that never produce application logs because the container never started.
3. Why is Kubernetes troubleshooting fundamentally about mapping events to the component that emitted them?
Short answer: Because failures can originate at any layer — app code, image, Pod spec, scheduler, node health, network policy, or the control plane — and the symptom rarely names its own root cause.
Detail: Effective troubleshooting requires understanding what each component (scheduler, kubelet, kube-proxy, CNI, CSI) is actually responsible for, so an event like "FailedScheduling" versus "Failed" (kubelet) points you toward a completely different layer to investigate.
Production example: A Service not routing traffic could be a kube-proxy issue, a NetworkPolicy blocking it, or a CoreDNS failure — reading which component logged the relevant event narrows it immediately.
Common mistake: Guessing at fixes based on the surface symptom (e.g. "Pod Pending") without identifying which specific component's decision caused it.
4. What's the difference between checking current logs versus --previous logs, and when does it matter?
Short answer: --previous retrieves the logs of the last crashed instance of a container, which is essential for CrashLoopBackOff since the current attempt's logs may be empty right after a restart.
Detail: When a container is in CrashLoopBackOff, kubectl logs without --previous may only show a freshly restarted, mostly-empty log; the actual failure output is in the terminated instance's logs.
Production example: kubectl logs <pod> -n <ns> --previous --tail=200 is listed as the specific command to see the last crashed instance's final output.
Common mistake: Concluding "there are no logs, so we can't diagnose this" when the real issue is simply forgetting the --previous flag.
5. What are the main categories of production failures this topic organizes troubleshooting around?
Short answer: Scheduling failures, image/registry failures, application crash loops, readiness/liveness probe failures, networking issues, and node-level issues.
Detail: Each category has a distinct signature (Pending, ImagePullBackOff, CrashLoopBackOff, probe-triggered restarts, Service/DNS/NetworkPolicy problems, and disk/memory pressure evictions) and a distinct place in the stack to investigate.
Production example: The code example provides one manifest per category to intentionally reproduce during practice: an oversized-resource-request Pod, a bad-image-tag Pod, and an overly aggressive liveness probe Pod.
Common mistake: Treating all failures as one undifferentiated "something's broken" problem instead of first identifying which of these categories the symptom fits.
6. Why can an overly aggressive liveness probe cause CrashLoopBackOff even when the application has no bug?
Short answer: A probe with too short an initialDelaySeconds or too strict a failureThreshold can kill the container before the app finishes starting or on one slow response.
Detail: This is explicitly called out as a false-positive cause of CrashLoopBackOff — the restart is caused by kubelet acting on probe failures, not by the application crashing on its own.
Production example: The flaky-liveness-pod example sets initialDelaySeconds: 0, periodSeconds: 2, failureThreshold: 1 against nginx, causing kubelet to kill the container before nginx finishes starting.
Common mistake: Assuming CrashLoopBackOff is always an app bug and debugging application code first, instead of checking probe configuration.
7. Why does a Pod requesting resources larger than any node's allocatable capacity get stuck Pending rather than erroring immediately?
Short answer: The scheduler can never find a node that satisfies the resource request, so the Pod simply stays unscheduled, logging an "Insufficient cpu"/"Insufficient memory" event on each scheduling attempt.
Detail: This is a scheduling failure category — the Pod object itself is valid, but no node passes the scheduler's filtering step, so it waits indefinitely rather than being rejected outright.
Production example: The oversized-pod example requests 32 CPU and 128Gi memory, deliberately larger than any real node, to reproduce this event.
Common mistake: Assuming a Pending Pod means something is "broken" in the cluster rather than checking whether the requested resources simply don't fit anywhere.
8. What does node condition monitoring add to Pod-level troubleshooting, and when should you check it?
Short answer: Node conditions (Ready, MemoryPressure, DiskPressure, PIDPressure) reported by kubelet drive eviction/scheduling decisions and explain failures that aren't visible at the Pod level — check them when the Pod-level story doesn't add up.
Detail: Some failures, like DiskPressure-triggered evictions, are node-wide and affect every Pod on that node simultaneously, which is easy to miss if you're only ever looking at one Pod at a time.
Production example: kubectl describe node <node> reveals conditions, taints, and allocatable vs. allocated resources that explain why several unrelated Pods on the same node are being evicted.
Common mistake: Ignoring node conditions and only ever looking at the Pod, missing node-wide issues like disk pressure evictions.
9. When should you pivot from live kubectl commands to the observability stack (Grafana/Loki/Tempo)?
Short answer: Once Pod-level tools (describe, logs, exec) and node conditions are exhausted and still inconclusive, pivot to Grafana/Loki/Tempo for historical trends and cross-service context.
Detail: Live kubectl commands only show current or very recent state; the observability stack is needed for historical patterns or correlating behavior across multiple services, which is exactly what metrics, logs, and traces (covered in the recent topics) are built for.
Production example: Correlating a Pod's failure timestamp against a Grafana dashboard before assuming the issue is isolated to one Pod is listed as a best practice.
Common mistake: Never bothering to check historical dashboards and assuming a failure is a one-off, when the same pattern has actually been recurring for days.
10. How do metrics, logs, and traces each contribute differently during an incident, according to this topic?
Short answer: Metrics tell you when and roughly where a problem occurred; logs tell you what the application said; traces tell you which downstream call was actually slow or failing.
Detail: This topic is explicitly framed as the practical synthesis point where architecture, controllers, networking, storage, and observability knowledge from the rest of the course all come together into one repeatable incident-response skill.
Production example: An alert fires from a metrics threshold, a Grafana dashboard narrows down which service is affected, Loki logs show the application's own error message, and a Tempo trace pinpoints the exact slow downstream call.
Common mistake: Using only one of the three signals throughout an entire incident instead of pivoting between them as the investigation narrows.
