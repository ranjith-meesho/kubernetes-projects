DNS

The cluster-wide naming convention that lets every Pod and Service be addressed by a stable, predictable hostname.
← Services Course Index CoreDNS →

Explanation

Cluster DNS is the naming layer that lets Pods find Services (and, less commonly, other Pods) by a predictable hostname instead of a raw, ever-changing IP address. Every Kubernetes cluster runs a cluster-wide DNS service (in practice, always CoreDNS today) that automatically creates and updates DNS records as Services and Pods are created, deleted, or rescheduled — closing the loop that starts with a Service's stable virtual IP: the IP is stable, but you still need a human/application-friendly name to reach it, and that name still needs to survive the Service being recreated with a new IP.

Kubernetes needs this because hardcoding IPs anywhere in application config is fragile and defeats the entire point of a dynamic, self-healing platform — an IP baked into a config file becomes wrong the moment the underlying object is recreated. DNS names, by contrast, are derived deterministically from object names and namespaces, so they never change even as the underlying IPs do. This is what lets a manifest say http://checkout.payments.svc.cluster.local:8080 and have that URL remain correct forever, regardless of how many times the checkout Service's ClusterIP changes underneath it.

The naming convention follows a fixed hierarchy: a Service named checkout in namespace payments gets the DNS name checkout.payments.svc.cluster.local (often shortened to just checkout or checkout.payments when resolving from within the same or a different namespace, thanks to the search-domain suffixes injected into every Pod's /etc/resolv.conf). Headless Services (ClusterIP: None, used by StatefulSets) get one DNS record per backing Pod instead of one record for the whole Service, in the form <pod-hostname>.<service>.<namespace>.svc.cluster.local — this is exactly the mechanism StatefulSet replicas use to find each other by stable name.

Every Pod, unless explicitly configured otherwise, has its container's /etc/resolv.conf populated by the kubelet with the cluster DNS service's IP as the nameserver, plus a search list of (<namespace>.svc.cluster.local, svc.cluster.local, cluster.local) so short, unqualified names resolve correctly depending on context. This file is what actually wires an application's ordinary DNS resolution calls (getaddrinfo, etc.) into the Kubernetes-aware DNS system without the application needing any Kubernetes-specific code.

DNS also resolves Pods directly (if their IP is dash-substituted into a hostname under a headless Service) and supports SRV records for named Service ports — useful for clients that need to discover both host and port dynamically (e.g. some service-mesh sidecars and legacy SRV-aware clients).

Internal Working / Interactions

  • kubelet — populates each Pod's /etc/resolv.conf with the cluster DNS Service IP and search domains at Pod creation time, based on the Pod's dnsPolicy (default: ClusterFirst).
  • CoreDNS (or equivalent) — the actual DNS server; watches Services/Endpoints/Pods via the API Server and answers queries by constructing records on the fly rather than from a static zone file.
  • kube-dns Service — CoreDNS itself is exposed via a stable ClusterIP Service (conventionally named kube-dns), which is the IP kubelet injects into every Pod's resolv.conf.
  • Search domain chaining — an unqualified name like checkout is tried against each search suffix in order until one resolves, which is why same-namespace lookups can omit the namespace entirely.
  • dnsPolicy variants — ClusterFirst (default, cluster DNS + fallback to upstream), Default (inherit node's own resolv.conf), None (fully custom via dnsConfig), ClusterFirstWithHostNet (for hostNetwork Pods).

Code Example

apiVersion: v1
kind: Pod
metadata:
  name: custom-dns-pod
spec:
  dnsPolicy: "None"              # Ignore cluster defaults entirely; use only dnsConfig below.
  dnsConfig:
    nameservers:
      - 10.96.0.10                 # Explicit cluster DNS IP (normally injected automatically).
      - 8.8.8.8                    # Extra upstream resolver, e.g. for external-only lookups.
    searches:
      - payments.svc.cluster.local
      - svc.cluster.local
      - cluster.local
    options:
      - name: ndots
        value: "2"                 # Fewer dots before trying search domains -> fewer wasted DNS queries.
  containers:
    - name: app
      image: registry.internal/payments-app:1.0
---
# Typical resolv.conf INSIDE a normal Pod (dnsPolicy: ClusterFirst, the default) — generated by kubelet, not user YAML:
# nameserver 10.96.0.10
# search payments.svc.cluster.local svc.cluster.local cluster.local
# options ndots:5

kubectl Commands

# Inspect a Pod's actual resolv.conf as generated by kubelet
kubectl exec -it my-pod -- cat /etc/resolv.conf

# Resolve a Service name from inside the cluster
kubectl run -it --rm dns-test --image=busybox --restart=Never -- nslookup checkout.payments.svc.cluster.local

# Resolve a specific StatefulSet Pod via its headless-Service DNS name
kubectl run -it --rm dns-test --image=busybox --restart=Never -- nslookup kafka-0.kafka-headless.default.svc.cluster.local

# Check the cluster DNS Service's ClusterIP (what's injected into every Pod)
kubectl get svc -n kube-system kube-dns

# Confirm CoreDNS Pods are healthy
kubectl get pods -n kube-system -l k8s-app=kube-dns

# Show a Pod's effective dnsPolicy/dnsConfig
kubectl get pod my-pod -o jsonpath='{.spec.dnsPolicy}{"\n"}{.spec.dnsConfig}'

Best Practices & Common Pitfalls

Best Practices

  • Use the shortest name that resolves correctly for your use case (same-namespace: just the Service name).
  • Use fully-qualified names (svc.ns.svc.cluster.local) for cross-namespace calls to avoid ambiguity.
  • Tune ndots only when you understand the extra-query cost tradeoff it creates for external lookups.
  • Rely on headless-Service per-Pod DNS for StatefulSet peer discovery rather than hardcoding Pod IPs.

Pitfalls

  • High default ndots:5 causing every external domain lookup (e.g. api.stripe.com) to first try several useless internal search-domain queries, adding latency.
  • Assuming DNS caching is instant — a Pod's cached negative DNS lookup can briefly outlive a just-created Service.
  • Hardcoding a Service's ClusterIP instead of its DNS name, breaking the moment the Service is recreated.
  • Forgetting cross-namespace calls need the namespace segment — checkout alone won't resolve from a different namespace.

50 Interview Questions

1. Why do Services need DNS names on top of an already-stable ClusterIP?
Short answer: The ClusterIP is stable for the Service's lifetime, but the Service object itself can still be deleted and recreated with a new IP — a DNS name derived deterministically from the object's name/namespace survives that.
Detailed explanation: Hardcoding an IP anywhere in application config becomes wrong the moment the underlying Service is recreated; DNS names are derived from object identity, not the current IP, so they never need updating.
Production example: A manifest referencing http://checkout.payments.svc.cluster.local:8080 remains correct forever, regardless of how many times the checkout Service's ClusterIP changes underneath it.
Common mistake: Hardcoding a Service's ClusterIP directly in application config instead of its DNS name.
2. What is the full DNS naming hierarchy for a Service, and what do the shortened forms rely on?
Short answer: <service>.<namespace>.svc.cluster.local is the full form; shortened forms like just checkout rely on search-domain suffixes injected into every Pod's resolv.conf.
Detailed explanation: The resolver tries the unqualified name against each search suffix in order until one resolves, which is why same-namespace lookups can omit the namespace entirely while cross-namespace calls need at least the namespace segment.
Production example: From a Pod in the payments namespace, just checkout resolves via the payments.svc.cluster.local search suffix without needing the fully-qualified form.
Common mistake: Using an unqualified Service name across namespaces and getting an unexpected resolution failure since the short form only works within the same namespace.
3. How does a headless Service's DNS behavior differ from a normal Service's, and why does that matter for StatefulSets?
Short answer: A headless Service returns one DNS record per backing Pod instead of one shared record for the whole Service.
Detailed explanation: The per-Pod record takes the form <pod-hostname>.<service>.<namespace>.svc.cluster.local, which is exactly the mechanism StatefulSet replicas use to find and address specific peers by stable name rather than "some randomly chosen instance."
Production example: kafka-1.kafka-headless.default.svc.cluster.local always resolves to the specific Kafka broker replica 1, never any other broker.
Common mistake: Creating a normal ClusterIP Service for a StatefulSet expecting per-replica DNS resolution, and getting one shared virtual IP instead.
4. What is dnsPolicy: ClusterFirst, and what happens for domains it doesn't recognize?
Short answer: It's the default policy — queries first go to cluster DNS (CoreDNS), and any domain CoreDNS doesn't own falls through to an upstream resolver.
Detailed explanation: This lets a single Pod resolve both internal Service names and external domains (e.g. api.stripe.com) through the same nameserver entry, since CoreDNS itself forwards non-cluster queries onward.
Production example: A Pod querying both checkout.payments.svc.cluster.local and api.stripe.com uses the exact same nameserver IP in its resolv.conf — CoreDNS internally decides which path to take.
Common mistake: Assuming a separate resolver configuration is needed for external domains — ClusterFirst already handles the fallback transparently.
5. What is kubelet's role in making cluster DNS "just work" for every Pod?
Short answer: kubelet populates each Pod's /etc/resolv.conf with the cluster DNS Service IP and search domains at Pod creation time, based on the Pod's dnsPolicy.
Detailed explanation: This is what actually wires an application's ordinary DNS resolution calls (getaddrinfo, etc.) into the Kubernetes-aware DNS system without the application needing any Kubernetes-specific code or configuration.
Production example: kubectl exec -it my-pod -- cat /etc/resolv.conf shows a nameserver entry pointing at the cluster DNS Service's ClusterIP (conventionally 10.96.0.10), injected automatically by kubelet.
Common mistake: Assuming applications need custom code to be "Kubernetes-DNS-aware" — the standard OS resolver library already does the right thing given the injected resolv.conf.
6. Why does a high default ndots value (5) sometimes cause latency for external domain lookups?
Short answer: With ndots:5, a name with fewer than 5 dots is first tried against every internal search-domain suffix before being tried as an absolute external name, wasting several DNS round-trips for genuinely external domains.
Detailed explanation: A lookup for api.stripe.com (2 dots) under ndots:5 first tries api.stripe.com.payments.svc.cluster.local, api.stripe.com.svc.cluster.local, and api.stripe.com.cluster.local — all guaranteed failures — before finally trying the real absolute name, adding real latency per request if uncached.
Production example: A high-traffic service calling many third-party APIs directly (not through an internal proxy) can see measurable added p99 latency purely from this wasted internal search-domain churn.
Common mistake: Not tuning ndots for workloads that make heavy use of external, non-cluster domain lookups.
7. What's the difference between dnsPolicy: Default and dnsPolicy: None?
Short answer: Default inherits the node's own resolv.conf (bypassing cluster DNS entirely); None ignores cluster defaults entirely and requires you to specify everything explicitly via dnsConfig.
Detailed explanation: Default is useful when a Pod genuinely needs the host's own DNS view (rare); None gives full manual control, useful for a Pod needing a custom nameserver list, custom search domains, or specific resolver options like ndots tuning.
Production example: A Pod using dnsPolicy: None with dnsConfig explicitly listing both the cluster DNS IP and an external resolver (8.8.8.8) gets fully controlled, predictable resolution behavior.
Common mistake: Confusing "Default" with "the cluster default" — counterintuitively, Default means the node's resolv.conf, not ClusterFirst.
8. Why can a Pod's cached negative DNS lookup briefly outlive a just-created Service?
Short answer: DNS caching (both in CoreDNS's cache plugin and potentially in the client's own resolver) isn't instant — a lookup performed just before a Service existed can be cached as "not found" for a short TTL window.
Detailed explanation: This is a race condition between Service creation and any earlier failed resolution attempt for that same name; it typically resolves itself once the negative cache TTL expires, but can cause confusing transient failures right after deploying a new Service.
Production example: A Pod that started up and failed to resolve a dependency's Service name moments before that Service was created might keep failing for a few seconds even after the Service becomes available.
Common mistake: Assuming DNS changes propagate instantly and not accounting for brief caching-related races during rapid successive deploys.
9. Why do SRV records matter for some clients even though most just use A/AAAA records via the Service name?
Short answer: SRV records let a client discover both host and port dynamically for a named Service port, useful when the connecting port isn't known/fixed in advance.
Detailed explanation: Some service-mesh sidecars and legacy SRV-aware clients query SRV records (in the form _<port-name>._<protocol>.<svc>.<ns>.svc.cluster.local) specifically to avoid hardcoding a port number that might change or vary by named port.
Production example: A client needing to discover the correct port for a Service's named "http" port dynamically can query the corresponding SRV record instead of assuming a fixed port number.
Common mistake: Assuming SRV record support is universal — most application code just uses plain A-record resolution plus a hardcoded/expected port instead.
10. Why must cross-namespace calls always include at least the namespace segment in the DNS name?
Short answer: Because the search-domain suffixes injected into a Pod's resolv.conf are scoped to that Pod's own namespace first — an unqualified name only resolves against the calling Pod's namespace.
Detailed explanation: A Pod in frontend namespace querying just checkout would try checkout.frontend.svc.cluster.local first (and likely fail, since checkout lives in payments) — it must instead query checkout.payments or the fully-qualified name.
Detailed explanation cont'd: This is a frequent source of "works in this namespace, breaks in that one" bugs when manifests are copy-pasted across environments without adjusting the DNS name's namespace segment.
Production example: Migrating a Deployment from a shared namespace into its own dedicated namespace commonly breaks any hardcoded unqualified Service names in its config that assumed same-namespace resolution.
Common mistake: Forgetting the namespace segment when referencing a Service cross-namespace and assuming Kubernetes DNS is somehow namespace-agnostic.
11–50. Additional rapid-fire questions
11. What's the conventional name of the cluster DNS Service? — kube-dns (even when CoreDNS is the actual server implementation).
12. What file does kubelet write DNS configuration into inside each Pod? — /etc/resolv.conf.
13. What are the three default search domains injected for a Pod? — <namespace>.svc.cluster.local, svc.cluster.local, cluster.local.
14. What's the default ndots value inside a Pod's resolv.conf? — 5.
15. What DNS record type resolves a Service name to its ClusterIP? — A (or AAAA for IPv6).
16. Does a Pod get its own DNS A record by default? — Not by default under a normal Service — only under a headless Service with per-Pod records, or if explicitly enabled.
17. What dnsPolicy variant is used for hostNetwork Pods that still want cluster DNS? — ClusterFirstWithHostNet.
18. Why do hostNetwork Pods need a special dnsPolicy variant at all? — Because hostNetwork Pods share the node's network namespace, so the normal ClusterFirst injection needs an explicit variant to still apply correctly.
19. How would you check what dnsPolicy a running Pod is using? — kubectl get pod <name> -o jsonpath='{.spec.dnsPolicy}'.
20. Can you override individual resolver options without setting dnsPolicy: None? — Yes — dnsConfig.options can be combined with ClusterFirst to tweak things like ndots without discarding the cluster defaults entirely.
21. What's a realistic reason to add an extra nameserver via dnsConfig? — Needing to resolve an internal corporate domain not known to cluster DNS, alongside normal cluster resolution.
22. Does cluster DNS resolve external, non-cluster domains directly itself? — No — it forwards them to an upstream resolver (its own forward plugin handles this, not raw cluster DNS logic itself).
23. What happens if a Pod's dnsPolicy is None but dnsConfig is empty? — The Pod effectively has no usable DNS configuration at all, so name resolution fails entirely.
24. Why is it risky to set dnsPolicy: Default for most workloads? — It bypasses cluster DNS entirely, so cluster-internal Service names won't resolve at all from that Pod.
25. What's the resolution order when multiple search domains could match? — Domains are tried in the listed order until one succeeds or all are exhausted.
26. How would a Pod resolve a specific StatefulSet replica from a completely different namespace? — Using the fully-qualified per-Pod headless-Service name including the target namespace, e.g. kafka-0.kafka-headless.data.svc.cluster.local.
27. Does DNS resolution depend on kube-proxy at all? — No — DNS resolution and kube-proxy's traffic routing are independent systems; DNS just returns an IP, kube-proxy handles routing to it separately.
28. What's a common symptom of CoreDNS being completely down? — Cluster-wide new-connection failures for anything addressed by name, while already-established connections keep working.
29. Why do already-established connections survive a DNS outage? — DNS is only consulted at connection-initiation time; existing TCP connections don't re-resolve names mid-session.
30. What's the purpose of testing with kubectl run ... -- nslookup? — Quickly verifying DNS resolution behavior from inside the cluster's actual DNS context, rather than from an external machine.
31. Can a Pod be configured to skip cluster DNS entirely for all lookups? — Yes, via dnsPolicy: Default or a fully custom dnsConfig under None.
32. What's the risk of a very low ndots setting for internal-only workloads? — Short internal names might be misinterpreted as absolute/external before search-domain suffixes are tried, causing internal resolution failures.
33. Why does the DNS naming convention matter for GitOps/portable manifests? — Deterministic names derived from object identity let the same manifest work unchanged across environments (dev/staging/prod) as long as namespace/service names match.
34. What's a subtle bug from copy-pasting a manifest into a new namespace without updating unqualified DNS references? — Any hardcoded unqualified Service name assumed to be "same namespace" now silently points to nothing or the wrong dependency.
35. Does every Kubernetes distribution use CoreDNS specifically? — Overwhelmingly yes today, though the DNS "concept" (cluster-wide naming) is implementation-agnostic; CoreDNS became the de facto standard since replacing kube-dns.
36. What's the effect of a Pod explicitly setting hostNetwork: true without adjusting dnsPolicy? — It may not get proper cluster DNS resolution unless ClusterFirstWithHostNet is set.
37. How does DNS caching interact with a Service's ClusterIP changing after recreation? — Cached positive answers for the old IP can serve stale data briefly until TTL expiry, even though the DNS name is otherwise "always correct" long-term.
38. What's a good practice for cross-namespace dependency references? — Always use at least the namespace-qualified form (svc.namespace) rather than relying on implicit same-namespace shortening.
39. Can Pod-level DNS resolution be tested without deploying a real workload? — Yes — an ephemeral debug Pod (kubectl run --rm) with nslookup/dig is the standard quick test.
40. What's the significance of "cluster.local" as the default cluster domain? — It's the conventional root suffix for all cluster-internal names, though it's technically configurable per-cluster (rare in practice).
41. Why might a multi-cluster setup need to change the default cluster domain? — To avoid naming collisions/ambiguity when cross-cluster service discovery or federation is involved.
42. What's the relationship between dnsConfig.searches and resolv.conf's search line? — dnsConfig.searches is exactly what gets written as the search directive in the Pod's resolv.conf.
43. Does DNS resolution require any RBAC permissions from the querying Pod? — No — DNS queries go straight to CoreDNS over the network; no Kubernetes API RBAC is involved in a client Pod merely resolving a name.
44. What's a common reason external API calls from Pods are slower than expected? — Uninvestigated ndots-driven wasted internal search-domain queries before the real external lookup succeeds.
45. How would you verify the actual search domains a Pod is using at runtime? — kubectl exec -it <pod> -- cat /etc/resolv.conf.
46. What happens if a custom dnsConfig nameserver is unreachable? — Resolution for that Pod fails or times out for any query relying on that nameserver, independent of cluster DNS health.
47. Is DNS involved in Pod-to-Pod communication that doesn't go through a Service? — Not necessarily — direct Pod IP communication (e.g. via a headless Service's per-Pod record, or a hardcoded IP) doesn't require the normal Service-name DNS path, though it's still commonly resolved via headless-Service DNS.
48. What's a good monitoring signal for cluster DNS health specifically? — Query latency and error-rate metrics scraped from CoreDNS, since app-level symptoms (timeouts) can otherwise be misattributed to networking or the target service itself.
49. Why is understanding DNS naming conventions important before troubleshooting connectivity? — Many "network" issues are actually simple naming/namespace-qualification mistakes rather than real connectivity problems.
50. What's the first thing to check when a Pod can't resolve a Service name at all? — Whether the correct namespace-qualified (or fully-qualified) name was used, before assuming CoreDNS itself is broken.
