Vertical Pod Autoscaler (VPA)

Right-sizing container CPU and memory requests/limits automatically instead of guessing once and never revisiting.
← HPA Course Index Cluster Autoscaler →

Explanation

The Vertical Pod Autoscaler (VPA) is a set of components (not built into core Kubernetes — it's a separate project under kubernetes/autoscaler) that automatically adjusts a container's resource requests and limits based on observed historical usage, rather than relying on a developer's initial guess. Where HPA (topic 37) changes how many Pods run, VPA changes how big each Pod is allowed to be — the two are complementary, not competing, though they should generally not both manage the exact same metric on the exact same workload simultaneously.

Kubernetes needs this because manually set resource requests are almost always wrong in one direction or the other: set too high and you waste cluster capacity (paying for CPU/memory nobody uses); set too low and the Pod gets throttled or OOM-killed under real load. Getting requests right requires observing actual usage over time — exactly the kind of continuous, data-driven adjustment a human operator is bad at doing repeatedly across hundreds of services, and exactly what VPA automates.

VPA is composed of three cooperating components, each with a distinct job:

  • Recommender — watches historical and current resource usage (via metrics-server/Prometheus) for Pods matching a VPA object's target, and computes recommended requests (lower bound, target, upper bound) using a decayed histogram model.
  • Updater — checks running Pods against the Recommender's current recommendation; if a Pod's requests are significantly out of line, it evicts the Pod (respecting PodDisruptionBudgets) so it can be recreated with updated values — because requests are immutable on a running Pod, updating them requires recreation.
  • Admission Controller (webhook) — intercepts Pod creation and mutates the incoming Pod's resource requests/limits to match the current recommendation, before it ever reaches the Scheduler.

The updateMode field controls how aggressively VPA acts: Off (recommendations only, nothing applied automatically — a very common safe starting point), Initial (apply only at Pod creation, never evict running Pods), Recreate (evict and recreate Pods when recommendations drift), and Auto (currently equivalent to Recreate, reserved for future in-place resize support). Because updating requests today requires deleting and recreating the Pod, VPA in Recreate/Auto mode causes real disruption — this is the single biggest operational caveat teams need to plan around (PodDisruptionBudgets, multiple replicas, avoiding VPA on singleton stateful workloads).

In production, VPA's most common use is running in Off mode purely for its recommendations — teams pipe the Recommender's output into dashboards or CI checks to catch egregiously over/under-provisioned workloads, then apply changes deliberately through code review rather than letting VPA evict pods automatically. Full Auto mode is more common for stateless, horizontally-redundant workloads with generous PodDisruptionBudgets, and is explicitly discouraged for anything combined with HPA on the same resource dimension.

VPA + HPA conflict: Never let VPA manage CPU requests for a workload where HPA also scales on CPU utilization — VPA changing the request changes the denominator HPA uses for utilization, causing the two controllers to fight. If you need both, have HPA scale on a custom/external metric and let VPA manage CPU/memory sizing.

Internal Working / Interactions

  • metrics-server / Prometheus: the Recommender's raw usage data source — same pipeline HPA depends on for resource metrics.
  • VPA Recommender: maintains an in-memory (and optionally checkpointed) decayed histogram of usage per container, computing target/lower-bound/upper-bound recommendations continuously.
  • VPA Updater: periodically compares live Pods against current recommendations; if a Pod is far enough out of range and its update policy allows it, evicts the Pod via the Eviction API (respecting PodDisruptionBudget).
  • VPA Admission Controller: registered as a mutating admission webhook; intercepts Pod CREATE requests matching an active VPA object and rewrites resources.requests/limits before the API Server persists the object.
  • Scheduler: sees only the already-mutated resource values by the time it schedules the recreated Pod — it has no direct awareness of VPA.
  • PodDisruptionBudget: consulted by the Updater before evicting — VPA will not evict a Pod if doing so would violate the target workload's PDB.

Code Example

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: checkout-vpa
  namespace: payments
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: checkout-service
  updatePolicy:
    updateMode: "Off"          # SAFE default: only produce recommendations, never evict/mutate pods
  resourcePolicy:
    containerPolicies:
      - containerName: checkout
        minAllowed:               # floor - never recommend below this
          cpu: "100m"
          memory: "128Mi"
        maxAllowed:               # ceiling - never recommend above this
          cpu: "2"
          memory: "2Gi"
        controlledResources: ["cpu", "memory"]

apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: batch-worker-vpa
  namespace: batch
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: batch-worker
  updatePolicy:
    updateMode: "Auto"          # actively evict + resize - only safe because batch-worker is
                                  # stateless, horizontally redundant, and has a permissive PDB
  resourcePolicy:
    containerPolicies:
      - containerName: worker
        controlledValues: RequestsAndLimits   # update both requests and limits, not just requests
        minAllowed: { cpu: "200m", memory: "256Mi" }
        maxAllowed: { cpu: "4", memory: "4Gi" }

kubectl Commands

# View current VPA recommendation
kubectl describe vpa checkout-vpa -n payments

# See just the recommendation block
kubectl get vpa checkout-vpa -n payments -o jsonpath='{.status.recommendation}'

# List all VPA objects and their target workloads
kubectl get vpa -A

# Check VPA component health (recommender/updater/admission-controller pods)
kubectl get pods -n kube-system -l app=vpa-recommender
kubectl get pods -n kube-system -l app=vpa-updater
kubectl get pods -n kube-system -l app=vpa-admission-controller

# Confirm the admission webhook is registered
kubectl get mutatingwebhookconfigurations | grep vpa

# Watch a pod get evicted and recreated with new resources
kubectl get pods -n payments -l app=checkout -w

Best Practices & Common Pitfalls

Do's

  • Start every VPA rollout in updateMode: Off and review recommendations before enabling eviction.
  • Always set minAllowed/maxAllowed bounds to prevent runaway recommendations from a noisy usage spike.
  • Ensure a permissive PodDisruptionBudget exists before enabling Auto/Recreate mode.
  • Reserve VPA for workloads where usage is relatively stable/predictable over time — it reacts slowly to sudden shifts.

Don'ts

  • Don't run VPA in Auto mode on singleton StatefulSets or workloads with no redundancy — evictions cause real downtime.
  • Don't let VPA and HPA both control CPU on the same workload — they will fight and cause instability.
  • Don't expect VPA to resize a running pod in place today — it always works via evict + recreate, which is disruptive.
  • Don't skip monitoring VPA's own resource usage — the Recommender keeps histograms per container and can become memory-heavy in very large clusters.

50 Interview Questions

1. What does VPA change about a Pod, and how is that different from HPA?
Short answer: VPA changes a container's resource requests/limits ("how big" a Pod is) rather than the replica count ("how many" Pods exist), which is HPA's job.
Detail: The two are complementary axes of autoscaling — vertical (size) vs horizontal (count) — and generally should not both manage the same metric on the same workload.
Production example: checkout-vpa adjusts the checkout container's CPU/memory requests based on observed usage, while a separate HPA (topic 37) would scale replica count on an unrelated metric.
Common mistake: Treating VPA as "just HPA but for resources" rather than understanding it operates on a completely different, and disruptive, mechanism (evict + recreate).
2. Name and describe VPA's three components.
Short answer: Recommender (computes recommendations from historical usage), Updater (evicts out-of-range running Pods), and Admission Controller (mutates new Pods' resources at creation time).
Detail: Each has a single responsibility: the Recommender never touches live Pods, the Updater only decides *whether* to evict, and the Admission Controller is the only component that actually writes new resource values, and only at Pod creation.
Production example: A recommendation drift is computed by the Recommender, acted on by the Updater (eviction), and then applied by the Admission Controller when the replacement Pod is created.
Common mistake: Assuming the Updater directly resizes a Pod — it can only evict; the Admission Controller does the actual mutation on the recreated Pod.
3. Why can't VPA resize a running Pod in place today?
Short answer: Because resource requests/limits are immutable on a running Pod spec — the only way to change them is to delete and recreate the Pod.
Detail: This is why Auto/Recreate mode is inherently disruptive, and why updateMode: Off (recommendations only) is a common safe starting point.
Production example: batch-worker-vpa in Auto mode evicts and recreates Pods whenever the recommendation drifts, relying on the workload being stateless and horizontally redundant to absorb the disruption.
Common mistake: Expecting VPA in Auto mode to behave like a live, in-place resize with zero disruption.
4. What are the four updateMode values and what does each do?
Short answer: Off (recommend only), Initial (apply only at Pod creation, never evict), Recreate (evict and recreate when recommendations drift), Auto (currently equivalent to Recreate).
Detail: Auto is reserved for future in-place resize support but today behaves identically to Recreate — both cause eviction-driven disruption.
Production example: checkout-vpa uses updateMode: "Off" so teams review recommendations before ever letting VPA touch a running Pod.
Common mistake: Assuming Auto already supports true in-place resizing rather than still relying on evict + recreate under the hood.
5. Where does the Recommender get its usage data from, and what does it compute?
Short answer: It watches historical/current usage via metrics-server or Prometheus and computes recommended requests as lower bound, target, and upper bound using a decayed histogram model.
Detail: This is the same underlying metrics pipeline HPA depends on for resource metrics, just consumed for a different purpose — sizing rather than scaling.
Production example: checkout-vpa's Recommender maintains a per-container histogram feeding .status.recommendation, which teams pipe into dashboards even in Off mode.
Common mistake: Assuming the Recommender needs its own dedicated metrics pipeline rather than reusing metrics-server/Prometheus.
6. Why does the Updater check PodDisruptionBudgets before evicting?
Short answer: To avoid violating availability guarantees — VPA will not evict a Pod if doing so would drop the workload below its PDB's minAvailable.
Detail: This is the same Eviction API mechanism used elsewhere in Kubernetes (e.g. node drains); VPA doesn't bypass it just because it's an autoscaler.
Production example: batch-worker-vpa is only safe in Auto mode because batch-worker has a permissive PDB allowing evictions without breaching availability.
Common mistake: Enabling Auto/Recreate mode on a workload with no PDB or a singleton replica count, causing real downtime on every eviction.
7. What happens architecturally when the Admission Controller intercepts a Pod?
Short answer: It's registered as a mutating admission webhook that rewrites resources.requests/limits on matching Pod CREATE requests before the API Server persists the object.
Detail: Because this happens before persistence, the Scheduler only ever sees the already-mutated values — it has no direct awareness that VPA exists.
Production example: When batch-worker-vpa evicts a Pod, the ReplicaSet controller's replacement Pod gets its CPU/memory rewritten by the webhook before the Scheduler ever places it.
Common mistake: Assuming the Scheduler applies VPA's recommendation itself — it never does; mutation happens strictly earlier, at admission.
8. Why is it dangerous to run VPA and HPA on the same metric for the same workload?
Short answer: VPA changing a Pod's CPU request changes the denominator HPA uses to compute utilization percentage, causing the two controllers to fight and destabilize replica count.
Detail: The fix is to have HPA scale on a custom/external metric unrelated to the dimension VPA manages, or use VPA only in Off mode for recommendations.
Production example: A workload with both HPA (CPU) and VPA (CPU, Auto mode) churned replicas erratically until VPA was switched to Off mode.
Common mistake: Enabling VPA Auto mode on a workload already scaled by CPU-based HPA without redesigning which controller owns which dimension.
9. Why do most production teams run VPA in Off mode?
Short answer: To get the Recommender's sizing insights (piped into dashboards or CI checks) without letting the Updater evict Pods automatically.
Detail: This lets teams apply changes deliberately through code review, catching egregiously over/under-provisioned workloads without risking unplanned disruption.
Production example: checkout-vpa runs in Off mode purely so kubectl describe vpa output can inform a manual resource-request PR.
Common mistake: Jumping straight to Auto mode on a new workload without first observing what Off-mode recommendations look like.
10. What is the purpose of minAllowed/maxAllowed in a VPA's resourcePolicy?
Short answer: They bound the Recommender's output so a noisy usage spike can't produce a runaway recommendation far outside sane limits.
Detail: minAllowed acts as a floor (never recommend below this) and maxAllowed as a ceiling (never recommend above this), regardless of what the histogram computes.
Production example: checkout-vpa bounds CPU between 100m and 2 cores so a brief traffic anomaly can't push the recommendation to an absurd value.
Common mistake: Omitting min/maxAllowed bounds entirely, letting a transient spike produce an unrealistic recommendation that then gets applied.
11–50. Additional rapid-fire questions
11. Is VPA built into core Kubernetes? — No, it's a separate project under kubernetes/autoscaler, installed separately.
12. What API group/kind defines a VPA object? — autoscaling.k8s.io/v1, kind VerticalPodAutoscaler.
13. What field on a VPA object points at the workload it manages? — spec.targetRef.
14. What problem does VPA solve that manual resource-request tuning doesn't scale to? — Continuously re-evaluating actual usage across hundreds of services, which a human operator can't do repeatedly and accurately.
15. What happens if requests are set too high manually? — Wasted cluster capacity — paying for CPU/memory nobody uses.
16. What happens if requests are set too low manually? — The Pod gets throttled (CPU) or OOM-killed (memory) under real load.
17. What command shows a VPA's current recommendation? — kubectl describe vpa <name> -n <namespace>.
18. What command extracts just the recommendation block via jsonpath? — kubectl get vpa <name> -o jsonpath='{.status.recommendation}'.
19. What labels identify the three VPA component Pods in kube-system? — app=vpa-recommender, app=vpa-updater, app=vpa-admission-controller.
20. How do you confirm the VPA admission webhook is registered? — kubectl get mutatingwebhookconfigurations | grep vpa.
21. What field controls whether both requests and limits are updated, or requests only? — controlledValues (e.g. RequestsAndLimits).
22. What field restricts VPA to only certain resource types? — controlledResources (e.g. ["cpu", "memory"]).
23. Why is VPA discouraged on singleton StatefulSets? — Evictions in Auto/Recreate mode cause real downtime with no redundant replica to absorb it.
24. What kind of workloads is full Auto mode most appropriate for? — Stateless, horizontally-redundant workloads with generous PodDisruptionBudgets.
25. Does the Recommender ever directly modify a running Pod? — No — it only writes recommendations to the VPA object's status; the Updater and Admission Controller act on them.
26. What model does the Recommender use internally? — A decayed histogram of usage per container, maintained in-memory (optionally checkpointed).
27. What's a known operational cost of running VPA in very large clusters? — The Recommender's per-container histograms can become memory-heavy at scale.
28. What API does the Updater use to remove a Pod? — The Eviction API, same mechanism used for node drains.
29. If a Pod's requests are already close to the recommendation, does the Updater evict it? — No — eviction only happens when a Pod is "significantly out of line" with the current recommendation.
30. What's the risk of using Initial mode instead of Recreate/Auto? — Recommendations are only applied at Pod creation — a long-running Pod whose usage drifts afterward never gets resized until it's naturally recreated.
31. Why does the Scheduler have "no direct awareness" of VPA? — By the time the Scheduler sees a Pod, its resources have already been mutated by the Admission Controller — VPA's involvement is invisible to scheduling logic.
32. What kubectl command lists all VPA objects across the cluster? — kubectl get vpa -A.
33. What's the safest way to roll out VPA on a new workload? — Start in updateMode: Off, review recommendations, then graduate to Recreate/Auto deliberately.
34. Why must PodDisruptionBudgets exist before enabling Recreate/Auto mode? — Because the Updater consults the PDB before evicting, and without one (or a too-strict one) evictions may be blocked or cause outages.
35. What's the interview one-liner distinguishing VPA's axis from HPA and Cluster Autoscaler? — HPA changes replica count, VPA changes per-Pod resource sizing, Cluster Autoscaler changes node count — three independent layers of the autoscaling stack.
36. Is VPA a good fit for workloads with rapidly, unpredictably changing usage? — No — VPA reacts slowly (historical/decayed data), so it suits relatively stable, predictable usage patterns.
37. What happens to a Pod's requests if no VPA object targets it? — Nothing — VPA only mutates Pods matched by an active VerticalPodAutoscaler's targetRef.
38. Why is watching a Pod get evicted and recreated a useful way to validate VPA end-to-end? — It confirms the full chain — Recommender computed a new value, Updater evicted, Admission Controller applied it on the replacement Pod.
39. What's the danger of setting maxAllowed too low? — VPA can never recommend enough resources for genuine growth in usage, causing chronic throttling/OOM despite VPA being enabled.
40. What's the danger of setting minAllowed too low? — A quiet period could cause VPA to recommend unrealistically small requests, risking throttling when load returns.
41. How does VPA's evict+recreate approach interact with a Deployment's rolling update logic? — The recreated Pod goes through the same Deployment/ReplicaSet reconciliation as any other replacement Pod — VPA doesn't bypass rollout mechanics.
42. Why is VPA described as "one Pod at a time" disruptive rather than a whole-Deployment resize? — The Updater evicts individual Pods based on their own drift from the recommendation, not the Deployment as a whole.
43. What's the production reason for piping Recommender output into CI checks rather than acting on it live? — To catch egregiously over/under-provisioned workloads and apply the fix deliberately via code review, avoiding automatic disruption.
44. Does VPA require metrics-server to function? — Effectively yes for the Recommender's usage data, mirroring HPA's dependency on the same resource-metrics pipeline.
45. What's a realistic incident VPA in Auto mode without a PDB could cause? — A workload gets evicted below its minimum safe replica count with no PDB to block it, causing an availability drop.
46. Why is Auto currently "reserved for future in-place resize support"? — Kubernetes doesn't yet support resizing a running Pod's resources without recreation, so Auto behaves exactly like Recreate until that capability lands.
47. What's the relationship between VPA's recommendation and QoS class? — Changing requests/limits changes which QoS class (Guaranteed/Burstable/BestEffort) a Pod falls into, tying VPA activity to eviction-priority behavior elsewhere in the cluster.
48. Why should a monitoring/observability team care about VPA recommendations even in Off mode? — They reveal systemic over/under-provisioning trends across services without requiring any live risk of eviction.
49. What's the fastest way to check if VPA's components themselves are healthy? — Check the three component Pods' status in kube-system via their respective app labels.
50. What's the one-line interview summary of VPA? — Three cooperating components (Recommender, Updater, Admission Controller) that observe real usage and right-size a container's requests/limits via evict-and-recreate, gated by an updateMode that ranges from purely advisory to fully automatic.
