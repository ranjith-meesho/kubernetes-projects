Resource Management

Requests, limits, QoS classes, and cluster-wide guardrails that keep noisy neighbors from starving each other.
← Topology Spread Course Index HPA →

Explanation

Kubernetes resource management is the system of requests, limits, Quality of Service (QoS) classes, and cluster-wide guardrails (LimitRange, ResourceQuota) that together decide how CPU and memory are shared across every Pod on every node. A request is what a container is guaranteed to get and what the Scheduler uses to decide which node has room; a limit is the hard ceiling the container is not allowed to exceed, enforced by the kernel via cgroups on the node. These are two entirely separate mechanisms solving two entirely separate problems — placement (requests) and containment (limits) — and conflating them is the single most common source of scheduling and stability confusion in Kubernetes.

Kubernetes needs this because a node's physical CPU and memory are finite and shared across every Pod scheduled to it. Without requests, the Scheduler has no way to reason about whether a node has room for another Pod — it would be bin-packing blind. Without limits, a single misbehaving container (memory leak, runaway CPU loop) can starve every other Pod on the same node, since the kernel has no idea it should be constrained. Requests protect the Scheduler's decision-making; limits protect node stability.

CPU and memory behave very differently once a limit is exceeded, and this distinction matters enormously in production: CPU is a compressible resource — a container exceeding its CPU limit is simply throttled (CFS bandwidth throttling), it keeps running, just slower. Memory is incompressible — a container exceeding its memory limit gets OOM-killed by the kernel, immediately and unconditionally, because there's no way to "throttle" memory usage after the fact.

Every Pod is assigned one of three QoS classes automatically, derived purely from how requests and limits are set (no separate field to configure):

  • Guaranteed — every container has requests == limits, for both CPU and memory. Highest protection; last to be OOM-killed or evicted under node pressure.
  • Burstable — at least one container has a request set, but requests != limits (or limits are unset for some resource). Evicted after BestEffort but before Guaranteed under pressure.
  • BestEffort — no requests or limits set at all. First to be OOM-killed/evicted under node pressure.

Beyond individual Pods, LimitRange objects set per-container/per-Pod defaults and min/max bounds within a namespace (so a forgotten request doesn't default to "unbounded"), while ResourceQuota objects cap the total resource consumption across an entire namespace — the tool platform teams use to prevent one team's namespace from consuming an entire cluster's capacity. Together, these four mechanisms form the complete resource-governance toolkit: requests/limits govern one Pod, LimitRange governs defaults within a namespace, ResourceQuota governs totals within a namespace.

Production incident pattern: a container with no memory limit set (BestEffort or Burstable-with-unset-limit) develops a memory leak. Because there's no ceiling, it slowly consumes the entire node's memory, eventually triggering node-level memory pressure that evicts other, healthy Pods on that node — a single misconfigured Deployment takes down unrelated services. This is why "always set limits" is close to a universal rule.

Internal Working / Interactions

  • Scheduler: reads requests only (never limits) during the Filter phase to determine which nodes have enough allocatable capacity; scoring plugins like NodeResourcesFit use requests to bin-pack or spread.
  • kubelet: configures cgroups (v1 or v2) on the node according to both requests (for the cgroup's relative CPU share / OOM score adjustment) and limits (for the hard cgroup ceiling), then relies on the Linux kernel to enforce them.
  • Kernel (cgroups + OOM killer): throttles CPU via CFS bandwidth control when a container exceeds its CPU limit; invokes the OOM killer immediately when a container exceeds its memory limit, using each Pod's QoS-derived oom_score_adj to decide kill order under system-wide pressure.
  • API Server / admission (LimitRange, ResourceQuota): LimitRange is enforced at admission time (defaults injected, min/max validated) when a Pod is created; ResourceQuota is enforced at admission time by rejecting the request outright if the namespace total would be exceeded.
  • Metrics pipeline (metrics-server): reads actual usage (not requests/limits) from kubelet's cAdvisor-derived stats, feeding HPA/VPA decisions (topics 37-38) — a separate data path from scheduling.

Code Example

apiVersion: v1
kind: Pod
metadata:
  name: checkout-guaranteed
spec:
  containers:
    - name: checkout
      image: registry.internal/checkout-service:2.4.1
      resources:
        requests:            # Scheduler placement guarantee
          cpu: "500m"
          memory: "512Mi"
        limits:               # requests == limits for BOTH cpu and memory -> QoS: Guaranteed
          cpu: "500m"
          memory: "512Mi"

apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: payments
spec:
  limits:
    - type: Container
      default:                 # applied if a container omits limits
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:           # applied if a container omits requests
        cpu: "250m"
        memory: "256Mi"
      max:                      # hard ceiling any container in this namespace may request
        cpu: "2"
        memory: "2Gi"
      min:                      # hard floor
        cpu: "100m"
        memory: "128Mi"
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: payments-quota
  namespace: payments
spec:
  hard:
    requests.cpu: "20"          # sum of all container requests in namespace cannot exceed this
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
    pods: "50"                  # also caps total pod count

kubectl Commands

# See a pod's QoS class directly
kubectl get pod checkout-guaranteed -o jsonpath='{.status.qosClass}'

# View actual resource usage vs requests/limits (requires metrics-server)
kubectl top pod -n payments
kubectl top node

# Inspect a namespace's ResourceQuota usage
kubectl describe resourcequota payments-quota -n payments

# Inspect a namespace's LimitRange defaults
kubectl describe limitrange default-limits -n payments

# Find pods without resource requests/limits set (BestEffort candidates)
kubectl get pods -A -o json | jq -r '.items[] | select(.spec.containers[].resources.requests == null) | .metadata.name'

# See why a pod was OOMKilled
kubectl describe pod  | grep -A5 "Last State"

Best Practices & Common Pitfalls

Do's

  • Always set both requests and limits for memory (incompressible) to avoid one leaky pod destabilizing a node.
  • Use Guaranteed QoS for latency-critical or stateful workloads that must never be preempted casually.
  • Set a LimitRange in every namespace as a safety net against forgotten resource fields.
  • Set a ResourceQuota per team/namespace to enforce fair-share capacity in shared clusters.

Don'ts

  • Don't set a CPU limit far above the request for latency-sensitive services — CFS throttling under contention can cause severe tail-latency spikes even with a generous limit.
  • Don't leave memory limits unset "to be safe" — it removes the isolation that requests/limits are meant to guarantee.
  • Don't confuse requests (scheduling) with limits (runtime enforcement) — Scheduler never looks at limits.
  • Don't set requests far below real usage just to fit more pods per node — it causes node-level overcommit and cascading evictions under load.

50 Interview Questions

1. What's the fundamental difference between a resource request and a resource limit?
Short answer: A request is what the Scheduler uses to guarantee placement and reserve capacity; a limit is the hard runtime ceiling enforced by the kernel via cgroups.
Detailed explanation: Requests solve placement — deciding which node has room; limits solve containment — preventing a container from consuming unbounded resources once running. Conflating the two is the most common source of scheduling/stability confusion.
Production example: checkout-service requests 500m/512Mi (used by the Scheduler) and limits at the same values for Guaranteed QoS.
Common mistake: Assuming the Scheduler considers limits when deciding placement — it only ever reads requests.
2. Why do CPU and memory behave differently when a limit is exceeded?
Short answer: CPU is compressible — the container is throttled but keeps running; memory is incompressible — the container is OOM-killed immediately.
Detailed explanation: CFS bandwidth throttling can pause CPU cycles without killing the process, but there's no equivalent way to "throttle" memory usage after the fact, so the kernel's OOM killer intervenes instead.
Production example: A CPU-limited batch job ran slower under load but stayed up; a memory-limited service with a leak was OOM-killed and restarted repeatedly.
Common mistake: Setting a CPU limit far above request for latency-sensitive services, not realizing CFS throttling under contention can still cause severe tail-latency spikes.
3. What are the three QoS classes and how are they determined?
Short answer: Guaranteed (requests == limits for all resources), Burstable (some requests set but requests != limits), BestEffort (no requests/limits at all) — derived automatically, never set directly.
Detailed explanation: QoS class determines eviction/OOM-kill order under node pressure: BestEffort first, then Burstable, Guaranteed last — it's purely a byproduct of how requests/limits are configured.
Production example: A latency-critical payment service is deliberately configured Guaranteed so it's the last thing evicted under node memory pressure.
Common mistake: Trying to set QoS class as a field directly — it can only be influenced indirectly through requests/limits configuration.
4. What is the difference between a LimitRange and a ResourceQuota?
Short answer: LimitRange sets per-container/per-Pod defaults and min/max bounds within a namespace; ResourceQuota caps total resource consumption across the entire namespace.
Detailed explanation: LimitRange prevents "unbounded by accident" (a forgotten request defaults sensibly); ResourceQuota prevents "one team consumes everything" at the aggregate namespace level — they compose rather than overlap.
Production example: payments namespace has a LimitRange defaulting unset requests to 250m/256Mi, and a ResourceQuota capping total namespace requests.cpu at 20.
Common mistake: Assuming a LimitRange alone prevents a namespace from consuming excessive total cluster capacity — that's ResourceQuota's job.
5. Why is a container with no memory limit set considered dangerous in production?
Short answer: A memory leak in that container can slowly consume the entire node's memory, triggering node-level memory pressure that evicts other, healthy Pods on the same node.
Detailed explanation: Without a ceiling, there's no isolation mechanism protecting neighbors — a single misconfigured Deployment can take down unrelated services sharing the node.
Production example: A BestEffort Pod with a slow memory leak triggered node-wide memory pressure, causing an unrelated Burstable Pod to be evicted first.
Common mistake: Leaving memory limits unset "to be safe," which actually removes the isolation requests/limits are meant to guarantee.
6. How does the Scheduler use resource requests during Filter and Score?
Short answer: It reads requests (never limits) to determine which nodes have enough allocatable capacity, and scoring plugins like NodeResourcesFit use requests to bin-pack or spread.
Detailed explanation: This is why unset requests make a Pod effectively invisible to capacity-aware scheduling — the Scheduler treats it as needing zero resources, which can lead to overcommit.
Production example: A Pod with no CPU request was scheduled onto an already-busy node because the Scheduler saw zero declared demand.
Common mistake: Setting requests far below real usage just to fit more pods per node, causing node-level overcommit and cascading evictions under load.
7. How do requests and limits translate into actual kernel enforcement?
Short answer: kubelet configures cgroups based on both — request informs relative CPU share/OOM score adjustment, limit sets the hard cgroup ceiling — and the Linux kernel enforces them at runtime.
Detailed explanation: The kernel's CFS bandwidth controller throttles CPU when a limit is exceeded, and the OOM killer uses each Pod's QoS-derived oom_score_adj to decide kill order under system-wide memory pressure.
Production example: Under system-wide pressure, the kernel's OOM killer targeted the BestEffort Pod first because of its higher oom_score_adj value.
Common mistake: Assuming Kubernetes itself enforces limits — enforcement is delegated entirely to the kernel via cgroups, Kubernetes just configures them.
8. When is a ResourceQuota enforced, and what happens if it's exceeded?
Short answer: At admission time — the API Server rejects the Pod-creation request outright if the namespace total would exceed the quota.
Detailed explanation: This is a hard, synchronous check during the same admission pipeline stage as LimitRange defaulting, not an after-the-fact alert.
Production example: A deploy attempting to scale past the namespace's pods: "50" quota was rejected immediately with an admission error, not silently queued.
Common mistake: Expecting ResourceQuota violations to show up as a warning or Pending state rather than an outright rejected create request.
9. Why does metrics-server read actual usage rather than requests/limits, and how does that differ from scheduling?
Short answer: metrics-server reports real, observed CPU/memory usage from kubelet's cAdvisor-derived stats — a completely separate data path from the Scheduler's request-based placement decisions.
Detailed explanation: This distinction matters because HPA/VPA (topics 37-38) act on actual usage trends, while the Scheduler only ever reasons about declared requests at placement time — the two systems never share the same numbers.
Production example: A Pod requesting 250m CPU but actually using 800m would schedule fine (Scheduler only sees the request) while HPA reacts to the real 800m usage.
Common mistake: Assuming the Scheduler "sees" real usage the way HPA/VPA do — it only ever sees the declared request value.
10. How would you find Pods without resource requests set as part of a cluster audit?
Short answer: kubectl get pods -A -o json | jq filtering for containers whose resources.requests is null.
Detailed explanation: These are BestEffort candidates — the highest-risk Pods for both scheduling blindness and first-in-line eviction under node pressure.
Production example: A platform-wide audit used this pattern to find and remediate dozens of Deployments silently running BestEffort in production namespaces.
Common mistake: Only checking for missing limits during an audit while overlooking Pods missing requests entirely, which is the more scheduling-relevant gap.
