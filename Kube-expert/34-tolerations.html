Tolerations

The Pod-side permission slip that cancels out a node's taint.
← Taints Course Index Topology Spread →

Explanation

A toleration is a Pod-spec field that declares "I am willing to be scheduled on a node with this specific taint." Tolerations don't attract a Pod to a tainted node the way affinity does — they merely remove the repulsion that the taint would otherwise cause. This distinction trips up almost everyone learning Kubernetes for the first time: adding a toleration for dedicated=ml:NoSchedule to a Pod does not mean that Pod will land on a GPU node — it only means the Scheduler is now allowed to consider that node, alongside every other untainted node in the cluster. To actually steer the Pod onto the tainted node, you must pair the toleration with a node affinity or nodeSelector targeting that pool.

Kubernetes needs tolerations as the mechanism that makes taints usable at all — without them, a taint would be an absolute wall with no door. The toleration is that door, and its fields make the match precise: key, operator, value, and effect must line up with the taint for the toleration to "cancel" it. The operator field has two modes:

  • Equal (default) — requires key, value, and effect to all match the taint exactly.
  • Exists — matches any taint with the given key (and effect, if specified), regardless of value; leaving key empty as well matches all taints, which is how DaemonSets like kube-proxy tolerate everything to guarantee they run on every node including tainted ones.

The tolerationSeconds field only applies to NoExecute taints, and it is one of the more subtle production-relevant knobs in the scheduling system: it defines how long a Pod is allowed to keep running on a node after a NoExecute taint appears (e.g. the node goes NotReady), before it gets forcibly evicted. This is the exact mechanism that determines how quickly a StatefulSet or Deployment reacts to a node failure — set it too high and failover is slow; set it too low (or omit it, letting the default 300s from Kubernetes' built-in tolerations kick in) and you risk premature eviction during brief network blips.

In production, tolerations show up constantly in three patterns: (1) node dedication — a workload tolerates a dedication taint plus carries matching affinity, (2) infrastructure DaemonSets — kube-proxy, CNI plugins, and log shippers tolerate everything (operator: Exists with no key) so they run on every node regardless of taints, and (3) fast failover tuning — critical Deployments explicitly set a short tolerationSeconds on the not-ready/unreachable taints to force faster rescheduling than the cluster default.

Mental model: Taint = "keep out unless." Toleration = "the unless." Affinity/nodeSelector = "please put me here." You almost always need toleration + affinity together for true node dedication — toleration alone just opens the door, it doesn't walk the Pod through it.

Internal Working / Interactions

  • Scheduler (TaintToleration plugin): during Filter, compares each node's taint list against the Pod's toleration list; any taint without a matching toleration eliminates that node as a candidate (for NoSchedule/NoExecute effects).
  • API Server: stores tolerations as part of Pod.spec.tolerations; validated for shape (key/operator/value/effect/tolerationSeconds) but not semantically checked against real node taints at admission time.
  • Node Controller: when it applies an automatic NoExecute taint (node NotReady/unreachable), the eviction logic checks each running Pod's tolerations and tolerationSeconds to schedule (or skip) eviction.
  • kubelet: executes the actual eviction once the eviction controller's timer expires for a given Pod — it does not decide timing itself.
  • DaemonSet controller: automatically injects tolerations for several built-in node conditions (memory pressure, disk pressure, not-ready, unreachable) onto DaemonSet pods, unless the DaemonSet spec overrides them, so DaemonSets keep running everywhere by default.

Code Example

apiVersion: v1
kind: Pod
metadata:
  name: ml-training-job
  labels:
    workload: ml
spec:
  tolerations:
    - key: "dedicated"          # must match the node's taint key exactly
      operator: "Equal"         # requires value to match too
      value: "ml"
      effect: "NoSchedule"      # only cancels NoSchedule taints with this key/value
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: node-pool
                operator: In
                values: ["gpu-pool"]   # toleration opens the door, affinity walks through it
  containers:
    - name: trainer
      image: registry.internal/ml-trainer:3.1
      resources:
        limits:
          nvidia.com/gpu: 1

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-shipper
  namespace: kube-system
spec:
  selector:
    matchLabels: { app: log-shipper }
  template:
    metadata:
      labels: { app: log-shipper }
    spec:
      tolerations:
        # Tolerate EVERYTHING (no key specified with Exists) so this DaemonSet
        # runs on every node in the cluster, including control-plane and dedicated pools.
        - operator: "Exists"
        # Explicit short grace period for the not-ready taint, so log shipping stops
        # quickly on a genuinely failed node instead of lingering for the 300s default.
        - key: "node.kubernetes.io/not-ready"
          operator: "Exists"
          effect: "NoExecute"
          tolerationSeconds: 30
      containers:
        - name: shipper
          image: registry.internal/log-shipper:1.2

Best Practices & Common Pitfalls

Do's

  • Always pair a toleration with node affinity/nodeSelector when the goal is true node dedication.
  • Set explicit tolerationSeconds for latency-sensitive workloads instead of relying on cluster defaults.
  • Use operator: Exists with no key sparingly and only for genuinely cluster-wide infrastructure (CNI, kube-proxy, log/metrics agents).
  • Keep toleration keys/values consistent with a documented taint naming convention across the org.

Don'ts

  • Don't assume a toleration alone will place a Pod on the tainted node — it only removes the block.
  • Don't tolerate the built-in not-ready/unreachable taints indefinitely on stateless services — you'll keep routing traffic to a pod on a dead node longer than necessary.
  • Don't copy-paste broad "tolerate everything" configs onto regular application workloads — it defeats the purpose of node dedication taints.
  • Don't forget effect matters — a toleration for NoSchedule does not cancel a NoExecute taint with the same key/value.

50 Interview Questions

1. What is a toleration and what does it actually do?
Short answer: It declares that a Pod is willing to be scheduled on a node carrying a specific taint — it removes a repulsion, it does not create attraction.
Detailed explanation: Adding a toleration only makes the Scheduler *consider* the tainted node alongside every other untainted node; it never steers the Pod there specifically.
Production example: An ML Pod tolerates dedicated=ml:NoSchedule but also needs a node affinity/selector to actually land on the GPU pool.
Common mistake: Believing a toleration alone guarantees placement on the tainted node — this is the single most common first-time misunderstanding.
2. What is the difference between the Equal and Exists toleration operators?
Short answer: Equal (default) requires key, value, and effect to all match exactly; Exists matches any taint with the given key (and effect, if specified) regardless of value.
Detailed explanation: Leaving key empty as well with Exists matches every taint on the node — a "tolerate everything" toleration.
Production example: kube-proxy uses operator: Exists with no key so it runs on every node including tainted ones.
Common mistake: Using Exists broadly on regular application workloads, accidentally defeating the purpose of node dedication taints.
3. What is tolerationSeconds and when does it apply?
Short answer: It only applies to NoExecute taints, defining how long a Pod may keep running after the taint appears before being evicted.
Detailed explanation: It directly tunes failover speed after a node goes NotReady/unreachable — too high delays rescheduling, too low risks premature eviction during brief blips.
Production example: A log-shipper DaemonSet sets tolerationSeconds: 30 on the not-ready taint to stop shipping quickly from a genuinely failed node instead of the 300s default.
Common mistake: Setting a critical stateful workload's tolerationSeconds too low, causing premature eviction during transient network issues.
4. Why do DaemonSets like kube-proxy and CNI plugins tolerate everything?
Short answer: Because they must run on literally every node — including control-plane and dedicated pools — to provide cluster-wide infrastructure.
Detailed explanation: A CNI plugin missing from even one node breaks networking for every Pod scheduled there; the DaemonSet controller also auto-injects tolerations for several built-in node conditions to reinforce this.
Production example: log-shipper's DaemonSet spec includes a bare operator: Exists toleration with no key, ensuring cluster-wide coverage.
Common mistake: Copy-pasting this "tolerate everything" pattern onto regular application Deployments, defeating dedication taints.
5. Why is toleration + affinity the correct pairing for true node dedication?
Short answer: The toleration opens the door (removes the block); the affinity/nodeSelector walks the Pod through it (actually places it there).
Detailed explanation: Without the affinity half, the tolerating Pod is merely eligible for the tainted node among all other candidates — it could still land anywhere else.
Production example: ml-training-job carries both a toleration for dedicated=ml:NoSchedule and a required nodeAffinity for node-pool=gpu-pool.
Common mistake: Adding only the toleration and being confused when Pods don't consistently land on the dedicated pool.
6. What validates a Pod's toleration fields at admission time?
Short answer: The API Server validates shape only — key/operator/value/effect/tolerationSeconds structure — not semantic correctness against real node taints.
Detailed explanation: A toleration referencing a taint key that doesn't exist anywhere in the cluster is still accepted; it simply has no effect until such a taint exists.
Production example: A typo'd toleration key silently did nothing for months because the API Server never rejected it, and no matching taint ever existed.
Common mistake: Assuming a syntactically valid toleration must be semantically meaningful in the current cluster.
7. How does the eviction timer for NoExecute taints actually work internally?
Short answer: When a NoExecute taint appears, the eviction logic starts a per-Pod timer based on that Pod's tolerationSeconds, and kubelet terminates the Pod locally once it expires.
Detailed explanation: Different Pods on the same node can have different tolerationSeconds, so eviction timing is computed independently per Pod, not once for the whole node.
Production example: On a node going NotReady, Pod A (tolerationSeconds=0) was evicted immediately while Pod B (tolerationSeconds=300) kept running for 5 more minutes until the node recovered.
Common mistake: Assuming all Pods on a node are evicted simultaneously the instant a NoExecute taint appears.
8. What's the difference between omitting effect and specifying it in a toleration?
Short answer: Omitting effect makes the toleration match the key/value combination against any effect; specifying it restricts the match to that exact effect only.
Detailed explanation: A toleration for NoSchedule explicitly does not cancel a NoExecute taint with the same key/value — effect must line up unless intentionally left blank.
Production example: A team tolerated only the NoSchedule effect of a taint, and was surprised when their Pods were evicted anyway once the same key/value taint was later reapplied with NoExecute.
Common mistake: Assuming tolerating one effect for a key/value automatically covers all effects for that same key/value.
9. Why is a toleration alone insufficient to reason about where a Pod will actually run?
Short answer: Because tolerations only cancel repulsion — the actual placement decision still depends on affinity, resource fit, and every other Filter/Score criterion.
Detailed explanation: A tolerating Pod remains eligible for every untainted node too, so without a targeting mechanism it could land anywhere satisfying its other constraints.
Production example: A tolerating-only Pod (no affinity) occasionally landed on a general-purpose node instead of the intended dedicated pool, surprising the on-call engineer.
Common mistake: Treating "has a toleration for the pool's taint" as proof the Pod will run in that pool.
10. How would you audit which Pods tolerate a given taint across a namespace?
Short answer: kubectl get pods -n <ns> -o custom-columns=NAME:.metadata.name,TOLERATIONS:.spec.tolerations.
Detailed explanation: This surfaces the raw toleration list per Pod for manual cross-referencing against the node taints returned by kubectl get node <name> -o jsonpath='{.spec.taints}'.
Production example: A platform team audited kube-system to confirm every DaemonSet's auto-injected tolerations still matched current node taints after a taint-naming convention change.
Common mistake: Trying to infer toleration coverage from Deployment YAML alone without checking the DaemonSet controller's auto-injected tolerations too.
11–50. Additional rapid-fire questions
11. What four fields make up a toleration? — key, operator, value, and effect (plus optional tolerationSeconds for NoExecute).
12. What's the default operator if none is specified? — Equal.
13. Does a toleration ever cause a Pod to be scheduled where it otherwise wouldn't be? — No, it only removes a block; it never adds attraction.
14. Which Scheduler plugin evaluates tolerations against node taints? — The TaintToleration plugin, during the Filter phase.
15. What happens if a Pod's toleration key doesn't match any real taint? — Nothing — it's inert; the toleration simply has no taint to cancel.
16. Why does the DaemonSet controller auto-inject tolerations for certain node conditions? — So DaemonSets keep running everywhere by default, including on nodes under memory/disk pressure or not-ready.
17. What's the risk of setting tolerationSeconds too high on a customer-facing service? — Traffic keeps routing to a Pod on a dead node longer than necessary before failover occurs.
18. What's the risk of setting it too low? — Premature eviction during brief, recoverable network blips.
19. Can a Pod have multiple tolerations? — Yes, spec.tolerations is a list, one entry per taint it needs to cancel.
20. How do you view a Pod's tolerations directly? — kubectl get pod <name> -o jsonpath='{.spec.tolerations}'.
21. What's a realistic three-pattern summary of production toleration usage? — Node dedication (toleration + affinity), infrastructure DaemonSets (tolerate everything), and fast-failover tuning (short tolerationSeconds).
22. Why is toleration described as "the door" in the mental model callout? — Because the taint is the wall, and the toleration is what specifically opens a path through it for one Pod.
23. Does kubelet decide when to evict a Pod under a NoExecute taint? — No, it only executes the eviction once the controller's timer expires; it doesn't decide the timing itself.
24. What's the danger of a broad "operator: Exists" toleration with no key on a regular app Deployment? — It bypasses every dedication taint in the cluster, letting the workload land anywhere, defeating isolation intent.
25. How would you debug a Pod stuck Pending that you expect to tolerate a specific taint? — Compare the Pod's tolerations against the target node's taints field-by-field, and check FailedScheduling events.
26. What's the interaction between tolerations and Cluster Autoscaler node group selection? — CA must know which node groups a Pending Pod could tolerate before deciding which group to scale up.
27. Why is the "not-ready" built-in toleration relevant to StatefulSet failover speed? — Its tolerationSeconds setting (or lack thereof) directly determines how fast a StatefulSet reacts to a node going down.
28. Does a toleration for NoSchedule cancel a NoExecute taint with the same key/value? — No, effect must match (or be omitted) for the toleration to apply.
29. What operator would you use to tolerate a taint regardless of its value? — Exists.
30. Is there any way to make a toleration also express a scheduling preference? — No — that requires pairing it separately with a node affinity or nodeSelector.
31. What's the biggest first-time misconception new engineers have about tolerations? — Assuming a toleration alone places the Pod on the tainted node.
32. How do infrastructure DaemonSets avoid manually listing every possible node condition taint? — The DaemonSet controller auto-injects tolerations for several built-in conditions unless overridden.
33. What's a good naming-convention practice across an org for tolerations? — Keep toleration keys/values consistent with a documented taint naming convention so teams don't guess at meaning.
34. Why would you explicitly override the DaemonSet controller's default not-ready tolerationSeconds? — To make node-failure response faster (or slower) than the cluster default of 300s for that specific workload.
35. What's the relationship between toleration key/value/effect and a taint's key/value/effect? — They must correspond exactly (for Equal) or partially (for Exists) for the toleration to cancel that specific taint.
36. Can a toleration be added after a Pod is already running, to stop a pending eviction? — No — toleration changes require the Pod to be recreated with the new spec; you can't hot-patch a running Pod's tolerations.
37. What's a scenario where toleration alone (no affinity) is actually the right design? — Infrastructure DaemonSets that genuinely need to run everywhere, where "everywhere" is exactly the desired placement.
38. Why is a short tolerationSeconds appropriate for a log-shipper but not for a stateful database? — Log shipping tolerates fast disruption with no data-consistency risk; a database needs longer grace to avoid unnecessary failover churn.
39. What kubectl command cross-checks toleration vs taint fastest? — Running kubectl get pod ... -o jsonpath='{.spec.tolerations}' next to kubectl get node ... -o jsonpath='{.spec.taints}'.
40. Does a missing toleration ever cause a Pod that's already running to be evicted retroactively? — Only if a new NoExecute taint is subsequently applied to its node — the Pod's existing tolerations are then checked against it.
41. What's the failure mode of forgetting to add a toleration when moving a workload to a newly dedicated pool? — The Pod simply can't schedule there at all, staying Pending or landing elsewhere.
42. How does the built-in default toleration for not-ready/unreachable interact with custom short tolerationSeconds? — A Pod's own explicit tolerationSeconds overrides the cluster's implicit default for that specific taint.
43. Is toleration evaluation part of the Score phase or only Filter? — Primarily Filter (eligibility); it doesn't itself contribute scoring weight the way PreferNoSchedule taint handling does.
44. Why would a platform team want to keep toleration usage minimal on regular application teams' Deployments? — To preserve the isolation guarantees of dedication taints — indiscriminate tolerations dilute their purpose.
45. What's the operational value of auditing all tolerations in kube-system periodically? — Confirming infrastructure DaemonSets still match current taint conventions after naming or effect changes.
46. How does a toleration's tolerationSeconds field behave if the node recovers before the timer expires? — The eviction timer is cancelled since the NoExecute taint is removed once the node is healthy again.
47. What's the core one-line distinction for interviews between taint and toleration? — Taint = "keep out unless"; toleration = "the unless" — the permission slip that cancels a specific taint.
48. Why is it misleading to call tolerations a "scheduling preference"? — Because they express permission/eligibility, not preference — they never influence which eligible node is actually chosen.
49. What's a realistic interview follow-up scenario: "your pod isn't landing on the dedicated pool despite a toleration"? — Check whether a corresponding node affinity/selector exists — the toleration alone never guarantees placement.
50. What's the safest default posture for toleration usage across a large platform? — Reserve broad "tolerate everything" tolerations for genuine cluster-wide infrastructure only, and require explicit key/value/effect matches everywhere else.
