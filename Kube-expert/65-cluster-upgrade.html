Cluster Upgrade

Moving a live cluster to a new Kubernetes version safely: version skew rules, control plane first, then nodes, one at a time.
← Disaster Recovery Course Index GKE →

Explanation

Kubernetes ships a new minor version roughly every four months, and each minor version is supported (patched) for about 14 months. This forces every operator into a recurring upgrade cadence whether they like it or not — staying put is not actually an option, because falling off the support window means no more CVE patches. The core discipline of a safe cluster upgrade is respecting the version skew policy: kube-apiserver is the "most current" component; kube-controller-manager, kube-scheduler, and cloud-controller-manager may be up to one minor version older than the API server; kubelet may be up to three minor versions older (as of recent skew policy); kubectl may be one minor version newer or older. This policy exists because upgrading everything atomically across a live, multi-thousand-node fleet is operationally impossible — the skew window is what makes rolling upgrades safe at all.

The practical consequence of the skew policy is a strict ordering: you always upgrade the control plane first (etcd, then API server, then controller-manager/scheduler), one minor version at a time — you cannot skip from 1.27 to 1.29 directly, each minor version must be applied in sequence because internal APIs and storage formats change incrementally and skipping breaks that assumption. Only after the control plane is fully upgraded do you upgrade worker nodes, and you never upgrade nodes past the API server's version (a newer kubelet than API server is not supported). Nodes are upgraded one at a time (or one batch at a time in a large fleet), always by first draining the node — cordoning it so no new pods schedule there, then evicting existing pods so they reschedule onto still-current nodes — before touching its kubelet/kubeadm/OS packages.

kubeadm upgrade plan and kubeadm upgrade apply encode this entire workflow for self-managed clusters: plan tells you what versions are available and what the skew constraints allow next, apply performs the actual control-plane component upgrade (including any etcd or API storage migrations) on the first control-plane node, and subsequent control-plane nodes use kubeadm upgrade node. Worker nodes get their kubelet/kubeadm packages upgraded via the OS package manager, then kubeadm upgrade node updates the local kubelet config, followed by a kubelet restart.

The operational risk in a cluster upgrade is almost never the Kubernetes version bump itself — it's the ripple effects: deprecated/removed API versions breaking manifests or third-party controllers that still reference them (e.g. a Helm chart still using extensions/v1beta1 after it's removed), admission webhooks that fail closed during the upgrade window, or a CNI/CSI driver version incompatible with the new kubelet. This is why every serious upgrade runbook starts with running kubectl-convert or tools like pluto/kube-no-trouble against the cluster to find deprecated API usage, and tests the upgrade against a staging cluster that mirrors production traffic and manifests before touching anything live.

Managed Kubernetes (EKS/GKE/AKS) automates the control-plane half of this entirely — you click or CLI-trigger a version bump and the provider handles etcd/API-server/controller-manager upgrades with their own tested rollback paths. What remains your responsibility even on managed clusters is node-pool upgrades (often also automatable, but you choose the surge/drain strategy) and, critically, fixing deprecated API usage in your own manifests before the provider forces the version forward.

Internal Working / Interactions

  • Version skew enforcement: kube-apiserver rejects connections from components outside the supported skew window at startup/handshake.
  • kubeadm upgrade plan: queries the current cluster version and the kubeadm binary's known target versions, and reports the safe next step.
  • kubeadm upgrade apply: pulls new control-plane component images, updates static pod manifests in /etc/kubernetes/manifests, and the kubelet on that node restarts the pods automatically since it watches that directory.
  • Node drain: cordon marks the node unschedulable; evict triggers the eviction API (respecting PodDisruptionBudgets), and the ReplicaSet/Deployment controller recreates evicted pods on other nodes.
  • kubelet upgrade + restart: new kubelet re-registers with the API server; DaemonSets and system pods are recreated by their controllers on the now-current node.
  • Uncordon: the node becomes schedulable again only after you've confirmed its health.

Code Example

#############################################
# 1. kubeadm control-plane upgrade (first CP node)
#############################################
apt-get update && apt-get install -y kubeadm=1.29.0-1.1   # install target kubeadm version first
kubeadm upgrade plan                                       # shows allowed next version + component versions
kubeadm upgrade apply v1.29.0                               # upgrades control plane components on THIS node

# On additional control-plane nodes (not the first)
apt-get install -y kubeadm=1.29.0-1.1
kubeadm upgrade node                                        # applies the already-decided upgrade locally

# Upgrade kubelet + kubectl on every control-plane node
apt-get install -y kubelet=1.29.0-1.1 kubectl=1.29.0-1.1
systemctl daemon-reload && systemctl restart kubelet

---
# 2. Worker node upgrade loop (run per node, one at a time)
kubectl cordon worker-3                                     # stop new pods from scheduling here
kubectl drain worker-3 --ignore-daemonsets --delete-emptydir-data   # evict pods, respect PDBs

ssh worker-3 -- "apt-get install -y kubeadm=1.29.0-1.1 && \
  kubeadm upgrade node && \
  apt-get install -y kubelet=1.29.0-1.1 kubectl=1.29.0-1.1 && \
  systemctl daemon-reload && systemctl restart kubelet"

kubectl uncordon worker-3                                   # allow scheduling again once healthy

---
# 3. Pre-upgrade deprecated-API check (run BEFORE touching anything)
# pluto detects use of deprecated/removed Kubernetes APIs across manifests in-cluster
pluto detect-all-in-cluster -t k8s=v1.29.0

kubectl / CLI Commands

# See what kubeadm believes is the safe next upgrade step
kubeadm upgrade plan

# Check current node versions across the fleet (spot skew issues early)
kubectl get nodes -o custom-columns=NAME:.metadata.name,VERSION:.status.nodeInfo.kubeletVersion

# Cordon a node (stop new scheduling) ahead of upgrade
kubectl cordon worker-3

# Drain a node safely, respecting PodDisruptionBudgets
kubectl drain worker-3 --ignore-daemonsets --delete-emptydir-data

# Confirm PDBs won't block a drain unexpectedly
kubectl get pdb -A

# Bring a node back into service after upgrade
kubectl uncordon worker-3

# Watch overall rollout / node readiness during the upgrade window
kubectl get nodes -w

Best Practices & Common Pitfalls

Best Practices

  • Upgrade one minor version at a time, control plane before nodes, always.
  • Scan for deprecated/removed API usage (pluto, kube-no-trouble) before every upgrade.
  • Test the exact upgrade path on a staging cluster with production-like manifests first.
  • Always cordon + drain before touching a node's kubelet, respecting PodDisruptionBudgets.
  • Take an etcd snapshot immediately before any control-plane upgrade.

Common Pitfalls

  • Skipping minor versions (e.g. 1.27 straight to 1.29) — unsupported and risky.
  • Upgrading kubelets ahead of the API server, violating version skew policy.
  • Draining without --ignore-daemonsets, causing the drain to hang forever.
  • No pre-upgrade check for deprecated APIs, breaking Helm charts/webhooks mid-upgrade.
  • Upgrading all worker nodes simultaneously instead of a controlled rolling batch.

50 Interview Questions

1. What is the Kubernetes version skew policy, and why does it exist?
Short answer: kube-apiserver is the "most current" component; controller-manager/scheduler/cloud-controller-manager may lag by up to one minor version; kubelet may lag by up to three; kubectl may be one minor version newer or older.
Detail: It exists because upgrading every component atomically across a live, multi-thousand-node fleet is operationally impossible — the skew window is what makes rolling, staged upgrades safe at all.
Production example: During a rolling worker-node upgrade, some nodes still run an older kubelet while the already-upgraded API server continues to serve them correctly within the supported skew.
Common mistake: Upgrading kubelets ahead of the API server, which violates the policy since kubelet must never be newer than the API server.
2. Why must minor versions be upgraded one at a time instead of skipping (e.g. 1.27 straight to 1.29)?
Short answer: Internal APIs and storage formats change incrementally between minor versions, and skipping breaks the assumption that each upgrade step is a well-tested, supported delta.
Detail: kubeadm upgrade plan and apply are built around sequential minor-version steps — there's no supported path that jumps two minors at once.
Production example: A cluster on 1.27 must go through 1.28 before reaching 1.29, even if the end goal is 1.29.
Common mistake: Trying to save time by jumping minor versions directly, which is unsupported and can leave the cluster in an inconsistent state.
3. Why is the control plane always upgraded before worker nodes, never after?
Short answer: A newer kubelet than the API server is not supported — the API server must always be at or above the version of every other component.
Detail: The order is etcd, then API server, then controller-manager/scheduler, then worker nodes one at a time — each step depends on the previous one being complete and healthy.
Production example: kubeadm upgrade apply v1.29.0 runs on the first control-plane node before any worker node's kubeadm/kubelet packages are touched.
Common mistake: Upgrading a worker node's kubelet before the control plane is fully on the target version, breaking version skew.
4. What does draining a node actually do, and why is it required before upgrading its kubelet?
Short answer: Cordoning marks the node unschedulable (no new Pods land there); draining then evicts existing Pods so they reschedule onto still-current nodes before you touch the kubelet/OS packages.
Detail: The eviction respects PodDisruptionBudgets, so a drain can pause or fail if it would violate availability guarantees rather than silently causing an outage.
Production example: kubectl cordon worker-3 && kubectl drain worker-3 --ignore-daemonsets --delete-emptydir-data is run before any package upgrade on that node.
Common mistake: Draining without --ignore-daemonsets, which causes the drain to hang forever since DaemonSet-managed Pods aren't meant to be evicted that way.
5. What's the difference between kubeadm upgrade apply and kubeadm upgrade node?
Short answer: apply performs the actual control-plane component upgrade (including any etcd/API storage migrations) on the first control-plane node; node applies the already-decided upgrade on subsequent control-plane or worker nodes.
Detail: Only the first control-plane node "decides" the upgrade via apply; every other node just follows that decision locally via node, keeping the process consistent across the fleet.
Production example: After running kubeadm upgrade apply v1.29.0 on control-plane node 1, nodes 2 and 3 each run kubeadm upgrade node instead of apply again.
Common mistake: Running upgrade apply on every control-plane node, which is unnecessary and not the documented flow.
6. Why is scanning for deprecated/removed API usage (e.g. with pluto) a required pre-upgrade step?
Short answer: The Kubernetes version bump itself is rarely the risky part — the ripple effect of removed API versions breaking manifests, Helm charts, or third-party controllers is the real operational risk.
Detail: A Helm chart still referencing a removed API group (e.g. old extensions/v1beta1) simply fails to apply post-upgrade if nobody checked beforehand.
Production example: pluto detect-all-in-cluster -t k8s=v1.29.0 is run against the live cluster before touching anything, to catch deprecated API usage in advance.
Common mistake: Skipping the deprecated-API check and discovering broken webhooks or Helm releases only after the upgrade is already in progress.
7. Why do admission webhooks pose a specific risk during a cluster upgrade?
Short answer: A webhook that fails closed (denies requests when unreachable) during the upgrade window can block legitimate API operations, including parts of the upgrade process itself.
Detail: Webhook endpoints can become briefly unavailable during a rolling upgrade of the workloads that host them, so their failure policy and timeout configuration matter as much as the webhook logic itself.
Production example: An upgrade runbook explicitly checks webhook health and considers temporarily setting failurePolicy: Ignore for non-critical webhooks during the maintenance window.
Common mistake: Assuming webhooks will "just keep working" through an upgrade without verifying their availability and failure policy beforehand.
8. Why should you take an etcd snapshot immediately before any control-plane upgrade?
Short answer: A control-plane upgrade can include etcd/API storage migrations, and having a pre-upgrade snapshot gives you a known-good rollback point if anything goes wrong.
Detail: This links directly to the Backup & Restore discipline — an upgrade is exactly the kind of planned, higher-risk operation that should always be preceded by a fresh, verified backup.
Production example: The upgrade runbook's first executable step is etcdctl snapshot save, before kubeadm upgrade apply is ever run.
Common mistake: Relying on whatever the last scheduled snapshot happened to be instead of taking a fresh one immediately before the upgrade.

9. How often does Kubernetes ship a new minor version, and how long is each supported? — Roughly every four months, with about 14 months of patch support per minor version.
10. Why is "staying on the current version forever" not actually an option? — Falling off the support window means no more CVE patches, forcing an eventual upgrade regardless of preference.
11. What does kubeadm upgrade plan actually check? — The current cluster version against the kubeadm binary's known target versions, reporting the safe next upgrade step.
12. Why must kubelet never be newer than the API server? — The API server enforces version skew at connection/handshake time and this direction of skew isn't supported.
13. What triggers static pods to restart after a control-plane component upgrade via kubeadm? — kubeadm updates the manifest files in /etc/kubernetes/manifests, and kubelet watches that directory, automatically restarting the pods.
14. Why should worker nodes be upgraded one at a time (or in controlled batches), not all simultaneously? — Simultaneous upgrades remove capacity everywhere at once, risking an outage instead of a controlled rolling change.
15. What happens to Pods on a node when it's cordoned but not yet drained? — Existing Pods keep running; only new scheduling onto that node is blocked.
16. Why does draining respect PodDisruptionBudgets rather than forcing eviction unconditionally? — To prevent an upgrade-triggered drain from itself causing an availability-violating outage.
17. What's the risk of testing an upgrade only on a minimal staging cluster with no real manifests? — It won't surface deprecated API usage or webhook issues that only exist in production's actual manifest set.
18. Why does managed Kubernetes (EKS/GKE/AKS) reduce upgrade risk but not eliminate it? — The provider automates the control-plane half with tested rollback paths, but you're still responsible for node-pool upgrades and fixing deprecated API usage in your own manifests.
19. What's the correct order of component upgrade within the control plane itself? — etcd first, then API server, then controller-manager/scheduler.
20. Why is kubectl allowed to be one minor version newer or older than the API server? — Client tooling has looser compatibility requirements than server-side components that actively coordinate cluster state.
21. What's a realistic symptom of skipping the deprecated-API check before upgrading? — A Helm release or controller referencing a removed API version fails to apply or reconcile right after the upgrade completes.
22. Why does `kubeadm upgrade node` not require re-deciding the target version? — The version decision was already made by `kubeadm upgrade apply` on the first control-plane node; other nodes just apply that same decision.
23. What's the purpose of `kubectl uncordon` at the end of a node upgrade? — Marks the node schedulable again, but only after confirming it's healthy post-upgrade — restoring it to service.
24. Why should uncordoning happen only after health verification, not immediately after the kubelet restarts? — A kubelet that restarted "successfully" could still be unhealthy in ways only visible a few minutes later (e.g. CNI/CSI compatibility issues).
25. What's a version-skew-related reason a CNI or CSI driver upgrade might be needed alongside a Kubernetes upgrade? — A new kubelet version may require a compatible CNI/CSI driver version — mismatches can break networking or storage post-upgrade.
26. Why is "the Kubernetes version bump itself" rarely the actual risk in a real upgrade? — Ripple effects — deprecated APIs, incompatible webhooks, driver mismatches — cause the vast majority of real upgrade incidents, not the core version change.
27. What does `pluto` or `kube-no-trouble` actually scan for? — Usage of Kubernetes APIs that are deprecated or already removed in the target version, across live cluster manifests.
28. Why is testing the exact upgrade path (not just "some" upgrade) on staging important? — Different version jumps can have different deprecations/migrations; testing the specific path you'll actually run in production is what catches real issues.
29. What's the risk of not tracking kubelet version drift across a large fleet? — Nodes could silently fall outside the supported skew window relative to a newly upgraded API server, causing connection issues.
30. How do you check current kubelet versions across all nodes quickly? — kubectl get nodes -o custom-columns=NAME:.metadata.name,VERSION:.status.nodeInfo.kubeletVersion.
31. Why does upgrading in controlled batches matter even on a large fleet with automation? — Even automated rolling upgrades need bounded batch sizes to avoid removing too much capacity at once if something goes wrong mid-batch.
32. What's a common cause of a drain hanging indefinitely? — Missing `--ignore-daemonsets`, or a PodDisruptionBudget that can never be satisfied given current replica counts.
33. Why should you scan for deprecated APIs against the *target* version, not just current best practices? — Deprecation/removal is version-specific; a manifest fine on 1.27 might use an API removed exactly in 1.29.
34. What operational responsibility remains yours even on a fully managed control plane? — Node-pool upgrades (surge/drain strategy) and fixing deprecated API usage in your own manifests before the provider forces the version forward.
35. Why is "one minor version at a time" specifically enforced by kubeadm tooling rather than just documented policy? — `kubeadm upgrade plan`/`apply` are built to only offer the next sequential minor version as a valid target, preventing accidental version skipping.
36. What's the purpose of installing the target kubeadm package version before running `kubeadm upgrade apply`? — The kubeadm binary itself needs to be the target version to correctly plan and execute that version's upgrade steps.
37. Why does `systemctl restart kubelet` follow a kubelet package upgrade rather than happening automatically? — The new kubelet binary must be explicitly loaded into the running process via a restart; installing the package alone doesn't swap the running binary.
38. What's a good way to validate an upgrade didn't silently break workloads? — Checking `kubectl get pods -A` for unexpected restarts/crashes and reviewing logs for API deprecation warnings post-upgrade, not just confirming node readiness.
39. Why is the version skew policy described as "what makes rolling upgrades safe at all"? — Without a tolerated skew window, every component across the entire fleet would need to upgrade atomically, which is operationally impossible at scale.
40. What's the risk of upgrading all worker nodes in a single batch instead of a controlled rolling batch? — Loses the safety margin of catching an issue on a small batch before it affects the whole fleet.
41. Why must cloud-controller-manager also respect the version skew policy? — It's a control-plane component interacting with the API server just like controller-manager/scheduler, subject to the same one-minor-version lag limit.
42. What does an upgrade runbook typically verify right after `kubeadm upgrade apply` completes? — That control-plane pods restarted successfully and the API server is healthy before proceeding to the next control-plane node.
43. Why should Helm charts be reviewed for API version compatibility specifically before a cluster upgrade? — Charts pin specific `apiVersion` values in templates, and a chart last updated years ago may still target a removed API group.
44. What's the value of `kubectl get pdb -A` before starting a rolling worker upgrade? — Confirms PodDisruptionBudgets won't unexpectedly block drains partway through the upgrade sequence.
45. Why is watching `kubectl get nodes -w` useful during an upgrade window? — Gives real-time visibility into node readiness transitions, letting you catch a stuck or failed node upgrade quickly.
46. What's the danger of treating a cluster upgrade as "just a version bump" without a runbook? — Skips the deprecated-API scan, staging test, and etcd snapshot steps that catch the actual common failure modes.
47. Why does the skew policy allow more lag for kubelet than for controller-manager/scheduler? — Worker nodes vastly outnumber control-plane components in a real fleet, and upgrading thousands of kubelets instantly is impractical — a wider skew window accommodates realistic rolling timelines.
48. What's a realistic reason to delay an upgrade even if support window pressure exists? — A critical deprecated-API dependency or an unpatched third-party controller compatibility issue that needs remediation first.
49. Why is "test on staging with production-like manifests" stronger advice than "test on staging"? — Generic staging manifests won't surface the deprecated-API or webhook issues specific to your actual production configuration.
50. What's the overarching operational discipline this topic teaches about upgrades? — Respect strict ordering and version-skew rules, and treat the ripple effects (deprecated APIs, webhooks, drivers) as the real risk, not the version number itself.
