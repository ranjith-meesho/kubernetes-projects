Topology Spread Constraints

Fine-grained, quantifiable even distribution of replicas across zones, nodes, or any topology domain.
← Tolerations Course Index Resource Management →

Explanation

Topology spread constraints (topologySpreadConstraints in the Pod spec) give the Scheduler a quantitative, self-contained way to distribute a group of Pods evenly across a topology domain — a node, a zone, a region, or any custom label you choose as the topologyKey. They solve the same class of availability problem as pod anti-affinity, but with a crucial difference: anti-affinity is binary (co-locate or don't), while topology spread constraints reason about balance using a numeric maxSkew — the maximum allowed difference between the topology domain with the most matching Pods and the domain with the fewest.

Kubernetes needs this because anti-affinity alone doesn't scale well to "spread N replicas evenly across M zones." With required anti-affinity you either forbid all co-location (impossible once replicas outnumber zones) or you don't constrain it at all. Topology spread constraints instead let you say precisely: "I have 3 zones, I want no more than 1 extra replica in any zone compared to the least-loaded zone" — expressed as maxSkew: 1. This produces genuinely balanced distributions (2/2/2, or 3/2/2, never 5/1/0) in a way anti-affinity rules were never designed to guarantee.

The four core fields define the whole behavior:

  • maxSkew — the maximum permitted imbalance between the busiest and least-busy matching topology domain.
  • topologyKey — the node label defining the domain (e.g. topology.kubernetes.io/zone, kubernetes.io/hostname).
  • whenUnsatisfiable — DoNotSchedule (hard: Pod stays Pending if placing it would violate maxSkew) or ScheduleAnyway (soft: the Scheduler still prefers balance via scoring, but places the Pod regardless).
  • labelSelector — which Pods count towards the skew calculation (usually the Deployment's own label, so unrelated Pods in the same domain don't distort the count).

Since Kubernetes 1.19+, this is a first-class scheduler feature (no longer alpha), and since 1.24 there's a cluster-level default constraint (defaultConstraints in the scheduler config, or auto-applied via PodTopologySpread plugin defaults) that applies basic zone/host spreading automatically even if you set nothing — but production teams should still declare explicit constraints for anything availability-critical, since defaults are intentionally soft and generic.

In production, topology spread constraints are the modern replacement for most anti-affinity use cases: a single Deployment usually wants "spread my own replicas evenly," which is exactly what this feature does natively, with better guarantees and lower Scheduler overhead than an equivalent required pod anti-affinity rule at scale. Anti-affinity remains the right tool specifically when the requirement is about a relationship between two different workloads (e.g. "never put my cache and my app on the same node"), which topology spread constraints don't express.

maxSkew intuition: with 3 zones and maxSkew: 1, valid distributions for 5 replicas include 2/2/1 but not 3/1/1 (skew of 2 between busiest and least-busy zone). Lower maxSkew = tighter balance, at the cost of more Pending pods if some zones lack capacity.

Internal Working / Interactions

  • Scheduler (PodTopologySpread plugin): during Filter, computes current Pod counts per topology domain (via its Pod cache) and rejects candidate nodes whose domain would exceed maxSkew when DoNotSchedule is set; during Score, favors domains with fewer matching Pods.
  • API Server: validates and persists topologySpreadConstraints as part of the Pod template; no separate object type.
  • Node labels: the whole mechanism depends on nodes being correctly labeled with the topologyKey (e.g. cloud-provider-injected topology.kubernetes.io/zone) — missing/incorrect labels silently break spreading.
  • Cluster Autoscaler: must account for topology spread constraints when deciding which node group to scale, since adding a node in the wrong zone won't unblock a Pending pod constrained to a different zone.
  • Deployment/ReplicaSet controller: creates Pods with the constraint inherited from the Pod template; it has no awareness of the constraint's effect — that's entirely the Scheduler's job.

Code Example

apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service
  namespace: payments
spec:
  replicas: 6
  selector:
    matchLabels: { app: checkout }
  template:
    metadata:
      labels: { app: checkout }
    spec:
      topologySpreadConstraints:
        - maxSkew: 1                                     # busiest zone can have at most 1 more pod than the least-busy
          topologyKey: topology.kubernetes.io/zone         # domain = AZ
          whenUnsatisfiable: DoNotSchedule                 # hard rule: block scheduling rather than violate skew
          labelSelector:
            matchLabels:
              app: checkout                                # only count OTHER checkout pods, ignore unrelated workloads
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname               # ALSO spread across individual nodes within a zone
          whenUnsatisfiable: ScheduleAnyway                  # soft rule: best-effort, don't block on this one
          labelSelector:
            matchLabels:
              app: checkout
      containers:
        - name: checkout
          image: registry.internal/checkout-service:2.4.1
          resources:
            requests: { cpu: "300m", memory: "256Mi" }

# Cluster-scoped default (kube-scheduler config, not a Pod field) that applies a baseline
# zone + hostname spread to ALL pods lacking their own explicit constraints.
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
profiles:
  - pluginConfig:
      - name: PodTopologySpread
        args:
          defaultConstraints:
            - maxSkew: 3
              topologyKey: topology.kubernetes.io/zone
              whenUnsatisfiable: ScheduleAnyway

Best Practices & Common Pitfalls

Do's

  • Prefer topology spread constraints over pod anti-affinity for single-Deployment even distribution — cheaper for the Scheduler and gives quantifiable balance.
  • Combine a hard zone-level constraint with a soft node-level constraint for layered guarantees.
  • Always scope labelSelector tightly to the workload's own labels to avoid miscounting unrelated pods.
  • Verify node topology labels are present and correct before relying on spreading — missing labels silently disable the constraint.

Don'ts

  • Don't set maxSkew: 0 expecting perfect balance in clusters with uneven zone capacity — it will just produce Pending pods.
  • Don't use DoNotSchedule everywhere without capacity headroom in every zone — it turns capacity shortages into scheduling failures.
  • Don't forget existing pods are never rebalanced — constraints only affect placement of new/rescheduled pods.
  • Don't mix an overly broad labelSelector that accidentally includes canary or debug pods in the skew count.

50 Interview Questions

1. What problem do topology spread constraints solve that anti-affinity doesn't?
Short answer: They provide quantifiable, even distribution of N replicas across M topology domains, rather than a binary co-locate/don't-co-locate rule.
Detailed explanation: Required anti-affinity can't express "spread evenly" once replica count exceeds domain count — it can only forbid all co-location, which becomes infeasible. maxSkew reasons numerically about balance instead.
Production example: checkout-service with 6 replicas across 3 zones uses maxSkew: 1 to guarantee something like 2/2/2, never 5/1/0.
Common mistake: Reaching for anti-affinity to spread a Deployment's own replicas when topology spread constraints are the purpose-built, cheaper tool.
2. What does maxSkew actually mean?
Short answer: The maximum allowed difference between the topology domain with the most matching Pods and the domain with the fewest.
Detailed explanation: With maxSkew: 1 and 3 zones, 5 replicas could land as 2/2/1 but never 3/1/1 (skew of 2 is disallowed).
Production example: Lowering maxSkew from 3 to 1 tightened balance across AZs at the cost of more Pending pods when a zone lacked capacity.
Common mistake: Setting maxSkew: 0 expecting perfect balance regardless of zone capacity — it just produces more Pending pods in uneven clusters.
3. What are the four core fields of a topology spread constraint?
Short answer: maxSkew, topologyKey, whenUnsatisfiable, and labelSelector.
Detailed explanation: maxSkew sets the imbalance tolerance, topologyKey defines the domain (zone/hostname/custom), whenUnsatisfiable decides hard vs soft enforcement, labelSelector scopes which Pods count toward the skew calculation.
Production example: checkout-service scopes labelSelector to app: checkout so unrelated workloads in the same zone don't distort the skew count.
Common mistake: Using an overly broad labelSelector that accidentally includes canary or debug pods in the skew count.
4. What's the difference between DoNotSchedule and ScheduleAnyway?
Short answer: DoNotSchedule is a hard rule — the Pod stays Pending if placing it would violate maxSkew; ScheduleAnyway is soft — the Scheduler still prefers balance via scoring but places the Pod regardless.
Detailed explanation: This mirrors required vs preferred affinity semantics but expressed through the maxSkew/whenUnsatisfiable combination instead.
Production example: checkout-service uses a hard zone-level constraint (DoNotSchedule) layered with a soft node-level constraint (ScheduleAnyway) for defense in depth.
Common mistake: Using DoNotSchedule everywhere without capacity headroom in every zone, turning normal capacity shortages into scheduling failures.
5. Why does topology spread depend on correct node labeling?
Short answer: The whole mechanism relies on nodes being labeled with the chosen topologyKey (e.g. topology.kubernetes.io/zone) — missing/incorrect labels silently disable the constraint.
Detailed explanation: If a node lacks the topologyKey label, the Scheduler can't group it into any domain for skew calculation, effectively excluding it from spread accounting without raising an error.
Production example: A newly added bare-metal node pool lacked zone labels, silently skewing distribution until labels were backfilled.
Common mistake: Assuming topology spread "just works" without first verifying node topology labels are present and correct.
6. When should you still use anti-affinity instead of topology spread constraints?
Short answer: When the requirement is about a relationship between two *different* workloads, not even distribution of one Deployment's own replicas.
Detailed explanation: Topology spread constraints only reason about one workload's own Pod count per domain; they can't express "never put my cache and my app on the same node."
Production example: A cache-and-app co-location exclusion rule stayed on pod anti-affinity even after the app's own replica spreading moved to topology spread constraints.
Common mistake: Trying to force a cross-workload exclusion rule into a topology spread constraint, which it fundamentally can't express.
7. Does Kubernetes apply any topology spreading by default with no explicit constraint?
Short answer: Since 1.24, there's a cluster-level default constraint via the PodTopologySpread plugin that applies basic zone/host spreading automatically.
Detailed explanation: These defaults are intentionally soft and generic — production teams should still declare explicit constraints for anything availability-critical rather than relying on the baseline.
Production example: A team assumed default spreading was "good enough" for a payments-critical service, then added explicit maxSkew constraints after an incident review.
Common mistake: Relying on cluster-level default constraints for availability-critical workloads instead of declaring explicit, workload-specific constraints.
8. Are existing Pods ever rebalanced when a new topology spread constraint is added?
Short answer: No — constraints only affect placement of new or rescheduled Pods, never already-running ones.
Detailed explanation: If a Deployment was already imbalanced before the constraint existed, that imbalance persists until Pods naturally churn or are manually recreated.
Production example: After adding a maxSkew constraint, the existing 4/1/1 distribution stayed put until a rolling restart triggered rescheduling under the new rule.
Common mistake: Expecting topology spread constraints to actively rebalance a cluster the moment they're applied.
9. How would you diagnose a Pod stuck Pending due to a topology spread violation?
Short answer: kubectl describe pod and check Events for a skew-violation message identifying the constraint.
Detailed explanation: The FailedScheduling event explains which topologyKey/maxSkew combination no candidate node could satisfy, similar in style to affinity failure messages.
Production example: "0/6 nodes are available: 3 node(s) didn't satisfy existing pods anti-affinity rules" style output but referencing topology spread skew, traced to a zone at capacity.
Common mistake: Debugging resource requests first when the actual blocker is a skew constraint clearly stated in the Events output.
10. How do Cluster Autoscaler and topology spread constraints interact?
Short answer: Cluster Autoscaler must account for topology spread constraints when choosing which node group to scale, since adding a node in the wrong zone won't unblock a Pod constrained to a different zone.
Detailed explanation: CA simulates scheduling feasibility per node group; a zone-constrained Pending Pod only gets unblocked if CA scales the specific zone/node group that satisfies the skew requirement.
Production example: CA scaled a node group in zone A while the Pending Pod actually needed capacity in zone C to satisfy maxSkew, leaving it Pending despite a successful scale-up elsewhere.
Common mistake: Assuming any Cluster Autoscaler scale-up resolves a Pending Pod, regardless of which zone/node group it targeted.

25. Can topologyKey be a custom label instead of a built-in one? — Yes, any consistently-applied node label can serve as the topology domain.
26. What's the tradeoff of choosing hostname vs zone as topologyKey? — Hostname gives per-node balance (protects against single-node failure); zone gives per-AZ balance (protects against zone outage) — often both are combined.
27. Why might a team prefer topology spread constraints as the org-wide default over anti-affinity? — Lower Scheduler overhead and quantifiable balance guarantees for the common "spread my own replicas" case.
28. What's the impact of setting maxSkew very high (e.g. 10)? — It effectively allows almost any imbalance, making the constraint nearly meaningless for availability purposes.
29. Does a ScheduleAnyway constraint ever block a Pod from scheduling? — No, by definition it only affects scoring/preference, never eligibility.
30. What's the first thing to check if spreading "isn't working" despite a correct constraint? — Whether node topology labels are actually present and correct on all candidate nodes.
31. How does topology spread interact with resource requests during Filter? — Both are independent Filter-phase checks — a node must satisfy skew constraints and have sufficient allocatable resources.
32. What's a common production pattern combining node-level and zone-level constraints? — Hard zone spreading for availability plus soft node-level spreading for even finer-grained balance within a zone.
33. Why is topology spread described as "quantifiable" compared to anti-affinity? — Because maxSkew gives a precise numeric bound on imbalance, rather than anti-affinity's binary allow/forbid co-location.
34. Does topology spread rebalance a cluster after a node is removed? — No, only new/rescheduled Pods are placed under the constraint; existing surviving Pods aren't moved.

38. Why is topology spread considered the "modern replacement" for most anti-affinity use cases? — Because a single Deployment's own even-distribution need maps directly onto maxSkew, with better guarantees than an equivalent anti-affinity rule.
39. What happens to skew calculation if labelSelector matches Pods from multiple unrelated Deployments? — It inflates the perceived count per domain, potentially causing incorrect scheduling decisions for the intended workload.
40. Can you set different maxSkew values for different topologyKeys in the same Pod spec? — Yes, topologySpreadConstraints is a list, each entry independently configured.
41. What's the interview-ready explanation of why maxSkew=0 is often a mistake? — It requires perfectly equal counts across all domains regardless of actual capacity, producing unnecessary Pending pods in uneven clusters.
