Control Plane

The brain of the cluster: how API Server, etcd, Scheduler, Controller Manager, and Cloud Controller Manager operate together as one unit.
← Kubernetes Architecture Course Index Worker Nodes →

Explanation

The control plane is not a single process — it's a set of independent components that collectively make global decisions about the cluster and detect/respond to cluster-wide events. In a self-managed cluster (e.g. kubeadm), these components are: the API Server (the front door), etcd (the source of truth), the Scheduler (decides pod placement), the Controller Manager (runs reconciliation loops), and — if running on a cloud provider — the Cloud Controller Manager (bridges cluster concepts like LoadBalancer Services to cloud-specific APIs).

Kubernetes needs this as a distinct layer because someone has to hold the cluster's desired state durably, decide where workloads run, and continuously reconcile reality toward that desired state — all while tolerating the failure of any individual node, including a control-plane node itself. Separating these concerns into independently replaceable processes (rather than one monolithic binary) is what lets each piece scale, fail, and be upgraded on its own schedule.

Architecturally, every control-plane component except etcd is designed to be stateless — they hold no durable data of their own, only in-memory caches populated by watching the API Server. That's what makes the API Server horizontally scalable (any number of replicas behind a load balancer) while the Scheduler and Controller Manager use leader election to guarantee only one active instance makes decisions at a time, even though multiple replicas run for HA.

In managed offerings (EKS, GKE, AKS) the cloud provider runs and hides all of this for you — you never SSH into a control-plane node. In self-managed clusters bootstrapped with kubeadm, these components run as static pods: kubelet on the control-plane node reads manifests directly from /etc/kubernetes/manifests/ and starts them without needing the API Server to exist yet — solving the chicken-and-egg bootstrapping problem.

Production teams treat the control plane as the single highest-value thing to protect: if it's down, nothing new can be scheduled, no failures self-heal, and no config changes apply — even though already-running Pods keep serving traffic for a while. This asymmetry (data plane survives briefly, control plane must be restored quickly) drives most HA design decisions for Kubernetes.

Internal Working / Interactions

  • API Server ↔ etcd: the only component allowed to read/write etcd directly; every write goes through Raft consensus before being acknowledged.
  • Scheduler ↔ API Server: watches for unscheduled Pods, writes back the chosen nodeName.
  • Controller Manager ↔ API Server: runs dozens of independent controllers (Deployment, Node, Job, ...), each watching one resource type and reconciling it.
  • Cloud Controller Manager ↔ Cloud APIs: provisions ELBs/ALBs for LoadBalancer Services, labels nodes with cloud topology, removes Node objects when the underlying VM is terminated.
  • Leader election: Scheduler and Controller Manager each acquire a Lease object in kube-system; only the lease holder is active, others stand by.

Code Example

# A control-plane component running as a static pod manifest,
# read directly by kubelet from /etc/kubernetes/manifests/kube-apiserver.yaml
apiVersion: v1
kind: Pod
metadata:
  name: kube-apiserver
  namespace: kube-system
  labels:
    component: kube-apiserver
    tier: control-plane
spec:
  containers:
    - name: kube-apiserver
      image: registry.k8s.io/kube-apiserver:v1.29.0
      command:
        - kube-apiserver
        - --etcd-servers=https://127.0.0.1:2379   # only the API Server talks to etcd
        - --service-cluster-ip-range=10.96.0.0/12 # ClusterIP allocation range
        - --client-ca-file=/etc/kubernetes/pki/ca.crt
        - --enable-admission-plugins=NodeRestriction,ResourceQuota
      livenessProbe:
        httpGet:
          path: /livez
          port: 6443
          scheme: HTTPS
      resources:
        requests:
          cpu: 250m       # control-plane components should have guaranteed CPU
  hostNetwork: true         # binds directly to the node's network namespace
  priorityClassName: system-cluster-critical

kubectl Commands

# List control-plane static pods (kubeadm clusters)
kubectl get pods -n kube-system -l tier=control-plane

# Check which replica currently holds each leader-election lease
kubectl get lease -n kube-system

# View raw control-plane component health
kubectl get --raw='/healthz?verbose'

# Describe a specific control-plane pod to see restarts/events
kubectl describe pod kube-scheduler-controlplane-1 -n kube-system

# Check API server version skew vs kubelet
kubectl version --short

# Watch control-plane pod status live
kubectl get pods -n kube-system -w

Best Practices & Common Pitfalls

Best Practices

  • Run 3 (or 5) control-plane nodes for etcd quorum and API Server HA.
  • Put a load balancer or VIP in front of multiple API Server replicas.
  • Reserve dedicated CPU/memory for control-plane static pods (system-cluster-critical priority class).
  • Prefer a managed control plane (EKS/GKE/AKS) unless you have a dedicated platform team.

Pitfalls

  • Running only 1 or 2 control-plane nodes — no real quorum benefit over 1, and worse resilience than 3.
  • Co-locating untrusted workloads on control-plane nodes (security + noisy-neighbor risk).
  • Forgetting the Scheduler/Controller Manager only ever have ONE active leader — scaling replicas doesn't parallelize their work.
  • Ignoring leader-election flapping, which silently halts reconciliation cluster-wide.

50 Interview Questions

1. What components make up the Kubernetes control plane?
Short answer: API Server, etcd, Scheduler, Controller Manager, and (on cloud) Cloud Controller Manager.
Detailed explanation: API Server is the front door and only component talking to etcd; etcd is the durable store; Scheduler assigns Pods to nodes; Controller Manager runs the reconciliation loops; CCM bridges cloud-specific APIs (LoadBalancers, node lifecycle) when running on a cloud provider.
Production example: In a kubeadm cluster these run as static pods under /etc/kubernetes/manifests/, bootstrapped by kubelet before the API Server even exists.
Common mistake: Thinking of "the control plane" as a single process rather than several independently replaceable components.
2. Why are control-plane components (except etcd) designed to be stateless?
Short answer: Statelessness lets the API Server scale horizontally behind a load balancer with no coordination needed between replicas.
Detailed explanation: Each stateless component only holds in-memory caches populated from watching the API Server; durable truth lives solely in etcd. Any replica can be killed and replaced without data loss.
Production example: Running 3 API Server replicas behind an ELB/VIP — any one can be terminated and traffic just reroutes.
Common mistake: Assuming Scheduler/Controller Manager replicas also share load like the API Server does — they don't, only one is ever active (leader election).
3. How does leader election work for the Scheduler and Controller Manager?
Short answer: Each acquires a Lease object in kube-system; only the lease holder is active, others stand by.
Detailed explanation: Standby replicas continuously try to acquire the lease; if the active holder fails to renew before expiry, a standby wins and becomes active, avoiding two instances making conflicting decisions.
Production example: kubectl get lease -n kube-system shows the current holder identity and renewal time for kube-scheduler/kube-controller-manager.
Common mistake: Expecting multiple Scheduler/Controller Manager replicas to parallelize work — they never do; more replicas only improve failover speed.
4. What is a static pod and why does the control plane depend on it?
Short answer: A Pod started directly by kubelet from a local manifest file, with no API Server involved.
Detailed explanation: kubeadm uses static pods to solve the chicken-and-egg bootstrap problem: kubelet needs to start kube-apiserver itself before any API Server exists to schedule it normally, by reading manifests from /etc/kubernetes/manifests/.
Production example: kubectl get pods -n kube-system -l tier=control-plane lists these mirrored (read-only-from-kubectl) static pods.
Common mistake: Trying to kubectl delete a static pod expecting it to stay gone — kubelet just recreates it from the file.
5. Why does only the API Server talk to etcd directly?
Short answer: Centralizing etcd access gives one enforcement point for authn/authz/admission and schema validation.
Detailed explanation: If Scheduler, Controller Manager, or kubelets wrote to etcd directly, there'd be no consistent place to check RBAC or run webhooks — every write path would need to reimplement security independently.
Production example: A ValidatingWebhookConfiguration blocking Pods without resource limits only works because every write is forced through the API Server.
Common mistake: Editing etcd directly for a "quick fix" — this bypasses all admission control and can corrupt cluster state.
6. What is the Cloud Controller Manager and why was it split from Controller Manager?
Short answer: CCM runs cloud-provider-specific logic (LoadBalancer provisioning, node lifecycle, topology labels) separately from core, cloud-agnostic controllers.
Detailed explanation: Splitting it out lets cloud vendors version and ship their integration independently of core Kubernetes releases, keeping the core Controller Manager free of vendor-specific code.
Production example: AWS's CCM provisions an ELB the moment you create a type: LoadBalancer Service.
Common mistake: Assuming core Kubernetes controllers talk to cloud APIs directly — only CCM ever does.
7. Why run 3 or 5 control-plane nodes instead of 2 or 4?
Short answer: etcd's Raft consensus needs a majority quorum, and even numbers add no fault tolerance over the odd number below them.
Detailed explanation: A 3-node etcd cluster tolerates 1 failure (needs 2/3 for quorum); a 4-node cluster still only tolerates 1 failure (needs 3/4) while adding replication overhead — strictly worse than 3.
Production example: Production kubeadm HA topologies standardize on 3 control-plane nodes across 3 AZs.
Common mistake: Adding a 4th control-plane node "for extra safety" without realizing it doesn't raise fault tolerance.
8. What happens to running workloads if the entire control plane goes down temporarily?
Short answer: Already-running Pods keep serving traffic; nothing new can be scheduled, reconciled, or self-healed.
Detailed explanation: kubelet already knows what Pods it should run and continues running/restarting them locally; but Scheduler can't place new Pods, and controllers can't reconcile drift, so this asymmetry means the data plane briefly survives while the control plane must be restored quickly.
Production example: A brief EKS control-plane maintenance window doesn't crash running checkout-service Pods, but a node failure during that window wouldn't reschedule them.
Common mistake: Believing a control-plane outage means total cluster downtime — it means the cluster stops changing, not stops running.
9. Why should you reserve dedicated CPU for control-plane static pods?
Short answer: CPU starvation on a control-plane node can cause leader-election lease loss, halting reconciliation cluster-wide.
Detailed explanation: Static pods run with priorityClassName: system-cluster-critical and guaranteed CPU requests specifically so noisy neighbors can't starve the Scheduler/Controller Manager into losing their lease.
Production example: A runaway logging sidecar on a control-plane node causing scheduler leader-election flapping and stalled Pod scheduling cluster-wide.
Common mistake: Co-locating untrusted or resource-hungry workloads on control-plane nodes to save cost.
10–50. Additional rapid-fire questions
10. What's the difference between a managed and self-managed control plane? — Managed (EKS/GKE/AKS) hides all control-plane operations from you; self-managed (kubeadm) requires you to run and upgrade every component yourself.
11. Why is the API Server the only horizontally-scaled control-plane component in the traditional sense? — It's the only one without leader election; all replicas are simultaneously active because it's stateless.
12. What object type backs leader election? — A Lease object (coordination.k8s.io/v1) in the kube-system namespace.
13. What is the risk of running only 1 control-plane node in production? — Single point of failure — losing that node loses etcd, API Server, scheduling, and reconciliation simultaneously.
14. How do you check control-plane component health without SSH? — kubectl get --raw='/healthz?verbose' and kubectl get pods -n kube-system -l tier=control-plane.
15. What is hostNetwork: true used for in the API Server's static pod spec? — Binds the API Server directly to the node's network namespace/IP rather than a Pod-private IP, needed since it must be reachable before cluster networking (CNI) is even up.
16. Why does the API Server need --client-ca-file configured? — To validate client certificates presented during authentication (mutual TLS).
17. What does --service-cluster-ip-range control? — The CIDR block the API Server allocates ClusterIPs from for Services.
18. Why is control-plane version skew policy important? — kubelet, Scheduler, and Controller Manager must stay within supported minor-version distance of the API Server or undefined behavior/incompatibility can occur.
19. What's a symptom of leader-election flapping? — Repeated "leader election lost"/"became leader" log entries and stalled reconciliation despite the process itself being up.
20. Why put a load balancer/VIP in front of multiple API Server replicas? — So clients (kubectl, kubelets, controllers) have one stable endpoint regardless of which replica handles the request.
21. What's the operational risk of co-locating workloads on control-plane nodes? — Security exposure (workload compromise reaching control-plane secrets) and noisy-neighbor resource contention affecting etcd/API Server latency.
22. How does kubeadm bootstrap the very first control-plane node? — kubelet starts as a systemd service, reads static pod manifests for etcd/API Server/Scheduler/Controller Manager, and they come up without any pre-existing API Server.
23. What is the priority class typically assigned to control-plane static pods? — system-cluster-critical, which protects them from eviction under node resource pressure.
24. Why does the Controller Manager bundle dozens of controllers into one binary? — Pragmatic choice: one leader-election lease, one deployable artifact, shared informer caching — at the cost of shared fate if the process crashes.
25. What's the first thing to check when Pods are stuck Pending cluster-wide? — Whether the Scheduler pod is running and currently holds its leader-election lease.
26. How do you upgrade a self-managed HA control plane safely? — Upgrade one control-plane node at a time (API Server, then Controller Manager/Scheduler, then kubelet), maintaining etcd quorum throughout.
27. What's the relationship between the API Server and RBAC in the control plane? — The API Server evaluates every request's Role/ClusterRoleBindings during its authorization stage before admission/persistence.
28. Why is the control plane described as "the brain" vs worker nodes as "the muscle"? — The control plane decides what should run where; worker nodes execute that decision — a clean separation of decision-making from execution.
29. What happens if you scale the Scheduler Deployment to multiple replicas in a kubeadm cluster? — All replicas run, but leader election ensures only one is ever actively scheduling — no throughput gain.
30. Why must etcd be reachable from every API Server replica? — Every mutating and consistent-read request needs to reach etcd; if etcd is unreachable, that API Server replica can't serve authoritative reads/writes.
31. What's a common cause of "context deadline exceeded" errors from the API Server? — Slow etcd responses (often disk fsync latency) cascading into API Server request timeouts.
32. How does the control plane isolate a compromised worker node? — Cordon/drain removes it from scheduling consideration; it never gains access to control-plane-only resources like etcd directly.
33. What is the purpose of the --enable-admission-plugins flag on the API Server? — Enables specific admission controllers (e.g. NodeRestriction, ResourceQuota) as part of the control-plane's policy enforcement.
34. Why do managed control planes (EKS) charge a flat hourly fee regardless of node count? — The control plane's operating cost (API Server, etcd, Scheduler, Controller Manager) doesn't scale directly with worker node count up to a point.
35. What's the difference between Node Controller and Scheduler responsibilities? — Node Controller monitors node health/heartbeats and evicts Pods off unhealthy nodes; Scheduler only decides initial placement of new Pods.
36. Why is etcd excluded from the "stateless" description of the control plane? — It's the one component that must durably persist data across restarts — everything else rebuilds its view from etcd via the API Server.
37. What's the risk of a webhook with failurePolicy: Fail and no HA? — A single webhook outage blocks all matching API Server requests cluster-wide, effectively halting part of the control plane's write path.
38. How would you detect that the Controller Manager lost its lease? — Check kubectl get lease -n kube-system kube-controller-manager for holder identity/renewal time, and controller logs for "leader election lost".
39. Why does a self-managed control plane require more platform-team investment than a managed one? — You own etcd backup/restore, cert rotation, version-skew management, and HA topology design yourself.
40. What's the impact of running the control plane in a single AZ? — A single AZ outage takes down etcd quorum and the entire API Server, unlike a 3-AZ spread which tolerates one AZ loss.
41. Why does the Scheduler need to watch both Pods and Nodes? — It needs Pods with empty nodeName as input and current Node capacity/conditions to filter and score candidate placements.
42. What's a control-plane-specific metric worth alerting on? — API Server request latency and etcd fsync/commit latency — both are leading indicators of control-plane degradation.
43. Why is "one active leader" important for the Controller Manager specifically? — Two active instances reconciling the same Deployment/ReplicaSet could issue conflicting scale-up/scale-down actions simultaneously.
44. What's the significance of the system:masters group in control-plane security? — It's a built-in RBAC group with implicit cluster-admin access, typically restricted to bootstrap/emergency use only.
45. How does the control plane's design support zero-downtime upgrades? — Rolling replacement of stateless API Server replicas behind a load balancer while etcd quorum and leader-elected components fail over independently.
46. What's a symptom that indicates etcd (not the API Server itself) is the bottleneck? — API Server logs showing slow etcd request latency, and etcdctl endpoint status showing high fsync/commit duration.
47. Why does kubeadm store control-plane manifests in a specific well-known directory? — kubelet is configured to watch /etc/kubernetes/manifests/ specifically for static pods — it's a hard-coded bootstrapping convention.
48. What's the difference between control-plane HA and control-plane scalability? — HA is about surviving component failure (leader election, etcd quorum); scalability is about serving more load (API Server horizontal scaling, etcd write throughput limits).
49. Why shouldn't you treat control-plane nodes like regular worker nodes for capacity planning? — They need guaranteed, isolated resources for etcd/API Server latency-sensitive operations, not bin-packed alongside arbitrary workloads.
50. What's the very first diagnostic step when the whole cluster seems "frozen" (no reconciliation happening)? — Check Controller Manager and Scheduler leader-election leases/logs — a stuck leader election halts nearly all cluster-wide reconciliation.
