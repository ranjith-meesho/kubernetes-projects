Topic 23 of 70

Storage

The big picture: how volumes, PV, PVC, StorageClass, and CSI fit together to give Pods durable, portable storage.
← Ingress Course Index PersistentVolume →

Explanation

Containers are ephemeral by design — a Pod's writable layer disappears the moment the container restarts. Kubernetes' volume abstraction exists to decouple storage lifetime from container lifetime. The simplest volumes are tied to the Pod: emptyDir is a scratch directory that lives as long as the Pod does (useful for caches or sidecar-to-main-container file sharing, but wiped on Pod deletion), and hostPath mounts a path from the node's own filesystem (useful for node-level agents, dangerous for anything else because it ties a Pod to one specific node and gives it access to the host).

Real applications — databases, message queues, anything stateful — need storage that outlives the Pod and can follow it if it's rescheduled elsewhere. That's what the PersistentVolume (PV) / PersistentVolumeClaim (PVC) pair provides: a PV represents an actual piece of storage (an EBS volume, a GCE PD, an NFS share, a Ceph RBD image), and a PVC is a namespaced request for storage that gets bound to a matching PV. Pods never reference storage directly — they reference a PVC, which is the layer of indirection that lets storage be provisioned, resized, or swapped without touching Pod specs.

StorageClass removes the last manual step: instead of a cluster admin pre-creating PVs, a StorageClass names a provisioner (a cloud disk API, a CSI driver) and parameters (disk type, filesystem, encryption), and PVs are created on demand the moment a PVC asks for one. This is "dynamic provisioning," and it's the default in virtually every real cluster today — static PV creation is now mostly reserved for pre-existing storage you need to import.

Underneath all of this sits the Container Storage Interface (CSI) — a standard gRPC contract between Kubernetes and any storage backend. Before CSI, every storage vendor's integration had to live in Kubernetes' own binary tree ("in-tree" volume plugins), which meant a driver bug required a Kubernetes release to fix. CSI moved all of that logic out-of-tree into independently versioned driver Pods, which is why almost every cloud and storage vendor now ships a CSI driver instead of relying on legacy in-tree plugins.

Together the stack reads top-down: Pod mounts a PVC, the PVC is bound to a PV, the PV was created (statically or dynamically via a StorageClass) by a CSI driver, and the CSI driver talks to the real storage system (EBS, Ceph, Portworx, etc.). Each of the next four topics dives into one layer of this stack.

Internal Working / Interactions

  • kubelet — mounts the final volume into the Pod's container filesystem at Pod startup, calling out to CSI node plugins for CSI-backed volumes.
  • PV controller (kube-controller-manager) — binds PVCs to PVs and enforces reclaim policy on release.
  • External provisioner (CSI sidecar) — watches unbound PVCs referencing a StorageClass and calls the CSI driver to create backing storage.
  • Cloud/storage backend — the actual disk/share/API that the CSI driver provisions and attaches.

Code Example

# emptyDir — ephemeral, Pod-lifetime-only scratch space (no PV/PVC involved)
apiVersion: v1
kind: Pod
metadata:
  name: cache-demo
spec:
  containers:
  - name: app
    image: nginx:1.27
    volumeMounts:
    - name: scratch
      mountPath: /cache             # writable scratch dir shared across containers in this Pod
  volumes:
  - name: scratch
    emptyDir:
      sizeLimit: 1Gi                # optional cap to stop it filling the node disk

---
# Pod consuming durable storage via a PVC (dynamic provisioning end-to-end)
apiVersion: v1
kind: Pod
metadata:
  name: db-demo
spec:
  containers:
  - name: postgres
    image: postgres:16
    volumeMounts:
    - name: data
      mountPath: /var/lib/postgresql/data   # durable data directory
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: db-data-pvc                # indirection: Pod never touches PV/StorageClass directly

kubectl Commands

kubectl get pv                         # list all PersistentVolumes cluster-wide
kubectl get pvc -A                     # list all PersistentVolumeClaims
kubectl get storageclass               # list available StorageClasses / provisioners
kubectl get csidrivers                 # list installed CSI drivers
kubectl describe pod db-demo           # check volume mount events, mount failures
kubectl get volumeattachments           # see which PVs are attached to which nodes
kubectl exec db-demo -- df -h /var/lib/postgresql/data  # verify the volume is actually mounted

Best Practices & Common Pitfalls

Do
  • Prefer dynamic provisioning (StorageClass) over hand-created PVs in every environment.
  • Use emptyDir only for genuinely disposable data.
  • Set sizeLimit on emptyDir to protect node disk pressure.
  • Treat CSI driver version/compatibility as a first-class upgrade dependency.
Pitfalls
  • Using hostPath for anything beyond node-local agents — it breaks Pod portability and is a security risk.
  • Assuming storage "just resizes" — many backends need explicit PVC resize support enabled.
  • Forgetting that deleting a PVC can delete the underlying disk depending on reclaim policy.
  • Mixing zones: a PV provisioned in one AZ can't be mounted by a Pod scheduled in another.

50 Interview Questions

1. Why does Kubernetes have both ephemeral volumes (emptyDir, hostPath) and persistent storage (PV/PVC) as separate concepts?
Short answer: They solve different problems — ephemeral volumes exist for the lifetime of a Pod for scratch/shared data, while PV/PVC exist for data that must outlive and follow the workload across reschedules.
Detailed explanation: Not every volume need is "durable data." Caches, sidecar hand-off files, and temp scratch space are naturally throwaway and shouldn't carry the weight (or cost) of provisioning real disks. Persistent storage exists specifically for the subset of workloads — databases, queues, anything stateful — where losing data on Pod deletion is unacceptable.
Production example: A build-cache sidecar pattern uses emptyDir to share compiled artifacts between an init container and the main container, while the Postgres Pod next to it uses a PVC so a rescheduled Pod reattaches to the same data.
Common mistake: Reaching for a PVC by default even when the data is genuinely disposable, adding unnecessary provisioning overhead and cost.
2. What is the key architectural reason Pods reference a PVC instead of a PV directly?
Short answer: The PVC is a layer of indirection that decouples the Pod spec from the actual storage implementation, so storage can be provisioned, resized, or swapped without touching the Pod.
Detailed explanation: A PVC is just a namespaced request for storage with size/access-mode requirements; it gets bound to whatever PV satisfies it, whether that PV was hand-created or dynamically provisioned. Because the Pod only knows the PVC's name, the underlying disk type or backend can change without any Pod spec edit.
Production example: Migrating a stateful workload from one cloud disk type to another can be done by rebinding the PVC to a new PV, without editing every Deployment/StatefulSet that mounts it.
Common mistake: Hardcoding storage-backend-specific volume fields directly in Pod specs instead of going through a PVC, which defeats the portability the abstraction provides.
3. How does StorageClass turn PV creation from a manual admin task into "dynamic provisioning"?
Short answer: A StorageClass names a provisioner and parameters; when a PVC references it, a PV is created on demand instead of an admin pre-creating one.
Detailed explanation: Without StorageClass, a cluster admin must manually create PVs ahead of time and hope their size/type matches future PVC requests. With StorageClass, the external provisioner watches for unbound PVCs and calls the CSI driver to create matching storage automatically, the moment it's requested.
Production example: A new namespace's PVC referencing an ssd-encrypted StorageClass gets a brand-new encrypted EBS volume provisioned within seconds, with no admin involvement.
Common mistake: Continuing to manually pre-create PVs "just in case" in clusters that already have dynamic provisioning configured, adding operational toil for no benefit.
4. What problem did CSI (Container Storage Interface) solve that in-tree volume plugins did not?
Short answer: CSI moved storage-vendor integration code out of Kubernetes' own binary tree into independently versioned driver Pods, so driver fixes no longer require a Kubernetes release.
Detailed explanation: Before CSI, every storage backend's logic lived inside kubelet/kube-controller-manager itself ("in-tree"). A bug in one vendor's plugin could only be fixed by shipping a new Kubernetes version. CSI defines a standard gRPC contract so any vendor can ship, version, and patch their driver independently of the Kubernetes release cycle.
Production example: An EBS CSI driver security patch can be rolled out by upgrading the driver DaemonSet/Deployment alone, with zero Kubernetes control-plane upgrade required.
Common mistake: Assuming an old in-tree plugin reference (e.g. kubernetes.io/aws-ebs) still applies on modern clusters where it's deprecated in favor of the CSI equivalent.
5. Walk through the full top-down flow from Pod to backend storage.
Short answer: Pod mounts a PVC, the PVC is bound to a PV, the PV is created (statically or via a StorageClass) by a CSI driver, and the CSI driver talks to the real storage system.
Detailed explanation: Each layer only knows about the one below it: the Pod knows a PVC name, the PVC knows a StorageClass name (or is bound directly to a pre-existing PV), the PV knows a CSI driver identity and volume handle, and only the CSI driver knows the storage backend's actual API (EBS, Ceph, Portworx, etc.).
Production example: Tracing a mount failure means walking this chain in order: check Pod events, then PVC status, then PV status, then CSI driver Pod logs, then the cloud backend's own console/API.
Common mistake: Jumping straight to cloud-provider logs when troubleshooting, skipping the intermediate PVC/PV binding status that often reveals the real failure point.
6. What's the practical difference between emptyDir and hostPath, and why is hostPath considered risky?
Short answer: emptyDir is a Pod-scoped scratch directory on whatever node the Pod lands on; hostPath mounts an actual path from the node's filesystem, tying the Pod to that specific node and exposing host-level access.
Detailed explanation: emptyDir's contents are wiped when the Pod is deleted and never persist across reschedules — it's genuinely ephemeral. hostPath instead reads/writes real files on the node, meaning a rescheduled Pod on a different node sees a completely different (or missing) directory, and a compromised container can potentially reach sensitive host files.
Production example: A legitimate hostPath use is a node-level log-shipping DaemonSet reading /var/log on each node; using hostPath for a stateless web app's cache would break the moment the Pod is rescheduled to another node.
Common mistake: Using hostPath as a shortcut for "persistent storage" instead of a PVC, which silently breaks Pod portability and creates a security exposure.
7. Why does the internal-working diagram show the kubelet calling out to CSI node plugins during mount, rather than mounting the volume itself?
Short answer: kubelet doesn't know how to talk to every storage backend, so for CSI-backed volumes it delegates the actual mount operations (NodeStageVolume/NodePublishVolume) to the CSI driver's node plugin.
Detailed explanation: kubelet's job is to orchestrate Pod lifecycle generically; the storage-specific logic of attaching a disk, formatting/staging it, and bind-mounting it into the container's filesystem is delegated via the standard CSI gRPC calls, keeping kubelet decoupled from any specific backend.
Production example: A stuck Pod in ContainerCreating due to a volume mount issue is usually diagnosed by checking the CSI node plugin Pod's logs on that node, not kubelet logs alone.
Common mistake: Debugging volume mount failures purely from kubectl describe pod events without also checking the CSI driver's own logs, missing the actual root cause.
8. What is the role of the PV controller inside kube-controller-manager, and why does reclaim policy matter?
Short answer: The PV controller binds PVCs to matching PVs and, when a PVC is released, enforces the PV's reclaim policy (Retain, Delete, or Recycle) on what happens to the underlying storage.
Detailed explanation: Binding is a matching process based on size, access modes, and StorageClass; once bound, the PV controller also watches for PVC deletion and acts according to reclaim policy — either keeping the disk around (Retain), deleting it entirely (Delete, common with dynamic provisioning), or the now-deprecated Recycle.
Production example: A PVC accidentally deleted for a dynamically-provisioned PV with reclaim policy Delete results in the underlying cloud disk being destroyed immediately — no soft-delete window.
Common mistake: Assuming deleting a PVC is a safe, reversible operation without first checking the bound PV's reclaim policy.
9. Why can't a Pod in one availability zone mount a PV that was provisioned in another zone?
Short answer: Most cloud block storage (EBS, GCE PD, etc.) is zone-local, so a PV backed by a disk in zone A physically cannot be attached to a node running in zone B.
Detailed explanation: Dynamic provisioning typically happens without zone awareness unless the StorageClass or scheduler constraints account for topology; if the Pod later gets scheduled to a node in a different zone than the disk, the volume attach step fails outright.
Production example: A StatefulSet Pod rescheduled after a node drain lands in a different zone than its PV and gets stuck in ContainerCreating with a volume attach error until manually rescheduled back to a compatible zone.
Common mistake: Not using topology-aware provisioning (volumeBindingMode: WaitForFirstConsumer) or zone-aware scheduling, so the disk and Pod end up mismatched.
10. Why is "storage just resizes" a dangerous assumption in production?
Short answer: PVC resize (volume expansion) is an opt-in capability that depends on the StorageClass having allowVolumeExpansion: true and the CSI driver supporting it — it isn't automatic for every backend.
Detailed explanation: Even when supported, expanding a PVC's requested size doesn't always immediately reflect in the filesystem; depending on the driver, a Pod restart or an online filesystem-resize step may be required to actually make the extra space usable inside the container.
Production example: A team edits a PVC's spec.resources.requests.storage expecting more space immediately, but the underlying StorageClass has expansion disabled, so the request is rejected or silently ignored.
Common mistake: Treating volume resize as a universal, always-available operation rather than checking the specific StorageClass/CSI driver's capabilities first.
11–50. Additional rapid-fire questions
11. What does emptyDir.sizeLimit protect against? — Prevents an unbounded scratch volume from filling up the node's disk and causing disk-pressure evictions.
12. Is emptyDir data shared between containers in the same Pod? — Yes, all containers in the Pod can mount and share the same emptyDir volume.
13. What happens to an emptyDir's contents when a container crashes and restarts? — They survive container restarts, but are wiped entirely when the Pod itself is deleted.
14. Is a PersistentVolume a namespaced object? — No, PV is cluster-scoped; PVC is namespaced.
15. What access modes can a PVC request? — ReadWriteOnce, ReadOnlyMany, ReadWriteMany (and ReadWriteOncePod on newer clusters).
16. What binds a PVC to a specific PV? — The PV controller matches based on requested size, access modes, and StorageClass, then sets the binding.
17. Can a PVC remain unbound indefinitely? — Yes, if no PV/provisioner can satisfy its request, it stays in Pending status.
18. What field on a PVC selects which StorageClass provisions it? — spec.storageClassName.
19. What happens if a PVC omits storageClassName entirely? — It uses the cluster's default StorageClass if one is marked default, otherwise it may stay unbound.
20. What two main components make up a typical CSI driver deployment? — A controller plugin (provisioning/attaching) and a node plugin (mounting on each node), often as sidecars alongside the driver container.
21. What is the external-provisioner sidecar's job? — Watches unbound PVCs referencing its StorageClass and calls the CSI driver's CreateVolume RPC.
22. What is the external-attacher sidecar's job? — Calls the CSI driver's ControllerPublishVolume RPC to attach a provisioned volume to the target node.
23. Why are CSI sidecars separate containers rather than built into the driver? — They implement common, standardized Kubernetes-watching logic so each vendor's driver only needs to implement the storage-specific gRPC calls.
24. What does kubectl get csidrivers show? — The CSI drivers currently registered with the cluster.
25. What does kubectl get volumeattachments tell you? — Which PVs (volumes) are currently attached to which nodes.
26. Why might a Pod using a PVC get stuck in ContainerCreating specifically at the mount step? — The CSI node plugin failed to stage/publish the volume, often due to attach failure, zone mismatch, or permissions.
27. What's a fast way to check if a volume is actually mounted inside a running container? — kubectl exec -- df -h .
28. Why is dynamic provisioning considered the default in virtually every real cluster today? — It removes manual PV pre-creation, scales with demand automatically, and matches how cloud storage APIs already work.
29. When would you still use static PV creation instead of dynamic provisioning? — When importing pre-existing storage (e.g. an existing NFS share or disk snapshot) that must be referenced as-is.
30. What's the risk of hostPath giving a container access to the underlying node? — A compromised or misconfigured container could read/write sensitive host files outside the intended sandbox.
31. Why does hostPath tie a Pod to a specific node? — Because it references the node's local filesystem directly, the same data won't exist if the Pod is scheduled elsewhere.
32. What StorageClass parameter controls provisioning timing relative to Pod scheduling? — volumeBindingMode: WaitForFirstConsumer, which delays provisioning until a Pod using the PVC is scheduled.
33. Why does WaitForFirstConsumer help avoid zone-mismatch issues? — It provisions the volume in the same zone the Pod actually gets scheduled to, instead of guessing ahead of time.
34. What's the difference between reclaim policy Retain and Delete? — Retain keeps the underlying disk (and data) after the PVC is released; Delete destroys it automatically.
35. Why is Recycle reclaim policy considered deprecated? — It was a rudimentary generic scrub-and-reuse mechanism, superseded by dynamic provisioning which just creates fresh volumes instead.
36. What does the PV's spec.csi.driver field identify? — Which CSI driver is responsible for operations on that PV.
37. Can a single Pod mix ephemeral and persistent volumes? — Yes, a Pod spec can declare an emptyDir volume and a PVC-backed volume side by side for different mount paths.
38. Why is CSI described as a "gRPC contract" specifically? — Kubernetes and the storage driver communicate over a defined set of gRPC services (Identity, Controller, Node) rather than a Kubernetes-specific in-tree API.
39. What's a sign that a CSI driver version is incompatible with the cluster's Kubernetes version? — Volume operations failing with RPC errors or the driver Pod crash-looping after a Kubernetes upgrade, per the driver's compatibility matrix.
40. Why treat CSI driver upgrades as a first-class dependency in cluster upgrade planning? — Because all persistent storage operations depend on it; an incompatible driver can break provisioning/mounting cluster-wide.
41. What's a quick command to see all StorageClasses and their default status? — kubectl get storageclass (the default one is annotated in its output).
42. Why might two StorageClasses point to the same CSI provisioner? — To offer different parameters (e.g. disk type, encryption, filesystem) from the same underlying driver.
43. What's a common reason a PVC stays Pending with a dynamic StorageClass configured? — The provisioner/CSI driver Pod is down, misconfigured, or lacks cloud IAM permissions to create the disk.
44. Why is emptyDir a poor choice for anything requiring durability guarantees? — Its entire contents are lost on Pod deletion, with no backing persistent disk at all.
45. How does the PV/PVC model support portability across environments (e.g. dev vs prod)? — The Pod spec stays identical; only the StorageClass/backend referenced by the PVC changes per environment.
46. What's the purpose of kubectl describe pod when troubleshooting storage? — Surfaces volume mount events and scheduling/attach failures directly tied to that Pod.
47. Why does the architecture diagram show StorageClass pointing to PV rather than PVC pointing to PV directly for dynamic cases? — The StorageClass supplies the provisioner and parameters used to create the PV; the PVC then gets bound to that resulting PV.
48. What's a real risk of granting overly broad hostPath access on a shared multi-tenant cluster? — One tenant's Pod could read or tamper with another workload's files or node-level secrets on the same node.
49. Why does CSI separate "Controller" and "Node" plugin responsibilities? — Controller-side operations (create/attach) can run centrally, while Node-side operations (mount/stage) must run on the specific node where the Pod lands.
50. If a stateful workload's data must survive even a full node loss, which storage choice is correct — emptyDir, hostPath, or PVC-backed PV? — A PVC-backed PV on a network-attached backend (e.g. EBS, Ceph), since it exists independently of any single node's lifecycle.
