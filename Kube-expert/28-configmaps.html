ConfigMaps

Externalizing non-secret configuration — literals, files, env vars, volume mounts, and how updates propagate.
← CSI Course Index Secrets →

Explanation

A ConfigMap is a key-value store for non-sensitive configuration, decoupled from the container image. The core problem it solves: without it, changing a config value (a feature flag, a log level, a database hostname) means rebuilding and re-pushing a container image, which is slow and couples "config" to "code" in a way that doesn't match how operations teams actually work — the same image should be deployable to dev/staging/prod with different config, not rebuilt per environment.

ConfigMaps can be created from literal key-value pairs (--from-literal), from whole files (--from-file, useful for a full nginx.conf or application.yaml), or declaratively via YAML with a data map. They're namespaced, so the same ConfigMap name can exist independently in dev and prod namespaces with different values — this is the standard pattern for per-environment configuration with identical manifests otherwise.

There are two fundamentally different ways to consume a ConfigMap, and the choice has real operational consequences. As environment variables (envFrom or individual valueFrom.configMapKeyRef), values are injected once at container start — simple, and what most 12-factor apps expect, but a change to the ConfigMap has zero effect on already-running Pods; you must trigger a rollout to pick up new values. As a volume mount, the ConfigMap's keys appear as files in a directory inside the container, and kubelet actively watches for ConfigMap changes and updates the mounted files in place (usually within about a minute, via the kubelet sync loop) — without restarting the Pod. This makes volume-mounted ConfigMaps the right choice for apps that can hot-reload config from disk (e.g. watching a file for changes), while env vars remain the right choice for values read once at startup.

ConfigMaps are also the natural place to store whole configuration files consumed by sidecars or the main container's binary — Envoy config, Prometheus scrape configs, Fluent Bit pipelines — letting the same generic container image be reused across many different deployments purely by swapping the mounted ConfigMap.

Internal Working / Interactions

  • API Server / etcd — ConfigMaps are stored as plain (unencrypted, unlike Secrets which get optional encryption-at-rest treatment) objects.
  • kubelet — for env-var consumption, reads the ConfigMap once at Pod creation to populate the container's environment; for volume mounts, keeps a watch and syncs updated keys to the mounted files periodically.
  • Deployment/rollout controllers — do NOT automatically restart Pods on ConfigMap change; teams commonly use a checksum annotation on the Pod template (hashed from the ConfigMap contents) to force a rollout when config changes.

Code Example

# ConfigMap holding both simple key-values and a whole config file
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: shop
data:
  LOG_LEVEL: "info"                 # simple key, consumable as an env var
  FEATURE_NEW_CHECKOUT: "true"
  app.properties: |                 # a whole file, consumable via volume mount
    server.port=8080
    cache.ttl=300
    retry.max=3

---
# Pod consuming the ConfigMap both ways
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
  namespace: shop
spec:
  containers:
  - name: app
    image: myco/app:1.4.0
    envFrom:
    - configMapRef:
        name: app-config            # LOG_LEVEL and FEATURE_NEW_CHECKOUT become env vars automatically
    env:
    - name: SPECIFIC_KEY
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL             # pull just one key under a custom env var name
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config          # app.properties appears here as a real file, live-updated
  volumes:
  - name: config-volume
    configMap:
      name: app-config

kubectl Commands

kubectl create configmap app-config --from-literal=LOG_LEVEL=info   # imperative creation
kubectl create configmap nginx-conf --from-file=nginx.conf          # from a whole file
kubectl get configmap app-config -n shop -o yaml                    # inspect full contents
kubectl describe configmap app-config -n shop                       # summary view
kubectl edit configmap app-config -n shop                           # live edit (triggers volume sync, not env)
kubectl rollout restart deployment/app -n shop                      # force Pods to pick up new env values
kubectl exec app-pod -n shop -- cat /etc/config/app.properties      # verify mounted file contents

Best Practices & Common Pitfalls

Do
  • Use a content-hash annotation on the Pod template so ConfigMap changes trigger an automatic rollout.
  • Prefer volume mounts for config an app can hot-reload; env vars for values read once at startup.
  • Keep ConfigMaps small and focused per component rather than one giant cluster-wide config object.
  • Version-control ConfigMap manifests the same as any other Kubernetes object (GitOps).
Pitfalls
  • Editing a ConfigMap and assuming env-var-consuming Pods immediately see the change — they won't until restarted.
  • Storing secrets in a ConfigMap "temporarily" — it has no encryption and is visible to anyone with read access.
  • Exceeding the ~1MiB etcd object size limit by stuffing huge files into a single ConfigMap.
  • Relying on the exact kubelet sync interval for volume-mounted updates — it's eventually consistent, not instant.

50 Interview Questions

1. What problem does a ConfigMap actually solve that hardcoding values in the image doesn't?
Short answer: It decouples configuration from the container image so the same built artifact can run in dev, staging, and prod with different values.
Detailed explanation: Without ConfigMaps, changing a log level or a feature flag means rebuilding and re-pushing the image — slow, and it wrongly couples "config" with "code." A ConfigMap lets the image stay identical while an external key-value object supplies environment-specific values.
Production example: The same myco/app:1.4.0 image is deployed to dev and prod namespaces, each with its own app-config ConfigMap setting different LOG_LEVEL or feature-flag values.
Common mistake: Baking environment-specific values into the image and maintaining separate image tags per environment instead of externalizing config.
2. What are the three ways to create a ConfigMap, and when would you pick each?
Short answer: --from-literal for individual key-value pairs, --from-file for whole files, and a declarative YAML data map for anything version-controlled.
Detailed explanation: --from-literal is quick for one-off imperative key=value pairs. --from-file is used when you need to hand a whole config file (like nginx.conf) to a container as-is. Declarative YAML with a data map is the production-grade approach since it's diffable and lives in Git.
Production example: A ConfigMap holding both LOG_LEVEL: "info" as a literal-style key and a full app.properties file under one key, defined declaratively in a checked-in manifest.
Common mistake: Relying on imperative kubectl create configmap --from-literal commands for production config with no YAML source of truth in Git.
3. Why can the same ConfigMap name exist in both dev and prod namespaces with different values?
Short answer: ConfigMaps are namespaced objects, so identical names in different namespaces are entirely independent resources.
Detailed explanation: This is the standard pattern for per-environment configuration: the Deployment manifest referencing app-config stays identical across environments, while the actual ConfigMap object with that name differs per namespace.
Production example: app-config in the dev namespace sets FEATURE_NEW_CHECKOUT: "false" while the same-named ConfigMap in prod sets it to "true", with no changes needed to the Pod spec.
Common mistake: Assuming a ConfigMap is cluster-scoped and can be referenced across namespaces without duplicating it into each one.
4. What's the fundamental difference between consuming a ConfigMap as env vars versus a volume mount?
Short answer: Env vars are injected once at container start and never update; volume-mounted files are actively kept in sync by kubelet as the ConfigMap changes.
Detailed explanation: With envFrom or configMapKeyRef, kubelet reads the ConfigMap once when the container starts. With a volume mount, kubelet maintains a watch and periodically syncs updated keys to the mounted files in place, without touching the running process.
Production example: A Pod using envFrom for LOG_LEVEL keeps logging at the old level even after kubectl edit configmap, until the Pod restarts; a Pod mounting app.properties as a volume sees the file update within roughly a minute.
Common mistake: Editing a ConfigMap and expecting env-var-based Pods to reflect the change immediately.
5. Why don't Deployment controllers automatically restart Pods when a referenced ConfigMap changes?
Short answer: The Deployment controller only watches its own Pod template's spec; a ConfigMap reference by name doesn't change when the ConfigMap's contents change, so no diff is detected.
Detailed explanation: Kubernetes has no built-in "config changed, roll the Deployment" mechanism. The Pod template still says configMapRef: app-config before and after the edit — identical spec, so the controller sees nothing to reconcile.
Production example: An engineer edits app-config expecting a rollout, but kubectl get pods shows the same Pod ages as before — nothing happened until a manual kubectl rollout restart.
Common mistake: Assuming Kubernetes "just knows" to redeploy when config changes, without any explicit rollout trigger in place.
6. How does the content-hash annotation pattern force a rollout on ConfigMap change?
Short answer: An annotation on the Pod template is set to a hash of the ConfigMap's contents, so any content change alters the Pod template spec and triggers a real rollout.
Detailed explanation: Tools like Helm or Kustomize (or a custom CI step) compute a checksum of the ConfigMap data and inject it as e.g. metadata.annotations.configHash on the Deployment's spec.template. Since the Pod template itself now differs, the Deployment controller detects a genuine spec change and performs a normal rolling update.
Production example: A Helm chart templates checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }} as a Pod annotation, so every helm upgrade that changes config also rolls the Pods.
Common mistake: Forgetting to wire up a hash annotation and relying purely on volume-mount syncing for changes that actually require a full app restart (e.g. changed startup-only settings).
7. Why would you deliberately choose env vars over a volume mount even though volume mounts auto-update?
Short answer: Many 12-factor apps only read configuration once at startup, so auto-updating files would have no effect anyway, and env vars are simpler to wire up and inspect.
Detailed explanation: If the application code never re-reads a file after boot, a live-updating volume mount just sits there stale in application memory — the update propagation is wasted. Env vars are also easier to `exec` in and check, and match how most language runtimes expect config to arrive.
Production example: A Java service that reads System.getenv() once during Spring context initialization gains nothing from a volume mount, so it uses envFrom for simplicity.
Common mistake: Wiring config as a volume mount for an app that never watches the filesystem for changes, then being confused why "hot reload" doesn't work.
8. Why are ConfigMaps a good fit for whole configuration files like Envoy or Prometheus configs?
Short answer: Storing a whole file as one ConfigMap key and mounting it as a volume lets a single generic container image be reused across many deployments purely by swapping the mounted config.
Detailed explanation: Instead of baking a Prometheus scrape config or Fluent Bit pipeline into a custom image, the ConfigMap supplies the file and the volume mount makes it appear at the expected path — the sidecar image itself never changes between environments or teams.
Production example: Every team's Fluent Bit sidecar uses the exact same fluent/fluent-bit image, differing only in which fluent-bit.conf ConfigMap is mounted into /fluent-bit/etc/.
Common mistake: Building custom container images per team just to embed a different config file, instead of reusing one image with a mounted ConfigMap.
9. What is the etcd object size limit for a ConfigMap, and why does it matter?
Short answer: Roughly 1MiB per object, because that's etcd's practical limit for a single stored value.
Detailed explanation: Since a ConfigMap is stored as one object in etcd, stuffing large files (bundled binaries, huge JSON blobs, large certs bundles) into it risks hitting this ceiling, causing creation or update failures.
Production example: An attempt to store a multi-megabyte GeoIP database as a ConfigMap key fails outright — it needs to be handled via a volume/init-container download instead.
Common mistake: Treating a ConfigMap as a general file store for arbitrarily large assets rather than small-to-medium text configuration.
10. Why should secrets never be stored in a ConfigMap, even "temporarily"?
Short answer: ConfigMaps are stored as plain, unencrypted objects in etcd and are visible in cleartext to anyone with read access, unlike Secrets which support optional encryption-at-rest.
Detailed explanation: A ConfigMap has no base64-obscuring convention and no encryption-at-rest treatment path the way Secrets do; `kubectl get configmap -o yaml` shows every value in plain text immediately.
Production example: A database password accidentally placed in a ConfigMap instead of a Secret is visible to any user or service account with `get configmaps` RBAC permission in that namespace — far broader exposure than intended.
Common mistake: Using a ConfigMap for a database URL that embeds credentials, rather than splitting the credential into a Secret and only the non-sensitive host/port into the ConfigMap.
11–50. Additional rapid-fire questions
11. What flag creates a ConfigMap from an entire directory of files at once? — kubectl create configmap <name> --from-file=<dir>, adding one key per file in the directory.
12. Can a ConfigMap key contain a full multi-line file's contents? — Yes, using the YAML block scalar (|) syntax under data, as with app.properties.
13. What command shows the full contents of a ConfigMap? — kubectl get configmap app-config -n shop -o yaml.
14. What's the difference between envFrom and valueFrom.configMapKeyRef? — envFrom imports every key in the ConfigMap as an env var automatically; configMapKeyRef pulls just one specific key, optionally under a different env var name.
15. How often does kubelet sync updated ConfigMap volume-mount contents to the container filesystem? — Roughly every ~60 seconds via its periodic sync loop, not instantly.
16. Does editing a ConfigMap trigger an immediate update for env-var consumers? — No — env vars are set once at container start and never refresh without a restart.
17. What command forces Pods to pick up new env-var values from a changed ConfigMap? — kubectl rollout restart deployment/app -n shop.
18. Are ConfigMaps namespaced or cluster-scoped? — Namespaced — a Pod can only reference a ConfigMap in its own namespace.
19. What happens if a Pod references a ConfigMap key that doesn't exist? — The Pod fails to start (env var case) or the volume mount omits that file, depending on how it's referenced and whether it's marked optional.
20. Can you mark a ConfigMap reference as optional so the Pod still starts if it's missing? — Yes, via optional: true on the configMapRef/configMapKeyRef.
21. What tool commonly auto-generates the content-hash annotation for a Pod template? — Helm (via a checksum template function) or Kustomize's ConfigMap generator with hash suffixes.
22. How does Kustomize's ConfigMap generator force a rollout differently from a manual hash annotation? — It appends a content hash to the generated ConfigMap's name itself, so the Pod template's reference literally changes, triggering a natural rollout.
23. Why might mounting an entire ConfigMap as a volume hide previously existing files in that directory? — By default, a ConfigMap volume mount replaces the entire target directory's contents with only the ConfigMap's keys, unless subPath is used.
24. What does using subPath in a ConfigMap volume mount let you avoid? — It mounts a single file from the ConfigMap without wiping out the rest of the target directory's existing contents.
25. Why does subPath disable the live-update behavior of ConfigMap volume mounts? — kubelet's sync mechanism relies on updating a symlinked directory, which subPath mounts bypass, so subPath files don't auto-update.
26. How would you verify a mounted ConfigMap file actually updated in a running container? — kubectl exec <pod> -- cat /etc/config/app.properties and compare against the current ConfigMap contents.
27. What's a fast way to see a summary of a ConfigMap's keys without dumping full YAML? — kubectl describe configmap app-config -n shop.
28. Why should you version-control ConfigMap manifests the same as application code? — So config changes are reviewed, diffable, and auditable via GitOps rather than made ad hoc with kubectl edit.
29. What risk does kubectl edit configmap in production carry compared to applying from Git? — It bypasses code review and leaves the cluster's live state diverged from the source-controlled manifest.
30. Why is it good practice to keep ConfigMaps small and focused per component? — Smaller, scoped ConfigMaps reduce blast radius and merge conflicts compared to one giant cluster-wide config object.
31. Can multiple Pods/Deployments reference the same ConfigMap simultaneously? — Yes, any number of Pods in the same namespace can mount or reference the same ConfigMap.
32. What happens to Pods already running if you delete a ConfigMap they reference via volume mount? — The mounted files typically remain as last-synced (stale) until kubelet's next failed sync attempt or Pod restart, though behavior can vary by kubelet version.
33. Why is a ConfigMap unsuitable for storing a TLS private key? — It has no encryption and is stored as plain text — a Secret with restricted RBAC is the appropriate object for sensitive material.
34. What's the benefit of using --from-env-file when creating a ConfigMap? — It bulk-imports an entire `.env`-style file of KEY=value lines into ConfigMap data in one command.
35. Why might an application misbehave right after a ConfigMap update even with a volume mount in place? — If the app caches config in memory at startup and never re-reads the file, the on-disk update has no runtime effect without a restart.
36. What's a common pitfall when referencing a ConfigMap key with a name containing dots, like app.properties? — Environment variable names generally can't contain dots, so such keys are only usable via volume mount, not as env vars.
37. How can a mounted ConfigMap volume be made read-only, and why would you want that? — Volume mounts from ConfigMaps are read-only by default, preventing the container from modifying what it perceives as its own config files.
38. Why is it easy to accidentally shadow one ConfigMap key with another when merging multiple ConfigMaps via envFrom? — If two ConfigMaps referenced in the same Pod share a key name, the later one in the list silently overrides the earlier value with no error.
39. How would you diagnose a Pod stuck in CreateContainerConfigError related to a ConfigMap? — Check `kubectl describe pod` events — it usually indicates a referenced ConfigMap or key doesn't exist in that namespace.
40. What's the advantage of declarative YAML ConfigMaps over imperative `kubectl create configmap` commands in CI/CD? — They're reproducible, diffable, and reapplyable via `kubectl apply`, matching GitOps workflows instead of one-off shell commands.
41. Why does a typo in a ConfigMap's key name often fail silently rather than erroring loudly? — For optional references or volume mounts, a missing key can simply be absent from the environment or directory rather than throwing a visible error.
42. What's the relationship between a ConfigMap and an `initContainer` in some config patterns? — An initContainer can read a mounted ConfigMap and transform/render it into a final config file before the main container starts, for cases needing pre-processing.
43. Why would you avoid embedding cluster-specific hostnames directly inside a versioned ConfigMap manifest? — It breaks the "same manifest across environments" pattern; better to template it (Helm/Kustomize) so only values change per environment.
44. What's a good way to validate a ConfigMap's YAML `data` map before applying it in CI? — Run `kubectl apply --dry-run=client -f configmap.yaml` or a schema/lint check as part of the pipeline.
45. Why might two teams sharing a namespace clash over ConfigMap names? — ConfigMaps are namespace-scoped, so identical names in a shared namespace collide and one team can unintentionally overwrite another's config.
46. What's the operational benefit of separating ConfigMaps per microservice rather than one shared "global-config"? — Limits blast radius — an unrelated service's config change can't accidentally roll or break a different service's Pods.
47. How does immutable ConfigMap support (the immutable: true field) help at scale? — It signals to the API server the ConfigMap's data won't change, letting kubelet skip watching it for updates and reducing API server load in large clusters.
48. Why would an immutable ConfigMap be combined with the Kustomize hash-suffix pattern? — Since immutable ConfigMaps can't be edited in place, a new hash-suffixed ConfigMap is created per change and the Deployment is updated to reference the new name, still triggering a proper rollout.
49. What's a quick way to confirm which Pods are actually using a given ConfigMap before deleting it? — Search Deployment/Pod specs (e.g. via `kubectl get pods -o yaml` or a tool like `kubectl-who-uses`) for references to that ConfigMap's name.
50. Why is it best practice to keep environment-specific values in the ConfigMap but keep the Deployment/Pod spec identical across environments? — It maximizes manifest reuse and reduces drift — the only thing that should differ between dev and prod is the data inside the environment's own ConfigMap, not the workload spec itself.
