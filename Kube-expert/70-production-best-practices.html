Production Best Practices

The capstone checklist: resource management, security, observability, high availability, and cost, synthesized across the whole course.
← CKA Exam Preparation Course Index (last topic)

Explanation

Everything in this course has been building toward one question: what does it actually take to run Kubernetes in production, for real users, for years, without it becoming a source of constant incidents? The honest answer is that no single feature makes a cluster "production ready" — it's the compounding effect of getting five categories right simultaneously: resource management, security, observability, high availability, and cost control. Neglecting any one of them eventually produces an incident, no matter how well the other four are handled — a beautifully secured cluster with no resource limits still gets taken down by a noisy-neighbor Pod; a highly available cluster with no observability still pages someone at 3 AM to debug blind.

Resource management means every workload declares realistic requests and limits (not copy-pasted defaults), namespaces carry ResourceQuotas and LimitRanges so one team can't starve another, and the Cluster Autoscaler / HPA / VPA are tuned against real traffic patterns rather than left at framework defaults. The first-principles reason this matters: the scheduler makes placement decisions purely off requests, so wrong requests mean wrong bin-packing, which means either wasted capacity or, worse, real resource contention discovered only under production load.

Security is the sum of many small disciplines this course covered individually — RBAC scoped to least privilege per team/service-account, NetworkPolicies enforcing default-deny between namespaces, Pod Security Standards/admission enforcing non-root and no privilege escalation, Secrets encrypted at rest and never embedded in images, image scanning and signed/verified images in the supply chain, and Workload Identity (IRSA/Workload Identity/Azure AD Workload Identity) instead of static cloud credentials anywhere in the cluster. None of these individually stops every attack; together they mean a single mistake doesn't cascade into a full compromise.

Observability is what turns "the cluster is unhealthy" into "here is exactly what's wrong and where" — metrics (Prometheus/cloud-native equivalents) for the golden signals (latency, traffic, errors, saturation) per service, centralized logging that survives Pod restarts, distributed tracing for latency root-causing across service boundaries, and alerting tied to symptoms users actually feel (error rate, latency) rather than only to raw infrastructure metrics nobody acts on at 3 AM. The test of good observability is whether an on-call engineer with no prior context on the incident can find the root cause from dashboards alone within minutes.

High availability — odd-numbered etcd quorums, multi-AZ control planes and node pools, PodDisruptionBudgets, topology spread, tested backup/restore, and a rehearsed disaster recovery runbook with real RTO/RPO numbers — is what keeps the previous three categories from being undone by a single AZ failure or a bad upgrade. And cost control — right-sized requests, Spot/preemptible capacity for fault-tolerant workloads, cluster autoscaling tuned to scale down as aggressively as it scales up, and visibility into per-team/per- namespace spend — is what keeps the other four categories affordable enough that leadership keeps investing in them rather than cutting corners under budget pressure. Production readiness is not a single gate you pass once; it's an ongoing discipline of chaos-testing your assumptions (kill a node, fail a zone, corrupt a backup on purpose) before production does it for you on its own schedule.

Internal Working / Interactions

  • Requests/limits → scheduler: the scheduler's bin-packing and the Vertical/Horizontal Pod Autoscalers all key off declared requests — get these wrong and every downstream automation misbehaves.
  • RBAC + NetworkPolicy + Pod Security → blast radius: each layer independently limits what a compromised Pod or credential can reach; they compose rather than substitute for each other.
  • Metrics/logs/traces → alerting → on-call: golden-signal alerting should page on user-facing symptoms, with dashboards and traces providing the drill-down path to root cause.
  • HA + backup/DR → incident containment: HA absorbs single-component failures automatically; backup/DR is the fallback when HA itself is insufficient (data corruption, full-region loss).
  • Cost visibility → sustained investment: without per-team cost attribution, the other four pillars quietly erode over time as "we can't afford it" becomes the path of least resistance.

Code Example

# Capstone checklist as a single annotated Namespace manifest
# (illustrating several pillars applied together on one workload)
#############################################
apiVersion: v1
kind: Namespace
metadata:
  name: checkout
  labels:
    pod-security.kubernetes.io/enforce: restricted    # Security: enforce Pod Security Standard
---
apiVersion: v1
kind: ResourceQuota
metadata:
  name: checkout-quota
  namespace: checkout
spec:
  hard:
    requests.cpu: "20"          # Resource mgmt: cap total requested CPU for this team's namespace
    requests.memory: 40Gi
    limits.cpu: "40"
    limits.memory: 80Gi
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny
  namespace: checkout
spec:
  podSelector: {}
  policyTypes: ["Ingress", "Egress"]   # Security: default-deny, explicit allow-lists added per service
---
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: checkout-pdb
  namespace: checkout
spec:
  minAvailable: 80%              # HA: survive voluntary disruption (drains/upgrades) without an outage
  selector:
    matchLabels: { app: checkout }
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-api
  namespace: checkout
spec:
  replicas: 6
  selector: { matchLabels: { app: checkout } }
  template:
    metadata:
      labels: { app: checkout }
      annotations:
        prometheus.io/scrape: "true"     # Observability: scraped for golden-signal metrics
    spec:
      topologySpreadConstraints:          # HA: spread across zones
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector: { matchLabels: { app: checkout } }
      containers:
        - name: checkout
          image: registry.example.com/checkout:v9.2.0
          resources:                       # Resource mgmt: explicit, load-tested requests/limits
            requests: { cpu: "250m", memory: "256Mi" }
            limits: { cpu: "500m", memory: "512Mi" }
          securityContext:                 # Security: least privilege at container level
            runAsNonRoot: true
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true

kubectl / CLI Commands

# Audit a namespace's resource requests/limits vs actual usage
kubectl top pods -n checkout
kubectl describe resourcequota checkout-quota -n checkout

# Confirm Pod Security Standard enforcement is active
kubectl get ns checkout -o jsonpath='{.metadata.labels}'

# Check NetworkPolicies actually applied to a namespace
kubectl get networkpolicy -n checkout

# Verify PodDisruptionBudget status before any planned disruption
kubectl get pdb -n checkout

# Confirm topology spread is actually achieved across zones
kubectl get pods -n checkout -o wide -L topology.kubernetes.io/zone

# Spot-check cost/capacity: idle vs requested vs limit across the cluster
kubectl describe nodes | grep -A5 "Allocated resources"

# Run a full readiness self-audit combining several signals at once
kubectl get all,pdb,networkpolicy,resourcequota -n checkout

Best Practices & Common Pitfalls

Best Practices

  • Treat resource requests as load-tested numbers, not copy-pasted defaults.
  • Layer RBAC, NetworkPolicy, Pod Security, and Workload Identity together — never rely on one alone.
  • Alert on user-facing symptoms first; use traces/logs for drill-down, not primary paging.
  • Rehearse HA and DR failure scenarios on a schedule, not just after an incident.
  • Give every team visibility into their own namespace's cost and resource footprint.

Common Pitfalls

  • Treating any single pillar (e.g. "we have HA") as sufficient for production readiness alone.
  • Alerting only on infrastructure metrics nobody ties back to real user impact.
  • Skipping chaos/game-day testing until a real incident forces the first real-world test.
  • No cost attribution, letting budget pressure quietly erode security/observability investment.
  • Treating this course's topics as one-time setup rather than an ongoing operational discipline.

50 Interview Questions

1. Why does this topic insist that no single feature makes a cluster "production ready"?
Short answer: Production readiness is the compounding effect of getting five categories right simultaneously — resource management, security, observability, high availability, and cost control — neglecting any one eventually produces an incident.
Detail: The categories are independent failure modes: a beautifully secured cluster with no resource limits still gets taken down by a noisy-neighbor Pod, and a highly available cluster with no observability still pages someone at 3 AM to debug blind.
Production example: A team with excellent RBAC and NetworkPolicy but no ResourceQuotas suffers an outage when one team's misconfigured Job consumes all available node capacity.
Common mistake: Treating any single pillar (e.g. "we have HA") as sufficient for production readiness alone.
2. Why does the scheduler's correctness fundamentally depend on accurate resource requests?
Short answer: The scheduler makes placement decisions purely off declared requests, so wrong requests mean wrong bin-packing — either wasted capacity or real resource contention discovered only under production load.
Detail: This is a first-principles dependency, not a best-practice suggestion: HPA/VPA and the Cluster Autoscaler all key off the same declared requests, so a single bad number cascades into every downstream automation misbehaving.
Production example: A Deployment copy-pasted with generic `requests: { cpu: "250m" }` regardless of actual load causes either wasted node capacity (over-requested) or noisy-neighbor CPU contention (under-requested).
Common mistake: Copy-pasting default resource requests/limits across services rather than setting load-tested, workload-specific numbers.
3. Why do RBAC, NetworkPolicy, and Pod Security Standards "compose rather than substitute" for each other?
Short answer: Each layer independently limits what a compromised Pod or credential can reach — RBAC scopes API access, NetworkPolicy scopes network reachability, Pod Security Standards scope container-level capabilities — so a single mistake in one layer doesn't cascade into a full compromise.
Detail: None of these individually stops every attack; their value comes specifically from stacking, so that a gap in one layer (e.g. an overly broad RBAC role) is still contained by another (e.g. default-deny NetworkPolicy).
Production example: A namespace with `pod-security.kubernetes.io/enforce: restricted`, a default-deny NetworkPolicy, and scoped RBAC means a compromised Pod still can't escalate privileges, reach unrelated services, or access the API beyond its bound role.
Common mistake: Relying on RBAC alone (or any single layer alone) as "our security," treating the other layers as optional rather than complementary.
4. Why should alerting be tied to symptoms users actually feel rather than raw infrastructure metrics?
Short answer: Golden-signal alerting (error rate, latency) directly reflects real user impact, whereas raw infrastructure metrics nobody acts on at 3 AM generate noise without corresponding action.
Detail: The test of good observability is whether an on-call engineer with no prior incident context can find the root cause from dashboards alone within minutes — that requires alerts anchored to what's actually broken for users, with infra metrics/traces/logs as the drill-down path, not the primary trigger.
Production example: Paging on `error rate > 5%` (a symptom) rather than on `node disk IO latency` alone (an infra metric that may or may not be user-impacting).
Common mistake: Alerting only on infrastructure metrics nobody ties back to real user impact, generating alert fatigue with low actionability.
5. What's the relationship between high availability and backup/DR, and why are they not interchangeable?
Short answer: HA absorbs single-component failures automatically (a node dies, a zone fails); backup/DR is the fallback when HA itself is insufficient — data corruption, full-region loss, or logical errors that HA cannot protect against.
Detail: HA mechanisms (odd-numbered etcd quorums, multi-AZ control planes, PDBs, topology spread) assume the underlying data is intact and simply need to survive infrastructure failure; they do nothing against a bad deploy that corrupts data or deletes resources.
Production example: A multi-AZ HA cluster survives a zone outage automatically, but only a tested backup/restore process recovers from an accidental `kubectl delete` of critical resources or corrupted etcd state.
Common mistake: Assuming HA alone covers disaster recovery, skipping tested backup/restore because "we're already highly available."
6. Why is cost control framed as what "funds continued investment" in the other four pillars?
Short answer: Without per-team/per-namespace cost visibility and attribution, budget pressure quietly erodes security/observability/HA investment over time as "we can't afford it" becomes the path of least resistance.
Detail: This reframes cost control from a purely financial concern into a sustainability mechanism for the other pillars — teams facing unattributed, opaque cluster costs tend to cut corners on the categories that don't have an obvious immediate cost (until an incident makes the cost visible the hard way).
Production example: A team with clear per-namespace cost attribution can justify continued investment in dedicated observability tooling; a team with no cost visibility gets that tooling budget cut first during a cost-cutting round.
Common mistake: Treating cost control as disconnected from security/observability/HA, rather than as the mechanism that keeps leadership funding them.
7. Why does the capstone Namespace example enforce Pod Security Standards at the namespace label level rather than per-Pod?
Short answer: A namespace-level `pod-security.kubernetes.io/enforce: restricted` label applies admission enforcement to every Pod created in that namespace automatically, rather than relying on each team to correctly configure per-Pod securityContext fields individually.
Detail: This is a defense-in-depth technique: even if an individual Deployment's securityContext is missing or misconfigured, the namespace-level enforcement still blocks non-compliant Pods from being admitted at all.
Production example: A Deployment that forgets `runAsNonRoot: true` in its own manifest still gets rejected by admission control if the namespace enforces the `restricted` Pod Security Standard.
Common mistake: Relying solely on per-container securityContext settings without namespace-level Pod Security Standard enforcement as a backstop.
8. Why does the capstone example use topologySpreadConstraints alongside a PodDisruptionBudget rather than either alone?
Short answer: topologySpreadConstraints proactively spreads replicas across zones to reduce correlated-failure risk; the PodDisruptionBudget protects against voluntary disruption (drains/upgrades) once replicas are already spread — they address different failure timings.
Detail: Spread alone doesn't prevent an upgrade from draining too many replicas simultaneously; a PDB alone doesn't prevent all replicas from initially landing in the same zone — together they cover both the initial placement and any later disruption event.
Production example: `maxSkew: 1` with `topologyKey: topology.kubernetes.io/zone` combined with `minAvailable: 80%` on the PDB ensures replicas start zone-spread and stay above a safe availability floor during planned node drains.
Common mistake: Setting a PodDisruptionBudget without topology spread, so all replicas end up concentrated in one zone despite the PDB "protecting" them.
9. Why does the topic recommend chaos-testing assumptions (killing a node, failing a zone, corrupting a backup on purpose) rather than trusting configuration alone?
Short answer: Production readiness is an ongoing discipline, not a one-time gate — untested HA/backup/DR configuration is an assumption until it's actually exercised, and production will test it on its own schedule if you don't do it deliberately first.
Detail: A PodDisruptionBudget, a backup job, or a multi-AZ topology can all be misconfigured in ways that only surface during an actual failure — deliberately rehearsing failure scenarios (game days) surfaces those gaps under controlled, non-incident conditions.
Production example: A team that has never tested restoring from its etcd backup discovers, only during a real incident, that the backup has been silently failing for months.
Common mistake: Skipping chaos/game-day testing until a real incident forces the first real-world test of HA/backup/DR assumptions.
10. How does this capstone topic's five-pillar framework synthesize the rest of the course?
Short answer: Resource management, security, observability, HA, and cost control each map directly to specific earlier topics (scheduling/HPA/VPA, RBAC/NetworkPolicy/Pod Security/Workload Identity, Prometheus/logging/tracing, etcd/multi-AZ/PDBs/backup, autoscaling/Spot) — this topic is explicitly the integration point, not new material.
Detail: The value of the capstone is showing these previously-separate topics interacting on a single manifest/namespace simultaneously, since real production readiness requires applying them together, not as isolated checklist items studied independently.
Production example: The capstone's single annotated Namespace example combines a Pod Security Standard label, a ResourceQuota, a default-deny NetworkPolicy, a PodDisruptionBudget, and topology spread — five different course topics on one page.
Common mistake: Treating this course's topics as one-time setup rather than an ongoing operational discipline that must be revisited as workloads and scale change.
