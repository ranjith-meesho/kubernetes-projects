Container Storage Interface (CSI)

The standard contract between Kubernetes and storage backends — controller/node plugins and the gRPC calls kubelet makes.
← StorageClass Course Index ConfigMaps →

Explanation

Before CSI, every storage integration (AWS EBS, GCE PD, Azure Disk, Ceph, etc.) had to be compiled directly into the core Kubernetes binaries as "in-tree" volume plugins. That coupling was painful: a bug fix or new feature in a storage driver required waiting for the next Kubernetes release, vendors couldn't ship or version their own code independently, and the core codebase accumulated dozens of storage-specific branches it didn't really own. The Container Storage Interface (CSI) is the industry-standard fix: a vendor-neutral gRPC API that any container orchestrator (not just Kubernetes) can speak, and any storage vendor can implement once and reuse everywhere.

A CSI driver is just a set of Pods you deploy like any other workload, split into two roles. The Controller plugin runs as a Deployment (usually 1-2 replicas) and handles cluster-level operations that don't need to happen on a specific node: CreateVolume, DeleteVolume, ControllerPublishVolume (attach a disk to a node), snapshotting, and resizing. The Node plugin runs as a DaemonSet (one Pod per node) and handles node-local operations: NodeStageVolume (format and mount the disk to a staging path) and NodePublishVolume (bind-mount it into the specific Pod's directory).

Kubernetes talks to these plugins not directly, but through a set of well-known "sidecar" containers that translate Kubernetes API objects into CSI gRPC calls: external-provisioner (watches PVCs, calls CreateVolume), external-attacher (watches VolumeAttachment objects, calls ControllerPublishVolume), external-resizer (handles PVC resize requests), external-snapshotter (VolumeSnapshot support), and node-driver-registrar (registers the node plugin with kubelet via a Unix domain socket). This sidecar pattern means the actual vendor code only needs to implement the CSI gRPC interface — all the Kubernetes-specific watch/reconcile logic is reusable boilerplate shared across every driver.

kubelet's role is to call the Node plugin directly over a Unix domain socket whenever a Pod using a CSI volume starts or stops on that node — it never talks to the cloud API itself. This is why CSI driver Pods must be healthy on every node that hosts stateful workloads: if the node plugin crashes, Pods needing volume mounts on that node will fail to start even if the cloud disk itself is fine.

Internal Working / Interactions

  • External provisioner / attacher / resizer sidecars — translate PVC/VolumeAttachment/resize events from the API Server into CSI controller RPCs.
  • CSI Controller plugin — talks to the cloud/storage backend API to create, delete, attach, detach, resize, snapshot volumes.
  • node-driver-registrar — tells kubelet where the node plugin's Unix socket lives so kubelet can call it.
  • kubelet — calls NodeStageVolume/NodePublishVolume on the CSI Node plugin over gRPC when a Pod starts; calls the unstage/unpublish equivalents on Pod termination.
  • CSI Node plugin — performs the actual mount/format/bind-mount on the node's filesystem.

Code Example

# CSIDriver object — declares driver capabilities to the cluster (usually installed by the driver's Helm chart)
apiVersion: storage.k8s.io/v1
kind: CSIDriver
metadata:
  name: ebs.csi.aws.com
spec:
  attachRequired: true          # this driver needs ControllerPublishVolume (attach) before mounting
  podInfoOnMount: true          # pass Pod metadata to NodePublishVolume calls
  volumeLifecycleModes:
    - Persistent                # standard PVC-backed lifecycle (vs Ephemeral inline volumes)

---
# DaemonSet snippet for the CSI Node plugin (illustrative — real charts are longer)
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: ebs-csi-node
  namespace: kube-system
spec:
  selector:
    matchLabels: { app: ebs-csi-node }
  template:
    metadata:
      labels: { app: ebs-csi-node }
    spec:
      containers:
      - name: node-driver-registrar         # registers this node plugin's socket with kubelet
        image: registry.k8s.io/sig-storage/csi-node-driver-registrar:v2.10.0
        args:
        - --csi-address=/csi/csi.sock
        - --kubelet-registration-path=/var/lib/kubelet/plugins/ebs.csi.aws.com/csi.sock
      - name: ebs-plugin                     # the actual vendor driver binary
        image: public.ecr.aws/ebs-csi-driver/aws-ebs-csi-driver:v1.32.0
        args: ["node"]
        securityContext:
          privileged: true                   # needed to mount/format block devices on the host

Best Practices & Common Pitfalls

Do
  • Monitor CSI node plugin DaemonSet health as closely as kubelet itself — a crashed node plugin blocks all volume mounts on that node.
  • Keep the CSI driver version compatible with your Kubernetes minor version per the vendor's support matrix.
  • Use vendor-provided Helm charts/operators to install CSI drivers rather than hand-rolling manifests.
  • Enable volume snapshot support (external-snapshotter) if you need backup/restore workflows.
Pitfalls
  • Draining/rebooting a node while a CSI attach/detach operation is mid-flight, leaving a stuck VolumeAttachment.
  • Forgetting the node plugin typically needs privileged: true — restrictive PodSecurity policies can silently block it.
  • Running an old in-tree volume plugin config alongside a CSI migration without checking migration flags.
  • Ignoring controller plugin logs when PVCs hang in Pending — the root cause is usually a cloud API throttling or permission error, not Kubernetes itself.

50 Interview Questions

1. Why was the Container Storage Interface (CSI) introduced when in-tree volume plugins already worked?
Short answer: In-tree plugins forced every storage vendor's code to live inside core Kubernetes, coupling their release cycle to Kubernetes releases.
Detailed explanation: A bug fix or feature in an AWS EBS or Ceph driver had to wait for the next Kubernetes minor release, and the core codebase accumulated storage-specific code it didn't own. CSI replaces this with a vendor-neutral gRPC contract any orchestrator can speak and any vendor can implement and ship independently.
Production example: A storage vendor can release a CSI driver patch and roll it out via a Helm chart upgrade the same day, without waiting on a Kubernetes release train.
Common mistake: Assuming CSI is Kubernetes-specific — it's an industry-standard interface any container orchestrator can adopt.
2. What is the split of responsibilities between the CSI Controller plugin and the Node plugin?
Short answer: The Controller plugin handles cluster-level operations like CreateVolume/DeleteVolume/attach; the Node plugin handles node-local mount operations.
Detailed explanation: The Controller plugin runs as a Deployment (1-2 replicas) since its operations don't need to run on a specific node — it talks to the cloud/storage backend API. The Node plugin runs as a DaemonSet, one Pod per node, because formatting and mounting a device only makes sense on the node where the Pod will actually run.
Production example: ControllerPublishVolume (attaching a disk to a node) runs from the Controller Deployment, while NodeStageVolume (mounting that attached disk) runs from the Node Pod on that exact node.
Common mistake: Assuming a single CSI Pod handles the whole lifecycle — the split is deliberate and both must be healthy for volumes to work end to end.
3. What role do the CSI sidecar containers (external-provisioner, external-attacher, etc.) play?
Short answer: They translate Kubernetes API objects (PVCs, VolumeAttachments) into CSI gRPC calls, so vendor code only needs to implement the CSI interface itself.
Detailed explanation: external-provisioner watches PVCs and calls CreateVolume, external-attacher watches VolumeAttachment objects and calls ControllerPublishVolume, external-resizer handles PVC resize requests, and external-snapshotter handles VolumeSnapshot support. This reusable boilerplate is shared across every vendor's driver.
Production example: A new storage vendor only writes the CreateVolume/DeleteVolume gRPC handlers; the sidecars they bundle from sig-storage handle all Kubernetes watch/reconcile logic.
Common mistake: Thinking the vendor plugin container alone implements Kubernetes integration — without the correct sidecars, PVCs never trigger the vendor's gRPC calls at all.
4. How does node-driver-registrar fit into the CSI Node plugin DaemonSet?
Short answer: It tells kubelet where the Node plugin's Unix domain socket lives so kubelet can call it directly.
Detailed explanation: Each Node plugin exposes a gRPC endpoint over a Unix socket; node-driver-registrar registers that socket path with kubelet's plugin registration mechanism so kubelet knows exactly which socket to dial for that driver on that node.
Production example: The DaemonSet's --kubelet-registration-path arg points to /var/lib/kubelet/plugins/ebs.csi.aws.com/csi.sock, which kubelet then uses for every subsequent NodeStageVolume/NodePublishVolume call.
Common mistake: Forgetting node-driver-registrar crashing means kubelet loses its registration for that driver on that node, even if the vendor plugin container is still running.
5. Why does kubelet talk to the CSI Node plugin over a Unix domain socket instead of calling the cloud API directly?
Short answer: kubelet is storage-agnostic by design under CSI — it only knows how to make generic gRPC calls to whatever Node plugin is registered, never the cloud provider's own API.
Detailed explanation: This indirection is exactly what makes CSI vendor-neutral: kubelet's logic doesn't change per storage backend, it just calls NodeStageVolume/NodePublishVolume on the local socket and the Node plugin translates that into whatever the backend actually requires.
Production example: Swapping from an EBS-backed StorageClass to a Ceph-backed one requires zero kubelet changes — only a different Node plugin DaemonSet is running that socket.
Common mistake: Assuming kubelet has cloud-provider credentials or logic baked in for volume mounting under CSI — it does not, all backend-specific work lives in the plugin.
6. Why does a crashed CSI Node plugin block Pod scheduling with volumes on that node, even if the cloud disk itself is healthy?
Short answer: kubelet has no way to mount the volume without a live Node plugin process to call over the Unix socket, regardless of the underlying disk's state.
Detailed explanation: NodeStageVolume and NodePublishVolume are gRPC calls to that specific local Pod; if it's crashed or its socket is gone, kubelet's calls fail or hang, so any Pod scheduled onto that node needing the volume gets stuck in ContainerCreating.
Production example: A misconfigured resource limit OOM-kills the node plugin Pod on one node; every StatefulSet Pod rescheduled there afterward hangs waiting for its volume to mount.
Common mistake: Troubleshooting a stuck Pod by checking only the cloud provider's volume/disk status and ignoring the local Node plugin DaemonSet's health on that specific node.
7. What is the difference between NodeStageVolume and NodePublishVolume?
Short answer: NodeStageVolume formats and mounts the device to a shared per-volume staging path; NodePublishVolume bind-mounts that staging path into the specific Pod's directory.
Detailed explanation: Staging happens once per volume per node regardless of how many Pods use it (for shareable volumes), while publish happens per Pod, giving each container its own bind-mounted view without re-formatting or re-mounting the underlying device.
Production example: If two Pods on the same node share a ReadWriteMany volume, the driver stages it once, then calls NodePublishVolume twice, once per Pod's target path.
Common mistake: Assuming these are one combined operation — separating them is what lets a driver avoid redundant format/mount work per Pod.
8. What does the CSIDriver object's attachRequired field actually control?
Short answer: It tells Kubernetes whether this driver needs a ControllerPublishVolume (attach) step before the Node plugin can mount the volume.
Detailed explanation: Block-storage-backed drivers like EBS need the disk attached to the node's instance first, so attachRequired: true triggers external-attacher to create a VolumeAttachment object and call ControllerPublishVolume; drivers backed by network filesystems that don't need node-level attachment can set it to false and skip that step.
Production example: An EFS-backed CSI driver typically sets attachRequired: false since there's no block device attach step, unlike EBS.
Common mistake: Assuming every CSI driver goes through an attach phase — it's conditional on whether the backend actually requires node-level attachment.
9. Why does the CSI Node plugin container typically need privileged: true, and what production risk does that create?
Short answer: Formatting and mounting block devices on the host filesystem requires host-level privileges the container runtime doesn't grant by default.
Detailed explanation: Operations like creating a filesystem on a raw block device or performing host mount namespace changes need capabilities beyond a normal container's sandbox, so vendor Node plugins commonly run with privileged: true and host mounts.
Production example: A cluster tightening PodSecurity admission to the "restricted" profile can silently block the CSI Node plugin DaemonSet from being (re)scheduled, breaking all volume mounts cluster-wide.
Common mistake: Rolling out stricter Pod security standards without carving out an exemption for the kube-system namespace hosting CSI Node plugins.
10. What can cause a VolumeAttachment to get "stuck" during a node drain, and why is this a common production incident?
Short answer: If a node is drained or rebooted while an attach/detach gRPC call is mid-flight, the CSI Controller plugin can be left waiting on a Node/Controller operation that never completes cleanly.
Detailed explanation: Detaching a volume requires the Controller plugin to call ControllerUnpublishVolume against the cloud API, but if the node disappears abruptly (forced termination, spot reclaim) before that finishes, the VolumeAttachment object can remain in a stuck state, blocking the Pod from rescheduling elsewhere with the same volume.
Production example: A StatefulSet Pod fails to reschedule on a new node because Kubernetes won't attach the same PVC there until the old VolumeAttachment is cleaned up.
Common mistake: Force-deleting the Pod without investigating the underlying VolumeAttachment object, which can leave the volume unusable on any node until manually resolved.
