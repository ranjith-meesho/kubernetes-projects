PersistentVolumeClaim (PVC)

The namespaced storage request: how binding works, access modes, and how Pods actually consume a PVC.
← PersistentVolume Course Index StorageClass →

Explanation

A PersistentVolumeClaim (PVC) is the developer-facing side of the storage story. Where a PV is a cluster-admin/provisioner concern full of infrastructure detail (disk IDs, mount options, CSI driver names), a PVC is deliberately simple and namespaced: "I need 20Gi, ReadWriteOnce, on the fast-ssd class." Kubernetes needs this split so application authors never have to know or care what's underneath — the same Deployment manifest can run unmodified on AWS, GCP, on-prem, or a local kind cluster, because the PVC abstracts the difference away.

When a PVC is created, the PV controller looks for a matching Available PV (matching capacity, access mode, storage class, and any label selector). If found, it binds them 1:1 — a PV can only ever be bound to one PVC at a time. If no matching PV exists and the PVC references a StorageClass, dynamic provisioning kicks in and a brand-new PV is created and bound automatically, usually within seconds.

Access modes describe how many nodes can mount the volume simultaneously and in what mode: ReadWriteOnce (RWO) — one node at a time (most cloud block storage: EBS, GCE PD, Azure Disk); ReadOnlyMany (ROX) — many nodes, read-only; ReadWriteMany (RWX) — many nodes, read-write (needs a real shared filesystem like NFS, EFS, Ceph FS, Azure Files — not plain block storage); and the newer ReadWriteOncePod, which restricts mounting to a single Pod (not just a single node), useful for StatefulSets where you must guarantee exclusive access even across node failures.

Pods never mount a PV directly — they declare a volume of type persistentVolumeClaim naming the PVC, and kubelet resolves that to the bound PV and its CSI driver at mount time. This indirection is what makes PVCs portable: the same Pod spec works whether the PVC is bound to a static NFS PV or a dynamically-provisioned cloud disk. It's also why deleting and recreating a PVC with the same name doesn't automatically get you the same data back unless you deliberately re-bind it to the same (Retained) PV.

Internal Working / Interactions

  • PV controller — matches the PVC against existing PVs or triggers dynamic provisioning.
  • API Server / etcd — PVC spec (request) vs status (bound volume, capacity) are tracked like any other object.
  • Scheduler — for topology-aware provisioning (WaitForFirstConsumer), the scheduler's chosen node influences which zone the PV gets created in.
  • kubelet — at Pod startup, resolves the PVC's bound PV and calls the CSI node plugin to stage and publish the volume into the container.

Architecture Diagram

flowchart TD
  Dev[Developer writes Pod + PVC manifest] --> PVC[PVC created: 20Gi, RWO, fast-ssd]

  Check -->|No, StorageClass set| Provision[Dynamic provisioning via CSI]
  Provision --> NewPV[New PV created]
  NewPV --> Bind
  Bind --> Ready[PVC status: Bound]
  Ready --> PodMount[Pod references PVC in volumes]
  PodMount --> Kubelet[kubelet mounts via CSI node plugin]

Code Example

# PVC requesting dynamically-provisioned block storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: db-data-pvc
  namespace: shop
spec:
  accessModes:
    - ReadWriteOnce                # one node at a time — correct for a single-writer database
  storageClassName: fast-ssd       # triggers dynamic provisioning via this class's provisioner
  resources:
    requests:
      storage: 20Gi                # requested minimum capacity

---
# Pod consuming the PVC above — no knowledge of the underlying disk/CSI driver
apiVersion: v1
kind: Pod
metadata:
  name: postgres-pod
  namespace: shop
spec:
  containers:
  - name: postgres
    image: postgres:16
    volumeMounts:
    - name: data
      mountPath: /var/lib/postgresql/data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: db-data-pvc       # only reference needed; PV/CSI details are hidden

---
# RWX PVC for a shared filesystem consumed by multiple replicas
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-assets-pvc
  namespace: shop
spec:
  accessModes:
    - ReadWriteMany                # multiple Pods on multiple nodes can mount simultaneously
  storageClassName: efs-sc
  resources:
    requests:
      storage: 50Gi

kubectl Commands

kubectl get pvc -n shop                       # list PVCs and their bound status
kubectl describe pvc db-data-pvc -n shop      # binding events, capacity, access mode
kubectl get pvc -n shop -o wide               # shows bound PV name and storage class inline
kubectl get pv | grep db-data-pvc             # find the PV backing a given PVC
kubectl delete pvc db-data-pvc -n shop        # release the claim (check reclaim policy first!)
kubectl patch pvc db-data-pvc -n shop -p '{"spec":{"resources":{"requests":{"storage":"30Gi"}}}}'  # expand (if allowVolumeExpansion is set)
kubectl get events -n shop --field-selector involvedObject.kind=PersistentVolumeClaim  # provisioning/binding failures

Best Practices & Common Pitfalls

Do
  • Pick the narrowest access mode that works (RWO over RWX) — RWX backends are usually slower and pricier.
  • Set resource requests conservatively but enable volume expansion for growth instead of over-provisioning upfront.
  • Use ReadWriteOncePod for StatefulSet workloads needing strict single-Pod exclusivity.
  • Name PVCs predictably (e.g. tied to StatefulSet ordinal) for easier operational tracing.
Pitfalls
  • PVC stuck Pending — usually a StorageClass typo, zone mismatch, or no provisioner running.
  • Assuming deleting a Pod deletes its PVC — it doesn't; PVCs (and their data) persist independently.
  • Requesting RWX on a StorageClass backed by block storage that physically can't support it.
  • Forgetting allowVolumeExpansion: true is required on the StorageClass before a PVC resize will work.

50 Interview Questions

1. Why is a PVC namespaced while a PV is cluster-scoped?
Short answer: A PVC represents an application's storage request and belongs to whichever namespace that application lives in, while a PV represents an actual piece of infrastructure that isn't owned by any single team.
Detailed explanation: This split lets cluster admins (or dynamic provisioners) manage the pool of underlying disks centrally, while developers only ever interact with the namespaced PVC object — they request capacity and access mode, never touch disk IDs, mount options, or CSI driver details.
Production example: Two teams in shop and billing namespaces can both create a db-data-pvc without collision, because PVC names only need to be unique within a namespace.
Common mistake: Expecting to reference a PVC from a different namespace in a Pod's volumes block — it must live in the same namespace as the Pod.
2. What criteria does the PV controller use to bind a PVC to an existing PV?
Short answer: It matches capacity (PV must have at least the requested amount), access mode, storage class name, and any label selector on the PVC.
Detailed explanation: Only PVs in Available phase are considered. The controller picks the smallest PV that satisfies all criteria rather than an exact match, and binding is exclusive — once bound, that PV is removed from consideration for any other PVC.
Production example: A PVC requesting 20Gi RWO with class fast-ssd will bind to a pre-provisioned 50Gi fast-ssd PV rather than wait for an exact 20Gi match, "wasting" the extra 30Gi that PV had.
Common mistake: Assuming the PVC's requested storage becomes the bound volume's exact size — it's actually whatever capacity the matched PV declares.
3. What happens when a PVC references a StorageClass but no matching Available PV exists?
Short answer: Dynamic provisioning kicks in — the CSI provisioner named in the StorageClass creates a brand-new PV sized to the request and binds it automatically.
Detailed explanation: This happens without any admin pre-creating volumes; the PVC's storageClassName tells the external-provisioner sidecar which backend to call, and it usually completes within seconds for cloud block storage.
Production example: Creating a db-data-pvc with storageClassName: fast-ssd in a fresh namespace results in a new EBS/GCE-PD-backed PV appearing moments later with no admin action.
Common mistake: Forgetting to set storageClassName and expecting dynamic provisioning to still occur — an empty/omitted class only works if a default StorageClass is configured cluster-wide.
4. Why do Pods reference a PVC instead of a PV directly?
Short answer: The indirection lets the same Pod spec work regardless of what's actually backing the storage — a static NFS PV, a dynamically-provisioned cloud disk, or anything else.
Detailed explanation: A Pod's volumes entry only names persistentVolumeClaim.claimName; kubelet resolves that PVC to its bound PV and CSI driver at mount time. This means portability across environments and storage backends without touching the Pod manifest.
Production example: The exact same postgres-pod manifest referencing db-data-pvc runs unmodified on a cloud cluster (EBS-backed PV) or a local kind cluster (hostPath-backed PV).
Common mistake: Hardcoding a PV name directly in a Pod's volume spec, bypassing the PVC layer and losing that portability.
5. Why doesn't recreating a PVC with the same name restore the old data?
Short answer: A new PVC has no memory of the previous binding — the PV controller treats it as a fresh claim and will match/provision independently unless you deliberately re-bind it.
Detailed explanation: If the original PV's reclaim policy was Delete, the underlying disk is gone entirely. If it was Retain, the PV enters Released phase holding the old data, but it won't auto-rebind — an admin must manually clear its claimRef and set it back to Available (or reference it explicitly) before a new PVC can bind to it.
Production example: After accidentally deleting db-data-pvc, a team recreates it expecting their database files back, only to get an empty new volume because the reclaim policy was Delete.
Common mistake: Deleting a PVC to "reset" storage without first checking the reclaim policy on its bound PV.
6. What's the practical difference between ReadWriteOnce (RWO) and ReadWriteOncePod?
Short answer: RWO restricts mounting to a single node (multiple Pods on that same node can still mount it); ReadWriteOncePod restricts mounting to a single Pod, cluster-wide.
Detailed explanation: RWO's node-level restriction has historically caused subtle bugs during StatefulSet Pod rescheduling, where an old Pod's leftover mount on a node and a new Pod scheduled to the same node could both attach. ReadWriteOncePod closes that gap by enforcing true single-Pod exclusivity.
Production example: A StatefulSet Pod that crash-loops and gets rescheduled to the same node could, under plain RWO, end up with two containers briefly both accessing the same volume; ReadWriteOncePod prevents this outright.
Common mistake: Assuming RWO alone guarantees exclusive single-Pod access — it only guarantees single-node access.
7. Why can't most cloud block storage (EBS, GCE PD, Azure Disk) support ReadWriteMany?
Short answer: Block storage volumes are attached to one compute instance at a time at the hypervisor/OS level — they aren't a shared filesystem multiple hosts can mount concurrently.
Detailed explanation: RWX requires a networked filesystem protocol (NFS, EFS, Azure Files, Ceph FS) that natively supports concurrent multi-client mounts with its own locking semantics; plain block devices have no such mechanism.
Production example: Requesting RWX on an ebs-sc StorageClass either fails provisioning outright or the CSI driver rejects the access mode, depending on driver validation.
Common mistake: Assuming any StorageClass can serve any access mode — access modes are constrained by the underlying storage technology, not just a YAML field you can set freely.
8. When would you choose ReadWriteMany over ReadWriteOnce for a workload?
Short answer: When multiple Pods across multiple nodes genuinely need concurrent read-write access to the same files, like shared static assets or a shared upload directory.
Detailed explanation: RWX needs a real shared filesystem backend, which typically costs more and can add latency versus block storage, so it should only be chosen when the access pattern truly requires simultaneous multi-node writers rather than out of convenience.
Production example: A shared-assets-pvc backed by efs-sc lets multiple replicas of a content service all read and write to the same image/CDN-origin directory.
Common mistake: Defaulting to RWX "just in case" for workloads that actually only need RWO, incurring unnecessary cost and complexity.
9. Why would a PVC get stuck in Pending status?
Short answer: Most commonly a StorageClass name typo, no provisioner available for that class, a zone/topology mismatch, or simply no matching Available PV when dynamic provisioning isn't configured.
Detailed explanation: kubectl describe pvc surfaces the specific reason via events — e.g. "no persistent volumes available for this claim" or a provisioning error from the CSI driver. With WaitForFirstConsumer binding mode, Pending is also normal and expected until a Pod referencing the PVC gets scheduled.
Production example: A PVC referencing storageClassName: fast-ssd stays Pending forever because the actual class name in the cluster is fast-ssd-v2 — a simple typo.
Common mistake: Panicking and deleting/recreating the PVC repeatedly instead of first running kubectl describe pvc to read the actual binding/provisioning event.
10. What does allowVolumeExpansion have to do with resizing a PVC, and why can shrinking not work the same way?
Short answer: A PVC can only be resized larger if its StorageClass has allowVolumeExpansion: true; even then, most CSI drivers only support growing a volume, never shrinking it.
Detailed explanation: Editing spec.resources.requests.storage to a larger value on a bound PVC triggers the CSI controller to expand the underlying volume, and then kubelet resizes the filesystem on next mount (or via online expansion if the driver supports it). Shrinking is unsafe because there's no generic way to guarantee data on the "removed" capacity isn't lost, so it's blocked at the API level.
Production example: Patching db-data-pvc from 20Gi to 30Gi succeeds instantly if allowVolumeExpansion is set on fast-ssd; without it, the patch is accepted by the API but never actually resizes anything.
Common mistake: Assuming a PVC resize request that gets accepted by kubectl patch means the expansion actually happened — it silently no-ops if the StorageClass doesn't allow expansion.
11–50. Additional rapid-fire questions
11. What field identifies which StorageClass a PVC should use? — spec.storageClassName.
12. Can a PVC bind to two PVs at once? — No, binding is always strictly one-to-one.
13. What phase is a PVC in before it's bound? — Pending.
14. What command shows which PV a given PVC is bound to? — kubectl get pvc -o wide or kubectl describe pvc.
15. Does deleting a Pod delete the PVC it referenced? — No, PVCs (and their data) persist independently of the Pods that use them.
16. What happens if you delete a PVC that's still mounted by a running Pod? — The PVC enters Terminating and stays there until no Pod is using it (a finalizer blocks actual removal).
17. Why would you set a label selector on a PVC? — To force binding only to PVs carrying matching labels, useful for pinning to specific pre-provisioned storage.
18. What does WaitForFirstConsumer binding mode change about PVC Pending behavior? — Binding (and provisioning) is deliberately delayed until a Pod using the PVC is scheduled, so the volume can be created in the correct topology zone.
19. Can a single Pod mount multiple PVCs? — Yes, a Pod's volumes list can reference any number of distinct PVCs.
20. What's the risk of requesting more storage in a PVC than any available PV provides, with no StorageClass set? — It stays Pending indefinitely since there's no provisioner to create new capacity.
21. How do StatefulSets typically manage PVCs per replica? — Via volumeClaimTemplates, which auto-generate a uniquely-named PVC per ordinal Pod.
22. What happens to a StatefulSet's PVCs when the StatefulSet is scaled down then back up? — The same PVCs (and their PVs/data) are reattached to the recreated ordinal Pods, since scaling down doesn't delete PVCs by default.
23. Why might a database workload specifically want ReadWriteOncePod over RWO? — To guarantee true single-writer exclusivity and avoid any risk of the volume being briefly double-mounted during a rescheduling race.
24. What object shows PVC provisioning failures if events don't reveal enough? — Logs of the external-provisioner sidecar Pod for the relevant CSI driver.
25. Can you change a PVC's access mode after creation? — No, access modes are immutable once the PVC is created; you'd need to create a new PVC.
26. Can you change a PVC's storageClassName after creation? — No, that field is immutable post-creation.
27. What's the effect of specifying no storageClassName and no default StorageClass exists? — The PVC will only bind to a manually pre-created PV matching its criteria; it can never be dynamically provisioned.
28. Why is capacity requested in a PVC treated as a minimum, not exact? — Static binding matches the smallest sufficient PV, which may exceed the requested size, since PVs aren't split or resized to fit exactly.
29. What's a common mistake when migrating a PVC's data to a larger disk without volume expansion support? — Manually provisioning a new larger PVC/PV and copying data across (e.g. via a temporary Pod), since in-place resize isn't possible.
30. How would you verify a resize actually took effect on the filesystem, not just the PVC object? — Check kubectl get pvc for updated status.capacity, then exec into the Pod and run df -h on the mount path.
31. What field in the PVC status reflects the actual bound PV's capacity, versus the request? — status.capacity versus spec.resources.requests.storage.
32. Why is a PVC considered the "contract" between application and storage infrastructure? — It expresses intent (size, access mode, class) without leaking implementation details, letting either side change independently.
33. What's a reason a PVC resize could fail even with allowVolumeExpansion set? — The underlying cloud provider or CSI driver may have its own limits (e.g. minimum time between resizes, or hitting max volume size).
34. Why should teams avoid requesting RWX by default across all PVCs? — RWX-capable backends are typically slower and more expensive than block storage, so it should be reserved for genuinely shared-access workloads.
35. What does it mean if a PVC shows Bound but the Pod using it fails to start with a mount error? — The binding succeeded at the API level, but the actual CSI mount at the node failed — check kubelet/CSI node plugin logs, not the PVC object.
36. Why would a PVC in one availability zone fail to bind to a Pod scheduled in another zone? — Zonal block storage (e.g. EBS) can't be attached across zones; the PV's node affinity restricts which nodes can use it.
37. How does topology-aware provisioning avoid the zone-mismatch problem above? — With WaitForFirstConsumer, provisioning waits until the consuming Pod is scheduled, so the new PV is created in that Pod's actual zone.
38. What's a good reason to name PVCs predictably, tied to a workload's identity? — It makes operational tracing (which PVC backs which database instance) far easier during incident response.
39. Can a PVC be shared between two independent Deployments if it's RWO? — Not safely for concurrent writes — RWO only guarantees single-node mount, and two Deployments' Pods could land on different nodes and conflict or fail to mount.
40. What kubectl command would you use to check for PVC provisioning/binding failures across a namespace? — kubectl get events -n <ns> --field-selector involvedObject.kind=PersistentVolumeClaim.
41. Why is it important to check reclaim policy before deleting a "temporary" PVC in a shared cluster? — A Delete policy destroys the underlying disk irreversibly, while Retain leaves recoverable data behind.
42. What happens to a PVC's bound PV if the PVC is deleted and the policy is Retain? — The PV moves to Released phase, keeping its data, but won't auto-bind to a new PVC until manually reset.
43. Why might over-provisioning storage size upfront be discouraged in favor of expansion? — It wastes capacity and cost immediately, whereas enabling expansion lets you grow storage precisely when actually needed.
44. What's the effect of a Pod referencing a PVC that doesn't exist yet? — The Pod stays stuck in ContainerCreating / Pending until the PVC is created and bound.
45. Why does a PVC's namespace matter for RBAC and multi-tenancy? — It scopes who can create, view, or delete storage requests to the same access boundaries as other namespaced resources like Pods and Secrets.
46. What's a subtle danger of relying on a default StorageClass without explicitly setting storageClassName? — If the cluster default changes later, new PVCs silently start provisioning against a different class than intended.
47. How would you confirm whether a PVC was dynamically provisioned or statically bound? — Check whether its bound PV's name matches the dynamic-provisioner naming pattern (e.g. pvc-<uid>) versus a hand-created PV name.
48. Why can't a PVC's access mode list include RWX for a StorageClass whose provisioner only creates block volumes? — The CSI driver validates and rejects access modes it can't physically support at provisioning time.
49. What's a real operational reason to prefer many small PVCs over one large shared PVC across services? — Isolates failure/resize/access-mode requirements per service instead of coupling unrelated workloads to the same volume's lifecycle.
50. Why is understanding PVC binding critical before debugging a "Pod won't start" issue involving storage? — Most storage-related Pod scheduling/startup failures trace back to the PVC never reaching Bound, so checking PVC status is the correct first diagnostic step, before looking at the Pod itself.
