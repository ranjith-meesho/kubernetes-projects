Grafana

The visualization, dashboards-as-code, and alerting layer that unifies metrics, logs, and traces into one pane of glass.
← VictoriaMetrics Course Index OpenTelemetry →

Explanation

Grafana is an open-source visualization and dashboarding platform that queries one or more backend data sources — Prometheus, VictoriaMetrics, Loki, Tempo, Elasticsearch, CloudWatch, and dozens more — and renders the results as graphs, tables, heatmaps, and alerts, all through a single web UI. Grafana itself stores no metrics or logs; it is a pure query-and-render layer, which is precisely why it composes so well with the rest of the observability stack — you can point the same Grafana instance at Prometheus for metrics, Loki for logs, and Tempo/Jaeger for traces, and pivot between them using consistent labels (e.g. jump from a metric spike straight to the logs for that pod).

Kubernetes needs Grafana because raw PromQL output and `kubectl top` numbers are not how humans reason about system health during an incident. Engineers need trends over time, correlated views (CPU spike next to error-rate spike next to a deployment marker), and pre-built dashboards that don't require re-deriving a query from scratch at 3 AM. Grafana turns the metrics collected by Prometheus/VictoriaMetrics into an operational nervous system: dashboards for steady-state observation, and alerting rules that page humans when thresholds are breached.

A key production practice is dashboards-as-code: rather than clicking through the UI to build a dashboard (which is not version-controlled and easily drifts), teams define dashboards as JSON models (the exact format Grafana's UI itself produces and consumes) and store them in git, provisioned into Grafana automatically via a sidecar or the provisioning API. This makes dashboards reviewable, reproducible across environments, and resistant to "someone edited prod and nobody knows why."

Grafana's alerting engine (unified alerting, Grafana 9+) evaluates queries against any configured datasource on a schedule and routes firing alerts to notification channels (Slack, PagerDuty, webhook) — for teams standardized on Grafana already, this can replace or complement Prometheus's own Alertmanager, giving one alerting UI across multiple metric backends instead of one per backend.

In the ecosystem, Grafana is the topmost, human-facing layer: vmagent/Prometheus scrape and store, Loki/Fluent Bit collect logs, OpenTelemetry/Tempo collect traces — and Grafana is where an engineer actually looks during an incident, correlating all three signal types (the "three pillars of observability") in one interface.

Internal Working / Interactions

  • Data sources — Grafana's server-side proxies each datasource's query API (PromQL for Prometheus/VM, LogQL for Loki, TraceQL for Tempo); the browser never talks to the datasource directly, Grafana's backend does.
  • Dashboard JSON model — every panel, query, and layout is one JSON document; this is what gets version-controlled and provisioned.
  • Provisioning — Grafana watches a config directory (or ConfigMaps mounted via a sidecar like k8s-sidecar) for datasource and dashboard definitions and loads them on startup/change, no manual UI clicking required.
  • Alerting — a background scheduler evaluates alert rule queries against the relevant datasource, tracks state (pending/firing/resolved), and pushes notifications via configured contact points.
  • Explore mode lets you pivot directly from a metrics panel to correlated logs/traces if datasources share exemplar/trace-ID links.

Code Example

# Grafana datasource provisioning — auto-registers VictoriaMetrics + Loki on startup,
# so no manual "Add data source" clicking is needed across environments.
apiVersion: v1
kind: ConfigMap
metadata:
  name: grafana-datasources
  namespace: monitoring
  labels:
    grafana_datasource: "1"          # picked up by the k8s-sidecar watching this label
data:
  datasources.yaml: |
    apiVersion: 1
    datasources:
      - name: VictoriaMetrics
        type: prometheus                       # VM speaks the Prometheus API, so use the prometheus plugin
        access: proxy
        url: http://vmselect.monitoring.svc:8481/select/0/prometheus/
        isDefault: true
        jsonData:
          timeInterval: "30s"
      - name: Loki
        type: loki
        access: proxy
        url: http://loki.monitoring.svc:3100
        jsonData:
          derivedFields:                       # extracts trace IDs from log lines to link Loki -> Tempo
            - datasourceUid: tempo
              matcherRegex: "trace_id=(\\w+)"
              name: TraceID
              url: "$${__value.raw}"
---
# Dashboard provisioning — tells Grafana to load JSON dashboards from a mounted directory,
# which a sidecar keeps in sync with a ConfigMap generated from a git repo.
apiVersion: v1
kind: ConfigMap
metadata:
  name: grafana-dashboard-provider
  namespace: monitoring
data:
  dashboards.yaml: |
    apiVersion: 1
    providers:
      - name: default
        orgId: 1
        folder: "Kubernetes"
        type: file
        options:
          path: /var/lib/grafana/dashboards   # sidecar writes fetched dashboard JSON here

kubectl / CLI Commands

kubectl get pods -n monitoring -l app.kubernetes.io/name=grafana   # check Grafana pod health
kubectl port-forward -n monitoring svc/grafana 3000:80              # access UI locally
kubectl logs -n monitoring deploy/grafana --tail=50                 # check provisioning/datasource errors
kubectl get configmap -n monitoring -l grafana_dashboard=1           # list dashboards being provisioned
kubectl exec -n monitoring deploy/grafana -- cat /etc/grafana/provisioning/datasources/datasources.yaml
kubectl rollout restart deploy/grafana -n monitoring                 # reload provisioning after a ConfigMap change
curl -s http://admin:admin@localhost:3000/api/health                # Grafana health endpoint

Best Practices & Common Pitfalls

Best Practices

  • Store dashboards as JSON in git and provision them — never hand-build production dashboards in the UI.
  • Use variables ($namespace, $pod) in dashboards so one dashboard serves every service/team.
  • Link Loki/Tempo datasources via derived fields so engineers can pivot from a log line straight to a trace.
  • Put Grafana behind SSO (OIDC) rather than local accounts for production access control.
  • Version-pin dashboard JSON schema and test provisioning in a staging Grafana before rolling to prod.

Pitfalls

  • Building dozens of one-off dashboards with copy-pasted queries that drift from the underlying metric names.
  • Alerting directly from Grafana and Alertmanager on the same condition, causing duplicate pages.
  • Not setting dashboard folder permissions, letting anyone edit shared production dashboards.
  • Ignoring query performance — dashboards with many high-cardinality panels can overload the datasource on load.

50 Interview Questions

1. What role does Grafana play in the observability stack, and what does it NOT do?
Short answer: Grafana is a pure visualization/query-and-render layer; it stores no metrics or logs itself.
Detail: It queries backend data sources (Prometheus, VictoriaMetrics, Loki, Tempo, Elasticsearch, CloudWatch, etc.) and renders results as graphs, tables, heatmaps, and alerts through one web UI.
Production example: Pointing one Grafana instance at Prometheus for metrics, Loki for logs, and Tempo/Jaeger for traces simultaneously.
Common mistake: Assuming Grafana persists data — losing the underlying datasource means Grafana has nothing to show, regardless of dashboard configuration.
2. Why can't engineers just rely on raw PromQL output or kubectl top during an incident?
Short answer: Humans need trends over time and correlated views, not raw numbers, especially at 3 AM.
Detail: Grafana turns collected metrics into an operational nervous system — dashboards for steady-state observation, and alerting that pages humans when thresholds are breached.
Production example: A dashboard showing CPU spike next to error-rate spike next to a deployment marker, letting an on-call engineer correlate cause and effect instantly.
Common mistake: Expecting engineers to re-derive PromQL queries from scratch during a live incident instead of having pre-built dashboards.
3. What is "dashboards-as-code" and why is it a key production practice?
Short answer: Defining dashboards as JSON models stored in git rather than building them by clicking through the UI.
Detail: UI-built dashboards aren't version-controlled and drift easily; JSON-as-code makes dashboards reviewable, reproducible across environments, and resistant to undocumented prod edits.
Production example: Dashboards provisioned automatically via a sidecar (like k8s-sidecar) or the provisioning API from a git-backed ConfigMap.
Common mistake: Hand-building production dashboards in the UI, leading to "someone edited prod and nobody knows why."
4. How does Grafana's provisioning mechanism work in Kubernetes?
Short answer: Grafana watches a config directory (or ConfigMaps mounted via a sidecar) for datasource and dashboard definitions and loads them on startup/change.
Detail: No manual UI clicking is required — a sidecar like k8s-sidecar syncs ConfigMaps labeled appropriately (e.g. grafana_datasource: "1") into files Grafana reads.
Production example: A ConfigMap named grafana-datasources with datasources.yaml auto-registering VictoriaMetrics and Loki on startup.
Common mistake: Forgetting the required label on the ConfigMap, so the sidecar never picks up the new datasource/dashboard.
5. How does Grafana's server-side proxy model affect security and datasource access?
Short answer: Grafana's backend proxies each datasource's query API; the browser never talks to the datasource directly.
Detail: This centralizes datasource credentials/network access on the server side rather than exposing them to end-user browsers.
Production example: A VictoriaMetrics datasource configured with access: proxy and a URL only the Grafana server needs network access to.
Common mistake: Assuming a browser-based direct connection is used, and misconfiguring network policies that only need to allow Grafana-server-to-datasource traffic.
6. What is Grafana's unified alerting engine and how does it relate to Alertmanager?
Short answer: Grafana 9+'s built-in alerting evaluates queries against any configured datasource on a schedule and routes firing alerts to notification channels.
Detail: For teams standardized on Grafana, this can replace or complement Prometheus's Alertmanager, giving one alerting UI across multiple metric backends instead of one per backend.
Production example: A background scheduler tracking alert state (pending/firing/resolved) and pushing notifications via configured contact points like Slack or PagerDuty.
Common mistake: Alerting from Grafana AND Alertmanager on the same condition, causing duplicate pages.
7. How does Grafana link logs and traces together via derived fields?
Short answer: Derived fields extract identifiers (like trace IDs) from log lines to link one datasource to another, e.g. Loki to Tempo.
Detail: A regex matcher (e.g. trace_id=(\w+)) pulls the trace ID out of a log line and turns it into a clickable link to the tracing datasource.
Production example: A Loki datasource config with derivedFields pointing a TraceID field at a Tempo datasource UID.
Common mistake: Not configuring derived fields, forcing engineers to manually copy trace IDs between separate UI tabs.
8. What is Grafana's Explore mode used for?
Short answer: It lets you pivot directly from a metrics panel to correlated logs/traces.
Detail: This works when datasources share exemplar/trace-ID links, enabling ad-hoc investigation without building a formal dashboard.
Production example: Jumping from a metric spike in a dashboard panel straight into Explore to view the logs for that pod at the same timeframe.
Common mistake: Building a whole new dashboard for one-off investigation instead of using Explore for ad-hoc correlation.
9. Why should dashboards use template variables like $namespace or $pod instead of hardcoded values?
Short answer: Variables let one dashboard serve every service/team instead of duplicating dashboards per target.
Detail: This avoids "dozens of one-off dashboards with copy-pasted queries" that drift from the underlying metric names over time.
Production example: A single Kubernetes dashboard with a $namespace variable dropdown reused across every team's services.
Common mistake: Copy-pasting a dashboard per service instead of parameterizing it, so metric name changes require updating every copy.
10. What are common pitfalls that degrade Grafana's reliability or security in production?
Short answer: Missing dashboard folder permissions, duplicate alerting, drifted ad-hoc dashboards, and unbounded high-cardinality panel queries.
Detail: Best practices counter these: put Grafana behind SSO/OIDC, version-pin dashboard JSON, and test provisioning in staging first.
Production example: Not setting dashboard folder permissions lets anyone edit shared production dashboards; ignoring query performance lets high-cardinality panels overload the datasource on load.
Common mistake: Relying on local Grafana accounts instead of SSO for production access control.
