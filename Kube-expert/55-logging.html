Logging

Cluster-wide logging architecture: where container stdout/stderr actually lives, and the patterns for shipping it off-node.
← OpenTelemetry Course Index Fluent Bit →

Explanation

Kubernetes itself does not provide cluster-wide log storage — it only guarantees that whatever a container writes to stdout/stderr is captured by the container runtime and written to a file on the node's local disk. By default, that's the entire logging story: kubectl logs reads those local files (via the kubelet), and if the node dies or the pod is evicted, the logs are gone. Any durable, searchable, cluster-wide logging capability is something you must deliberately build using a log collection agent plus a backend store — Kubernetes gives you the raw material (files on disk), not the pipeline.

Concretely, when a container writes to stdout, the container runtime (containerd/CRI-O) redirects it to a log file under /var/log/pods/<namespace>_<pod>_<uid>/<container>/0.log (JSON-lines or CRI log format), typically symlinked from /var/log/containers/. The kubelet manages log rotation on that file (size- and count-based, configurable via kubelet flags) so a chatty container can't fill node disk. This is why "the log disappeared" after a pod restart or eviction is expected behavior, not a bug — node-local log files are ephemeral by design.

There are three classic architectural patterns for getting logs off the node before they're lost: (1) node-level agent — one DaemonSet pod per node (Fluent Bit, Fluentd, Promtail) tails every container's log file on that node and ships it centrally; this is the dominant pattern because it's low-overhead (one agent per node, not per pod) and requires zero application changes. (2) sidecar container — a logging agent runs inside the same Pod as the app, useful when the app writes logs to a file instead of stdout, or when per-app log processing rules differ significantly. (3) direct shipping from the application — the app itself pushes logs straight to a backend (e.g. via an OTel SDK log exporter), bypassing node files entirely; simplest to reason about but requires every app to implement it, and loses logs generated before a crash if buffering isn't handled carefully.

Kubernetes needs a deliberate logging architecture because troubleshooting production incidents fundamentally depends on correlating what an application said (logs) with what the system observed (metrics) and how a request traveled (traces). Without centralized, durable, searchable logs, engineers are reduced to SSHing into nodes or racing `kubectl logs` before a CrashLoopBackOff pod's log buffer rotates away — untenable at any real scale.

In the wider ecosystem, this topic is the "why" behind Fluent Bit, Loki, and OpenTelemetry logs pipelines covered next: node-level agents (Fluent Bit) read the files this topic describes, ship them to an aggregator/backend (Loki, or an OTel Collector's logs pipeline, or Elasticsearch), and Grafana (or Kibana) is where humans query them — often correlated with Prometheus/VictoriaMetrics metrics and Tempo/Jaeger traces via shared labels like pod name and trace ID.

Internal Working / Interactions

  • Container runtime (containerd/CRI-O) writes each container's stdout/stderr to a per-container log file under /var/log/pods/..., in CRI log format (timestamp, stream, tag, message).
  • kubelet owns log rotation policy (--container-log-max-size, --container-log-max-files) and is what kubectl logs reads from via its API on that node.
  • Node-level agent (DaemonSet) tails /var/log/containers/*.log, enriches each line with Kubernetes metadata (namespace, pod, labels) by querying the API server, and forwards to a backend.
  • Backend (Loki, Elasticsearch, OTel Collector logs pipeline) indexes and stores logs durably, independent of node lifecycle.
  • Grafana/Kibana queries the backend for search, and can correlate a log line's trace ID with a trace in Tempo/Jaeger.

Code Example

# Kubelet configuration controlling node-local log rotation — this is what determines
# how long a container's logs survive on disk before the shipping agent must have read them.
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
containerLogMaxSize: "10Mi"        # rotate once a single log file hits this size
containerLogMaxFiles: 5            # keep at most 5 rotated files per container before deleting the oldest
---
# Minimal DaemonSet pattern for a node-level logging agent (conceptually what Fluent Bit runs as —
# see Topic 56 for the full Fluent Bit config). This shows the volume mounts that make node-level
# log collection possible: read-only access to the node's log directories.
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-agent
  namespace: logging
spec:
  selector:
    matchLabels: { app: log-agent }
  template:
    metadata:
      labels: { app: log-agent }
    spec:
      serviceAccountName: log-agent      # needs RBAC to read Pod metadata for enrichment
      containers:
        - name: agent
          image: fluent/fluent-bit:3.0
          volumeMounts:
            - name: varlog
              mountPath: /var/log          # symlinked container log files live here
              readOnly: true
            - name: varlibcontainers
              mountPath: /var/lib/docker/containers   # underlying log files on some runtimes
              readOnly: true
      volumes:
        - name: varlog
          hostPath: { path: /var/log }
        - name: varlibcontainers
          hostPath: { path: /var/lib/docker/containers }

kubectl / CLI Commands

kubectl logs my-pod                              # tail current container's stdout/stderr via kubelet
kubectl logs my-pod --previous                    # logs from the last crashed instance (before restart)
kubectl logs my-pod -c my-container --since=1h    # scope to a specific container and time window
kubectl logs -f deploy/my-app                     # follow logs live across the deployment's pods
kubectl exec -it my-pod -- ls -la /var/log        # inspect log files an app writes internally (non-stdout case)
kubectl get pods -n logging -l app=log-agent -o wide   # verify the log agent DaemonSet runs on every node
kubectl describe node <node> | grep -A5 "Allocated resources"  # check node disk pressure affecting log retention

Best Practices & Common Pitfalls

Best Practices

  • Always log to stdout/stderr from applications — it's the only pattern Kubernetes natively supports and rotates.
  • Ship logs off-node continuously via a DaemonSet agent; never rely on node disk as durable storage.
  • Include structured fields (JSON logs) so downstream backends can index and query efficiently.
  • Tune containerLogMaxSize/containerLogMaxFiles so rotation doesn't outrun your shipping agent during bursts.
  • Tag every log line with a trace ID when available, to enable log-to-trace correlation later.

Pitfalls

  • Writing logs to a file inside the container instead of stdout, silently losing them (no agent reads it by default).
  • Assuming kubectl logs --previous always works — it depends on the crashed container's log file surviving rotation.
  • Under-provisioning node disk for logs on very chatty workloads, causing disk-pressure evictions.
  • Not setting log level appropriately in production, drowning the pipeline (and budget) in DEBUG-level noise.

50 Interview Questions

1. Does Kubernetes provide cluster-wide log storage out of the box?
Short answer: No — it only guarantees stdout/stderr is captured to a file on the node's local disk.
Detail: Any durable, searchable, cluster-wide logging capability must be deliberately built with a collection agent plus a backend store; Kubernetes gives raw material (files on disk), not a pipeline.
Production example: A fresh cluster with no logging agent installed can still serve kubectl logs, but has zero durable log history once pods are evicted or nodes recycle.
Common mistake: Assuming logs are centrally searchable by default just because kubectl logs works.
2. Where does a container's stdout/stderr actually end up on a node?
Short answer: A per-container log file under /var/log/pods/<namespace>_<pod>_<uid>/<container>/0.log, typically symlinked from /var/log/containers/.
Detail: The container runtime (containerd/CRI-O) redirects stdout/stderr into this file in CRI log format (timestamp, stream, tag, message).
Production example: A node-level agent DaemonSet tails /var/log/containers/*.log to pick up every container's output on that node.
Common mistake: Looking for logs only under /var/log/containers/ and not realizing it's a symlink to the real files under /var/log/pods/.
3. Why do logs disappear after a pod restart or eviction?
Short answer: Node-local log files are ephemeral by design — they live on that node's disk only.
Detail: If the node dies or the pod is evicted, the underlying log file goes with it unless something already shipped the contents elsewhere.
Production example: A CrashLoopBackOff pod's earlier crash logs vanish once kubelet rotates or the pod is rescheduled to another node.
Common mistake: Treating a disappeared log as a bug report-worthy anomaly instead of expected ephemeral-storage behavior.
4. What are the three architectural patterns for getting logs off a node?
Short answer: Node-level agent (DaemonSet), sidecar container, and direct shipping from the application.
Detail: Node-level agent is dominant (one agent per node, zero app changes); sidecar suits apps writing to files instead of stdout; direct shipping bypasses node files entirely but needs every app to implement it.
Production example: Fluent Bit as a DaemonSet is the standard node-level agent pattern; an OTel SDK log exporter pushing straight to a backend is the direct-shipping pattern.
Common mistake: Choosing sidecar-per-pod as the default pattern, multiplying agent overhead by pod count instead of running one agent per node.
5. Who owns log rotation on a node, and what controls it?
Short answer: The kubelet, via --container-log-max-size and --container-log-max-files (or the equivalent KubeletConfiguration fields).
Detail: Rotation prevents a chatty container from filling node disk; once the max file count is reached, the oldest rotated file is deleted.
Production example: containerLogMaxSize: "10Mi" and containerLogMaxFiles: 5 in a KubeletConfiguration bounds each container to 5 rotated 10Mi files.
Common mistake: Setting rotation thresholds too aggressive for a chatty workload, causing the shipping agent to lose lines that rotate out before being read.
6. How does kubectl logs actually retrieve output — does it query a central store?
Short answer: No — it reads the local log file on that pod's node via the kubelet's API, not any centralized store.
Detail: This is why kubectl logs stops working the moment the node or its log files are gone, and why it has nothing to do with your Loki/Elasticsearch backend.
Production example: kubectl logs my-pod --previous pulls the last crashed instance's log, but only if that file survived rotation on the node.
Common mistake: Assuming kubectl logs --previous always works — it depends entirely on the crashed container's log file surviving rotation.
7. Why is the node-level DaemonSet agent pattern considered the dominant approach?
Short answer: It's low-overhead (one agent per node, not per pod) and requires zero application changes.
Detail: The agent tails every container's log file on its node, enriches with Kubernetes metadata, and forwards centrally — apps never need to know it exists.
Production example: A Fluent Bit DaemonSet mounts /var/log and /var/lib/docker/containers read-only to tail every container's output on that node.
Common mistake: Believing a sidecar-per-app gives better isolation without weighing the multiplied resource cost across every pod in the cluster.
8. Why does a node-level logging agent need RBAC permissions?
Short answer: To query the API server and enrich each log line with Kubernetes metadata like namespace, pod, and labels.
Detail: Without metadata enrichment, a raw log line is just a timestamp and message with no way to filter/search by workload identity.
Production example: The example DaemonSet sets serviceAccountName: log-agent, which needs RBAC to read Pod metadata for enrichment before forwarding.
Common mistake: Deploying the agent with a default/no service account and getting logs shipped with no pod/namespace context at all.
9. What's the risk of writing application logs to a file instead of stdout?
Short answer: The logs are silently lost — no node-level agent reads them by default.
Detail: Kubernetes' native log capture and rotation only applies to stdout/stderr; a file inside the container is invisible to the standard pipeline unless a sidecar is added specifically to read it.
Production example: kubectl exec -it my-pod -- ls -la /var/log is how you'd inspect such internally-written log files that bypass the standard capture.
Common mistake: An app team logging to a local file "for safety" and being surprised their logs never reach the central backend.
10. How does log-to-trace correlation actually work in this architecture?
Short answer: Tagging every log line with a trace ID when available, so Grafana/Kibana can jump from a log line to the matching trace in Tempo/Jaeger.
Detail: This requires the application (or its OTel instrumentation) to inject the active trace ID into structured log output; the backend and query UI do the rest.
Production example: A structured JSON log line carrying a trace_id field lets an engineer pivot straight from an error log in Grafana to the full distributed trace.
Common mistake: Shipping unstructured plain-text logs with no trace ID field, losing the ability to correlate logs with traces even though both pipelines exist.
