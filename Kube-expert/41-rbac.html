RBAC (Role-Based Access Control)

Who can do what, to which resources, in which namespaces — enforced at the API Server on every single request.
← Health Probes Course Index Service Accounts →

Explanation

RBAC (Role-Based Access Control) is Kubernetes' native authorization system: it decides, for every request that reaches the API Server, whether the authenticated identity is allowed to perform the requested verb (get, list, create, delete, etc.) on the requested resource (pods, secrets, deployments, etc.) in the requested namespace. RBAC runs as the authorization step of the API Server's request pipeline — strictly after authentication (who are you?) and strictly before admission control (should this specific object be allowed?). If RBAC denies a request, it never even reaches admission webhooks or etcd.

Kubernetes needs RBAC because a cluster is a shared, multi-tenant system by default — every user, CI pipeline, in-cluster controller, and Pod-attached identity (via a ServiceAccount, topic 42) talks to the exact same API Server. Without an authorization layer, anyone who could authenticate at all could do anything: read every Secret, delete any Deployment, escalate their own privileges. RBAC is what makes "least privilege" enforceable rather than aspirational — every identity gets exactly the verbs on exactly the resources it needs, nothing more.

RBAC is built from four object types, always used in pairs:

  • Role — a set of permission rules (verbs + resources + apiGroups) scoped to a single namespace.
  • ClusterRole — the same kind of rule set, but scoped cluster-wide (or reusable across namespaces); required for cluster-scoped resources like Nodes or PersistentVolumes, which have no namespace to bind a Role to.
  • RoleBinding — grants the permissions in a Role (or a ClusterRole, used namespace-scoped) to a specific user, group, or ServiceAccount, within one namespace.
  • ClusterRoleBinding — grants the permissions in a ClusterRole to a subject cluster-wide, across every namespace.

A Role/ClusterRole's rules are built from three dimensions that must all match for a permission to apply: apiGroups (e.g. "" for core resources like Pods, apps for Deployments, rbac.authorization.k8s.io for RBAC objects themselves), resources (the plural resource name, e.g. pods, pods/log for the log subresource, secrets), and verbs (get, list, watch, create, update, patch, delete, deletecollection). RBAC in Kubernetes is purely additive/allow-based — there is no explicit "deny" rule; a request is authorized if any binding grants it, and denied by default otherwise.

In production, least-privilege RBAC design means granting the narrowest possible resources/verbs combination per identity — a monitoring ServiceAccount gets get, list, watch on Pods/Nodes/metrics, never create or delete; a CI/CD ServiceAccount deploying to one namespace gets a Role (not a ClusterRole) scoped to that namespace only. A common and dangerous anti-pattern is binding cluster-admin broadly "to get things working," which silently grants unrestricted access to every resource in every namespace, including Secrets and RBAC objects themselves (a direct privilege-escalation path).

Privilege escalation risk: any identity with create/update permission on roles, rolebindings, clusterroles, or clusterrolebindings can, in many configurations, grant itself additional permissions — Kubernetes has explicit escalation-prevention checks, but the safest posture is to treat RBAC-object write access itself as one of the most sensitive permissions in the cluster and restrict it to platform admins only.

Internal Working / Interactions

  • API Server: runs the RBAC authorizer as part of its request-handling chain, for every single request, immediately after authentication and before admission control — this happens on the hot path of literally every API call.
  • Authentication layer: supplies the identity (username, groups, or ServiceAccount) that RBAC then checks bindings against — RBAC never authenticates, it only authorizes an already-established identity.
  • etcd: Roles, ClusterRoles, RoleBindings, and ClusterRoleBindings are ordinary API objects stored in etcd like anything else, watched/cached by the API Server for fast lookup on every request.
  • ServiceAccount controller / kubelet: ServiceAccounts (topic 42) are the identity RBAC most commonly authorizes for in-cluster callers (Pods, controllers) — the RBAC layer treats a ServiceAccount exactly like a user for binding purposes.
  • Admission webhooks: run strictly after RBAC authorization succeeds — a webhook can further restrict or mutate a request, but RBAC has already gated whether the request is considered at all.

Code Example

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: deployer-role
  namespace: payments             # Role is namespace-scoped - only applies within "payments"
rules:
  - apiGroups: ["apps"]            # Deployments live in the "apps" API group
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: [""]                # core API group (empty string) for Pods
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
    # Deliberately NO "delete" or "create" on pods - CI can update Deployments,
    # but pod lifecycle is left to the Deployment controller, not the CI identity directly.
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: ci-deployer-binding
  namespace: payments
subjects:
  - kind: ServiceAccount
    name: ci-deployer
    namespace: payments
roleRef:                          # binds the Role above to the subject above
  kind: Role
  name: deployer-role
  apiGroup: rbac.authorization.k8s.io

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: read-only-monitoring
rules:
  - apiGroups: [""]
    resources: ["pods", "nodes", "services", "endpoints"]
    verbs: ["get", "list", "watch"]      # strictly read-only, cluster-wide
  - apiGroups: ["metrics.k8s.io"]
    resources: ["pods", "nodes"]
    verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: monitoring-binding
subjects:
  - kind: ServiceAccount
    name: prometheus
    namespace: monitoring
roleRef:
  kind: ClusterRole
  name: read-only-monitoring
  apiGroup: rbac.authorization.k8s.io

kubectl Commands

# Check whether an identity CAN perform an action (the single most useful RBAC debug command)
kubectl auth can-i create deployments --namespace payments --as system:serviceaccount:payments:ci-deployer

# List all roles/rolebindings in a namespace
kubectl get roles,rolebindings -n payments

# List all clusterroles/clusterrolebindings cluster-wide
kubectl get clusterroles,clusterrolebindings

# Describe a role to see exact rules granted
kubectl describe role deployer-role -n payments

# Find every binding referencing a specific ServiceAccount
kubectl get rolebindings,clusterrolebindings -A -o json | jq '.items[] | select(.subjects[]?.name=="ci-deployer")'

# Audit who has cluster-admin (a critical security review command)
kubectl get clusterrolebindings -o json | jq '.items[] | select(.roleRef.name=="cluster-admin")'

Best Practices & Common Pitfalls

Do's

  • Design Roles around the minimum verb/resource set an identity actually needs — audit and tighten regularly.
  • Prefer namespaced Role + RoleBinding over ClusterRole + ClusterRoleBinding unless cluster-wide scope is genuinely required.
  • Use kubectl auth can-i in CI to validate expected permissions before deploying RBAC changes.
  • Treat write access to RBAC objects themselves (roles, bindings) as a highly privileged permission, restricted to platform admins.

Don'ts

  • Don't bind cluster-admin to application ServiceAccounts "to unblock" a permission error — diagnose the actual missing rule instead.
  • Don't grant wildcard resources: ["*"]/verbs: ["*"] rules except for genuinely cluster-admin-equivalent identities.
  • Don't forget Secrets read access is extremely sensitive — get on secrets is close to equivalent to broad credential access.
  • Don't assume RBAC alone is sufficient isolation — combine with NetworkPolicies and Pod Security Standards for full workload isolation.

50 Interview Questions

1. Where does RBAC fit in the API Server's request pipeline?
Short answer: RBAC runs as the authorization step, strictly after authentication (who are you?) and strictly before admission control (should this specific object be allowed?).
Detail: If RBAC denies a request, it never reaches admission webhooks or etcd — authorization is a hard gate that happens on every single API request.
Production example: A request to get pods as ServiceAccount ci-deployer passes authentication, then RBAC checks bindings before the request is ever considered by any admission webhook.
Common mistake: Assuming admission webhooks run before or independently of RBAC — they strictly run after RBAC has already authorized the request.
2. Name RBAC's four object types and explain how they pair up.
Short answer: Role (namespaced rule set) pairs with RoleBinding (grants it to a subject within one namespace); ClusterRole (cluster-wide rule set) pairs with ClusterRoleBinding (grants it to a subject cluster-wide).
Detail: A ClusterRole can also be used namespace-scoped via a RoleBinding — useful for reusing a common rule set across many namespaces without duplicating it as separate Roles.
Production example: deployer-role (Role) + ci-deployer-binding (RoleBinding) scope CI's permissions to the payments namespace only; read-only-monitoring (ClusterRole) + monitoring-binding (ClusterRoleBinding) grant Prometheus cluster-wide read access.
Common mistake: Assuming a ClusterRole always grants cluster-wide access — it only does so when bound via a ClusterRoleBinding; bound via a RoleBinding, its scope is limited to that one namespace.
3. Why must ClusterRole be used for resources like Nodes or PersistentVolumes?
Short answer: Because those resources are cluster-scoped — they have no namespace at all, so there's no namespace to bind a Role to.
Detail: A Role's rules can only ever apply within the single namespace it's created in; any resource kind without a namespace field is structurally incompatible with a Role, requiring a ClusterRole regardless of how it's bound.
Production example: read-only-monitoring includes nodes in its rules, which only makes sense as a ClusterRole since Nodes aren't namespaced.
Common mistake: Attempting to grant access to a cluster-scoped resource via a namespaced Role — it simply can't reference such resources correctly.
4. What three dimensions must all match for an RBAC rule to authorize a request?
Short answer: apiGroups (e.g. "" for core, apps for Deployments), resources (plural resource name, e.g. pods, pods/log), and verbs (get, list, watch, create, update, patch, delete, deletecollection).
Detail: All three must match simultaneously — a rule granting get on pods in the core group says nothing about get on pods/log, which is a distinct resource requiring its own explicit grant.
Production example: deployer-role explicitly lists both pods and pods/log as separate resource entries because the log subresource needs its own rule.
Common mistake: Assuming a grant on a resource automatically covers its subresources (like /log or /exec) — subresources require explicit separate entries.
5. Is RBAC allow-based or deny-based, and what does that imply about combining rules?
Short answer: RBAC is purely additive/allow-based — there is no explicit "deny" rule; a request is authorized if any binding grants it, denied by default otherwise.
Detail: This means multiple Roles/ClusterRoles bound to the same subject are cumulative — you can't use a second binding to subtract or restrict a permission already granted elsewhere.
Production example: If ci-deployer somehow had two RoleBindings, one broad and one narrow, the broad one's grants still apply — the narrow one can't revoke them.
Common mistake: Trying to "restrict" an identity's access by adding a more limited binding, expecting it to override a broader one already in place — RBAC has no such override/deny mechanism.
6. Why is granting an identity write access to RBAC objects themselves considered high-risk?
Short answer: Any identity with create/update permission on roles, rolebindings, clusterroles, or clusterrolebindings can, in many configurations, grant itself additional permissions — a direct privilege-escalation path.
Detail: Kubernetes has explicit escalation-prevention checks, but the safest posture is treating RBAC-object write access as one of the most sensitive permissions in the cluster, restricted to platform admins only.
Production example: A ServiceAccount able to create RoleBindings could bind itself to a broader ClusterRole it wasn't originally granted, effectively escalating its own privileges.
Common mistake: Granting a CI/CD identity broad create/update access on RBAC objects "to manage its own permissions," inadvertently opening a self-escalation path.
7. Why does RBAC treat a ServiceAccount exactly like a user for binding purposes?
Short answer: ServiceAccounts (topic 42) are the identity RBAC most commonly authorizes for in-cluster callers (Pods, controllers), and the RBAC layer subjects them to the exact same Role/ClusterRole binding mechanism as human users or groups.
Detail: This uniformity means the same least-privilege design principles apply regardless of whether the "subject" in a binding is a human, a group, or a ServiceAccount.
Production example: ci-deployer-binding lists a ServiceAccount as its subject in exactly the same subjects array shape a User or Group subject would use.
Common mistake: Assuming ServiceAccounts get special implicit permissions beyond what's explicitly bound — they're authorized through the identical RBAC mechanism as any other subject.
8. What is the most useful single command for debugging an RBAC permission issue, and why?
Short answer: kubectl auth can-i <verb> <resource> --as <identity> — it directly answers "would this specific request be authorized" without needing to manually trace bindings and rules.
Detail: It's also usable proactively in CI to validate expected permissions before deploying RBAC changes, rather than discovering a gap only when a real deployment fails with 403.
Production example: kubectl auth can-i create deployments --namespace payments --as system:serviceaccount:payments:ci-deployer confirms whether ci-deployer-binding actually grants the expected access.
Common mistake: Manually cross-referencing Roles and RoleBindings by hand to guess whether access is granted, instead of directly asking the API Server via auth can-i.
9. Why is binding cluster-admin broadly considered a dangerous anti-pattern?
Short answer: It silently grants unrestricted access to every resource in every namespace, including Secrets and RBAC objects themselves — a direct privilege-escalation path if bound to anything less than a fully trusted platform-admin identity.
Detail: Teams often reach for it "to get things working" when they hit an unexpected 403, which papers over the real missing rule instead of diagnosing and granting the specific narrow permission actually needed.
Production example: Auditing via kubectl get clusterrolebindings -o json | jq '.items[] | select(.roleRef.name=="cluster-admin")' is called out as a critical security review command specifically because of this risk.
Common mistake: Binding cluster-admin to an application ServiceAccount to unblock a permission error rather than diagnosing the actual missing rule.
10. Why is read access to Secrets treated as an especially sensitive RBAC grant?
Short answer: get on secrets is close to equivalent to broad credential access, since Secrets typically store the very credentials that would otherwise require separate compromise to obtain.
Detail: This makes Secret-read grants deserve the same scrutiny as granting direct credential access, not routine data-read access — the resource name matters as much as the verb.
Production example: deployer-role deliberately grants no access to Secrets at all, keeping the CI identity's blast radius limited to Deployments and read-only Pod visibility.
Common mistake: Treating get/list on Secrets as routine read access equivalent in sensitivity to reading Pods or Deployments.
