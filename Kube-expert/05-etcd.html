etcd

The single source of truth: Raft consensus, key layout, snapshots, and quorum guarantees underlying the whole cluster.
← API Server Course Index Scheduler →

Explanation

etcd is a distributed, strongly-consistent key-value store, and it is the only place Kubernetes durably stores cluster state. Every object you create — Pods, Secrets, Deployments, Nodes — ends up as a serialized protobuf value under a structured key in etcd. Nothing in Kubernetes is "real" until it's committed to etcd; the API Server itself holds no durable state of its own.

Kubernetes needs etcd's specific guarantees — not just any database — because a distributed control system requires linearizable reads/writes (every client sees a single consistent order of changes) and a watch API (clients get notified of changes instead of polling). etcd provides both natively via the Raft consensus algorithm: one node is elected leader, all writes go through the leader, and a write is only acknowledged once a majority of members have durably persisted it. This is what prevents split-brain — two different parts of the cluster can never both believe conflicting things are true.

Because Raft requires a majority (quorum) to commit any write, etcd clusters are run with an odd number of members — 3 or 5. A 3-member cluster tolerates 1 failure; a 5-member cluster tolerates 2. Adding a 4th member to a 3-member cluster does not add resilience — it only raises the quorum threshold. This is a frequently misunderstood point: more members is not free — each additional member adds replication overhead and doesn't scale throughput (all writes still funnel through the leader).

etcd stores Kubernetes objects under keys shaped like /registry/{resource}/{namespace}/{name}, e.g. /registry/pods/default/nginx-abc123. This flat, path-like key layout is why etcd's watch API can efficiently support "watch everything under /registry/pods" style queries that the API Server relies on. Because Secrets are stored the same way as everything else, they are plaintext by default unless encryption-at-rest is explicitly configured — a fact every security review should catch.

etcd is extremely latency-sensitive: every write requires an fsync to disk before it's acknowledged, so slow disks directly translate into slow API Server responses cluster-wide. This is the single most common root cause of "the whole cluster feels sluggish" incidents, and it's why production etcd clusters mandate low-latency SSDs and tight network proximity between members.

Internal Working / Interactions

  • etcd ↔ API Server only: no other component ever talks to etcd directly — this is enforced by convention and network policy, not a technical restriction.
  • Raft leader election: one etcd member is leader; all writes are proposed to it, replicated to followers, and committed once a quorum acknowledges.
  • Watch stream: the API Server maintains a long-lived watch on etcd's key space and fans changes out to its own watchers (controllers, kubelets).
  • Compaction/defrag: etcd keeps historical revisions until compacted; periodic compaction + defragmentation reclaims disk space and keeps performance stable.
  • Snapshotting: etcdctl snapshot save captures a point-in-time backup of the entire keyspace for disaster recovery.

Code Example

# etcd is not configured via Kubernetes YAML — it's configured via its own
# static pod manifest (kubeadm) or systemd unit flags. Key production flags:

# /etc/kubernetes/manifests/etcd.yaml (relevant command args)
- --data-dir=/var/lib/etcd
- --initial-cluster=cp1=https://10.0.1.10:2380,cp2=https://10.0.1.11:2380,cp3=https://10.0.1.12:2380
- --initial-cluster-state=new
- --listen-peer-urls=https://10.0.1.10:2380       # member-to-member Raft traffic
- --listen-client-urls=https://10.0.1.10:2379     # API Server connects here
- --quota-backend-bytes=8589934592                # raise beyond default 2GB for large clusters
- --auto-compaction-mode=periodic
- --auto-compaction-retention=1h                    # compact old revisions hourly to bound DB growth

# Backup command (run on any etcd member, requires TLS certs)
etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /backup/etcd-snapshot-$(date +%F).db

# Restore from snapshot into a fresh data directory (used during DR)
etcdctl snapshot restore /backup/etcd-snapshot-2026-07-19.db \
  --data-dir=/var/lib/etcd-restored \
  --initial-cluster=cp1=https://10.0.1.10:2380 \
  --initial-advertise-peer-urls=https://10.0.1.10:2380

kubectl Commands

# etcd health is not exposed via kubectl directly — use etcdctl on a control-plane node:
etcdctl endpoint health --cluster -w table       # per-member health
etcdctl endpoint status --cluster -w table       # shows leader, DB size, raft term
etcdctl member list -w table                     # list all etcd members and their URLs

# Indirect visibility via kubectl:
kubectl get pods -n kube-system -l component=etcd    # etcd static pods (kubeadm)
kubectl get --raw='/healthz/etcd'                    # API Server's view of etcd reachability
kubectl top node                                     # slow etcd often manifests as slow describe/get here first
kubectl logs etcd-controlplane-1 -n kube-system      # etcd's own logs (fsync latency warnings appear here)

Best Practices & Common Pitfalls

Best Practices

  • Run 3 or 5 members on dedicated low-latency SSD storage.
  • Automate regular snapshot backups to off-cluster object storage.
  • Enable encryption-at-rest for Secrets stored in etcd.
  • Monitor fsync and network round-trip latency continuously — they predict issues before quorum is lost.

Pitfalls

  • Running an even number of members — no resilience benefit, higher split-vote risk.
  • Using network-attached storage with high/variable latency for the data directory.
  • Letting the DB grow unbounded without compaction/defrag — leads to mvcc: database space exceeded.
  • Exposing etcd's client port beyond the control-plane network — anyone with access effectively has cluster-admin.

50 Interview Questions

1. Why does Kubernetes use etcd specifically instead of any other key-value store?
Short answer: etcd natively provides linearizable reads/writes via Raft consensus and a built-in watch API, both required by Kubernetes' architecture.
Detailed explanation: Kubernetes needs every client to see a single consistent order of changes (linearizability) and needs to notify watchers of changes instead of polling. Raft gives strict write ordering through a single elected leader, and etcd's watch API is a first-class primitive, not bolted on.
Production example: The API Server's own watch mechanism for controllers/kubelets is built directly on top of etcd's native watch stream.
Common mistake: Assuming any consistent database would work — the watch API and Raft-based consistency guarantees are both load-bearing, not incidental.
2. Explain how Raft consensus makes an etcd write durable and consistent.
Short answer: A write is proposed to the elected leader, replicated to followers, and only acknowledged once a majority (quorum) durably persists it.
Detailed explanation: One etcd member is leader at any time; all writes flow through it. The leader replicates the entry to followers and waits for acknowledgment from a majority before considering the write committed — this prevents split-brain because no two conflicting values can both achieve quorum simultaneously.
Production example: In a 3-member cluster, a write is durable once 2 of 3 members (leader + 1 follower) have persisted it to disk.
Common mistake: Believing a write is safe as soon as the leader accepts it — it isn't durable until quorum acknowledges.
3. Why must etcd clusters run an odd number of members, and why doesn't adding more always help?
Short answer: Raft needs a strict majority; odd numbers give the best fault-tolerance-to-member ratio, and each additional member increases replication overhead without adding throughput.
Detailed explanation: A 3-member cluster tolerates 1 failure (needs 2/3); a 5-member cluster tolerates 2 (needs 3/5). A 4-member cluster still only tolerates 1 failure (needs 3/4) while requiring more replication work — strictly worse than 3. All writes still funnel through one leader regardless of member count, so more members never increases write throughput.
Production example: Production etcd topologies standardize on 3 (small/medium clusters) or 5 (very large clusters needing to tolerate 2 failures).
Common mistake: Adding a 4th member "for extra safety" — it raises the quorum threshold without adding real resilience.
4. How are Kubernetes objects actually stored as keys in etcd?
Short answer: Under a flat, path-like key structure: /registry/{resource}/{namespace}/{name}.
Detailed explanation: This layout, e.g. /registry/pods/default/nginx-abc123, is why etcd's range-watch capabilities can efficiently support "watch everything under /registry/pods" style queries that the API Server relies on for its own watch fan-out to clients.
Production example: Direct etcdctl get against this key prefix on a lab cluster reveals the raw serialized object the API Server would otherwise abstract away.
Common mistake: Thinking Kubernetes objects are stored in some Kubernetes-specific format rather than plain etcd keys with serialized (protobuf) values.
5. Why are Secrets in etcd plaintext by default, and how do you fix that?
Short answer: Secrets are stored the same way as every other object (just base64-encoded, not encrypted) unless encryption-at-rest is explicitly configured.
Detailed explanation: Kubernetes' generic object storage doesn't special-case Secrets for encryption — you must configure an EncryptionConfiguration on the API Server to encrypt Secret values before they're written to etcd.
Production example: A security review flags an EKS cluster without KMS-backed etcd encryption as a compliance gap for PCI-DSS/SOC2.
Common mistake: Assuming Secrets are inherently "secure" just because they're a distinct resource type — anyone with direct etcd access can read them in plaintext without encryption-at-rest.
6. Why is etcd so sensitive to disk and network latency?
Short answer: Every write requires an fsync to disk before acknowledgment, and Raft replication between members is latency-sensitive to network round-trips.
Detailed explanation: A slow disk directly delays every write's commit time, which cascades into slow API Server responses cluster-wide since the API Server can't acknowledge a write until etcd does. Network jitter between members can also trigger unnecessary leader re-elections.
Production example: Switching etcd's data directory from network-attached storage to local NVMe SSD resolves cluster-wide "kubectl feels sluggish" complaints.
Common mistake: Running etcd on network storage or spinning disks to save cost, not realizing it degrades the entire cluster's responsiveness.
7. What is etcd compaction and defragmentation, and why are both needed?
Short answer: Compaction discards old historical revisions (MVCC); defragmentation reclaims the freed disk space at the storage-engine level.
Detailed explanation: etcd's MVCC model keeps every historical revision of every key until compacted, so disk usage grows unbounded without periodic compaction; even after compaction, the underlying bbolt database file doesn't shrink until defragmented.
Production example: --auto-compaction-mode=periodic --auto-compaction-retention=1h compacts hourly; a separate scheduled defrag job then reclaims disk space.
Common mistake: Enabling compaction but never defragmenting, then still hitting mvcc: database space exceeded because the file itself never shrank.
8. How do you back up and restore etcd, and when would you need to?
Short answer: etcdctl snapshot save captures a point-in-time backup; etcdctl snapshot restore rebuilds a data directory from it during disaster recovery.
Detailed explanation: Restore is typically needed after total quorum loss, accidental mass deletion, or corruption — it rebuilds a fresh data directory that a new etcd cluster is then bootstrapped from, not a live in-place operation.
Production example: A platform team automates hourly snapshots to S3, with a documented runbook to restore from the latest snapshot during a DR drill.
Common mistake: Taking backups but never testing the restore procedure until an actual outage, discovering gaps in the runbook under pressure.
9. What happens operationally when etcd loses quorum?
Short answer: All writes fail cluster-wide; reads may still succeed from a stale follower, but nothing can change.
Detailed explanation: Quorum loss (more than (N-1)/2 members down) means no majority can commit new entries via Raft, so the API Server can't persist any change — Pods can't be created, scaled, or deleted, though already-running workloads keep functioning.
Production example: Losing 2 of 3 etcd members simultaneously (e.g. a bad AZ-wide network event) halts all cluster mutations until members are restored.
Common mistake: Forcing a single surviving member into a fake "quorum" as a workaround without understanding the data-loss and split-brain risk this introduces.
10–50. Additional rapid-fire questions
10. What consensus algorithm does etcd use? — Raft.
11. How many members can a 5-node etcd cluster lose and still function? — 2 (needs 3 of 5 for quorum).
12. What's the default etcd quota-backend-bytes limit? — 2GB, raised via --quota-backend-bytes for large clusters.
13. What error indicates the etcd DB size limit was hit? — etcdserver: mvcc: database space exceeded.
14. What command shows per-member health across an etcd cluster? — etcdctl endpoint health --cluster -w table.
15. What command shows which member is currently the Raft leader? — etcdctl endpoint status --cluster -w table.
16. Why should etcd's client port never be exposed outside the control-plane network? — Anyone with access effectively has cluster-admin, since it holds all cluster state including Secrets.
17. What's the difference between etcd's peer URL and client URL? — Peer URLs (--listen-peer-urls) handle member-to-member Raft traffic; client URLs (--listen-client-urls) are where the API Server connects.
18. Why is MVCC relevant to etcd's watch API? — MVCC keeps historical revisions, letting watchers resume from a specific revision instead of losing events during reconnects.
19. What's a typical symptom of etcd fsync latency problems? — Slow kubectl commands cluster-wide, especially writes (create/update/delete), even though reads may feel fine.
20. Why does etcd prefer local SSD over network-attached storage? — fsync latency and consistency guarantees are far more predictable on local, low-latency storage.
21. What's the risk of etcd members being geographically far apart? — Raft replication round-trip latency increases, slowing every write and increasing the chance of unnecessary leader re-elections.
22. Why does Kubernetes never let controllers write to etcd directly? — Only the API Server should enforce authn/authz/admission before any persistence — direct writes would bypass all of that.
23. What does --initial-cluster-state=new signify during bootstrap? — Tells etcd this is a brand-new cluster being formed, not joining an existing one.
24. How would you add a new member to an existing etcd cluster safely? — Use etcdctl member add first to register it, then start the new member pointing at the updated cluster list — order matters to avoid quorum issues.
25. What's the effect of an etcd leader crashing? — Remaining members hold a new election; a brief write unavailability window occurs until a new leader is elected.
26. Why is etcd described as "the single source of truth"? — Nothing in Kubernetes is considered real until it's committed to etcd — the API Server itself holds no durable state.
27. What's a good way to test a DR runbook without touching production? — Take a snapshot from a lab/staging etcd cluster, deliberately corrupt/delete data, then practice the full restore procedure end to end.
28. Why does etcd compaction need to run periodically rather than never? — Without it, historical revisions accumulate indefinitely and eventually exceed the DB size quota.
29. What's the relationship between etcd revision numbers and resourceVersion in Kubernetes? — Kubernetes' resourceVersion is derived from etcd's internal revision, used for optimistic concurrency and watch resumption.
30. Why might you see "etcdserver: request timed out" errors? — etcd is overloaded or its disk I/O is too slow to commit writes within the client's timeout.
31. What's the purpose of TLS certificates in etcd's configuration? — Encrypts and authenticates both peer (member-to-member) and client (API Server-to-etcd) traffic.
32. Why is running etcd co-located with the API Server on the same control-plane node common? — Minimizes network latency between the two, since API Server performance is bottlenecked heavily by etcd round-trip time.
33. What's a leading indicator metric to alert on for etcd health? — fsync/commit duration and Raft proposal failure rate.
34. Why is it risky to restore an etcd snapshot into a running cluster without care? — It replaces the entire keyspace state, potentially reverting recent legitimate changes or causing conflicts with already-running Pods/controllers.
35. What's the difference between etcd's "leader" and Kubernetes controller "leader election"? — etcd's Raft leader handles consensus for etcd's own writes; Kubernetes leader election (via Lease objects) is a separate, higher-level mechanism used by Scheduler/Controller Manager, implemented using etcd/API Server underneath.
36. Why shouldn't you scale etcd cluster size the way you'd scale stateless services? — More members add Raft replication overhead without adding write throughput, since all writes still go through one leader.
37. What's the impact of a slow etcd disk on Pod scheduling latency? — Scheduler bindings must be persisted through the API Server to etcd, so slow etcd commits directly delay how fast Pods transition out of Pending.
38. How can you verify an etcd snapshot is valid before relying on it for DR? — etcdctl snapshot status reports hash, revision, and size, letting you sanity-check it before an actual restore.
39. Why does etcd use a term/revision-based model rather than plain timestamps? — Logical revisions avoid the coordination problems of relying on potentially-skewed wall-clock time across distributed members.
40. What's a real consequence of exposing etcd without TLS? — Any network-adjacent attacker could read or write raw cluster state, including all Secrets, completely bypassing Kubernetes RBAC.
41. Why is etcd backup automation (not just manual snapshotting) considered a best practice? — Manual processes are forgotten or done inconsistently; automated, tested, off-cluster backups are the only reliable DR guarantee.
42. What's the practical difference between etcd being "down" vs "quorum lost"? — A single member being down with quorum intact is transparent to clients; quorum loss halts all writes cluster-wide.
43. Why does etcd's watch mechanism matter beyond just the API Server? — It's the foundational primitive the API Server's own client watch/informer system is built on top of.
44. What's a symptom in etcd logs that predicts an upcoming performance issue? — Repeated warnings about slow fsync duration or applied entries taking too long.
45. Why is restoring from a very old snapshot risky? — It can revert the cluster to a stale desired state, causing controllers to "correct" already-changed real-world resources back to outdated configurations.
46. What's the operational reason production etcd clusters avoid public cloud "burstable" storage tiers? — Inconsistent/variable IOPS and latency directly translate into unpredictable etcd (and thus cluster-wide) performance.
47. How does etcd's design relate to the CAP theorem? — etcd favors consistency and partition tolerance (CP) over availability — it will refuse writes rather than risk inconsistent state during a partition.
48. Why might restoring etcd not fully recover a cluster by itself? — Some controller-managed external resources (cloud LoadBalancers, PVs) may be orphaned or duplicated if their corresponding etcd records were rolled back to a stale state.
49. What's the first command to run when investigating "the cluster feels slow" that points at etcd specifically? — etcdctl endpoint status --cluster -w table to check DB size, leader, and latency across members.
50. Why is etcd considered the highest-value single component to protect operationally? — It's the sole durable source of truth for the entire cluster — its loss (without backups) means total, unrecoverable loss of cluster state.
