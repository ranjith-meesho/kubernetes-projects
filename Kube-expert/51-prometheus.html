Prometheus

Pull-based metrics collection, ServiceMonitors, PromQL, and alerting via Alertmanager.
← Observability Course Index VictoriaMetrics →

Explanation

Prometheus is the de facto standard metrics system for Kubernetes — a CNCF graduated project that implements the metrics pillar of observability (Topic 50) with a specific, deliberate design choice: it pulls (scrapes) metrics from targets over HTTP on a schedule, rather than having applications push metrics to it. Kubernetes needs a metrics system with exactly this shape because the cluster is inherently dynamic — Prometheus's service-discovery integration lets it automatically find scrape targets as Pods come and go, rather than every workload needing to know a metrics endpoint's address to push to.

Every scrape target exposes a plain-text /metrics HTTP endpoint listing current values for its counters, gauges, histograms, and summaries. Prometheus scrapes each target on an interval (commonly 15-30s), storing each sample with a timestamp in its own purpose-built time-series database. The pull model has real operational advantages in Kubernetes: Prometheus itself can detect a target is down (a failed scrape is a strong, unambiguous signal), and targets don't need any awareness of where Prometheus lives or need network egress to it — Prometheus just needs to know how to find and reach them.

Manually configuring scrape targets for every microservice doesn't scale, so most clusters run the Prometheus Operator (Topic 46's Operator pattern applied to Prometheus itself), which introduces the ServiceMonitor CRD: a declarative object saying "scrape any Service matching these labels, on this port, at this path." The Operator watches ServiceMonitors and automatically regenerates Prometheus's underlying scrape configuration — so onboarding a new microservice's metrics is "add a ServiceMonitor," not "hand-edit a giant central Prometheus config file and reload it."

PromQL is Prometheus's query language, built around the concept of a time series identified by a metric name plus a set of label key/value pairs. Basic queries select a metric (http_requests_total); label matchers filter it (http_requests_total{service="checkout"}); range vectors and functions compute rates over time (rate(http_requests_total[5m]) — the per-second average request rate over the last 5 minutes, essential because raw counters only ever increase and are meaningless without a rate). Aggregation operators (sum by (service) (...)) roll many series into fewer for dashboards and alerts.

Alerting is a separate concern handled by evaluating PromQL expressions as alerting rules (e.g. "fire if error rate > 5% for 5 minutes") inside Prometheus itself, which then forwards firing alerts to Alertmanager — a distinct component responsible for deduplication, grouping, silencing, and routing alerts to notification channels (PagerDuty, Slack, email). This separation means Prometheus's job stops at "is this condition true," while Alertmanager owns the entirely different problem of "who should be told, how, and how often."

Internal Working / Interactions

  • Prometheus server: scrapes targets on an interval, stores samples in its local TSDB, evaluates alerting/recording rules.
  • Prometheus Operator: watches ServiceMonitor/PodMonitor/PrometheusRule CRDs and regenerates the running Prometheus server's config automatically.
  • Kubernetes API Server / kube-state-metrics / cAdvisor: common scrape targets exposing cluster-object state and container resource usage respectively.
  • Alertmanager: receives firing alerts pushed from Prometheus, handles dedup/grouping/silencing/routing to external notification channels.
  • Grafana: queries Prometheus's HTTP API (PromQL) to render dashboards — a separate, stateless visualization layer.

Code Example

# 1. ServiceMonitor — tells the Prometheus Operator to scrape checkout-service Pods
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: checkout-service
  namespace: payments
  labels:
    release: prometheus            # Must match the Prometheus CR's serviceMonitorSelector.
spec:
  selector:
    matchLabels:
      app: checkout-service         # Selects the Service (not the Pod directly) with this label.
  endpoints:
    - port: metrics                 # Named port on the Service, e.g. "metrics" -> containerPort 9100.
      path: /metrics
      interval: 30s
---
# 2. PrometheusRule — alerting rule evaluated by Prometheus, routed via Alertmanager
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: checkout-service-alerts
  namespace: payments
  labels:
    release: prometheus
spec:
  groups:
    - name: checkout-service.rules
      rules:
        - alert: CheckoutHighErrorRate
          expr: |
            sum(rate(http_requests_total{service="checkout",status=~"5.."}[5m]))
            /
            sum(rate(http_requests_total{service="checkout"}[5m])) > 0.05
          for: 5m                    # Must stay true for 5 minutes before firing (avoids noisy blips).
          labels:
            severity: critical
          annotations:
            summary: "Checkout service 5xx rate above 5% for 5 minutes"

Best Practices & Common Pitfalls

Best Practices

  • Use ServiceMonitors/PodMonitors via the Operator rather than hand-editing Prometheus's scrape config directly.
  • Always wrap counters in rate() over a window before graphing/alerting — raw counter values are meaningless.
  • Use for: durations on alerting rules to avoid paging on transient blips.
  • Keep label cardinality bounded — avoid labels with unbounded values like raw user IDs or request IDs.

Pitfalls

  • Forgetting the ServiceMonitor selects a Service, not a Pod directly — a missing/mislabeled Service means silent scrape failure.
  • Running a single non-HA Prometheus instance for production alerting with no redundancy — a single point of failure for your alerting pipeline.
  • Confusing Prometheus's local TSDB retention limits with long-term storage needs — long-term/high-cardinality workloads usually need a remote-write backend (see VictoriaMetrics, next topic).
  • Routing every alert straight to a human pager without grouping/silencing in Alertmanager, causing alert fatigue.

50 Interview Questions

1. Why does Prometheus pull (scrape) metrics rather than have applications push them?
Short answer: The pull model lets Prometheus itself detect a target is down (a failed scrape is unambiguous), and targets never need to know where Prometheus lives or need network egress to it.
Detail: This fits Kubernetes's dynamic nature — service discovery lets Prometheus automatically find scrape targets as Pods come and go, instead of every workload needing to know a metrics endpoint's address to push to.
Production example: A crashed Pod simply stops responding to scrapes, and the resulting `up == 0` series is itself alertable — no separate "did the push stop" heuristic needed.
Common mistake: Assuming Prometheus needs inbound network access configured per-application rather than the reverse — it's Prometheus reaching out, not applications pushing in.
2. What does a scrape target's /metrics endpoint actually expose?
Short answer: A plain-text HTTP endpoint listing current values for its counters, gauges, histograms, and summaries.
Detail: Prometheus scrapes each target on an interval (commonly 15-30s) and stores each sample with a timestamp in its own purpose-built time-series database (TSDB).
Production example: `curl localhost:9100/metrics` on a Pod shows raw text like `http_requests_total{method="GET"} 4231`.
Common mistake: Expecting /metrics to return JSON — it's a specific plain-text exposition format, not a generic API response.
3. What problem does the Prometheus Operator's ServiceMonitor CRD solve?
Short answer: It replaces hand-editing a giant central Prometheus scrape config with a declarative object saying "scrape any Service matching these labels, on this port, at this path."
Detail: The Operator watches ServiceMonitors and automatically regenerates Prometheus's underlying scrape configuration, so onboarding a new microservice's metrics becomes "add a ServiceMonitor," not a manual central-file edit and reload.
Production example: A ServiceMonitor with `matchLabels: { app: checkout-service }` and `labels: { release: prometheus }` gets picked up automatically the moment it's applied.
Common mistake: Forgetting the ServiceMonitor's own label (e.g. `release: prometheus`) must match the Prometheus CR's `serviceMonitorSelector` — otherwise the Operator silently ignores it.
4. Does a ServiceMonitor select Pods directly or Services?
Short answer: A ServiceMonitor's selector matches a Service, not a Pod directly.
Detail: The `endpoints.port` field references a named port on that Service (e.g. "metrics" mapping to containerPort 9100) — the Operator resolves the Service's endpoints to find actual scrape targets.
Production example: A missing or mislabeled Service in front of otherwise-correct Pods means the ServiceMonitor matches nothing, and metrics silently stop being scraped.
Common mistake: Debugging a "missing metrics" issue by checking the Pod's labels only, without checking whether the fronting Service itself has the labels the ServiceMonitor selects on.
5. Why is `rate()` essential when working with counters in PromQL?
Short answer: Raw counters only ever increase, so a raw counter value is meaningless on its own — you need `rate()` over a time window to get a meaningful per-second measure.
Detail: `rate(http_requests_total[5m])` computes the per-second average request rate over the last 5 minutes, correctly handling counter resets (e.g. from a Pod restart) that a naive subtraction would not.
Production example: Graphing `http_requests_total` directly just shows an ever-climbing line; graphing `rate(http_requests_total[5m])` shows the actual traffic pattern.
Common mistake: Alerting or graphing directly on a raw counter value instead of wrapping it in `rate()` first.
6. How do alerting rules relate to Alertmanager, and why is that split deliberate?
Short answer: Prometheus evaluates PromQL expressions as alerting rules and decides "is this condition true"; Alertmanager is a separate component that owns "who should be told, how, and how often."
Detail: This separation means Prometheus stays focused on detection while Alertmanager independently handles deduplication, grouping, silencing, and routing to notification channels like PagerDuty, Slack, or email.
Production example: A `CheckoutHighErrorRate` PrometheusRule fires, and Alertmanager groups it with related alerts before routing a single deduplicated notification to Slack instead of paging once per replica.
Common mistake: Routing every fired alert straight to a human pager without grouping/silencing configured in Alertmanager, causing alert fatigue.
7. What does the `for:` duration on a PrometheusRule accomplish?
Short answer: It requires the alerting expression to stay true for a sustained period (e.g. 5 minutes) before the alert actually fires, avoiding noisy pages on transient blips.
Detail: Without `for:`, a single scrape interval's momentary spike above threshold would immediately fire and notify, even if it self-resolved within seconds.
Production example: `expr: ... > 0.05` combined with `for: 5m` means a brief 10-second error spike doesn't page anyone, but a sustained 5%+ error rate does.
Common mistake: Setting `for: 0s` or omitting it entirely on noisy metrics, resulting in frequent false-positive pages for self-resolving blips.
8. Why is running a single non-HA Prometheus instance a production risk?
Short answer: It becomes a single point of failure for the entire alerting pipeline — if that one instance goes down, both metrics collection and alert evaluation stop simultaneously.
Detail: Unlike a stateless dashboard layer, Prometheus is both the scraper and the rule evaluator, so its unavailability isn't just a visibility gap, it's a silent alerting gap too.
Production example: An incident that coincides with the Prometheus Pod itself crashing means no alert fires for the real incident, compounding the outage.
Common mistake: Assuming Grafana dashboards being down is the only risk of a Prometheus outage, overlooking that alerting itself also silently stops.
9. Why can't Prometheus's local TSDB substitute for long-term metrics storage?
Short answer: Prometheus's local TSDB has retention limits designed for operational, recent-window querying, not indefinite long-term or high-cardinality historical storage.
Detail: Long-term or high-cardinality workloads typically need a remote-write backend (e.g. VictoriaMetrics, Topic 52) that Prometheus forwards samples to, decoupling long-term storage/query scale from the scraping instance itself.
Production example: A team needing 2 years of metrics history for capacity planning configures Prometheus to remote-write into VictoriaMetrics rather than extending local retention indefinitely.
Common mistake: Confusing Prometheus's local retention window with a long-term storage guarantee and losing historical data when local disk-based retention rolls off.
10. Why is label cardinality a specific operational concern for Prometheus?
Short answer: Every unique combination of metric name + label values creates a distinct time series, so unbounded-value labels (raw user IDs, request IDs) can silently blow up storage and query cost.
Detail: Prometheus's TSDB is optimized for a bounded set of label value combinations; high-cardinality labels defeat that assumption and degrade both ingestion and query performance.
Production example: Adding a raw `user_id` label to `http_requests_total` multiplies the metric's series count by the number of distinct users, overwhelming the TSDB within days.
Common mistake: Adding high-cardinality identifiers directly as PromQL labels instead of keeping that granularity in logs or traces where it belongs.
