Custom Resource Definitions (CRDs)

How the API Server learns to serve entirely new resource types without a single line of core Kubernetes code changing.
← Operators Course Index GitOps →

Explanation

Kubernetes ships with built-in resource types — Pod, Deployment, Service — but real platforms need to extend that vocabulary: an Application object for Argo CD, a ServiceMonitor for Prometheus, a PostgresCluster for a database Operator. A CustomResourceDefinition (CRD) is the API object that teaches the API Server a brand-new kind, complete with its own schema, versioning, and REST endpoints — turning "I want a new kind of object" from a fork of Kubernetes source code into a single YAML file.

Kubernetes needs this because its core cannot anticipate every domain's needs, and forcing every extension into core would make the API server bloated, slow to evolve, and impossible to govern safely. CRDs (alongside the Aggregated API Server model, a heavier alternative) are the sanctioned extension mechanism: once a CRD is applied, the API Server dynamically registers a new REST path (e.g. /apis/postgres-operator.example.com/v1/postgresclusters), and from that point on the new kind behaves exactly like any built-in resource — it gets etcd storage, watch/list support, RBAC enforcement, kubectl get/describe/apply, and admission control, all for free.

The schema is defined using OpenAPI v3 validation embedded directly in the CRD (spec.versions[].schema.openAPIV3Schema) — required fields, types, enums, min/max values. The API Server validates every write against this schema at admission time, rejecting malformed custom resources before they're ever persisted — the same guarantee built-in types get from their compiled-in Go structs, but declared as data instead of code.

CRDs support multiple versions simultaneously (e.g. v1alpha1, v1beta1, v1), which matters because a CRD's schema evolves over time just like any API. When multiple versions coexist, exactly one is marked the storage version (what's actually persisted in etcd), and a conversion webhook handles translating objects between versions on read/write when the schemas differ meaningfully — this is the same version-skew problem core Kubernetes APIs solve internally, exposed to CRD authors as an explicit extension point.

A CRD alone only gives you storage and validation — it does nothing on its own. The "brains" that make a custom resource meaningful come from a controller/Operator watching it and reconciling real-world state (see Topic 46). A CRD with no controller watching it is just structured data sitting inertly in etcd; the pairing of "CRD defines the shape" + "controller defines the behavior" is the entire extension model.

Internal Working / Interactions

  • API Server: reads the CRD object at startup/apply time and dynamically registers a new REST endpoint + OpenAPI schema for the new kind — no restart, no compiled code.
  • etcd: stores instances of the custom resource exactly like any built-in object, keyed under the CRD's group/version/resource path.
  • Admission control: the API Server's built-in OpenAPI-schema validation runs on every custom-resource write; optional conversion webhooks run when multiple CRD versions are served.
  • RBAC: Roles/ClusterRoles reference the CRD's apiGroups/resources exactly like built-in types — custom resources are first-class RBAC citizens.
  • Controllers/Operators: watch the new kind via the same informer/watch mechanism used for Pods or Deployments — from client-go's perspective, a CRD-backed object is indistinguishable from a built-in one.

Code Example

# CustomResourceDefinition — teaches the API Server about "PostgresCluster"
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: postgresclusters.postgres-operator.example.com   # Must be .
spec:
  group: postgres-operator.example.com
  names:
    kind: PostgresCluster
    plural: postgresclusters
    singular: postgrescluster
    shortNames: ["pgc"]                # Enables `kubectl get pgc`
  scope: Namespaced
  versions:
    - name: v1
      served: true                     # This version is reachable via the API.
      storage: true                    # This is the version actually persisted in etcd.
      schema:
        openAPIV3Schema:                # OpenAPI v3 validation — enforced at admission time.
          type: object
          properties:
            spec:
              type: object
              required: ["version", "instances"]
              properties:
                version:
                  type: string
                  enum: ["13", "14", "15", "16"]   # Rejects unsupported Postgres versions up front.
                instances:
                  type: integer
                  minimum: 1
                  maximum: 9
                storage:
                  type: object
                  properties:
                    size: { type: string, pattern: "^[0-9]+Gi$" }
            status:
              type: object
              properties:
                phase: { type: string }
                currentPrimary: { type: string }
      subresources:
        status: {}                     # Enables the separate /status subresource, same pattern as built-in types.

kubectl Commands

kubectl apply -f postgrescluster-crd.yaml           # Register the new CRD
kubectl get crd | grep postgres-operator.example.com  # Confirm the CRD is established
kubectl explain postgrescluster.spec                  # API Server serves live schema, just like built-in types
kubectl get postgresclusters -A                        # List custom resources across all namespaces
kubectl get pgc orders-db -n payments -o yaml           # Full object incl. status subresource
kubectl api-resources | grep postgres                   # Confirm the new REST resource is registered
kubectl delete crd postgresclusters.postgres-operator.example.com   # Removes the type AND all its instances — dangerous

Best Practices & Common Pitfalls

Best Practices

  • Always define a strict openAPIV3Schema with required fields and enums — don't leave it permissive.
  • Use the status subresource so RBAC can separate who can write spec vs. who (the controller) can write status.
  • Version CRDs deliberately (v1alpha1 → v1beta1 → v1) and provide conversion webhooks before dropping an old version.
  • Document custom resources like any public API — consumers (other teams) will treat the CRD as a contract.

Pitfalls

  • Deleting a CRD deletes every instance of that custom resource cluster-wide — treat CRD deletion as destructive as dropping a database table.
  • Shipping a CRD with no controller watching it, leaving custom resources as inert, unenforced data.
  • Skipping schema validation ("just accept anything") and pushing validation entirely into the controller — admission-time rejection is cheaper and safer.
  • Forgetting CRDs are cluster-scoped objects themselves even when the custom resources they define are namespaced — plan RBAC for who can install/modify CRDs.

50 Interview Questions

1. What does applying a CustomResourceDefinition actually do to the API Server?
Short answer: It dynamically registers a brand-new REST endpoint and OpenAPI schema for the new kind, with no restart or compiled code needed.
Detail: Once established, the new kind behaves exactly like any built-in resource — etcd storage, watch/list support, RBAC enforcement, and kubectl get/describe/apply all work for free.
Production example: Applying a PostgresCluster CRD instantly makes /apis/postgres-operator.example.com/v1/postgresclusters a live, functioning REST path.
Common mistake: Assuming extending the API requires forking or recompiling Kubernetes source code.
2. How does the API Server validate custom resource writes without compiled Go structs?
Short answer: Via OpenAPI v3 validation embedded directly in the CRD's spec.versions[].schema.openAPIV3Schema field.
Detail: Required fields, types, enums, and min/max values are declared as data instead of code, and the API Server enforces them at admission time before persisting to etcd — the same guarantee built-in types get, but declarative.
Production example: A version field with enum ["13","14","15","16"] rejecting any PostgresCluster spec requesting an unsupported Postgres version.
Common mistake: Leaving the schema permissive ("just accept anything") and pushing all validation into the controller instead — admission-time rejection is cheaper and safer.
3. What happens when a CRD serves multiple versions simultaneously?
Short answer: Exactly one version is marked the storage version (what's actually persisted); a conversion webhook translates between versions on read/write when schemas differ meaningfully.
Detail: This mirrors the version-skew problem core Kubernetes APIs solve internally, exposed to CRD authors as an explicit extension point they must implement themselves for non-trivial schema changes.
Production example: A CRD serving both v1beta1 and v1, with v1 as the storage version and a conversion webhook translating v1beta1 requests transparently.
Common mistake: Dropping an old served version before providing a conversion webhook, breaking clients still using that version.
4. What's the difference between "served" and "storage" in a CRD version's spec?
Short answer: served means the version is reachable via the API; storage means it's the one actually persisted in etcd — only one version can be storage: true at a time.
Detail: Multiple versions can be served simultaneously for compatibility, but etcd only ever holds objects in the single designated storage version's shape, converting on the fly for other served versions.
Production example: v1 marked storage: true while v1alpha1 remains served: true for backward compatibility during a migration window.
Common mistake: Marking multiple versions as storage: true, which is invalid — only one is allowed.
5. Why does deleting a CRD count as one of the most destructive operations in a cluster?
Short answer: It deletes every instance of that custom resource cluster-wide, not just the CRD's own schema definition.
Detail: This is comparable to dropping a database table — all data stored under that type across every namespace disappears along with the type itself.
Production example: Accidentally deleting the PostgresCluster CRD wiping out every PostgresCluster object cluster-wide, even though the underlying StatefulSets/PVCs might survive depending on ownerReference/finalizer behavior.
Common mistake: Treating `kubectl delete crd` as a routine cleanup action instead of a destructive, cluster-wide operation requiring extreme caution.
6. What's the purpose of the status subresource on a CRD?
Short answer: It splits spec and status into separately addressable endpoints, letting RBAC distinguish who can write spec versus who (typically only the controller) can write status.
Detail: This is the same pattern built-in resources like Deployment use — users declare desired state via spec, controllers report observed state via status, and RBAC can enforce that separation cleanly.
Production example: A user's RoleBinding allows updating a PostgresCluster's spec, but only the Operator's ServiceAccount can write to its status subresource.
Common mistake: Omitting the status subresource, which means any client with general write access to the object can also freely overwrite status fields meant to be controller-owned.
7. Why is a CRD alone insufficient to make a custom resource "do" anything?
Short answer: A CRD only provides storage, schema validation, and REST endpoints — it has no behavior; a controller/Operator watching it is what reconciles real-world state.
Detail: Without a controller, instances of the custom resource are just structured data sitting inertly in etcd — the pairing of "CRD defines the shape" plus "controller defines the behavior" is the entire extension model.
Production example: Applying a CRD for a new kind with no Operator deployed yet results in objects that can be created/listed but never actually provision anything.
Common mistake: Shipping a CRD to a team before the corresponding controller is deployed and running, leading to confusion about why "nothing happens."
8. Are CRDs themselves namespaced or cluster-scoped, even when they define namespaced resources?
Short answer: CRDs are always cluster-scoped objects, even if the custom resource type they define has scope: Namespaced.
Detail: This distinction matters for RBAC planning — anyone who can create/modify CRDs affects the whole cluster's API surface, regardless of whether the resulting resource instances are namespace-scoped.
Production example: Restricting `create`/`update`/`delete` on CustomResourceDefinition objects to platform-team RBAC only, while individual PostgresCluster instances remain manageable by app teams within their namespace.
Common mistake: Assuming a namespaced custom resource type implies the CRD object itself is also namespace-scoped.
9. How does RBAC treat custom resources compared to built-in types like Pods?
Short answer: Identically — Roles/ClusterRoles reference the CRD's apiGroups/resources exactly as they would core or built-in extension types.
Detail: Custom resources are first-class RBAC citizens; there's no special-casing required to grant or restrict access to them versus any core Kubernetes type.
Production example: A Role granting get/list/watch on resources: ["postgresclusters"] under apiGroups: ["postgres-operator.example.com"].
Common mistake: Assuming custom resources need a different or special RBAC mechanism instead of the standard Role/RoleBinding pattern.
10. What's the alternative extension mechanism to CRDs, and why are CRDs preferred for most cases?
Short answer: The Aggregated API Server model — running your own API server that Kubernetes' API Server proxies to — is heavier and reserved for cases needing custom storage backends or non-CRUD semantics.
Detail: CRDs are far simpler to author (a single YAML file, no need to run and maintain a separate API server binary) and cover the vast majority of extension needs; aggregated API servers are used only when CRDs' constraints (e.g. etcd-backed storage, standard REST semantics) don't fit.
Production example: The Kubernetes metrics-server uses an aggregated API server because it serves live, non-persisted metric data rather than stored objects — a poor fit for CRDs.
Common mistake: Reaching for an aggregated API server when a standard CRD would have been sufficient and far simpler to maintain.
11–50. Additional rapid-fire questions
11. What must a CRD's metadata.name match? — <plural>.<group>.
12. What field enables `kubectl get pgc` as a shorthand? — spec.names.shortNames.
13. What kubectl command shows live schema for a custom resource, just like built-in types? — kubectl explain <kind>.spec.
14. What confirms a CRD's new REST resource is registered? — kubectl api-resources | grep <name>.
15. What confirms a CRD is fully established after applying it? — kubectl get crd | grep <group>.
16. Does client-go/informers treat CRD-backed objects differently from built-in ones? — No, they're indistinguishable from a client's perspective — same watch/informer mechanism.
17. What API group manages CRD objects themselves? — apiextensions.k8s.io.
18. Can a CRD define a resource with scope: Cluster instead of Namespaced? — Yes, custom resources can be cluster-scoped just like Node or PersistentVolume.
19. What happens to admission control for writes to a custom resource? — Built-in OpenAPI-schema validation always runs; optional conversion webhooks run additionally when multiple versions are served.
20. Is a conversion webhook required if a CRD only ever serves one version? — No, it's only needed when multiple versions with differing schemas coexist.
21. What's the risk of a CRD schema with no required fields at all? — Nearly anything gets accepted, pushing all real validation downstream into the controller, losing admission-time safety.
22. Can you add new optional fields to a CRD schema without a version bump? — Yes, additive optional fields are typically safe to add to an existing version without bumping it.
23. What's a risky schema change that does require careful version handling? — Removing or renaming a previously required field, which can break existing stored objects or clients.
24. How does a CRD's subresources.status: {} setting affect PATCH behavior? — Updates to spec and status become separately addressable, each via their own subresource endpoint.
25. Who typically writes to a CR's status field in practice? — The controller/Operator reconciling that CR, not end users.
26. What's the analogy for "CRD defines shape, controller defines behavior"? — Like a class definition (schema) versus the running program instance that acts on it (controller).
27. Can a single CRD support both v1alpha1 and v1 being served at once? — Yes, multiple served versions can coexist during a migration window.
28. What's a practical reason to keep an old CRD version served temporarily? — Giving existing clients/tooling time to migrate to the new version before it's removed.
29. Does deleting a CRD affect any owned native resources (like StatefulSets) that a controller created? — Only indirectly — those resources may become orphaned or garbage-collected depending on ownerReferences and finalizers, but they aren't defined by the CRD itself.
30. What's the purpose of a validating admission webhook alongside a CRD's built-in schema validation? — Enforcing business rules beyond what static OpenAPI schema can express (e.g. cross-field consistency checks).
31. Is a CRD's OpenAPI schema enforced only on create, or also on update? — Both — every write (create and update) is validated against the schema at admission time.
32. What's a good practice for documenting a custom resource? — Treat it like a public API contract since other teams will consume it, documenting fields and expected behavior.
33. Can RBAC restrict who can install/modify CRDs separately from who can create instances of them? — Yes, since CRDs are cluster-scoped objects with their own resource type (customresourcedefinitions) distinct from the custom resource instances.
34. What's the typical progression pattern for CRD versions? — v1alpha1 → v1beta1 → v1, with conversion webhooks bridging as needed before dropping old versions.
35. Does a CRD support default values for fields not explicitly set? — Yes, OpenAPI v3 schemas can specify defaults applied at admission time.
36. What's a common pitfall with enum fields in a CRD schema? — Forgetting to add a new valid value when the underlying software adds support for it, causing valid configs to be rejected.
37. How does a CRD interact with `kubectl get -o yaml` output compared to built-in types? — Identical behavior — full object including status is returned the same way.
38. What's the effect of pattern validation (e.g. regex) in an OpenAPI v3 schema field? — Rejects string values that don't match the given pattern at admission time, e.g. enforcing a storage size format like "100Gi".
39. Can a CRD schema enforce cross-field relationships (e.g. "field A required only if field B is set")? — Limited support via newer OpenAPI features (like CEL validation rules in recent Kubernetes versions); classic schema validation alone is more limited.
40. What's the risk of skipping schema validation entirely on a CRD? — Malformed custom resources get persisted, and the controller may fail unpredictably trying to act on invalid data.
41. Does a CRD need a controller running at the moment it's applied? — No, the CRD registers the schema/API immediately; the controller can be deployed separately, before or after.
42. What's a "finalizer" in the context of a CR managed by a controller? — A marker blocking deletion from completing until the controller finishes cleanup of external/owned resources.
43. Why might a CR appear "stuck deleting"? — A finalizer set by its controller hasn't been removed yet, often because the controller is down or failing cleanup.
44. Can two different CRDs use the same group but different resource names? — Yes, a single API group can host multiple distinct custom resource kinds.
45. What's the relationship between a CRD's `kind` and `plural` fields? — kind is the singular, capitalized Go-style type name; plural is the lowercase name used in REST paths and kubectl commands.
46. Is etcd storage format for custom resources different from built-in types? — No, custom resources are stored under their own group/version/resource key path exactly like built-in objects.
47. What's a good CI practice for CRD schema changes? — Validate new schemas against sample manifests and run conversion webhook tests before rolling out to production clusters.
48. Can a custom resource type be queried across all namespaces like built-in namespaced resources? — Yes, `kubectl get <kind> -A` works identically to built-in resources.
49. What's the danger of a controller having overly permissive RBAC on a CRD's resources? — It can read/write far more than its intended scope, increasing blast radius if the controller is compromised or buggy.
50. What's the first thing to check when `kubectl apply` on a custom resource fails with a schema error? — Compare the submitted YAML against the CRD's openAPIV3Schema for missing required fields, wrong types, or invalid enum values.
