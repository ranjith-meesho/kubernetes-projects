Fluent Bit

A lightweight, high-performance log forwarder built for exactly the node-level DaemonSet pattern described in the logging topic.
← Logging Course Index Loki →

Explanation

Fluent Bit is a lightweight (written in C, small memory footprint, sub-second startup) log and metrics forwarder from the Fluentd/CNCF ecosystem, purpose-built to run as the node-level agent described in the Logging topic. Where Fluentd (its heavier Ruby-based predecessor) is a full-featured aggregator often run centrally, Fluent Bit is optimized to run on every node as a DaemonSet with minimal CPU/memory overhead — a critical property, since a log agent that consumes significant resources on every node in a large cluster becomes a tax on the entire fleet.

Kubernetes needs a tool like Fluent Bit because raw log files on disk (as covered in the Logging topic) are not useful until they're parsed, enriched with Kubernetes metadata, and routed to a durable backend. Fluent Bit does all three: it tails the node's container log files, parses them (JSON or plain text with configurable parsers), calls the Kubernetes API to attach pod/namespace/label metadata to each line (so a log line in Loki/Elasticsearch is filterable by namespace="checkout" without any application code change), and forwards the result to one or more outputs simultaneously.

Fluent Bit's configuration model is built around three plugin types chained into a pipeline: Input plugins (most commonly tail, reading log files matching a glob) produce records; Filter plugins transform records in-flight — the kubernetes filter is the most important one in this context, enriching each record with pod metadata by querying the Kubernetes API and caching results; and Output plugins deliver the final records to a destination — Loki, Elasticsearch, an OpenTelemetry Collector's OTLP logs receiver, S3, Kafka, or many others. This plugin architecture is why Fluent Bit fits equally well feeding Loki (Topic 57) or feeding an OTel Collector pipeline (Topic 54) — it's a transport and enrichment layer that's backend-agnostic.

In production, Fluent Bit typically runs with a small, tuned configuration: Mem_Buf_Limit to cap memory if the output is slow or unreachable, Skip_Long_Lines On to avoid one giant log line blocking the pipeline, and multiple [OUTPUT] blocks with Match patterns routing different namespaces/labels to different destinations (e.g. security-relevant namespaces to a separate, longer-retention index).

Within the broader ecosystem, Fluent Bit is one of two dominant choices for the "collector" role alongside Promtail (Loki's own purpose-built agent) and the OpenTelemetry Collector's filelog receiver; many teams standardize on Fluent Bit specifically because of its plugin breadth (it can output to almost anything) and its proven low resource footprint at very large node counts.

Internal Working / Interactions

  • Input (tail) — reads /var/log/containers/*.log, tracking file offsets in a local SQLite/flat-file database so it resumes correctly after a restart without re-reading or losing lines.
  • Kubernetes filter — calls the Kubernetes API server (via a ServiceAccount token) to look up pod metadata by parsing the log file's path/tag, caching results locally to avoid hammering the API server.
  • Parser — converts raw text/JSON into structured fields (e.g. docker or cri parser depending on runtime log format).
  • Output plugins — deliver to Loki (via the loki output), Elasticsearch, or an OTel Collector (via opentelemetry/forward output), often with buffering and retry on failure.
  • Runs as a DaemonSet so exactly one instance per node handles that node's own log volume — horizontally scales automatically as nodes are added.

Code Example

# Fluent Bit ConfigMap — classic DaemonSet config: tail container logs, enrich with
# Kubernetes metadata, and ship to Loki.
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
  namespace: logging
data:
  fluent-bit.conf: |
    [SERVICE]
        Flush         5                 # flush buffered records every 5s
        Log_Level     info
        Parsers_File  parsers.conf

    [INPUT]
        Name              tail
        Path              /var/log/containers/*.log   # matches the symlinked per-container log files
        Tag               kube.*
        Parser            cri                          # matches containerd's CRI log line format
        Mem_Buf_Limit     15MB                          # caps memory if downstream output stalls
        Skip_Long_Lines   On                            # don't block the pipeline on one oversized line
        Refresh_Interval  10

    [FILTER]
        Name                kubernetes
        Match               kube.*
        Kube_URL            https://kubernetes.default.svc:443   # talks to the API server for metadata
        Kube_CA_File        /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
        Kube_Token_File     /var/run/secrets/kubernetes.io/serviceaccount/token
        Merge_Log           On                          # parse JSON app logs into structured fields
        K8S-Logging.Parser  On

    [OUTPUT]
        Name            loki
        Match           kube.*
        Host            loki.logging.svc
        Port            3100
        Labels          job=fluentbit, namespace=$kubernetes['namespace_name'], pod=$kubernetes['pod_name']
        Line_Format     json
---
# DaemonSet running Fluent Bit with the config above mounted, and read-only access to node log dirs.
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
spec:
  selector:
    matchLabels: { app: fluent-bit }
  template:
    metadata:
      labels: { app: fluent-bit }
    spec:
      serviceAccountName: fluent-bit           # RBAC: get/list/watch on pods (for the kubernetes filter)
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:3.0
          resources:
            requests: { cpu: "50m", memory: "64Mi" }   # deliberately tiny — this is per-node overhead
            limits:   { cpu: "200m", memory: "128Mi" }
          volumeMounts:
            - name: varlog
              mountPath: /var/log
              readOnly: true
            - name: config
              mountPath: /fluent-bit/etc
      volumes:
        - name: varlog
          hostPath: { path: /var/log }
        - name: config
          configMap: { name: fluent-bit-config }

kubectl / CLI Commands

kubectl get pods -n logging -l app=fluent-bit -o wide       # confirm one pod per node, all Running
kubectl logs -n logging fluent-bit-xxxxx --tail=100          # check for parse/output errors
kubectl exec -n logging fluent-bit-xxxxx -- cat /fluent-bit/etc/fluent-bit.conf   # verify loaded config
kubectl top pod -n logging -l app=fluent-bit                  # confirm resource usage stays within limits
kubectl describe ds fluent-bit -n logging                     # check rollout status across all nodes
kubectl get sa fluent-bit -n logging -o yaml                  # verify RBAC ServiceAccount is bound correctly
kubectl rollout restart ds/fluent-bit -n logging               # reload config after a ConfigMap change

Best Practices & Common Pitfalls

Best Practices

  • Set tight, explicit resource requests/limits — Fluent Bit should be a lightweight per-node tax, not a heavyweight pod.
  • Use Mem_Buf_Limit and storage.type filesystem for buffering to survive backend hiccups without dropping logs.
  • Cache Kubernetes API lookups (the filter does this by default) to avoid overloading the API server across hundreds of nodes.
  • Route different namespaces/log types to different outputs using Match tags rather than one firehose destination.
  • Pin parser choice (cri vs docker) to match your actual container runtime's log format.

Pitfalls

  • Forgetting Skip_Long_Lines On, letting one massive stack-trace log line stall the whole pipeline.
  • Not setting Mem_Buf_Limit, causing OOM on the DaemonSet pod during an output outage.
  • Granting the ServiceAccount cluster-admin instead of the minimal get/list/watch on pods needed for enrichment.
  • Running conflicting Fluent Bit and Promtail DaemonSets simultaneously, double-shipping the same logs.

50 Interview Questions

1. Why is Fluent Bit written in C instead of a higher-level language like Fluentd's Ruby?
Short answer: To achieve a small memory footprint and sub-second startup as a per-node agent.
Detail: Since Fluent Bit runs on every node in the cluster, any per-instance overhead is multiplied by node count — C gives it the low CPU/memory profile needed to avoid taxing the whole fleet.
Production example: A 500-node cluster running Fluent Bit at ~64Mi/50m requests versus a heavier JVM/Ruby agent that could cost gigabytes of aggregate memory across the fleet.
Common mistake: Assuming any log agent's resource cost is negligible without checking it against total node count.
2. What are the three plugin types in a Fluent Bit pipeline, and what does each do?
Short answer: Input, Filter, and Output plugins.
Detail: Input plugins (e.g. tail) produce records; Filter plugins transform records in-flight (e.g. the kubernetes filter enriches with metadata); Output plugins deliver final records to a destination.
Production example: tail reads /var/log/containers/*.log, the kubernetes filter attaches namespace/pod labels, and the loki output ships the enriched record.
Common mistake: Configuring an Output without a matching Match tag against the right Input/Filter chain, silently dropping records.
3. Why is the Kubernetes filter considered the most important filter in this context?
Short answer: It attaches pod/namespace/label metadata to raw log lines, making them filterable without any application code change.
Detail: It queries the Kubernetes API server to resolve metadata for each record based on the log file path/tag, then caches results to avoid overloading the API server.
Production example: A log line in Loki becomes filterable by namespace="checkout" purely from container path parsing, with no app-side logging changes.
Common mistake: Disabling or misconfiguring the kubernetes filter and losing the ability to filter logs by namespace/pod in the backend.
4. Why does Fluent Bit run as a DaemonSet rather than a Deployment?
Short answer: Exactly one instance per node is needed to handle that node's own log volume.
Detail: A DaemonSet automatically scales horizontally as nodes are added or removed, matching the node-local nature of log tailing.
Production example: Adding worker nodes to a cluster automatically schedules a new Fluent Bit pod on each, with no manual intervention.
Common mistake: Running it as a Deployment with replicas, which wouldn't guarantee coverage of every node's log files.
5. How does the tail input plugin avoid re-reading or losing lines after a restart?
Short answer: It tracks file offsets in a local SQLite/flat-file database.
Detail: On restart, the tail plugin resumes reading from the last recorded offset for each tracked file instead of starting from the beginning or losing new lines written in the gap.
Production example: A Fluent Bit pod restart during a node reboot resumes tailing exactly where it left off, with no duplicate or missing log lines.
Common mistake: Wiping the offset database/volume on every restart, causing either duplicate re-ingestion or lost logs.
6. What is the purpose of Mem_Buf_Limit?
Short answer: Caps memory usage if the output destination is slow or unreachable.
Detail: Without a buffer cap, records accumulate in memory unboundedly during an output outage, risking OOM on the DaemonSet pod.
Production example: A Loki outage causes buffered records to hit the 15MB cap rather than growing until the pod is OOMKilled.
Common mistake: Not setting Mem_Buf_Limit, which is explicitly called out as a pitfall leading to OOM during output outages.
7. What does Skip_Long_Lines On protect against?
Short answer: Prevents one oversized log line (e.g. a giant stack trace) from stalling the whole pipeline.
Detail: Without it, a single line exceeding internal buffer expectations can block processing of all subsequent records on that input.
Production example: A Java service emitting a multi-KB stack trace as one line doesn't halt log shipping for the rest of the node's containers.
Common mistake: Forgetting this setting, explicitly listed as a pitfall that stalls the pipeline on a single massive line.
8. Why does the Kubernetes filter cache its API lookups?
Short answer: To avoid overloading the Kubernetes API server across potentially hundreds of nodes each running a Fluent Bit instance.
Detail: Since every node's Fluent Bit needs metadata for its local pods, unrestrained per-record API calls would multiply into significant API server load cluster-wide.
Production example: A large cluster with hundreds of nodes relies on this caching to keep the enrichment filter from becoming an API server bottleneck.
Common mistake: Assuming metadata lookups are free and not accounting for their cumulative cost at scale.
9. How does routing to multiple outputs work in Fluent Bit, and why is it used?
Short answer: Multiple [OUTPUT] blocks with Match patterns route different namespaces/labels to different destinations.
Detail: This lets teams send security-relevant namespaces to a separate, longer-retention index while routing general application logs elsewhere (e.g. Loki).
Production example: Application logs go to Loki while audit/security-relevant logs are routed to Elasticsearch with longer retention.
Common mistake: Routing everything through one firehose destination instead of differentiating by Match tags, called out explicitly as a best practice violation.
10. Why must the parser choice (cri vs docker) match the container runtime?
Short answer: Each runtime writes log lines in a different on-disk format, so the wrong parser will fail to correctly structure fields.
Detail: The cri parser matches containerd's CRI log line format, while a docker-formatted log requires the docker parser instead.
Production example: A cluster running containerd uses Parser cri in the INPUT block to correctly split timestamp/stream/message fields.
Common mistake: Copy-pasting a config from a Docker-runtime cluster onto a containerd cluster without updating the parser, producing garbled structured fields.
