Pod Anti-Affinity

Spreading replicas apart so a single node or zone failure never takes down your whole service.
← Affinity Course Index Taints →

Explanation

Pod anti-affinity is a scheduling rule that tells the Scheduler to avoid placing a Pod on a node that already runs (or doesn't run) Pods matching a given label selector, evaluated relative to a topology domain such as kubernetes.io/hostname or topology.kubernetes.io/zone. Where affinity pulls Pods together, anti-affinity pushes them apart. It is the single most important primitive for turning "3 replicas" into "3 replicas that actually survive a node failure" — without it, the Scheduler is free to bin-pack all three replicas of a Deployment onto the same node purely because that node happened to have the most free capacity at the time.

Kubernetes needs this because the default scheduling objective is efficient bin-packing, not availability. Bin-packing and high availability are often in direct tension: the most "efficient" placement for 3 replicas is frequently all on one node, which is exactly the placement that turns a single node failure into a full outage. Anti-affinity lets you explicitly encode the availability requirement as a scheduling constraint instead of hoping the scorer happens to spread things out.

There are two flavors, mirroring node affinity: requiredDuringSchedulingIgnoredDuringExecution (a hard rule — if no node satisfies it, the Pod stays Pending) and preferredDuringSchedulingIgnoredDuringExecution (a soft rule — the Scheduler tries to honor it via scoring, but will still place the Pod if no ideal node exists). The "IgnoredDuringExecution" suffix means that once a Pod is running, Kubernetes never evicts it just because the anti-affinity rule is later violated (e.g. another Pod is later scheduled that breaks the rule) — it only affects the initial placement decision.

Anti-affinity fits into the wider architecture at the Scheduler's filtering and scoring phases, exactly like node affinity and taints/tolerations. It reads PodAffinityTerm objects containing a labelSelector, a topologyKey, and optionally a namespaces list, and the Scheduler must query the current placement of every other Pod matching that selector across the cluster before it can decide — this makes pod anti-affinity computationally more expensive than node affinity, which is one reason large clusters are cautioned against overusing "required" pod anti-affinity at scale.

In production, anti-affinity is what makes topology-aware statements like "no two replicas of payment-service on the same node" or "no two replicas in the same availability zone" actually enforceable, rather than best-effort. It composes naturally with topologySpreadConstraints (topic 35), which is the more modern, flexible tool for the same general goal — many teams today prefer topology spread constraints for even distribution and reserve anti-affinity for strict "never co-locate these two workloads" rules (e.g. keep an app and its adjacent cache off the same node).

Anti-affinity vs topology spread constraints: Anti-affinity answers "keep me away from Pods with label X." Topology spread constraints answer "spread my Pods evenly across a topology domain." They can be combined, but if all you want is even distribution of one Deployment's own replicas, topology spread constraints (topic 35) are usually the cleaner, more scalable choice.

Internal Working / Interactions

  • API Server: validates the affinity.podAntiAffinity block against the Pod spec schema and persists it to etcd unchanged — the Scheduler reads it back on the next scheduling attempt.
  • Scheduler: the InterPodAffinity plugin runs during the Filter phase (for required rules) and the Score phase (for preferred rules). It must list existing Pods across the relevant topology domain to evaluate the selector — this is a live query against the Scheduler's Pod cache (informer), not etcd directly.
  • kubelet: has no role in anti-affinity — by the time a Pod reaches a node, the placement decision is already final.
  • etcd: stores the anti-affinity rule as part of the Pod object; no separate storage.
  • Cluster Autoscaler: must simulate anti-affinity rules when deciding whether adding a new node would actually let a Pending Pod schedule — a common source of "cluster autoscaler added a node but the pod is still Pending" confusion.

Code Example

apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service
  namespace: payments
spec:
  replicas: 3
  selector:
    matchLabels:
      app: checkout
  template:
    metadata:
      labels:
        app: checkout
    spec:
      affinity:
        podAntiAffinity:
          # HARD rule: never place two "app=checkout" pods on the same node.
          # If no node can satisfy this, new pods stay Pending instead of co-locating.
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchExpressions:
                  - key: app
                    operator: In
                    values: ["checkout"]
              topologyKey: "kubernetes.io/hostname"   # domain = individual node
      containers:
        - name: checkout
          image: registry.internal/checkout-service:2.4.1
          resources:
            requests: { cpu: "250m", memory: "256Mi" }

apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service-multi-az
  namespace: payments
spec:
  replicas: 6
  selector:
    matchLabels:
      app: checkout
  template:
    metadata:
      labels:
        app: checkout
    spec:
      affinity:
        podAntiAffinity:
          # SOFT rule: prefer spreading across zones, but don't block scheduling
          # if capacity forces some replicas to share a zone.
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100                              # relative weight vs other preferred terms
              podAffinityTerm:
                labelSelector:
                  matchLabels:
                    app: checkout
                topologyKey: "topology.kubernetes.io/zone"   # domain = availability zone
          # Combine with a hard per-node rule so no single node ever gets 2 replicas.
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchLabels:
                  app: checkout
              topologyKey: "kubernetes.io/hostname"
      containers:
        - name: checkout
          image: registry.internal/checkout-service:2.4.1

kubectl Commands

# See which node each replica landed on — quick visual anti-affinity check
kubectl get pods -l app=checkout -o wide

# Confirm no two replicas share a node
kubectl get pods -l app=checkout -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName

# Inspect why a pod is Pending (anti-affinity conflicts show up here)
kubectl describe pod checkout-abcde | grep -A5 Events

# View the raw affinity rule on a running pod
kubectl get pod checkout-abcde -o jsonpath='{.spec.affinity.podAntiAffinity}'

# Check node zone labels used as the topology domain
kubectl get nodes -L topology.kubernetes.io/zone

# Dry-run validate a manifest's affinity block against the live schema
kubectl explain deployment.spec.template.spec.affinity.podAntiAffinity

Best Practices & Common Pitfalls

Do's

  • Use hard (required) anti-affinity by hostname for critical, low-replica-count services (databases, leader-elected controllers).
  • Prefer soft (preferred) anti-affinity by zone for larger fleets where strict enforcement could cause Pending pods.
  • Combine with topologySpreadConstraints when you need both "spread evenly" and "never co-locate with X".
  • Always pair required anti-affinity with enough node/zone capacity — verify with cluster-autoscaler dry-run before rollout.

Don'ts

  • Don't use required anti-affinity on high-replica-count Deployments in small clusters — pods will get stuck Pending once nodes run out.
  • Don't forget podAffinityTerm evaluation cost scales with cluster size — avoid required inter-pod anti-affinity at very large scale (1000s of nodes) without testing scheduler latency.
  • Don't rely on anti-affinity alone for rebalancing after scale-up — it only affects initial placement, not existing running pods.
  • Don't mix labelSelectors so broad they accidentally match unrelated pods across namespaces (remember to scope with namespaces field).

50 Interview Questions

1. What is pod anti-affinity and how does it differ from node affinity?
Short answer: Anti-affinity repels a Pod from nodes already running Pods matching a label selector, relative to a topology domain; node affinity attracts a Pod toward nodes matching node-level labels.
Detailed explanation: Node affinity compares a Pod's rules against static node labels. Pod anti-affinity compares against the live placement of *other Pods*, which the Scheduler must query dynamically via its Pod cache — a fundamentally more expensive operation.
Production example: Spreading 3 replicas of checkout-service across 3 different nodes using podAntiAffinity with topologyKey: kubernetes.io/hostname.
Common mistake: Assuming anti-affinity reads node labels — it reads other Pods' labels and positions instead.
2. What do requiredDuringSchedulingIgnoredDuringExecution and preferredDuringSchedulingIgnoredDuringExecution mean?
Short answer: "Required" is a hard constraint — the Pod stays Pending if unmet; "preferred" is a soft scoring hint that never blocks scheduling.
Detailed explanation: "IgnoredDuringExecution" means once a Pod is running, Kubernetes never evicts it even if the anti-affinity rule is later violated by a subsequent scheduling decision — it only governs initial placement.
Production example: A payments team uses required hostname anti-affinity for a 3-replica database and preferred zone anti-affinity for a 20-replica stateless API.
Common mistake: Expecting a required rule to rebalance already-running Pods after the rule is violated — it never does.
3. Why is required pod anti-affinity more expensive for the Scheduler than node affinity or taints?
Short answer: It requires listing every existing Pod matching the label selector across the topology domain before a placement decision can be made.
Detailed explanation: Node affinity/taints only compare against static node metadata already cached; inter-pod affinity/anti-affinity forces the InterPodAffinity plugin to query live Pod placement, which scales with cluster and Pod count.
Production example: Large clusters (1000s of nodes) are cautioned against heavy use of required pod anti-affinity because of Scheduler latency impact.
Common mistake: Applying required pod anti-affinity broadly across hundreds of Deployments without load-testing scheduler throughput.
4. What is the role of topologyKey in an anti-affinity rule?
Short answer: It defines the domain across which the anti-affinity rule is evaluated — e.g. per-node or per-zone.
Detailed explanation: kubernetes.io/hostname means "never share a node"; topology.kubernetes.io/zone means "never share a zone." The Scheduler groups nodes by this label's value to compute matches.
Production example: checkout-service uses hostname-level required anti-affinity plus zone-level preferred anti-affinity, layering two domains.
Common mistake: Choosing a topologyKey that isn't consistently applied to all nodes, silently breaking the rule for unlabeled nodes.
5. Why would the Scheduler bin-pack all 3 replicas of a Deployment onto one node without anti-affinity?
Short answer: Because the default scheduling objective optimizes for efficient resource packing, not high availability.
Detailed explanation: The scorer picks the "best fit" node by capacity criteria; nothing about that objective inherently spreads replicas of the same workload apart.
Production example: A 3-replica Deployment landed entirely on one large, mostly-empty node until anti-affinity was added, and a single node failure took the whole service down.
Common mistake: Assuming "3 replicas" automatically implies "3 different nodes" — it does not without an explicit constraint.
6. How does anti-affinity interact with Cluster Autoscaler?
Short answer: Cluster Autoscaler must simulate anti-affinity rules when deciding if a new node would actually let a Pending Pod schedule.
Detailed explanation: If it doesn't correctly account for required anti-affinity, it may add a node that still can't satisfy the rule, leaving the Pod Pending even after scale-up.
Production example: Cluster Autoscaler added a node in an already-occupied zone, but the Pod's required hostname anti-affinity still blocked it because another replica already sat there.
Common mistake: Assuming "Cluster Autoscaler added a node" always resolves a Pending Pod — anti-affinity mismatches are a common exception.
7. When should you prefer topology spread constraints over pod anti-affinity?
Short answer: When the goal is even distribution of one Deployment's own replicas rather than avoiding co-location with a *different* workload.
Detailed explanation: Anti-affinity is binary (co-locate or not); topology spread constraints reason quantitatively via maxSkew and scale better for large replica counts.
Production example: checkout-service replaced its zone-level anti-affinity with a topology spread constraint once replica count grew past the number of zones.
Common mistake: Reaching for required anti-affinity to spread N replicas across M zones when N > M — it becomes impossible to satisfy.
8. What happens if a required anti-affinity rule can't be satisfied by any node?
Short answer: The Pod stays Pending indefinitely until a satisfying node appears.
Detailed explanation: The Filter phase eliminates every node that violates the rule; if none remain, the Scheduler has nothing left to bind the Pod to.
Production example: A 4th replica of a hostname-anti-affinity-constrained Deployment stayed Pending in a 3-node cluster until a 4th node was added.
Common mistake: Setting required per-node anti-affinity on a Deployment whose replica count can exceed the node count.
9. How would you diagnose a Pod stuck Pending due to anti-affinity?
Short answer: Run kubectl describe pod and check the Events section for a scheduling failure reason referencing affinity.
Detailed explanation: The FailedScheduling event typically states something like "didn't match pod anti-affinity rules," identifying exactly which constraint filtered out every candidate node.
Production example: Ops saw "0/5 nodes are available: 3 node(s) didn't match pod anti-affinity rules" and traced it to insufficient node count for the hostname rule.
Common mistake: Debugging by checking resource requests first when the real blocker is an affinity/anti-affinity mismatch shown clearly in Events.
10. Can anti-affinity rules span namespaces, and why does that matter?
Short answer: Yes, unless scoped — the optional namespaces field on the PodAffinityTerm restricts which namespaces' Pods count toward the selector match.
Detailed explanation: Without scoping, a broad labelSelector could match unrelated Pods in other namespaces, causing unexpected scheduling failures.
Production example: Two unrelated teams both labeled Pods app=worker; one team's anti-affinity rule accidentally counted the other team's Pods until namespaces was added.
Common mistake: Writing a labelSelector broad enough to accidentally match Pods across the whole cluster.
11–50. Additional rapid-fire questions
11. Does anti-affinity have any runtime effect after a Pod is scheduled? — No, it only affects the initial Filter/Score decision; running Pods are never evicted for violating it.
12. What plugin implements pod anti-affinity in the Scheduler? — The InterPodAffinity plugin, active during both Filter and Score phases.
13. Does kubelet play any role in anti-affinity? — No, by the time a Pod reaches a node the scheduling decision is already final.
14. How is a preferred anti-affinity rule expressed? — Via preferredDuringSchedulingIgnoredDuringExecution with a weight and podAffinityTerm.
15. What does the weight field control in a preferred rule? — Its relative influence versus other preferred terms during the Score phase.
16. Why is anti-affinity described as "computationally more expensive" than node affinity? — It requires querying live Pod placement across the cluster rather than static node metadata.
17. What's a common production use for combining required + preferred anti-affinity? — Hard per-node spread plus soft per-zone spread, layering two topology domains.
18. Can anti-affinity be used to keep two *different* workloads apart? — Yes — e.g. never co-locate an app and its adjacent cache on the same node; this is the use case anti-affinity retains over topology spread constraints.
19. What object stores an anti-affinity rule? — It's persisted as part of the Pod spec (spec.affinity.podAntiAffinity) in etcd, unchanged from what was submitted.
20. Does the API Server validate anti-affinity rules semantically against live nodes? — No, only shape/schema validation; semantic feasibility is entirely the Scheduler's job at scheduling time.
21. What happens during a rolling update if the new replica's anti-affinity can't be satisfied? — The new Pod stays Pending, which can stall the rollout depending on maxUnavailable/maxSurge settings.
22. How do you check which node each replica landed on to verify anti-affinity worked? — kubectl get pods -l app=X -o wide or custom-columns showing NODE.
23. Why is a hard per-node anti-affinity risky for a high-replica-count Deployment in a small cluster? — Replicas can outnumber nodes, leaving some Pods permanently Pending.
24. What field limits which Pods count toward the anti-affinity match? — labelSelector on the podAffinityTerm.
25. Is anti-affinity Pod-spec-level or Deployment-level? — It's set in the Pod template's spec, inherited by every replica the Deployment creates.
26. What's the risk of using an overly broad labelSelector? — It may match unrelated Pods across namespaces, distorting the anti-affinity decision.
27. How does anti-affinity differ conceptually from a taint? — Anti-affinity is a Pod-side rule about other Pods; a taint is a node-side rule that repels any Pod lacking a matching toleration.
28. Why do database StatefulSets commonly use required hostname anti-affinity? — To guarantee no single node failure takes out more than one replica of a low-replica-count, stateful workload.
29. What's the interaction between anti-affinity and node draining? — Draining a node relocates Pods elsewhere; anti-affinity rules are re-evaluated as if the Pod were newly scheduled.
30. Can a preferred anti-affinity rule ever block scheduling? — No — by definition it only affects scoring, never filtering.
31. What log/event message typically indicates an anti-affinity scheduling failure? — Something like "didn't match pod anti-affinity rules" inside the FailedScheduling event.
32. Why should required inter-pod anti-affinity be tested before scaling to very large clusters? — Its evaluation cost scales with cluster size and can measurably increase scheduler latency.
33. How does anti-affinity combine with resource requests during scheduling? — Both are independent Filter-phase checks; a node must pass both to be a valid candidate.
34. What's the danger of relying on anti-affinity alone for "rebalancing" after a scale event? — It never moves already-running Pods, so imbalance introduced before the rule existed persists.
35. Why might anti-affinity and topology spread constraints be combined in one spec? — To get both "never co-locate with X" and "spread my own replicas evenly" simultaneously.
36. What's the effect of setting topologyKey to a custom node label instead of a built-in one? — The Scheduler groups nodes purely by that label's value — any custom domain works as long as nodes are labeled consistently.
37. Does anti-affinity require the CNI or networking layer to do anything? — No, it's purely a Scheduler-side placement decision, unrelated to networking.
38. How would you validate an anti-affinity manifest against the live API schema before applying? — kubectl explain deployment.spec.template.spec.affinity.podAntiAffinity.
39. What's a realistic symptom of forgetting the namespaces scoping field? — Pods stuck Pending because an unrelated namespace's same-labeled Pods are being counted against the rule.
40. Why is "3 replicas that survive a node failure" not automatic in Kubernetes? — Because the default Scheduler objective is efficient packing, not availability, unless an explicit anti-affinity/spread constraint is added.
41. What's the tradeoff of choosing zone-level over node-level anti-affinity? — Zone-level survives node failures within the same zone but not a full zone outage; it also leaves more scheduling flexibility than node-level.
42. Can you mix Equal/In-style matchExpressions in a podAffinityTerm's labelSelector? — Yes, it uses standard Kubernetes label selector matchExpressions syntax (In, NotIn, Exists, DoesNotExist).
43. What's a good way to prove anti-affinity is actually enforced in a running cluster? — Compare each replica's node via custom-columns output and confirm none repeat under a per-node rule.
44. Why does the course callout distinguish "keep me away from X" (anti-affinity) vs "spread evenly" (topology spread)? — Because conflating them leads to picking a scaling-unfriendly tool (anti-affinity) for a problem topology spread solves more cleanly.
45. What happens to anti-affinity rules when a Deployment is scaled down then back up? — Each new Pod re-evaluates the rule fresh against whatever Pods are currently running at that moment.
46. Is there a way to make anti-affinity affect already-running Pods? — Not natively — "IgnoredDuringExecution" is a permanent property of this API today; achieving that requires external tooling (e.g. a descheduler).
47. Why is combining toleration + anti-affinity uncommon compared to combining toleration + node affinity? — Anti-affinity concerns other Pods' placement, not node properties, so it doesn't need a toleration to bypass a taint — they solve different problems entirely.
48. What's the first thing you'd check if Cluster Autoscaler added nodes but a Pod is still Pending? — Whether the new node(s) actually satisfy the Pod's anti-affinity/affinity/taint constraints, not just raw capacity.
49. Why would a platform team standardize on topology spread constraints as the default and reserve anti-affinity for exceptions? — Lower Scheduler overhead and quantifiable balance at scale, keeping anti-affinity only for genuine cross-workload exclusion rules.
50. What's the key interview-ready one-liner distinguishing all three spreading tools? — Node affinity = attract to nodes; anti-affinity = repel from other Pods; topology spread constraints = quantifiably balance a Deployment's own replicas.
