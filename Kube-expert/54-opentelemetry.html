OpenTelemetry

A vendor-neutral standard and toolset for instrumenting, collecting, and exporting metrics, logs, and traces under one spec.
← Grafana Course Index Logging →

Explanation

OpenTelemetry (OTel) is a CNCF project that standardizes how applications generate telemetry — metrics, logs, and distributed traces — and how that telemetry is transported, processed, and exported to backends like Prometheus, VictoriaMetrics, Loki, Tempo, Jaeger, or commercial vendors. Before OTel, every signal type had its own competing instrumentation library and wire format (StatsD for metrics, OpenTracing and OpenCensus for traces, ad-hoc logging libraries), forcing teams to instrument code differently per backend and re-instrument entirely when switching vendors. OTel merges OpenTracing and OpenCensus into one spec and extends it to cover metrics and logs, giving one instrumentation API, one wire protocol (OTLP), and one deployable pipeline component (the Collector) for all three signal types.

Kubernetes needs this because a modern cluster runs polyglot microservices (Go, Java, Python, Node) each historically emitting telemetry in incompatible formats. Without a common standard, correlating "this trace is slow" with "this pod's CPU spiked" with "this log line has an error" requires stitching together three separate toolchains by hand. OTel's common semantic conventions (e.g. every span and log line can carry the same k8s.pod.name, service.name resource attributes) make cross-signal correlation possible by default rather than by convention teams invent themselves.

The centerpiece most platform teams deploy is the OpenTelemetry Collector: a vendor-agnostic proxy that receives telemetry (via OTLP, but also Prometheus scrape, Jaeger, Zipkin, and other protocols), applies processing (batching, filtering, sampling, adding Kubernetes metadata via the k8sattributes processor), and exports it to one or more backends. This decouples application code from backend choice entirely — apps emit OTLP to a local Collector (often a DaemonSet agent), and platform teams can swap Loki for Elasticsearch, or Tempo for a commercial APM, by changing Collector config only, with zero application redeploys.

OTel SDKs, embedded in application code, auto-instrument common frameworks (HTTP servers, gRPC, database clients) to emit spans and metrics with minimal manual code, and expose an API for custom spans/metrics where auto-instrumentation isn't enough. This is the layer application developers actually interact with; the Collector is what platform/SRE teams operate.

Within the ecosystem, OpenTelemetry is the connective tissue: it doesn't replace Prometheus, Loki, or Tempo — it standardizes how telemetry gets to them, often more efficiently and with richer context than each tool's native agent alone (e.g. richer trace-to-log correlation than a hand-rolled Fluent Bit + Prometheus setup provides).

Internal Working / Interactions

  • SDK / auto-instrumentation in the application process generates spans, metrics, and logs, exporting them via OTLP (gRPC or HTTP) to a local Collector.
  • Collector receivers accept OTLP, Prometheus-format scrapes, Jaeger, Zipkin, and other protocols — often deployed as both a DaemonSet (node-local agent, low-latency hop) and a central Gateway Deployment (cluster-wide aggregation, sampling, fan-out).
  • Collector processors — batch (efficiency), k8sattributes (enriches telemetry with pod/namespace/node metadata by querying the Kubernetes API), tail_sampling (keep only interesting traces), memory_limiter (backpressure protection).
  • Collector exporters send processed data onward — prometheusremotewrite to VictoriaMetrics, loki for logs, otlp to Tempo/Jaeger for traces.
  • The Collector itself is often managed via the OpenTelemetry Operator, which also handles auto-injecting SDK sidecars into application Pods.

Code Example

# OpenTelemetry Collector config — receives OTLP from apps, enriches with k8s metadata,
# and fans out metrics/traces/logs to their respective backends.
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317      # apps send traces/metrics/logs here via OTLP/gRPC
      http:
        endpoint: 0.0.0.0:4318      # OTLP/HTTP alternative for clients that can't do gRPC

processors:
  k8sattributes:                    # enriches every span/log/metric with pod, namespace, node labels
    extract:
      metadata:
        - k8s.pod.name
        - k8s.namespace.name
        - k8s.deployment.name
  batch:
    timeout: 5s                     # batches telemetry before export to reduce network overhead
    send_batch_size: 1000
  memory_limiter:                   # protects the Collector itself from OOM under telemetry bursts
    check_interval: 1s
    limit_mib: 512

exporters:
  prometheusremotewrite:
    endpoint: "http://vminsert.monitoring.svc:8480/insert/0/prometheus/"
  otlp/tempo:
    endpoint: "tempo.monitoring.svc:4317"
    tls: { insecure: true }
  loki:
    endpoint: "http://loki.monitoring.svc:3100/loki/api/v1/push"

service:
  pipelines:
    metrics:
      receivers: [otlp]
      processors: [k8sattributes, memory_limiter, batch]
      exporters: [prometheusremotewrite]
    traces:
      receivers: [otlp]
      processors: [k8sattributes, memory_limiter, batch]
      exporters: [otlp/tempo]
    logs:
      receivers: [otlp]
      processors: [k8sattributes, memory_limiter, batch]
      exporters: [loki]

Best Practices & Common Pitfalls

Best Practices

  • Deploy a two-tier Collector topology: node-local agent (DaemonSet) plus a central gateway for heavier processing.
  • Always include memory_limiter as the first processor to prevent OOMs during traffic spikes.
  • Use k8sattributes so every signal carries consistent pod/namespace context for correlation.
  • Use tail-based sampling for traces to keep only error/slow traces instead of sampling uniformly.
  • Standardize service.name resource attributes across teams for consistent dashboards.

Pitfalls

  • Sending 100% of traces to the backend uninspected — cost and storage explode at scale.
  • Running the Collector without resource limits, letting a telemetry burst take down the node.
  • Mixing OTel auto-instrumentation with hand-rolled StatsD/custom metrics, causing duplicate/conflicting series.
  • Not versioning Collector config alongside app deploys, causing silent pipeline breakage after upgrades.

50 Interview Questions

1. What problem does OpenTelemetry actually solve that Prometheus, Loki, and Tempo don't?
Short answer: It standardizes instrumentation and transport across all three signal types instead of replacing any of the backends.
Detail: Before OTel, metrics/traces/logs each had their own competing instrumentation library and wire format (StatsD, OpenTracing, OpenCensus, ad-hoc logging), forcing re-instrumentation per vendor.
Production example: Swapping Loki for Elasticsearch, or Tempo for a commercial APM, by changing Collector exporter config only, with zero app redeploys.
Common mistake: Thinking OTel is a monitoring backend — it's the connective tissue that feeds Prometheus/Loki/Tempo, not a replacement for them.
2. What is OTLP and why does it matter?
Short answer: OTLP is OpenTelemetry's own wire protocol (gRPC or HTTP) for transporting metrics, traces, and logs.
Detail: It's the one common protocol apps emit to and Collectors receive on, replacing the need for a separate protocol per signal type.
Production example: An app's OTel SDK exports spans/metrics/logs over OTLP/gRPC to a local Collector on port 4317, or OTLP/HTTP on 4318 for clients that can't do gRPC.
Common mistake: Assuming OTLP is required end-to-end — the Collector's receivers also accept Prometheus scrape, Jaeger, and Zipkin formats from legacy sources.
3. What are the three stages of an OpenTelemetry Collector pipeline?
Short answer: Receivers, processors, and exporters, wired together per signal type under service.pipelines.
Detail: Receivers accept incoming telemetry (OTLP, Prometheus, Jaeger, Zipkin), processors transform/enrich/filter it in-flight, and exporters send it onward to one or more backends.
Production example: The example config's traces pipeline: receivers: [otlp] → processors: [k8sattributes, memory_limiter, batch] → exporters: [otlp/tempo].
Common mistake: Forgetting each signal type (metrics/traces/logs) needs its own pipeline entry, even if they share the same receiver.
4. Why deploy the Collector as both a DaemonSet agent and a central Gateway Deployment?
Short answer: The DaemonSet agent gives every node a low-latency local hop; the Gateway does cluster-wide aggregation, sampling, and fan-out.
Detail: Apps emit OTLP to the node-local agent first (cheap, no cross-node hop), which batches and forwards to a central Gateway for heavier processing like tail-based sampling.
Production example: Agent DaemonSet receives OTLP from every pod on its node, forwards batched OTLP onward to the Gateway Deployment which applies k8sattributes/tail_sampling/batch before exporting.
Common mistake: Running only a Gateway with no local agent, forcing every application pod to reach across the cluster network for every span/metric emitted.
5. What does the k8sattributes processor actually do, and why put it early in the pipeline?
Short answer: It enriches every span, metric, and log with pod, namespace, and node metadata by querying the Kubernetes API.
Detail: Without it, telemetry arrives with no cluster context — you can't correlate a slow trace or CPU spike to a specific pod/deployment.
Production example: The example extracts k8s.pod.name, k8s.namespace.name, k8s.deployment.name onto every signal, enabling cross-signal correlation in dashboards.
Common mistake: Adding k8sattributes only to the metrics pipeline and forgetting traces/logs, breaking the promised cross-signal correlation.
6. Why should memory_limiter be the first processor in a Collector pipeline?
Short answer: It protects the Collector itself from OOMing under a telemetry burst, before any other processing happens.
Detail: memory_limiter checks memory usage on an interval and starts refusing/dropping data before the process is killed, acting as backpressure.
Production example: The example config sets check_interval: 1s and limit_mib: 512 to bound Collector memory during traffic spikes.
Common mistake: Running the Collector with no resource limits and no memory_limiter, letting a telemetry burst take down the node it runs on.
7. What is tail_sampling and why use it instead of uniform sampling?
Short answer: Tail-based sampling decides which traces to keep after seeing the whole trace, so it can retain only error/slow traces instead of a random percentage.
Detail: Uniform (head) sampling might drop the exact trace that shows a production incident; tail sampling inspects outcome first.
Production example: Gateway processors chain includes tail_sampling before batch, keeping "interesting" traces while discarding routine successful ones.
Common mistake: Sending 100% of traces to the backend uninspected — cost and storage explode at scale without any sampling strategy.
8. What role does the OpenTelemetry Operator play beyond running the Collector?
Short answer: It manages Collector deployments via a CRD and also auto-injects SDK instrumentation into application Pods.
Detail: Rather than hand-writing Collector Deployments/DaemonSets, teams manage an otelcol custom resource; the Operator also handles auto-instrumentation injection via an Instrumentation CRD.
Production example: kubectl get otelcol -n monitoring and kubectl get instrumentation -n default show the Operator-managed Collector and auto-instrumentation config respectively.
Common mistake: Manually maintaining Collector YAML alongside an installed Operator, causing drift between the CRD's desired state and what's actually running.
9. Why does OTel's semantic conventions concept (e.g. service.name) matter operationally?
Short answer: Consistent resource attributes across every span/metric/log make cross-signal correlation and dashboarding possible by default.
Detail: If every team names their service attribute differently, dashboards and correlation queries break per-team instead of working cluster-wide.
Production example: Standardizing service.name so Grafana dashboards built for one service work unmodified for any other instrumented service.
Common mistake: Mixing OTel auto-instrumentation with hand-rolled StatsD or custom metrics, causing duplicate or conflicting series with inconsistent labels.
10. If Collector logs show export errors, what's the systematic way to diagnose it?
Short answer: Check agent and gateway pod health, then inspect logs for receive/export errors, then check resource pressure.
Detail: A failure could originate at receipt (bad OTLP payload), in a processor (memory_limiter dropping data), or at export (backend unreachable).
Production example: kubectl logs -n monitoring daemonset/otel-collector-agent to inspect errors, then kubectl top pod -n monitoring -l app=otel-collector and kubectl describe pod to check for memory_limiter-driven drops from CPU/memory starvation.
Common mistake: Assuming a missing trace/metric is an app instrumentation bug before checking whether the Collector itself dropped it under memory pressure.

35. Why standardize on OTLP instead of each backend's native agent format? — One instrumentation and wire format means one set of SDKs to maintain instead of a different exporter per backend per language.
36. What's a common outcome of mixing OTel auto-instrumentation with hand-rolled StatsD metrics? — Duplicate or conflicting metric series for the same underlying signal.
37. Why is the OpenTelemetry Collector described as "vendor-agnostic"? — It receives and exports via config-driven receivers/exporters, so no backend vendor is hardcoded into the pipeline design.
38. What does the Collector's describe output help diagnose that logs alone might not show? — Resource limits and memory_limiter-driven drops, visible via kubectl describe pod on the Collector.
39. Why is the Collector often the single most critical add-on in an observability stack? — Every signal — metrics, traces, and logs — for every app flows through it; if it's down or dropping data, all three go dark simultaneously.
40. What's the first thing to check if dashboards suddenly stop correlating traces with logs for a service? — Whether the k8sattributes processor (or its metadata extraction config) is still applied consistently across all three pipelines.
41. Why does the example Collector config define separate exporters per signal (prometheusremotewrite, otlp/tempo, loki) instead of one universal exporter? — Each backend speaks its own ingestion protocol/format, so metrics, traces, and logs each need an exporter matching their destination.
42. What's the operational benefit of apps only ever needing to know about a local Collector endpoint? — Application code is fully decoupled from backend choice — no redeploy needed when the platform team changes where telemetry ultimately lands.
43. Why might a Collector Gateway need more CPU/memory than the per-node agents? — It performs heavier processing like tail_sampling across aggregated traffic from every agent, not just a single node's local telemetry.
44. What's a sign that the k8sattributes processor is failing to enrich telemetry? — Spans, metrics, or logs arriving at the backend with missing or generic pod/namespace/node context, breaking correlation.
45. Why is a receiver capable of accepting both OTLP and Prometheus scrape formats useful during a migration? — It lets teams onboard legacy Prometheus-scraped services onto the same Collector pipeline before fully re-instrumenting with OTel SDKs.
46. What's the purpose of the batch processor's send_batch_size setting? — It caps how many items accumulate before a batch is flushed, balancing latency against network efficiency.
47. Why would a platform team prefer otlp/tempo over a generic otlp exporter name in config? — Naming the exporter instance conveys its destination intent (Tempo) when multiple otlp exporters target different backends in the same pipeline set.
48. What's a reasonable way to validate Collector connectivity without a real application? — Port-forward to the Collector's OTLP/HTTP port and send a manually crafted test span/metric.
49. Why is memory_limiter placed before k8sattributes and batch in the processor chain? — So backpressure protection applies before more expensive enrichment/batching work is done on data that might get dropped anyway.
50. What's the risk of an Operator-injected auto-instrumentation sidecar going unnoticed by app teams? — Increased pod resource consumption and potential startup behavior changes that the app team didn't explicitly provision for or expect.
