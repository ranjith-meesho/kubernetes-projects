Deployments

The workhorse controller for stateless workloads — rolling updates, rollbacks, and revision history on top of ReplicaSets.
← ReplicaSets Course Index StatefulSets →

Explanation

A Deployment is a higher-level controller that manages ReplicaSets on your behalf to provide declarative updates for stateless applications. Where a bare ReplicaSet only guarantees "N identical Pods exist," a Deployment adds the crucial missing piece: a safe, controlled way to change what those Pods run — a new image version, new env vars, new resource limits — without downtime and with the ability to roll back.

Kubernetes needs this because real applications change constantly. If you edited a ReplicaSet's Pod template directly, nothing would happen to already-running Pods (templates only apply to newly created Pods), so you'd have to manually delete old Pods and let the RS recreate new ones — with no control over ordering, batching, or a safety net if the new version is broken. Deployments solve this by never editing a ReplicaSet's Pods directly; instead, on every spec change, the Deployment controller creates a brand new ReplicaSet with the updated template and gradually shifts replica counts from the old RS to the new one according to a configurable strategy.

The default and most common strategy is RollingUpdate: the Deployment controller incrementally scales the new ReplicaSet up and the old one down, bounded by maxSurge (how many extra Pods above the desired count are allowed temporarily) and maxUnavailable (how many Pods below desired count are tolerated temporarily). This guarantees that at every point during the rollout, enough healthy Pods exist to serve traffic — turning a risky "stop everything, start everything new" update into a gradual, observable, reversible process.

Because every update creates a new ReplicaSet rather than mutating the old one, Deployments get revision history almost for free: old ReplicaSets are kept around (scaled to zero) up to revisionHistoryLimit, letting you kubectl rollout undo back to any previous template instantly by simply scaling an old RS back up and the current one down — no rebuild, no re-pull needed if the image is still cached.

This gives Kubernetes a clean three-layer ownership chain: Deployment → ReplicaSet → Pod. The Deployment owns ReplicaSets via owner references and orchestrates the rollout strategy; each ReplicaSet independently guarantees its own replica count is met; Pods are the actual running units. This separation of concerns is why Deployments are the default choice for essentially all stateless workloads in production.

Internal Working / Interactions

  • API Server / etcd — the Deployment object, and every ReplicaSet + Pod it creates, are all persisted as ordinary objects; the Deployment controller only ever talks to the API Server.
  • Deployment controller (in kube-controller-manager) — watches Deployment objects; on spec change, computes a new pod-template hash, checks if a ReplicaSet with that hash already exists (reuse on rollback), and otherwise creates a new ReplicaSet, then drives the surge/unavailable math each reconcile tick.
  • ReplicaSet controller — completely unaware a Deployment exists; it just keeps whichever replica count the Deployment controller last set on each RS.
  • pod-template-hash label — the Deployment controller injects this label into every ReplicaSet's selector and Pod template, which is how it distinguishes "old" vs "new" ReplicaSets/Pods during a rollout.
  • Scheduler / kubelet — unchanged; they operate purely at the Pod level and have no concept of Deployments or ReplicaSets.

Code Example

apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service
  namespace: payments
  labels:
    app: checkout-service
spec:
  replicas: 6                        # Total desired Pods once rollout completes.
  revisionHistoryLimit: 5            # Keep 5 old ReplicaSets around for fast rollback.
  strategy:
    type: RollingUpdate              # Default; alternative is "Recreate" (kill all, then start all — has downtime).
    rollingUpdate:
      maxSurge: 2                    # Up to 2 EXTRA Pods above desired count during rollout.
      maxUnavailable: 0              # Never fewer than desired count -> zero-downtime rollout.
  minReadySeconds: 10                # Pod must stay Ready this long before counted "available" (guards flapping).
  selector:
    matchLabels:
      app: checkout-service          # Immutable — must match template labels below.
  template:
    metadata:
      labels:
        app: checkout-service
    spec:
      containers:
        - name: checkout
          image: registry.internal/checkout-service:2.4.1   # Changing this tag triggers a new rollout.
          readinessProbe:
            httpGet: { path: /healthz, port: 8080 }
            initialDelaySeconds: 5
          resources:
            requests: { cpu: "500m", memory: "512Mi" }
            limits:   { cpu: "1",    memory: "1Gi" }

kubectl Commands

# Trigger a new rollout by updating the image
kubectl set image deployment/checkout-service checkout=registry.internal/checkout-service:2.4.2 -n payments

# Watch the rollout progress live
kubectl rollout status deployment/checkout-service -n payments

# View revision history (each entry = one ReplicaSet the Deployment created)
kubectl rollout history deployment/checkout-service -n payments

# Roll back to the previous revision
kubectl rollout undo deployment/checkout-service -n payments

# Roll back to a specific revision number
kubectl rollout undo deployment/checkout-service --to-revision=3 -n payments

# Pause/resume a rollout to batch multiple template changes into one rollout
kubectl rollout pause deployment/checkout-service -n payments
kubectl rollout resume deployment/checkout-service -n payments

# See the ReplicaSets a Deployment owns (old + new)
kubectl get rs -l app=checkout-service -n payments

Best Practices & Common Pitfalls

Best Practices

  • Always set readinessProbe — the rollout's "wait for Pods Ready" step is meaningless without one.
  • Use maxUnavailable: 0 for latency-sensitive services; accept a slower rollout for zero downtime.
  • Set revisionHistoryLimit deliberately — too high wastes etcd storage with dead ReplicaSets.
  • Prefer kubectl apply with version-controlled YAML over imperative kubectl set image in production.

Pitfalls

  • Forgetting minReadySeconds lets a rollout race ahead even if new Pods crash shortly after starting.
  • Editing the Deployment's ReplicaSet or Pods directly — changes are overwritten on the next reconcile.
  • Using Recreate strategy for a service that needs zero downtime.
  • Not setting resource requests, causing rollout Pods to fail scheduling mid-rollout and stall it.

50 Interview Questions

1. What problem does a Deployment solve that a bare ReplicaSet cannot?
Short answer: Safe, controlled updates to a running workload — new image, new config — without downtime and with rollback.
Detailed explanation: A ReplicaSet has no update mechanism; editing its Pod template does nothing to existing Pods. A Deployment instead creates a brand-new ReplicaSet on every spec change and shifts replica counts between old and new according to a strategy.
Production example: Changing checkout-service's image tag via kubectl set image creates a new ReplicaSet and gradually scales it up while scaling the old one down.
Common mistake: Editing a ReplicaSet's Pod template directly and expecting a rollout to happen — nothing occurs until Pods are manually deleted.
2. How does the RollingUpdate strategy guarantee availability during a rollout?
Short answer: By bounding surge (extra Pods allowed above desired count) and unavailable (Pods allowed below desired count) at every step.
Detailed explanation: maxSurge and maxUnavailable together define the safe range the Deployment controller operates within while incrementally scaling the new ReplicaSet up and the old one down.
Production example: Setting maxSurge: 2, maxUnavailable: 0 guarantees the full desired capacity is always available, at the cost of temporarily running up to 2 extra Pods.
Common mistake: Setting maxUnavailable too high for a latency-sensitive service, causing real capacity dips during rollout.
3. Why does every spec change create a brand new ReplicaSet instead of mutating the existing one?
Short answer: Because a ReplicaSet's template changes never affect already-running Pods, so mutating in place wouldn't actually update anything.
Detailed explanation: By creating a new RS with a new pod-template-hash, the Deployment controller can gradually shift traffic between two independently-reconciling ReplicaSets and retain the old one for instant rollback.
Production example: Rolling back is just re-scaling a previous ReplicaSet back up (if still retained within revisionHistoryLimit) rather than rebuilding anything.
Common mistake: Assuming a Deployment update "patches" the existing ReplicaSet in place rather than creating a new one.
4. What is the pod-template-hash label and why does the Deployment controller need it?
Short answer: A hash of the Pod template that the Deployment controller injects into each ReplicaSet's selector and Pods to distinguish "old" from "new" during a rollout.
Detailed explanation: Without this label, the controller would have no reliable way to tell which ReplicaSet corresponds to which template revision, especially across multiple past rollouts.
Production example: kubectl get rs -l app=checkout-service shows multiple ReplicaSets, each identifiable by a distinct pod-template-hash suffix in its name.
Common mistake: Manually setting a label that collides with the auto-injected pod-template-hash, breaking the Deployment's own bookkeeping.
5. How does kubectl rollout undo actually perform a rollback?
Short answer: By reusing a previously retained ReplicaSet (if the template hash matches) and swapping replica counts back — scaling the old RS up and current RS down.
Detailed explanation: Because the Deployment controller checks for a matching pod-template-hash before creating a new RS, rolling back to an identical previous template reuses the existing RS instead of building a new one, making rollback instant if the image is still cached.
Production example: kubectl rollout undo deployment/checkout-service --to-revision=3 targets a specific retained revision by number.
Common mistake: Assuming rollback always requires re-pulling an image — it doesn't, if the old ReplicaSet's Pods can still reference an already-cached image.
6. Why is revisionHistoryLimit important and what's the tradeoff in setting it?
Short answer: It bounds how many old ReplicaSets (scaled to zero) are kept for fast rollback — too high wastes etcd storage, too low limits rollback depth.
Detailed explanation: Each retained ReplicaSet is a full object in etcd even at zero replicas; keeping dozens around for workloads that deploy frequently adds unnecessary storage/watch overhead.
Production example: A team deploying 20 times a day sets revisionHistoryLimit: 5 deliberately rather than the implicit unlimited default some tools assume.
Common mistake: Never setting it explicitly, letting hundreds of dead ReplicaSets accumulate over months of frequent deploys.
7. Why must readinessProbe be set for a rolling update to behave safely?
Short answer: Because the rollout's "wait for new Pods to be Ready before continuing" step depends entirely on readiness reporting.
Detailed explanation: Without a readinessProbe, a Pod is considered Ready the instant its containers start, so the rollout can race ahead scaling down old (working) Pods before new ones are actually able to serve traffic.
Production example: A checkout service without a readinessProbe rolling out a broken image would replace all healthy old Pods before anyone notices the new ones are failing.
Common mistake: Relying only on a livenessProbe, which restarts a crashing container but does nothing to gate rollout progression.
8. What does minReadySeconds add on top of a readinessProbe?
Short answer: A minimum duration a Pod must stay continuously Ready before being counted "available" for rollout progression.
Detailed explanation: This guards against a Pod that passes its readiness probe once, then immediately crashes — without minReadySeconds, the rollout could count it as done before the crash is detected.
Production example: minReadySeconds: 10 on checkout-service forces new Pods to prove stability for 10 seconds before the rollout proceeds to the next batch.
Common mistake: Omitting minReadySeconds and having a rollout "succeed" moments before new Pods start crash-looping.
9. Why is Recreate strategy dangerous for services that need zero downtime?
Short answer: It kills all old Pods before creating any new ones, guaranteeing a window with zero available capacity.
Detailed explanation: Unlike RollingUpdate, Recreate does not interleave old/new Pod lifecycles at all — it's simplest but sacrifices availability entirely, appropriate only for workloads that genuinely cannot run two versions simultaneously (e.g. exclusive-lock singleton processes).
Production example: A legacy batch coordinator that can't tolerate two active instances at once might deliberately use Recreate despite the downtime.
Common mistake: Using Recreate by default/inertia for a normal stateless HTTP service that could easily use RollingUpdate instead.
10. What is the three-layer ownership chain of a Deployment, and why does that separation of concerns matter?
Short answer: Deployment → ReplicaSet → Pod, where each layer owns exactly one concern: rollout orchestration, replica-count guarantee, and actual execution.
Detailed explanation: This lets each layer reconcile independently — a Pod crash is handled at the RS layer without the Deployment controller getting involved, while template/version changes are handled at the Deployment layer without the RS needing any rollout awareness.
Production example: During a rollout, if a brand-new Pod crashes, its owning ReplicaSet replaces it immediately — the Deployment's surge/unavailable accounting is unaffected by which specific Pod failed.
Common mistake: Debugging rollout issues by looking only at the Deployment object, missing that the actual state (which Pods exist, why) lives at the ReplicaSet layer.
11–50. Additional rapid-fire questions
11. What API group/version is Deployment under? — apps/v1.
12. Can you pause a Deployment mid-rollout? — Yes, kubectl rollout pause, useful for batching multiple template changes into a single rollout.
13. What does kubectl rollout resume do? — Un-pauses a paused Deployment, letting queued spec changes roll out as one combined rollout.
14. Does the Deployment controller talk to Pods directly? — No — it only ever creates/scales ReplicaSets; ReplicaSets manage Pods.
15. What triggers a new rollout besides image changes? — Any change to the Pod template: env vars, resource limits, labels, volumes, etc.
16. Does changing spec.replicas alone trigger a new ReplicaSet? — No — scaling doesn't change the template hash, so it just resizes the existing active ReplicaSet.
17. What's the default value of maxSurge and maxUnavailable? — Both default to 25% of desired replicas.
18. How do you check if a rollout is stuck? — kubectl rollout status deployment/<name> blocks and reports progress or a timeout.
19. What field bounds how long a rollout can be "stuck" before being marked failed? — progressDeadlineSeconds.
20. Can a Deployment manage Pods across multiple namespaces? — No — Deployment, its ReplicaSets, and Pods are all in the same namespace.
21. What happens if you delete a Deployment? — Cascading delete removes its ReplicaSets and their Pods by default.
22. How would you keep a Deployment's Pods running while removing just the Deployment object? — kubectl delete deployment <name> --cascade=orphan.
23. What's the difference between kubectl apply and kubectl set image for triggering rollouts? — apply is declarative (full manifest, GitOps-friendly); set image is an imperative patch, harder to track in version control.
24. Why is kubectl apply with version-controlled YAML preferred in production? — Ensures the desired state in Git matches the cluster, supporting GitOps and auditability.
25. Does a Deployment support canary releases natively? — Not directly — true weighted canary requires something like Argo Rollouts or a service mesh; Deployment only supports coarse partial-rollout via surge math.
26. What happens to HPA if it targets a Deployment mid-rollout? — HPA continues adjusting Deployment.spec.replicas, which the rollout math incorporates into surge/unavailable calculations.
27. Can two Deployments accidentally select the same Pods? — Yes, if their selectors overlap — causes the same flapping fight seen with overlapping ReplicaSets, but one layer up.
28. Is spec.selector on a Deployment mutable? — No — immutable after creation, same constraint as ReplicaSet.
29. What's the purpose of kubectl rollout history? — Lists retained revisions (backed by retained ReplicaSets) with change-cause annotations if set.
30. How do you annotate a rollout with a human-readable change reason? — kubectl annotate deployment <name> kubernetes.io/change-cause="..." or the --record flag (deprecated in newer versions).
31. What happens if a new ReplicaSet's Pods never become Ready? — The rollout stalls; after progressDeadlineSeconds it's marked as failed progress (but doesn't auto-rollback unless external tooling does so).
32. Does Kubernetes auto-rollback a failed Deployment rollout? — No, not by default — you must run kubectl rollout undo manually or use external automation.
33. What's the risk of not setting resource requests on the Pod template during a rollout? — New Pods may fail to schedule mid-rollout, stalling progress with a partially-updated fleet.
34. How does a Deployment interact with PodDisruptionBudgets during a rollout? — The rollout process itself isn't bound by PDBs (those govern voluntary evictions like drains); it's bounded by maxUnavailable/maxSurge instead.
35. Can you run two different container images simultaneously under one Deployment intentionally? — No — a Deployment's whole model assumes one template; you'd need two Deployments (or a mesh-based canary) for that.
36. What's the effect of setting maxUnavailable: 100% and maxSurge: 0%? — Equivalent to a Recreate-like behavior — old Pods are killed before new ones exist, with no surge capacity.
37. Why does a Deployment need its own controller separate from the ReplicaSet controller? — Different responsibilities — Deployment computes surge/rollout math and decides when to create new RS objects; RS controller only manages a fixed replica count.
38. What does kubectl rollout restart do differently from a normal update? — Forces a new rollout (new ReplicaSet, new Pods) without changing any spec fields — useful for picking up ConfigMap/Secret changes that aren't otherwise detected.
39. Why doesn't editing a mounted ConfigMap trigger a rollout automatically? — The Deployment only reacts to changes in its own Pod template spec, not to changes in referenced objects like ConfigMaps/Secrets.
40. What's a common pattern to force a rollout on ConfigMap change? — Add a checksum/hash annotation of the ConfigMap's content into the Pod template so its hash changes whenever the ConfigMap does.
41. Does scaling to zero replicas delete the Deployment? — No — the Deployment object and its active ReplicaSet remain, just with zero running Pods.
42. What status field tells you the count of Pods that have successfully rolled out to the new spec? — status.updatedReplicas, compared against status.replicas.
43. How does kubectl diff help before applying a Deployment change? — Shows exactly what will change server-side before committing, catching unintended template drift.
44. Is topologySpreadConstraints set on the Deployment or the Pod template? — On the Pod template (spec.template.spec) — the Deployment object itself has no such field.
45. What happens if you set replicas higher than the cluster can schedule? — Extra Pods stay Pending; rollout progress may stall depending on maxUnavailable/maxSurge accounting.
46. Can a Deployment reference a ServiceAccount? — Yes, via the Pod template's spec.serviceAccountName, same as any Pod.
47. What's the safest way to test a risky rollout in production? — Use maxUnavailable: 0 with a small maxSurge and closely watch rollout status/metrics before letting it fully complete, or pause mid-rollout.
48. Why would a team choose Argo Rollouts over plain Deployment? — For advanced strategies (canary with automated analysis, blue/green) that plain Deployment's RollingUpdate/Recreate can't express.
49. How do you verify a rollback actually reused an old ReplicaSet instead of creating a new one? — Compare pod-template-hash and creationTimestamp on the scaled-up ReplicaSet — a reused RS retains its original timestamp.
50. What's the first thing to check when a Deployment rollout appears stuck? — kubectl rollout status plus kubectl describe rs on the new ReplicaSet for scheduling/readiness failures blocking progression.
