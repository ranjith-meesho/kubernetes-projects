Horizontal Pod Autoscaler (HPA)

Automatically scaling replica count up and down based on observed metrics, closing the loop between load and capacity.
← Resource Management Course Index VPA →

Explanation

The Horizontal Pod Autoscaler (HPA) is a control loop that automatically adjusts the replicas count of a Deployment, ReplicaSet, or StatefulSet based on observed metrics — most commonly CPU or memory utilization, but also custom and external metrics (queue depth, requests-per-second, etc.). It is "horizontal" because it scales by adding/removing whole Pod replicas, in contrast to the Vertical Pod Autoscaler (topic 38), which resizes the resources of existing Pods.

Kubernetes needs HPA because static replica counts are either wasteful (provisioned for peak load, idle most of the time) or risky (provisioned for average load, falls over during spikes). HPA closes the loop: it continuously compares current metric values against a target, computes how many replicas would be needed to bring the metric back to target, and updates the workload's replica count — turning capacity planning from a manual, reactive process into an automatic, continuous one.

Architecturally, HPA is itself just a controller — HorizontalPodAutoscaler is a Kubernetes API object, and the control loop runs inside the Controller Manager (or as a separate controller in some distributions), polling on a fixed interval (default 15s, controlled by --horizontal-pod-autoscaler-sync-period). It does not collect metrics itself — it depends entirely on the metrics APIs: the resource metrics API (metrics.k8s.io, served by metrics-server) for CPU/memory, and the custom/external metrics APIs (typically backed by Prometheus Adapter or a cloud provider's metrics adapter) for anything else. Without metrics-server installed, HPA cannot function at all for CPU/memory-based scaling — this is one of the most common "HPA shows <unknown> targets" support issues.

The core scaling formula HPA uses is: desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue)). For example, if 4 replicas are averaging 80% CPU against a target of 50%, HPA computes ceil(4 * (80/50)) = 7 replicas. HPA also applies stabilization windows and scale-up/scale-down policies (behavior field) to avoid flapping — rapidly scaling up and down in response to noisy, short-lived metric spikes.

In production, HPA is almost always paired with well-set resource requests (topic 36) — CPU-based HPA computes utilization as a percentage of the Pod's request, not its limit, so an incorrectly-set request directly skews scaling decisions. It's also common to combine HPA with the Cluster Autoscaler (topic 39): HPA decides how many Pods are needed, Cluster Autoscaler decides whether the cluster has enough nodes to actually place them.

Common trap: HPA cannot scale a workload with no resource requests set for CPU — it has nothing to compute a percentage against, and the target shows <unknown> forever. Always set CPU requests on any Deployment you intend to autoscale.

Internal Working / Interactions

  • metrics-server: scrapes kubelet's /stats/summary (cAdvisor-derived) endpoint on every node every ~15-60s and exposes aggregated CPU/memory usage via the metrics.k8s.io API — the resource metrics API HPA queries for CPU/memory targets.
  • Prometheus Adapter (or similar): exposes custom (custom.metrics.k8s.io) and external (external.metrics.k8s.io) metrics APIs backed by Prometheus queries, for non-resource metrics like queue length or request rate.
  • HPA controller: polls the relevant metrics API on its sync period, computes desired replicas per the scaling formula, and applies stabilization/behavior policies before acting.
  • API Server: HPA controller PATCHes the target Deployment/StatefulSet's spec.replicas via the API Server, exactly like a human running kubectl scale.
  • Deployment/ReplicaSet controller: reacts to the new replica count exactly as it would to a manual scale — creates or deletes Pods, which then flow through the normal Scheduler/kubelet pipeline.
  • Cluster Autoscaler: if new HPA-triggered Pods can't fit on existing nodes, they go Pending, which triggers Cluster Autoscaler to add capacity (topic 39) — the two systems are decoupled but chained.

Code Example

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: checkout-hpa
  namespace: payments
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: checkout-service        # must have CPU requests set on its pod template
  minReplicas: 3                  # never scale below this, even at zero load
  maxReplicas: 20                 # hard ceiling regardless of metric pressure
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 50   # target 50% of requested CPU across all pods
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300   # wait 5 min of sustained low usage before scaling down
      policies:
        - type: Pods
          value: 2                        # remove at most 2 pods per step
          periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0       # scale up immediately, no delay
      policies:
        - type: Percent
          value: 100                      # can double replica count per step
          periodSeconds: 60

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: checkout-hpa-custom
  namespace: payments
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: checkout-service
  minReplicas: 3
  maxReplicas: 30
  metrics:
    - type: External                       # external metric from Prometheus Adapter, e.g. SQS queue depth
      external:
        metric:
          name: sqs_queue_messages_visible
          selector:
            matchLabels:
              queue: checkout-orders
        target:
          type: AverageValue
          averageValue: "30"                # aim for ~30 messages per replica

kubectl Commands

# Create an HPA imperatively (quick testing)
kubectl autoscale deployment checkout-service --cpu-percent=50 --min=3 --max=20

# View current HPA status - current vs target metric, replica count
kubectl get hpa -n payments

# Detailed view including scaling events
kubectl describe hpa checkout-hpa -n payments

# Check whether metrics-server is installed and reporting
kubectl top pods -n payments
kubectl get apiservices | grep metrics

# Watch replica count change live during a load test
kubectl get deployment checkout-service -n payments -w

# List available custom/external metrics (if Prometheus Adapter installed)
kubectl get --raw "/apis/custom.metrics.k8s.io/v1beta1" | jq .

Best Practices & Common Pitfalls

Do's

  • Always set CPU requests on any workload targeted by HPA — utilization is computed as a percentage of the request.
  • Set sensible minReplicas for baseline availability, not 1, for anything customer-facing.
  • Use behavior stabilization windows to prevent flapping on noisy metrics.
  • Prefer external/custom metrics (queue depth, RPS) over CPU alone for workloads where CPU doesn't correlate well with load.

Don'ts

  • Don't set maxReplicas without checking cluster/node-group capacity — HPA will happily request more pods than the cluster can schedule.
  • Don't combine HPA and VPA on the same metric (e.g. both managing CPU) without care — they can fight each other.
  • Don't forget HPA needs metrics-server (or an adapter) running and healthy — silent <unknown> targets mean no scaling happens at all.
  • Don't set overly aggressive scale-down stabilization windows for cost-sensitive batch-like workloads — you'll pay for idle capacity longer than necessary.

50 Interview Questions

1. What is HPA and why is it called "horizontal"?
Short answer: A control loop that adjusts a workload's replica count based on observed metrics — "horizontal" because it scales by adding/removing whole Pod replicas, not resizing existing ones.
Detailed explanation: This contrasts with VPA (topic 38), which is "vertical" — it resizes the resource requests/limits of existing Pods instead of changing how many there are.
Production example: checkout-hpa scales the checkout-service Deployment between 3 and 20 replicas based on CPU utilization.
Common mistake: Confusing HPA and VPA's axes of scaling — one changes Pod count, the other changes Pod size.
2. Why can't HPA function for CPU/memory scaling without metrics-server?
Short answer: HPA doesn't collect metrics itself — it depends entirely on the resource metrics API (metrics.k8s.io), which is served by metrics-server.
Detailed explanation: Without metrics-server installed and healthy, HPA has no data source for CPU/memory targets, resulting in the classic "" target symptom.
Production example: A fresh cluster without metrics-server showed every HPA's current CPU value as <unknown> until it was installed.
Common mistake: Assuming HPA has some built-in metrics collection — it's purely a consumer of the metrics APIs, never a producer.
3. Walk through the HPA scaling formula with an example.
Short answer: desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue)).
Detailed explanation: With 4 replicas averaging 80% CPU against a 50% target: ceil(4 * (80/50)) = 7 replicas — the formula scales proportionally to how far current usage is from target.
Production example: checkout-hpa recomputes this every 15s (the default sync period) against live metrics-server data.
Common mistake: Assuming HPA scales by a fixed step count rather than proportionally to the ratio between current and target metric value.
4. Why does CPU-based HPA require accurate resource requests to work correctly?
Short answer: Utilization is computed as a percentage of the Pod's *request*, not its limit — an incorrectly-set request directly skews scaling decisions.
Detailed explanation: If the request is set too low, a small absolute usage increase appears as a huge percentage spike, causing premature/excessive scale-up; too high, and real load never appears to cross the target.
Production example: A Deployment with an artificially low CPU request scaled to maxReplicas almost immediately under modest load because utilization percentage was distorted.
Common mistake: Deploying HPA on a workload without first validating its CPU request reflects realistic baseline usage.
5. What's the difference between resource, custom, and external metrics in HPA?
Short answer: Resource metrics (CPU/memory) come from metrics-server via metrics.k8s.io; custom and external metrics come from an adapter (typically Prometheus Adapter) via custom.metrics.k8s.io/external.metrics.k8s.io.
Detailed explanation: Custom metrics are tied to a Kubernetes object (e.g. Pod-level requests-per-second); external metrics originate outside the cluster entirely (e.g. an SQS queue depth).
Production example: checkout-hpa-custom scales on sqs_queue_messages_visible, an external metric from Prometheus Adapter reflecting queue backlog, not CPU at all.
Common mistake: Assuming all non-CPU/memory metrics are automatically available — they require deploying and correctly configuring an adapter first.
6. How does the behavior field prevent flapping?
Short answer: It defines separate stabilization windows and scale-up/scale-down policies so HPA doesn't rapidly oscillate replica count in response to noisy, short-lived metric spikes.
Detailed explanation: A scale-down stabilization window (e.g. 300s) requires sustained low usage before removing replicas, while scale-up can be configured to react immediately since under-provisioning is usually riskier than over-provisioning briefly.
Production example: checkout-hpa sets scaleDown.stabilizationWindowSeconds: 300 with a max of 2 pods removed per 60s, but scaleUp.stabilizationWindowSeconds: 0 for immediate response to spikes.
Common mistake: Setting overly aggressive scale-down stabilization windows for cost-sensitive batch-like workloads, paying for idle capacity longer than necessary.
7. How do HPA and Cluster Autoscaler interact in production?
Short answer: HPA decides how many Pods are needed; Cluster Autoscaler decides whether the cluster has enough nodes to place them — they're chained, not directly coupled.
Detailed explanation: If HPA-triggered Pods can't fit on existing nodes, they go Pending, which is the signal Cluster Autoscaler reacts to by adding capacity (topic 39). This chain has real latency — cloud instance boot time is often the dominant factor (1-3 minutes), motivating headroom buffers or placeholder Pods.
Production example: A load spike triggered HPA to request 10 more replicas; 4 went Pending until Cluster Autoscaler added 2 nodes roughly 90 seconds later.
Common mistake: Assuming HPA scaling is instantaneous end-to-end when the cluster lacks headroom — the Cluster Autoscaler chain adds real, sometimes multi-minute, latency.
8. What is the risk of running HPA and VPA on the same metric for the same workload?
Short answer: They can fight each other — VPA changing a Pod's CPU request changes the denominator HPA uses for utilization, causing instability.
Detailed explanation: This is why the best-practice pattern is to have HPA scale on CPU/memory OR let VPA manage sizing while HPA scales on an unrelated custom/external metric — never both governing the same resource dimension simultaneously.
Production example: A workload with both HPA (CPU) and VPA (CPU, Auto mode) enabled churned replicas erratically until VPA was switched to Off mode.
Common mistake: Enabling VPA in Auto mode on a workload already scaled by CPU-based HPA without redesigning which controller owns which dimension.
9. Where does the HPA control loop actually run, and how often does it evaluate?
Short answer: Inside the Controller Manager (or a separate controller in some distributions), polling on a fixed interval — default 15s via --horizontal-pod-autoscaler-sync-period.
Detailed explanation: HorizontalPodAutoscaler is itself just a Kubernetes API object; the controller watches it, queries the relevant metrics API each sync period, and PATCHes the target's spec.replicas exactly like a manual kubectl scale.
Production example: Lowering the sync period cluster-wide would make every HPA in the cluster react faster, at the cost of more frequent metrics API load.
Common mistake: Assuming HPA reacts instantly to metric changes rather than on its fixed sync-period cadence.
10. How would you troubleshoot an HPA stuck showing "" for its target metric?
Short answer: Check whether metrics-server (or the relevant adapter) is installed and healthy, and confirm the target Deployment has resource requests set.
Detailed explanation: kubectl describe hpa shows the exact metric and its current status; kubectl top pods and kubectl get apiservices | grep metrics confirm whether the underlying metrics pipeline is even functioning.
Production example: An HPA showed targets for a week because metrics-server's pod was CrashLoopBackOff from a resource limit misconfiguration.
Common mistake: Debugging the HPA object itself first instead of first verifying the metrics pipeline it depends on is healthy.

36. What's a common cost-optimization tradeoff with scale-down stabilization windows? — Longer windows avoid flapping but pay for idle capacity longer; shorter windows save cost but risk premature scale-down on noisy metrics.
37. Does HPA work without any resource requests set, using only external metrics? — Yes — external/custom metrics don't require CPU/memory requests, only resource-type metrics do.
38. What's the risk of combining HPA and Cluster Autoscaler without headroom planning? — Scale-up latency compounds — HPA creates Pods instantly, but they stay Pending until Cluster Autoscaler provisions and boots new nodes.
39. How would you watch replica count change live during a load test? — kubectl get deployment <name> -w.
40. What's the significance of HPA being "just a controller" architecturally? — It reinforces that HorizontalPodAutoscaler is an ordinary API object reconciled like any other resource, not a special built-in mechanism.
41. What's the danger of not setting any behavior/stabilization config at all? — HPA falls back to Kubernetes' defaults, which may not suit noisy or bursty workloads, risking flapping.
42. Can HPA scale to zero replicas? — No, standard HPA respects a minReplicas floor of at least 1 in typical configurations; scale-to-zero requires additional tooling (e.g. KEDA).
43. What's a realistic production incident caused by HPA maxReplicas misconfiguration? — A traffic spike hit maxReplicas and stayed there, capping capacity below actual demand and degrading latency.
44. Why does HPA depend on the same metrics pipeline as VPA for resource metrics? — Both consume metrics-server's metrics.k8s.io API as their resource usage data source, just for different purposes (scaling vs sizing).

47. Why is it good practice to test HPA behavior under synthetic load before production rollout? — To validate the scaling formula, stabilization windows, and metrics pipeline actually produce the intended replica trajectory under realistic traffic patterns.
