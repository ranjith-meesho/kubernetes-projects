Argo CD

A CNCF GitOps controller: the Application CRD, continuous sync/self-heal, and the App-of-Apps pattern.
← GitOps Course Index Observability →

Explanation

Argo CD is the most widely adopted implementation of the GitOps pattern (Topic 48) for Kubernetes — it is itself a Kubernetes-native controller, installed as a set of Pods in your cluster, that watches its own CRD, Application, and continuously reconciles a target cluster/namespace to match a given Git path. Where the previous topic described GitOps conceptually, Argo CD is the concrete, production-grade tool most teams reach for to implement it.

An Application object declares three things: a source (Git repo + path + revision, optionally a Helm chart or Kustomize overlay), a destination (which cluster and namespace to deploy into — Argo CD can manage many clusters from one control instance), and a sync policy (manual, or automated with optional self-heal and pruning). Argo CD's core loop continuously renders the source, compares it against the live destination via the API Server, and reports a SyncStatus (Synced / OutOfSync) plus a HealthStatus (aggregated from each resource's own health — e.g. a Deployment is "Healthy" once its rollout completes and Pods are Ready).

With automated sync enabled, Argo CD applies any detected diff without human intervention; self-heal goes further and actively reverts manual out-of-band changes (like a rogue kubectl edit) back to what Git specifies, closing the drift gap discussed in the GitOps topic. Manual sync mode instead just surfaces OutOfSync and waits for a human (or a promotion pipeline) to click "Sync" — commonly used for production environments where a deliberate approval gate is wanted even though the desired state is already in Git.

The App-of-Apps pattern solves the "how do I manage a whole platform's worth of Applications, not just one" problem: a parent Application object's source is itself a Git directory containing many child Application manifests. Applying the parent causes Argo CD to create/reconcile all the children — this is how platform teams manage dozens or hundreds of Applications (one per microservice, or one per tenant/environment) declaratively, using the exact same CRD recursively, rather than clicking through a UI or scripting argocd app create calls.

Argo CD's comparison of "desired (Git)" vs "live (cluster)" is continuous and visible — its UI/CLI literally shows a diff per resource, which is a major operational advantage over CI-push deployment: at any moment, anyone can see exactly what's out of sync and why, rather than trusting that the last pipeline run "probably worked."

Internal Working / Interactions

  • repo-server component: clones the Git repo and renders manifests (plain YAML, Helm, Kustomize, or plugin-based) into final Kubernetes objects.
  • application-controller component: the actual reconciliation loop — diffs rendered manifests against live cluster state via the API Server, computes SyncStatus/HealthStatus, and applies changes per the Application's sync policy.
  • API Server (target cluster): receives Argo CD's apply calls via a ServiceAccount/cluster-credential scoped to what that Application is allowed to manage.
  • argocd-server (API/UI): exposes the CLI/UI/API for humans and automation to inspect Applications and trigger manual syncs.
  • Notifications/webhooks: Argo CD can push Slack/webhook notifications on sync state changes, and can be triggered immediately on Git push via a repo webhook instead of waiting for the poll interval.

Code Example

# 1. A standard Argo CD Application — one microservice deployed from a Kustomize overlay
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: checkout-service
  namespace: argocd
spec:
  project: payments
  source:
    repoURL: https://github.com/meesho/infra-gitops.git
    targetRevision: main
    path: apps/checkout-service/overlays/prod
  destination:
    server: https://kubernetes.default.svc   # In-cluster; could also be a remote cluster's API URL.
    namespace: payments-prod
  syncPolicy:
    automated:
      prune: true       # Delete resources removed from Git.
      selfHeal: true     # Revert manual out-of-band changes back to Git's desired state.
    syncOptions:
      - CreateNamespace=true
---
# 2. App-of-Apps root — one Application whose "source" is a directory of other Application manifests
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: platform-root
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/meesho/infra-gitops.git
    targetRevision: main
    path: apps/                # Directory containing checkout-service.yaml, catalog-service.yaml, etc.
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

kubectl Commands

argocd app list                                  # List all managed Applications and their Sync/Health status
argocd app get checkout-service                    # Detailed status + per-resource diff for one Application
argocd app sync checkout-service                    # Manually trigger a sync (for manual sync-policy Applications)
argocd app diff checkout-service                    # Show exactly what differs between Git and the live cluster
argocd app history checkout-service                  # List past sync revisions (for rollback)
argocd app rollback checkout-service 4               # Roll back to a previous synced revision
kubectl get applications -n argocd                    # Applications are just CRs — inspect them with plain kubectl too
kubectl describe application checkout-service -n argocd  # See conditions/events straight from the API Server

Best Practices & Common Pitfalls

Best Practices

  • Use App-of-Apps to manage fleets of Applications declaratively instead of manual argocd app create scripting.
  • Enable selfHeal in production so drift is actively corrected, not just flagged.
  • Scope each Application to an ArgoCD AppProject restricting which repos/destinations/resource kinds it may touch.
  • Use manual sync + required approvals for the highest-risk production Applications if a human gate is desired.

Pitfalls

  • Enabling prune: true without understanding it will delete cluster resources no longer present in Git — dangerous on shared namespaces.
  • Granting Argo CD's ServiceAccount cluster-admin across every managed cluster instead of least-privilege per AppProject.
  • Ignoring persistently OutOfSync Applications — it usually means either drift is happening or the rendered manifests are non-deterministic.
  • Storing Helm values with plaintext secrets referenced by an Application's source — pair with Sealed Secrets/external-secrets.

50 Interview Questions

1. What three things does an Argo CD Application object declare?
Short answer: A source (Git repo/path/revision or chart), a destination (target cluster and namespace), and a sync policy (manual or automated with self-heal/prune options).
Detail: These three fields fully describe "this Git path should be reconciled into this cluster/namespace," which is the entire contract an Application represents — Argo CD is itself just a controller reconciling instances of this one CRD.
Production example: An Application named checkout-service pointing at apps/checkout-service/overlays/prod, targeting the payments-prod namespace with automated sync and selfHeal enabled.
Common mistake: Assuming the Application object is just configuration metadata rather than the actual object Argo CD's controller watches and reconciles like any CR.
2. What's the difference between SyncStatus and HealthStatus on an Application?
Short answer: SyncStatus (Synced/OutOfSync) reflects whether live state matches Git; HealthStatus reflects whether the deployed resources are actually functioning correctly.
Detail: A Deployment can be Synced (exactly what Git says) yet still Degraded in health if its Pods are crash-looping — the two statuses are orthogonal and both need to be green for a truly healthy deployment.
Production example: An Application showing Synced but Progressing/Degraded health while a new rollout's Pods are still becoming Ready.
Common mistake: Only checking SyncStatus and assuming "Synced" means the application is actually working correctly.
3. What's the difference between automated sync and self-heal?
Short answer: Automated sync applies any detected diff without human intervention; self-heal goes further and actively reverts manual out-of-band changes back to Git's desired state.
Detail: You can have automated sync without self-heal (new Git changes auto-apply, but manual kubectl edits are only flagged, not reverted) — self-heal is the stricter setting that actively fights drift from any source.
Production example: selfHeal: true reverting a rogue `kubectl scale` back to the replica count declared in Git within one reconcile cycle.
Common mistake: Assuming automated: {} alone (without selfHeal: true) will revert manual cluster edits — it won't, it only handles new Git-driven changes.
4. What problem does the App-of-Apps pattern solve?
Short answer: Managing fleets of many Applications declaratively — a parent Application's source is itself a directory of child Application manifests.
Detail: Applying the parent causes Argo CD to create/reconcile all the children, letting platform teams manage dozens or hundreds of Applications the same declarative way, using the same CRD recursively, rather than scripting `argocd app create` calls.
Production example: A platform-root Application whose source path is apps/, containing checkout-service.yaml and catalog-service.yaml as child Application manifests.
Common mistake: Manually creating dozens of individual Applications via the UI/CLI instead of composing them declaratively through a parent App-of-Apps.
5. What do the repo-server and application-controller components each do?
Short answer: repo-server clones Git and renders manifests (Helm/Kustomize/plain YAML); application-controller performs the actual diff-and-sync reconciliation loop.
Detail: This separation means rendering (potentially expensive, template-heavy work) is isolated from the core reconciliation logic that computes SyncStatus/HealthStatus and applies changes per the sync policy.
Production example: repo-server rendering a Helm chart's templates, then application-controller diffing the rendered output against the live cluster via the target API Server.
Common mistake: Debugging a stuck sync by only checking application-controller logs when the actual failure is in repo-server's manifest rendering.
6. What's the risk of enabling prune: true on an Application's sync policy?
Short answer: It deletes any cluster resource no longer present in Git, which can be dangerous on shared namespaces if something was created outside of Argo CD's management.
Detail: Prune makes Git a truly exhaustive source of truth — anything not declared there for that Application gets removed, so shared namespaces need careful scoping to avoid accidentally deleting resources another team or process manages.
Production example: Enabling prune on an Application scoped too broadly accidentally deletes a manually-created debugging ConfigMap another team relied on.
Common mistake: Enabling prune: true without first confirming exactly what resources the Application's manifests currently include versus what actually exists live in that namespace.
7. Why might a team choose manual sync mode for a production environment even though Git already has the desired state?
Short answer: To keep a deliberate human approval gate before changes are actually applied, even though what would be applied is already fully declared and reviewed in Git.
Detail: Manual sync surfaces OutOfSync and waits for a human (or a promotion pipeline) to click "Sync" — useful when teams want a final go/no-go moment distinct from the PR merge itself, e.g. coordinating with a release window.
Production example: A prod Application using manual sync so a release manager can trigger the actual deploy at a scheduled maintenance window, independent of when the PR merged.
Common mistake: Assuming manual sync mode means the change isn't "GitOps compliant" — it still tracks Git as the source of truth, it just adds an explicit apply gate.
8. What's the purpose of an AppProject in Argo CD?
Short answer: It scopes which repos, destinations, and resource kinds an Application (or group of Applications) is allowed to use, providing multi-tenant guardrails.
Detail: Without AppProject scoping, any Application could theoretically point at any repo or destination cluster the Argo CD instance has credentials for — AppProject constrains that per team/tenant.
Production example: A payments AppProject restricting Applications to only the infra-gitops repo's payments/ path and only the payments-prod destination namespace.
Common mistake: Leaving every Application in the default AppProject with no restrictions, giving any team's Application effectively unlimited destination/repo reach.
9. How does Argo CD get notified of Git changes faster than its default poll interval?
Short answer: Via a repo webhook that immediately notifies Argo CD of a push, rather than waiting for the next periodic poll.
Detail: Without a webhook, Argo CD's application-controller still eventually detects changes on its regular poll cycle, but a webhook shortens the feedback loop from "next poll" to "near-instant."
Production example: A GitHub webhook configured against the argocd-server endpoint triggering an immediate refresh after any push to the tracked repo.
Common mistake: Assuming Argo CD requires a webhook to detect changes at all — polling alone works, just with more latency.
10. Why is Argo CD's continuous, visible diff considered a major operational advantage over CI-push deployment?
Short answer: At any moment, anyone can see exactly what's out of sync and why, rather than trusting that the last pipeline run "probably worked."
Detail: The UI/CLI shows a per-resource diff between desired (Git) and live (cluster) state continuously, not just at deploy time — turning "is prod in the state we think it's in" from a guess into a directly observable fact.
Production example: During an incident, immediately checking `argocd app diff checkout-service` to rule in/out configuration drift as a root cause.
Common mistake: Still relying on "the last deploy pipeline succeeded" as proof of current cluster state instead of checking Argo CD's live diff.

20. Can Argo CD manage multiple remote clusters from one control instance? — Yes, that's a core capability — many destination clusters from one Argo CD install.
21. What's the purpose of syncPolicy.automated.prune? — Deletes resources removed from Git during automated sync.
22. What's the purpose of syncPolicy.automated.selfHeal? — Reverts manual out-of-band changes back to Git's desired state.
23. Does the App-of-Apps parent Application itself need automated sync enabled to create its children? — Typically yes, or a manual sync of the parent to trigger child Application creation.
24. What's a HealthStatus value indicating a rollout is still in progress? — Progressing.
25. What's a HealthStatus value indicating something is actively broken? — Degraded.
26. Can Argo CD send Slack/webhook notifications on sync state changes? — Yes, via its notifications integration.
27. What's the risk of granting Argo CD's ServiceAccount cluster-admin across every managed cluster? — Massively expands blast radius versus least-privilege per AppProject scoping.
28. What does a persistently OutOfSync Application usually indicate? — Either genuine drift happening repeatedly, or non-deterministic rendered manifests causing false diffs.
29. What's the source field's targetRevision typically set to for a floating branch? — A branch name like main, or a fixed tag/commit SHA for stricter pinning.
30. Does Argo CD support Helm charts as an Application source, not just plain YAML/Kustomize? — Yes, Helm charts (including with value overrides) are a supported source type.
31. What's the effect of scoping an AppProject's destinations narrowly? — Prevents an Application in that project from ever targeting a cluster/namespace outside the allowed list.
32. How would you find the history of past sync revisions for rollback purposes? — argocd app history <name>.
33. What's a good practice for the highest-risk production Applications regarding sync mode? — Manual sync with required human approval, even if automated sync is used elsewhere.
34. Can secrets be safely referenced by an Argo CD Application's Helm values? — Only if paired with Sealed Secrets/external-secrets — plaintext secrets in values remain a risk regardless of Argo CD.
35. What's the relationship between Argo CD and the underlying GitOps pattern (Topic 48)? — Argo CD is one concrete, production-grade implementation of that general pattern.
36. Does Argo CD's application-controller talk to etcd directly? — No, it only talks to the target cluster's API Server, same as any client.
37. What's a reason an Application might fail to sync even though the manifests look correct? — Insufficient RBAC on Argo CD's ServiceAccount for the destination cluster/namespace/resource kind.
38. How does Argo CD handle CRDs referenced by an Application's manifests? — It applies them like any other resource, subject to the same sync/health evaluation, though CRD install ordering can require sync waves/hooks.
39. What's a "sync wave" in Argo CD? — An ordering mechanism (via annotations) controlling which resources apply before others within one sync operation.
40. Why might you use sync waves for a CRD and its consuming custom resource in the same Application? — To ensure the CRD is established before the API Server needs to validate instances of that custom resource.
41. What's the effect of deleting an Application object itself (not its resources)? — By default, only removes Argo CD's tracking of it; whether managed resources are also deleted depends on cascade/finalizer settings.
42. Can an Application track a specific Git tag instead of a branch? — Yes, targetRevision can be any valid Git ref — branch, tag, or commit SHA.
43. What's a common cause of "false" OutOfSync status? — Non-deterministic rendering (e.g. timestamps or random values injected by a Helm template) causing every diff to look different.
44. How does Argo CD differ operationally from Flux in basic architecture? — Both are GitOps controllers with their own CRDs, but Argo CD centers on the Application CRD with a strong UI, while Flux is more toolkit-composed (separate Kustomization/HelmRelease/Source controllers).
45. What's the benefit of the App-of-Apps root itself being version-controlled? — Even the set of Applications managed becomes auditable and reviewable via the same Git PR process.
46. Can an Argo CD Application's destination be a namespace that doesn't exist yet? — Yes, if CreateNamespace=true is set in syncOptions.
47. What's a risk of using manual sync for every single Application in a large platform? — Operational overhead — someone has to remember to manually sync potentially hundreds of Applications after every relevant Git change.
48. What's the value of Argo CD's per-resource diff view during an incident? — Immediately distinguishes "cluster matches Git but something is still broken" from "cluster has actually drifted from Git."
49. Does self-heal interfere with legitimate automated changes made by other controllers (e.g. HPA adjusting replicas)? — It can, unless the Application is configured to ignore differences on fields managed by another controller (via ignoreDifferences).
50. What's the first thing to check when an Application is stuck Progressing and never reaches Healthy? — The underlying resource's own events/status (e.g. Pod CrashLoopBackOff) via kubectl describe, since Argo CD's health is only aggregated from the resource's own reported health.
