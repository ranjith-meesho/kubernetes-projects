Node & Pod Affinity

nodeAffinity vs podAffinity, and the required-vs-preferred distinction that controls how strict a rule is.
← Scheduling Course Index Anti-Affinity →

Explanation

Affinity rules let you express placement preferences richer than a flat nodeSelector label match. Kubernetes needs this because real placement decisions are rarely a single equality check — you often want "prefer SSD nodes but fall back to HDD," or "run this cache Pod on the same node as the app it serves," or "spread these replicas so no two land in the same zone." Two distinct rule families exist because they answer two different questions: nodeAffinity asks "what does the node itself look like" (labels, instance type, zone), while podAffinity/podAntiAffinity ask "what other Pods are already running nearby" — a relational constraint the scheduler resolves by looking at currently-running Pods' labels, not static node metadata.

nodeAffinity is essentially nodeSelector with boolean logic (In, NotIn, Exists, Gt, Lt) and a topology dimension. It's the natural fit for constraints like "must run on amd64 nodes," "prefer nodes in us-east-1a," or "must run on nodes with an SSD-backed instance type."

podAffinity pulls Pods together — e.g. co-locating a web Pod with its Redis cache Pod on the same node (minimizing network latency) or, more commonly, within the same zone (topologyKey: topology.kubernetes.io/zone) to avoid cross-AZ data transfer costs and latency. podAntiAffinity pushes Pods apart — the standard mechanism for spreading replicas of the same Deployment across different nodes or zones so a single node/zone failure doesn't take out every replica at once. (Topic 32 goes deeper on anti-affinity specifically.)

Every affinity rule — node or pod — comes in two flavors that control how strictly it's enforced. requiredDuringSchedulingIgnoredDuringExecution is a hard constraint: it behaves like a Filter — any node that doesn't satisfy it is eliminated outright, and the Pod stays Pending if no node qualifies. preferredDuringSchedulingIgnoredDuringExecution is a soft constraint fed into Scoring with a weight (1-100): the scheduler tries to satisfy it but will still place the Pod elsewhere if nothing else fits, just with a lower score. The "IgnoredDuringExecution" suffix on both is a deliberate design detail: once a Pod is running, Kubernetes does not re-evaluate or move it if the affinity condition later becomes false (e.g. the node's label changes) — affinity governs initial placement only, not ongoing enforcement.

Internal Working / Interactions

  • kube-scheduler Filter plugins — enforce all "required" node/pod affinity rules, eliminating non-matching nodes.
  • kube-scheduler Score plugins — evaluate "preferred" rules and add weighted scores to surviving candidate nodes.
  • API Server / etcd — the scheduler reads live Pod and Node label state from its informer cache (backed by watches against the API Server) to evaluate podAffinity, which requires knowing what's currently running everywhere.
  • Node labels vs Pod labels — nodeAffinity matches against Node object labels; podAffinity/AntiAffinity match against labels of other Pod objects, correlated via a topologyKey (e.g. node label topology.kubernetes.io/zone) that groups nodes into domains.

Architecture Diagram

Code Example

# nodeAffinity: hard requirement + soft preference combined
apiVersion: v1
kind: Pod
metadata:
  name: web-frontend
  labels:
    app: web-frontend
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/arch
            operator: In
            values: ["amd64"]                 # hard: must be amd64, no exceptions
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80                             # soft: try hard to satisfy, but not mandatory
        preference:
          matchExpressions:
          - key: node.kubernetes.io/instance-type
            operator: In
            values: ["m6i.large", "m6i.xlarge"]  # prefer these instance types if available
  containers:
  - name: web
    image: myco/web:2.3

---
# podAffinity: co-locate with cache Pods in the same zone (not necessarily same node)
apiVersion: v1
kind: Pod
metadata:
  name: web-frontend-2
  labels:
    app: web-frontend
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values: ["redis-cache"]             # must land near a Pod labeled app=redis-cache
        topologyKey: topology.kubernetes.io/zone  # "near" = same zone, not necessarily same node
  containers:
  - name: web
    image: myco/web:2.3

kubectl Commands

kubectl get nodes --show-labels                              # inspect labels used by nodeAffinity rules
kubectl label nodes node-1 topology.kubernetes.io/zone=us-east-1a  # set/verify topology labels
kubectl describe pod web-frontend                             # scheduling events, unmet affinity reasons
kubectl get pods -o wide -l app=redis-cache                   # see where podAffinity target Pods are running
kubectl get pod web-frontend -o jsonpath='{.spec.affinity}'   # dump the affinity spec actually applied
kubectl get events --field-selector reason=FailedScheduling   # find Pods rejected due to affinity constraints

Best Practices & Common Pitfalls

Do
  • Default to "preferred" rules unless a hard requirement is truly non-negotiable — required rules can strand Pods Pending.
  • Use a broad topologyKey (zone) for podAffinity when node-level co-location isn't strictly necessary.
  • Combine nodeAffinity with taints/tolerations for dedicated pools rather than affinity alone.
  • Test affinity rules in a staging cluster with realistic node label sets before rolling to prod.
Pitfalls
  • Required podAffinity rules referencing a Pod label that doesn't exist yet, causing a chicken-and-egg scheduling deadlock.
  • Expecting affinity to react to runtime changes — "IgnoredDuringExecution" means it won't.
  • Overusing required node affinity for minor preferences, fragmenting capacity and causing Pending Pods during node churn.
  • Forgetting topologyKey must be a label that actually exists and is consistent across all relevant nodes.

50 Interview Questions

1. Why does Kubernetes split placement rules into nodeAffinity and podAffinity/podAntiAffinity instead of one unified mechanism?
Short answer: They answer fundamentally different questions — nodeAffinity asks what a node looks like, podAffinity/podAntiAffinity ask what other Pods are already running nearby.
Detailed explanation: nodeAffinity is evaluated purely against static Node object labels (instance type, zone, arch) — it's essentially nodeSelector with richer boolean logic. podAffinity/podAntiAffinity are relational: the scheduler must inspect the labels of currently-running Pods and correlate them through a topologyKey, which is a fundamentally different kind of lookup requiring live cluster state rather than a fixed label set.
Production example: A cluster uses nodeAffinity to pin a Pod to amd64 nodes, and separately uses podAffinity to co-locate that same Pod with its Redis cache in the same zone — two independent decisions answered by two independent mechanisms.
Common mistake: Trying to express "run near another Pod" using nodeSelector/nodeAffinity by hand-labeling nodes, instead of using podAffinity which tracks actual Pod placement dynamically.
2. What role does topologyKey play in podAffinity and podAntiAffinity rules?
Short answer: topologyKey defines the granularity of "nearby" by naming a node label that groups nodes into domains, such as zone or hostname.
Detailed explanation: podAffinity/podAntiAffinity don't compare Pods directly to each other; they compare via nodes. The scheduler finds nodes running Pods that match the labelSelector, reads each of those nodes' topologyKey label value, and treats any node sharing that same value as satisfying (or violating, for anti-affinity) the rule. A node-level key like kubernetes.io/hostname means "same node"; a broader key like topology.kubernetes.io/zone means "same zone."
Production example: Using topologyKey: topology.kubernetes.io/zone to co-locate a web Pod with redis-cache Pods lets the scheduler pick any node in the same AZ, not necessarily the exact same node, avoiding cross-AZ costs while still allowing flexibility.
Common mistake: Choosing a topologyKey label that isn't consistently applied across all nodes, causing some nodes to be silently excluded from evaluation.
3. What's the practical difference between requiredDuringSchedulingIgnoredDuringExecution and preferredDuringSchedulingIgnoredDuringExecution?
Short answer: Required is a hard Filter — nodes that don't match are eliminated and the Pod stays Pending if none qualify; preferred is a soft, weighted Score that influences ranking without blocking placement.
Detailed explanation: Required rules run in the scheduler's Filter phase, pruning the candidate node list before scoring even begins. Preferred rules run in the Score phase: each rule carries a weight from 1-100, and nodes satisfying more/higher-weighted preferences rank higher, but a Pod will still be placed somewhere even if no preference is met.
Production example: A Pod with a required amd64 nodeAffinity rule and a preferred instance-type rule will never land on an arm64 node, but will happily land on an m6i.2xlarge if the preferred m6i.large/xlarge types are all busy.
Common mistake: Marking a rule as required when a preferred rule would achieve the intent with far less risk of Pods becoming stuck Pending during capacity crunches.
4. What does the "IgnoredDuringExecution" suffix actually mean, and why does it matter operationally?
Short answer: It means affinity rules only govern the moment of initial scheduling — Kubernetes never re-evaluates or evicts/moves a running Pod if the condition later becomes false.
Detailed explanation: Once a Pod is bound and running, changes to node labels or the disappearance/appearance of other Pods that would have changed the original scheduling decision have zero effect on that already-placed Pod. There is no controller that continuously reconciles affinity against live state for running Pods; it's a one-time placement filter/score, not an ongoing constraint.
Production example: If a node's zone label is corrected after a Pod was scheduled with a required podAffinity rule keyed on that zone, the already-running Pod stays put even though it would no longer pass scheduling if re-evaluated today.
Common mistake: Assuming affinity rules act like a continuously enforced policy (similar to NetworkPolicy) that will react to drift, and being surprised when a stale placement isn't automatically corrected.
5. In a production scenario, why would you use podAffinity to co-locate a web tier with a Redis cache in the same zone rather than the same node?
Short answer: Same-zone co-location captures most of the latency/cost benefit of proximity while avoiding the fragility of pinning to one exact node.
Detailed explanation: Cross-AZ network calls typically carry both extra latency and per-GB data transfer costs in cloud environments. Requiring the exact same node (kubernetes.io/hostname topologyKey) ties both Pods' fate to a single node's capacity and failure domain; requiring only the same zone still eliminates cross-AZ hops while letting the scheduler freely choose among many nodes.
Production example: A web-frontend Pod with a required podAffinity rule matching app: redis-cache and topologyKey: topology.kubernetes.io/zone guarantees zone-local Redis calls without forcing both Pods onto one physical node.
Common mistake: Using kubernetes.io/hostname for co-location "for lowest latency" without considering that it also removes scheduling flexibility and couples both Pods to the same node's health.
6. What is the "chicken-and-egg" deadlock that required podAffinity rules can cause, and how does it happen?
Short answer: A required podAffinity rule referencing a Pod label that no Pod currently satisfies leaves every node ineligible, so the new Pod stays Pending forever.
Detailed explanation: podAffinity is relational — it needs at least one currently-running Pod matching the labelSelector to anchor a topology domain as valid. If the target Pods (e.g. app: redis-cache) haven't been scheduled yet, or were deleted, there's no node satisfying the constraint, and the Filter phase eliminates every candidate node.
Production example: Deploying a web tier with required podAffinity on redis-cache before the Redis Deployment has been applied causes every web Pod to sit Pending until at least one Redis Pod is scheduled somewhere.
Common mistake: Applying manifests with an ordering assumption baked into required podAffinity without verifying deployment order, or without a preferred fallback for first-time bootstraps.
7. How does nodeAffinity's boolean logic (In, NotIn, Exists, Gt, Lt) go beyond what nodeSelector can express?
Short answer: nodeSelector only supports exact key=value equality across all entries (implicit AND), while nodeAffinity's matchExpressions support set-membership, existence checks, and numeric comparisons.
Detailed explanation: Operators like In/NotIn allow matching against a list of acceptable values (e.g. several instance types), Exists/DoesNotExist check for label presence regardless of value, and Gt/Lt allow numeric comparisons on label values. Multiple nodeSelectorTerms are OR'd together, while expressions within one term are AND'd, giving compound logic nodeSelector cannot express at all.
Production example: A rule requiring instance type In ["m6i.large", "m6i.xlarge"] cannot be written with plain nodeSelector, which would need an exact single value per key.
Common mistake: Continuing to use nodeSelector for constraints that actually need set-membership or negation logic, then working around it with multiple near-duplicate Pod specs.
8. A Pod is stuck Pending and you suspect an affinity rule is the cause — how would you confirm and troubleshoot it?
Short answer: Check the Pod's scheduling events for a FailedScheduling reason describing unmet affinity, then verify the actual labels on nodes or target Pods against the rule.
Detailed explanation: kubectl describe pod surfaces scheduler event messages like "node(s) didn't match Pod's node affinity/selector" or "didn't match pod affinity rules." From there, cross-check kubectl get nodes --show-labels for nodeAffinity issues, or kubectl get pods -o wide -l to see whether any Pod matching the podAffinity labelSelector is even running, and in which topology domain.
Production example: A required podAffinity rule targeting app: redis-cache fails because the Redis Deployment was scaled to zero for a maintenance window, removing the only anchor Pod satisfying the rule.
Common mistake: Only checking node labels and missing that podAffinity failures stem from the absence of a matching Pod, not a node label problem at all.
9. What's the tradeoff between using podAffinity versus podAntiAffinity for the same Deployment?
Short answer: podAffinity pulls related Pods together to optimize for latency/cost between cooperating workloads; podAntiAffinity pushes replicas of the same workload apart to optimize for failure isolation.
Detailed explanation: They serve opposite goals and are typically applied to different relationships — podAffinity between a service and its dependency (web + cache), podAntiAffinity among replicas of the same Deployment so a single node or zone outage doesn't remove every replica simultaneously. Using podAffinity on replicas of the same Deployment would actively work against resilience by clustering them together.
Production example: A production Deployment combines podAffinity toward its Redis cache (same zone) with podAntiAffinity among its own replicas (different nodes) to get both proximity and resilience at once.
Common mistake: Confusing the two and applying podAffinity where podAntiAffinity was intended, accidentally concentrating all replicas onto the same node or zone.
10. Internally, how do the scheduler's Filter and Score plugins process affinity rules, and why does podAffinity evaluation require reading live cluster state?
Short answer: Filter plugins enforce required rules by eliminating non-conforming nodes; Score plugins evaluate preferred rules with their weights; podAffinity specifically needs the scheduler's informer cache of current Pod and Node labels since it depends on what's actually running, not static config.
Detailed explanation: nodeAffinity Filter/Score plugins only need to read Node object labels, which change rarely. podAffinity/podAntiAffinity plugins must additionally enumerate currently running Pods across the cluster (via the scheduler's informer cache backed by API Server watches) to find which nodes host Pods matching the labelSelector, then map those nodes to topology domains via topologyKey before filtering or scoring candidates.
Production example: In a large cluster, required podAffinity rules with a broad topologyKey (e.g. zone) are computationally cheaper than node-level ones because the scheduler needs to correlate fewer distinct topology domains.
Common mistake: Assuming affinity evaluation cost is negligible at scale — heavy use of podAffinity/podAntiAffinity across many Pods and nodes measurably increases scheduler latency.
11–50. Additional rapid-fire questions
11. What are the two rule families under spec.affinity? — nodeAffinity and podAffinity/podAntiAffinity.
12. What operators does nodeAffinity's matchExpressions support? — In, NotIn, Exists, DoesNotExist, Gt, Lt.
13. Does podAffinity support the same rich operator set as nodeAffinity? — Its labelSelector supports In, NotIn, Exists, DoesNotExist (matchExpressions), but not Gt/Lt.
14. What weight range is valid for preferredDuringSchedulingIgnoredDuringExecution entries? — 1 to 100.
15. Can multiple nodeSelectorTerms be specified under requiredDuringSchedulingIgnoredDuringExecution? — Yes, and they are OR'd together; expressions within one term are AND'd.
16. Does a Pod need any affinity rule at all to be scheduled? — No, affinity is entirely optional; without it the scheduler uses default Filter/Score plugins only.
17. What happens to a Pod with an unmet required nodeAffinity rule? — It stays in Pending state with a FailedScheduling event.
18. What happens to a Pod with an unmet preferred nodeAffinity rule? — It still gets scheduled, just potentially on a lower-scored node.
19. What field on podAffinity restricts the labelSelector match to specific namespaces? — namespaces (or namespaceSelector for label-based namespace matching).
20. What's the default namespace scope for a podAffinity labelSelector if namespaces is omitted? — The same namespace as the Pod being scheduled.
21. Is nodeAffinity evaluated against Node labels or Node annotations? — Node labels only.
22. Can nodeAffinity match on built-in labels like topology.kubernetes.io/zone? — Yes, any Node label including standard well-known labels.
23. Is requiredDuringSchedulingRequiredDuringExecution a real, GA field? — No, it has existed only as an alpha/experimental concept; the stable fields both use IgnoredDuringExecution.
24. What scheduler phase handles required rules — Filter or Score? — Filter (they eliminate candidate nodes before scoring).
25. What scheduler phase handles preferred rules — Filter or Score? — Score (they influence ranking of surviving candidates).
26. Why can heavy podAntiAffinity usage across many replicas increase scheduling latency? — Each additional replica adds more Pods the scheduler must cross-reference against the topology domains of every candidate node.
27. What's a simpler, less scheduler-intensive alternative to podAntiAffinity for spreading replicas evenly? — Pod Topology Spread Constraints (topologySpreadConstraints).
28. Can nodeAffinity and podAffinity rules be combined on the same Pod? — Yes, they're independent fields under spec.affinity and are all evaluated together.
29. What happens if required nodeAffinity and required podAffinity conflict such that no node satisfies both? — The Pod remains Pending — both are hard Filter constraints applied jointly.
30. How would you check which node labels are available before writing a nodeAffinity rule? — kubectl get nodes --show-labels.
31. How would you verify the affinity spec Kubernetes actually stored for a Pod? — kubectl get pod -o jsonpath='{.spec.affinity}'.
32. Why is topologyKey mandatory on podAffinity/podAntiAffinity but not on nodeAffinity? — nodeAffinity matches Node labels directly with no grouping concept; podAffinity needs a key to define what "nearby" means across nodes.
33. What's a typical topologyKey used for node-level (same-machine) co-location? — kubernetes.io/hostname.
34. What's a typical topologyKey used for zone-level co-location? — topology.kubernetes.io/zone.
35. Does changing a node's label after Pods are already scheduled retroactively evict Pods that no longer satisfy affinity? — No — "IgnoredDuringExecution" means already-running Pods are never re-evaluated or moved.
36. What is the risk of using required podAffinity as the very first workload deployed into a fresh namespace? — No anchor Pod exists yet to satisfy the labelSelector, so the Pod can deadlock Pending.
37. Would a preferred (not required) podAffinity rule cause the same first-deploy deadlock? — No — since it's a soft preference, the Pod still schedules even if no matching Pod exists yet.
38. What's the general guidance on defaulting to required vs preferred rules? — Default to preferred unless the constraint is truly non-negotiable, since required rules can strand Pods Pending.
39. Why should podAffinity typically use a broad topologyKey unless node-level co-location is essential? — It gives the scheduler more candidate nodes to choose from, reducing the chance of Pending Pods.
40. What complements nodeAffinity well for dedicating a node pool to specific workloads? — Taints and tolerations, used together with nodeAffinity rather than affinity alone.
41. Why is it good practice to test affinity rules in staging with realistic node labels before production rollout? — Label naming/values often differ across environments, and an untested rule can silently fail to match any node.
42. What's a pitfall of overusing required nodeAffinity for minor preferences? — It fragments available capacity and causes Pending Pods during normal node churn or scaling events.
43. Why must a topologyKey label be consistent across all relevant nodes? — If some nodes lack the label, the scheduler can't correlate them into a topology domain, effectively excluding them from evaluation.
44. How do you find which nodes currently host Pods matching a podAffinity labelSelector? — kubectl get pods -o wide -l to see their node placement.
45. What kubectl command surfaces affinity-related scheduling failures across the cluster? — kubectl get events --field-selector reason=FailedScheduling.
46. Does nodeAffinity get re-evaluated by the kubelet after the Pod is bound to a node? — No — once bound, the Pod runs regardless of subsequent node label changes.
47. What is the effect of a preferred rule's weight being higher relative to another preferred rule? — Nodes satisfying the higher-weighted preference get proportionally more added score during the Score phase.
48. Why might a Pod with only preferred rules still land on a node matching none of them? — All preferred rules are soft; if no node satisfies any, the Pod is still scheduled on the best-scoring node available by other criteria.
49. What's a good way to detect that a required affinity rule (not resource shortage) is the actual cause of Pending Pods? — The FailedScheduling event text explicitly references node/pod affinity mismatch rather than insufficient CPU/memory.
50. Why is affinity described as governing "initial placement only"? — Because both required and preferred variants are suffixed IgnoredDuringExecution — they're consulted once at scheduling time and never enforced again for a running Pod.
