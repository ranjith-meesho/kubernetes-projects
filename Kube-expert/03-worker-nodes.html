Worker Nodes

Where containers actually run: kubelet, kube-proxy, and the container runtime working together on every machine.
← Control Plane Course Index API Server →

Explanation

A worker node is any machine (VM or bare metal) that runs your actual workloads. Every worker node runs the same trio of agents: kubelet (the node agent that starts/stops containers and reports status), kube-proxy (programs node-local networking rules for Services), and a container runtime (containerd or CRI-O, which actually creates namespaces/cgroups and runs the container process). None of these talk to each other directly except through well-defined interfaces — kubelet drives the runtime via the CRI (Container Runtime Interface) gRPC API, and drives networking setup via the CNI (Container Network Interface) plugin.

Kubernetes needs this layer because the control plane only stores intent — nothing runs until some agent on a physical machine turns that intent into a real Linux process. Splitting "decide what to run" (control plane) from "actually run it" (worker node) is what allows the cluster to scale to thousands of machines: the control plane never needs to know how to start a container, it only needs to write "Pod X belongs on Node Y" and trust that Node Y's kubelet will do the rest.

Each worker node registers itself with the API Server as a Node object, advertising its capacity (CPU, memory, pods) and allocatable resources (capacity minus reserved system/kube overhead). The Scheduler reads this Node object when deciding placement. The node then sends periodic heartbeats — if these stop, the Node Controller (part of Controller Manager) marks it NotReady and eventually evicts its Pods for rescheduling elsewhere.

In production, worker nodes are typically organized into node pools/groups with different instance types (e.g. a general-purpose pool and a GPU pool), scaled by a cluster autoscaler that adds/removes whole nodes based on pending Pod pressure. Because kubelet, kube-proxy, and the runtime run as systemd services or static processes (not Kubernetes-managed Pods themselves, in most distros), node-level upgrades typically mean cordoning, draining, and replacing the node's OS image rather than in-place patching.

The health and correct configuration of these three agents is the single biggest source of "why is my pod stuck / slow / unreachable" production incidents — more so than control-plane issues in most clusters, simply because there are far more worker nodes than control-plane nodes and each one is an independent point of failure.

Internal Working / Interactions

  • kubelet ↔ API Server: watches for Pods bound to its node name; posts Node status/heartbeats every ~10s (configurable via --node-status-update-frequency).
  • kubelet ↔ Container Runtime: talks over the CRI gRPC socket (/run/containerd/containerd.sock) to create/stop containers and pull images.
  • kubelet ↔ CNI plugin: invokes the CNI binary to set up the Pod's network namespace and assign an IP.
  • kube-proxy ↔ API Server: watches Services/EndpointSlices, translates them into iptables/IPVS rules on the local node.
  • Node Controller (control plane) ↔ kubelet: monitors heartbeats; marks Node NotReady and triggers Pod eviction after the configured timeout if heartbeats stop.

Code Example

# kubelet systemd unit config — a real worker node has no "Node YAML" to apply;
# node registration/config is driven by kubelet's own startup flags/config file.
# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
address: 0.0.0.0
authentication:
  anonymous:
    enabled: false            # disallow unauthenticated API calls to kubelet itself
  webhook:
    enabled: true             # delegate authn/authz decisions to the API Server
clusterDNS:
  - 10.96.0.10                # CoreDNS ClusterIP, injected into every Pod's resolv.conf
clusterDomain: cluster.local
maxPods: 110                  # hard cap on pods this node will accept (affects allocatable calc)
evictionHard:
  memory.available: "200Mi"   # kubelet proactively evicts pods before the node truly OOMs
  nodefs.available: "10%"
cgroupDriver: systemd         # MUST match the container runtime's cgroup driver, a classic mismatch bug

kubectl Commands

# List all nodes with role, version, internal/external IP
kubectl get nodes -o wide

# Inspect one node's capacity, allocatable resources, and conditions
kubectl describe node worker-3

# See what's actually running on a given node
kubectl get pods -A --field-selector spec.nodeName=worker-3

# Safely take a node out of scheduling for maintenance
kubectl cordon worker-3
kubectl drain worker-3 --ignore-daemonsets --delete-emptydir-data
kubectl uncordon worker-3

# Check kubelet version reported by each node (skew vs API Server)
kubectl get nodes -o custom-columns=NAME:.metadata.name,KUBELET:.status.nodeInfo.kubeletVersion

Best Practices & Common Pitfalls

Best Practices

  • Match kubelet and container-runtime cgroup driver (systemd on modern distros) to avoid silent instability.
  • Set maxPods and reserve CPU/memory for system daemons (--system-reserved, --kube-reserved).
  • Always cordon + drain before maintenance rather than force-terminating a node.
  • Spread nodes across availability zones and use topology spread constraints for real resilience.

Pitfalls

  • Letting kubelet and runtime versions drift out of the supported skew window.
  • Ignoring disk pressure — a full node-fs triggers aggressive, disruptive eviction.
  • Assuming a NotReady node instantly reschedules Pods — there's a default eviction grace period (~5 min).
  • Running workloads directly on control-plane nodes in self-managed clusters.

50 Interview Questions

1. What three agents run on every worker node and what does each do?
Short answer: kubelet (node agent), kube-proxy (networking rules), and a container runtime (containerd/CRI-O).
Detailed explanation: kubelet starts/stops containers and reports status; kube-proxy programs iptables/IPVS rules for Services; the container runtime creates namespaces/cgroups and runs the actual process. None talk to each other directly — kubelet drives the runtime via CRI and networking via CNI.
Production example: Diagnosing a "Pod stuck Pending on this node" issue means checking kubelet logs; diagnosing "Service unreachable" means checking kube-proxy rules instead.
Common mistake: Assuming kube-proxy or the runtime can independently decide what runs — only kubelet decides that, driven by the API Server.
2. Why does the control plane never need to know how to start a container?
Short answer: That responsibility is entirely delegated to worker-node kubelets, keeping the control plane's job limited to writing intent.
Detailed explanation: The control plane only writes "Pod X belongs on Node Y" to etcd; kubelet on Node Y independently notices this and does all the real work. This split is what lets the cluster scale to thousands of machines without the control plane becoming a bottleneck.
Production example: Scheduler binds a Pod in milliseconds regardless of how long the image pull or container startup takes afterward — those are fully decoupled.
Common mistake: Believing the Scheduler or API Server "creates" the container — they only ever record a decision.
3. How does a Node register itself and what does the Scheduler read from it?
Short answer: Each worker node registers as a Node object advertising capacity and allocatable resources.
Detailed explanation: Allocatable resources are capacity minus reserved system/kube overhead (--system-reserved/--kube-reserved); the Scheduler reads this Node object during filtering/scoring to decide if a Pod fits.
Production example: kubectl describe node worker-3 shows both Capacity and Allocatable — the Scheduler only reasons about the latter.
Common mistake: Forgetting reserved resources exist and being surprised the node has less schedulable capacity than raw hardware specs suggest.
4. What happens when a node stops sending heartbeats?
Short answer: The Node Controller marks it NotReady and eventually evicts its Pods for rescheduling elsewhere.
Detailed explanation: kubelet posts heartbeats roughly every 10s (--node-status-update-frequency); after a grace period with no heartbeat, Node Controller applies a NoExecute taint and, after the pod-eviction-timeout, evicts Pods lacking a matching toleration.
Production example: A network partition between a worker node and the control plane shows as NotReady within ~40s, with eviction following after ~5 minutes by default.
Common mistake: Expecting instant rescheduling the moment a node goes NotReady — there's a deliberate grace period to avoid overreacting to transient blips.
5. Why must kubelet's cgroup driver match the container runtime's exactly?
Short answer: Both must manage the same cgroup hierarchy consistently, or resource enforcement becomes unstable.
Detailed explanation: kubelet's cgroupDriver: systemd setting must match containerd's SystemdCgroup = true option — a mismatch causes subtle, hard-to-diagnose instability (double cgroup hierarchies, resource limits not applied correctly).
Production example: After migrating from cgroup v1 to v2, forgetting to align both configs causes containers to silently ignore CPU/memory limits.
Common mistake: Changing one side (e.g. upgrading the runtime) without verifying the other still matches.
6. What is maxPods and why does it matter for allocatable calculation?
Short answer: A hard cap on how many Pods a node will accept, which factors into how the Scheduler and cluster-autoscaler reason about node capacity.
Detailed explanation: Even if CPU/memory technically allow more Pods, kubelet refuses to exceed maxPods (default 110). This is especially relevant on cloud nodes where IP-per-Pod networking (e.g. AWS VPC CNI) further constrains real Pod density below the default.
Production example: An EKS node type with limited ENI/IP capacity hitting its practical Pod limit well before maxPods or CPU/memory limits are reached.
Common mistake: Sizing nodes purely by CPU/memory and being surprised Pods won't schedule due to the Pod-count or IP-address ceiling.
7. Why is cordon + drain the correct way to take a node out of service?
Short answer: Cordon stops new scheduling; drain gracefully evicts existing Pods, respecting PodDisruptionBudgets and graceful termination.
Detailed explanation: kubectl cordon marks the node unschedulable without touching running Pods; kubectl drain --ignore-daemonsets --delete-emptydir-data then evicts Pods one by one, letting controllers reschedule them elsewhere safely.
Production example: Rolling OS-image upgrades across a node pool: cordon, drain, replace, uncordon, one node at a time.
Common mistake: Force-terminating or deleting a node directly instead of draining first, causing ungraceful Pod termination and potential PDB violations.
8. Why are node-level upgrades typically "replace the node" rather than "patch in place"?
Short answer: kubelet, kube-proxy, and the runtime run as systemd services/host processes, not Kubernetes-managed Pods, so in-place upgrades are harder to automate safely at scale.
Detailed explanation: Cloud node pools typically use immutable machine images — a new node image with upgraded components is rolled out, and old nodes are cordoned/drained/terminated rather than patched live.
Production example: EKS managed node groups perform "replace" upgrades: new nodes join with the new AMI, old nodes drain and terminate.
Common mistake: Trying to live-upgrade kubelet via config management across a fleet instead of using the cordon-drain-replace pattern, risking partial, inconsistent states.
9. Why do worker-node issues cause more production incidents than control-plane issues in most clusters?
Short answer: There are far more worker nodes than control-plane nodes, and each is an independent point of failure.
Detailed explanation: Misconfiguration of any single node's kubelet, kube-proxy, or runtime (cgroup mismatch, disk pressure, PLEG issues) only affects that node's Pods, but such issues are individually far more numerous and varied than centralized control-plane failures.
Production example: A single node hitting disk pressure triggers aggressive local eviction, disrupting only Pods scheduled there.
Common mistake: Focusing all monitoring effort on control-plane health while under-monitoring per-node kubelet/runtime health.
10–50. Additional rapid-fire questions
10. What interface does kubelet use to talk to the container runtime? — CRI (Container Runtime Interface), a gRPC API.
11. What interface does kubelet use to set up Pod networking? — CNI (Container Network Interface) plugins.
12. What's the default node heartbeat interval? — Roughly every 10 seconds, configurable via --node-status-update-frequency.
13. What's the difference between Capacity and Allocatable on a Node? — Capacity is total hardware resources; Allocatable is Capacity minus reserved system/kube overhead, which the Scheduler actually uses.
14. Which component evicts Pods off a NotReady node? — The Node Controller, part of the Controller Manager.
15. Why does kube-proxy run on every node instead of centrally? — Routing decisions must be applied inline per-packet at line rate, which a centralized service can't do fast enough.
16. What's the danger of ignoring disk pressure on a node? — kubelet triggers aggressive, disruptive eviction of Pods to reclaim space once nodefs.available thresholds are breached.
17. How do you check a node's kubelet version vs the API Server's? — kubectl get nodes -o custom-columns=...KUBELET:.status.nodeInfo.kubeletVersion compared against kubectl version.
18. What's a node pool/group used for? — Grouping nodes by instance type/purpose (e.g. general-purpose vs GPU) so different workloads land on appropriately-provisioned hardware.
19. Why does the cluster-autoscaler add/remove whole nodes rather than resize existing ones? — Cloud instances aren't elastically resizable in place at the speed needed; adding/removing whole nodes based on pending-Pod pressure is the practical mechanism.
20. What does evictionHard: memory.available: "200Mi" do? — Proactively evicts Pods once available memory drops below 200Mi, before the node truly OOMs.
21. Why should you avoid running workloads on control-plane nodes in self-managed clusters? — Security isolation and to avoid noisy-neighbor resource contention affecting etcd/API Server.
22. What's the risk of letting kubelet and runtime versions drift out of supported skew? — Undefined or broken behavior in CRI calls, subtle bugs that are hard to reproduce and diagnose.
23. How do topology spread constraints relate to worker nodes? — They instruct the Scheduler to spread Pods across node failure domains (zones) rather than clustering them for real resilience.
24. What field on the Pod tells you which worker node it's running on? — spec.nodeName, set once by the Scheduler and never changed afterward.
25. Why can't a Pod be "moved" to another node? — Pods are bound permanently once scheduled; a controller creates an entirely new Pod object elsewhere if the original dies.
26. What's the purpose of --kube-reserved? — Reserves resources for Kubernetes system daemons (kubelet, kube-proxy) so they aren't starved by user workloads.
27. How does kubelet find out a Pod is assigned to its node? — It watches the API Server for Pods with spec.nodeName matching its own node name.
28. What is the practical effect of a full node filesystem? — Aggressive Pod eviction and potential inability to pull new images or write container logs.
29. Why is --ignore-daemonsets needed during drain? — DaemonSet Pods are meant to run on every node including this one; drain would otherwise fail trying to evict Pods that will just be recreated.
30. What's a symptom that a worker node's kube-proxy is misconfigured? — Services resolve/route inconsistently or not at all from Pods on that specific node, while other nodes work fine.
31. How would you find all Pods currently running on a specific node? — kubectl get pods -A --field-selector spec.nodeName=worker-3.
32. What does it mean that kubelet, kube-proxy, and the runtime are "not Kubernetes-managed Pods"? — They typically run as systemd services or host processes bootstrapping the node itself, not objects the Scheduler places.
33. Why might a Pod become stuck in ContainerCreating on a specific node? — Often CNI plugin failures, image pull issues, or CRI socket problems local to that node's runtime.
34. What's the relationship between Node conditions and taints? — Certain conditions (e.g. MemoryPressure, DiskPressure) can automatically apply corresponding taints that repel new scheduling.
35. Why is worker-node capacity planning about more than raw CPU/memory? — Pod density is also limited by maxPods, IP address availability (CNI), and reserved resources.
36. What's the value of spreading worker nodes across availability zones? — Tolerates an entire AZ failure without losing all capacity, assuming topology spread constraints are also used.
37. How does a worker node's kubelet contribute to cluster-wide DNS resolution? — It injects the configured clusterDNS IP (CoreDNS ClusterIP) into every Pod's resolv.conf on that node.
38. What's the risk of uncordoning a node too early after maintenance? — New Pods may schedule onto it before the underlying issue is confirmed fixed, causing repeat failures.
39. Why does a NotReady node not instantly lose all its Pods? — There's a default eviction grace period (pod-eviction-timeout, ~5 min) to tolerate transient network blips without unnecessary reschedule storms.
40. What tool would you use to debug the container runtime directly on a worker node, bypassing kubectl? — crictl, the CRI-level CLI.
41. Why is per-node monitoring of kubelet resource usage important? — A starved kubelet delays every Pod lifecycle operation (start, probe, status report) on that entire node.
42. What's a common root cause when only one node in a cluster has slow Pod startup? — Local image pull bottlenecks, disk I/O contention, or that node's specific runtime/CNI misconfiguration.
43. Why does the Scheduler need up-to-date Node objects, not stale ones? — Placing a Pod based on stale capacity data risks overcommitting a node that's actually already full.
44. What's the effect of setting maxPods too low on a large-capacity node? — Wastes CPU/memory capacity because the Pod-count ceiling is hit before resource limits are.
45. How does worker-node design support horizontal scalability of the whole cluster? — Adding more worker nodes linearly adds capacity without any control-plane component needing to know how containers actually run.
46. What's a good first troubleshooting step for "Pod runs fine on some nodes but not others"? — Compare kubelet/runtime versions and cgroup driver configuration across the affected nodes.
47. Why should GPU workloads typically run in a separate node pool? — Different hardware/driver requirements and cost profile than general-purpose nodes, and to avoid GPU nodes being wasted on non-GPU workloads.
48. What happens to DaemonSet Pods during a drain? — By default drain would try to evict them too; --ignore-daemonsets skips them since they'll simply be recreated by their controller.
49. Why is "kubectl describe node" often the first command in node troubleshooting? — It surfaces Capacity/Allocatable, Conditions, Taints, and recent Events in one place, covering most common node-level failure signatures.
50. What's the risk of scaling worker nodes without adjusting cluster-autoscaler settings? — Manual node additions/removals can conflict with autoscaler decisions, causing thrash or nodes being removed unexpectedly.
