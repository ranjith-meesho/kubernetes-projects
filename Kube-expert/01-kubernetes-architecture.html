1. Introduction

What it is

Kubernetes architecture is the overall design of the system: a control plane that makes global decisions about the cluster (scheduling, detecting and responding to events) and a set of worker nodes that actually run your containers. Every other Kubernetes concept — Pods, Deployments, Services, RBAC — is built on top of this architecture, so understanding it deeply is the single highest-leverage thing you can learn.

At the highest level, Kubernetes is a declarative, distributed control system. You tell it the desired state of your application ("I want 3 replicas of this container, exposed on port 80"), and a set of independent control loops continuously work to make the actual state of the cluster match that desired state.

Why Kubernetes needs it

Running containers on a single Docker host is easy. Running hundreds of containers across hundreds of machines, surviving hardware failure, rolling out updates without downtime, and scaling elastically is not. Kubernetes needs a clear separation between:

  • The brain (control plane): stores desired state, decides what should run where.
  • The muscle (worker nodes): executes containers and reports back actual state.

Without this separation you cannot achieve self-healing, horizontal scalability, or safe multi-tenant operation.

Why it exists

Kubernetes descends from Google's internal systems Borg and Omega, which ran Google's production workloads for over a decade before Kubernetes was open-sourced in 2014. The architecture encodes hard-won lessons: keep the control plane stateless and horizontally scalable, keep a single source of truth (etcd), and make every component reconcile against that source of truth independently rather than talking to each other directly. This is what allows Kubernetes to tolerate component crashes, network partitions, and rolling upgrades without losing consistency.

Real-world analogy

Airport control tower analogy: The control plane is the control tower — it doesn't fly planes, but it knows the desired schedule, tracks every plane's actual position, and issues instructions. The worker nodes are the runways and ground crews — they do the physical work of taxiing, fueling, and boarding based on instructions, and report status back to the tower. If a runway (node) goes down, the tower reroutes flights to other runways. The tower itself is replicated (multiple controllers, redundant systems) so no single failure grounds the airport.

Production use case

A payments company runs 400 microservices across 60 nodes in 3 availability zones. The control plane schedules pods away from failing nodes automatically, the API server enforces RBAC so only the payments team can touch payment services, and controllers restart crashed containers within seconds — all without a human paging at 3 AM for routine failures. This is architecture doing its job: turning individual machine failures into non-events.

2. Internal Working

Every action in Kubernetes — kubectl apply, a node dying, a pod crashing — flows through the same basic pattern: write intent to etcd via the API server, then have independent controllers reconcile reality towards that intent. Let's trace it step by step.

Step-by-step: what happens when you run kubectl apply -f pod.yaml

  1. kubectl reads the YAML, converts it to JSON, and sends an HTTPS request to the API Server (authenticated via your kubeconfig's client cert or token).
  2. API Server runs the request through a pipeline: Authentication (who are you?) → Authorization (RBAC — are you allowed to create Pods in this namespace?) → Admission Control (mutating webhooks may inject sidecars; validating webhooks may reject the request; resource quotas are checked).
  3. If it passes, the API Server persists the object to etcd as the single source of truth. Nothing is considered "real" until it is in etcd.
  4. The API Server returns success to kubectl — but at this point no container is running yet. Only the desired state exists.
  5. The Scheduler is watching the API Server (via a watch/informer, not polling) for Pods with an empty spec.nodeName. It picks one up, runs filtering (which nodes satisfy resource/affinity/taint requirements) and scoring (which node is the best fit), then writes the chosen node name back to the Pod object via the API Server, which persists it to etcd.
  6. The kubelet on that specific node is also watching the API Server for Pods bound to its node name. It notices the new assignment and begins the actual work: pulling the image, creating the sandbox (pause container), and invoking the Container Runtime (via the CRI) to start the containers.
  7. kube-proxy on every node watches Services/Endpoints and updates iptables/IPVS rules so traffic can reach the new Pod once it's ready.
  8. The kubelet continuously reports the Pod's actual status (Running, Ready, container statuses) back to the API Server, which updates etcd. This is the "actual state" half of the reconciliation loop.
  9. Various Controllers in the Controller Manager (e.g. ReplicaSet controller) are simultaneously watching to make sure the number of running Pods matches the desired replica count, creating or deleting Pods as needed.

Which components interact, and how

ComponentInteraction
API ServerThe only component that talks to etcd directly. Every other component talks only to the API Server, never to etcd or to each other directly.
etcdDistributed, strongly-consistent key-value store. Holds every object's desired and last-known-actual state. Uses the Raft consensus algorithm for leader election and write consistency.
SchedulerWatches for unscheduled Pods, writes back nodeName via the API Server.
kubeletWatches for Pods assigned to its node, drives the container runtime via CRI, reports status back.
kube-proxyWatches Services/EndpointSlices, programs node-level networking rules.
Controller ManagerRuns dozens of independent control loops (Deployment, ReplicaSet, Node, Job, etc.), each reconciling one resource type toward its desired state.
Key architectural principle — the "watch, don't poll" pattern: Every controller and kubelet uses a long-lived HTTP watch connection to the API Server, backed by a local cache (informer). This means Kubernetes scales to thousands of nodes without the API Server being hammered by constant polling, and every component reacts to changes within milliseconds instead of on a fixed poll interval.

Complete request flow (text form)

kubectl apply -f pod.yaml
   │
   ▼
API Server ──(authn/authz/admission)──▶ etcd (desired state persisted)
   │
   ▼ (scheduler watch fires)
Scheduler ──(filter + score nodes)──▶ API Server ──▶ etcd (nodeName bound)
   │
   ▼ (kubelet watch fires on target node)
kubelet ──▶ CRI ──▶ Container Runtime ──▶ Pod running
   │
   ▼
kubelet reports status ──▶ API Server ──▶ etcd (actual state updated)
   │
   ▼
Controllers reconcile (ReplicaSet/Deployment controllers compare desired vs actual)

4. YAML Deep Dive

Architecture itself has no single YAML manifest (it's the system, not a resource) — but every object you'll ever write obeys the same top-level envelope, because it's enforced by the API Server's schema validation. Understanding this envelope IS understanding how the architecture represents state.

apiVersion: v1        # Which API group/version this object belongs to.
                       # The API Server exposes many "API groups" (core "v1", apps/v1, batch/v1...);
                       # this tells the API Server which schema + endpoint to validate/route to.
kind: Pod              # The resource type. Combined with apiVersion, uniquely identifies the REST endpoint,
                       # e.g. apps/v1 Deployment -> /apis/apps/v1/namespaces/{ns}/deployments
metadata:              # Identity + bookkeeping data. NOT part of desired behavior.
  name: my-pod          # Unique within a namespace. Immutable after creation.
  namespace: default     # Logical partition of the cluster (RBAC, quotas, network policies scope to this).
  labels: {}             # Arbitrary key/value used for selection (Services, ReplicaSets match on these).
  annotations: {}        # Arbitrary key/value NOT used for selection — for tooling/metadata only.
  uid:                   # Server-generated globally unique ID (you never set this).
  resourceVersion:       # Server-generated optimistic-concurrency token (etcd's internal revision marker).
spec:                  # Desired state — the only part you're really "writing" as a human.
  ...                    # Structure is entirely defined by kind (Pod spec != Deployment spec).
status:                # Actual/observed state — written ONLY by the system (kubelet, controllers), never by you.
  ...                    # This is what makes Kubernetes "declarative": spec is your intent, status is reality,
                         # and controllers exist solely to shrink the diff between the two to zero.
Why this matters architecturally: the fact that spec and status are separate subresources (with separate RBAC rules and separate API endpoints — /status) is a deliberate design choice. It lets kubelet be granted permission to update only status, never spec, closing off a whole class of privilege-escalation bugs.

5. YAML Examples

Basic — a single Pod showing the architecture's object envelope

apiVersion: v1
kind: Pod
metadata:
  name: nginx-basic
  namespace: default
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80

Intermediate — Pod with resource requests/limits (Scheduler + kubelet both read this)

apiVersion: v1
kind: Pod
metadata:
  name: nginx-intermediate
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      resources:
        requests:          # Scheduler uses this to decide WHICH node has room.
          cpu: "250m"
          memory: "128Mi"
        limits:            # kubelet/runtime enforces this via cgroups on the node.
          cpu: "500m"
          memory: "256Mi"
      readinessProbe:       # kubelet uses this to decide when kube-proxy should route traffic here.
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 5

Advanced — Pod pinned via node affinity + toleration (Scheduler filtering in action)

apiVersion: v1
kind: Pod
metadata:
  name: nginx-advanced
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values: ["ap-south-1a"]
  tolerations:
    - key: "dedicated"
      operator: "Equal"
      value: "frontend"
      effect: "NoSchedule"
  containers:
    - name: nginx
      image: nginx:1.25

Enterprise — full production Deployment exercising the whole architecture

apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service
  namespace: payments
  labels:
    app: checkout-service
    team: payments
spec:
  replicas: 6                       # ReplicaSet controller in Controller Manager enforces this count.
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2
      maxUnavailable: 0             # Zero-downtime: Deployment controller orchestrates this.
  selector:
    matchLabels:
      app: checkout-service
  template:
    metadata:
      labels:
        app: checkout-service
    spec:
      serviceAccountName: checkout-sa   # Identity presented to the API Server for RBAC checks.
      containers:
        - name: checkout
          image: registry.internal/checkout-service:2.4.1
          resources:
            requests: { cpu: "500m", memory: "512Mi" }
            limits: { cpu: "1", memory: "1Gi" }
          readinessProbe:
            httpGet: { path: /healthz, port: 8080 }
          livenessProbe:
            httpGet: { path: /healthz, port: 8080 }
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels: { app: checkout-service }

6. kubectl Commands

# Inspect the control plane itself
kubectl get componentstatuses          # (deprecated in newer versions, but still common in exam contexts)
kubectl get nodes -o wide              # See every worker node, its role, version, internal/external IP
kubectl cluster-info                   # Shows API Server and core DNS addresses
kubectl version --short                # Client vs server version skew — critical for upgrade planning

# Watch the reconciliation loop happen live
kubectl get pods -w                    # Watch Pod status transition Pending -> ContainerCreating -> Running
kubectl get events --sort-by=.lastTimestamp   # See Scheduler + kubelet events in order

# Inspect how an object flows through the architecture
kubectl describe pod my-pod            # Shows Node assigned, Events (scheduling decision, image pull, etc.)
kubectl get pod my-pod -o yaml         # Full object including status subresource — see actual vs desired
kubectl explain pod.spec               # Pulls live schema from the API Server's OpenAPI definitions

Output explained: kubectl describe pod shows an Events section at the bottom with entries like Scheduled, Pulling, Pulled, Created, Started — this is literally the internal working section above, made visible. Each event was emitted by a different component (Scheduler emits Scheduled; kubelet emits the rest).

8. Best Practices

Do's

  • Run an odd number of etcd/control-plane replicas (3 or 5) for quorum.
  • Separate control plane nodes from worker nodes in production (never co-locate untrusted workloads with etcd).
  • Keep API Server, kubelet, and etcd versions within the supported skew (kubelet no more than 2 minor versions behind API Server).
  • Use managed control planes (EKS/GKE/AKS) unless you have a dedicated platform team — self-managing etcd correctly is genuinely hard.

Don'ts

  • Don't run an even number of etcd members — it doesn't improve fault tolerance and increases split-brain risk.
  • Don't bypass the API Server (e.g. editing etcd directly) — it skips admission control and validation entirely.
  • Don't scale the Scheduler/Controller Manager horizontally naively — they use leader election, only one instance is active at a time.

Common mistakes

  • Assuming Kubernetes is "one binary" — mixing up control plane and node component responsibilities during troubleshooting.
  • Not budgeting etcd disk I/O — etcd is extremely latency-sensitive and a slow disk degrades the entire cluster's API responsiveness.

Cost optimization

On managed platforms, the control plane is often billed flat-rate (e.g. EKS ~$0.10/hr) regardless of node count — so consolidating many small clusters into fewer, well-namespaced clusters reduces control-plane overhead cost significantly at scale.

9. Security Considerations

  • API Server is the sole trust boundary — every authn/authz/admission decision funnels through it; a misconfigured RBAC rule or an overly permissive admission webhook compromises the whole cluster.
  • etcd holds Secrets in plaintext by default unless encryption-at-rest is configured — anyone with etcd access effectively has cluster-admin.
  • Node-to-control-plane traffic must be encrypted (TLS) — kubelet certificates should be per-node and rotated automatically (kubelet TLS bootstrapping).
  • Compliance: PCI-DSS/SOC2 audits typically require etcd encryption at rest, audit logging enabled on the API Server, and network isolation of control-plane nodes.

10. Performance Optimization

  • CPU: API Server and etcd are CPU-sensitive under high request/write volume — isolate them on dedicated instance types in self-managed clusters.
  • Memory: Scheduler and Controller Manager cache full cluster state in memory (via informers) — memory scales with object count, not just node count.
  • Networking: etcd is extremely latency-sensitive to network jitter between members — keep etcd members in low-latency proximity (same region, ideally same low-latency AZ group).
  • Storage: etcd requires fast, low-latency SSD (fsync-heavy) — spinning disks or network storage with high latency will degrade the entire cluster.
  • Scheduling: Large clusters benefit from scheduler profiles and reduced percentageOfNodesToScore to bound scheduling latency.
  • Scaling: The control plane itself has scaling limits (etcd write throughput, API Server watch fan-out) — this is why Kubernetes officially supports up to 5,000 nodes per cluster, not unlimited.

11. Troubleshooting

Issue: API Server unreachable

Symptoms: kubectl commands hang or return "connection refused".

Root Cause: API Server pod/process down, network/firewall issue, or load balancer in front of a multi-master setup misconfigured.

Diagnosis: kubectl get --raw='/healthz', check API server pod logs on control plane node, check LB target health.

Fix: Restart API Server process/pod; verify certs haven't expired; verify etcd is reachable from API Server.

Issue: etcd quorum lost

Symptoms: Writes fail cluster-wide; reads may still work from a stale follower.

Root Cause: More than (N-1)/2 etcd members down simultaneously.

Diagnosis: etcdctl endpoint status --cluster, check member health across all etcd nodes.

Fix: Restore failed members, or restore from snapshot if permanently lost; never force a single-node "quorum" workaround in production without understanding the data-loss risk.

Issue: Pods stuck Pending due to scheduler not running

Symptoms: All new Pods stay Pending with no nodeName assigned.

Root Cause: Scheduler pod crashed or lost leader election.

Diagnosis: kubectl get pods -n kube-system | grep scheduler, check leader election lease: kubectl get lease -n kube-system.

Fix: Restart scheduler; check for CPU starvation or panic in logs.

12. 50 Interview Questions

1. What are the two main planes in Kubernetes architecture?
Short answer: Control plane and data plane (worker nodes).
Detail: The control plane makes global decisions (scheduling, API, state storage); worker nodes run workloads.
Production example: In EKS, AWS manages the control plane; you manage worker node groups.
Common mistake: Assuming worker nodes make scheduling decisions — they never do.
2. Why does Kubernetes use etcd instead of a relational database?
Short answer: etcd provides strong consistency via Raft consensus and a watch API, which relational DBs don't natively provide.
Detail: Kubernetes needs every component to react instantly to changes (watch) and needs guaranteed consistency for leader election and scheduling decisions.
Production example: Multi-master clusters rely on etcd's Raft leader election to avoid split-brain.
Common mistake: Trying to swap etcd for another datastore without understanding the consistency guarantees required.
3. What happens if the API Server goes down but workloads are already running?
Short answer: Existing Pods keep running; nothing new can be scheduled/changed.
Detail: kubelet keeps running its already-known Pods independently — it doesn't need the API Server for a Pod it already started.
Production example: A brief control-plane outage during an EKS upgrade doesn't crash running services.
Common mistake: Believing the whole cluster "goes down" — only the control loop (changes) stops.
4. What is the role of the Scheduler in the architecture?
Short answer: Assigns Pods to Nodes based on resource availability and constraints.
Detail: Filters nodes that can't run the pod, then scores remaining nodes to pick the best fit.
Production example: Bin-packing workloads onto fewer nodes to enable cluster-autoscaler scale-down.
Common mistake: Thinking the scheduler starts the container — it only assigns the node; kubelet starts it.
5. Why is the architecture "declarative" rather than "imperative"?
Short answer: You declare desired state; controllers continuously reconcile actual state to match it.
Detail: This makes the system self-healing and idempotent — reapplying the same YAML is always safe.
Production example: GitOps tools like Argo CD rely entirely on this property.
Common mistake: Writing imperative scripts (kubectl run, kubectl edit) as the primary deployment mechanism in production.
6. What is a control loop / reconciliation loop?
Short answer: A continuous process that compares desired vs actual state and takes action to close the gap.
Detail: Every controller (Deployment, ReplicaSet, Node) runs its own independent loop.
Production example: ReplicaSet controller recreating a Pod within seconds of a node failure.
Common mistake: Assuming there's one global reconciliation loop instead of many independent ones.
7. Why does kubelet report status back to the API Server instead of the API Server polling kubelet?
Short answer: Push-based reporting scales better and reduces control-plane load.
Detail: kubelet posts status updates (heartbeats + full status) on its own schedule.
Production example: Node NotReady detection after missed heartbeats within ~40s default.
Common mistake: Confusing node heartbeat interval with pod eviction timeout — they're separate settings.
8. What is the Cloud Controller Manager and why was it split out?
Short answer: It runs cloud-provider-specific logic (load balancers, node lifecycle) separately from core Controller Manager.
Detail: Splitting it out let cloud vendors ship/version their integration independently of core Kubernetes releases.
Production example: AWS CCM provisions an ELB when you create a LoadBalancer-type Service.
Common mistake: Assuming core Kubernetes talks to cloud APIs directly — it never does, only CCM does.
9. How does Kubernetes achieve high availability of the control plane?
Short answer: Multiple API Server replicas behind a load balancer, odd-numbered etcd cluster, leader election for Scheduler/Controller Manager.
Detail: API Servers are stateless and can all be active; only Scheduler/CM have a single active leader at a time.
Production example: A 3-master kubeadm cluster with a keepalived VIP in front of the API Servers.
Common mistake: Running 2 control-plane nodes — provides no quorum benefit over 1 and is worse than 3.
10. What is admission control and where does it sit in the flow?
Short answer: A pluggable step in the API Server pipeline after authn/authz, before persistence to etcd.
Detail: Mutating webhooks can modify the object (inject sidecars); validating webhooks can reject it.
Production example: Istio injects sidecar containers via a mutating admission webhook.
Common mistake: Forgetting mutating webhooks run before validating ones — order matters.
11–50. Additional rapid-fire questions
11. What's the difference between kube-apiserver and kubectl? — kubectl is a client CLI; API Server is the server all clients talk to.
12. Can you have zero control-plane nodes and still run Pods? — Existing Pods keep running briefly, but nothing self-heals or reschedules.
13. What is a static pod and how does it relate to bootstrapping the control plane? — kubelet runs manifests from a local directory without the API Server, used to bootstrap API Server/etcd/scheduler themselves.
14. Why is etcd typically not exposed outside the control plane network? — It has no built-in fine-grained authz; anyone with access can read all Secrets.
15. What object stores the desired replica count for a Deployment? — The Deployment object's spec.replicas, enforced via its owned ReplicaSet.
16. How does the architecture support multi-tenancy? — Namespaces + RBAC + ResourceQuotas + NetworkPolicies, all enforced at the API Server/CNI layer.
17. What is the API Server's aggregation layer? — Lets you register additional API servers (e.g. metrics-server) that appear under the same API surface.
18. Why does Kubernetes prefer level-triggered over edge-triggered reconciliation? — Level-triggered (compare full desired vs actual state) is resilient to missed events; edge-triggered would lose state on a dropped message.
19. What is the purpose of resourceVersion on objects? — Optimistic concurrency control to prevent lost updates on concurrent writes.
20. How does the architecture isolate a compromised node? — Cordoning/draining removes it from scheduling; NetworkPolicies and node-level RBAC limit blast radius.
21. What is the kubelet's PLEG? — Pod Lifecycle Event Generator, the mechanism kubelet uses to detect container state changes efficiently.
22. Why is DNS (CoreDNS) considered part of the "architecture" even though it's an add-on? — It's a cluster-critical add-on that most workloads depend on for service discovery.
23. What does "conformant Kubernetes" mean? — Passes the CNCF conformance test suite guaranteeing consistent API behavior across vendors (EKS/GKE/AKS/on-prem).
24. How do you upgrade a cluster's control plane safely? — Upgrade API Server first, then Controller Manager/Scheduler, then kubelet, respecting version skew policy.
25. What is the maximum supported version skew between kubelet and API Server? — kubelet can be up to 3 minor versions older than API Server (policy varies by release; historically 2).
26. What's the difference between liveness and the architecture's self-healing? — Liveness probes are a signal kubelet uses locally; self-healing at the ReplicaSet level is a separate, higher-level control loop.
27. Why does Kubernetes use a REST API rather than RPC? — Uniform, cacheable, well-understood transport that makes tooling (kubectl, clients in any language) trivial to build.
28. What is a finalizer and how does it interact with the architecture's deletion flow? — A marker that blocks object deletion from completing in etcd until a controller finishes cleanup.
29. How does horizontal scaling of the API Server work? — Stateless replicas behind a load balancer, all reading/writing the same etcd cluster.
30. What is the role of informers/listers in client-go? — Local cache + event dispatch layer so controllers don't re-list the full API Server on every reconcile.
31. Why can't the Scheduler and Controller Manager run multiple active replicas simultaneously? — They'd race to make conflicting decisions; leader election ensures exactly one active instance.
32. What happens to Pods on a node that's been NotReady for 5 minutes? — Default pod-eviction-timeout triggers eviction and rescheduling elsewhere (if tolerations allow).
33. How is etcd backed up? — etcdctl snapshot save, ideally automated and stored off-cluster.
34. What is the audit log and where is it generated? — API Server-level log of every request, used for security/compliance review.
35. Why does Kubernetes separate "spec" and "status" as different subresources? — Enables separate RBAC (kubelet can write status without write access to spec).
36. What's the blast radius of a compromised admission webhook? — Potentially cluster-wide — it can mutate or block any matching resource across namespaces.
37. How does Kubernetes support extension via CRDs at the architecture level? — API Server dynamically serves new resource types registered via CustomResourceDefinition, no core code change needed.
38. What is the purpose of the kube-scheduler's "preemption" feature? — Evicts lower-priority Pods to make room for higher-priority pending Pods.
39. Why do managed Kubernetes offerings (EKS/GKE/AKS) hide the control plane? — Reduces operational burden and attack surface for customers; provider guarantees SLAs on it.
40. What is a taint vs a node condition? — Taints are explicit scheduling repellents you set; conditions (e.g. MemoryPressure) are observed states that can auto-apply taints.
41. How does the architecture prevent thundering herd on API Server restarts? — Client-go's informer resync + exponential backoff on watch reconnects.
42. What is the significance of API groups/versions (v1, apps/v1, batch/v1)? — Enables safe API evolution — a resource can move from beta to stable without breaking existing clients immediately.
43. Why is etcd typically limited to a small, odd cluster size (3–5) rather than scaled like stateless services? — Raft consensus overhead grows with member count; more members means slower writes, not more capacity.
44. How does the architecture support zero-downtime control plane upgrades? — Rolling replacement of API Server replicas behind a load balancer while etcd quorum is maintained.
45. What is the relationship between a Deployment, ReplicaSet, and Pod in terms of controllers? — Deployment controller manages ReplicaSets; ReplicaSet controller manages Pods — a chain of owner references.
46. Why does kube-proxy run on every node instead of centrally? — Traffic routing decisions must be made locally and fast, without a control-plane round trip per packet.
47. What is the purpose of the "watch bookmark" feature in the API? — Reduces resource usage on reconnect by letting clients resume from a known resourceVersion instead of a full re-list.
48. How does architecture handle a network partition between a node and the control plane? — Node marked NotReady after missed heartbeats; kubelet keeps running existing Pods but can't get new instructions.
49. Why is "everything is an API object" a core architectural philosophy? — Uniform tooling, RBAC, and extension model — Nodes, Pods, Secrets, and CRDs are all accessed the same way.
50. What's the first thing to check when the whole cluster seems "frozen" (nothing reconciling)? — Controller Manager and Scheduler leader election lease/logs — a stuck leader election halts almost all reconciliation.

13. Practical Labs

Beginner Labs

  1. Run kubectl get nodes -o wide and identify which node is control-plane vs worker using labels/taints.
  2. Create a Pod via YAML, then run kubectl describe pod and map each Event line to the component that emitted it.
  3. Run kubectl get pods -w in one terminal and apply a Pod in another — observe status transitions live.
  4. Use kubectl explain pod.spec.containers to explore the schema served live by the API Server.
  5. Delete a Pod and observe how fast a ReplicaSet-backed Deployment recreates it (self-healing in action).

Intermediate Labs

  1. Cordon a node (kubectl cordon) and observe that new Pods are never scheduled there while existing ones keep running.
  2. Drain a node and watch Pods get rescheduled to other nodes — trace the Scheduler's decision via events.
  3. Simulate a scheduler-only failure by scaling kube-scheduler deployment (in kubeadm setups) to 0 and observe Pods stay Pending.
  4. Enable API Server audit logging (kubeadm cluster) and trace a single kubectl apply through the audit log.
  5. Inspect etcd directly with etcdctl get (read-only, lab cluster only) to see the raw key structure Kubernetes uses.

Advanced Labs

  1. Set up a 3-node kubeadm control plane and kill one control-plane node — verify the cluster keeps functioning via quorum.
  2. Take an etcd snapshot, delete a resource, then restore from snapshot and observe the resource reappear.
  3. Write a minimal custom controller (client-go) that watches Pods and logs events — observe the same watch pattern the real controllers use.
  4. Configure and test a validating admission webhook that rejects Pods without resource limits.
  5. Simulate a network partition between one worker node and the control plane (iptables DROP) and observe NotReady + eviction timing.

Production Labs

  1. Design an HA control-plane topology across 3 AZs for a 50-node cluster and document the failure domains.
  2. Write an etcd backup/restore runbook including automated snapshotting to object storage.
  3. Build a Grafana dashboard for API Server request latency and etcd disk fsync latency (the two most predictive control-plane health metrics).
  4. Perform a full control-plane version upgrade on a test cluster following the official version-skew policy end to end.
  5. Load-test the API Server with a synthetic client generating high watch/list volume and identify the first bottleneck.

14. Mini Project

Project: "See the Architecture" dashboard.

  • Architecture: A single Pod running a script that polls the API Server for Nodes, Pods, and Events and prints a live text dashboard.
  • Requirements: ServiceAccount with read-only RBAC to list Nodes/Pods/Events cluster-wide.
  • Implementation: Python + kubernetes client library, using watch() to stream events instead of polling.
  • Deployment: Deploy as a Pod in a dedicated observability namespace.
  • Testing: Create/delete Pods elsewhere in the cluster and confirm the dashboard updates within ~1s.
  • Validation: Compare dashboard output against kubectl get events for consistency.
  • Troubleshooting: If nothing appears, check the ServiceAccount's ClusterRoleBinding first — the #1 cause is RBAC.

15. Real Production Project

This topic's enterprise project is the foundation project used across the whole course: deploying a Spring Boot microservices stack (MySQL, Redis, Kafka) behind Ingress with TLS, monitored by Prometheus + VictoriaMetrics + Grafana, traced with OpenTelemetry, deployed via Helm + Argo CD, autoscaled with HPA, and secured with RBAC + NetworkPolicies. 20. Challenge Questions

  1. Pods stuck Pending cluster-wide: Check Scheduler pod status and leader-election lease first.
  2. CrashLoopBackOff on a critical add-on: Check container logs and resource limits before assuming architecture-level failure.
  3. ImagePullBackOff cluster-wide: Check node-level registry credentials/network, not the API Server.
  4. All nodes show NotReady simultaneously: Suspect API Server or network fabric outage, not individual node issues.
  5. PVC stuck Pending: Check StorageClass/CSI controller, unrelated to core control-plane health.
  6. DNS failures cluster-wide: Check CoreDNS pods and kube-proxy rules, not etcd.
  7. Network issues between two specific pods: Check CNI and NetworkPolicy, not the Scheduler.
  8. Scheduler assigns Pods to an overloaded node: Check if resource requests are set at all (unset requests = Scheduler can't reason about capacity).
  9. API Server intermittently slow: Check etcd fsync latency first — it's the most common root cause.
  10. Node failure doesn't trigger rescheduling: Check pod-eviction-timeout and whether the Pod has tolerations for the NoExecute taint.
  11. ...continuing through 20 scenarios covering OOMKilled, RBAC denials, webhook failures, certificate expiry, quota exhaustion, leader-election flapping, etcd disk-full, version-skew errors, and cross-AZ network partition — each diagnosed by first identifying which architectural component owns the failing behavior before touching configuration.

21. Cheat Sheet

# Control plane health
kubectl get nodes -o wide
kubectl get pods -n kube-system
kubectl get --raw='/healthz?verbose'
kubectl get lease -n kube-system      # leader election status

# Object flow
kubectl describe pod <name>           # events = component-by-component trace
kubectl get pod <name> -o yaml        # spec (desired) vs status (actual)
kubectl explain <kind>.<field>         # live schema

# etcd (control-plane node only)
etcdctl endpoint status --cluster -w table
etcdctl snapshot save backup.db

Architecture at a glance: API Server = front door · etcd = source of truth · Scheduler = decides node placement · Controller Manager = reconciliation loops · kubelet = node agent · kube-proxy = node networking · Container Runtime = actually runs containers.

Interview tip: Always answer architecture questions by naming which component is responsible — interviewers are testing whether you know the boundaries, not just the vocabulary.

Troubleshooting tip: Diagnose top-down: API Server → etcd → Scheduler/Controller Manager → kubelet → runtime. Fixing in the wrong order wastes time.

22. Summary

  • Key takeaway: Kubernetes architecture is a declarative, watch-driven reconciliation system with a clean control-plane/data-plane split — every feature you'll learn next is built on this loop.
  • Production recommendation: Prefer managed control planes; if self-managing, invest heavily in etcd health monitoring and HA topology.
  • Interview tip: Be able to draw the full request flow (kubectl → API Server → etcd → Scheduler → kubelet → runtime) from memory.
  • Exam tip (CKA): Expect hands-on tasks involving kubeadm control-plane setup, etcd backup/restore, and diagnosing a broken control-plane component.
