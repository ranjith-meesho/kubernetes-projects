Secrets

Secret types, why base64 is not encryption, encryption-at-rest, and the RBAC boundary that actually protects them.
← ConfigMaps Course Index Scheduling →

Explanation

A Secret is structurally almost identical to a ConfigMap — a namespaced key-value object consumable as env vars or volume mounts — but Kubernetes treats it as a distinct kind so it can be handled differently: excluded from kubectl describe output by default, eligible for encryption at rest, and subject to tighter RBAC conventions. The type field matters: Opaque is the generic catch-all for arbitrary key-value secrets (API keys, passwords); kubernetes.io/dockerconfigjson holds container registry credentials and is what imagePullSecrets references so kubelet can pull private images; kubernetes.io/tls holds a certificate and private key pair and is what Ingress objects reference for TLS termination; kubernetes.io/service-account-token holds a token tied to a ServiceAccount for authenticating to the API Server.

The single most misunderstood fact about Secrets: values are stored base64-encoded, not encrypted. Base64 is an encoding, not a cipher — anyone with read access to the Secret object (via kubectl get secret -o yaml or direct etcd access) can trivially decode it. This is a deliberate design choice (it lets binary data like certs be stored as text in YAML/JSON) but it means the real security boundary is RBAC, not the encoding. By default, etcd itself stores Secrets in plaintext (base64 only) unless the cluster admin explicitly enables encryption at rest via an EncryptionConfiguration on the API Server, which transparently encrypts Secret data before it's written to etcd (using a local key, KMS-backed key, or an external provider like Vault/AWS KMS).

Because Secrets often hold the actual keys to production systems, the RBAC model around them deserves more scrutiny than almost any other resource: get/list/watch on Secrets should never be granted broadly (a Role with list on all Secrets in a namespace effectively exposes every credential in that namespace to anyone bound to it), and cluster-admin-style ClusterRoles that touch Secrets cluster-wide are one of the highest-value targets in a cluster compromise. Many organizations also externalize truly sensitive material entirely — using tools like External Secrets Operator or Vault's CSI/Agent Injector — treating the native Kubernetes Secret object as a synced cache rather than the source of truth.

Internal Working / Interactions

  • API Server — enforces RBAC on Secret access and applies encryption-at-rest transforms (if configured) before writing to etcd.
  • etcd — physical storage; plaintext base64 unless EncryptionConfiguration is enabled.
  • kubelet — fetches referenced Secrets when starting a Pod, injects as env vars or mounts as tmpfs-backed volume files (never written to the node's persistent disk).
  • ServiceAccount admission controller — auto-mounts a projected service account token Secret/volume unless disabled.
  • External Secrets Operator / Vault (optional) — syncs secrets from an external store into native Secret objects, or injects them directly at runtime.

Code Example

# Opaque Secret — generic key-value credentials
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: shop
type: Opaque
data:
  username: YWRtaW4=              # base64("admin") — NOT encrypted, just encoded
  password: c3VwZXJzZWNyZXQ=      # base64("supersecret")

---
# TLS Secret — referenced by an Ingress for HTTPS termination
apiVersion: v1
kind: Secret
metadata:
  name: shop-tls
  namespace: shop
type: kubernetes.io/tls
data:
  tls.crt: 
  tls.key: 

---
# Pod consuming a Secret as env vars and as a mounted volume
apiVersion: v1
kind: Pod
metadata:
  name: app-pod
  namespace: shop
spec:
  containers:
  - name: app
    image: myco/app:1.4.0
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: password            # injected as an env var at container start
    volumeMounts:
    - name: db-creds-vol
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: db-creds-vol
    secret:
      secretName: db-credentials    # mounted as tmpfs files, never touches node disk
      defaultMode: 0400

kubectl Commands

kubectl create secret generic db-credentials --from-literal=password=supersecret  # imperative creation
kubectl create secret docker-registry regcred --docker-server=... --docker-username=...  # registry pull secret
kubectl create secret tls shop-tls --cert=cert.pem --key=key.pem                   # TLS secret from files
kubectl get secret db-credentials -n shop -o jsonpath='{.data.password}' | base64 -d  # decode a value
kubectl describe secret db-credentials -n shop  # metadata only, values hidden by default
kubectl auth can-i get secrets -n shop --as=system:serviceaccount:shop:app-sa  # verify RBAC scope
kubectl get secrets -A | wc -l                  # audit total secret sprawl across the cluster

Best Practices & Common Pitfalls

Do
  • Enable encryption at rest for Secrets on every cluster handling real credentials.
  • Scope RBAC to specific named Secrets, never blanket list/get on all Secrets in a namespace.
  • Rotate Secrets regularly and automate rotation where possible (short-lived tokens over static passwords).
  • Consider External Secrets Operator/Vault for true secret-of-record management.
Pitfalls
  • Treating base64 as encryption and pasting decoded Secret contents into tickets/logs/Slack.
  • Committing Secret YAML (even base64-encoded) into a public or shared git repo.
  • Granting cluster-admin-equivalent roles just to let a CI pipeline read one Secret.
  • Forgetting that Secret volume mounts update on change just like ConfigMaps — stale in-memory copies in the app are a common source of "why didn't my password rotation work" bugs.

50 Interview Questions

1. Why does Kubernetes treat Secrets as a distinct kind instead of just using ConfigMaps for everything?
Short answer: Secrets are structurally almost identical to ConfigMaps, but the distinct kind lets Kubernetes apply different handling — hiding values from default output, enabling encryption at rest, and inviting tighter RBAC conventions.
Detailed explanation: If sensitive and non-sensitive config shared one kind, there'd be no clean way to selectively encrypt, audit, or restrict access to just the sensitive subset. Splitting them into a separate API kind gives operators a single, consistent signal ("this is a Secret") to hang security policy off of.
Production example: A cluster-wide encryption-at-rest policy or a restrictive RBAC Role can target resources: ["secrets"] specifically, without needing to distinguish "sensitive" ConfigMaps from harmless ones.
Common mistake: Storing credentials in a ConfigMap because it's "just easier," bypassing whatever encryption/RBAC posture is applied specifically to Secrets.
2. Is data stored in a Kubernetes Secret encrypted?
Short answer: No — by default it's only base64-encoded, which is a reversible encoding, not a cipher.
Detailed explanation: Base64 exists so binary data (like TLS keys) can be represented as text inside YAML/JSON. It provides zero confidentiality: anyone who can read the Secret object can trivially decode it with a single command.
Production example: kubectl get secret db-credentials -n shop -o jsonpath='{.data.password}' | base64 -d instantly reveals the plaintext password to anyone with get permission on that Secret.
Common mistake: Telling an auditor or teammate "it's encrypted" because the YAML looks like gibberish base64 text.
3. If base64 isn't encryption, what actually protects a Secret's contents?
Short answer: RBAC is the real security boundary — restricting who and what can get/list/watch Secret objects.
Detailed explanation: Because the encoding provides no confidentiality, access control is the only thing standing between an attacker (or an overly broad service account) and the plaintext credential. Encryption at rest adds a second layer for data sitting in etcd, but it doesn't change who the API Server will hand decoded values to once authorized.
Production example: A compromised Pod with a ServiceAccount bound to a Role that includes get on Secrets can read every credential that Role permits, regardless of encryption-at-rest settings.
Common mistake: Investing effort in encryption at rest while leaving RBAC on Secrets broad and unaudited — the weaker control dominates the overall risk.
4. What does enabling encryption at rest for Secrets actually change?
Short answer: It makes the API Server encrypt Secret data with a configured provider before writing it to etcd, so raw etcd access (backups, disk snapshots, etc.) doesn't expose plaintext.
Detailed explanation: This is configured via an EncryptionConfiguration passed to the API Server, specifying a provider — a local static key, a KMS-backed key, or an external provider like Vault/AWS KMS. It is off by default; etcd stores base64-only plaintext unless an admin explicitly turns this on.
Production example: An etcd snapshot exfiltrated from a misconfigured backup job is far less damaging on a cluster with KMS-backed encryption at rest than on one storing Secrets as bare base64.
Common mistake: Assuming a managed Kubernetes offering enables this automatically without checking — many require explicit opt-in or a specific tier.
5. What does the Opaque Secret type mean, and when should you use it?
Short answer: Opaque is the generic catch-all type for arbitrary key-value secret data with no special semantics — used for things like API keys and database passwords.
Detailed explanation: Unlike kubernetes.io/tls or kubernetes.io/dockerconfigjson, which have required key names that consumers (Ingress, kubelet) expect, Opaque imposes no structure — the keys under data are entirely up to the author.
Production example: A db-credentials Secret with username and password keys, consumed via secretKeyRef in a Pod's env vars.
Common mistake: Using Opaque for TLS certs or registry credentials instead of the dedicated typed Secrets, which then breaks components (like Ingress or imagePullSecrets) expecting specific key names and type-based validation.
6. How does kubernetes.io/dockerconfigjson differ from a generic Opaque Secret?
Short answer: It's a typed Secret holding container registry credentials in Docker's config.json format, referenced by a Pod's imagePullSecrets so kubelet can authenticate to private registries.
Detailed explanation: The type enforces a specific key (.dockerconfigjson) and structure that kubelet's image-pull logic knows how to parse, unlike an arbitrary Opaque Secret which has no guaranteed shape.
Production example: kubectl create secret docker-registry regcred --docker-server=... --docker-username=... creates this type, then a Pod spec references it under spec.imagePullSecrets: [{name: regcred}] to pull a private image.
Common mistake: Forgetting to attach imagePullSecrets to the Pod (or its ServiceAccount) and getting an ImagePullBackOff despite the Secret existing.
7. How does a kubernetes.io/tls Secret relate to Ingress?
Short answer: An Ingress object's tls block references a kubernetes.io/tls Secret by name, and the Ingress Controller uses its tls.crt/tls.key to terminate HTTPS at the edge.
Detailed explanation: The Secret must live in the same namespace as the Ingress. The type enforces exactly two required keys — tls.crt and tls.key — which controllers expect by convention.
Production example: cert-manager automates issuing and renewing a Let's Encrypt certificate and writes it directly into this Secret, so the Ingress Controller picks up renewed certs without manual intervention.
Common mistake: Creating the TLS Secret in the wrong namespace, or hand-rotating it and letting the certificate silently expire because nothing automated the renewal.
8. What is a kubernetes.io/service-account-token Secret, and how is it different from the modern default?
Short answer: It's a Secret holding a token tied to a ServiceAccount used to authenticate to the API Server; modern clusters instead default to short-lived, auto-rotated projected tokens rather than long-lived Secret-backed ones.
Detailed explanation: Older Kubernetes versions auto-created a non-expiring token Secret per ServiceAccount. Since the introduction of the TokenRequest API and projected service account volumes, kubelet mounts a time-bound, audience-scoped token directly into the Pod instead, reducing the blast radius of a leaked token.
Production example: A Pod's /var/run/secrets/kubernetes.io/serviceaccount/token file today is typically a projected, auto-refreshing token rather than a static Secret-mounted one, limiting how long a stolen copy remains valid.
Common mistake: Assuming every ServiceAccount still has a long-lived token Secret automatically created for it, and building tooling that depends on that legacy behavior.
9. Why is a Secret volume mount safer than an environment variable for sensitive data?
Short answer: Volume-mounted Secrets are written to a tmpfs (in-memory) filesystem inside the Pod and never touch the node's persistent disk, whereas env vars are more easily leaked via process listings, crash dumps, or logging.
Detailed explanation: Environment variables are visible to any process in the container, get inherited by child processes, and often show up in debugging output or `/proc//environ`. A file mount under a restrictive defaultMode can be scoped more tightly and doesn't propagate the same way.
Production example: Mounting db-credentials at /etc/secrets with defaultMode: 0400 and readOnly: true limits exposure compared to an env var that a misbehaving library might log verbatim on startup.
Common mistake: Passing database passwords via env vars purely for convenience and then having them show up in error-reporting tools that dump the full environment on crash.
10. Why should RBAC never grant broad list/get on all Secrets in a namespace?
Short answer: Because Secret values decode trivially, a Role with unrestricted list/get on Secrets effectively hands out every credential stored in that namespace to anyone bound to it.
Detailed explanation: Unlike a resource where broad read access is low-risk, "read access to Secrets" is functionally equivalent to "read access to every password, token, and key in the namespace." RBAC should instead scope rules to specific named Secrets via resourceNames, or split sensitive Secrets into narrower namespaces.
Production example: A CI service account that only needs one deploy token should get a Role scoped to that single Secret name, not a namespace-wide secrets: get, list, watch grant.
Common mistake: Copy-pasting a broad "view" ClusterRole that includes blanket Secret access onto a service account that only ever needed one credential.
11–50. Additional rapid-fire questions
11. What command decodes a Secret value from the CLI? — kubectl get secret <name> -o jsonpath='{.data.<key>}' | base64 -d.
12. Does kubectl describe secret show the actual values? — No, it shows metadata and key names only, with values hidden by default.
13. What field configures encryption at rest for Secrets? — An EncryptionConfiguration resource passed to the API Server via --encryption-provider-config.
14. Is encryption at rest enabled by default in vanilla Kubernetes? — No, it must be explicitly configured by the cluster administrator.
15. What are the three main encryption-at-rest provider options? — A local static key, a KMS-backed key (via the KMS provider), or an external provider like Vault/AWS KMS.
16. Why can binary data like a TLS private key be stored in a Secret's YAML? — Because base64 encoding lets arbitrary binary content be represented safely as printable text.
17. What is the required key name inside a kubernetes.io/tls Secret for the certificate? — tls.crt, paired with tls.key for the private key.
18. What key does kubernetes.io/dockerconfigjson require? — .dockerconfigjson, containing the registry auth config.
19. How do you create a TLS Secret imperatively? — kubectl create secret tls shop-tls --cert=cert.pem --key=key.pem.
20. How do you create a registry pull Secret imperatively? — kubectl create secret docker-registry regcred --docker-server=... --docker-username=....
21. What Pod field references an imagePullSecret? — spec.imagePullSecrets, a list of Secret names.
22. Can imagePullSecrets be attached at the ServiceAccount level instead of per-Pod? — Yes, a ServiceAccount's imagePullSecrets field applies automatically to Pods using it.
23. Where does kubelet mount volume-based Secrets inside a container? — Into a tmpfs (RAM-backed) filesystem at the specified mountPath, never onto the node's persistent disk.
24. What does defaultMode: 0400 on a Secret volume achieve? — Restricts the mounted files to read-only for the owning user, limiting other processes/users from reading them.
25. Why might an application not notice a rotated Secret? — If it only reads env vars or reads the mounted file once at startup, it holds a stale in-memory copy until restarted.
26. Do Secret volume mounts update automatically after the Secret changes? — Yes, kubelet periodically syncs updated Secret content to the mounted files, similar to ConfigMaps.
27. What component auto-mounts a ServiceAccount token into a Pod? — The ServiceAccount admission controller, via a projected volume.
28. How can you disable automatic ServiceAccount token mounting? — Set automountServiceAccountToken: false on the ServiceAccount or Pod spec.
29. What is one advantage of projected, short-lived service account tokens over the legacy static token Secret? — They expire and auto-rotate, reducing the impact of a leaked token compared to a non-expiring one.
30. What tool commonly automates issuing certs into a kubernetes.io/tls Secret? — cert-manager, via ACME issuers like Let's Encrypt.
31. Why is committing Secret YAML to a public git repo dangerous even though it's base64? — Base64 is trivially reversible, so committing it is functionally equivalent to committing plaintext credentials.
32. What's a common alternative to storing the source of truth for secrets natively in Kubernetes? — External Secrets Operator or Vault, syncing into native Secret objects or injecting at runtime.
33. When using External Secrets Operator, what role does the native Kubernetes Secret play? — It becomes a synced cache of the externally-managed secret, not the authoritative source.
34. How would you audit total secret sprawl across a cluster? — kubectl get secrets -A | wc -l to see the raw count across all namespaces.
35. How do you verify whether a specific ServiceAccount can read Secrets in a namespace? — kubectl auth can-i get secrets -n <ns> --as=system:serviceaccount:<ns>:<sa>.
36. Why is granting cluster-admin just to let CI read one Secret a serious risk? — It grants full cluster control far beyond what's needed, turning a single leaked CI credential into a total cluster compromise.
37. What's the risk of pasting a decoded Secret value into a support ticket or Slack message? — It permanently leaks the credential outside Kubernetes' access controls into a system with its own (often broader) audience.
38. Does enabling encryption at rest protect a Secret from someone with valid RBAC read access? — No, encryption at rest only protects raw etcd/disk access; the API Server still decrypts and returns plaintext to any authorized caller.
39. What is a practical way to rotate a database password stored in a Secret with minimal downtime? — Update the Secret, then trigger a rolling restart of consumers so they pick up the new mounted/env value.
40. Why might a Secret update not immediately affect a Pod using envFrom/secretKeyRef? — Env vars are injected once at container start; unlike volume mounts, they don't update live and require a Pod restart.
41. What's a security reason to avoid one giant Secret holding many unrelated credentials? — It violates least privilege — any consumer granted access to that Secret gains every credential inside it, not just the one it needs.
42. How does the API Server enforce Secret confidentiality on read? — Via standard RBAC authorization checks (`get`/`list`/`watch` on the `secrets` resource) before returning any data.
43. What's a quick way to check which type a given Secret is? — kubectl get secret <name> -o jsonpath='{.type}'.
44. Why do etcd backups need the same security scrutiny as live Secrets? — Without encryption at rest, a backup snapshot contains the same base64-decodable plaintext as the live cluster.
45. What happens if you reference a Secret name in a Pod spec that doesn't exist? — The Pod fails to start (env var case) or the volume fails to mount, and the Pod stays in a pending/error state until the Secret exists.
46. Is it possible to make a Secret immutable? — Yes, setting immutable: true prevents updates to its data, protecting against accidental modification and reducing watch load.
47. Why would you prefer Vault-issued dynamic database credentials over a long-lived Secret? — Dynamic credentials are short-lived and unique per lease, drastically shrinking the exposure window if one leaks.
48. What's a common misconfiguration that causes ImagePullBackOff despite a correct registry Secret existing? — The Secret isn't referenced in the Pod's (or ServiceAccount's) imagePullSecrets, or it's in the wrong namespace.
49. Why is namespace scoping relevant to Secrets and TLS certs used by Ingress? — A Secret must exist in the same namespace as the Ingress referencing it — cross-namespace references aren't supported.
50. What's the first thing to check when a Pod can't decode/use a mounted Secret correctly? — Confirm the key names in the Secret's data match exactly what the consuming Pod/controller expects (e.g. tls.crt/tls.key), since typos in key names fail silently as missing files.
