CronJobs

Time-based scheduling for Jobs — cron syntax, concurrency control, and missed-schedule handling.
← Jobs Course Index Services →

Explanation

A CronJob creates Jobs on a recurring time-based schedule, using standard cron syntax. It is not a workload type of its own — it is purely a scheduling wrapper: at each tick that matches its schedule, the CronJob controller stamps out a new Job object from its embedded jobTemplate, and that Job then runs exactly like any other Job (completions, parallelism, backoffLimit all apply the same way).

Kubernetes needs this because a huge share of batch/operational work is inherently periodic — nightly backups, hourly cache warmups, daily report emails, periodic cleanup of stale data. Without CronJobs, you'd need an external scheduler (a cron daemon somewhere, or a CI pipeline) just to call kubectl create job on a timer, adding an extra system and failure domain outside Kubernetes' own reconciliation model. A CronJob keeps scheduling itself as just another declarative object living in etcd, reconciled the same way as everything else.

The schedule is a standard 5-field cron expression (minute hour day-of-month month day-of-week), evaluated in the CronJob controller's configured time zone (settable per-CronJob since spec.timeZone was added, avoiding the classic "cluster is UTC but ops team is IST" confusion). Because a controller could miss a tick (was down, cluster was under load), startingDeadlineSeconds bounds how late a missed run is still allowed to start — beyond that window, it's simply skipped and counted as a missed schedule rather than an infinite pile-up of catch-up runs.

concurrencyPolicy governs what happens if a previous run's Job is still active when the next scheduled time arrives: Allow (default — let them run concurrently), Forbid (skip the new run entirely if the old one is still active — critical for tasks that must never overlap, like a report that reads and rewrites the same file), or Replace (kill the still-running old Job and start the new one). Getting this setting wrong is one of the most common production CronJob bugs — e.g. an Allow nightly backup job whose runs occasionally overlap and corrupt a shared destination.

successfulJobsHistoryLimit / failedJobsHistoryLimit bound how many completed Job objects are kept around for inspection (defaults 3 and 1) — old ones beyond the limit are garbage collected automatically.

Internal Working / Interactions

  • CronJob controller (in kube-controller-manager) — on every reconcile tick, computes whether the schedule has fired since the last recorded run, and if so (and within startingDeadlineSeconds if set), creates a new Job from spec.jobTemplate.
  • Job controller — takes over from there exactly as with any manually created Job: creates Pods, tracks completions/retries.
  • API Server / etcd — CronJob spec/status, each generated Job, and their Pods are all ordinary watched/persisted objects.
  • concurrencyPolicy enforcement — the CronJob controller lists currently-active Jobs it owns before creating a new one, and applies Allow/Forbid/Replace logic accordingly.
  • History garbage collection — the controller deletes old completed/failed Job objects beyond the configured history limits on each reconcile.

Code Example

apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"           # Standard cron: 02:00 every day.
  timeZone: "Asia/Kolkata"        # Evaluate the schedule in IST, not cluster default (UTC).
  concurrencyPolicy: Forbid       # Never run two backups concurrently — protects the shared destination.
  startingDeadlineSeconds: 300    # If controller was down, only start a missed run within 5 minutes of schedule.
  successfulJobsHistoryLimit: 3   # Keep last 3 successful Jobs for inspection.
  failedJobsHistoryLimit: 5       # Keep more failed ones — useful for debugging.
  jobTemplate:                    # Everything below is a normal Job spec.
    spec:
      backoffLimit: 2
      activeDeadlineSeconds: 1800
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: registry.internal/db-backup:1.0
              command: ["/scripts/backup.sh"]
              resources:
                requests: { cpu: "250m", memory: "256Mi" }
                limits:   { cpu: "500m", memory: "512Mi" }

kubectl Commands

# List CronJobs and their schedule/last-run status
kubectl get cronjobs

# Show recent scheduling decisions and events
kubectl describe cronjob nightly-backup

# Manually trigger one run immediately, outside the schedule (great for testing)
kubectl create job --from=cronjob/nightly-backup manual-test-run

# See the Jobs a CronJob has created
kubectl get jobs -l app.kubernetes.io/managed-by=cronjob-controller

# Temporarily disable a CronJob without deleting it
kubectl patch cronjob nightly-backup -p '{"spec":{"suspend":true}}'

# Re-enable it
kubectl patch cronjob nightly-backup -p '{"spec":{"suspend":false}}'

# Tail logs of the most recent run's Pod
kubectl logs -l job-name=$(kubectl get jobs -l app.kubernetes.io/managed-by=cronjob-controller -o jsonpath='{.items[-1:].metadata.name}')

Best Practices & Common Pitfalls

Best Practices

  • Set concurrencyPolicy: Forbid for anything that mutates shared state — most backup/report jobs.
  • Always set startingDeadlineSeconds to avoid a storm of catch-up runs after a controller outage.
  • Set explicit timeZone rather than relying on cluster-default UTC for business-hour-sensitive jobs.
  • Use spec.suspend: true to pause a CronJob during maintenance instead of deleting it.

Pitfalls

  • Leaving concurrencyPolicy: Allow (the default) on jobs that must never overlap.
  • Forgetting the underlying Job's backoffLimit/activeDeadlineSeconds still apply — a stuck CronJob run can block the next schedule under Forbid.
  • Not setting history limits, letting hundreds of old completed Jobs accumulate in etcd over months.
  • Assuming exactly-once execution — clock skew or controller restarts can occasionally cause a missed or (with Allow) overlapping run.

50 Interview Questions

1. What exactly does a CronJob object manage, and what doesn't it manage?
Short answer: It manages the schedule and creation of Job objects from a template — it is not itself a workload type, and it does not run containers directly.
Detailed explanation: At each tick matching its cron schedule, the CronJob controller stamps out a new Job from spec.jobTemplate; from that point on, the ordinary Job controller and all Job semantics (completions, parallelism, backoffLimit) take over completely.
Production example: A nightly-backup CronJob at 02:00 creates a Job named something like nightly-backup-28470870, which then behaves exactly like any manually created Job.
Common mistake: Debugging a failed scheduled run by only looking at the CronJob object instead of inspecting the generated Job and its Pods.
2. Why does Kubernetes need CronJobs instead of relying on an external cron daemon calling kubectl create job?
Short answer: Keeping scheduling inside Kubernetes avoids introducing an extra system and failure domain outside the platform's own reconciliation model.
Detailed explanation: An external scheduler would need its own credentials, availability guarantees, and monitoring — entirely separate from etcd/API Server state — whereas a CronJob is just another declarative object reconciled the same way as everything else.
Production example: A CI pipeline calling kubectl create job on a timer would need its own uptime monitoring independent of the cluster; a CronJob's reliability is instead tied to kube-controller-manager's own HA.
Common mistake: Building an external triggering system for periodic Kubernetes tasks when a CronJob already solves it natively.
3. What problem does concurrencyPolicy: Forbid solve, and what commonly goes wrong without it?
Short answer: It prevents a new scheduled run from starting while a previous run's Job is still active — critical for tasks that must never overlap.
Detailed explanation: The default policy, Allow, lets runs overlap freely; for a task that reads and rewrites a shared destination (like a backup or report file), overlapping runs can corrupt that shared state.
Production example: A nightly-backup CronJob set to Forbid simply skips a run (and logs it as missed) if the previous night's backup Job hasn't finished yet, rather than running two backups concurrently against the same destination.
Common mistake: Leaving the default Allow policy on a job that mutates shared state, only discovering the corruption after overlapping runs occur in production.
4. What's the difference between concurrencyPolicy Forbid and Replace?
Short answer: Forbid skips the new run entirely if the old one is still active; Replace kills the still-running old Job and starts the new one instead.
Detailed explanation: Forbid preserves the in-progress work but sacrifices timeliness (the new schedule tick is simply missed); Replace prioritizes running the latest scheduled version but discards whatever progress the previous run had made.
Production example: A cache-warmup CronJob might use Replace since only the freshest run matters, while a backup CronJob uses Forbid since a half-finished backup being killed mid-write would be worse than a skipped run.
Common mistake: Using Replace for a task where a partially-completed previous run leaves the system in a bad intermediate state when killed.
5. Why does startingDeadlineSeconds exist, and what happens without it?
Short answer: It bounds how late a missed scheduled run is still allowed to start; without it, a controller outage could otherwise cause an unbounded pile-up of catch-up runs once it recovers.
Detailed explanation: If the CronJob controller itself was down or the cluster was under load for hours, on recovery it needs a rule for how many "missed" ticks to still honor versus simply skip — startingDeadlineSeconds is that rule.
Production example: startingDeadlineSeconds: 300 means if the controller was down for 3 hours, only a run within the last 5 minutes of the outage's end is still attempted — the rest are logged as missed schedules, not queued up.
Common mistake: Leaving it unset and assuming Kubernetes automatically avoids catch-up storms — behavior without an explicit deadline can still result in unexpected missed/skipped run handling depending on version and load.
6. Why is spec.timeZone important, and what problem did it solve?
Short answer: It lets a CronJob's schedule be evaluated in a specific time zone rather than the cluster's default (commonly UTC), avoiding the classic "cluster is UTC but ops team is IST" confusion.
Detailed explanation: Before per-CronJob timeZone existed, teams had to manually offset cron expressions to account for UTC, which was error-prone and broke silently across daylight-saving transitions in some regions.
Production example: timeZone: "Asia/Kolkata" on a nightly-backup CronJob ensures "02:00" genuinely means 2 AM IST, not 2 AM UTC (7:30 AM IST).
Common mistake: Writing a schedule assuming local business hours without setting timeZone, causing the job to run at an unintended UTC-shifted time.
7. What do successfulJobsHistoryLimit and failedJobsHistoryLimit control, and why keep more failed history than successful?
Short answer: They bound how many completed Job objects are retained for inspection; failed runs are often worth keeping longer for debugging than routine successes.
Detailed explanation: Old Jobs beyond these limits are automatically garbage collected by the CronJob controller on each reconcile, preventing unbounded accumulation of historical Job objects in etcd.
Production example: successfulJobsHistoryLimit: 3, failedJobsHistoryLimit: 5 keeps more failure evidence around since failures are rarer and more valuable to investigate than routine successful runs.
Common mistake: Not setting history limits at all, letting hundreds of old completed Jobs accumulate in etcd over months of daily runs.
8. How would you manually trigger an out-of-schedule run of a CronJob for testing?
Short answer: kubectl create job --from=cronjob/<name> <manual-run-name>, which copies the jobTemplate into a fresh, immediately-created Job.
Detailed explanation: This is the standard way to test a CronJob's actual behavior without waiting for its schedule to fire, and it produces an ordinary Job indistinguishable in behavior from a normally-scheduled one.
Production example: Before trusting a new nightly-backup CronJob's first scheduled run, an engineer manually triggers it once during business hours to verify it succeeds and inspect the resulting Job's logs.
Common mistake: Editing the CronJob's schedule temporarily to "run soon" for testing instead of just using the --from=cronjob shortcut.
9. What's the effect of spec.suspend: true on a CronJob, and how does it differ from deleting it?
Short answer: It pauses future scheduled runs without deleting the CronJob object, its configuration, or its history — deleting would remove all of that permanently.
Detailed explanation: Suspend is the correct mechanism for temporarily halting a CronJob during a maintenance window, since the object (and any already-running Jobs) remains intact and can be un-suspended instantly.
Production example: kubectl patch cronjob nightly-backup -p '{"spec":{"suspend":true}}' pauses backups during a planned database migration, then suspend:false resumes the normal schedule afterward.
Common mistake: Deleting and later recreating a CronJob to "pause" it, losing its Job history and needing to redefine it from scratch.
10. Why does the CronJob controller need to list currently-active Jobs before creating a new one?
Short answer: To correctly enforce concurrencyPolicy — it must know whether a previous run is still active before deciding to Allow, Forbid, or Replace the new one.
Detailed explanation: This list operation happens on every reconcile tick where the schedule has fired, since the concurrency decision fundamentally depends on real-time knowledge of currently-running Jobs the CronJob owns.
Production example: With Forbid set, the controller's list-then-decide step is exactly what causes a skipped (not queued) run when the prior night's Job is still executing at the next scheduled tick.
Common mistake: Assuming concurrencyPolicy is enforced by some separate locking mechanism rather than a straightforward "check active Jobs, then decide" step inside the controller's own reconcile loop.
11–50. Additional rapid-fire questions
11. What API group/version is CronJob under? — batch/v1.
12. What's the standard cron expression field order? — minute hour day-of-month month day-of-week.
13. What does the schedule "0 2 * * *" mean? — Every day at 02:00.
14. Does concurrencyPolicy Allow guarantee overlapping runs never cause problems? — No — it just permits overlap; correctness depends entirely on the workload tolerating concurrent execution.
15. What's the default concurrencyPolicy if unset? — Allow.
16. Can a suspended CronJob still be manually triggered via --from=cronjob? — Yes — suspend only blocks the automatic schedule, not manual Job creation from its template.
17. What happens to already-running Jobs when you suspend a CronJob? — They continue running to completion; suspend only prevents new scheduled runs from starting.
18. How do you find the Jobs a specific CronJob created? — kubectl get jobs -l app.kubernetes.io/managed-by=cronjob-controller or by checking ownerReferences.
19. Is CronJob namespaced or cluster-scoped? — Namespaced.
20. What happens if you delete a CronJob? — By default its owned Jobs (and their Pods) are cascade-deleted too.
21. Does a CronJob support activeDeadlineSeconds? — Indirectly — it's set inside jobTemplate.spec, applying to each generated Job individually.
22. Can startingDeadlineSeconds and concurrencyPolicy interact in a confusing way? — Yes — a missed run within the deadline might still be skipped anyway if concurrencyPolicy is Forbid and a prior run is active.
23. What's a subtle gotcha with concurrencyPolicy: Forbid and a stuck previous Job? — A hung Job blocks every subsequent scheduled run indefinitely until its own backoffLimit/activeDeadlineSeconds finally ends it.
24. Why is it important to set activeDeadlineSeconds on the underlying Job template for Forbid CronJobs? — Without it, one stuck run can silently block all future scheduled runs with no automatic recovery.
25. What's the risk of setting failedJobsHistoryLimit to 0? — You lose all evidence of failed runs immediately after they're garbage collected, hampering debugging.
26. Does CronJob support the same restartPolicy rules as Job? — Yes — the jobTemplate.spec.template.spec.restartPolicy must still be OnFailure or Never.
27. Can you change a CronJob's schedule without recreating it? — Yes — spec.schedule is mutable; the next reconcile picks up the new schedule going forward.
28. What happens to already-scheduled-but-not-yet-fired ticks when you change the schedule? — They're simply recalculated based on the new schedule from that point forward — no special migration needed.
29. Is there a way to see a CronJob's "last successful run" time? — Yes — status.lastSuccessfulTime (and status.lastScheduleTime for the last scheduling attempt).
30. What's a good practice before relying on a new CronJob in production? — Manually trigger it once via --from=cronjob to validate the jobTemplate actually succeeds end to end.
31. Why might clock skew across nodes matter for CronJob scheduling? — The CronJob controller's own clock (in kube-controller-manager) drives scheduling decisions — skew there, not on worker nodes, is what could cause off-schedule runs.
32. Can two CronJobs share the same jobTemplate content safely? — Yes — they're independent objects; sharing template content causes no conflict since each generates its own separately-named Jobs.
33. What field would tell you if a CronJob missed its schedule due to controller downtime? — Comparing status.lastScheduleTime against the expected schedule times reveals gaps.
34. Does a CronJob retry a missed schedule automatically once startingDeadlineSeconds passes? — No — once beyond the deadline, that specific missed run is simply skipped, not queued for later.
35. What's the interaction between CronJob and leader election in kube-controller-manager? — Only the active (elected leader) controller-manager instance evaluates and fires CronJob schedules, avoiding duplicate-run races across HA control planes.
36. Why would a team choose an external workflow engine (e.g. Argo Workflows) over CronJob for complex batch pipelines? — CronJob only creates one Job per tick with no built-in multi-step dependency/DAG orchestration between steps.
37. What's a realistic incident caused by Allow concurrencyPolicy on a report job? — Two concurrent runs both read-modify-write the same output file, corrupting or truncating the final report.
38. How would you debug a CronJob that never seems to fire? — Check kubectl describe cronjob for schedule parsing errors, confirm it isn't suspended, and check controller-manager logs for the leader's health.
39. Can a CronJob's jobTemplate reference a ConfigMap for its schedule-sensitive config? — Yes — same as any Job, via volume mounts or env vars in the Pod template.
40. What's the effect of a very tight schedule (e.g. every minute) combined with a slow job and Forbid? — Most ticks get skipped since the previous run is almost always still active when the next one would fire.
41. Why should ttlSecondsAfterFinished still be considered on the jobTemplate even with history limits set? — History limits bound Job *object* count, but TTL provides faster, time-based cleanup independent of run frequency.
42. What happens to in-flight Jobs if the whole CronJob controller crashes mid-tick? — Already-created Jobs continue running under the ordinary Job controller; only the CronJob's own scheduling/bookkeeping is paused until the controller recovers.
43. Is a CronJob's schedule field validated at creation time? — Yes — the API Server validates cron syntax and rejects malformed expressions immediately.
44. What's a good monitoring signal specifically for CronJobs? — Alerting on status.lastScheduleTime falling too far behind the expected interval, indicating missed or stuck runs.
45. Can a CronJob's generated Job names collide? — No — each generated Job gets a unique, timestamp-derived suffix appended to the CronJob name.
46. What's the difference between a "missed schedule" and a "failed Job" from a CronJob's perspective? — A missed schedule means no Job was even created for that tick (deadline/concurrency skip); a failed Job means one was created but its Pods ultimately failed.
47. Why is Forbid often the safer default for unfamiliar/new batch workloads? — It fails safe by skipping rather than risking silent data corruption from unexpected overlap, until the workload's concurrency-safety is proven.
48. Does suspending a CronJob affect its history/garbage collection? — No — history limits and TTL cleanup for already-completed Jobs continue to apply normally even while suspended.
49. What's a common mistake when migrating an existing cron-based script into a CronJob? — Forgetting the script needs to tolerate at-least-once execution (Job semantics), same idempotency concern as plain Jobs.
50. What's the first thing to check when a CronJob appears to have stopped running entirely? — Confirm spec.suspend is false, then check status.lastScheduleTime and controller-manager health/leader election.
