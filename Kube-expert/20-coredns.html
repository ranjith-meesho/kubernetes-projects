CoreDNS

The pluggable DNS server that implements cluster DNS — its Corefile plugin chain, live watches, and caching/forwarding behavior.
← DNS Course Index Networking →

Explanation

CoreDNS is the DNS server implementation that provides cluster DNS in essentially every modern Kubernetes distribution (it replaced kube-dns as the default around Kubernetes 1.13). Where the previous topic described the naming convention (how DNS names are structured), this topic is about the actual server that answers those queries: a general-purpose, plugin-chain-based DNS server written in Go, running as a Deployment inside the cluster itself, exposed via its own Service.

Kubernetes needs CoreDNS specifically (rather than a generic DNS server with a static zone file) because cluster DNS records are not static — they must reflect the live, constantly-changing set of Services, Endpoints, and optionally Pods. A traditional DNS server reads records from disk or a database that something else must keep in sync; CoreDNS instead has a first-class kubernetes plugin that talks directly to the Kubernetes API Server via a watch connection, so DNS answers are generated on the fly from live cluster state with no separate synchronization step and no stale-zone-file problem.

CoreDNS's defining architectural idea is the plugin chain, configured via a Corefile (analogous to nginx.conf but for DNS). Each plugin handles one concern and the request flows through them in the order they're listed: errors logs errors, health/ready expose liveness/readiness endpoints for the Pod's own probes, kubernetes answers queries about cluster Services/Pods by watching the API Server, prometheus exposes metrics, forward sends any query CoreDNS doesn't own (e.g. an external domain like api.stripe.com) upstream to the node's configured resolver (often the cloud provider's DNS, e.g. AmazonProvidedDNS on EKS), cache caches both the cluster-internal and forwarded answers for a configurable TTL to cut both API Server load and outbound query latency, and loop detects and breaks DNS forwarding loops that would otherwise crash CoreDNS with an infinite request storm.

Because CoreDNS is deployed as an ordinary Kubernetes Deployment (typically 2 replicas for HA, often with the cluster-autoscaler-aware topologySpreadConstraints to spread across nodes), it benefits from — and is subject to — every property already covered: it's exposed via a stable Service so kubelet can inject one durable IP into every Pod's resolv.conf, and it self-heals like any Deployment-managed workload. This is a nice demonstration of the platform's own composability: even a piece of "core infrastructure" like cluster DNS is just another workload running on top of Kubernetes' regular primitives.

Internal Working / Interactions

  • API Server — CoreDNS's kubernetes plugin maintains a watch/informer against Services, EndpointSlices, and (if the plugin is configured to) Pods, to build in-memory DNS answers.
  • kube-dns Service — CoreDNS Pods sit behind this stable ClusterIP Service; it's this IP that kubelet writes into every other Pod's /etc/resolv.conf.
  • kubelet — indirectly depends on CoreDNS being healthy; if CoreDNS is down, name resolution cluster-wide fails even though existing established connections keep working.
  • Upstream resolver — for non-cluster domains, the forward plugin sends queries to whatever DNS server is configured on the CoreDNS Pod's own node (often /etc/resolv.conf of the host, passed through).
  • Deployment / ReplicaSet controller — manage CoreDNS's own Pods exactly like any other Deployment; a crashed CoreDNS Pod is replaced the same way any other workload's Pod would be.

Code Example

# ConfigMap holding the Corefile — CoreDNS's actual configuration (edit via kubectl edit cm coredns -n kube-system)
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {                          # Listen on port 53 for all domains (".").
        errors                        # Log DNS errors to stdout.
        health {
           lameduck 5s                 # Report unhealthy 5s before shutdown, for clean rolling updates.
        }
        ready                         # Readiness endpoint used by CoreDNS's own Pod readinessProbe.
        kubernetes cluster.local in-addr.arpa ip6.arpa {   # Handles cluster.local + reverse DNS.
           pods insecure                 # Allow resolving Pod IPs directly (pod-ip.ns.pod.cluster.local).
           fallthrough in-addr.arpa ip6.arpa
           ttl 30                        # TTL for records served by this plugin.
        }
        prometheus :9153               # Expose Prometheus metrics on :9153.
        forward . /etc/resolv.conf {    # Anything not matched above -> forward to node's upstream resolver.
           max_concurrent 1000
        }
        cache 30                       # Cache answers for 30s to cut API Server load + latency.
        loop                           # Detect and break forwarding loops.
        reload                        # Auto-reload Corefile changes without restarting the Pod.
        loadbalance                   # Round-robin shuffle of A/AAAA records returned, for basic load spreading.
    }

kubectl Commands

# Check CoreDNS Pod health and replica count
kubectl get pods -n kube-system -l k8s-app=kube-dns

# View live logs (great for diagnosing resolution failures)
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=100 -f

# View or edit the active Corefile
kubectl get configmap coredns -n kube-system -o yaml
kubectl edit configmap coredns -n kube-system

# Check CoreDNS's own Service (the IP injected into every Pod's resolv.conf)
kubectl get svc kube-dns -n kube-system

# Restart CoreDNS to pick up a Corefile change immediately (reload plugin usually handles this automatically)
kubectl rollout restart deployment coredns -n kube-system

# Scrape CoreDNS's Prometheus metrics endpoint for query volume / error rates
kubectl port-forward -n kube-system deploy/coredns 9153:9153
curl localhost:9153/metrics | grep coredns_dns_requests_total

Best Practices & Common Pitfalls

Best Practices

  • Run at least 2 CoreDNS replicas with pod anti-affinity/topology spread for HA.
  • Tune the cache plugin's TTL to balance API Server load against DNS staleness tolerance.
  • Monitor coredns_dns_request_duration_seconds and error-rate metrics — DNS latency spikes often precede wider outages.
  • Use the kubernetes-autoscaler/CoreDNS-autoscaler add-on to scale replica count with cluster size.

Pitfalls

  • Forgetting the loop plugin, risking a forwarding loop that crash-loops CoreDNS under certain node resolv.conf misconfigurations.
  • Setting cache TTL too high, causing stale answers after fast Service/Endpoint churn (e.g. during rollouts).
  • Not monitoring CoreDNS at all — since it's "just infrastructure," teams often notice it's down only via app-wide DNS failures.
  • Editing the Corefile without validating syntax — a malformed Corefile can crash-loop every CoreDNS replica simultaneously.

50 Interview Questions

1. Why can't cluster DNS be implemented with a traditional static zone file?
Short answer: Cluster DNS records must reflect the live, constantly-changing set of Services, Endpoints, and Pods — a static file would immediately go stale.
Detailed explanation: CoreDNS's kubernetes plugin instead talks directly to the API Server via a watch connection, generating DNS answers on the fly from live cluster state with no separate synchronization step and no stale-zone-file problem.
Production example: The moment a checkout Service's ClusterIP-backing Pods change, the next DNS query reflects current state — there's no "reload the zone file" step required.
Common mistake: Assuming CoreDNS needs to be manually reloaded or restarted whenever Services/Pods change — the kubernetes plugin's watch handles this automatically.
2. What is the Corefile plugin chain, and why does plugin order matter?
Short answer: The Corefile lists plugins (errors, health, kubernetes, forward, cache, etc.) that a request flows through in listed order, each handling one concern.
Detailed explanation: Since each plugin can inspect, answer, or pass along a query, ordering determines behavior — e.g. kubernetes must come before forward so cluster-domain queries are answered locally instead of being needlessly forwarded upstream.
Production example: In the standard Corefile, kubernetes handles cluster.local/in-addr.arpa/ip6.arpa, and only queries it doesn't recognize fall through to forward for upstream resolution.
Common mistake: Reordering the Corefile without understanding fallthrough semantics, accidentally causing cluster-internal queries to be forwarded upstream (and fail) instead of answered locally.
3. What does the forward plugin do, and where does it send queries it handles?
Short answer: It sends any query CoreDNS doesn't own (e.g. an external domain like api.stripe.com) upstream to the node's configured resolver.
Detailed explanation: By default this reads from /etc/resolv.conf on the CoreDNS Pod's own node, meaning external resolution ultimately depends on whatever DNS server the underlying infrastructure provides (e.g. AmazonProvidedDNS on EKS).
Production example: A Pod querying api.stripe.com gets answered via CoreDNS's kubernetes plugin recognizing it's not a cluster domain, then forward relaying it to the node's upstream resolver.
Common mistake: Assuming CoreDNS resolves external domains itself rather than merely forwarding to an upstream server it doesn't control.
4. Why does the loop plugin exist, and what specific failure mode does it prevent?
Short answer: It detects and breaks DNS forwarding loops that would otherwise crash CoreDNS with an infinite request storm.
Detailed explanation: A misconfigured node resolv.conf (e.g. pointing back at CoreDNS itself as an upstream) can create a forwarding cycle; loop detects this pattern and fails fast instead of letting CoreDNS spiral into a crash-loop under the resulting query storm.
Production example: A cluster where the node's own resolv.conf was accidentally pointed at the cluster DNS IP would trigger loop's detection, surfacing a clear error instead of CoreDNS silently melting down.
Common mistake: Removing the loop plugin from a Corefile "to simplify it," losing this safety net against a subtle but severe misconfiguration.
5. What does the cache plugin actually cache, and what's the tradeoff in setting its TTL?
Short answer: It caches both cluster-internal and forwarded (external) answers for a configurable TTL, cutting API Server load and outbound query latency.
Detailed explanation: Too high a TTL risks serving stale answers after fast Service/Endpoint churn (e.g. during rollouts); too low a TTL reduces the caching benefit, increasing load and latency for repeated queries.
Production example: A Corefile with cache 30 balances a 30-second staleness window against meaningfully reduced query volume for high-traffic Service names.
Common mistake: Setting cache TTL far too high for a cluster with fast-churning Services (frequent rollouts), causing clients to briefly resolve to already-gone Pod-backed IPs.
6. Why is CoreDNS deployed as an ordinary Deployment rather than some special always-on system process?
Short answer: To demonstrate and benefit from Kubernetes' own composability — even core infrastructure is just another workload managed by the same primitives.
Detailed explanation: Running as a Deployment means CoreDNS self-heals like any other workload (ReplicaSet controller replaces a crashed Pod), can be scaled via replica count, and benefits from standard rollout/rollback mechanics for Corefile/image changes.
Production example: A crashed CoreDNS Pod is replaced within seconds by the ReplicaSet controller exactly as any other Deployment-managed Pod would be — no special-cased recovery logic exists.
Common mistake: Treating CoreDNS as untouchable "system magic" instead of applying the same operational rigor (monitoring, resource limits, topology spread) used for any other critical Deployment.
7. Why does CoreDNS typically run with at least 2 replicas and topology spread constraints?
Short answer: For high availability — since literally every Pod in the cluster depends on it for name resolution, a single point of failure would be catastrophic.
Detailed explanation: Spreading replicas across nodes (via topologySpreadConstraints or anti-affinity) ensures a single node failure can't take down all CoreDNS replicas simultaneously, which would cause cluster-wide new-connection failures.
Production example: A CoreDNS Deployment with 2 replicas pinned via anti-affinity to different nodes survives a single node failure without any DNS resolution outage.
Common mistake: Running CoreDNS with a single replica "since it's just DNS," creating a severe single point of failure for the entire cluster.
8. What happens cluster-wide if CoreDNS goes down entirely, and why do existing connections keep working?
Short answer: New name resolutions fail cluster-wide, but already-established connections keep working since DNS is only consulted at connection-initiation time.
Detailed explanation: This creates a deceptive symptom pattern — existing traffic looks fine while anything attempting a fresh connection (including retries, reconnects, and new Pods starting up) fails, often masking the true root cause during an incident.
Production example: An outage where CoreDNS crash-loops might show healthy dashboards for long-lived connections while every new Pod rollout or reconnect attempt silently fails.
Common mistake: Ruling out a DNS outage too early during an incident because "some things still seem to be working."
9. Why is monitoring CoreDNS's own metrics (via the prometheus plugin) important, given it's "just infrastructure"?
Short answer: DNS latency spikes and error rates often precede wider outages, but teams frequently don't monitor CoreDNS directly and only notice it's degraded via downstream app-wide failures.
Detailed explanation: Metrics like coredns_dns_request_duration_seconds and request error rates give an early, direct signal of DNS health, rather than waiting to infer it indirectly from application-level symptoms across the whole cluster.
Production example: A Grafana dashboard scraping CoreDNS's :9153 metrics endpoint can alert on rising query latency minutes before it manifests as broad application timeouts.
Common mistake: Not scraping CoreDNS's Prometheus endpoint at all, treating it as invisible infrastructure until it's already causing a visible cluster-wide incident.
10. Why can a malformed Corefile crash-loop every CoreDNS replica simultaneously?
Short answer: All CoreDNS replicas typically share the exact same Corefile (via the same ConfigMap), so a syntax error affects every replica identically at once.
Detailed explanation: Unlike application config errors that might only affect a subset of replicas due to rolling deployment timing, a Corefile error is baked into the ConfigMap all replicas mount, meaning a bad edit + rollout restart can take down 100% of DNS capacity simultaneously.
Production example: An engineer editing the Corefile ConfigMap with a typo and then running kubectl rollout restart deployment coredns can crash every CoreDNS Pod at once, causing a full cluster DNS outage.
Common mistake: Editing the live Corefile ConfigMap directly in production without validating syntax first or testing in a non-production cluster.
11–50. Additional rapid-fire questions
11. What did CoreDNS replace as the Kubernetes DNS default? — kube-dns, around Kubernetes 1.13.
12. What language is CoreDNS written in? — Go.
13. What's the name of the plugin that handles cluster.local resolution? — The kubernetes plugin.
14. What port does CoreDNS listen on for DNS traffic? — 53 (both TCP and UDP).
15. What port exposes CoreDNS's Prometheus metrics by convention? — 9153.
16. What plugin exposes CoreDNS's own liveness/readiness endpoints? — health (liveness) and ready (readiness).
17. What does the errors plugin do? — Logs DNS errors to stdout for visibility/debugging.
18. What does the reload plugin enable? — Auto-reloading Corefile changes without restarting the CoreDNS Pod.
19. What does the loadbalance plugin do? — Round-robin shuffles the order of A/AAAA records returned, for basic client-side load spreading.
20. What does pods insecure do inside the kubernetes plugin block? — Allows resolving Pod IPs directly via a pod-ip-derived DNS name, without stricter verification.
21. How do you view the currently active Corefile? — kubectl get configmap coredns -n kube-system -o yaml.
22. How do you edit the Corefile? — kubectl edit configmap coredns -n kube-system.
23. Does editing the Corefile ConfigMap immediately affect running CoreDNS Pods? — With the reload plugin present, yes, automatically within its check interval — otherwise a manual rollout restart is needed.
24. What's the CoreDNS-autoscaler add-on used for? — Automatically scaling CoreDNS replica count in proportion to cluster size (node/core count).
25. Why does CoreDNS replica count need to scale with cluster size? — Larger clusters generate more DNS query volume and more watched Service/Endpoint objects, increasing both query load and memory footprint.
26. What happens if the kubernetes plugin loses its watch connection to the API Server? — It attempts to reconnect/resync; until then, served answers may become stale or queries may fail depending on how the plugin handles the gap.
27. Is CoreDNS aware of NetworkPolicies? — No — DNS resolution and NetworkPolicy enforcement are entirely separate systems; NetworkPolicy could still block the actual traffic even after a successful DNS answer.
28. What's the lameduck setting under the health plugin for? — Reports unhealthy for a grace period (e.g. 5s) before shutdown, allowing load balancers/kube-proxy to stop routing to a terminating Pod cleanly.
29. Why does CoreDNS's own readiness probe matter during its rolling updates? — Ensures traffic isn't routed to a CoreDNS Pod that isn't fully initialized yet, avoiding self-inflicted resolution failures during CoreDNS's own deploys.
30. What's a realistic reason to add a custom zone/plugin block to the Corefile? — Resolving an internal corporate domain (e.g. an on-prem service) that isn't part of the standard cluster.local zone.
31. Can CoreDNS answer queries for a completely custom internal TLD? — Yes — additional server blocks in the Corefile can serve arbitrary zones alongside the standard cluster.local one.
32. What's the relationship between CoreDNS and the kube-dns Service name? — CoreDNS Pods sit behind a Service still conventionally named kube-dns for backward compatibility, even though kube-dns (the old server) is gone.
33. Why might CoreDNS's memory usage grow with cluster size even without more query volume? — It caches an in-memory representation of watched Services/Endpoints/Pods, which scales with object count, not just query rate.
34. What's a good first diagnostic step for slow DNS resolution cluster-wide? — Check CoreDNS Pod CPU/memory usage and its own request-duration metrics before assuming an application-level problem.
35. Does CoreDNS support DNS over TLS (DoT) for its own queries or forwarding? — Yes, in more recent configurations forward can be configured to use TLS to upstream resolvers, though it's not the historical default.
36. What's the effect of accidentally listing forward before kubernetes in the Corefile? — Cluster-internal queries could be sent upstream unnecessarily (and fail, since the upstream doesn't know cluster.local) before ever reaching the kubernetes plugin.
37. Why is fallthrough important in the kubernetes plugin's config? — Without it, unmatched queries within the configured zones might not correctly pass to subsequent plugins like forward.
38. What's a good rollout practice for Corefile changes in production? — Test in a non-production cluster first, validate syntax, and apply during low-traffic windows given the blast radius of a mistake.
39. Does CoreDNS itself enforce any RBAC on who can query it? — No — DNS queries are unauthenticated network requests; access control is a network/firewall concern, not a CoreDNS-level one.
40. What RBAC does CoreDNS's own ServiceAccount need? — Read (list/watch) access to Services, EndpointSlices, and optionally Pods/Namespaces, to populate its in-memory records.
41. Why would you scrape coredns_dns_requests_total specifically? — To track query volume trends and detect abnormal spikes that might indicate a misbehaving client or a forwarding loop starting to form.
42. What's a subtle risk of setting an extremely low cache TTL? — Increased load on both CoreDNS and the API Server watch/informer layer from far more frequent re-resolution, undermining the point of caching.
43. Can CoreDNS run with more plugins than the defaults shown in a typical install? — Yes — it's fully pluggable; third-party and custom plugins can be compiled in for specialized behavior.
44. What's the operational risk of running an outdated CoreDNS version long-term? — Missing bug fixes/performance improvements and potentially losing compatibility with newer Kubernetes API behaviors.
45. How would you validate a Corefile change before applying it in production? — Use a local/staging CoreDNS instance or dry-run tooling to check syntax and basic query behavior first.
46. What's the danger of manually restarting CoreDNS during a live incident without understanding the root cause? — Could briefly worsen an outage further (zero DNS capacity during the restart window) without fixing the underlying config/resource issue.
47. Does CoreDNS support IPv6 (AAAA) resolution? — Yes — the kubernetes plugin can serve AAAA records for dual-stack clusters.
48. What's a good way to load-test CoreDNS before a major cluster size increase? — Simulate expected query volume/Service count in a staging cluster and observe latency/error metrics before scaling production.
49. Why is CoreDNS considered a "cluster-critical add-on" despite being a normal Deployment? — Because nearly every other workload's ability to discover dependencies transitively depends on it being healthy.
50. What's the first thing to check when DNS resolution is failing cluster-wide? — kubectl get pods -n kube-system -l k8s-app=kube-dns for CoreDNS Pod health, followed by its logs for errors/crash-loops.
