Taints

Node-side repellents that keep pods away unless they explicitly tolerate them.
← Anti-Affinity Course Index Tolerations →

Explanation

A taint is a property applied to a node (not a Pod) that repels Pods from scheduling onto it unless the Pod carries a matching toleration. Where affinity/anti-affinity are Pod-side preferences, taints are the node's way of saying "I don't want to run arbitrary workloads — only Pods that have explicitly agreed to tolerate my condition." This inverts the usual relationship: instead of every Pod having to know about every special node, the node advertises a taint once, and only Pods that opt in via a toleration can land there.

Kubernetes needs taints because clusters are rarely homogeneous. Some nodes have GPUs and should only run ML workloads; some nodes are being drained for maintenance and should run nothing new; some nodes are dedicated to a particular team or workload class for cost-allocation or compliance reasons; and some nodes are simply unhealthy and must repel all pods immediately. Taints give the platform team a first-class, declarative mechanism for all of these cases without touching every Deployment's affinity rules.

A taint has three parts: key, value, and effect. The effect determines the severity of the repulsion:

  • NoSchedule — the Scheduler will not place new Pods here, but existing Pods already running are left alone.
  • PreferNoSchedule — a soft version; the Scheduler tries to avoid the node but will use it if there's no better option.
  • NoExecute — the strongest effect: new Pods are not scheduled, and existing Pods without a matching toleration are evicted. This is also what Kubernetes uses automatically for node health conditions (e.g. node.kubernetes.io/not-ready, node.kubernetes.io/unreachable).

Taints and tolerations exist as a pair by design — a taint alone can't be "matched," it needs a Pod's toleration to cancel it out for that specific Pod. This is deliberately asymmetric to node affinity's nodeSelector/ requiredDuringScheduling, which is Pod-driven attraction; taints are node-driven repulsion, and the two mechanisms are often combined: taint the GPU nodes so only GPU-workload Pods (which tolerate the taint and have a node affinity/selector pulling them there) can land.

In production, the most common taint pattern is node dedication: taint a pool of nodes with dedicated=ml:NoSchedule, then only Pods carrying the matching toleration plus a node affinity/selector for that pool will actually run there — the taint keeps everyone else out, the affinity pulls the intended workload in. Control-plane nodes in kubeadm clusters are tainted with node-role.kubernetes.io/control-plane:NoSchedule by default for exactly this reason: keep untrusted workloads off the machines running etcd and the API Server.

NoExecute is destructive by default. A NoExecute taint without a tolerationSeconds escape hatch on the Pod evicts it immediately. Kubernetes itself relies on this for the built-in not-ready/unreachable taints — which is exactly how Pods get rescheduled off a node that just went offline.

Internal Working / Interactions

  • Node Controller (Controller Manager): automatically adds/removes built-in taints like node.kubernetes.io/not-ready and node.kubernetes.io/unreachable based on node heartbeat status — this is the automatic self-healing path.
  • API Server: stores taints as part of the Node object's spec.taints array; kubectl taint is just a PATCH against this field.
  • Scheduler: the TaintToleration plugin runs during Filter (for NoSchedule) and Score (for PreferNoSchedule) to decide eligibility; it never overrides a hard taint mismatch.
  • kubelet: when a NoExecute taint appears on its node (or the node condition changes), the kubelet (via the API Server-driven eviction controller) starts a timer per Pod based on that Pod's tolerationSeconds, then terminates the Pod locally when the timer expires.
  • etcd: holds the current taint list as part of the Node object, watched by both Scheduler and Controller Manager.

Code Example

# Applying a taint to a node (imperative — taints are node-level, not a YAML resource type
# you create standalone; they live under Node.spec.taints)
# kubectl taint nodes gpu-node-1 dedicated=ml:NoSchedule

# Equivalent declarative view of the resulting Node object (read-only status you'd see via
# `kubectl get node gpu-node-1 -o yaml`):
apiVersion: v1
kind: Node
metadata:
  name: gpu-node-1
spec:
  taints:
    - key: "dedicated"        # arbitrary key identifying the reason for the taint
      value: "ml"             # arbitrary value, must match toleration's value for Equal operator
      effect: "NoSchedule"    # repels new pods; existing pods on this node are unaffected

apiVersion: apps/v1
kind: Deployment
metadata:
  name: control-plane-critical-addon
  namespace: kube-system
spec:
  replicas: 1
  selector:
    matchLabels: { app: critical-addon }
  template:
    metadata:
      labels: { app: critical-addon }
    spec:
      # Explicitly tolerate the built-in control-plane taint so this addon can run there
      # (common for CNI plugins, kube-proxy, and other node-critical DaemonSets).
      tolerations:
        - key: "node-role.kubernetes.io/control-plane"
          operator: "Exists"     # matches any value for this key
          effect: "NoSchedule"
        - key: "node.kubernetes.io/not-ready"
          operator: "Exists"
          effect: "NoExecute"
          tolerationSeconds: 300   # stay for 5 min after node goes NotReady before evicting
      containers:
        - name: addon
          image: registry.internal/critical-addon:1.0

kubectl Commands

# Add a taint imperatively
kubectl taint nodes gpu-node-1 dedicated=ml:NoSchedule

# Remove a taint (note the trailing minus sign)
kubectl taint nodes gpu-node-1 dedicated=ml:NoSchedule-

# List all taints across all nodes
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.taints}{"\n"}{end}'

# See the built-in control-plane taint
kubectl describe node  | grep Taints

# Check whether a specific pod would be affected by a taint (look at Events on describe)
kubectl describe pod  | grep -A5 Events

# View default cluster taints (e.g. node.kubernetes.io/not-ready) live
kubectl get nodes -o custom-columns=NAME:.metadata.name,TAINTS:.spec.taints

Best Practices & Common Pitfalls

Do's

  • Use taints for node dedication (GPU pools, spot-instance pools, team-specific pools) and pair with matching tolerations plus affinity.
  • Set tolerationSeconds explicitly on latency-sensitive workloads so you control eviction delay rather than relying on defaults.
  • Use PreferNoSchedule for soft guidance (e.g. "prefer not to put general workloads here") vs NoSchedule for hard boundaries.
  • Document every custom taint's purpose — undocumented taints are a common cause of "why won't my pod schedule" tickets.

Don'ts

  • Don't taint nodes with NoExecute casually — it evicts running workloads immediately, unlike NoSchedule.
  • Don't forget the built-in not-ready/unreachable taints already handle most failure-based eviction — don't duplicate this logic manually.
  • Don't use taints as a security boundary by themselves — they control scheduling, not runtime isolation; combine with NetworkPolicies/RBAC for real isolation.
  • Don't forget to remove taints left over from decommissioned dedication schemes — stale taints silently block scheduling for years.

50 Interview Questions

1. What is a taint and what does it do?
Short answer: A property on a Node that repels Pods from scheduling there unless the Pod has a matching toleration.
Detailed explanation: Taints invert the usual model — instead of Pods declaring what they want, the node declares what it will refuse, and only Pods that explicitly opt in via a toleration can land.
Production example: A GPU node pool tainted dedicated=ml:NoSchedule keeps ordinary web workloads off expensive hardware.
Common mistake: Thinking a taint alone routes the intended workload to the node — it only blocks everyone else; affinity is what actually pulls the target workload in.
2. What are the three taint effects and how do they differ?
Short answer: NoSchedule (block new Pods only), PreferNoSchedule (soft avoidance), NoExecute (block new Pods and evict existing ones without a matching toleration).
Detailed explanation: NoSchedule and PreferNoSchedule only affect the Filter/Score phases at scheduling time; NoExecute is the only effect that acts on already-running Pods, actively evicting them.
Production example: Kubernetes automatically applies node.kubernetes.io/not-ready:NoExecute when a node's heartbeats stop, evicting Pods that don't tolerate it.
Common mistake: Applying NoExecute casually, not realizing it immediately evicts running workloads unlike NoSchedule.
3. What are the three parts of a taint?
Short answer: key, value, and effect.
Detailed explanation: The key/value form an arbitrary label-like identifier for the reason behind the taint; the effect determines the scheduling/eviction severity applied against Pods lacking a matching toleration.
Production example: dedicated=ml:NoSchedule — key=dedicated, value=ml, effect=NoSchedule.
Common mistake: Forgetting effect is part of the match — a toleration must match key, value (for Equal), and effect together.
4. Why does Kubernetes automatically taint control-plane nodes?
Short answer: To keep untrusted or arbitrary workloads off the machines running etcd and the API Server.
Detailed explanation: kubeadm clusters apply node-role.kubernetes.io/control-plane:NoSchedule by default so regular application Pods never compete for resources with, or gain proximity to, the control plane.
Production example: Only cluster-critical DaemonSets (CNI, kube-proxy) carry tolerations for this taint so they still run everywhere.
Common mistake: Manually removing the control-plane taint to "get more capacity," inadvertently exposing etcd/API Server nodes to arbitrary workloads.
5. How does the node dedication pattern work using taints?
Short answer: Taint the dedicated pool so unrelated Pods are repelled, then give the intended workload both a matching toleration and a node affinity/selector pulling it there.
Detailed explanation: The taint alone only opens/closes the door; the affinity is what actually walks the intended Pod through it. Without both, you either get unwanted Pods leaking onto the pool (missing taint) or the intended Pod not landing there at all (missing affinity).
Production example: ML training Pods tolerate dedicated=ml:NoSchedule and carry a nodeAffinity for node-pool=gpu-pool.
Common mistake: Adding only the toleration and assuming it's enough to route the workload onto the dedicated nodes.
6. Which component automatically manages the built-in not-ready/unreachable taints?
Short answer: The Node Controller inside the Controller Manager.
Detailed explanation: It watches node heartbeat status and adds/removes these NoExecute taints based on missed heartbeats, driving the automatic eviction/rescheduling path for node failures.
Production example: A node losing network connectivity gets tainted node.kubernetes.io/unreachable:NoExecute within the node-monitor-grace-period window.
Common mistake: Assuming kubelet applies these taints — kubelet only executes eviction once the timer expires, it doesn't decide node health status itself.
7. What is tolerationSeconds and why does it only apply to NoExecute?
Short answer: It's a Pod-side grace period before eviction after a NoExecute taint appears; it's meaningless for NoSchedule/PreferNoSchedule since those never evict.
Detailed explanation: It controls how long a Pod is allowed to keep running after a node goes NotReady/unreachable before being forcibly removed, directly tuning failover speed.
Production example: A critical add-on tolerates not-ready:NoExecute with tolerationSeconds: 300, riding out brief network blips.
Common mistake: Setting it too high for latency-sensitive services, delaying failover after a genuine node failure.
8. Are taints a sufficient security boundary on their own?
Short answer: No — taints only control scheduling, not runtime isolation.
Detailed explanation: A Pod that somehow lands on a tainted node (or one already running before the taint was applied) still shares the kernel/network unless NetworkPolicies, RBAC, and other isolation controls are also applied.
Production example: A compliance review flagged that GPU-pool taints alone didn't prevent lateral network access from co-located Pods, requiring NetworkPolicy as well.
Common mistake: Treating "tainted node" as equivalent to "isolated node" for security purposes.
9. How would you remove a taint from a node?
Short answer: kubectl taint nodes <node> key=value:effect- — note the trailing minus sign.
Detailed explanation: The API Server treats this as a PATCH removing that entry from Node.spec.taints; the Scheduler and Node Controller pick up the change on their next watch event.
Production example: Decommissioning a dedicated pool required removing dedicated=ml:NoSchedule from every node in the pool before repurposing it.
Common mistake: Forgetting the trailing -, which causes kubectl to try to add rather than remove the taint.
10. What's a common production incident caused by stale taints?
Short answer: Undocumented, leftover taints from a decommissioned dedication scheme silently block scheduling on otherwise healthy nodes.
Detailed explanation: Because taints are node-level and invisible unless explicitly inspected, they can persist for years after their original purpose is gone, confusing later engineers about why Pods won't schedule.
Production example: A "why won't my pod schedule" ticket traced back to a two-year-old taint from a since-abandoned team-dedication experiment.
Common mistake: Not documenting every custom taint's purpose and owner at creation time.
11–50. Additional rapid-fire questions
11. Where are taints stored? — In Node.spec.taints, an array field on the Node object in etcd.
12. Which Scheduler plugin evaluates taints? — The TaintToleration plugin, during Filter (NoSchedule/NoExecute) and Score (PreferNoSchedule).
13. Does a taint affect Pods already running on the node when first applied with NoSchedule? — No, NoSchedule only blocks new Pods; existing Pods are left alone.
14. Does NoExecute affect already-running Pods? — Yes, it evicts any running Pod lacking a matching toleration.
15. What's the imperative command to add a taint? — kubectl taint nodes <node> key=value:effect.
16. How do you list taints across all nodes? — kubectl get nodes -o jsonpath pulling .spec.taints, or custom-columns.
17. Is a taint a standalone API object? — No, it's a field within the Node object's spec, not its own resource kind.
18. What effect does PreferNoSchedule have during the Filter phase? — None — it only influences scoring, never filters out a node.
19. Why is dedicating nodes with taints preferred over relying on affinity alone? — Affinity alone doesn't stop unrelated workloads from landing on the pool; the taint is what actively repels them.
20. What happens if you taint a node but forget to add a matching toleration to the intended workload? — The intended workload also gets blocked, since it has no toleration either.
21. Which component executes the actual container termination during a NoExecute eviction? — kubelet, once the eviction controller's timer for that Pod expires.
22. Do DaemonSets automatically tolerate all taints? — Not automatically for custom taints, but the DaemonSet controller injects tolerations for several built-in node conditions by default.
23. What's the relationship between taints and the API Server's admission control? — None directly — taints are evaluated by the Scheduler and Node Controller, not by admission webhooks.
24. Can a single node have multiple taints? — Yes, spec.taints is an array; a Pod's tolerations must cover every taint that would otherwise block it.
25. What's the default grace period before eviction if tolerationSeconds is unset on a not-ready taint? — Kubernetes' built-in default toleration typically applies 300 seconds unless the Pod specifies otherwise.
26. Why might a platform team use PreferNoSchedule instead of NoSchedule for a new pool? — To softly steer workloads away during a migration without immediately blocking scheduling if no other capacity exists.
27. How does a spot-instance pool commonly use taints? — Tainted so only workloads that explicitly tolerate interruption/spot-eviction risk are scheduled there.
28. What's the risk of taints without documentation across a large platform team? — Engineers waste time diagnosing "why won't this pod schedule" tickets caused by unexplained legacy taints.
29. Do taints have any effect on Services/Endpoints? — No, taints only affect scheduling eligibility, not traffic routing.
30. What's the difference between a taint and a node label used in nodeSelector? — A label is neutral/attractive metadata a Pod can select on; a taint is an active repellent that blocks Pods by default.
31. Why is NoExecute the effect used for automatic failure handling? — Because on real node failure you want existing Pods actively evicted and rescheduled, not just blocked from new placement.
32. What field would you check to see why a pod isn't scheduling onto a specific node? — kubectl describe pod Events, cross-referenced with kubectl describe node | grep Taints.
33. Can you taint a node to isolate it for maintenance? — Yes, applying a NoSchedule or NoExecute taint is a common way to prevent new workloads while draining for maintenance.
34. What's the interaction between a taint and Cluster Autoscaler node groups? — CA must simulate whether a new node's taints would still block a Pending pod before scaling that node group up.
35. Is a taint enough to guarantee compliance isolation for regulated workloads? — No — taints are scheduling-only; combine with RBAC and NetworkPolicies for real compliance-grade isolation.
36. How do you verify the built-in control-plane taint exists? — kubectl describe node <control-plane-node> | grep Taints.
37. What happens to a Pod with no tolerations at all when its node gets any taint applied? — It's affected by every effect present — blocked from re-scheduling and evicted if any taint is NoExecute.
38. Why is manually removing the not-ready taint logic risky? — It's the exact automatic mechanism that reschedules Pods off failed nodes; disabling it removes self-healing for node failures.
39. What's a realistic multi-tenant use of taints? — Dedicating a node pool per team for cost-allocation/compliance, with each team's Pods carrying the matching toleration.
40. How does a taint's effect interact with a toleration missing that same effect? — No match occurs — the toleration must specify the same effect (or omit it, matching all effects for that key) to cancel the taint.
41. Why are taints described as "node-driven repulsion" versus affinity's "Pod-driven attraction"? — Because the node declares the repellent once via a taint, rather than every Pod needing to know about every special node.
42. What's the operational cost of leaving stale taints on decommissioned pools? — They silently block scheduling on otherwise usable capacity, sometimes for years, wasting cluster resources.
43. How would an interviewer expect you to answer "what's the difference between a taint and anti-affinity"? — Taint = node-side repulsion against any non-tolerating Pod; anti-affinity = Pod-side rule about other Pods' placement.
44. What's the risk of applying NoExecute to a node hosting a singleton stateful workload with no PDB? — Immediate eviction with no availability guard, potentially causing real downtime.
45. Can a taint be scoped to only affect specific namespaces? — No, taints are node-level and apply cluster-wide to any Pod lacking a matching toleration, regardless of namespace.
46. What's the quickest way to confirm a taint is actually blocking a specific Pending pod? — Check the FailedScheduling event text in kubectl describe pod, which names the untolerated taint.
47. Why do platform teams often standardize taint key naming conventions? — To keep toleration keys/values consistent across the org and avoid confusion between similarly-purposed taints.
48. What's the failure mode of forgetting to re-taint a repaired node before returning it to service? — None by itself — the concern is the opposite: forgetting to *remove* a taint after a workaround, blocking future scheduling.
49. How does taint-based node dedication reduce noisy-neighbor risk? — By ensuring only intentionally-tolerating workloads land on a pool, isolating resource contention to a known set of workloads.
50. What's the one-line distinction to remember for interviews? — Taint = node says "keep out unless you tolerate me"; toleration = Pod's permission slip; affinity = Pod's explicit request to go there.
