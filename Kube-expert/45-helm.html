Helm

The package manager for Kubernetes: templated manifests bundled into versioned, installable charts.
← Pod Security Course Index Operators →

Explanation

A real application is rarely one YAML file — it's a Deployment, a Service, a ConfigMap, an Ingress, maybe an HPA and a NetworkPolicy, and every environment (dev/staging/prod) needs slightly different values (replica counts, resource limits, image tags). Copy-pasting and hand-editing YAML across environments doesn't scale and is error-prone. Helm solves this by treating a Kubernetes application as a chart: a directory of templated YAML plus a single values.yaml file of parameters, rendered into concrete manifests at install time.

A chart's structure is: Chart.yaml (name, version, dependencies — chart metadata), values.yaml (default parameter values), and a templates/ directory containing Go-template YAML files that reference those values ({{ .Values.replicaCount }}) plus built-in objects like {{ .Release.Name }}. Helm's templating engine is just Go's text/template with Sprig functions added, run entirely client-side (by the helm binary or, in cluster automation, by Argo CD/Flux) — the API Server never sees templates, only the final rendered manifests.

Kubernetes needs Helm (or something like it) because raw manifests have no concept of parameterization, versioning, or release lifecycle. Helm introduces a Release: a named, versioned instance of a chart installed into a cluster, tracked via a Secret Helm stores in the target namespace recording exactly what was applied. This is what makes helm upgrade and helm rollback possible — Helm diffs your new render against the last recorded release state and can revert to any previous revision.

The release lifecycle is: helm install renders the chart with your values and applies the resulting manifests, recording revision 1. helm upgrade re-renders with new values/chart version, applies the diff, and records a new revision. helm rollback <revision> re-applies a prior revision's rendered manifests without needing the original chart source. Helm also supports hooks (e.g. pre-install, post-upgrade) — special annotated resources (often Jobs) that run at specific lifecycle points, commonly used for database migrations before a new app version goes live.

Helm charts can declare dependencies on other charts (subcharts) in Chart.yaml, letting you compose a complex stack (e.g. an umbrella chart pulling in Redis, Postgres, and your app chart) from reusable public or internal building blocks — this is the same compositional idea GitOps tools later build the "App-of-Apps" pattern on top of.

Internal Working / Interactions

  • Helm client: renders templates locally using values.yaml + --set overrides + the chart's Go templates; the API Server is never aware templating happened.
  • API Server: receives only the final, rendered plain YAML/JSON manifests via normal kubectl apply-equivalent calls — same authn/authz/admission pipeline as any other request.
  • Release storage: Helm 3 stores each release's rendered manifest and metadata as a Secret (type helm.sh/release.v1) in the target namespace — this is Helm's "database," with no separate Tiller server (removed in Helm 3).
  • Controllers: once manifests are applied, ordinary Kubernetes controllers (Deployment, ReplicaSet, etc.) reconcile them exactly as if you'd written them by hand — Helm has no runtime involvement after apply.

Code Example

# Chart.yaml — chart metadata
apiVersion: v2
name: checkout-service
version: 1.4.0          # Chart version (semver) — bump on any template change.
appVersion: "2.4.1"      # Version of the application the chart deploys — informational.
description: Checkout microservice Helm chart

---
# values.yaml — default parameters, overridable per environment
replicaCount: 3
image:
  repository: registry.internal/checkout-service
  tag: "2.4.1"
resources:
  requests: { cpu: 250m, memory: 256Mi }
  limits:   { cpu: 500m, memory: 512Mi }
service:
  port: 8080

---
# templates/deployment.yaml — Go-templated manifest referencing values above
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-checkout       # {{ .Release.Name }} = the release name at install time.
  labels:
    app: {{ .Chart.Name }}
spec:
  replicas: {{ .Values.replicaCount }}      # Pulled straight from values.yaml (or --set override).
  selector:
    matchLabels: { app: {{ .Chart.Name }} }
  template:
    metadata:
      labels: { app: {{ .Chart.Name }} }
    spec:
      containers:
        - name: checkout
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          resources:
            {{- toYaml .Values.resources | nindent 12 }}   {# Sprig 'toYaml' + 'nindent' for nested blocks #}
          ports:
            - containerPort: {{ .Values.service.port }}

kubectl Commands

helm install checkout ./checkout-service -f values-prod.yaml   # Install chart with an environment-specific overrides file
helm upgrade checkout ./checkout-service --set replicaCount=6    # Upgrade an existing release with an inline override
helm rollback checkout 3                                          # Roll back to revision 3 of this release
helm history checkout                                              # List all revisions and their status
helm template ./checkout-service -f values-prod.yaml               # Render manifests locally without installing (great for review/CI)
helm diff upgrade checkout ./checkout-service -f values-prod.yaml   # (helm-diff plugin) preview exact changes before upgrading
helm uninstall checkout                                             # Remove the release and its resources
helm lint ./checkout-service                                        # Validate chart templates/values for common errors

Best Practices & Common Pitfalls

Best Practices

  • Keep one values file per environment (values-dev.yaml, values-prod.yaml) rather than scattering --set flags across CI scripts.
  • Always run helm template or helm diff in CI to review rendered output before applying.
  • Pin exact chart and image versions in production — never install against a floating latest tag or unpinned dependency.
  • Use helm lint and schema validation (values.schema.json) to catch bad values early.

Pitfalls

  • Overusing template logic (deeply nested conditionals) until charts become unreadable — prefer simple, composable templates.
  • Forgetting that helm upgrade doesn't automatically delete resources removed from templates in some edge cases (orphaned resources) — audit periodically.
  • Storing secrets in plaintext values.yaml committed to git — use Sealed Secrets, SOPS, or external-secrets instead.
  • Treating Helm as a full GitOps solution — it's a packaging/templating tool, not a continuous reconciler (that's what Argo CD/Flux add on top).

50 Interview Questions

1. What are the three core pieces of a Helm chart's structure?
Short answer: Chart.yaml (metadata), values.yaml (default parameters), and templates/ (Go-template YAML files).
Detail: Chart.yaml declares name/version/dependencies, values.yaml supplies the default parameter values templates reference, and templates/ contains the actual manifest templates rendered against those values.
Production example: A checkout-service chart with Chart.yaml at version 1.4.0, values.yaml defining replicaCount and image tag, and templates/deployment.yaml referencing both.
Common mistake: Hardcoding environment-specific values directly into templates instead of parameterizing via values.yaml.
2. Does the Kubernetes API Server ever see Helm's Go templates?
Short answer: No — templating happens entirely client-side; the API Server only ever receives fully-rendered plain YAML/JSON.
Detail: The helm binary (or Argo CD/Flux acting as the client) renders templates locally using values.yaml plus overrides, then applies the resulting concrete manifests via the normal API pipeline.
Production example: Running helm template to inspect exactly what would be sent to the API Server, with zero cluster interaction.
Common mistake: Assuming the cluster stores or is aware of the templated source — it only ever sees rendered output.
3. What is a Helm "Release" and how is its state tracked?
Short answer: A named, versioned instance of a chart installed into a cluster, tracked as a Secret of type helm.sh/release.v1 in the target namespace.
Detail: Each install/upgrade records a new revision in that Secret, storing the rendered manifest and metadata — this is what makes rollback possible without needing the original chart source again.
Production example: helm history checkout listing revisions 1 through 5, each backed by its own release Secret revision.
Common mistake: Manually deleting a release Secret, breaking Helm's ability to track or roll back that release.
4. What happened to Tiller, and why does Helm 3 not need it?
Short answer: Tiller was Helm 2's in-cluster server component; Helm 3 removed it entirely, doing all rendering client-side and storing state as Secrets instead.
Detail: Tiller required broad, often cluster-admin-level RBAC to operate, which was a major security concern; Helm 3's client-only architecture eliminates that standing privileged component.
Production example: Helm 3 clusters have no tiller-deploy Pod at all — install/upgrade calls go straight from the helm CLI to the API Server.
Common mistake: Following outdated Helm 2 documentation that references Tiller RBAC setup on a Helm 3 cluster.
5. How does helm rollback work without access to the original chart source?
Short answer: It re-applies the already-rendered manifest stored in a prior revision's release Secret, not the chart template.
Detail: Because each revision's fully-rendered output is persisted, Helm can revert to any past revision purely from that stored state, even if the chart repo has since changed or is unavailable.
Production example: helm rollback checkout 3 reverting to exactly what was applied at revision 3, regardless of what the chart looks like today.
Common mistake: Assuming rollback re-renders the chart with old values — it replays the stored rendered manifest directly.
6. What is a Helm hook, and what's a common production use case?
Short answer: Specially annotated resources (often Jobs) that run at specific lifecycle points like pre-install or post-upgrade.
Detail: Hooks let you sequence operations around the main install/upgrade — most commonly running database migrations before the new application version's Pods actually start serving traffic.
Production example: A pre-upgrade hook Job running Flyway/Django migrations before the Deployment's new image is rolled out.
Common mistake: Relying on hooks for ongoing reconciliation logic — hooks run once per lifecycle event, they're not a continuous controller.
7. Why is Helm described as "not a full GitOps solution"?
Short answer: Helm renders and applies once per command invocation — it has no continuous reconciliation loop watching for drift.
Detail: A GitOps tool like Argo CD or Flux continuously compares live cluster state against desired state and can auto-correct drift; Helm itself just performs a one-time templating+apply operation when invoked.
Production example: Argo CD invoking `helm template` under the hood to render manifests, then handling the ongoing sync/self-heal loop itself.
Common mistake: Treating a scheduled `helm upgrade` cron job as equivalent to a real GitOps reconciler.
8. What's the risk of "orphaned resources" after a helm upgrade?
Short answer: Resources removed from a chart's templates aren't always automatically deleted from the cluster on upgrade in certain edge cases, leaving stale objects behind.
Detail: This can lead to unexpected leftover ConfigMaps, Services, or other objects that no longer correspond to anything in the current chart version, silently consuming resources or causing confusion.
Production example: Renaming a ConfigMap in a template and later discovering the old-named ConfigMap still exists in the cluster from a previous release.
Common mistake: Never auditing for orphaned resources, assuming helm upgrade is always a perfectly clean state transition.
9. Why should charts avoid storing secrets directly in values.yaml committed to git?
Short answer: Plaintext secrets in values.yaml become plaintext secrets in git history, readable by anyone with repo access, forever (even if later removed).
Detail: The recommended pattern is to keep values.yaml free of real secret material and instead reference Sealed Secrets, SOPS-encrypted values, or an external-secrets operator that pulls from a vault at deploy time.
Production example: A values-prod.yaml referencing a Sealed Secret's name rather than embedding a raw database password.
Common mistake: Committing a values-prod.yaml with a literal API key or password "just for now" during a rushed deploy.
10. What's the purpose of subcharts / chart dependencies declared in Chart.yaml?
Short answer: They let one "umbrella" chart compose several reusable charts (e.g. Redis, Postgres, your app) into a single installable unit.
Detail: Each dependency is itself a normal chart, and the parent's values.yaml can override subchart values under a namespaced key — the same compositional idea GitOps tools later build "App-of-Apps" on top of.
Production example: An umbrella chart for a platform stack pulling in a public Redis chart plus an internal checkout-service chart as dependencies.
Common mistake: Duplicating a common dependency's manifests by hand across many charts instead of declaring it once as a shared subchart.
11–50. Additional rapid-fire questions
11. What file declares a chart's name, version, and dependencies? — Chart.yaml.
12. What templating engine does Helm use? — Go's text/template with Sprig functions added.
13. What built-in object gives the release's name inside a template? — {{ .Release.Name }}.
14. How do you render a chart locally without installing it? — helm template.
15. What command previews exact changes before an upgrade? — helm diff upgrade (helm-diff plugin).
16. What command validates a chart's templates/values for common errors? — helm lint.
17. What removes a release and its resources entirely? — helm uninstall.
18. What lists all revisions of a release and their status? — helm history.
19. What Kubernetes object type stores Helm 3 release state? — A Secret of type helm.sh/release.v1.
20. Does Helm 3 require a Tiller server component? — No, Tiller was removed entirely in Helm 3.
21. What Sprig function converts a values structure to YAML inside a template? — toYaml.
22. What does the `nindent` function commonly pair with toYaml for? — Correctly indenting nested YAML blocks inside a template.
23. What's a values.schema.json used for? — Validating values.yaml structure/types early, catching bad values before templating.
24. Can chart version and appVersion differ? — Yes — chart version tracks the chart's own template changes; appVersion tracks the application version it deploys.
25. What happens to the API Server's admission pipeline for Helm-applied manifests? — Nothing special — they go through the exact same authn/authz/admission pipeline as any kubectl apply.
26. What's the recommended way to manage per-environment overrides? — Separate values files per environment (values-dev.yaml, values-prod.yaml) rather than scattered --set flags.
27. What's a common failure mode of deeply nested chart template logic? — Charts become unreadable and hard to maintain — prefer simple, composable templates.
28. Does helm upgrade automatically pick up new default values from a new chart version? — Yes for any value not explicitly overridden by the caller; overridden values persist unless removed.
29. What's the purpose of pinning exact chart and image versions in production? — Avoiding unpredictable behavior from floating "latest" tags or unpinned dependency versions.
30. How does a subchart's values get referenced from the parent chart? — Under a key matching the subchart's name in the parent's values.yaml.
31. What's the risk of using `--set` flags scattered across CI scripts instead of values files? — Harder to audit/reproduce exactly what was deployed — no single reviewable source of truth.
32. Is a Helm hook Job automatically retried on failure? — Not by default — hook behavior/retry depends on annotations and the underlying Job's backoffLimit.
33. What's the effect of setting a hook's delete-policy annotation? — Controls whether the hook resource is cleaned up before creation, after success, or left behind on failure for debugging.
34. Can Argo CD or Flux use Helm charts as a GitOps source? — Yes — they invoke Helm's rendering logic and add continuous reconciliation and drift correction on top.
35. What's the difference between helm upgrade --install and plain helm upgrade? — --install creates the release if it doesn't already exist, otherwise it fails if no release is present.
36. Why is `helm lint` useful in CI? — Catches template/value errors (missing required values, YAML issues) before ever reaching the cluster.
37. What happens to a release's history if you helm uninstall it? — By default the release is fully removed including history, unless --keep-history is used.
38. What's a common reason for "helm upgrade" to hang? — A resource stuck waiting on --wait for readiness (e.g. a Deployment that never becomes Ready) if wait flags are set.
39. Can two different releases install the same chart into different namespaces? — Yes, releases are independent instances distinguished by release name and namespace.
40. What's the benefit of `helm template` in a CI pipeline specifically? — Lets you diff/review the exact manifests that would be applied without touching any cluster.
41. How does Helm handle CRDs shipped inside a chart's crds/ directory? — They're installed once on first install and not upgraded/deleted automatically on subsequent helm upgrade or uninstall.
42. What's a risk of relying on the crds/ directory's install-once behavior? — Schema changes to a chart's CRD in a new version won't automatically apply — you must update the CRD manually.
43. Does helm rollback re-run pre-install/pre-upgrade hooks? — It runs rollback-specific hooks (pre-rollback/post-rollback) rather than install/upgrade hooks.
44. What's the purpose of chart annotations vs labels in Chart.yaml? — Annotations carry non-identifying metadata (e.g. category), while labels can be used for selection/categorization by tooling.
45. Is Helm aware of resources it didn't create but that happen to match its manifests? — Generally no — Helm tracks ownership via its own release metadata, not by matching arbitrary existing resources.
46. What's a best practice for validating values before templating in CI? — Running `helm lint` plus schema validation against values.schema.json.
47. Why might `helm diff upgrade` show unexpected changes even with no values modified? — A new chart version's template logic itself changed, altering rendered output independent of values.
48. Can a single Helm chart deploy resources across multiple namespaces? — Yes, if templates explicitly set differing metadata.namespace fields, though this is uncommon and often discouraged for clarity.
49. What's the tradeoff of an umbrella chart pulling in many subcharts vs separate independent charts? — Simplifies coordinated deploys of a stack but couples their release lifecycle together, making independent upgrades harder.
50. What's the first thing to check when a helm upgrade silently doesn't change anything? — Whether the values/chart version actually changed, and whether `helm diff` shows any difference at all before assuming a bug.
