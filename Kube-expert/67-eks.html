EKS (Amazon Elastic Kubernetes Service)

AWS's managed Kubernetes: managed node groups vs Fargate, IRSA for pod-level IAM, and the VPC CNI's real-IP networking model.
← GKE Course Index AKS →

Explanation

EKS manages the control plane (etcd, API server, scheduler, controller-manager) across multiple AZs on AWS's behalf, billed as a flat hourly fee per cluster, while you retain full choice over how compute is provisioned. There are three compute options, and picking the wrong default costs real operational effort later. Managed node groups are AWS-provisioned Auto Scaling Groups of EC2 instances where AWS handles node provisioning, lifecycle, and (optionally) automated version upgrades, but you still choose instance types, and the nodes are real VMs you could SSH into if needed. Self-managed node groups give you full EC2/AMI control at the cost of doing everything yourself — rarely the right choice today. Fargate removes nodes entirely: each Pod gets its own isolated micro-VM sized to the Pod's resource requests, billed per-vCPU/memory-second with zero node patching or capacity planning — the tradeoff is no DaemonSets, no privileged Pods, and higher per-unit compute cost, making it best for spiky, less latency-sensitive, or security-isolation-sensitive workloads rather than your entire fleet.

IRSA (IAM Roles for Service Accounts) is EKS's answer to "how does a Pod get AWS credentials without static access keys." EKS runs an OIDC identity provider for the cluster; a Kubernetes ServiceAccount is annotated with an IAM role ARN, and that IAM role's trust policy is scoped to trust only that specific OIDC subject (namespace + service account name). When a Pod using that ServiceAccount calls AWS APIs via the SDK, the injected projected service-account token is exchanged (via sts:AssumeRoleWithWebIdentity) for short-lived AWS credentials scoped exactly to that IAM role's permissions — no shared node-level IAM role, no long-lived keys, and each workload's blast radius is limited to its own explicitly granted permissions. This is functionally identical in purpose to GKE's Workload Identity, just implemented through AWS STS and cluster OIDC federation.

The VPC CNI is what makes EKS networking distinctive versus most other Kubernetes networking options: instead of an overlay network, the VPC CNI assigns each Pod a real, routable VPC IP address directly from the node's ENI (Elastic Network Interface) secondary IP pool. This means Pods can be reached directly by other AWS resources (RDS, EC2, on-prem via Direct Connect) without NAT translation, security groups can apply directly to Pods (via SecurityGroupPolicy), and there's no encapsulation overhead — but it also means your VPC subnet IP space must be sized generously, since every Pod consumes a real VPC IP, and small subnets run out of IPs far faster than people expect at scale (a common EKS production incident: pods stuck in ContainerCreating because the subnet ran out of free IPs).

Practical defaults: use managed node groups for most steady-state workloads, add a Fargate profile for namespaces with spiky/untrusted/low-effort workloads (e.g. CI runners, batch jobs), always use IRSA instead of node-instance-profile IAM for workload permissions, and size VPC subnets (or enable custom networking / prefix delegation in the CNI) with real Pod-count headroom before you hit production scale.

Internal Working / Interactions

  • Control plane: fully managed by AWS across 3 AZs; exposed via a managed API server endpoint (public, private, or both) that you configure at cluster creation.
  • Managed node groups: backed by an ASG; AWS handles rolling AMI upgrades respecting PodDisruptionBudgets when you trigger a node-group update.
  • Fargate: a Fargate profile matches Pods by namespace/labels; matching Pods are scheduled onto AWS-managed Firecracker microVMs instead of your node groups, invisible to kubectl get nodes in the traditional sense (a virtual node appears per Pod).
  • IRSA / OIDC: EKS publishes an OIDC provider per cluster; IAM trust policies reference it; the injected AWS_WEB_IDENTITY_TOKEN_FILE lets the AWS SDK auto-assume the mapped role.
  • VPC CNI: the aws-node DaemonSet on each node manages ENI attachment and secondary-IP allocation; kubelet requests a Pod IP from the CNI plugin at Pod creation.

Code Example

#############################################
# 1. eksctl - create a cluster with a managed node group + Fargate profile
#############################################
eksctl create cluster \
  --name prod-eks \
  --region us-east-1 \
  --version 1.29 \
  --nodegroup-name general \
  --node-type m5.large \
  --nodes 3 --nodes-min 3 --nodes-max 10 \      # managed node group with autoscaling bounds
  --with-oidc                                    # provisions the OIDC provider needed for IRSA

# Add a Fargate profile for spiky/batch workloads in the "batch" namespace
eksctl create fargateprofile \
  --cluster prod-eks \
  --name batch-fp \
  --namespace batch                              # any Pod created in "batch" ns runs on Fargate, not EC2 nodes

---
# 2. IRSA - create IAM role + bind it to a ServiceAccount (eksctl shortcut)
eksctl create iamserviceaccount \
  --cluster prod-eks \
  --namespace orders \
  --name orders-sa \
  --attach-policy-arn arn:aws:iam::123456789012:policy/OrdersDynamoDBAccess \
  --approve                                       # creates IAM role + trust policy scoped to ns:orders-sa

apiVersion: v1
kind: Pod
metadata:
  name: orders-worker
  namespace: orders
spec:
  serviceAccountName: orders-sa    # Pod automatically gets AWS_WEB_IDENTITY_TOKEN_FILE + IRSA env vars injected
  containers:
    - name: worker
      image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/orders-worker:v1.4

---
# 3. VPC CNI - enable prefix delegation to avoid running out of Pod IPs
kubectl set env daemonset aws-node -n kube-system ENABLE_PREFIX_DELEGATION=true
# Assigns /28 IP prefixes per ENI instead of one IP at a time - dramatically raises
# max Pods per node without needing bigger instance types or subnets.

kubectl / CLI Commands

# Update local kubeconfig for the EKS cluster
aws eks update-kubeconfig --name prod-eks --region us-east-1

# List node groups and their scaling config
eksctl get nodegroup --cluster prod-eks

# List Fargate profiles
eksctl get fargateprofile --cluster prod-eks

# Confirm IRSA annotation on a ServiceAccount
kubectl get sa orders-sa -n orders -o yaml | grep eks.amazonaws.com/role-arn

# Check aws-node (VPC CNI) DaemonSet health across nodes
kubectl get pods -n kube-system -l k8s-app=aws-node

# See how many IPs are available/used per node (diagnose ContainerCreating IP exhaustion)
kubectl describe node  | grep -A5 Allocatable

# Trigger a managed node group version upgrade
eksctl upgrade nodegroup --cluster prod-eks --name general

Best Practices & Common Pitfalls

Best Practices

  • Use IRSA for every workload needing AWS access — never attach broad IAM to the node instance role.
  • Mix managed node groups (steady-state) with Fargate profiles (spiky/isolated workloads).
  • Size VPC subnets generously or enable prefix delegation before you hit IP exhaustion in production.
  • Use private API endpoint access (or both public+private with restricted CIDRs) for production clusters.
  • Enable EKS control-plane logging (audit, authenticator) to CloudWatch for security visibility.

Common Pitfalls

  • Granting the shared EC2 node IAM role broad permissions — every Pod on that node inherits them.
  • Undersized VPC subnets causing Pods to get stuck in ContainerCreating under load.
  • Assuming Fargate supports DaemonSets or privileged Pods — it does not.
  • Forgetting --with-oidc at cluster creation, then IRSA silently doesn't work.
  • Not monitoring ENI/IP allocation as a first-class capacity metric.

50 Interview Questions

1. What are the three EKS compute options, and when is each the right default?
Short answer: Managed node groups (AWS-provisioned ASGs of EC2, you choose instance types), self-managed node groups (full EC2/AMI control, rarely the right choice today), and Fargate (per-Pod micro-VMs, no nodes at all).
Detail: Managed node groups suit most steady-state workloads; Fargate suits spiky, less latency-sensitive, or security-isolation-sensitive workloads like CI runners or batch jobs, at higher per-unit compute cost and with no DaemonSets/privileged Pods.
Production example: A cluster runs managed node groups for the main API fleet and a Fargate profile scoped to the `batch` namespace for CI/batch jobs.
Common mistake: Defaulting to self-managed node groups today when managed node groups cover the same need with far less operational toil.
2. How does IRSA let a Pod get AWS credentials without static access keys?
Short answer: A ServiceAccount is annotated with an IAM role ARN whose trust policy trusts only that specific OIDC subject (namespace + service account name); the Pod's injected token is exchanged via `sts:AssumeRoleWithWebIdentity` for short-lived credentials.
Detail: EKS runs a per-cluster OIDC identity provider specifically to make this trust relationship possible — no shared node-level IAM role, and each workload's blast radius is limited to its own explicitly granted permissions.
Production example: `eksctl create iamserviceaccount --namespace orders --name orders-sa --attach-policy-arn ...` creates both the IAM role and its scoped trust policy in one step.
Common mistake: Forgetting `--with-oidc` at cluster creation, after which IRSA silently doesn't work because no OIDC provider exists to trust.
3. Why does IRSA limit blast radius better than granting IAM permissions to the node instance role?
Short answer: A shared node IAM role grants its permissions to every Pod running on that node; IRSA scopes permissions per-ServiceAccount, so only Pods using that specific ServiceAccount inherit those permissions.
Detail: This is the core reason IRSA is considered the correct default — without it, any compromised Pod on a node effectively has all of that node's IAM permissions, regardless of what that particular Pod actually needs.
Production example: An `orders-sa` ServiceAccount scoped to only DynamoDB access on the orders table means a compromised orders Pod can't touch unrelated S3 buckets the node role might otherwise have access to.
Common mistake: Granting broad permissions to the shared EC2 node IAM role "to make things work," giving every Pod on that node those same permissions.
4. How does the VPC CNI's real-IP model differ from an overlay network, and what's the tradeoff?
Short answer: The VPC CNI assigns each Pod a real, routable VPC IP directly from the node's ENI secondary IP pool instead of using an overlay — Pods are reachable by other AWS resources without NAT, but VPC subnet IP space must be sized generously.
Detail: Security groups can also apply directly to Pods via SecurityGroupPolicy, and there's no encapsulation overhead — but every Pod consumes a real VPC IP, so small subnets run out of IPs far faster than people expect at scale.
Production example: A subnet sized for the initial cluster size runs out of free IPs as the cluster scales, and Pods get stuck in `ContainerCreating` — a classic EKS production incident.
Common mistake: Sizing VPC subnets based on expected node count rather than expected total Pod count, since Pods (not nodes) are what consume the IP space.
5. What does enabling prefix delegation on the VPC CNI actually change?
Short answer: It assigns `/28` IP prefixes per ENI instead of allocating one secondary IP at a time, dramatically raising the max Pods per node without needing bigger instance types or larger subnets.
Detail: Without prefix delegation, Pod density per node is capped by the number of secondary IPs an ENI can hold individually; prefix delegation allocates blocks of IPs at once, relaxing that per-node ceiling.
Production example: `kubectl set env daemonset aws-node -n kube-system ENABLE_PREFIX_DELEGATION=true` raises Pod density on existing instance types without a fleet-wide resize.
Common mistake: Responding to IP exhaustion only by resizing the VPC subnet, when enabling prefix delegation or custom networking may solve it without a disruptive re-IP-plan.
6. Why does Fargate remove nodes entirely, and what capability does that cost you?
Short answer: Each Pod gets its own isolated micro-VM sized to its resource requests, billed per-vCPU/memory-second with zero node patching or capacity planning — but you lose DaemonSets and privileged Pods.
Detail: Because there's no persistent, shared node underneath a Fargate Pod, anything requiring host-level access (a DaemonSet, hostPath, privileged mode) has no place to run — this is a fundamental architectural tradeoff, not a configurable limit.
Production example: A Fargate profile scoped to the `batch` namespace runs CI jobs with zero node-capacity planning, but a log-collector DaemonSet still has to run on managed node groups instead.
Common mistake: Assuming Fargate supports DaemonSets or privileged Pods for some workloads "if configured right" — it categorically does not.
7. How do managed node groups handle AMI upgrades safely?
Short answer: AWS handles rolling AMI upgrades that respect PodDisruptionBudgets when you trigger a node-group update, rather than replacing all nodes at once.
Detail: This is one of the concrete operational benefits over self-managed node groups — you still choose instance types and trigger the upgrade, but the rollout mechanics (cordon/drain respecting PDBs) are handled for you.
Production example: `eksctl upgrade nodegroup --cluster prod-eks --name general` triggers a managed, PDB-aware rolling replacement of the group's nodes.
Common mistake: Assuming self-managed node groups are needed for "more control" over upgrade timing, when managed node groups already expose this via a single command with safer defaults.
8. What virtual node abstraction does Fargate present to kubectl, and why does that matter operationally?
Short answer: Matching Fargate Pods are scheduled onto AWS-managed Firecracker microVMs, invisible in the traditional node-fleet sense — a virtual node appears per Pod in `kubectl get nodes`.
Detail: This means node-count-based capacity reasoning (like `kubectl get nodes` to gauge total cluster capacity) doesn't apply the same way to Fargate workloads, since there's no shared, persistent node backing multiple Pods.
Production example: A cluster running mostly Fargate workloads shows an unusually large, constantly-changing node list, one "node" per running Fargate Pod, rather than a small stable fleet.
Common mistake: Trying to reason about Fargate capacity using traditional node-count or node-utilization dashboards designed for EC2-backed node groups.
9. Why is IRSA described as "functionally identical in purpose" to GKE Workload Identity?
Short answer: Both solve "how does a Pod get cloud credentials without static keys" via OIDC federation between a Kubernetes ServiceAccount and a cloud IAM identity — just implemented through different mechanisms (AWS STS vs GCP's metadata-server federation).
Detail: This is a common interview framing: the underlying problem and security properties (short-lived, auditable, revocable, no static keys) are the same across EKS, GKE, and AKS — only the plumbing differs.
Production example: A platform team documenting "workload identity" policy across a multi-cloud fleet writes one conceptual policy and maps it to IRSA on EKS, Workload Identity on GKE, and Azure AD Workload Identity on AKS.
Common mistake: Treating IRSA as an EKS-specific one-off concept instead of recognizing it as one implementation of a cross-cloud pattern.
10. What are the practical default recommendations this topic gives for a new production EKS cluster?
Short answer: Managed node groups for steady-state workloads, a Fargate profile for spiky/untrusted/low-effort workloads, IRSA everywhere instead of node-instance-profile IAM, and generously sized VPC subnets (or prefix delegation) before hitting production scale.
Detail: These defaults directly map to the topic's core risks: node-role over-permissioning (solved by IRSA), IP exhaustion (solved by subnet sizing/prefix delegation), and unnecessary operational toil (solved by managed node groups/Fargate over self-managed).
Production example: A new cluster is bootstrapped with `--with-oidc`, a managed node group, a Fargate profile for the `batch` namespace, and IRSA bindings for every workload needing AWS access from day one.
Common mistake: Treating these as optional hardening steps to add later, rather than defaults to establish at cluster creation before workloads onboard.
