Scheduler

How kube-scheduler decides which node a Pod runs on: filtering, scoring, profiles, and preemption.
← etcd Course Index Controller Manager →

Explanation

The Scheduler (kube-scheduler) has exactly one job: watch for Pods with an empty spec.nodeName and decide which node they should run on. It does not create Pods, does not start containers, and does not talk to the container runtime — it only writes a binding decision back through the API Server. Everything downstream (kubelet noticing the assignment, the runtime starting the container) is someone else's responsibility.

Kubernetes needs a dedicated scheduling component because placement is a genuinely hard constraint-satisfaction problem at scale: a Pod might require specific CPU/memory, GPU hardware, co-location with other Pods (affinity), avoidance of other Pods (anti-affinity), a specific zone, or exclusion from tainted nodes. Doing this well — and fast, for clusters with thousands of nodes and pending Pods — justifies a purpose-built component rather than folding placement logic into the API Server or kubelet.

Scheduling happens in two phases. Filtering (formerly "predicates") eliminates every node that cannot possibly run the Pod — insufficient CPU/memory, a taint the Pod doesn't tolerate, a node selector mismatch, port conflicts. Scoring (formerly "priorities") ranks the surviving nodes — spreading Pods across zones, preferring nodes with more free resources (or fewer, for bin-packing), respecting soft affinity/anti-affinity preferences — and the highest-scoring node wins. Both phases are built from a pluggable set of "plugins," configured via scheduling profiles, so you can run multiple scheduler configurations (e.g. a bin-packing profile for batch jobs, a spread profile for stateful services) side by side.

When no node passes filtering for a high-priority Pod, the scheduler can invoke preemption: it finds one or more lower-priority Pods on some node whose eviction would free enough room, evicts them, and schedules the higher-priority Pod in their place. This is why PriorityClass matters in multi-tenant clusters — without it, critical workloads have no mechanism to displace non-critical ones under resource pressure.

Like the Controller Manager, the Scheduler uses leader election when run with multiple replicas for HA — only one instance is ever actively scheduling at a time, avoiding the possibility of two schedulers racing to bind the same Pod to different nodes.

Internal Working / Interactions

  • Scheduler ↔ API Server: watches unscheduled Pods (empty nodeName) and the full Node/Pod state needed for filtering/scoring, via informer caches.
  • Scheduler → API Server (bind): writes the decision as a Binding subresource, which the API Server persists as spec.nodeName in etcd.
  • Scheduler ↔ kubelet: no direct interaction — kubelet independently notices the binding via its own watch on the API Server.
  • Scheduler ↔ PriorityClass/PodDisruptionBudget: reads Pod priority for preemption decisions and respects PDBs when evicting during preemption.
  • Extender/webhook scheduling: can call out to external scheduler extenders for custom constraints not expressible via built-in plugins.

Code Example

# KubeSchedulerConfiguration — defines a custom scheduling profile,
# e.g. a bin-packing profile used only for batch workloads.
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
leaderElection:
  leaderElect: true                  # only one scheduler replica active at a time
profiles:
  - schedulerName: default-scheduler
  - schedulerName: batch-binpack       # Pods opt in via spec.schedulerName
    pluginConfig:
      - name: NodeResourcesFit
        args:
          scoringStrategy:
            type: MostAllocated       # prefer already-busy nodes (bin-packing, saves cost)
    plugins:
      score:
        enabled:
          - name: NodeResourcesFit
            weight: 5
---
# PriorityClass + a Pod that uses it to enable preemption over lower-priority Pods
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority-payments
value: 1000000                        # higher value = more important, can preempt lower values
globalDefault: false
preemptionPolicy: PreemptLowerPriority
---
apiVersion: v1
kind: Pod
metadata:
  name: checkout-critical
spec:
  priorityClassName: high-priority-payments   # scheduler will evict lower-priority Pods if needed
  schedulerName: batch-binpack                 # opt into the custom profile above
  containers:
    - name: checkout
      image: registry.internal/checkout:2.4.1

kubectl get events --field-selector reason=Scheduled -A

Best Practices & Common Pitfalls

Best Practices

  • Always set resource requests — the scheduler cannot reason about capacity otherwise.
  • Use PriorityClasses deliberately in multi-tenant clusters so critical workloads can preempt.
  • Use topology spread constraints for real HA across zones, not just replica count.
  • Tune percentageOfNodesToScore in very large clusters to bound scheduling latency.

Pitfalls

  • Leaving requests unset, causing the scheduler to over-pack nodes that later hit real resource pressure.
  • Assuming the scheduler starts containers — it only binds; kubelet does the rest.
  • Setting overly aggressive preemption without PodDisruptionBudgets, causing unexpected outages.
  • Running multiple scheduler replicas expecting parallel throughput — only one is ever active.

50 Interview Questions

1. What exactly does the Scheduler do, and what does it explicitly NOT do?
Short answer: It watches for Pods with an empty spec.nodeName and decides which node they run on — it never creates Pods or starts containers.
Detailed explanation: The Scheduler's entire output is a single write: the Binding subresource that sets nodeName. Everything downstream (kubelet noticing the binding, pulling images, starting containers) is entirely someone else's responsibility.
Production example: kubectl describe pod shows a Scheduled event emitted by the Scheduler, followed by separate Pulling/Started events emitted by kubelet.
Common mistake: Believing the Scheduler starts the container — it only ever binds the Pod to a node.
2. Explain the two-phase filtering and scoring process.
Short answer: Filtering eliminates nodes that can't possibly run the Pod; scoring ranks the survivors to pick the best fit.
Detailed explanation: Filtering (formerly "predicates") checks hard constraints — insufficient CPU/memory, untolerated taints, node selector mismatches, port conflicts. Scoring (formerly "priorities") ranks the feasible nodes using weighted plugins (spreading across zones, resource balance, image locality, affinity preferences) and the highest-scoring node wins.
Production example: A Pod requiring a GPU is filtered down to only GPU nodes first, then scored among those based on remaining GPU/CPU headroom.
Common mistake: Conflating hard requirements (should be filters, e.g. required node affinity) with soft preferences (should be scoring plugins, e.g. preferred affinity) when designing placement rules.
3. Why can't the Scheduler reason about a Pod's resource needs if requests aren't set?
Short answer: Without resources.requests, the Scheduler has no numeric basis to filter/score nodes for capacity.
Detailed explanation: The filtering phase's NodeResourcesFit plugin compares a Pod's requests against a node's allocatable capacity; if requests are unset, the Scheduler effectively treats the Pod as needing ~0 resources, which can lead to overpacking a node that later hits real resource pressure.
Production example: A team omits requests on a memory-heavy batch job; the Scheduler happily stacks a dozen of them onto one node, which then OOMs under real load.
Common mistake: Assuming limits alone are enough — the Scheduler only reads requests for placement decisions, not limits.
4. What is preemption and when does it trigger?
Short answer: When no node passes filtering for a high-priority pending Pod, the Scheduler can evict lower-priority Pods elsewhere to make room.
Detailed explanation: The Scheduler identifies a node where evicting one or more lower-priority Pods (per PriorityClass) would free enough capacity, evicts them, and schedules the higher-priority Pod in their place — but it respects PodDisruptionBudgets during this process where possible.
Production example: A high-priority-payments PriorityClass Pod preempts lower-priority batch Pods on a resource-constrained node during a traffic spike.
Common mistake: Enabling aggressive preemption without setting PodDisruptionBudgets on the Pods that could be preempted, causing unexpected outages.
5. What are scheduling profiles and why would you run more than one?
Short answer: Profiles are named, independently configured sets of scheduling plugins that Pods opt into via spec.schedulerName.
Detailed explanation: You might run a bin-packing profile (MostAllocated scoring strategy) for batch/cost-sensitive workloads to consolidate nodes for autoscaler efficiency, alongside a spread profile (default) for stateful/critical services needing maximum HA.
Production example: A batch-binpack profile with scoringStrategy: MostAllocated intentionally packs jobs tightly to enable cluster-autoscaler scale-down.
Common mistake: Forgetting a Pod must explicitly set schedulerName to use anything other than default-scheduler — profiles aren't applied automatically.
6. Why does the Scheduler use leader election even though it's not doing distributed consensus itself?
Short answer: To guarantee exactly one active instance so two replicas never race to bind the same Pod to different nodes.
Detailed explanation: Multiple Scheduler replicas run for HA/failover speed, but only the lease holder actively schedules; standbys wait to take over if the leader fails, avoiding conflicting binding decisions.
Production example: kubectl get lease -n kube-system kube-scheduler -o yaml shows exactly which replica currently holds the lease.
Common mistake: Expecting more Scheduler replicas to parallelize scheduling throughput — they never do.
7. How do topology spread constraints differ from just increasing replica count?
Short answer: Topology spread constraints explicitly instruct the Scheduler to distribute Pods across failure domains (zones/nodes), while replica count alone says nothing about placement.
Detailed explanation: Without spread constraints, the Scheduler could legally place all replicas of a Deployment in the same zone if that zone scores highest, defeating the purpose of having multiple replicas for resilience; maxSkew and whenUnsatisfiable: DoNotSchedule enforce real distribution.
Production example: A payments Deployment with 6 replicas and a zone-based topology spread constraint (maxSkew: 1) guarantees no more than a 1-Pod imbalance between zones.
Common mistake: Assuming "6 replicas" automatically means HA across zones without an explicit spread constraint.
8. What's a scheduler extender and when would you use one?
Short answer: An external service the Scheduler calls out to for custom filtering/scoring logic not expressible via built-in plugins.
Detailed explanation: Extenders let organizations bolt on proprietary constraints (e.g. license availability, custom hardware topology) without forking or rebuilding the scheduler binary, at the cost of added latency per scheduling decision.
Production example: A company with specialized hardware licensing uses an extender to filter out nodes lacking a valid license token before the Scheduler considers them.
Common mistake: Reaching for an extender when a native scheduling plugin or affinity rule would solve the same problem with less operational overhead.
9. Why does percentageOfNodesToScore matter in very large clusters?
Short answer: It bounds scheduling latency by not scoring every single node when the cluster is huge.
Detailed explanation: In a cluster with thousands of nodes, scoring all of them for every pending Pod would be slow; this setting lets the Scheduler stop after evaluating a representative percentage of feasible nodes, trading a small amount of placement optimality for much faster scheduling decisions.
Production example: A 3,000-node cluster tunes this down from 100% to keep scheduling throughput acceptable during rapid autoscaling events.
Common mistake: Leaving it at default in an extremely large cluster and then being surprised scheduling latency climbs as node count grows.
10–50. Additional rapid-fire questions
10. What field does the Scheduler look for to find unscheduled Pods? — An empty spec.nodeName.
11. What subresource does the Scheduler write to bind a Pod? — The Binding subresource, which the API Server persists as spec.nodeName.
12. What were filtering and scoring plugins formerly called? — Predicates and priorities, respectively.
13. What plugin checks whether a node has enough CPU/memory for a Pod? — NodeResourcesFit.
14. What's a taint, and how does it interact with scheduling? — A repellent applied to a node that only Pods with a matching toleration can be scheduled onto.
15. What object lets a Pod preempt lower-priority Pods? — PriorityClass, referenced via spec.priorityClassName.
16. How does a Pod opt into a non-default scheduling profile? — By setting spec.schedulerName to match a configured profile.
17. What's the risk of setting preemptionPolicy: Never on a high-priority Pod? — It waits in the normal queue like any other Pod instead of being able to evict lower-priority Pods for room.
18. Why does the Scheduler have no direct interaction with kubelet? — kubelet independently discovers Pods bound to its node via its own watch on the API Server; the Scheduler never talks to kubelet directly.
19. What's ImageLocality scoring? — A plugin that favors nodes which already have the Pod's image cached, reducing pull time.
20. Why might a Pod stay Pending with "0/5 nodes are available: insufficient cpu"? — Every node failed the filtering phase due to a hard CPU resource constraint.
21. What's the difference between required and preferred node affinity? — Required affinity acts as a hard filter (Pod won't schedule if unmet); preferred affinity only influences scoring.
22. How would you check which scheduler profile a specific Pod used? — kubectl get pod my-pod -o jsonpath='{.spec.schedulerName}'.
23. What's the effect of running two Scheduler replicas without leader election disabled accidentally? — This shouldn't normally be possible with correct config, but if both became active it would risk race conditions binding the same Pod to conflicting nodes.
24. Why is PodAffinity/PodAntiAffinity evaluated during both filtering and scoring? — Required affinity/anti-affinity acts as a hard filter; preferred forms only influence the score.
25. What's a MostAllocated scoring strategy used for? — Bin-packing — preferring already-busy nodes to consolidate workloads and free up whole nodes for autoscaler removal.
26. What's a LeastAllocated scoring strategy used for? — Spreading Pods across nodes to keep headroom evenly available, the opposite goal of bin-packing.
27. How does the Scheduler discover Node capacity/allocatable info? — From cached Node objects populated via its own watch/informer against the API Server.
28. Why does preemption respect PodDisruptionBudgets where possible? — To avoid violating an application's minimum-availability guarantee even while making room for a higher-priority Pod.
29. What's the operational risk of disabling all preemption in a multi-tenant cluster? — Critical workloads have no mechanism to displace non-critical ones under resource pressure, risking outages during contention.
30. Why is "Scheduled" one of the very first events in a Pod's lifecycle? — It marks the moment the Scheduler's decision is persisted, before kubelet has done any actual work.
31. What happens if a Pod's node affinity requires a zone that has zero nodes? — The Pod remains Pending indefinitely — filtering finds zero feasible nodes.
32. How would you debug why the Scheduler picked a seemingly suboptimal node? — Check the scoring plugin weights in the scheduler profile and compare candidate node resource states at decision time.
33. Why does the Scheduler need to be aware of PodDisruptionBudgets during preemption specifically, not during normal scheduling? — Normal scheduling never removes existing Pods; only preemption does, so PDBs are only relevant there.
34. What's the risk of using scheduler extenders heavily? — Added per-scheduling-decision latency and an external dependency the Scheduler now relies on for every decision.
35. Why is globalDefault: false typically set on custom PriorityClasses? — To avoid silently making every Pod cluster-wide inherit an unintended elevated priority.
36. What's the difference between a Pod being Pending due to scheduling vs. due to image pulling? — Scheduling-phase Pending has no nodeName yet; image-pull Pending is already bound to a node and waiting on kubelet/runtime.
37. Why do batch workloads commonly use a separate bin-packing scheduler profile? — To maximize node consolidation for cost savings, since batch jobs are typically more tolerant of being packed tightly than latency-sensitive services.
38. What's the effect of setting an extremely high PriorityClass value on every workload? — It defeats the purpose of prioritization — if everything is "high priority," preemption can't meaningfully distinguish importance.
39. Why does port conflict count as a filtering-phase failure, not a scoring one? — It's a hard constraint (the Pod literally cannot run there), not a preference to be weighed.
40. How does InterPodAffinity scoring differ from required PodAffinity? — Scoring only nudges preference toward co-location; required affinity makes it a hard scheduling constraint.
41. What's a real production reason to lower percentageOfNodesToScore? — To keep scheduling throughput acceptable during rapid scale-out events in very large clusters.
42. Why is understanding owner references relevant when debugging scheduling issues? — A Pod's controller (ReplicaSet/Deployment) will recreate it if deleted, so scheduling problems often need fixing at the template level, not the individual Pod.
43. What's the risk of manually setting spec.nodeName on a Pod yourself? — It bypasses the Scheduler's filtering/scoring entirely, potentially placing the Pod on a node that can't actually satisfy its resource needs.
44. Why might increasing scheduler scoring plugin weights change placement behavior unexpectedly? — Weights determine relative influence between plugins; increasing one can dominate the total score and override other considerations.
45. What's the purpose of the kubectl get events --field-selector reason=Scheduled command? — Quickly audits all recent scheduling decisions cluster-wide for troubleshooting or capacity review.
46. Why doesn't the Scheduler retry a failed binding forever on the exact same node? — It re-evaluates filtering/scoring fresh on each scheduling attempt against current cluster state, which may have changed.
47. What's a symptom that the Scheduler itself (not resource constraints) is the problem? — All Pods across the cluster stay Pending with no nodeName, regardless of available capacity — check the Scheduler pod/lease first.
48. Why is affinity/anti-affinity evaluation more expensive than simple resource filtering? — It requires inspecting labels of other Pods across the cluster/topology domain, not just a single node's static capacity.
49. What's a good way to validate a new scheduling profile before rolling it out broadly? — Test it on a non-critical workload with schedulerName set, comparing actual placement against expectations before switching production Pods.
50. Why is "Pods stuck Pending cluster-wide" almost always a Scheduler-health issue rather than a resource issue? — True resource exhaustion normally produces per-Pod filtering-failure events; a total, cluster-wide freeze instead points to the Scheduler process/leader-election itself being down.
