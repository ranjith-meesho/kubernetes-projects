Pods

The smallest deployable unit: spec anatomy, multi-container patterns (sidecar/init), and the pod lifecycle phases.
← Container Runtime Course Index ReplicaSets →

Explanation

A Pod is the smallest deployable unit in Kubernetes — never a single container. A Pod is a group of one or more containers that share a network namespace (same IP, same localhost) and can share storage volumes, scheduled together onto the same node as an atomic unit. You never schedule a bare container in Kubernetes; the Pod is the abstraction the Scheduler places, kubelet runs, and Services route to.

Kubernetes needs this grouping because real applications often need tightly-coupled helper processes co-located with the main process — a log shipper, a proxy, a config-reloader — and those helpers need to share the same network identity and sometimes a filesystem with the main container, without the complexity of manually wiring up shared namespaces yourself. The Pod is Kubernetes' answer to "what is the unit of co-location and shared fate."

Every Pod actually starts with a hidden, first container: the pause container (sandbox), created by the runtime to hold the shared Linux namespaces (network, IPC, sometimes PID) before any of your containers start. All your containers then join those same namespaces — this is the literal mechanism behind "containers in a Pod share localhost."

Pods support two important multi-container patterns. Init containers run sequentially to completion before any main container starts — used for setup tasks like waiting on a dependency or seeding config, and if one fails, the Pod restarts from the first init container (per the Pod's restart policy). Sidecar containers (including native sidecars since 1.28, marked via restartPolicy: Always inside initContainers) run alongside the main container for the Pod's entire lifetime — a service mesh proxy (Envoy/Istio) is the canonical production example, injected automatically via a mutating admission webhook.

A Pod moves through defined lifecycle phases: Pending (accepted but not yet fully running — includes scheduling and image pull time), Running (bound to a node, at least one container running or starting), Succeeded/Failed (terminal, all containers exited — relevant for Jobs), and Unknown (kubelet lost contact with the node). Pods are also fundamentally ephemeral and disposable by design — they are never rescheduled onto a different node once bound; if a Pod dies, a controller (like ReplicaSet) creates an entirely new Pod object with a new identity, rather than the old one being "moved."

Internal Working / Interactions

  • Pod ↔ Scheduler: Pods with empty nodeName are the Scheduler's entire input; it writes the binding once.
  • Pod ↔ kubelet: kubelet on the bound node drives sandbox creation, init container sequencing, and main/sidecar container startup via the CRI.
  • Pod ↔ ReplicaSet/Deployment/Job: most Pods are owned by a higher-level controller via ownerReferences — direct "naked" Pods have no self-healing.
  • Pod ↔ Service/EndpointSlice: a Pod's IP is added to a Service's EndpointSlice only once it passes its readiness probe.
  • Pod ↔ Volumes: volumes are defined at the Pod level and mounted into one or more containers, enabling shared filesystem state (e.g. sidecar reading logs the main container writes).

Code Example

# A realistic multi-container Pod: init container + main container + sidecar,
# demonstrating shared network/volume and lifecycle ordering.
apiVersion: v1
kind: Pod
metadata:
  name: checkout-with-sidecar
  labels:
    app: checkout
spec:
  initContainers:
    - name: wait-for-db                # runs to completion BEFORE main containers start
      image: busybox:1.36
      command: ["sh", "-c", "until nc -z postgres 5432; do sleep 2; done"]
  containers:
    - name: checkout                    # main application container
      image: registry.internal/checkout:2.4.1
      ports:
        - containerPort: 8080
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/app
      readinessProbe:
        httpGet: { path: /healthz, port: 8080 }
    - name: log-shipper                  # sidecar: runs for the Pod's entire lifetime
      image: fluent-bit:2.2
      volumeMounts:
        - name: shared-logs              # shares the SAME volume to read what checkout writes
          mountPath: /var/log/app
          readOnly: true
  volumes:
    - name: shared-logs
      emptyDir: {}                       # ephemeral, Pod-lifetime-scoped shared storage

kubectl Commands

# Inspect Pod phase and per-container status
kubectl get pod checkout-with-sidecar -o wide
kubectl get pod checkout-with-sidecar -o jsonpath='{.status.phase}'

# See init container progress explicitly
kubectl describe pod checkout-with-sidecar | grep -A10 "Init Containers"

# View logs from a specific container within a multi-container Pod
kubectl logs checkout-with-sidecar -c checkout
kubectl logs checkout-with-sidecar -c log-shipper

# Exec into one specific container
kubectl exec -it checkout-with-sidecar -c checkout -- /bin/sh

# Watch the full lifecycle transition live
kubectl get pod checkout-with-sidecar -w

# See owner references (which controller, if any, owns this Pod)
kubectl get pod checkout-with-sidecar -o jsonpath='{.metadata.ownerReferences}'

Best Practices & Common Pitfalls

Best Practices

  • Never deploy bare/naked Pods in production — always through a Deployment/StatefulSet/Job for self-healing.
  • Use init containers for genuine sequential setup, not as a substitute for readiness probes.
  • Set resource requests/limits on every container in the Pod, including sidecars.
  • Prefer native sidecar containers (1.28+) over ad-hoc background processes in the main container.

Pitfalls

  • Forgetting a sidecar's resource usage counts toward the Pod's total scheduling footprint.
  • Assuming a "restarted" Pod kept its IP or identity — a new Pod object may be created entirely by its controller.
  • Using emptyDir for data that must survive a Pod restart/reschedule (it's ephemeral, node-local).
  • Not setting terminationGracePeriodSeconds appropriately for containers needing clean shutdown.

50 Interview Questions

1. Why is a Pod, not a container, the smallest deployable unit in Kubernetes?
Short answer: A Pod groups one or more containers that share a network namespace and can share storage, scheduled together as an atomic unit — you never schedule a bare container directly.
Detail: Real applications often need tightly-coupled helper processes (log shippers, proxies, config-reloaders) co-located with the main process, sharing the same network identity and sometimes a filesystem, without manually wiring shared namespaces yourself.
Production example: A checkout service container and its Fluent Bit log-shipper sidecar run in the same Pod, sharing localhost and a volume, scheduled together on the same node.
Common mistake: Treating "Pod" and "container" as interchangeable terms — a Pod is the scheduling/networking abstraction; it may contain multiple containers.
2. What is the pause container/sandbox, and why does every Pod start with one?
Short answer: A hidden first container created by the runtime to hold the shared Linux namespaces (network, IPC, sometimes PID) before any of your actual containers start.
Detail: All your containers then join those same namespaces — this is the literal mechanism behind "containers in a Pod share localhost," since the pause container's namespaces persist even if application containers restart.
Production example: If the main container in a Pod crashes and restarts, it rejoins the same network namespace held open by the pause container, so the Pod IP doesn't change.
Common mistake: Being confused by an unfamiliar "pause" container showing up in crictl ps -a output and assuming it's an application bug.
3. What is the difference between init containers and sidecar containers?
Short answer: Init containers run sequentially to completion before any main container starts; sidecar containers run alongside the main container for the Pod's entire lifetime.
Detail: Init containers are for setup tasks (waiting on a dependency, seeding config) and if one fails, the Pod restarts from the first init container per its restart policy. Sidecars, including native sidecars since 1.28 (marked via restartPolicy: Always inside initContainers), coexist with the main container throughout.
Production example: A wait-for-db init container blocks until Postgres is reachable, then a Fluent Bit sidecar runs continuously alongside the main checkout container.
Common mistake: Using an init container as a substitute for a readiness probe — init containers gate startup ordering, not ongoing health.
4. What is a native sidecar (1.28+) and how does it differ from an old-style "sidecar" container?
Short answer: A native sidecar is declared inside initContainers but marked with restartPolicy: Always, so it starts before the main container but keeps running for the Pod's entire life instead of exiting.
Detail: Before this feature, "sidecars" were just regular containers in the containers list with no special startup ordering guarantee — native sidecars solve the problem of ensuring a proxy/mesh sidecar is ready before the main container starts, and stays running until the Pod terminates.
Production example: A service mesh proxy (Envoy/Istio) injected as a native sidecar guarantees traffic interception is ready before the main application container begins accepting connections.
Common mistake: Assuming any container alongside the main one is automatically a "sidecar" in the native sense — without the native sidecar restart policy, there's no startup-ordering guarantee.
5. List the Pod lifecycle phases and what each one means.
Short answer: Pending (accepted but not fully running, includes scheduling/image pull), Running (bound to a node with at least one container running/starting), Succeeded/Failed (terminal, all containers exited), Unknown (kubelet lost contact with the node).
Detail: Succeeded/Failed are primarily relevant for Job-style Pods that are expected to terminate; a long-running Deployment Pod normally lives in Running until deleted.
Production example: A batch Job Pod transitions Pending → Running → Succeeded once its container exits 0, and the Job controller considers that completion satisfied.
Common mistake: Treating Pending as always meaning "scheduling problem" — it also legitimately covers image pull time, which can dominate for large images.
6. Why are Pods described as "ephemeral and disposable," and what does that mean operationally?
Short answer: Pods are never rescheduled onto a different node once bound — if a Pod dies, a controller creates an entirely new Pod object with a new identity rather than "moving" the old one.
Detail: This means Pod IPs, names (unless a controller reuses a naming pattern), and any node-local state (like emptyDir) are not preserved across a Pod's death and replacement.
Production example: When a node fails, the ReplicaSet controller creates a brand-new Pod object on a different node — it is not the "same" Pod resurrected, it's a new one with a new IP.
Common mistake: Assuming a Pod that gets recreated after a failure retains its original IP or identity — Services rely on EndpointSlice updates precisely because this isn't true.
7. Why should you avoid deploying "naked" Pods directly in production?
Short answer: A Pod with no owning controller has no self-healing — if it dies, nothing recreates it.
Detail: Most Pods should be owned by a higher-level controller (ReplicaSet, Deployment, Job, StatefulSet) via ownerReferences, which is what actually watches for Pod death and creates replacements.
Production example: A manually created Pod (kubectl run without a controller) that gets OOMKilled simply stays dead — no ReplicaSet exists to notice and replace it.
Common mistake: Using bare Pods for anything beyond quick, disposable debugging/testing purposes.
8. How does a Pod's IP become eligible to receive Service traffic?
Short answer: A Pod's IP is only added to a Service's EndpointSlice once it passes its readiness probe.
Detail: Being in the Running phase is not sufficient — the Pod must also report Ready via its readiness probe before kube-proxy's watched EndpointSlice includes it as a valid routing target.
Production example: A Pod that's Running but still initializing a cache on startup won't receive Service traffic until its readiness probe explicitly passes.
Common mistake: Assuming Running phase alone means a Pod is receiving traffic — readiness is a separate, additional gate.
9. How do containers within the same Pod share storage, and what's a common pitfall with emptyDir?
Short answer: Volumes are defined at the Pod level and mounted into one or more containers; emptyDir is a common choice but is ephemeral and node-local.
Detail: This enables patterns like a sidecar reading logs the main container writes to a shared volume, but emptyDir data does not survive a Pod restart/reschedule — it's tied to that specific Pod instance on that specific node.
Production example: The checkout container writes logs to a shared emptyDir volume mounted read-only by the log-shipper sidecar, which forwards them elsewhere before the Pod terminates.
Common mistake: Using emptyDir for data that must survive a Pod restart or reschedule — it's explicitly ephemeral, not a persistence mechanism.
10. What happens during Pod termination, and why does terminationGracePeriodSeconds matter?
Short answer: Containers receive a termination signal and get a grace period to shut down cleanly before being forcibly killed; terminationGracePeriodSeconds controls how long that window is.
Detail: Containers needing to drain in-flight connections, flush buffers, or close database connections cleanly require enough grace period to actually finish that work before the kernel sends a forceful kill.
Production example: A checkout container handling long-lived requests sets a longer terminationGracePeriodSeconds to allow in-flight checkouts to complete before shutdown.
Common mistake: Leaving the default grace period for containers that need meaningful clean-shutdown time, causing abrupt connection drops during rolling updates.
11–50. Additional rapid-fire questions
11. Do containers in the same Pod share the same IP address? — Yes, because they all join the same network namespace held by the pause/sandbox container.
12. Can containers in a Pod communicate over localhost? — Yes, since they share one network namespace, localhost works between them directly.
13. What creates the pause container? — The container runtime, during CRI sandbox creation (RunPodSandbox).
14. What happens if an init container fails? — The Pod restarts from the first init container, subject to the Pod's restart policy.
15. Do init containers run in parallel or sequentially? — Sequentially, one completes before the next starts.
16. Can a Pod have multiple init containers? — Yes, they run one after another in listed order.
17. What restartPolicy value marks a native sidecar inside initContainers? — restartPolicy: Always.
18. What's the canonical production example of a sidecar container? — A service mesh proxy like Envoy or Istio's sidecar.
19. How is a service mesh sidecar typically injected? — Automatically, via a mutating admission webhook.
20. What Pod phase covers image pull time? — Pending.
21. Which phases are terminal (Pod won't transition further)? — Succeeded and Failed.
22. What causes phase Unknown? — kubelet losing contact with the node the Pod is bound to.
23. Are Pod phases the same as container-level restart counts? — No, phase reflects the overall Pod, while individual containers can restart independently within the Running phase.
24. Once a Pod is bound to a node, can the Scheduler move it to another node? — No, binding is one-time; a new Pod object is required to run elsewhere.
25. What field indicates which controller owns a Pod? — metadata.ownerReferences.
26. What happens to a naked Pod (no owner) if its node fails? — It is simply gone — nothing recreates it.
27. Does deleting a ReplicaSet-owned Pod manually cause it to come back? — Yes, the ReplicaSet controller notices the missing replica and creates a new Pod.
28. What kubectl flag views logs from one specific container in a multi-container Pod? — -c <container-name>, e.g. kubectl logs pod -c checkout.
29. What kubectl flag execs into a specific container? — -c <container-name> with kubectl exec.
30. How would you watch a Pod's lifecycle transitions live? — kubectl get pod <name> -w.
31. What field shows a Pod's current phase directly? — status.phase, viewable via kubectl get pod -o jsonpath='{.status.phase}'.
32. Do sidecar containers count toward the Pod's total scheduling resource footprint? — Yes, their resource requests/limits are added to the Pod's overall footprint the Scheduler considers.
33. What's a risk of not setting resource requests/limits on sidecar containers? — Sidecars can silently consume disproportionate node resources, affecting the main container and other Pods on the node.
34. Can two containers in the same Pod use different images/versions? — Yes, each container has its own image, independently versioned.
35. Is a Pod's storage volume automatically shared with every container? — No, each container must explicitly declare a volumeMount for a given Pod-level volume.
36. What Pod-level construct enables a sidecar to read logs the main container writes? — A shared volume (e.g. emptyDir) mounted into both containers.
37. Does an emptyDir volume survive a Pod restart on the same node? — No, it is tied to that specific Pod instance and is deleted when the Pod is removed.
38. What happens to init container resources after the Pod reaches Running? — Init containers have already exited; their resources are no longer consumed once the main containers start.
39. Why is "restartPolicy" a Pod-level (not container-level) field for regular containers? — Because it governs how the whole Pod's containers are restarted on exit, applied uniformly per Pod semantics (Always/OnFailure/Never).
40. What Kubernetes object is the Scheduler's entire input regarding Pods? — Pods with an empty nodeName field, i.e. unscheduled Pods.
41. Does the Pod spec or the controller spec define replica count? — The controller (ReplicaSet/Deployment) defines replica count; the Pod spec only describes a single Pod template.
42. What's a key difference between Job-owned Pods and Deployment-owned Pods regarding phase? — Job Pods are expected to reach Succeeded/Failed as a terminal outcome; Deployment Pods are expected to stay Running indefinitely.
43. Why must a sidecar reading shared logs mount its volume readOnly in good practice? — To avoid the sidecar accidentally corrupting or interfering with data the main container is actively writing.
44. What happens if you don't declare a readiness probe at all? — The container is considered Ready as soon as it starts (default), which may add it to Service routing before it's actually able to serve traffic.
45. Does deleting a Pod object guarantee the underlying container process stops immediately? — No, there's a grace period (terminationGracePeriodSeconds) before a forceful kill.
46. Can a sidecar container observe the main container's process directly (PID namespace)? — Only if the Pod explicitly enables a shared PID namespace (shareProcessNamespace: true); by default PID namespaces aren't shared.
47. What's the significance of "one network namespace" for port conflicts within a Pod? — Containers in the same Pod cannot bind the same port simultaneously since they share one network namespace/IP.
48. Why is a Pod described as the "unit of co-location and shared fate"? — Because all its containers are scheduled together, live/die together as one unit, and share networking/storage — not independently placed or recovered.
49. What's the practical effect of a sidecar crashing repeatedly in a Pod? — It can trigger restart-policy-driven container restarts and CrashLoopBackOff status for that container, potentially affecting overall Pod readiness depending on how readiness is configured.
50. Why do interviewers commonly ask "what is a Pod" as a first Kubernetes question? — Because nearly every other concept (Services, controllers, scheduling, probes, volumes) is defined in terms of the Pod abstraction, making it the foundational building block to understand first.
