VictoriaMetrics

A Prometheus-compatible, far more storage- and compute-efficient time series database for long-term, high-cardinality metrics at scale.
← Prometheus Course Index Grafana →

Explanation

VictoriaMetrics (VM) is a time series database and monitoring solution that speaks the Prometheus remote-write/remote-read and PromQL protocols, but is implemented from scratch with a storage engine that is dramatically more efficient on disk, memory, and CPU than vanilla Prometheus's local TSDB. Teams don't adopt it because Prometheus is "wrong" — Prometheus's pull model, service discovery, and PromQL are all excellent — they adopt it because Prometheus's single-node local storage does not scale well past a certain cardinality and retention window, and federating many Prometheus servers becomes an operational headache.

The core problem VictoriaMetrics solves is long-term, high-cardinality storage without a linear cost/complexity explosion. Vanilla Prometheus stores everything on local disk per-instance; once you want 1+ years of retention, tens of millions of active series, or a single global query view across many clusters, you either run a fleet of sharded Prometheus servers with Thanos/Cortex-style sidecars, or you replace the storage layer entirely. VictoriaMetrics takes the second approach: keep Prometheus doing what it's good at (scraping, alerting rules, service discovery) and use `vmagent` to remote_write scraped samples into a purpose-built, horizontally scalable storage cluster.

Architecturally VM ships in two flavors: single-node (one binary, disk-based, extremely simple to operate, handles surprisingly large workloads) and cluster mode, split into three independently scalable components: vminsert (accepts writes, shards them by metric name/labels across storage nodes), vmstorage (actually persists compressed time series data, the workhorse), and vmselect (serves PromQL/MetricsQL queries by fanning out to the relevant vmstorage nodes and merging results). This separation lets you scale ingest and query paths independently — a classic write-heavy/read-light workload doesn't need to over-provision query capacity, and vice versa.

Within the broader observability ecosystem, VictoriaMetrics sits exactly where Prometheus's local TSDB sits, but as a drop-in remote backend: Prometheus (or `vmagent`, VM's own lightweight scrape agent) remote-writes into it, Grafana queries it via a Prometheus-compatible datasource, and Alertmanager still fires alerts (VM ships its own `vmalert` component that can also evaluate recording/alerting rules directly against VM storage, removing the need to run Prometheus at all in some setups). This is why many platform teams describe their migration as "Prometheus for scraping, VictoriaMetrics for everything else."

The efficiency gains come from VM's custom storage format: aggressive columnar compression tuned for time series data, deduplication, and a merge-tree-like structure (inspired by ClickHouse) that dramatically reduces disk I/O and storage footprint — commonly cited as 5-10x less disk space and RAM than the equivalent Prometheus TSDB for the same data. At Meesho-scale, where hundreds of microservices across multiple EKS clusters emit millions of active series, this difference is the line between a monitoring stack that fits in a reasonable budget and one that doesn't.

Internal Working / Interactions

  • vmagent — a lightweight, Prometheus-compatible scrape agent; discovers targets (via the same Kubernetes SD mechanisms Prometheus uses), scrapes them, and remote_writes samples onward. Can also relabel, shard, and buffer during backend outages.
  • vminsert — the write entrypoint for cluster mode; consistently hashes each time series (by metric name + labels) to the correct set of vmstorage nodes.
  • vmstorage — durable storage nodes; each holds a shard of the overall series and answers query fragments from vmselect.
  • vmselect — the read entrypoint; parses PromQL/MetricsQL, fans the query out to relevant vmstorage nodes, and merges/aggregates results before returning to Grafana or an alerting engine.
  • vmalert — evaluates recording and alerting rules directly against VM, sending firing alerts to Alertmanager, closing the loop without needing Prometheus itself for rule evaluation.
  • Grafana talks to vmselect (or the single-node binary) using the standard Prometheus datasource type, since the query API is wire-compatible.

Code Example

# vmagent Deployment — replaces (or complements) Prometheus's scrape role,
# remote_writing samples into the VictoriaMetrics cluster's insert endpoint.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vmagent
  namespace: monitoring
spec:
  replicas: 2                       # vmagent can run >1 replica; use --remoteWrite.label to dedupe if needed
  selector:
    matchLabels: { app: vmagent }
  template:
    metadata:
      labels: { app: vmagent }
    spec:
      serviceAccountName: vmagent          # needs RBAC to list/watch Pods, Services, Endpoints (like Prometheus)
      containers:
        - name: vmagent
          image: victoriametrics/vmagent:v1.101.0
          args:
            - "--promscrape.config=/etc/vmagent/scrape.yaml"   # same syntax as prometheus.yml scrape_configs
            - "--remoteWrite.url=http://vminsert.monitoring.svc:8480/insert/0/prometheus/"
            - "--remoteWrite.tmpDataPath=/tmpdata"             # local disk buffer if vminsert is unreachable
            - "--remoteWrite.maxDiskUsagePerURL=2GB"           # caps buffer growth during backend outages
          volumeMounts:
            - name: config
              mountPath: /etc/vmagent
            - name: tmp
              mountPath: /tmpdata
      volumes:
        - name: config
          configMap: { name: vmagent-scrape-config }
        - name: tmp
          emptyDir: {}
---
# vmstorage StatefulSet — the durable, shardable storage tier.
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: vmstorage
  namespace: monitoring
spec:
  serviceName: vmstorage
  replicas: 3                        # each replica is an independent shard; vminsert hashes series across all 3
  selector:
    matchLabels: { app: vmstorage }
  template:
    metadata:
      labels: { app: vmstorage }
    spec:
      containers:
        - name: vmstorage
          image: victoriametrics/vmstorage:v1.101.0-cluster
          args:
            - "--retentionPeriod=12"          # months of retention kept on this shard
            - "--storageDataPath=/storage"
          ports:
            - { containerPort: 8482, name: http }
            - { containerPort: 8400, name: vminsert-conn }
            - { containerPort: 8401, name: vmselect-conn }
          volumeMounts:
            - name: data
              mountPath: /storage
  volumeClaimTemplates:
    - metadata: { name: data }
      spec:
        accessModes: ["ReadWriteOnce"]
        resources: { requests: { storage: 200Gi } }   # size per shard; scale nodes horizontally rather than disks vertically

kubectl / CLI Commands

kubectl get pods -n monitoring -l app=vmstorage           # check all storage shards are Running
kubectl logs -n monitoring deploy/vmagent --tail=100       # see scrape/remote_write errors
kubectl port-forward -n monitoring svc/vmselect 8481:8481  # query vmselect locally
curl 'http://localhost:8481/select/0/prometheus/api/v1/query?query=up'   # ad-hoc PromQL query against VM
kubectl exec -n monitoring vmstorage-0 -- df -h /storage   # check disk usage on a storage shard
kubectl get vmagent,vmstorage,vmselect -n monitoring        # if using the VictoriaMetrics Operator CRDs
kubectl describe pod -n monitoring vmagent-xxxx             # diagnose CrashLoopBackOff / config errors

Best Practices & Common Pitfalls

Best Practices

  • Use the VictoriaMetrics Operator in Kubernetes rather than hand-rolling manifests — it manages upgrades and scaling for you.
  • Start with single-node VM; only move to cluster mode once you clearly exceed its vertical limits (millions of active series).
  • Set -remoteWrite.maxDiskUsagePerURL on vmagent so a backend outage doesn't fill node disks.
  • Keep vmstorage replicas at 3+ with replication factor set for HA in production.
  • Use recording rules (via vmalert) to pre-aggregate expensive high-cardinality queries used on dashboards.

Pitfalls

  • Migrating without validating label cardinality first — VM is efficient but unbounded cardinality still hurts any TSDB.
  • Forgetting vmselect and vminsert scale independently — over-provisioning one because the other is under load.
  • Not sizing vmstorage PVCs for the full retention period up front, causing disruptive resize operations later.
  • Assuming 100% PromQL compatibility for every advanced function — check MetricsQL docs for edge-case differences.

50 Interview Questions

1. Why do teams adopt VictoriaMetrics instead of just running more Prometheus servers?
Short answer: Prometheus's local TSDB doesn't scale well past a certain cardinality/retention, and federating many Prometheus servers is operationally painful.
Detail: VictoriaMetrics keeps Prometheus doing scraping/alerting rules well, but replaces the storage layer with a horizontally scalable, far more disk/CPU-efficient engine.
Production example: At Meesho-scale, hundreds of microservices across multiple EKS clusters emit millions of active series — a single global VM cluster avoids sharding dozens of independent Prometheus instances.
Common mistake: Assuming Prometheus itself is "wrong" — its pull model, service discovery, and PromQL are kept; only storage is swapped out.
2. What is the core problem VictoriaMetrics is designed to solve?
Short answer: Long-term, high-cardinality time series storage without a linear cost/complexity explosion.
Detail: Vanilla Prometheus stores everything on local per-instance disk; scaling retention or series count means either a sharded Prometheus fleet with Thanos/Cortex-style sidecars, or replacing the storage layer.

Detail: Single-node handles surprisingly large workloads with minimal operational overhead; cluster mode is needed once you clearly exceed single-node's vertical limits.
Production example: Best practice per the reference doc: start with single-node VM and only move to cluster mode once you hit millions of active series.
Common mistake: Jumping straight to cluster mode "for scale" before validating single-node actually can't handle the workload.

Common mistake: Not sizing vmstorage PVCs for the full retention period up front, causing disruptive resize operations later.
7. What does vmselect do, and why can Grafana query it without special configuration?

Detail: Because vmselect's query API is wire-compatible with Prometheus, Grafana talks to it using the standard Prometheus datasource type.
Production example: kubectl port-forward -n monitoring svc/vmselect 8481:8481 then querying /select/0/prometheus/api/v1/query?query=up ad-hoc.
Common mistake: Assuming Grafana needs a dedicated "VictoriaMetrics" plugin type instead of the ordinary Prometheus datasource.
8. Why does cluster mode split ingest (vminsert) and query (vmselect) into separate components?
Short answer: So ingest and query paths can be scaled independently based on actual load.
Detail: A write-heavy/read-light workload doesn't need to over-provision query capacity, and vice versa — a monolithic component would force both to scale together.
Production example: Scaling vminsert replicas during a traffic spike in scrape volume without touching vmselect at all.
