API Server

The single front door to Kubernetes: REST API, the authn/authz/admission pipeline, aggregation, and the watch mechanism.
← Worker Nodes Course Index etcd →

Explanation

The kube-apiserver is the only component in the entire cluster that talks to etcd. Every other piece — kubectl, kubelets, controllers, the scheduler, even other control-plane components — interacts with the cluster exclusively through the API Server's REST interface. It exposes every Kubernetes object (Pods, Services, Deployments, CRDs...) as a resource under a versioned HTTP path (e.g. /api/v1/namespaces/default/pods, /apis/apps/v1/namespaces/default/deployments), validates schema, and is the single enforcement point for security policy.

Kubernetes needs a centralized API gateway because a distributed system with dozens of independent controllers cannot safely allow direct database access — there'd be no consistent place to enforce RBAC, no consistent validation, and no way to intercept/mutate objects (webhooks) before they're persisted. By forcing every write through one code path, Kubernetes gets a single choke point where authentication, authorization, quota enforcement, and admission policy can all be applied uniformly regardless of which client is making the request.

Every request passes through a strict pipeline: Authentication (who are you — client cert, bearer token, OIDC) → Authorization (RBAC — are you allowed to do this verb on this resource in this namespace?) → Admission Control (mutating webhooks can modify the object, e.g. inject a sidecar; validating webhooks and built-in plugins like ResourceQuota or NodeRestriction can reject it) → schema validation → persisted to etcd. Only after all of this succeeds does the object become "real."

The API Server also supports an aggregation layer, letting you register additional API servers (like the metrics-server or custom API extensions) that appear under the same unified API surface at /apis/, and a watch API — long-lived HTTP connections that stream change events to clients instead of requiring them to poll. This watch mechanism, combined with client-side caches (informers), is what lets Kubernetes scale to thousands of watching clients without collapsing under polling load.

Because it is stateless (all durable state lives in etcd), the API Server scales horizontally trivially — production clusters run 3+ replicas behind a load balancer. Its performance is nonetheless gated by etcd write latency and by how many watch connections/list calls it must serve, which is why API Priority and Fairness (APF) exists to prevent one noisy client from starving others.

Internal Working / Interactions

  • API Server ↔ etcd: the only reader/writer; every mutating request becomes a Raft-replicated write.
  • API Server ↔ Authentication plugins: client certs, static tokens, OIDC, webhook token auth — pluggable chain, first success wins.
  • API Server ↔ RBAC: evaluates Role/ClusterRole bindings for the authenticated identity on every request.
  • API Server ↔ Admission webhooks: calls out to external HTTP services (mutating first, then validating) before persistence.
  • API Server ↔ every controller/kubelet/scheduler: serves long-lived watch streams so each can react to changes without polling.

Code Example

# ValidatingWebhookConfiguration — registers a policy check into the API Server's
# admission pipeline. This is how OPA Gatekeeper / Kyverno plug into the API Server.
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: require-resource-limits
webhooks:
  - name: require-limits.policy.example.com
    admissionReviewVersions: ["v1"]
    sideEffects: None
    failurePolicy: Fail        # reject the request if the webhook is unreachable (fail closed)
    clientConfig:
      service:
        name: policy-webhook    # Service fronting the webhook server
        namespace: policy-system
        path: "/validate-pods"
      caBundle: 
    rules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE"]
        resources: ["pods"]
    namespaceSelector:
      matchLabels:
        policy-enforced: "true"   # only applies to namespaces opting in
---
# Minimal RBAC that the API Server evaluates during authorization for a CI ServiceAccount
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: default
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]   # least-privilege: read-only, no create/delete

Best Practices & Common Pitfalls

Best Practices

  • Enable audit logging for compliance and incident forensics.
  • Set failurePolicy deliberately on webhooks — Fail for security-critical checks, Ignore for best-effort ones.
  • Use API Priority and Fairness to isolate noisy or misbehaving clients.
  • Keep admission webhook latency low (<1s) — every request blocks on it.

Pitfalls

  • Registering a webhook with failurePolicy: Fail and no HA — a single webhook outage blocks all matching API calls cluster-wide.
  • Bypassing the API Server (direct etcd edits) — skips all authz/admission entirely.
  • Overly broad RBAC bindings (cluster-admin for convenience) instead of least privilege.
  • Not accounting for aggregation-layer availability (e.g. metrics-server down breaks kubectl top).

50 Interview Questions

1. Why is the API Server the only component that talks to etcd?
Short answer: Centralizing etcd access is the single enforcement point for authn, authz, admission, and schema validation.
Detailed explanation: If controllers or kubelets wrote to etcd directly, there'd be no consistent place to check RBAC or run mutating/validating webhooks — every write path would need its own security logic, defeating the purpose of a uniform policy layer.
Production example: A ResourceQuota admission check only works because every namespace-scoped create request is forced through the same API Server code path.
Common mistake: Editing etcd directly to "fix" an object — this skips all admission control and validation entirely.
2. Walk through the full request pipeline for a kubectl apply.
Short answer: Authentication → Authorization (RBAC) → Mutating admission → Schema validation → Validating admission → persisted to etcd.
Detailed explanation: Authentication identifies the caller (client cert, bearer token, OIDC); Authorization checks if that identity can perform the verb on the resource; mutating webhooks can modify the object (e.g. inject a sidecar); the object is validated against its OpenAPI schema; validating webhooks can still reject it; only then is it written to etcd via Raft consensus.
Production example: Istio's sidecar injector runs as a mutating webhook early in this chain, before OPA Gatekeeper's validating webhook checks policy compliance.
Common mistake: Assuming validating webhooks run before mutating ones — mutating always runs first so validation sees the final object.
3. What is the difference between mutating and validating admission webhooks?
Short answer: Mutating webhooks can modify the incoming object; validating webhooks can only accept or reject it.
Detailed explanation: Mutating webhooks run first and can patch the object (add labels, inject containers); validating webhooks run after schema validation and see the final, mutated object, deciding admit/deny with no further changes allowed.
Production example: Istio's injector (mutating) adds an Envoy sidecar container; a policy engine (validating) then confirms the resulting Pod still meets security requirements.
Common mistake: Writing a validating webhook that tries to modify the object — validating webhooks cannot mutate, only allow/deny.
4. What is the watch API and why does it matter for scalability?
Short answer: Long-lived HTTP connections that stream change events to clients instead of requiring polling.
Detailed explanation: Combined with client-side informer caches, watch lets thousands of controllers/kubelets react to changes within milliseconds without hammering the API Server with repeated list calls — this is what allows Kubernetes to scale to large clusters.
Production example: Every kubelet watches only Pods bound to its own node, and every controller watches only its resource type, minimizing redundant traffic.
Common mistake: Writing a custom controller that polls the API Server on a timer instead of using watch/informers — this doesn't scale and adds unnecessary load.
5. What is the aggregation layer and what problem does it solve?
Short answer: It lets you register additional API servers that appear under the same unified API surface at /apis/.
Detailed explanation: Instead of every extension needing to be built into kube-apiserver itself, the aggregation layer allows components like metrics-server to serve their own API while looking, to clients, like part of the core API.
Production example: kubectl top works because metrics-server registers itself via the aggregation layer under metrics.k8s.io.
Common mistake: Forgetting that aggregation-layer availability matters too — if metrics-server is down, kubectl top fails even though the core API Server is healthy.
6. Why is the API Server horizontally scalable while Scheduler/Controller Manager are not (in the same way)?
Short answer: The API Server is stateless with no leader election — all replicas can serve traffic simultaneously; Scheduler/Controller Manager use leader election so only one is ever active.
Detailed explanation: Because durable state lives only in etcd, any API Server replica can serve any request; but Scheduler/Controller Manager making conflicting decisions concurrently would cause race conditions, so they intentionally run single-active via a Lease.
Production example: Running 3 API Server replicas behind a load balancer scales read/write throughput; running 3 Scheduler replicas only improves failover time, not throughput.
Common mistake: Assuming adding more Scheduler replicas increases scheduling throughput.
7. What is API Priority and Fairness (APF) and why does it exist?
Short answer: A mechanism to prevent one noisy or misbehaving client from starving other clients of API Server capacity.
Detailed explanation: APF classifies requests into priority levels/flow schemas and applies fair queuing, so a runaway controller issuing excessive list/watch calls can't degrade the whole cluster's API responsiveness.
Production example: A custom controller with a bug causing excessive full-list calls gets throttled by APF instead of taking down API Server responsiveness for everyone else.
Common mistake: Not enabling/tuning APF in large multi-tenant clusters and being surprised when one bad actor degrades cluster-wide API latency.
8. What's the risk of setting failurePolicy: Fail on an admission webhook without HA?
Short answer: A single webhook outage blocks every matching API request cluster-wide.
Detailed explanation: failurePolicy: Fail means the API Server rejects the request if it can't reach the webhook — appropriate for security-critical checks, but catastrophic if the webhook service itself has no redundancy and goes down.
Production example: A policy webhook Deployment scaled to a single replica crashes, and suddenly no Pods can be created anywhere in the cluster.
Common mistake: Deploying admission webhooks as single-replica services while setting failurePolicy: Fail.
9. How does RBAC authorization actually get evaluated for a request?
Short answer: The API Server checks Role/ClusterRole bindings for the authenticated identity against the requested verb, resource, and namespace.
Detailed explanation: After authentication establishes identity (a user or ServiceAccount), the authorization stage looks up all applicable RoleBindings/ClusterRoleBindings and checks if any grants the specific verb (get/list/create/delete/etc.) on the target resource.
Production example: kubectl auth can-i create pods --as=system:serviceaccount:default:ci-bot simulates exactly this check without making the real request.
Common mistake: Granting cluster-admin broadly for convenience instead of scoping least-privilege Roles per namespace/ServiceAccount.
10–50. Additional rapid-fire questions
10. What does kubectl api-versions show? — All API groups/versions currently served by the API Server, including aggregated ones.
11. How does kubectl explain get its schema information? — It queries the API Server's live OpenAPI schema definitions.
12. What authentication methods can the API Server accept? — Client certificates, static bearer tokens, OIDC tokens, and webhook token authentication, evaluated in a pluggable chain.
13. What happens if authentication fails? — The request is rejected with 401 before authorization or admission is ever evaluated.
14. What happens if authorization (RBAC) fails? — The request is rejected with 403 Forbidden before any admission webhook runs.
15. Why is schema validation performed after mutating admission? — Because mutating webhooks might change the object's shape, so validation must check the final version, not the original request body.
16. What object registers a validating webhook? — A ValidatingWebhookConfiguration.
17. What object registers a mutating webhook? — A MutatingWebhookConfiguration.
18. Why does the API Server need a caBundle in a webhook config? — To verify the TLS certificate presented by the webhook service when the API Server calls out to it.
19. What's the purpose of audit logging on the API Server? — Records every request for compliance and incident forensics, independent of whether it succeeded or failed.
20. Why is the API Server described as "stateless"? — It holds no durable data itself; all persistent state lives in etcd, which it's the sole gateway to.
21. What HTTP verb maps to a Kubernetes "watch"? — A GET request with ?watch=true, kept open as a streaming HTTP connection.
22. How does an informer reduce load on the API Server? — It maintains a local cache populated by an initial list + ongoing watch, so controllers query the cache instead of re-listing from the API Server repeatedly.
23. What's the risk of an admission webhook with high latency? — Every matching request blocks on it, so slow webhooks directly slow down all affected API operations cluster-wide.
24. Why might kubectl top fail even if core API Server health checks pass? — It depends on the aggregation layer and metrics-server being healthy, which is separate from core API Server health.
25. What does namespaceSelector on a webhook config do? — Scopes the webhook to only fire for objects in namespaces matching specific labels, letting some namespaces opt out.
26. Why is kubectl auth can-i useful before debugging a "Forbidden" error? — It directly simulates the RBAC decision without side effects, quickly confirming whether it's an authorization issue.
27. What's a typical cause of "the server doesn't have a resource type" errors? — The API group/version for that resource (often a CRD) isn't registered/installed on this cluster.
28. Why does the API Server enforce OpenAPI schema validation? — To reject malformed objects before they ever reach etcd, keeping stored data structurally consistent.
29. What's the effect of setting failurePolicy: Ignore on a webhook? — If the webhook is unreachable, the request proceeds as if the webhook had allowed it — safer for non-critical checks, riskier for security ones.
30. How would you find all currently registered admission webhooks? — kubectl get validatingwebhookconfigurations and kubectl get mutatingwebhookconfigurations.
31. Why does API Server performance depend heavily on etcd? — Every write must be committed to etcd via Raft before the request completes; slow etcd fsync directly slows every write-path API call.
32. What's the purpose of --enable-admission-plugins? — Enables specific built-in admission controllers (ResourceQuota, NodeRestriction, etc.) as part of the pipeline.
33. What kind of identity can a webhook's clientConfig.service point to? — An in-cluster Service fronting the webhook server, resolved and called by the API Server over HTTPS.
34. Why is it dangerous to grant broad RBAC to CI/CD ServiceAccounts? — A compromised CI pipeline with excessive permissions can create/delete/modify arbitrary cluster resources.
35. How does the API Server notify watchers after a successful write? — It dispatches the change event to all active watch streams matching that resource/selector immediately after the etcd commit is acknowledged.
36. What's a common reason for "429 Too Many Requests" from the API Server? — API Priority and Fairness throttling a client that's exceeded its allotted concurrency/queue capacity.
37. Why should webhook servers be deployed with multiple replicas? — To avoid a single point of failure blocking all matching admission requests, especially under failurePolicy: Fail.
38. What's the difference between a Role and a ClusterRole? — A Role is namespace-scoped; a ClusterRole can be applied cluster-wide or bound within a specific namespace via a RoleBinding.
39. Why does kubectl explain pod.spec --recursive matter for interview/exam prep? — It shows you can query the live schema instead of memorizing every field, reflecting real-world workflow.
40. What's the operational impact of API Server certificate expiry? — Clients (kubectl, kubelets, controllers) fail to authenticate, effectively causing a cluster-wide outage of new operations.
41. Why is rate-limiting/APF particularly important in multi-tenant clusters? — Multiple independent teams' controllers/CI pipelines share the same API Server, so one team's bug shouldn't degrade everyone else.
42. What's the purpose of the sideEffects field on a webhook? — Declares whether the webhook has side effects outside the admission request, which affects dry-run behavior compatibility.
43. How does the API Server support optimistic concurrency? — Via resourceVersion on objects — a write with a stale resourceVersion is rejected, forcing the client to re-fetch and retry.
44. Why is direct etcd access considered a security anti-pattern even for read-only debugging? — etcd has no RBAC of its own comparable to the API Server's; direct read access exposes plaintext Secrets unless encryption-at-rest is configured.
45. What's a good diagnostic first step when many unrelated kubectl commands suddenly start timing out? — Check API Server health (/healthz) and then etcd latency, since API Server slowness is very often downstream of etcd.
46. Why does dry-run (--dry-run=server) still invoke admission webhooks? — To accurately simulate what would happen, including mutations and validation rejections, without persisting to etcd.
47. What's the risk of a mutating webhook that isn't idempotent? — Reapplying the same manifest could keep adding/duplicating injected content on every mutation pass.
48. Why does OPA Gatekeeper/Kyverno typically run as validating (not mutating) webhooks for policy enforcement? — Policy decisions are about allow/deny; mutation is a separate concern usually handled by dedicated defaulting webhooks.
49. What's a sign that admission webhook latency is impacting cluster performance? — Elevated API Server request latency specifically for resource types matched by the webhook's rules, not across all resources.
50. Why is the API Server considered the cluster's single trust boundary? — Every authn/authz/admission decision funnels through it; a misconfiguration there (RBAC, webhook) compromises the whole cluster regardless of how secure other components are.
