Container Runtime

The CRI shim, containerd/CRI-O internals, the image pull flow, and the OCI runtime spec underneath it all.
← kube-proxy Course Index Pods →

Explanation

The container runtime is the layer that does the actual, literal work of running a container: pulling an image, unpacking its filesystem layers, creating Linux namespaces and cgroups, and executing the containerized process. Kubernetes itself has no idea how to do any of this — kubelet delegates every bit of it to the runtime through a standardized gRPC API called the CRI (Container Runtime Interface).

Kubernetes needs this separation because container runtimes are a genuinely separate, fast-moving ecosystem concern, and early Kubernetes paid a real cost for being tightly coupled to Docker specifically. The CRI was introduced precisely so kubelet's code never needs to change when the runtime changes — today, containerd and CRI-O are the dominant CRI-compliant runtimes (Docker Engine itself is no longer used directly; even Docker Desktop-based clusters run containerd underneath).

Underneath the CRI-compliant runtime sits an even lower-level layer: the OCI (Open Container Initiative) runtime spec, implemented by tools like runc (or crun, or gVisor/Kata for sandboxed isolation). containerd/CRI-O handle image management, storage, and the CRI gRPC surface, then shell out to an OCI runtime to actually create the low-level namespaces/cgroups and exec() the container's entrypoint process. This is a genuine layered architecture: CRI (Kubernetes-facing) → containerd/CRI-O (image + lifecycle management) → OCI runtime (actual process creation).

The image pull flow matters operationally because it's a common source of slow Pod startup and production incidents: kubelet asks the runtime to pull an image, the runtime checks its local content-addressable store first, and if missing, pulls each layer from the registry (with configurable pull policy — Always, IfNotPresent, Never) and unpacks it before the container can start. Registry rate limits, network latency to the registry, and large image sizes all directly translate into slower Pod scheduling-to-Running time — one reason production clusters run local registry mirrors/pull-through caches.

The runtime also enforces the resource limits you set in a Pod spec by configuring cgroups — CPU limits become CFS quota/period settings, memory limits become cgroup memory limits that trigger OOM-kill when exceeded. Understanding that this enforcement happens here, not in kubelet or the API Server, is key to correctly diagnosing OOMKilled and CPU-throttling incidents.

Internal Working / Interactions

  • kubelet ↔ Container Runtime (CRI): gRPC calls over a Unix socket — RunPodSandbox, PullImage, CreateContainer, StartContainer, ContainerStatus, StopContainer, RemoveContainer.
  • Container Runtime ↔ image registry: pulls image manifests + layers (OCI image spec) over HTTPS, verifies digests, caches layers in local content store.
  • Container Runtime ↔ OCI runtime (runc/crun): generates an OCI runtime spec (config.json) describing namespaces/cgroups/mounts, then invokes the OCI runtime to create and start the actual process.
  • Container Runtime ↔ cgroups: translates Pod resources.limits/requests into cgroup v1/v2 controller settings enforced by the kernel.
  • Container Runtime ↔ CNI (via kubelet): the runtime creates the network namespace during sandbox creation; kubelet then invokes CNI plugins to wire it up.

Code Example

# containerd config — the runtime's own config file, not a Kubernetes YAML.
# /etc/containerd/config.toml
version = 2

[plugins."io.containerd.grpc.v1.cri"]
  sandbox_image = "registry.k8s.io/pause:3.9"   # the "pause" container that holds the Pod's namespaces

[plugins."io.containerd.grpc.v1.cri".containerd]
  default_runtime_name = "runc"

[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
  runtime_type = "io.containerd.runc.v2"

[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
  SystemdCgroup = true                # MUST match kubelet's cgroupDriver=systemd setting

[plugins."io.containerd.grpc.v1.cri".registry.mirrors."docker.io"]
  endpoint = ["https://mirror.internal.example.com"]   # pull-through cache, avoids Docker Hub rate limits

# crictl — the CRI-level CLI, used to debug the runtime directly (bypassing kubectl)
# crictl pull nginx:1.25
# crictl images
# crictl ps -a
# crictl inspect

kubectl Commands

# See which runtime + version a node reports
kubectl get nodes -o custom-columns=NAME:.metadata.name,RUNTIME:.status.nodeInfo.containerRuntimeVersion

# Check image pull policy and status for a Pod
kubectl describe pod my-pod | grep -A3 Image

# See image pull failures/backoff
kubectl get events --field-selector reason=Failed,reason=BackOff

# Confirm resource limits are actually enforced (cgroup-level effect visible via kubectl top)
kubectl top pod my-pod --containers

# On the node (debug pod or SSH), talk to the runtime directly:
# crictl ps -a
# crictl logs 
# crictl inspect  | grep -i cgroup

kubectl describe node worker-3 | grep -i "Container Runtime"

Best Practices & Common Pitfalls

Best Practices

  • Run a registry pull-through cache/mirror to avoid rate limits and reduce Pod startup latency.
  • Keep the runtime's cgroup driver setting in sync with kubelet's.
  • Use imagePullPolicy: IfNotPresent for immutable, digest-pinned production images.
  • Set both CPU/memory requests and limits so cgroup enforcement behaves predictably.

Pitfalls

  • Mismatched cgroup drivers between kubelet and runtime — causes subtle, hard-to-diagnose instability.
  • Using imagePullPolicy: Always universally, adding registry round-trip latency to every Pod start.
  • Diagnosing "container won't start" purely via kubectl instead of dropping to crictl on the node when the issue is below kubelet.
  • Ignoring image layer bloat — huge images slow scheduling-to-Running time cluster-wide during traffic spikes/autoscale events.

50 Interview Questions

1. What is the container runtime's job, and why doesn't kubelet do this work directly?
Short answer: The runtime pulls images, unpacks filesystem layers, creates namespaces/cgroups, and executes the containerized process; kubelet delegates all of this via the CRI so it never needs runtime-specific code.
Detail: Kubernetes itself has no built-in knowledge of how to actually run a container process — that's a genuinely separate, fast-moving ecosystem concern handled entirely by the runtime layer.
Production example: kubelet issues a CRI CreateContainer call and never touches image layers or cgroup files itself — containerd does all of that work.
Common mistake: Assuming kubelet directly manipulates containers/images — it only ever talks CRI gRPC to a runtime that does the real work.
2. Why was the CRI introduced, and what problem did tight Docker coupling cause before it existed?
Short answer: Early Kubernetes paid a real cost being tightly coupled to Docker specifically; CRI standardized the interface so kubelet's code never needs to change when the runtime changes.
Detail: CRI is a gRPC API contract — any CRI-compliant runtime (containerd, CRI-O) can be swapped in without touching kubelet internals, decoupling the fast-moving runtime ecosystem from core Kubernetes development.
Production example: Migrating a fleet from dockershim to containerd required runtime-level configuration changes only, with zero kubelet code changes.
Common mistake: Believing Docker Engine is still used directly today — even Docker Desktop-based clusters run containerd underneath now.
3. What are the three layers in the runtime architecture, from Kubernetes-facing down to the actual process?
Short answer: CRI (Kubernetes-facing gRPC contract) → containerd/CRI-O (image + lifecycle management) → OCI runtime (actual process creation).
Detail: containerd/CRI-O handle image management, storage, and the CRI gRPC surface, then generate an OCI runtime spec and shell out to an OCI runtime like runc to create namespaces/cgroups and exec the entrypoint.
Production example: containerd generates a config.json OCI spec describing the container's namespaces/mounts, then invokes runc to actually create the process.
Common mistake: Conflating "container runtime" (containerd/CRI-O) with "OCI runtime" (runc) as if they're the same layer — they have distinct responsibilities.
4. What is runc, and what alternatives exist for sandboxed isolation?
Short answer: runc is the reference OCI runtime that creates namespaces/cgroups and execs the container process; gVisor and Kata provide stronger sandboxed isolation as alternatives.
Detail: runc runs the container as a regular (namespaced/cgrouped) Linux process sharing the host kernel; gVisor intercepts syscalls in userspace and Kata runs each container in a lightweight VM, trading some performance for stronger isolation boundaries.
Production example: A multi-tenant platform running untrusted workloads might configure a RuntimeClass that routes specific Pods to Kata containers instead of runc for kernel-level isolation.
Common mistake: Assuming all containers on a node share identical isolation guarantees — the underlying OCI runtime (runc vs gVisor vs Kata) materially changes that.
5. Walk through the image pull flow when kubelet asks the runtime to pull an image.
Short answer: The runtime checks its local content-addressable store first; if the image is missing, it pulls each layer from the registry, verifies digests, and unpacks before the container can start.
Detail: This flow is gated by imagePullPolicy (Always, IfNotPresent, Never), and registry rate limits, network latency, and large image sizes directly translate into slower Pod scheduling-to-Running time.
Production example: A cluster running a local registry pull-through cache/mirror to avoid Docker Hub rate limits and cut Pod startup latency during autoscale events.
Common mistake: Using imagePullPolicy: Always universally, adding an unnecessary registry round-trip to every single Pod start even when the image hasn't changed.
6. Where does cgroup-based resource enforcement actually happen, and why does that matter for diagnosing incidents?
Short answer: The container runtime configures cgroups based on Pod spec resources.limits — CPU limits become CFS quota/period settings, memory limits become cgroup memory limits enforced by the kernel.
Detail: Neither kubelet nor the API Server enforces resource limits directly — they only pass values down; the runtime translates them into kernel cgroup controller settings, and the kernel itself performs OOM-kill or CPU-throttling.
Production example: An OOMKilled Pod is the kernel's OOM killer acting on a cgroup memory limit the runtime configured — not a kubelet-level decision.
Common mistake: Trying to explain CPU throttling or OOMKilled purely by looking at kubelet logs instead of understanding it's a kernel/cgroup-level enforcement mechanism.
7. Why must the cgroup driver setting match between kubelet and the container runtime?
Short answer: Both write cgroup settings for the same containers; a mismatch means two different managers compete over the same cgroup hierarchy, causing subtle instability.
Detail: containerd's SystemdCgroup option must align with kubelet's cgroupDriver flag — if one uses systemd and the other cgroupfs, resource enforcement becomes inconsistent in ways that are hard to trace back to the root cause.
Production example: Setting SystemdCgroup = true in containerd's config specifically because kubelet is configured with cgroupDriver=systemd.
Common mistake: Changing one side's cgroup driver during a runtime upgrade without verifying the other side still matches.
8. What is the "pause" container/sandbox_image, and what is it for?
Short answer: A minimal container the runtime creates first to hold the Pod's shared namespaces (network, IPC) before any application containers start.
Detail: Configured via sandbox_image in containerd's config, the pause container's sole job is to exist and hold the namespaces open so all containers in the Pod can share them, even if application containers restart.
Production example: RunPodSandbox in the CRI creates this pause container first, before CreateContainer/StartContainer ever run for the actual application container.
Common mistake: Being confused by an extra "pause" process/container showing up per Pod in crictl ps -a output and thinking it's an application bug.
9. What is crictl, and why would you use it instead of kubectl for a debugging session?
Short answer: crictl is a CRI-level CLI that talks directly to the container runtime, bypassing kubelet and the API Server entirely.
Detail: When an issue is below kubelet — e.g. a stuck image pull, a runtime-level crash, or a cgroup misconfiguration — kubectl can't see far enough; crictl commands like crictl ps -a, crictl inspect, and crictl logs query the runtime directly on the node.
Production example: Diagnosing a container stuck in ContainerCreating by SSHing to the node and running crictl inspect <container-id> to see the actual CRI-level state and error.
Common mistake: Trying to diagnose "container won't start" purely via kubectl instead of dropping to crictl on the node when the failure is at the runtime layer.
10. Who creates the Pod's network namespace, and how does that interact with the CNI plugin?
Short answer: The container runtime creates the network namespace during sandbox creation; kubelet then invokes the CNI plugin to wire it up with actual networking.
Detail: This is a coordinated two-step process — the runtime provides the empty namespace as part of RunPodSandbox, and kubelet's separate call to the CNI plugin populates it with an IP address and routes.
Production example: A Pod stuck in ContainerCreating due to a CNI plugin failure still has a runtime-created sandbox/namespace — the failure is specifically in the kubelet-to-CNI step, not the runtime's sandbox creation.
Common mistake: Assuming the container runtime itself configures Pod networking end-to-end — it only creates the namespace; the CNI plugin (invoked by kubelet) does the actual wiring.
