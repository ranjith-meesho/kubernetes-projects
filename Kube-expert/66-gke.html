GKE (Google Kubernetes Engine)

Google's managed Kubernetes: Autopilot vs Standard, node pools, and Workload Identity for keyless GCP access.
← Cluster Upgrade Course Index EKS →

Explanation

GKE was the first managed Kubernetes offering (Google authored Kubernetes itself, drawing on Borg), and it exposes this maturity as a genuine choice of operating model: Standard mode gives you a fully managed control plane but you still own node pools — you choose machine types, node counts, autoscaling ranges, and you pay for the VMs whether or not they're fully utilized. Autopilot mode goes further and manages nodes too: you only submit Pod specs, GKE provisions right-sized, hardened, autoscaled node capacity behind the scenes, and you're billed per-pod-resource-request rather than per-VM. Autopilot removes an entire category of operational work (node OS patching, node pool sizing, bin-packing efficiency) at the cost of some flexibility — no DaemonSets that need host access, no privileged pods, and a curated set of allowed configurations enforced by built-in admission policies.

A node pool is GKE's unit of homogeneous worker capacity: a group of nodes sharing the same machine type, disk type, labels, taints, and (optionally) GPU/TPU accelerators. Real clusters run multiple node pools deliberately — e.g. a general-purpose pool for most workloads, a memory-optimized pool for data-processing jobs, and a GPU pool for ML inference, each with independent autoscaling min/max and often tainted so only intended workloads schedule there via matching tolerations. Node pools can also mix Spot VMs (deeply discounted, preemptible) for fault-tolerant batch work with on-demand pools for latency-sensitive services.

Workload Identity solves the oldest problem in cloud-native security: how does a Pod get GCP credentials without a long-lived service-account JSON key sitting in a Secret? Workload Identity binds a Kubernetes ServiceAccount to a GCP IAM service account via a trust relationship at the GKE metadata server level. When a Pod using that KSA calls a GCP API (e.g. via the client library, which auto-discovers credentials from the metadata server), GKE's metadata server transparently exchanges the Pod's Kubernetes identity for short-lived GCP IAM credentials — no static keys ever touch the cluster, and the binding is fully auditable and revocable via IAM alone. This is the same problem IRSA solves on EKS, just implemented via GCP's workload identity federation rather than AWS's OIDC federation.

Operationally, GKE's biggest practical decisions are: Standard vs Autopilot (choose Autopilot by default for new clusters unless you have a specific need for host-level access or unusual workloads — it removes real toil), regional vs zonal clusters (regional replicates the control plane and, typically, node pools across zones for free HA — almost always the right default), and release channels (Rapid/Regular/Stable trade newer features for stability; Stable is the right default for production, giving Google's own upgrade-testing signal time to catch issues before you inherit them).

Internal Working / Interactions

  • Control plane: fully managed and invisible — Google runs and upgrades etcd/API-server/scheduler; you interact only via the GKE API/CLI/console and the cluster's kube-apiserver endpoint.
  • Node pools (Standard): backed by a Managed Instance Group per pool; the Cluster Autoscaler resizes the MIG based on unschedulable-pod pressure and pool min/max bounds.
  • Autopilot scheduling: GKE's own controller provisions right-sized nodes on demand per Pod resource requests, applying its own bin-packing and security policies before your Pod is placed.
  • Workload Identity binding: a KSA is annotated with the target GCP service account; GKE's metadata-server proxy on each node intercepts credential requests from Pods using that KSA and mints short-lived tokens for the bound GSA.
  • GKE Autoscaler + VPA: Cluster Autoscaler handles node count, HPA/VPA handle Pod replica count and per-Pod resource sizing — all three cooperate but are independently configured.

Code Example

#############################################
# 1. Create a GKE Autopilot cluster (gcloud)
#############################################
gcloud container clusters create-auto prod-autopilot \
  --region us-central1 \                      # regional = HA control plane across zones, by default
  --release-channel stable                    # stable channel: slower, safer version rollout

# Create a GKE Standard cluster with an explicit node pool
gcloud container clusters create prod-standard \
  --region us-central1 \
  --num-nodes 2 \                              # 2 nodes per zone in the region
  --machine-type e2-standard-4 \
  --enable-autoscaling --min-nodes 2 --max-nodes 6 \
  --workload-pool=my-project.svc.id.goog       # enables Workload Identity for the cluster

# Add a second, GPU-backed node pool for inference workloads
gcloud container node-pools create gpu-pool \
  --cluster prod-standard --region us-central1 \
  --machine-type n1-standard-8 \
  --accelerator type=nvidia-tesla-t4,count=1 \
  --node-taints=nvidia.com/gpu=present:NoSchedule   # only GPU-tolerant pods land here

---
# 2. Workload Identity binding (Kubernetes ServiceAccount <-> GCP IAM ServiceAccount)
apiVersion: v1
kind: ServiceAccount
metadata:
  name: inference-ksa
  namespace: ml
  annotations:
    iam.gke.io/gcp-service-account: inference-gsa@my-project.iam.gserviceaccount.com
---
# Grant the GCP-side binding (run once via gcloud, not YAML):
# gcloud iam service-accounts add-iam-policy-binding inference-gsa@my-project.iam.gserviceaccount.com \
#   --role roles/iam.workloadIdentityUser \
#   --member "serviceAccount:my-project.svc.id.goog[ml/inference-ksa]"
apiVersion: v1
kind: Pod
metadata:
  name: inference-worker
  namespace: ml
spec:
  serviceAccountName: inference-ksa   # Pod inherits GCP creds via Workload Identity, no key files
  containers:
    - name: worker
      image: gcr.io/my-project/inference:v4

kubectl / CLI Commands

# Fetch cluster credentials into local kubeconfig
gcloud container clusters get-credentials prod-standard --region us-central1

# List node pools and their autoscaling config
gcloud container node-pools list --cluster prod-standard --region us-central1

# Check available GKE versions / release channel status
gcloud container get-server-config --region us-central1

# Verify Workload Identity is enabled on the cluster
gcloud container clusters describe prod-standard --region us-central1 --format="value(workloadIdentityConfig)"

# List nodes and confirm which pool/machine-type they belong to
kubectl get nodes -L cloud.google.com/gke-nodepool

# Check a Pod's effective GCP identity via the Workload Identity path
kubectl exec -it inference-worker -n ml -- curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/email"

# Manually trigger a node pool upgrade
gcloud container clusters upgrade prod-standard --node-pool general --region us-central1

Best Practices & Common Pitfalls

Best Practices

  • Default to Autopilot for new clusters; use Standard only when you need host-level control.
  • Use regional clusters for production — free HA across zones for the control plane.
  • Always use Workload Identity instead of service-account JSON key files.
  • Separate node pools by workload shape (general, memory, GPU) with taints/tolerations.
  • Pin production clusters to the Stable release channel.

Common Pitfalls

  • Mounting long-lived GCP service-account keys as Secrets instead of using Workload Identity.
  • Using zonal clusters for production and losing the control plane in a zone outage.
  • Expecting Autopilot to support privileged Pods, hostPath, or DaemonSets with host access.
  • One giant node pool for everything, causing noisy-neighbor and inefficient bin-packing.
  • Ignoring release-channel upgrade notifications until forced auto-upgrade happens.

50 Interview Questions

1. What's the fundamental difference between GKE Standard and Autopilot mode?
Short answer: Standard gives you a managed control plane but you still own and pay for node pools; Autopilot manages nodes too — you submit only Pod specs and get billed per-Pod-resource-request.
Detail: Autopilot removes node OS patching, node pool sizing, and bin-packing efficiency work at the cost of flexibility — no DaemonSets needing host access, no privileged Pods, and a curated set of allowed configurations enforced by built-in admission policies.
Production example: A team migrating from Standard to Autopilot has to remove a privileged DaemonSet it relied on for host-level monitoring, replacing it with a supported alternative.
Common mistake: Choosing Autopilot for a workload that fundamentally needs host-level access, then fighting the platform's restrictions instead of using Standard.
2. What is a node pool, and why do real GKE clusters run more than one?
Short answer: A node pool is GKE's unit of homogeneous worker capacity — nodes sharing machine type, disk type, labels, taints, and optional accelerators.
Detail: Real clusters deliberately run multiple pools (general-purpose, memory-optimized, GPU) each with independent autoscaling bounds, often tainted so only matching-toleration workloads land there, enabling workload-shape-specific bin-packing and cost control.
Production example: A general pool serves most services while a separate GPU pool, tainted with nvidia.com/gpu=present:NoSchedule, serves only ML inference Pods with a matching toleration.
Common mistake: Running one giant node pool for everything, causing noisy-neighbor effects and inefficient bin-packing across dissimilar workload shapes.
3. How does Workload Identity let a Pod get GCP credentials without a long-lived service-account key?
Short answer: It binds a Kubernetes ServiceAccount to a GCP IAM service account via a trust relationship at the metadata-server level; GKE transparently exchanges the Pod's Kubernetes identity for short-lived GCP IAM credentials.
Detail: No static keys ever touch the cluster, and the binding is fully auditable and revocable via IAM alone — client libraries auto-discover these credentials from the metadata server exactly as they would a normal GCE service account.
Production example: A KSA annotated with iam.gke.io/gcp-service-account: inference-gsa@my-project.iam.gserviceaccount.com lets Pods using it call GCP APIs with no Secret-mounted key file anywhere.
Common mistake: Mounting a downloaded service-account JSON key as a Secret instead of configuring Workload Identity, leaving a long-lived credential sitting in the cluster.
4. Why should regional clusters be the default over zonal clusters for production on GKE?
Short answer: Regional clusters replicate the control plane (and typically node pools) across zones for free HA; zonal clusters put the entire control plane in a single zone.
Detail: A zonal cluster's control plane becomes unavailable during that zone's outage, even though it costs less — regional is almost always the right default for anything production-facing.
Production example: A production cluster is created with gcloud container clusters create prod-standard --region us-central1 rather than --zone, spreading control-plane replicas across zones automatically.
Common mistake: Using zonal clusters for production to save cost and losing the entire control plane during a single zone outage.
5. What do GKE release channels (Rapid/Regular/Stable) control, and why default to Stable in production?
Short answer: They trade newer Kubernetes features for stability — Stable gives Google's own upgrade-testing signal time to catch issues before you inherit them.
Detail: Rapid gets new versions almost immediately (useful for testing new features early); Stable delays the same versions until they've proven themselves across the broader GKE fleet first.
Production example: gcloud container clusters create-auto prod-autopilot --release-channel stable pins a production cluster to the safer channel.
Common mistake: Leaving a production cluster on the Rapid channel and being surprised by an early, less-battle-tested version upgrade.
6. How does the Cluster Autoscaler interact with node pools on GKE Standard?
Short answer: Each node pool is backed by a Managed Instance Group; the Cluster Autoscaler resizes that MIG based on unschedulable-pod pressure, staying within the pool's configured min/max bounds.
Detail: This is distinct from Autopilot, where GKE's own controller provisions right-sized nodes on demand per Pod resource request, applying its own bin-packing rather than resizing a pre-defined pool.
Production example: --enable-autoscaling --min-nodes 2 --max-nodes 6 on a node pool lets it grow under load and shrink back down during quiet periods.
Common mistake: Setting min-nodes and max-nodes equal to each other, effectively disabling autoscaling while believing it's still active.
7. Why does Autopilot bill per-Pod-resource-request instead of per-VM?
Short answer: Since GKE manages node provisioning invisibly and bin-packs Pods onto right-sized capacity itself, billing is tied directly to what each Pod actually requests rather than to whole VM instances you'd otherwise have to size and pay for regardless of utilization.
Detail: This removes the classic "paying for idle VM capacity" problem inherent to Standard mode's node-pool model, at the cost of losing direct control over instance shape/bin-packing decisions.
Production example: A team moving batch workloads to Autopilot sees costs track actual CPU/memory requests rather than a fixed set of underutilized VMs.
Common mistake: Setting unrealistically high resource requests on Autopilot Pods "just in case," which directly and immediately inflates the bill since billing follows requests.
8. Why does the VPC subnet/IP model matter less on GKE than on EKS's VPC CNI approach?
Short answer: GKE's networking model differs from EKS's real-VPC-IP-per-Pod approach; GKE Pod IPs come from cluster-scoped alias IP ranges rather than consuming node ENI secondary IPs directly the same way, though IP planning still matters at cluster creation.
Detail: This is a genuine platform difference interviewers probe for — candidates who only know EKS's VPC CNI model sometimes wrongly assume identical IP-exhaustion mechanics apply identically on GKE.
Production example: Planning a large GKE cluster still requires sizing the cluster's pod and service CIDR ranges appropriately at creation time to avoid later exhaustion.
Common mistake: Assuming GKE has the exact same VPC-IP-exhaustion failure mode as EKS without checking the platform-specific networking model.

16. Why taint a GPU node pool rather than relying on node selectors alone? — Taints actively repel non-GPU workloads by default, whereas a node selector alone doesn't stop other Pods from being scheduled there if unconstrained.
17. What's the purpose of Spot VMs in a node pool? — Deeply discounted, preemptible capacity suited to fault-tolerant batch work, mixed alongside on-demand pools for latency-sensitive services.

19. What does the built-in admission policy in Autopilot actually enforce? — A curated set of allowed Pod configurations (no privileged mode, no arbitrary hostPath, etc.) applied automatically without you writing your own policies.
20. How is HPA/VPA related to Cluster Autoscaler in a GKE Standard cluster? — HPA/VPA handle Pod replica count and per-Pod resource sizing; Cluster Autoscaler handles node count — all three cooperate but are configured independently.
21. What's the GKE equivalent concept to EKS's IRSA? — Workload Identity, implemented via GCP's workload identity federation instead of AWS's OIDC/STS federation.
22. Why is granting the GCP-side IAM binding done via `gcloud`, not YAML? — The IAM policy binding lives on the GCP side (IAM service), not as a Kubernetes object — YAML only configures the Kubernetes-side ServiceAccount annotation.
23. What's a symptom of forgetting the GCP-side Workload Identity binding? — The Pod's calls to GCP APIs fail with permission/authentication errors despite the KSA annotation looking correct.
24. Why does Google recommend Stable channel specifically for production, not Regular? — Regular still adopts new versions faster than Stable; Stable maximizes the time other GKE users' clusters have already exercised that version first.
25. What happens if you ignore release-channel upgrade notifications for too long? — GKE eventually forces an automatic upgrade regardless, which is riskier to encounter unplanned than a scheduled, tested upgrade.
26. How would you add a GPU-backed node pool to an existing Standard cluster? — gcloud container node-pools create gpu-pool --accelerator type=...,count=... --node-taints=....
27. Why does a regional cluster typically cost more than a zonal one? — It runs control-plane replicas (and often node replicas) across multiple zones instead of one, trading cost for HA.
28. What's the risk of using Autopilot without understanding its Pod restrictions upfront? — Migrating an existing workload can require nontrivial rework to remove privileged/hostPath dependencies it silently relied on.
29. Why is Workload Identity described as "fully auditable and revocable via IAM alone"? — Because access can be reviewed or revoked entirely through IAM policy changes, without touching cluster-side Secrets or credentials.
30. What's the purpose of `gcloud container get-server-config`? — Shows available GKE versions and release channel status for a given region, useful for upgrade planning.
31. Why might a team deliberately keep a mixed fleet of Standard and Autopilot clusters? — Autopilot for most general workloads to reduce toil, Standard reserved specifically for workloads needing host-level access or unusual configurations.
32. What's a common mistake when sizing node pool autoscaling bounds? — Setting `max-nodes` too low, causing Pods to stay Pending under real load spikes despite autoscaling being technically enabled.
33. How do node pool taints and Pod tolerations work together for workload isolation? — The taint repels all Pods by default; only Pods with a matching toleration are allowed to schedule onto that tainted pool.
34. Why does Autopilot remove "node OS patching" as an operational burden specifically? — Because Google manages the underlying node lifecycle entirely; you never see or patch the OS layer yourself.
35. What's the tradeoff of Autopilot's flexibility restrictions in exchange for reduced toil? — You lose fine-grained control (privileged Pods, hostPath, custom node configuration) in exchange for GKE managing node-level operations for you.

43. Why is per-Pod billing in Autopilot considered more cost-transparent than per-VM billing? — Costs map directly and proportionally to what each workload actually requests, rather than to fixed VM sizes that may sit partially idle.
