StatefulSets

Stable network identity and ordered lifecycle for workloads that can't treat replicas as interchangeable.
← Deployments Course Index DaemonSets →

Explanation

A StatefulSet is a controller for workloads where each replica needs a stable, unique identity that survives rescheduling — think databases, message brokers, or anything using leader election or per-node persistent data (Kafka, Cassandra, ZooKeeper, Elasticsearch, PostgreSQL replicas). Deployments treat all Pods as identical and interchangeable, which is exactly wrong for these systems: replica "2" of a Kafka broker isn't just any broker, it's the specific broker holding a specific set of partitions on a specific volume, and it must keep coming back as "kafka-2" with the same disk every time.

Kubernetes needs a distinct mechanism for this because a Deployment/ReplicaSet's Pods get random name suffixes and random IPs, and their PersistentVolumeClaims (if any) are shared across all replicas rather than one-per-replica. Neither property works for stateful systems that need to find "the same peer" after a restart or reschedule. StatefulSets solve both problems: Pods get a predictable, stable name in the form <statefulset-name>-<ordinal> (e.g. kafka-0, kafka-1, kafka-2), and a stable DNS hostname via a required headless Service, and each replica gets its own PersistentVolumeClaim generated from a volumeClaimTemplates spec, so kafka-1 always reattaches to the same volume it had before, no matter which node it's rescheduled to.

StatefulSets also guarantee ordered, sequential Pod creation, scaling, and deletion by default: kafka-0 must be Running and Ready before kafka-1 is created, and on scale-down the highest-ordinal Pod is removed first. This matters for systems with bootstrap dependencies (e.g. a cluster's first node needs to exist and be healthy before others join) or where tearing down out of order could break quorum.

The pairing with a headless Service (a Service with clusterIP: None) is essential: instead of load-balancing across replicas behind one virtual IP, a headless Service's DNS returns one A/AAAA record per Pod, at a name like kafka-0.kafka-headless.default.svc.cluster.local. This is what lets a client — or a peer in the same cluster — address a specific, stable replica directly rather than "some randomly chosen instance."

Because ordering and identity add operational cost (rollouts are one Pod at a time by default, scaling is sequential), StatefulSets should only be used when a workload genuinely needs stable identity or per-replica storage — for pure stateless services, a Deployment remains simpler and faster to roll out.

Internal Working / Interactions

  • Headless Service — required alongside the StatefulSet; provides the per-Pod DNS names. CoreDNS answers queries for it directly from Endpoints, without a ClusterIP.
  • StatefulSet controller — watches StatefulSet objects, creates Pods one ordinal at a time, waiting for each to become Ready before creating the next (unless podManagementPolicy: Parallel).
  • PersistentVolumeClaim / CSI driver — for each ordinal, the controller instantiates a PVC from volumeClaimTemplates (e.g. data-kafka-0); the CSI driver provisions/attaches the actual volume; the PVC is NOT deleted automatically when the Pod/StatefulSet is deleted, by design.
  • API Server / etcd — as with all controllers, all state (StatefulSet spec, Pod objects, PVCs) lives in etcd via the API Server; the controller only watches/writes through it.
  • Scheduler / kubelet — bind and run each ordinal Pod exactly as any other Pod, but must respect the PVC's storage topology (e.g. a Pod may be forced onto a specific zone if its volume is zonal).

Code Example

apiVersion: v1
kind: Service
metadata:
  name: kafka-headless
spec:
  clusterIP: None                 # Headless — no load balancing, one DNS record per Pod.
  selector:
    app: kafka
  ports:
    - port: 9092
      name: broker
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: kafka
spec:
  serviceName: kafka-headless      # MUST match the headless Service above; used to build Pod DNS names.
  replicas: 3
  podManagementPolicy: OrderedReady  # Default: strictly sequential create/scale/delete.
  selector:
    matchLabels:
      app: kafka
  template:
    metadata:
      labels:
        app: kafka
    spec:
      containers:
        - name: kafka
          image: bitnami/kafka:3.6
          ports:
            - containerPort: 9092
          volumeMounts:
            - name: data
              mountPath: /bitnami/kafka   # Each replica's own persistent data directory.
  volumeClaimTemplates:               # Stamped out per-ordinal: data-kafka-0, data-kafka-1, data-kafka-2.
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 20Gi

kubectl Commands

# List StatefulSet Pods — note the stable, ordinal-suffixed names
kubectl get pods -l app=kafka -o wide

# Show current vs desired replicas and rollout status
kubectl describe statefulset kafka

# Scale a StatefulSet (ordinals added/removed in order)
kubectl scale statefulset kafka --replicas=5

# Watch the ordered rollout of a RollingUpdate on a StatefulSet
kubectl rollout status statefulset/kafka

# List PVCs — confirm one-per-ordinal, and that they survive Pod deletion
kubectl get pvc -l app=kafka

# Resolve a specific replica's stable DNS name from inside the cluster
kubectl run -it --rm dns-test --image=busybox -- nslookup kafka-1.kafka-headless.default.svc.cluster.local

# Force-delete a stuck StatefulSet Pod (use cautiously — bypasses graceful termination)
kubectl delete pod kafka-1 --grace-period=0 --force

Best Practices & Common Pitfalls

Best Practices

  • Always pair a StatefulSet with a dedicated headless Service — it will not work correctly without one.
  • Use volumeClaimTemplates rather than a single shared PVC across replicas.
  • Set Pod anti-affinity so ordinals spread across nodes/zones for real redundancy.
  • Plan PVC cleanup explicitly — deleting a StatefulSet never deletes its PVCs, which is usually what you want.

Pitfalls

  • Assuming scale-down deletes data — PVCs persist and will be reattached if you scale back up later.
  • Using podManagementPolicy: Parallel for systems that truly need ordered bootstrap (e.g. some ZooKeeper configs).
  • Forgetting that StatefulSet updates are one Pod at a time by default — a bad rollout is much slower to detect/roll back than a Deployment's.
  • Treating StatefulSets as a general-purpose "Deployment with storage" when the workload doesn't actually need identity or ordering.

50 Interview Questions

1. What core problem does a StatefulSet solve that a Deployment cannot?
Short answer: Giving each replica a stable, unique identity (name, network address, and storage) that survives rescheduling.
Detailed explanation: Deployments treat all Pods as interchangeable with random name suffixes and shared/random storage; stateful systems like Kafka or PostgreSQL need "replica 2" to always come back as the same peer with the same disk.
Production example: A Kafka broker rescheduled after a node failure must reattach as kafka-1 with the same partitions on the same volume, not a fresh random identity.
Common mistake: Trying to run a clustered database under a Deployment and being surprised peers can't find each other reliably after any restart.
2. Why is a headless Service required alongside a StatefulSet?
Short answer: It provides one DNS A/AAAA record per Pod instead of a single load-balanced virtual IP, enabling stable per-replica addressing.
Detailed explanation: A headless Service (clusterIP: None) tells CoreDNS to return individual Pod records at names like kafka-0.kafka-headless.default.svc.cluster.local rather than one shared ClusterIP that hides which specific replica you're talking to.
Production example: Kafka brokers use this DNS pattern to discover and address specific peers by ordinal during cluster bootstrap and rebalancing.
Common mistake: Creating a normal ClusterIP Service for a StatefulSet and being unable to address individual replicas by stable name.
3. How does volumeClaimTemplates differ from a single PVC referenced by a Deployment?
Short answer: It stamps out one dedicated PVC per ordinal (e.g. data-kafka-0, data-kafka-1) instead of one shared PVC across all replicas.
Detailed explanation: Each replica gets its own PersistentVolumeClaim generated from the template, and the CSI driver provisions/attaches a distinct volume per ordinal, so kafka-1 always reattaches to its own specific volume regardless of which node it lands on.
Production example: A 3-replica Elasticsearch StatefulSet ends up with three independent PVCs, each holding a different shard's data.
Common mistake: Assuming all replicas share one PVC the way they might share a ConfigMap — this would corrupt per-replica data.
4. Why does StatefulSet enforce ordered, sequential Pod creation by default?
Short answer: Many stateful systems have bootstrap dependencies — the first node must exist and be healthy before others join a cluster/quorum.
Detailed explanation: With the default podManagementPolicy: OrderedReady, ordinal N+1 is only created after ordinal N reports Ready, and scale-down removes the highest ordinal first — protecting quorum-sensitive systems from losing multiple members simultaneously.
Production example: A ZooKeeper ensemble relies on this ordering so node 0 (often the initial leader candidate) is fully up before node 1 attempts to join.
Common mistake: Setting podManagementPolicy: Parallel for a system that genuinely needs ordered bootstrap, causing join races.
5. What happens to a StatefulSet's PVCs when you scale it down or delete it?
Short answer: PVCs are never deleted automatically — they persist even after their owning Pod or the whole StatefulSet is gone.
Detailed explanation: This is a deliberate safety design: accidentally deleting a StatefulSet must not silently destroy stateful data; scaling back up later reattaches the same PVC to the recreated ordinal.
Production example: Scaling a Cassandra StatefulSet from 5 down to 3 and back up to 5 later reattaches data-cassandra-3 and data-cassandra-4 exactly as they were.
Common mistake: Assuming scale-down "cleans up" storage, then being surprised (or relieved) that old data reappears on scale-up.
6. Why should StatefulSet Pods typically use Pod anti-affinity across nodes/zones?
Short answer: Because stable identity alone doesn't guarantee redundancy — without spreading, all replicas could land on the same node and share its failure domain.
Detailed explanation: The Scheduler has no inherent knowledge that StatefulSet replicas are meant to be redundant; without explicit anti-affinity or topology spread constraints, three "redundant" Kafka brokers could all be co-located on one failing node.
Production example: A production Kafka StatefulSet adds pod anti-affinity keyed on app: kafka to force each broker onto a distinct node.
Common mistake: Assuming stable identity implies fault-tolerant placement — they are unrelated guarantees.
7. What's the operational cost of choosing a StatefulSet over a Deployment when it isn't strictly needed?
Short answer: Slower rollouts and scaling, since both default to one-Pod-at-a-time, ordered operations.
Detailed explanation: A Deployment can surge multiple new Pods and scale/rollout in parallel; a StatefulSet's default ordering makes both scaling and rolling updates strictly sequential, which is significantly slower for large replica counts.
Production example: Rolling out an image update across a 10-replica StatefulSet takes far longer than the equivalent Deployment rollout because each Pod must finish before the next starts.
Common mistake: Using a StatefulSet as a "Deployment with storage" for stateless workloads that don't actually need identity or ordering, paying the rollout-speed cost for nothing.
8. How does a StatefulSet's Pod naming scheme work, and why does it matter?
Short answer: Pods are named <statefulset-name>-<ordinal> (e.g. kafka-0, kafka-1) — predictable and stable, unlike a ReplicaSet's random suffixes.
Detailed explanation: This predictable naming is what lets other systems (config, DNS, peer-discovery logic embedded in the application itself) reference a specific replica reliably, since the name doesn't change across restarts or rescheduling.
Production example: A Kafka broker's own configuration can hardcode kafka-2.kafka-headless as a peer address, knowing it will always resolve to the same logical broker.
Common mistake: Assuming ordinal names are just cosmetic — applications often depend on them for identity-sensitive configuration (broker IDs, node IDs).
9. Why does a Pod might get forced onto a specific node/zone under a StatefulSet?
Short answer: Because its PVC may be a zonal volume, and the Scheduler must place the Pod where that volume is actually accessible.
Detailed explanation: Many CSI-backed block storage types (e.g. AWS EBS) are zone-scoped, so once a PVC is bound and provisioned in a zone, its Pod is effectively pinned to that zone on every future reschedule.
Production example: A kafka-1 Pod originally scheduled in ap-south-1a stays pinned there on every subsequent restart, because its EBS volume can't be attached from a different zone.
Common mistake: Not accounting for storage topology when planning multi-zone HA, leading to replicas that can't actually fail over across zones.
10. What is podManagementPolicy: Parallel and when is it appropriate?
Short answer: It launches/terminates all StatefulSet Pods concurrently instead of waiting for ordinal-by-ordinal readiness.
Detailed explanation: Appropriate for stateful workloads that need stable identity/storage but have no actual bootstrap-ordering dependency between replicas — you keep the identity guarantees while dropping the sequencing cost.
Production example: Some Cassandra deployments use Parallel since nodes can bootstrap concurrently once seed nodes are configured, unlike ZooKeeper's stricter sequential joining.
Common mistake: Blanket-applying Parallel to every StatefulSet for speed without checking whether the specific workload actually tolerates concurrent bootstrap.
11–50. Additional rapid-fire questions
11. What API group/version is StatefulSet under? — apps/v1.
12. Does a StatefulSet require serviceName to be set? — Yes — it must match the headless Service's name, used to construct Pod DNS.
13. What happens if you delete the headless Service but keep the StatefulSet? — Pod DNS resolution for individual replicas breaks even though the Pods keep running.
14. Can a StatefulSet's spec.selector be changed after creation? — No — immutable, same as Deployment/ReplicaSet.
15. What's the default update strategy for a StatefulSet? — RollingUpdate, applied in reverse ordinal order (highest ordinal updated first).
16. What does the partition field in RollingUpdate do? — Only ordinals >= partition are updated, letting you canary-test an update on just the highest-numbered replicas first.
17. How do you force-delete a stuck StatefulSet Pod safely? — kubectl delete pod <name> --grace-period=0 --force, used cautiously since it bypasses graceful shutdown.
18. Why is force-deleting a StatefulSet Pod risky? — The controller may believe two instances of the same ordinal identity exist briefly, risking split-brain for quorum-sensitive apps.
19. Does scaling a StatefulSet down remove the lowest or highest ordinal first? — The highest ordinal is always removed first.
20. What object actually creates the per-ordinal PVCs? — The StatefulSet controller, instantiating volumeClaimTemplates for each new ordinal.
21. Can two different StatefulSets share the same headless Service? — Not meaningfully — each StatefulSet expects its own serviceName-based DNS naming scheme to be unambiguous.
22. What field in a Pod's spec reflects its ordinal-derived identity? — Its metadata.name and hostname, both set to <statefulset>-<ordinal>.
23. Is StatefulSet rollout paused automatically on a failed Pod? — Yes for OrderedReady — the controller won't proceed to the next ordinal if the current one isn't Ready.
24. What CSI feature is essential for StatefulSets using local/zonal storage? — Topology-aware provisioning, ensuring PVCs are created in a zone the Scheduler can actually place the Pod in.
25. Does StatefulSet support HPA? — Yes, technically, though it's less common than scaling stateless Deployments due to the ordering/storage implications of adding/removing ordinals.
26. What happens to a StatefulSet's Pod hostname inside the container? — It's set to the Pod name (e.g. kafka-1), usable by the application for self-identification.
27. How do you resolve a specific replica's DNS name from another Pod? — <pod-name>.<headless-service>.<namespace>.svc.cluster.local, e.g. kafka-1.kafka-headless.default.svc.cluster.local.
28. What's a common reason a StatefulSet Pod gets stuck in Pending? — Its PVC can't be provisioned/attached, often due to zone mismatch or exhausted storage capacity.
29. Can StatefulSet Pods use ephemeral (non-persistent) storage instead? — Yes, if you skip volumeClaimTemplates and just use emptyDir — but then you lose the durable-per-replica-storage guarantee entirely.
30. Why is minReadySeconds still relevant for StatefulSets? — Same purpose as Deployments — ensures a Pod stays healthy for a minimum time before the controller proceeds to the next ordinal.
31. What's the risk of an application hardcoding assumptions about ordinal 0 being "special" (e.g. bootstrap leader)? — If ordinal 0 is ever deleted/recreated, re-electing or re-bootstrapping logic must handle that correctly, or the cluster can get stuck.
32. Does deleting a StatefulSet delete its headless Service? — No — the Service is a separate object you must manage/delete independently.
33. What's the effect of changing volumeClaimTemplates on an existing StatefulSet? — It only affects newly created ordinals going forward — existing PVCs are untouched.
34. How would you migrate a StatefulSet to larger storage per replica? — Typically via PVC resize (if the StorageClass supports expansion) rather than editing volumeClaimTemplates, since existing PVCs aren't retroactively changed.
35. What's the purpose of updateStrategy: OnDelete? — Disables automatic rolling updates entirely — Pods are only replaced with the new template when manually deleted, giving full manual control over timing.
36. Why might a team choose OnDelete for a critical database StatefulSet? — To control exactly when each replica is updated, coordinating with maintenance windows or manual failover checks.
37. Does the StatefulSet controller use owner references on Pods like a ReplicaSet does? — Yes — Pods carry ownerReferences back to the StatefulSet for garbage collection purposes.
38. What happens if you try to change serviceName after creation? — It's immutable — you'd need to recreate the StatefulSet to change it.
39. Can a StatefulSet's Pods be part of a NetworkPolicy the same way as any other Pod? — Yes — NetworkPolicy selection is purely label-based and doesn't distinguish controller type.
40. Why is it important to monitor PVC capacity for StatefulSet workloads specifically? — Databases/brokers commonly hit disk-full conditions gradually, and PVC resize/replacement is more involved than for stateless storage.
41. What's a realistic StatefulSet incident scenario? — A node hosting kafka-1's zonal volume fails permanently — the Pod can't reschedule elsewhere without manual PVC/volume intervention.
42. How does a StatefulSet differ from a Deployment in terms of Service requirement? — A Deployment works fine with just a normal ClusterIP Service; a StatefulSet specifically needs a headless one for per-Pod DNS to function.
43. What's the significance of stable network identity for leader election protocols? — Peers can reliably reference a specific, unchanging address for quorum/voting logic instead of re-discovering shifting IPs.
44. Is it possible to run a StatefulSet without any volumeClaimTemplates at all? — Yes — if the workload only needs stable naming/identity/ordering but no persistent per-replica storage.
45. What's the risk of accidentally deleting a PVC belonging to a live StatefulSet ordinal? — The next scheduling of that ordinal provisions a brand-new empty volume, silently losing that replica's data.
46. How do you safely decommission a StatefulSet replica and its data together? — Explicitly delete both the StatefulSet's replica count (scale down) and the corresponding orphaned PVC afterward.
47. Why do some teams run their own Operator instead of a bare StatefulSet for complex databases? — Operators encode domain-specific logic (backup, failover, version-aware upgrades) that a bare StatefulSet's generic ordering/identity model doesn't cover.
48. Can StatefulSet Pods be part of a Deployment's rollout at the same time? — No — they're mutually exclusive controller types; a Pod belongs to exactly one owning controller.
49. What field would show you the currently observed vs desired generation of a StatefulSet? — status.observedGeneration compared to metadata.generation.
50. What's the first thing to check when a StatefulSet is stuck creating ordinal N? — kubectl describe pod <name>-N for PVC binding failures or scheduling constraints blocking that specific ordinal.
