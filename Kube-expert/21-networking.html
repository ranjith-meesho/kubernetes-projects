Networking

The CNI model and the flat-network requirement that makes every Pod reachable from every other Pod, cluster-wide.
← CoreDNS Course Index Ingress →

Explanation

Kubernetes networking rests on one foundational rule, often called the Kubernetes networking model: every Pod gets its own IP address, and every Pod can reach every other Pod's IP directly, cluster-wide, without NAT — regardless of which node either Pod is on. This is a deliberately strong and unusual guarantee compared to plain Docker networking (where containers on different hosts can't reach each other by IP without extra port-mapping/NAT gymnastics), and it's what allows every other abstraction covered so far — Services, DNS, Endpoints — to work uniformly no matter where Pods happen to land.

Kubernetes itself does not implement this networking — it delegates entirely to the Container Network Interface (CNI), a plugin standard. The kubelet, when starting a Pod, invokes whatever CNI plugin binary is configured on that node to actually set up the Pod's network namespace, assign it an IP, and wire it into the cluster-wide network fabric. This plugin boundary is why you choose a CNI implementation (Calico, Cilium, Flannel, AWS VPC CNI, etc.) separately from choosing Kubernetes itself — the core API and scheduler have no opinion on network implementation, only on the contract every CNI plugin must satisfy.

CNI plugins fall broadly into two architectural families. Overlay networks (e.g. Flannel's VXLAN mode, Calico in IPIP mode) create a virtual network on top of the existing node network by encapsulating Pod-to-Pod packets inside node-to-node packets — this works on any underlying infrastructure (even ones you don't control, like arbitrary cloud VPCs) at the cost of some encapsulation overhead and slightly higher latency. Underlay/native-routed networks (e.g. Calico in BGP mode, AWS VPC CNI) instead assign Pods IPs directly from the real, routable network and use standard routing (BGP peers, or in AWS's case, actual ENI secondary IPs) so packets flow without encapsulation — better performance and simpler packet inspection, at the cost of needing more control over or integration with the underlying network fabric (route table capacity, IP address exhaustion in a VPC, etc.).

On top of basic connectivity, most CNI plugins (Calico and Cilium especially) also implement NetworkPolicy enforcement — the mechanism for restricting which Pods may talk to which other Pods, turning the default "everything can reach everything" model into an explicitly allow-listed one. Not every CNI plugin supports NetworkPolicy (e.g. plain Flannel does not), which is a key evaluation criterion when choosing one for a security-conscious production cluster.

Networking also has to solve node-to-Pod and Pod-to-external-internet paths, but the flat, NAT-less Pod-to-Pod guarantee is the one property every higher-level abstraction (Services, DNS, NetworkPolicy) implicitly assumes and builds on.

Internal Working / Interactions

  • kubelet — on Pod creation, calls the configured CNI plugin binary (per the /etc/cni/net.d config) to create the Pod's network namespace, assign an IP, and attach it to the cluster network.
  • CNI plugin (Calico/Cilium/Flannel/VPC CNI) — runs as its own DaemonSet-managed agent on every node; programs routes, encapsulation rules, or ENI attachments so the new Pod IP is reachable cluster-wide.
  • IPAM (IP Address Management) — a sub-component (often a separate CNI plugin chained after the main one) that allocates a Pod IP from a per-node CIDR block carved out of the cluster's overall Pod CIDR.
  • kube-proxy — layers Service virtual-IP routing on top of whatever the CNI plugin provides for raw Pod-to-Pod reachability; it does not replace the CNI, it depends on it.
  • NetworkPolicy controller (part of the CNI plugin, if supported) — watches NetworkPolicy objects and programs node-local firewall rules (iptables/eBPF) to enforce allow/deny between Pods.

Code Example

# Example: default-deny NetworkPolicy, enforced by the CNI plugin (Calico/Cilium), not core Kubernetes
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: payments
spec:
  podSelector: {}                 # Empty selector = applies to ALL Pods in this namespace.
  policyTypes:
    - Ingress                      # Only restricts inbound traffic; egress remains unaffected.
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-checkout-from-gateway
  namespace: payments
spec:
  podSelector:
    matchLabels:
      app: checkout                 # Applies to Pods labeled app=checkout.
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: api-gateway       # Only allow traffic from Pods labeled app=api-gateway...
      ports:
        - protocol: TCP
          port: 8080                 # ...and only on port 8080.

Best Practices & Common Pitfalls

Best Practices

  • Choose a CNI that supports NetworkPolicy if any workload requires network segmentation (most production clusters do).
  • Size the cluster's Pod CIDR generously up front — resizing it later usually requires cluster recreation.
  • Prefer native-routed/underlay CNI modes (e.g. Calico BGP, VPC CNI) when raw throughput/latency matters and the underlying network supports it.
  • Start with a default-deny NetworkPolicy per namespace, then explicitly allow required paths.

Pitfalls

  • Running out of IPs on cloud-native CNIs (e.g. AWS VPC CNI) that consume real VPC/subnet IPs per Pod.
  • Assuming NetworkPolicy is enforced when the installed CNI (e.g. plain Flannel) doesn't actually support it.
  • Overlapping the cluster's Pod CIDR with an existing on-prem/VPC range, breaking routing silently.
  • Not accounting for encapsulation overhead (overlay mode) when benchmarking latency-sensitive services.

50 Interview Questions

1. What is the core guarantee of the Kubernetes networking model?
Short answer: Every Pod gets its own IP, and every Pod can reach every other Pod's IP directly, cluster-wide, without NAT — regardless of which node either Pod is on.
Detail: This is a deliberately strong guarantee compared to plain Docker networking, where cross-host containers can't reach each other by IP without extra port-mapping/NAT gymnastics; it's the assumption every higher-level abstraction (Services, DNS, NetworkPolicy) builds on.
Production example: A Pod on Node 1 (10.244.1.5) can directly dial a Pod on Node 2 (10.244.2.9) with no NAT translation in between, letting Services and DNS work uniformly regardless of Pod placement.
Common mistake: Assuming Kubernetes networking behaves like Docker's default bridge networking, where cross-host container IPs aren't routable.
2. What is the CNI, and why does Kubernetes delegate networking to it instead of implementing it natively?
Short answer: CNI (Container Network Interface) is a plugin standard; Kubernetes has no built-in networking implementation, only a contract every CNI plugin must satisfy.
Detail: The kubelet invokes whatever CNI plugin binary is configured on a node whenever it starts a Pod, delegating network namespace setup, IP assignment, and cluster-wide wiring entirely to that plugin — this is what lets Calico, Cilium, Flannel, and AWS VPC CNI all be valid, interchangeable choices.
Production example: Two clusters can run identical Kubernetes versions but completely different CNI plugins (one Calico, one Cilium) with zero changes to the API Server or scheduler.
Common mistake: Assuming Kubernetes itself provides Pod networking, rather than realizing it's entirely delegated to a swappable external plugin.
3. What triggers a CNI plugin to run, and what does it actually do?
Short answer: The kubelet calls the configured CNI plugin binary on Pod creation, per the config in /etc/cni/net.d, to create the Pod's network namespace, assign it an IP, and attach it to the cluster network.
Detail: This is the "CNI ADD" operation — the plugin creates a veth pair, requests an IP from IPAM, attaches the veth to the Pod's network namespace, and programs whatever routing/encapsulation is needed for cluster-wide reachability.
Production example: The sequence diagram for a new Pod shows kubelet calling CNI ADD, IPAM assigning 10.244.1.5, and the CNI plugin programming the route before returning the IP config to kubelet.
Common mistake: Debugging a Pod stuck in ContainerCreating without checking CNI plugin logs/health, when a CNI failure is a common root cause of that state.
4. What is the difference between overlay and underlay (native-routed) CNI architectures?
Short answer: Overlay networks (e.g. Flannel VXLAN, Calico IPIP) encapsulate Pod-to-Pod packets inside node-to-node packets; underlay networks (e.g. Calico BGP, AWS VPC CNI) assign Pods real routable IPs and route natively without encapsulation.
Detail: Overlay works on any underlying infrastructure you don't control, at the cost of encapsulation overhead and slightly higher latency; underlay gives better performance and simpler packet inspection but needs tighter integration with (or control over) the underlying network fabric.
Production example: A cluster spanning multiple cloud VPCs with no BGP peering might use Flannel VXLAN overlay mode, while a single-VPC AWS cluster might use VPC CNI to assign Pods real ENI secondary IPs.
Common mistake: Choosing overlay mode by default without considering that native routing (when available) removes encapsulation overhead entirely.
5. What is IPAM's role in the CNI model?
Short answer: IPAM (IP Address Management) allocates a Pod's IP from a per-node CIDR block carved out of the cluster's overall Pod CIDR.
Detail: IPAM is often a separate plugin chained after the main CNI plugin in the CNI config, meaning the main plugin handles namespace/veth setup while IPAM specifically owns IP allocation bookkeeping per node.
Production example: kubectl get nodes -o jsonpath='{.items[*].spec.podCIDR}' shows each node's carved-out sub-range (e.g. 10.244.1.0/24) that IPAM allocates Pod IPs from.
Common mistake: Sizing the cluster's overall Pod CIDR too small up front — resizing it later usually requires cluster recreation, since per-node sub-ranges are carved out at cluster bring-up.
6. How does kube-proxy's role relate to the CNI plugin's role?
Short answer: kube-proxy layers Service virtual-IP routing on top of whatever raw Pod-to-Pod reachability the CNI plugin provides — it doesn't replace the CNI, it depends on it.
Detail: The CNI plugin solves "can Pod A reach Pod B's real IP at all"; kube-proxy solves a separate problem — "how does traffic to a Service's virtual IP get DNAT'd to one of several real Pod IPs" — and that DNAT only works because the underlying Pod-to-Pod path already exists thanks to the CNI.
Production example: Some CNIs like Cilium can run in "kube-proxy replacement" mode, implementing Service routing via eBPF instead of relying on a separate kube-proxy process — showing the two concerns are related but distinct.
Common mistake: Conflating kube-proxy and the CNI plugin as the same layer, rather than recognizing kube-proxy builds on top of CNI-provided connectivity.
7. What is NetworkPolicy, and why doesn't every CNI plugin support it?
Short answer: NetworkPolicy restricts which Pods may talk to which other Pods, turning the default "everything can reach everything" model into an explicitly allow-listed one — but enforcement is implemented by the CNI plugin, not core Kubernetes, so support varies.
Detail: Plain Flannel, for example, does not implement NetworkPolicy enforcement at all — a NetworkPolicy object would be accepted by the API Server but silently have no effect — making CNI choice a key evaluation criterion for security-conscious clusters.
Production example: A team running plain Flannel writes a default-deny NetworkPolicy expecting it to block traffic, but nothing is actually blocked because Flannel has no NetworkPolicy controller watching and enforcing those objects.
Common mistake: Assuming NetworkPolicy is enforced simply because the object was created and accepted by the API Server, without confirming the installed CNI actually supports enforcement.
8. What does an empty podSelector ({}) mean in a NetworkPolicy, and why is it used for default-deny?
Short answer: An empty podSelector: {} matches all Pods in the NetworkPolicy's namespace, so combined with policyTypes: [Ingress] and no ingress rules, it denies all inbound traffic to every Pod in that namespace by default.
Detail: This is the standard "default-deny-ingress" pattern — start by blocking everything, then layer specific allow rules (like allowing only the api-gateway to reach checkout) on top, following least-privilege network segmentation.
Production example: A payments namespace applies default-deny-ingress with an empty selector, then adds allow-checkout-from-gateway permitting only Pods labeled app: api-gateway to reach app: checkout on port 8080.
Common mistake: Writing a default-deny NetworkPolicy but forgetting to also add explicit allow rules afterward, silently breaking all legitimate traffic to that namespace.
9. Why can running out of IPs be a real production risk on cloud-native CNIs like AWS VPC CNI?
Short answer: AWS VPC CNI assigns Pods real IPs directly from VPC subnet ranges (via ENI secondary IPs) rather than from a separate overlay range, so Pod IP consumption directly competes with the VPC's finite subnet address space.
Detail: Unlike overlay CNIs where the Pod CIDR is independent of the real network, underlay/native CNIs tie Pod scaling directly to real, finite address space — a cluster that scales up heavily can exhaust available subnet IPs, blocking new Pod scheduling.
Production example: A cluster on AWS VPC CNI hits its subnet's IP capacity during a traffic spike autoscaling event, and new Pods fail to get IPs assigned even though compute capacity (nodes) is still available.
Common mistake: Sizing VPC subnets only for node count, not accounting for the much larger number of Pod IPs that a native-routed CNI will consume from the same address space.
10. Why does overlapping the cluster's Pod CIDR with an existing on-prem or VPC range cause silent routing failures?
Short answer: If the Pod CIDR overlaps a real, already-routed network range, packets meant for a Pod IP can be ambiguously routed toward the real overlapping network instead, breaking connectivity without any obvious error.
Detail: Because the CNI programs routes assuming the Pod CIDR is exclusively its own, any overlap with existing infrastructure creates routing table conflicts that are hard to diagnose since nothing rejects the configuration outright — it just doesn't work correctly for the overlapping ranges.
Production example: A cluster's default Pod CIDR (10.244.0.0/16) overlaps a company's existing on-prem VPN range, causing Pods in that overlapping subset to intermittently fail to reach on-prem systems and vice versa.
Common mistake: Accepting a CNI's default Pod CIDR without checking it against the organization's existing IP allocation plan before cluster creation.
