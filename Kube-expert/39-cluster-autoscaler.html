Cluster Autoscaler

Adding and removing whole nodes automatically based on pending pods and node utilization.
← VPA Course Index Health Probes →

Explanation

The Cluster Autoscaler (CA) is a component that automatically adjusts the number of nodes in a cluster based on scheduling pressure: it adds nodes when Pods can't be scheduled due to insufficient capacity, and removes nodes when they're significantly underutilized and their Pods could be safely moved elsewhere. Where HPA (topic 37) scales Pod count and VPA (topic 38) resizes Pod resource requests, Cluster Autoscaler operates one layer below both — it scales the actual infrastructure the Scheduler has to work with.

Kubernetes needs this because the Scheduler can only place Pods on nodes that already exist — if every node is full, new Pods simply sit Pending forever, no matter how correctly HPA or VPA sized them. Conversely, a cluster sized for peak load but running well below it wastes money on idle nodes around the clock. Cluster Autoscaler closes this last gap in the autoscaling stack, turning "add capacity" and "shed capacity" into fully automatic responses to real demand rather than a human watching dashboards and resizing node groups by hand.

Cluster Autoscaler is cloud-provider-aware by design — it doesn't manage raw VMs itself, it talks to a cloud-specific abstraction (AWS Auto Scaling Groups / EKS Managed Node Groups, GCP Managed Instance Groups, Azure VMSS) through a pluggable cloud provider interface. This is why it understands the concept of a "node group" (a set of nodes with identical capacity/labels/taints) — its core decisions are always phrased in terms of node groups, not individual nodes.

Scale-up logic: CA watches for Pods in Pending state that failed scheduling due to insufficient resources. For each node group, it simulates whether adding one more node of that group's instance type would allow the Pending Pod to schedule (respecting the Pod's actual constraints — affinity, taints/tolerations, topology spread, resource requests). If multiple node groups would work, an "expander" strategy (e.g. least-waste, most-pods, priority) picks which one to scale. Scale-down logic: CA periodically checks each node's utilization; if a node's Pods could all be rescheduled elsewhere within the same constraints, and the node has been underutilized for longer than scale-down-unneeded-time (default 10 min), CA cordons and drains it, then removes the underlying instance from the node group.

In production, Cluster Autoscaler and HPA are almost always deployed together: HPA reacts to load by creating more Pods; if the cluster lacks room, those Pods go Pending; CA reacts to Pending Pods by adding nodes; the Scheduler then places the Pods on the new capacity. The chain has real latency at every step (cloud instance boot time is often the dominant factor, sometimes 1-3 minutes), which is why over-provisioning a small buffer of headroom (or using low-priority placeholder Pods) is a common production pattern to smooth over this scale-up lag.

Cluster Autoscaler vs Karpenter: Karpenter (increasingly popular on AWS) is a newer alternative that provisions individual nodes directly rather than through fixed node groups, offering faster and more flexible bin-packing decisions. Cluster Autoscaler remains the CNCF-standard, cloud-agnostic option and is what most managed Kubernetes documentation and certification exams (CKA) reference.

Internal Working / Interactions

  • Scheduler: CA reads the Scheduler's simulated placement outcome (via its own internal scheduling simulation library, which mirrors the real Scheduler's filtering logic) to decide if a new node would actually unblock a Pending Pod.
  • API Server: CA watches Pods (for Pending ones) and Nodes (for utilization) via the standard watch/informer pattern, and issues cordon/drain-equivalent operations through the API Server before removing a node.
  • Cloud provider API (ASG/MIG/VMSS): the actual mechanism for adding/removing capacity — CA calls the cloud provider's scaling API, then waits for the new node to register with the API Server and become Ready.
  • PodDisruptionBudget: respected during scale-down — CA will not drain a node if doing so would violate a PDB for the Pods running on it.
  • HPA: upstream trigger in the common chain — HPA creates Pending Pods that CA then reacts to; the two systems don't talk to each other directly, they're chained via the shared Pod/Node state in the API Server.

Code Example

# Cluster Autoscaler is typically deployed as a Deployment in kube-system,
# configured via command-line flags/args rather than a CRD (on most cloud providers).
apiVersion: apps/v1
kind: Deployment
metadata:
  name: cluster-autoscaler
  namespace: kube-system
spec:
  replicas: 1
  selector:
    matchLabels: { app: cluster-autoscaler }
  template:
    metadata:
      labels: { app: cluster-autoscaler }
    spec:
      serviceAccountName: cluster-autoscaler
      containers:
        - name: cluster-autoscaler
          image: registry.k8s.io/autoscaling/cluster-autoscaler:v1.29.0
          command:
            - ./cluster-autoscaler
            - --cloud-provider=aws
            - --namespace=kube-system
            - --scale-down-enabled=true
            - --scale-down-unneeded-time=10m       # how long a node must be underused before removal
            - --scale-down-utilization-threshold=0.5   # below 50% utilization counts as "underused"
            - --skip-nodes-with-local-storage=false
            - --expander=least-waste                # scale-up strategy when multiple node groups qualify
            - --nodes=2:10:eks-node-group-general    # min:max:name - CA-managed node group bounds
            - --nodes=0:5:eks-node-group-gpu

# PodDisruptionBudget that Cluster Autoscaler respects during scale-down drains
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: checkout-pdb
  namespace: payments
spec:
  minAvailable: 2          # CA will not drain a node if it would drop below 2 available replicas
  selector:
    matchLabels:
      app: checkout

kubectl Commands

# Check Cluster Autoscaler pod health and logs
kubectl get pods -n kube-system -l app=cluster-autoscaler
kubectl logs -n kube-system deployment/cluster-autoscaler --tail=100

# See CA's own status/decisions (writes a status configmap)
kubectl describe configmap cluster-autoscaler-status -n kube-system

# Find pods currently Pending due to insufficient capacity
kubectl get pods -A --field-selector=status.phase=Pending

# See why a pod couldn't be scheduled (the trigger for CA scale-up)
kubectl describe pod  | grep -A5 Events

# List nodes and their current utilization
kubectl top nodes

# Manually cordon a node to test scale-down draining behavior
kubectl cordon

Best Practices & Common Pitfalls

Do's

  • Set realistic min/max bounds per node group to avoid runaway costs or under-provisioning.
  • Set PodDisruptionBudgets on every critical workload so scale-down never drops availability below your threshold.
  • Combine with HPA and correctly-set resource requests — CA's simulation is only as accurate as the requests it reads.
  • Use node group priority/expander strategy deliberately (e.g. prefer spot/cheaper groups first) for cost optimization.

Don'ts

  • Don't run workloads with local storage or no PDB on nodes you expect CA to freely scale down — they'll block or bypass drains unexpectedly.
  • Don't set scale-down-unneeded-time too low for spiky workloads — you'll get node churn (scale down, immediately scale back up).
  • Don't forget CA cannot help a Pod that's Pending due to affinity/taint mismatches with every node group — it only adds capacity of types that already exist as node groups.
  • Don't rely on CA for sub-minute scaling response — cloud instance boot time is a hard physical floor on scale-up latency.

50 Interview Questions

1. What does Cluster Autoscaler change, and how does that differ from HPA and VPA?
Short answer: It adjusts the number of nodes in the cluster, one layer below HPA (Pod count) and VPA (Pod resource sizing) — it scales the actual infrastructure the Scheduler works with.
Detail: HPA and VPA can only resize what already fits on existing nodes; if every node is full, no amount of correct Pod sizing helps — CA is what closes that gap.
Production example: A load spike triggers HPA to request 10 more replicas; 4 go Pending until CA adds 2 nodes roughly 90 seconds later.
Common mistake: Confusing CA with HPA/VPA's scope — CA never touches replica counts or resource requests, only node count.
2. Why does CA need a "node group" abstraction instead of managing individual VMs?
Short answer: CA is cloud-provider-aware by design and talks through a pluggable cloud provider interface (ASG, MIG, VMSS) — its decisions are always phrased in terms of a group of nodes with identical capacity/labels/taints, not individual instances.
Detail: This is why CA's core scale-up decision is "would adding one more node of this node group's instance type unblock the Pending Pod," rather than reasoning about arbitrary individual machines.
Production example: A configuration defines --nodes=2:10:eks-node-group-general and --nodes=0:5:eks-node-group-gpu as two separately-bounded node groups.
Common mistake: Assuming CA can provision an arbitrary new instance type on demand — it can only scale within node groups that already exist.
3. Walk through CA's scale-up logic step by step.
Short answer: CA watches for Pods stuck Pending due to insufficient resources, then simulates whether adding one more node of each node group's instance type would let that Pod schedule, respecting its actual constraints.
Detail: The simulation considers affinity, taints/tolerations, topology spread, and resource requests — the same constraints the real Scheduler enforces. If multiple node groups would work, an expander strategy picks which one.
Production example: A Pending Pod requiring a GPU triggers CA to simulate against eks-node-group-gpu specifically, since eks-node-group-general's instance type can't satisfy the constraint.
Common mistake: Assuming CA reacts to raw cluster utilization for scale-up — it only reacts to actual scheduling failures (Pending Pods), not general busyness.
4. Walk through CA's scale-down logic step by step.
Short answer: CA periodically checks each node's utilization; if a node has been underutilized longer than scale-down-unneeded-time (default 10 min) and all its Pods could be rescheduled elsewhere, CA cordons and drains it, then removes the instance from the node group.
Detail: "Could be rescheduled elsewhere" is itself a constraint-aware simulation — the same kind used for scale-up, just checking whether removing capacity is safe rather than whether adding it helps.
Production example: --scale-down-utilization-threshold=0.5 means a node below 50% utilization for over 10 minutes becomes a scale-down candidate.
Common mistake: Assuming CA scales down purely on a timer — it requires both sustained low utilization AND confirmation the node's Pods can be safely relocated.
5. What is an "expander" strategy and why does it matter?
Short answer: When multiple node groups could satisfy a Pending Pod's scheduling requirements, the expander strategy (e.g. least-waste, most-pods, priority) decides which node group CA actually scales up.
Detail: least-waste picks the option that leaves the least unused capacity after placement; priority lets teams deliberately prefer cheaper/spot node groups first for cost optimization.
Production example: The Deployment config sets --expander=least-waste as CA's scale-up strategy when multiple node groups qualify.
Common mistake: Assuming CA always picks the cheapest option by default — the expander strategy must be deliberately configured for cost-driven behavior.
6. Why do HPA and Cluster Autoscaler get deployed together in production, and how are they chained?
Short answer: HPA reacts to load by creating more Pods; if the cluster lacks room those Pods go Pending, which is the signal CA reacts to by adding nodes — they don't talk to each other directly, they're chained via shared Pod/Node state in the API Server.
Detail: This chain has real latency at every step, with cloud instance boot time often the dominant factor (1-3 minutes) — motivating headroom buffers or low-priority placeholder Pods in production.
Production example: HPA scaling checkout-service to 10 replicas leaves several Pending until CA provisions and boots new nodes roughly 90 seconds later.
Common mistake: Assuming HPA-triggered scaling is instantaneous end-to-end when the cluster lacks headroom — the chained CA response adds real, sometimes multi-minute, latency.
7. What role does the Scheduler's simulation logic play inside Cluster Autoscaler?
Short answer: CA maintains its own internal scheduling simulation library that mirrors the real Scheduler's filtering logic, used to decide whether a new node would actually unblock a specific Pending Pod.
Detail: This is a simulation, not a live call to the real Scheduler — CA needs to answer "would this pod fit" for hypothetical future nodes that don't exist yet.
Production example: Before scaling up eks-node-group-b, CA simulates whether that group's instance type and labels satisfy the Pending Pod's affinity and resource constraints.
Common mistake: Assuming CA asks the live Scheduler to test-place a Pod — it uses its own internal simulation, separate from the real scheduling pipeline.
8. Why does CA respect PodDisruptionBudgets during scale-down?
Short answer: CA will not drain a node if doing so would violate a PDB for the Pods running on it, preventing scale-down from silently breaching availability guarantees.
Detail: This mirrors VPA's Updater behavior (topic 38) — both autoscalers treat PDBs as a hard constraint before evicting/draining, not just a suggestion.
Production example: checkout-pdb sets minAvailable: 2, so CA will not drain a node if doing so would drop available checkout replicas below 2.
Common mistake: Running workloads with no PDB and assuming CA will "just be careful" — without a PDB, CA has no explicit availability floor to respect.
9. Why can't CA help a Pod that's Pending due to affinity or taint mismatches?
Short answer: CA only adds capacity of instance types that already exist as configured node groups — if no node group's labels/taints could ever satisfy the Pod's constraints, adding more nodes of any existing type won't help.
Detail: This is a direct consequence of CA's scale-up simulation — it explicitly checks whether a new node of a given node group would unblock the Pod; if none can, CA correctly does nothing.
Production example: A Pod requiring a node label that no configured node group has stays Pending indefinitely, regardless of how much CA scales other groups.
Common mistake: Assuming CA will eventually resolve any Pending Pod given enough time — it cannot fix scheduling constraints that no existing node group can satisfy.
10. How would you troubleshoot a Pod stuck Pending when Cluster Autoscaler should have added capacity?
Short answer: Check the Pod's scheduling failure event for the real constraint, check CA's own logs/status configmap for its decision, and check node group min/max bounds aren't already exhausted.
Detail: kubectl describe pod reveals why the Scheduler rejected placement (the same signal CA reacts to); cluster-autoscaler-status configmap and CA's logs reveal whether it even attempted a scale-up and why it may have declined.
Production example: A Pod stayed Pending because its node group was already at its configured max (--nodes=2:10:...), a bound CA respects even when scale-up would otherwise help.
Common mistake: Assuming a Pending Pod always means CA is broken, rather than checking whether node group bounds or unsatisfiable constraints are the actual cause.
