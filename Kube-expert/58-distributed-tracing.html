Distributed Tracing

Following a single request as it travels across dozens of microservices, using traces, spans, and context propagation.
← Loki Course Index Troubleshooting →

Explanation

Distributed tracing solves a problem that metrics and logs individually cannot: understanding the end-to-end journey and latency breakdown of a single request as it hops across many microservices. In a monolith, a slow request means one stack trace. In a Kubernetes-based microservices architecture, a single user-facing API call might fan out through an API gateway, an auth service, a catalog service, a pricing service, and a database — each running in its own Pod, each with its own logs and metrics. Without tracing, answering "why was this one request slow" means manually correlating timestamps across a dozen separate log streams — tedious and often impossible once concurrency is involved.

The core unit is a span: a single named, timed operation (e.g. "call pricing-service", "query Postgres") with a start time, duration, and metadata (tags, status). A trace is the entire tree of spans generated by one logical request, linked by a shared trace ID. Each span also carries a parent span ID, letting tools reconstruct the exact call graph and see, at a glance, which downstream call in the chain consumed most of the total latency.

The mechanism that makes this possible across process/network boundaries is context propagation: when service A calls service B over HTTP or gRPC, it injects the current trace ID and span ID into request headers (commonly the W3C traceparent header). Service B's instrumentation reads that header, creates a new child span linked to the same trace, and propagates it onward to whatever it calls next. This is why tracing requires either manual code changes or auto-instrumentation in every service in the call path — a single uninstrumented hop breaks the chain and produces a trace with a mysterious gap.

Sampling exists because tracing every single request at scale is prohibitively expensive to store and mostly unnecessary — most requests are fast and uninteresting. Head-based sampling decides at the start of a request (e.g. "trace 1% of all requests") which is cheap but can miss rare slow/erroring requests entirely. Tail-based sampling (done at the OpenTelemetry Collector, see Topic 54) buffers all spans for a trace and decides after the fact whether to keep it — commonly configured to always keep traces with errors or latency above a threshold, discarding the uninteresting majority. This gives much higher signal-to-noise at a fraction of head-based sampling's storage cost.

In the ecosystem, tracing is generated by OpenTelemetry SDKs (or older Jaeger/Zipkin clients), transported via the OTel Collector, and stored/queried in a tracing backend like Tempo or Jaeger. Grafana visualizes trace waterfalls and, critically, links traces to correlated logs (via trace ID in log lines, see Loki topic) and to the metrics spike that prompted the investigation in the first place — this three-way correlation is the practical payoff of the whole observability stack.

Internal Working / Interactions

  • Application SDK creates spans around instrumented operations (HTTP handlers, DB calls, outbound requests) and injects/extracts trace context via headers on every network call.
  • Context propagation — W3C Trace Context (traceparent header) or legacy formats (B3, Jaeger) carry trace ID/span ID across service boundaries; a service mesh (Istio/Linkerd) can also propagate/generate spans at the proxy level.
  • OTel Collector receives spans via OTLP, applies tail-based sampling, and exports to the tracing backend.
  • Tracing backend (Tempo/Jaeger) stores spans, indexed primarily by trace ID (Tempo, like Loki, avoids expensive full inverted indexing and relies on object storage + trace ID lookups).
  • Grafana queries the backend (TraceQL for Tempo) to render trace waterfalls, and cross-links to Loki (logs) and Prometheus/VictoriaMetrics (metrics) via shared trace/pod identifiers.

Code Example

# OTel Collector tail-sampling processor config — keeps traces with errors or high latency,
# discards the rest, dramatically reducing storage while preserving the traces that matter.
processors:
  tail_sampling:
    decision_wait: 10s                 # how long to buffer spans for a trace before deciding
    policies:
      - name: errors-policy
        type: status_code
        status_code: { status_codes: ["ERROR"] }   # always keep traces containing an error span
      - name: slow-requests-policy
        type: latency
        latency: { threshold_ms: 500 }              # always keep traces slower than 500ms
      - name: baseline-sample-policy
        type: probabilistic
        probabilistic: { sampling_percentage: 5 }    # keep 5% of everything else as a baseline sample

exporters:
  otlp/tempo:
    endpoint: "tempo.monitoring.svc:4317"
    tls: { insecure: true }

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [tail_sampling, batch]
      exporters: [otlp/tempo]
---
# Example: Node.js Express handler manually creating a child span for a DB call,
# on top of auto-instrumentation handling the inbound HTTP request span automatically.
const { trace } = require('@opentelemetry/api');
const tracer = trace.getTracer('catalog-service');

app.get('/products/:id', async (req, res) => {
  // Auto-instrumentation already started a span for this HTTP request and
  // propagated the parent trace context from the incoming traceparent header.
  const span = tracer.startSpan('db.query.products');   // manual child span for a specific operation
  try {
    const product = await db.query('SELECT * FROM products WHERE id = $1', [req.params.id]);
    span.setAttribute('db.rows_returned', product.rows.length);
    res.json(product.rows[0]);
  } catch (err) {
    span.recordException(err);       // marks this span (and the trace) as an error for tail sampling
    span.setStatus({ code: 2 });      // SpanStatusCode.ERROR
    res.status(500).send('error');
  } finally {
    span.end();                      // records duration; kept or dropped based on Collector's sampling policy
  }
});

Best Practices & Common Pitfalls

Best Practices

  • Use auto-instrumentation wherever available before writing manual spans — far less code, consistent conventions.
  • Always propagate trace context through async boundaries (queues, background jobs), not just synchronous HTTP calls.
  • Use tail-based sampling to keep 100% of error/slow traces while discarding routine traffic.
  • Tag spans with business-relevant attributes (order ID, user tier) to make traces searchable beyond just latency.
  • Ensure every service in a request path is instrumented — one gap breaks the whole trace's continuity.

Pitfalls

  • Instrumenting only some services, producing traces with confusing unexplained gaps.
  • Using head-based sampling only, missing the rare slow/erroring requests that matter most during incidents.
  • Not propagating context across message queues (Kafka/SQS), silently breaking traces for async workflows.
  • Storing 100% of traces without sampling, causing runaway tracing backend storage costs.

50 Interview Questions

1. What is a span in distributed tracing?
Short answer: A single named, timed operation with a start time, duration, and metadata (tags, status).
Detail: A span represents one unit of work, such as "call pricing-service" or "query Postgres", and is the atomic building block that traces are composed of.
Production example: A catalog-service handler manually creates a db.query.products span around a Postgres call, nested under the auto-instrumented HTTP request span.
Common mistake: Confusing a span with a log line — a span always has a duration and parent/child relationship, a log line does not.
2. What is a trace and how is it related to a span?
Short answer: A trace is the entire tree of spans generated by one logical request, linked by a shared trace ID.
Detail: Every span in the tree also carries a parent span ID, which lets tooling reconstruct the exact call graph and see which downstream call consumed the most latency.
Production example: A single user-facing API call fanning out through an API gateway, auth service, catalog service, and database produces one trace containing four (or more) linked spans.
Common mistake: Treating "trace" and "span" as interchangeable terms in conversation, which causes confusion when discussing sampling (sampling decisions are made per-trace, not per-span).
3. What problem does distributed tracing solve that metrics and logs individually cannot?
Short answer: Understanding the end-to-end journey and latency breakdown of a single request across many microservices.
Detail: In a monolith a slow request means one stack trace; in a microservices architecture, each hop runs in its own Pod with its own logs and metrics, so correlating timestamps manually across a dozen log streams is tedious and often impossible under concurrency.
Production example: A request fanning through an API gateway, auth service, catalog service, and pricing service before hitting a database — tracing shows exactly which hop was slow.
Common mistake: Assuming metrics dashboards alone can tell you which specific downstream call caused a particular slow request — metrics show aggregates, not individual request paths.
4. What mechanism makes trace context propagation possible across process/network boundaries?
Short answer: Context propagation via request headers, commonly the W3C traceparent header.
Detail: When service A calls service B over HTTP or gRPC, it injects the current trace ID and span ID into the request headers; service B's instrumentation reads that header, creates a new child span linked to the same trace, and propagates it onward.
Production example: The Node.js Express handler in the code example relies on auto-instrumentation to extract the incoming traceparent header before starting a manual child span for a DB call.
Common mistake: Assuming context propagates automatically without any code involvement — every hop must actively read and forward the header, or the chain breaks.
5. Why does a single uninstrumented service in a request path break tracing?
Short answer: Because tracing requires manual code changes or auto-instrumentation in every service in the call path to propagate context.
Detail: If a service doesn't read and forward the trace headers, the next hop starts a brand-new, disconnected trace instead of continuing the existing one, producing a trace with a mysterious gap.
Production example: A legacy service in the middle of a call chain that hasn't been instrumented yet causes the trace waterfall in Grafana to show a hole where that service's spans should be.
Common mistake: Assuming a partial trace with a gap means "no requests went through that service" rather than "that service wasn't instrumented."
6. What is the difference between head-based and tail-based sampling?
Short answer: Head-based sampling decides at the start of a request whether to trace it; tail-based sampling buffers all spans for a trace and decides after the fact.
Detail: Head-based sampling (e.g. "trace 1% of all requests") is cheap but can miss rare slow/erroring requests entirely, since the decision is made before anything is known about the outcome; tail-based sampling, done at the OTel Collector, can be configured to always keep traces with errors or high latency while discarding routine traffic.
Production example: The tail-sampling processor config in the code example keeps all traces with an ERROR status span and all traces slower than 500ms, while only keeping a 5% baseline sample of everything else.
Common mistake: Using head-based sampling only, which misses the rare slow/erroring requests that matter most during an incident.
7. Why does tracing require sampling at all — why not trace every request?
Short answer: Tracing every single request at scale is prohibitively expensive to store and mostly unnecessary since most requests are fast and uninteresting.
Detail: Sampling trades completeness for cost, and tail-based sampling in particular gives much higher signal-to-noise per byte stored by discarding the uninteresting majority after the outcome is known.
Production example: Storing 100% of traces without any sampling policy is listed as a pitfall because it causes runaway tracing backend storage costs.
Common mistake: Assuming sampling is purely a cost-cutting measure with no benefit — properly configured tail-based sampling actually improves the usefulness of what's stored.
8. What role does the OTel Collector play in the tracing pipeline?
Short answer: It receives spans via OTLP, applies tail-based sampling, and exports the surviving traces to the tracing backend.
Detail: Because tail-based sampling requires buffering all spans for a trace before deciding, the Collector — not the application SDK — is the component responsible for making and enforcing that sampling decision.
Production example: The pipeline config routes the otlp receiver through tail_sampling and batch processors before exporting to otlp/tempo.
Common mistake: Assuming applications themselves decide which traces to keep under tail-based sampling — the decision is centralized in the Collector, which is why it must buffer spans in memory.
9. Why does Tempo avoid full inverted indexing the way a log backend might?
Short answer: Because spans are stored and queried primarily by trace ID, so Tempo relies on object storage plus trace ID lookups rather than expensive full indexing.
Detail: This mirrors the design philosophy of Loki — both avoid full-text/full-field indexing to keep storage costs and operational complexity low, trading off some query flexibility for scale and cost efficiency.
Production example: Fetching a specific trace via curl "http://localhost:3200/api/traces/<trace_id>" is a fast, direct lookup rather than a full-index search.
Common mistake: Expecting Tempo to support arbitrary full-text search across span attributes as cheaply as a trace-ID lookup — TraceQL queries are more expensive than direct ID fetches.
10. How does Grafana tie tracing together with the rest of the observability stack?
Short answer: Grafana queries the tracing backend to render trace waterfalls and cross-links traces to correlated logs (via trace ID in log lines) and to the metrics spike that prompted the investigation.
Detail: This three-way correlation between traces, logs (Loki), and metrics (Prometheus/VictoriaMetrics) via shared trace/pod identifiers is described as the practical payoff of the whole observability stack.
Production example: An engineer sees a latency spike in a metrics dashboard, pivots to a trace waterfall to find the slow span, then jumps to the correlated log lines for that exact trace ID to read the application's own error message.
Common mistake: Treating metrics, logs, and traces as three separate tools investigated in isolation instead of using the shared trace ID to pivot between them during an incident.
11–50. Additional rapid-fire questions
11. What does the traceparent header carry? — The current trace ID and span ID (plus flags), injected by the calling service and read by the callee to create a linked child span.
12. What are B3 and Jaeger propagation formats? — Legacy trace-context header formats that predate the W3C Trace Context standard, still supported for compatibility.
13. Can a service mesh generate spans without application code changes? — Yes — a mesh like Istio or Linkerd can propagate and generate spans at the proxy level, independent of in-app instrumentation.
14. What is auto-instrumentation? — SDK or agent tooling that automatically creates spans for common operations (HTTP handlers, DB calls) without manual code changes.
15. Why prefer auto-instrumentation over manual spans where available? — Far less code to write and maintain, and it produces consistent span-naming conventions across services.
16. What is the OTel Operator's Instrumentation custom resource for? — Configures automatic injection of OpenTelemetry auto-instrumentation into Pods, checkable via kubectl get instrumentation.
17. What does decision_wait control in a tail-sampling processor? — How long the Collector buffers spans for a trace before making its keep/drop sampling decision.
18. What does the status_code tail-sampling policy do? — Always keeps traces containing a span marked with an ERROR status.
19. What does the latency tail-sampling policy do? — Always keeps traces whose total duration exceeds a configured threshold (e.g. 500ms).
20. What does a probabilistic tail-sampling policy do in combination with the others? — Keeps a small baseline percentage (e.g. 5%) of otherwise-uninteresting traces for general visibility.
21. How does a span get marked so tail-based sampling will keep it? — Calling span.recordException(err) and span.setStatus({code: 2}) (ERROR) inside application code when a failure occurs.
22. What does span.end() do? — Records the span's duration, finalizing it so it can be kept or dropped by the Collector's sampling policy.
23. Why must async boundaries like message queues explicitly propagate trace context? — Because there's no synchronous HTTP call carrying headers automatically — context has to be manually injected into the message and extracted by the consumer.
24. What happens if context propagation is skipped across a Kafka/SQS hop? — The trace silently breaks for that async workflow, producing disconnected traces instead of one continuous one.
25. What transport protocol do OTel SDKs typically use to send spans to the Collector? — OTLP (OpenTelemetry Protocol).
26. Where does the OTel Collector export sampled traces in the example config? — To Tempo, via the otlp/tempo exporter targeting tempo.monitoring.svc:4317.
27. What query language does Grafana use against Tempo? — TraceQL.
28. What is a "trace waterfall"? — The visual rendering of a trace's spans as nested, time-ordered bars showing which operation took how long relative to its parent.
29. Why is tagging spans with business-relevant attributes (order ID, user tier) a best practice? — It makes traces searchable beyond just latency, letting you find traces relevant to a specific customer or business scenario.
30. What is the risk of instrumenting only some services in a call path? — Traces end up with confusing unexplained gaps at the uninstrumented hop.
31. How would you confirm trace headers are actually propagating in production? — kubectl describe pod <pod> | grep -A3 traceparent or checking app logs for the header value.

35. Why should you monitor memory usage on the OTel Collector specifically for tracing? — Tail sampling buffers all spans for a trace in memory during the decision_wait period, so a spike in trace volume can spike Collector memory.
36. What's the danger of storing 100% of traces with no sampling policy at all? — Runaway tracing backend storage costs, since most traces are uninteresting routine traffic.
37. What is the relationship between a parent span ID and the call graph? — Each span's parent span ID lets tooling reconstruct exactly which operation called which, forming the trace's tree structure.
38. Why is a single trace ID shared across every span in a request? — It's the identifier that groups all spans from unrelated services back into one logical trace for reconstruction and lookup.
39. What's an example of a span attribute set for diagnostic value in the code example? — span.setAttribute('db.rows_returned', product.rows.length), recording how many rows a query returned.
40. Why is context propagation described as requiring "either manual code changes or auto-instrumentation" rather than being automatic at the network layer? — Because trace context lives in request headers that application-layer code or an instrumentation library must explicitly read and write — the network itself doesn't understand tracing.
41. What's the tradeoff of tail-based over head-based sampling in terms of infrastructure cost? — Tail-based needs a stateful buffering component (the Collector holding spans per trace), which head-based sampling avoids entirely.
42. How does tracing complement metrics during an incident, per this topic? — Metrics tell you when and roughly where a problem is; a trace tells you exactly which downstream call in a specific request was slow or failing.
43. How does tracing complement logs during an incident? — Once you find the slow/erroring span in a trace, its trace ID lets you pull the exact correlated log lines from that request across every service it touched.
44. What is the significance of the trace ID appearing in log lines? — It's the shared key that lets Grafana cross-link a trace to the exact log output produced during that same request.
45. Why might a request that "looks fine" in aggregate metrics still need tracing to diagnose? — Metrics are aggregates; a rare individual slow request (which tail-based sampling is designed to always capture) won't move the aggregate but is exactly what tracing preserves.
46. What does the sequence diagram show happens after each service finishes handling its part of the request? — Each service (gateway, auth, catalog) independently exports its spans to the Collector via OTLP, which then makes a tail-sampling decision before forwarding to Tempo.
47. Is the sampling decision made once per span or once per trace? — Once per trace — tail-based sampling buffers and evaluates all spans belonging to a trace together before deciding to keep or drop the whole thing.
48. What would you check first if a known-slow request doesn't appear in Tempo at all? — Whether the sampling policy (head-based percentage, or a tail-sampling policy misconfiguration) discarded it, or whether context propagation broke somewhere in the call path.
49. Why is the OTel Collector placed between application SDKs and the tracing backend rather than exporting directly? — It centralizes sampling, batching, and backend routing so applications don't need backend-specific export logic or per-app sampling logic.
50. What's the first thing to check if a trace shows a gap between two known-instrumented services? — Whether context propagation is actually wired through that specific hop (headers not injected/extracted, or an async boundary like a queue in between without manual context passing).
