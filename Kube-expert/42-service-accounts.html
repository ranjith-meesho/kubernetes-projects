Service Accounts

The identity a Pod presents to the API Server — and the foundation everything in RBAC is checked against.
← RBAC Course Index Network Policies →

Explanation

A ServiceAccount is a Kubernetes API object that gives a process running inside a Pod an identity it can use to authenticate to the API Server. This is distinct from a User account (a human, authenticated via cert or SSO token) — Kubernetes has no built-in concept of human users as API objects, but it does have a first-class object for machine/workload identity: the ServiceAccount. Every namespace automatically gets a default ServiceAccount, and every Pod that doesn't explicitly request one is assigned default at admission time.

Kubernetes needs this because workloads constantly need to talk back to the API Server: a controller watching resources, a CI job creating Deployments, a sidecar reading Secrets, a service mesh control plane reading Endpoints. Without a first-class identity primitive for these in-cluster clients, you'd be stuck minting human-style credentials for every piece of automation — unmanageable at scale and impossible to scope tightly. ServiceAccounts let you bind a narrowly-scoped identity to exactly the Pods that need it, and RBAC then grants that identity only the verbs/resources it actually requires.

Mechanically, when a Pod is admitted, the API Server (via the ServiceAccount admission controller) attaches a projected, time-bound, audience-bound token for that ServiceAccount as a volume mounted at /var/run/secrets/kubernetes.io/serviceaccount. Modern Kubernetes (1.24+) uses the TokenRequest API to issue short-lived JWTs (default ~1 hour, auto-refreshed by kubelet) instead of the old model of a long-lived, non-expiring token stored in a Secret — a major security improvement, since leaked long-lived tokens used to be a standing liability.

ServiceAccounts fit into the architecture as the subject half of every RBAC decision made for in-cluster clients: a RoleBinding or ClusterRoleBinding names a ServiceAccount as its subject, and the API Server's authorization layer checks the incoming request's ServiceAccount identity (embedded in the JWT) against the bound Roles. No ServiceAccount, no scoped access — this is why "least privilege" for workloads always starts with "does this Pod even need its own ServiceAccount, and does it need API access at all."

The most common production mistake is leaving automountServiceAccountToken: true (the default) on every Pod, even ones that never call the Kubernetes API. That token, sitting on disk inside the container, is a live credential an attacker who achieves code execution in that Pod can steal and use to enumerate or modify cluster resources — the actual blast radius of "container escape" is often "what could this ServiceAccount's token do."

Internal Working / Interactions

  • API Server (admission): the ServiceAccount admission controller assigns default if none is set, and requests a projected token from the TokenRequest API for the mount.
  • kubelet: mounts the projected token volume into the Pod's filesystem and is responsible for periodically refreshing it before expiry.
  • API Server (authn): on every incoming request from inside the cluster, validates the JWT signature and audience, extracting the ServiceAccount identity (system:serviceaccount:<namespace>:<name>).
  • API Server (authz/RBAC): checks that identity against RoleBindings/ClusterRoleBindings to decide allow/deny for the requested verb+resource.
  • Controller Manager (tokens controller): in legacy mode, watched ServiceAccounts and auto-created long-lived Secret-based tokens; largely superseded by TokenRequest-based projected tokens.

Code Example

# 1. A dedicated, minimally-scoped ServiceAccount for a payments reconciler
apiVersion: v1
kind: ServiceAccount
metadata:
  name: payments-reconciler-sa
  namespace: payments
automountServiceAccountToken: false   # Don't auto-mount by default; opt in per-Pod where actually needed.
---
# 2. Role granting only what this workload needs (read-only on its own ConfigMaps)
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: payments-reconciler-role
  namespace: payments
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: payments-reconciler-binding
  namespace: payments
subjects:
  - kind: ServiceAccount
    name: payments-reconciler-sa
    namespace: payments
roleRef:
  kind: Role
  name: payments-reconciler-role
  apiGroup: rbac.authorization.k8s.io
---
# 3. Pod that explicitly opts back INTO the token, using a scoped audience + short TTL
apiVersion: v1
kind: Pod
metadata:
  name: reconciler
  namespace: payments
spec:
  serviceAccountName: payments-reconciler-sa
  containers:
    - name: reconciler
      image: registry.internal/payments-reconciler:1.3.0
      volumeMounts:
        - name: sa-token
          mountPath: /var/run/secrets/tokens
          readOnly: true
  volumes:
    - name: sa-token
      projected:
        sources:
          - serviceAccountToken:
              path: token
              expirationSeconds: 3600   # Short-lived, auto-refreshed by kubelet before expiry.
              audience: api             # Bound to a specific audience, cannot be replayed elsewhere.

kubectl Commands

kubectl create serviceaccount payments-sa -n payments        # Create a ServiceAccount
kubectl get serviceaccounts -n payments                        # List ServiceAccounts in a namespace
kubectl describe sa payments-sa -n payments                    # Show associated secrets/annotations
kubectl create token payments-sa -n payments --duration=1h      # Mint a short-lived token manually (TokenRequest API)
kubectl get pod my-pod -o jsonpath='{.spec.serviceAccountName}'  # Check which SA a running Pod uses
kubectl auth can-i list pods --as=system:serviceaccount:payments:payments-sa -n payments  # Test effective RBAC
kubectl get rolebinding,clusterrolebinding -A -o wide | grep payments-sa   # Find every binding referencing this SA

Best Practices & Common Pitfalls

Best Practices

  • Create one dedicated ServiceAccount per workload/microservice — never share a single SA across unrelated apps.
  • Set automountServiceAccountToken: false by default; mount projected tokens explicitly only where the app truly calls the API Server.
  • Use short expirationSeconds and a specific audience on projected tokens for anything beyond default in-cluster calls.
  • Pair every ServiceAccount with a least-privilege Role, never a broad ClusterRole "just in case."

Pitfalls

  • Leaving every Pod on the default ServiceAccount with implicit broad bindings — a common lateral-movement path.
  • Forgetting that a stolen SA token is valid until expiry (or forever, on old long-lived Secret tokens) — rotate/revoke by deleting the SA or Secret.
  • Binding a ClusterRoleBinding to a namespaced ServiceAccount when a namespaced RoleBinding would have sufficed, exploding blast radius.
  • Assuming ServiceAccounts authenticate humans — they don't; use OIDC/SSO for human users instead.

50 Interview Questions

1. What is a ServiceAccount and how does it differ from a Kubernetes User?
Short answer: A ServiceAccount is an API object giving an identity to processes running inside Pods; a User is a human identity that Kubernetes has no built-in API object for.
Detail: Users authenticate via certs or SSO/OIDC tokens managed outside the cluster; ServiceAccounts are first-class objects stored in etcd, created per namespace, and referenced directly by RBAC bindings.
Production example: A CI pipeline uses an OIDC-federated human/service identity externally, while an in-cluster controller uses a dedicated ServiceAccount.
Common mistake: Trying to model human access control using ServiceAccounts instead of OIDC/SSO.
2. What happens if a Pod spec doesn't set serviceAccountName?
Short answer: It's assigned the namespace's default ServiceAccount automatically.
Detail: The ServiceAccount admission controller performs this assignment at admission time, and by default that account also gets a token auto-mounted.
Production example: Legacy clusters often have dozens of workloads all silently sharing the default SA with whatever RBAC happens to be bound to it.
Common mistake: Assuming an unset serviceAccountName means "no identity" — it always means the default one.
3. How did ServiceAccount tokens change with the TokenRequest API (1.24+)?
Short answer: Tokens moved from long-lived, non-expiring Secret-based JWTs to short-lived (~1h), audience-bound, auto-rotated projected tokens.
Detail: The TokenRequest API is called by kubelet to mint a JWT scoped to a specific audience and TTL, mounted as a projected volume instead of a static Secret.
Production example: A stolen token from a compromised Pod now expires within an hour instead of being valid forever.
Common mistake: Still writing automation that expects a permanent Secret-based SA token to exist.
4. Why is automountServiceAccountToken: true risky by default?
Short answer: Every Pod gets a live API credential mounted on disk even if it never calls the Kubernetes API.
Detail: An attacker who achieves code execution in the container can read the token file and use it to enumerate or modify whatever that SA's RBAC bindings allow.
Production example: A compromised frontend Pod with an unnecessary broad-RBAC default SA token being used to list Secrets cluster-wide.
Common mistake: Leaving the setting at its implicit default instead of setting automountServiceAccountToken: false and opting in per-Pod.
5. How does the API Server identify a ServiceAccount from an incoming request?
Short answer: It validates the JWT's signature and audience, extracting the identity string system:serviceaccount:<namespace>:<name>.
Detail: This identity string is exactly what RoleBindings/ClusterRoleBindings reference as their subject, so authn and authz share the same identifier format.
Production example: Running kubectl auth can-i --as=system:serviceaccount:payments:payments-sa to test effective permissions.
Common mistake: Assuming the token itself encodes RBAC permissions — it only encodes identity; permissions are looked up separately via bindings.
6. Why does the projected token support an "audience" field?
Short answer: Audience-binding prevents a token minted for one consumer (e.g. the API Server) from being replayed against a different service.
Detail: Without audience binding, a leaked token usable against the API Server could potentially be reused against any other service that trusted the same issuer.
Production example: A workload identity federation setup where a Pod's SA token is exchanged for a cloud IAM credential, scoped to that specific audience only.
Common mistake: Using the default audience for every purpose instead of minting narrowly-scoped tokens per consumer.
7. What's the risk of binding a ClusterRoleBinding to a namespaced ServiceAccount?
Short answer: It grants that identity the bound permissions across every namespace in the cluster, not just its own.
Detail: A namespaced RoleBinding would have scoped the same permissions to a single namespace; ClusterRoleBinding is cluster-wide regardless of where the SA itself lives.
Production example: A reconciler SA accidentally bound via ClusterRoleBinding can read ConfigMaps in every tenant's namespace, not just its own.
Common mistake: Reaching for ClusterRoleBinding out of convenience when a RoleBinding would have sufficed.
8. How do you revoke a ServiceAccount's access immediately?
Short answer: Delete or modify the RoleBinding/ClusterRoleBinding referencing it, or delete the ServiceAccount object itself.
Detail: Because modern tokens are short-lived and signed against the SA's existence, deleting the SA invalidates future authentication; existing in-flight tokens still work until they expire since Kubernetes doesn't do live token revocation by default.
Production example: Incident response removing a compromised reconciler's RoleBinding to immediately cut off further API access even before the token's TTL lapses.
Common mistake: Assuming deleting a Secret-based legacy token immediately invalidates it everywhere without checking if it's cached by a client.
9. Why should each microservice get its own dedicated ServiceAccount?
Short answer: It lets RBAC scope permissions precisely to what that one workload needs, limiting blast radius if it's compromised.
Detail: Sharing one SA across unrelated apps means a vulnerability in any one of them exposes the combined permissions of all of them.
Production example: A payments-reconciler-sa scoped only to get/list/watch its own ConfigMaps, unable to touch anything in the checkout namespace.
Common mistake: Reusing a general-purpose "app-sa" across many Deployments for convenience.
10. What's the difference between the Controller Manager's legacy tokens controller and the TokenRequest flow?
Short answer: The legacy tokens controller auto-created long-lived Secret-backed tokens; TokenRequest issues short-lived signed JWTs on demand via the API, no Secret needed.
Detail: The legacy path is largely superseded because standing long-lived Secrets in etcd were a persistent credential-leak risk.
Production example: Newer clusters no longer show a <sa-name>-token-xxxxx Secret automatically appearing for every ServiceAccount.
Common mistake: Writing tooling that expects to find a token inside a Secret object instead of using kubectl create token or a projected volume.
11–50. Additional rapid-fire questions
11. Where is a projected SA token mounted inside a container by default? — /var/run/secrets/kubernetes.io/serviceaccount.
12. Who mounts and refreshes the projected token? — kubelet, periodically before expiry.
13. Can two Pods in different namespaces share the same ServiceAccount name? — Yes, ServiceAccounts are namespace-scoped, so identity is always namespace+name combined.
14. What identity string format does RBAC match against a ServiceAccount? — system:serviceaccount:<namespace>:<name>.
15. Does a ServiceAccount need an explicit Secret in modern Kubernetes? — No, TokenRequest-based projected tokens don't require a backing Secret.
16. What command mints a short-lived token manually? — kubectl create token <sa> --duration=1h.
17. How do you check which SA a running Pod uses? — kubectl get pod -o jsonpath='{.spec.serviceAccountName}'.
18. How do you test effective RBAC for an SA without deploying a Pod? — kubectl auth can-i --as=system:serviceaccount:ns:sa.
19. Is automountServiceAccountToken settable at the Pod level, ServiceAccount level, or both? — Both; Pod-level setting overrides ServiceAccount-level default.
20. What's the actual security risk of a stolen SA token? — Whatever the bound RoleBindings allow — read Secrets, list Pods, create resources, etc., until expiry.
21. Why is default expirationSeconds a security improvement over old tokens? — It bounds the usable window of a leaked token instead of it being valid indefinitely.
22. Can a ServiceAccount be used to authenticate a human via kubectl? — Technically yes if you extract its token, but it's not the intended pattern and bypasses individual accountability.
23. What object type does a RoleBinding's subject reference for a ServiceAccount? — kind: ServiceAccount with name and namespace fields.
24. Does deleting a Pod invalidate its mounted SA token immediately? — No, the token remains valid until its TTL expires regardless of Pod lifecycle.
25. What's the risk of an overly broad default namespace RoleBinding? — Every Pod using the default SA in that namespace inherits those permissions, even unrelated workloads.
26. How does audience-binding prevent token replay across services? — The receiving service validates the JWT's aud claim matches its own expected audience, rejecting mismatched tokens.
27. What component enforces RBAC decisions using the SA identity? — The API Server's authorization layer (RBAC authorizer).
28. Why is "does this Pod even need API access" the first least-privilege question? — Many workloads never call the Kubernetes API at all, so any mounted token is pure unnecessary risk.
29. Can a ServiceAccount be bound to permissions in a namespace other than its own? — Yes, via a RoleBinding in the target namespace referencing the SA with its origin namespace specified.
30. What's the purpose of `kubectl describe sa`? — Shows associated secrets/annotations for the ServiceAccount object.
31. Is the ServiceAccount admission controller mutating or validating? — Mutating — it injects the default SA and token volume into the Pod spec.
32. Why can't a ServiceAccount token be scoped to "read-only for 10 minutes then read-write"? — Tokens carry only identity, not fine-grained embedded permissions; RBAC (a separate, static binding) governs what the identity can do at request time.
33. What's a common lateral-movement path involving ServiceAccounts? — A compromised Pod on the default SA with broad bindings reads Secrets across the namespace to pivot further.
34. How would you find every binding referencing a specific SA? — kubectl get rolebinding,clusterrolebinding -A -o wide | grep <sa-name>.
35. Does a ServiceAccount need to exist before a Pod references it? — Best practice yes, though some clusters allow referencing before creation with the Pod failing admission until it exists depending on config.
36. What field disables the automatic token mount at the ServiceAccount level? — automountServiceAccountToken: false in the ServiceAccount spec.
37. Why would you explicitly re-mount a token via a projected volume even with automount disabled? — To control TTL and audience explicitly per Pod rather than accepting cluster defaults.
38. What happens to RBAC access if you delete a ServiceAccount but not its bindings? — The bindings become dangling references; recreating an SA with the same name would silently regain those permissions.
39. Is ServiceAccount identity checked on every API request or only at Pod startup? — Every request — the token is validated per-call by the API Server's authn layer.
40. What's the danger of naming SAs generically like "app-sa" across many deployments? — It obscures which specific workload holds which permissions, complicating audits.
41. Can RBAC distinguish between two Pods using the same ServiceAccount? — No — RBAC only sees the SA identity, not which specific Pod presented the token.
42. Why does short TTL matter more for CI-minted tokens than in-cluster ones? — CI tokens are more likely to be persisted in logs or artifacts, so a short expiry limits exposure window.
43. What's the interaction between PSA/Pod Security Standards and ServiceAccount tokens? — They're unrelated — PSA governs container privilege settings, not API identity/token behavior.
44. How do workload-identity-federation systems (e.g. cloud IAM) use SA tokens? — They exchange a Kubernetes-issued, audience-bound SA JWT for a cloud provider's temporary credentials via a trust relationship.
45. What's the risk of setting an extremely long expirationSeconds on a projected token? — It approaches the same risk profile as the old long-lived tokens — a wide exposure window if leaked.
46. Does kube-proxy use a ServiceAccount? — Yes, like any in-cluster component it authenticates to the API Server via its own SA and scoped RBAC.
47. What's the quickest way to audit every Pod still on the default SA? — kubectl get pods -A -o jsonpath filtering for empty/`default` serviceAccountName, or a policy-engine query.
48. Why is "one SA per workload" also good for auditability, not just security? — Audit logs and RBAC bindings become traceable to a specific microservice instead of an ambiguous shared identity.
49. Can you use ServiceAccounts across clusters? — Not directly — a token is only valid against the API Server (and audience) that issued it, though federation patterns can bridge this.
50. What's the first thing to check when a Pod gets "Forbidden" calling the API? — Which ServiceAccount it's using (spec.serviceAccountName) and whether any RoleBinding actually grants the needed verb/resource.
