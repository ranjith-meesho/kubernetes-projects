High Availability

Designing a control plane and worker fleet that survives node, zone, and quorum-member failure without an outage.
← Multi-cluster Course Index Backup & Restore →

Explanation

High Availability (HA) in Kubernetes means the cluster keeps accepting API requests and keeps scheduling/running workloads even when individual machines, network links, or entire availability zones fail. There are two separate HA problems that people often conflate: control-plane HA (can I still talk to the API server and get new scheduling decisions?) and data-plane / workload HA (do my application Pods keep serving traffic?). Both need deliberate design — Kubernetes does not make either of these free by default on a single-node or single-AZ cluster.

The control plane's core dependency is etcd, a Raft-based distributed key-value store. Raft requires a strict majority (quorum) of members to agree before any write commits, which is why etcd is always run with an odd number of members — 3 or 5. With 3 members you can lose 1 and still have a majority (2/3); with 5 you can lose 2. An even number like 4 gives you no extra fault tolerance over 3 (you can still only lose 1 before losing majority) while adding an extra vote to coordinate — pure downside. This is the single most important number to get right in an HA design.

On top of etcd, you typically run 3 (or more) kube-apiserver replicas, all stateless and all reading/writing the same etcd cluster, fronted by a load balancer (cloud LB, or a simple keepalived+HAProxy VIP for on-prem/kubeadm clusters). Because the API server is stateless, this scales linearly and requires no leader election — any replica can serve any request. The controller-manager and scheduler, by contrast, are active-passive: multiple replicas run, but only one is active at a time (enforced via a lease-based leader election using the coordination.k8s.io/Lease object), because having two schedulers or two replica-set controllers acting simultaneously would cause conflicting decisions.

Worker-node HA is a different concern: spread your Deployment/StatefulSet replicas across multiple nodes AND multiple availability zones using topologySpreadConstraints or pod anti-affinity, so that losing one node — or one entire zone — doesn't take your whole application down. Pair this with PodDisruptionBudgets so voluntary disruptions (node drains, cluster upgrades) never take down more replicas than your minimum availability tolerates.

In production, the real failure modes worth designing for are: a single node dying (handled by the ReplicaSet controller + kube-scheduler rescheduling), an entire AZ losing power or network (handled by multi-AZ node groups + topology spread), and etcd leader loss (handled by Raft re-election, typically sub-second, invisible to users). The scenario people forget is network partition between AZs — a minority partition of etcd members will correctly stop accepting writes rather than risk split-brain, which is why quorum math (not just "more replicas is better") governs your AZ count too: 3 etcd members across 3 AZs, never 2, so no single AZ loss removes your quorum majority.

Internal Working / Interactions

  • etcd Raft group: members elect a leader; all writes go through the leader and must be replicated to a majority before ack. Losing the leader triggers a new election (typically 1-2s) using randomized election timeouts to avoid split votes.
  • kube-apiserver replicas: stateless, sit behind an L4/L7 load balancer or a VIP; each opens its own watch connections to etcd; no coordination needed between them.
  • Leader election for controller-manager/scheduler: each replica tries to acquire a Lease object; the holder renews it periodically; if it fails to renew (crash, GC pause, network issue) another replica takes over within the lease's leaseDurationSeconds.
  • kubelet/node health: node-lifecycle-controller marks a node NotReady after missed heartbeats, then after pod-eviction-timeout begins evicting pods so the ReplicaSet controller can recreate them elsewhere.
  • Cloud LB / kube-vip / keepalived: provides the single stable endpoint (e.g. the value in kubeconfig's server:) that clients use regardless of which API server replica is currently healthy.

Code Example

#############################################
# 1. kubeadm HA control-plane init (first node)
#############################################
kubeadm init \
  --control-plane-endpoint "k8s-api.internal.example.com:6443" \  # stable LB endpoint, NOT a single node IP
  --upload-certs \                                                 # share certs so other CP nodes can join
  --pod-network-cidr=10.244.0.0/16

# Join additional control-plane nodes (repeat on 2 more machines, one per AZ)
kubeadm join k8s-api.internal.example.com:6443 \
  --token  \
  --discovery-token-ca-cert-hash sha256: \
  --control-plane \                                                # joins as a control-plane node, not just worker
  --certificate-key 

---
# 2. topologySpreadConstraints — spread replicas across zones AND nodes
apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service
spec:
  replicas: 6
  selector:
    matchLabels: { app: checkout }
  template:
    metadata:
      labels: { app: checkout }
    spec:
      topologySpreadConstraints:
        - maxSkew: 1                              # allow at most 1 extra pod imbalance per zone
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule         # hard requirement: never pack all replicas in one zone
          labelSelector:
            matchLabels: { app: checkout }
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname      # also spread across individual nodes within a zone
          whenUnsatisfiable: ScheduleAnyway
          labelSelector:
            matchLabels: { app: checkout }
      containers:
        - name: checkout
          image: registry.example.com/checkout:v2.3.1

---
# 3. PodDisruptionBudget — protect availability during voluntary disruption (drains/upgrades)
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: checkout-pdb
spec:
  minAvailable: 4        # out of 6 replicas, never let voluntary drains take more than 2 down at once
  selector:
    matchLabels: { app: checkout }

kubectl / CLI Commands

# Check etcd member health and quorum status (from an etcd pod/host)
etcdctl member list --endpoints=https://127.0.0.1:2379 --cacert=... --cert=... --key=...

# Confirm Raft leader / cluster health across all 3 members
etcdctl endpoint status --cluster -w table

# See which controller-manager/scheduler replica currently holds the leader Lease
kubectl -n kube-system get lease kube-controller-manager -o yaml
kubectl -n kube-system get lease kube-scheduler -o yaml

# List control-plane nodes and confirm they're spread across zones
kubectl get nodes -l node-role.kubernetes.io/control-plane -L topology.kubernetes.io/zone

# Verify a Deployment's actual pod-to-node/zone distribution
kubectl get pods -o wide -l app=checkout

# Check PodDisruptionBudget status before performing a drain
kubectl get pdb checkout-pdb

# Safely drain a node while respecting PDBs
kubectl drain  --ignore-daemonsets --delete-emptydir-data

Best Practices & Common Pitfalls

Best Practices

  • Run etcd with 3 or 5 members, one per AZ, never an even count.
  • Put a real load balancer or VIP in front of API servers — never point kubeconfig at one node's IP.
  • Use topologySpreadConstraints plus PodDisruptionBudgets together, not either alone.
  • Monitor etcd leader-change frequency and Raft proposal latency as first-class SLIs.
  • Prefer managed control planes (EKS/GKE/AKS) unless you have a dedicated platform team to run etcd.

Common Pitfalls

  • Running 2 or 4 etcd members — no quorum benefit, extra fragility.
  • Co-locating all 3 control-plane nodes in one AZ "for low latency" — defeats the purpose.
  • Forgetting PodDisruptionBudgets, so a routine node drain or cluster upgrade causes a real outage.
  • Anti-affinity set to preferredDuringScheduling only, silently ignored under resource pressure.
  • Treating HA as "done" after setup — not chaos-testing node/AZ failure regularly.

50 Interview Questions

1. What are the two separate HA problems in Kubernetes, and why shouldn't they be conflated?
Short answer: Control-plane HA (can I still reach the API server and schedule?) and data-plane/workload HA (do my app Pods keep serving traffic?).
Detail: Control-plane HA depends on etcd quorum and API server/leader-election design; workload HA depends on how you spread replicas across nodes/zones and enforce disruption budgets. Solving one doesn't solve the other.
Production example: A 3-AZ etcd cluster with fully healthy quorum can still let an entire application go down if all 6 replicas of a Deployment happen to land in one AZ.
Common mistake: Assuming "the control plane is HA" means the applications are automatically HA too.
2. Why must etcd always run with an odd number of members?
Short answer: Raft requires a strict majority to commit writes, and an odd count maximizes fault tolerance per member added.
Detail: With 3 members you tolerate losing 1 (2/3 majority remains); with 5 you tolerate losing 2. A 4-member cluster still only tolerates losing 1 (needs 3/4) — the 4th member adds coordination overhead with zero extra fault tolerance.
Production example: A kubeadm HA cluster deliberately runs 3 etcd members, one per AZ, never 2 or 4.
Common mistake: Adding a 4th etcd member "for extra safety" — it makes the cluster strictly worse, not better.
3. Why is kube-apiserver active-active while controller-manager and scheduler are active-passive?
Short answer: The API server is stateless so any replica can serve any request; controller-manager/scheduler must have exactly one active instance to avoid conflicting decisions.
Detail: Multiple schedulers acting simultaneously could double-bind Pods to nodes, or two ReplicaSet controllers could race on scale decisions — leader election via a Lease object prevents this.
Production example: A 3-master kubeadm cluster runs 3 API server replicas behind a VIP, but only one controller-manager and one scheduler are ever "active" at a time.
Common mistake: Assuming more scheduler replicas means more scheduling throughput — it doesn't; only one is ever doing work.
4. How does leader election for controller-manager/scheduler actually work?
Short answer: Each replica races to acquire a coordination.k8s.io/Lease object; the holder renews it periodically, and another replica takes over if it fails to renew in time.
Detail: Failure to renew (crash, GC pause, network issue) surfaces within the lease's configured duration, so failover is typically seconds, not minutes.
Production example: kubectl -n kube-system get lease kube-scheduler -o yaml shows the current holder's identity and renewal timestamp.
Common mistake: Not monitoring lease renewal failures/flapping as a leading indicator of control-plane resource starvation.
5. What's the difference between topologySpreadConstraints and pod anti-affinity for zone spread?
Short answer: topologySpreadConstraints allows fine-grained, quantified skew control (maxSkew) across a topology key; anti-affinity is a binary "avoid/prefer to avoid" rule.
Detail: topologySpreadConstraints can enforce even distribution across an arbitrary number of zones/nodes with a hard DoNotSchedule or soft ScheduleAnyway policy, which scales better than anti-affinity rules as replica/zone counts grow.
Production example: A 6-replica Deployment spread with maxSkew: 1 across 3 zones guarantees no zone holds more than 1 extra replica versus any other.
Common mistake: Using only preferredDuringSchedulingIgnoredDuringExecution anti-affinity, which is silently ignored under resource pressure, defeating the intended spread.
6. Why are PodDisruptionBudgets necessary in addition to topology spread?
Short answer: Topology spread controls placement; PDBs control how many replicas can be voluntarily removed at once (drains, upgrades) regardless of placement.
Detail: Without a PDB, a node drain during a cluster upgrade could evict enough replicas simultaneously to cause an outage, even if replicas were well spread across zones.
Production example: minAvailable: 4 on a 6-replica Deployment ensures a rolling node drain never takes more than 2 replicas down at once.
Common mistake: Deploying topology spread but forgetting PDBs, so a routine kubectl drain becomes an unplanned outage.
7. What happens during a network partition that isolates a minority of etcd members?
Short answer: The minority partition correctly stops accepting writes rather than risking split-brain, since it cannot reach quorum.
Detail: This is why AZ placement of etcd members matters as much as the count — 3 members spread across 3 AZs ensures no single AZ loss removes the majority; 2 members in one AZ and 1 in another risks losing quorum on a single AZ failure.
Production example: An AZ-wide network blip isolating 1 of 3 etcd members causes zero disruption — the remaining 2 still hold quorum and continue serving writes.
Common mistake: Placing 2 of 3 control-plane nodes in the same AZ "for low latency," which reintroduces a single point of failure for quorum.
8. What is the purpose of --control-plane-endpoint in a kubeadm HA setup?
Short answer: It gives the cluster a single stable DNS/VIP endpoint that survives any individual API server replica going down.
Detail: Without it, kubeconfig would point at one control-plane node's IP directly, defeating HA entirely — that node's failure would break all client access even though other API servers are healthy.
Production example: kubeadm init --control-plane-endpoint "k8s-api.internal.example.com:6443" fronts 3 API server replicas via a load balancer or keepalived VIP.
Common mistake: Bootstrapping with a single node's IP, then discovering during a later HA upgrade that kubeconfig can't be cleanly repointed at a VIP.
9–50. Additional rapid-fire questions
9. Why is running exactly 2 control-plane nodes worse than running 1? — 2 etcd members require both to be up for quorum (majority of 2 is 2), so you've added a node with zero fault tolerance benefit.
10. What triggers a Raft leader re-election in etcd? — Loss of heartbeat from the current leader within a randomized election timeout, used to avoid split votes.
11. Why are etcd election timeouts randomized rather than fixed? — Prevents multiple followers from timing out simultaneously and starting competing elections at once.
12. What's the role of node-lifecycle-controller in worker HA? — Marks a node NotReady after missed heartbeats, then evicts its Pods after pod-eviction-timeout so they reschedule elsewhere.
13. Why does kube-vip/keepalived matter for on-prem/kubeadm HA? — Provides the stable VIP that a cloud load balancer would otherwise provide automatically in managed environments.
14. Can the API server scale horizontally without coordination? — Yes — it's stateless, so adding replicas behind the LB requires no leader election or state sync between them.
15. What does maxSkew control in a topologySpreadConstraint? — The maximum allowed difference in matching Pod count between any two topology domains.
16. What does whenUnsatisfiable: DoNotSchedule mean versus ScheduleAnyway? — DoNotSchedule hard-blocks placement that would violate maxSkew; ScheduleAnyway allows it as a best-effort preference.
17. Why combine a zone-level and a hostname-level topologySpreadConstraint? — Zone spread survives AZ failure; hostname spread also survives single-node failure within a zone.
18. What's the first SLI to monitor for etcd health? — Raft proposal/commit latency and leader-change frequency — both spike before a quorum problem becomes visible elsewhere.
19. Why should you avoid running application workloads on control-plane nodes? — They compete for CPU/memory with etcd and the API server, degrading control-plane responsiveness cluster-wide.
20. What happens to a Deployment's Pods if the ReplicaSet controller (via controller-manager) is briefly unavailable? — Existing Pods keep running; only new reconciliation (recreating failed Pods, scaling) is paused until a leader is re-elected.
21. How do you verify etcd quorum health from the CLI? — etcdctl endpoint status --cluster -w table shows leader, term, and raft index per member.
22. Why is chaos-testing node/AZ failure part of a real HA strategy? — HA design that's never exercised commonly has silent gaps (missing PDBs, anti-affinity ignored under pressure) that only surface during a real failure.
23. What's the risk of setting a PDB's minAvailable too high? — It can block legitimate node drains/upgrades entirely if the cluster can't temporarily spare more replicas.
24. Does a load balancer in front of API servers need health checks? — Yes — it must route only to API servers currently able to reach etcd, not just processes that are technically running.
25. What is topology.kubernetes.io/zone used for beyond spread constraints? — It's also the label the Scheduler and node-affinity rules use generically to reason about zone placement.
26. Why is losing the etcd leader typically invisible to end users? — Raft re-election is normally sub-second, and the API server transparently retries against the new leader.
27. What's the operational difference between a managed control plane (EKS/GKE/AKS) and self-managed for HA? — Managed control planes come with the HA topology, quorum management, and failover already engineered and SLA-backed.
28. Can you achieve HA with all worker nodes in a single AZ? — No — a single AZ outage takes down every replica regardless of how well they're spread across nodes within that zone.
29. What does --upload-certs do in a kubeadm HA join flow? — Shares control-plane certificates securely so additional control-plane nodes can join without manually copying cert material.
30. How many etcd members can a 5-node etcd cluster lose and still function? — 2 — majority of 5 is 3, so quorum survives with 3 remaining.
31. Why not just run etcd with 7 or 9 members for extra safety? — Each added member increases the number of nodes required to agree per write, raising write latency without proportional fault-tolerance gain in most deployments.
32. What's a symptom of etcd running on slow disks in an HA cluster? — Frequent, unexplained leader elections caused by fsync latency delaying heartbeats past the election timeout.
33. Why should control-plane nodes be tainted against scheduling regular workloads? — Prevents application Pods from starving etcd/API-server of CPU and memory during load spikes.
34. What's the practical effect of losing the active scheduler for 30 seconds? — New Pods stay Pending; already-scheduled Pods are completely unaffected.
35. How does a rolling cluster upgrade interact with HA design? — It relies on PDBs and topology spread to ensure control-plane and worker upgrades never remove more capacity than the cluster can tolerate at once.
36. What's the difference between a Deployment's replicas count and actual availability during a zone outage? — Replica count is desired state; actual availability depends on how those replicas were distributed before the outage happened.
37. Why is HA design considered incomplete without a tested backup/restore process? — HA absorbs component failures, not data corruption or catastrophic misconfiguration — that's what backup/restore protects against.
38. What object type coordinates leader election in modern Kubernetes (vs older ConfigMap-based election)? — coordination.k8s.io/Lease, a lighter-weight purpose-built object replacing the older ConfigMap/Endpoints annotation approach.
39. Why must you check PDB status before running kubectl drain? — A drain waits/fails if evicting the node's Pods would violate a PDB, so pre-checking avoids a stuck operation mid-maintenance.
40. What's the tradeoff of running 5 etcd members instead of 3? — Higher fault tolerance (tolerate 2 losses) at the cost of higher write latency and more nodes to operate.
41. How does the Scheduler use zone labels differently from topologySpreadConstraints? — Zone labels are the raw topology data; topologySpreadConstraints is the policy that consumes those labels to enforce even distribution.
42. What's a realistic RTO for a single etcd leader failure versus a full AZ loss? — Leader failure: sub-second to a few seconds (Raft re-election); full AZ loss: minutes, bounded by rescheduling and traffic rebalancing across surviving zones.
43. Why is "3 nodes total" not the same as "HA" if they're all worker nodes? — Worker-node count alone says nothing about control-plane quorum or Pod distribution across failure domains.
44. What's the risk of an HA design that was correct at launch but never revisited? — Team growth, new namespaces, or workload changes can silently violate original topology-spread/PDB assumptions over time.
45. How do you detect that anti-affinity rules are being silently ignored? — Inspect actual Pod-to-node/zone placement (kubectl get pods -o wide) rather than trusting the manifest's stated intent.
46. What's the relationship between etcd quorum size and the number of AZs you should use? — They should match — 3 etcd members should map to 3 AZs, not 3 members crammed into 2 AZs.
47. Why does stateless API-server scaling not require a Lease object? — There's no shared mutable decision being made between replicas — every replica independently serves whatever request it receives.
48. What's the first thing to check if PDBs seem to be blocking all drains cluster-wide? — Whether minAvailable/maxUnavailable values were set unrealistically tight relative to actual replica counts.
49. Why is monitoring leader-election flapping a leading indicator, not just a lagging one? — Frequent leader changes usually reflect underlying resource starvation or network instability that will cause bigger problems if untreated.
50. What's the single most important number to get right in an HA design, per this topic? — The etcd member count and its odd/even parity — it's the one decision that silently determines your entire quorum fault tolerance.
