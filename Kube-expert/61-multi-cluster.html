Multi-cluster

Fleet management, Cluster API, and service mesh federation — patterns for when one cluster is no longer enough.
← Kubernetes Internals Course Index High Availability →

Explanation

A single Kubernetes cluster has real ceilings — officially supported up to roughly 5,000 nodes, but far more commonly teams split into multiple clusters long before hitting that limit, for reasons that have nothing to do with raw capacity: blast-radius isolation (a control-plane incident or a bad CRD/webhook shouldn't take down every team's workloads at once), regulatory/data-residency boundaries (EU customer data must stay on EU-region clusters), multi-region latency and disaster recovery (serve users from the nearest region, survive a whole region going down), and organizational boundaries (giving each business unit or environment — dev/staging/prod — its own cluster rather than fighting over shared quota and RBAC in one).

Once you have more than one cluster, new problems appear that a single cluster's control plane never had to solve: how do you deploy the same application consistently across all of them? How do you discover and route to a service running in a different cluster? How do you get one unified view of health/metrics/logs across clusters instead of context-switching kubeconfigs all day? This is the "multi-cluster" problem space, and it has a few dominant patterns rather than one standard solution, because different organizations weight these problems differently.

Fleet management tools (Argo CD's ApplicationSets, Flux, Rancher Fleet, Google Anthos Config Management) solve the "deploy consistently everywhere" problem: one Git repository describes desired state, and a controller (running centrally or per-cluster) applies it across every registered cluster, often with per-cluster overlays for environment-specific values. This directly extends the GitOps model from earlier in the course, just targeting N clusters instead of one.

Cluster API (CAPI) solves a different problem: provisioning and lifecycle-managing the clusters themselves declaratively. Instead of clicking through a cloud console or running ad-hoc Terraform per cluster, CAPI represents clusters, node pools, and their infrastructure as Kubernetes custom resources managed by a controller running in a small "management cluster" — literally using Kubernetes' own reconciliation model (Topic 60) to manage the creation and upgrade of other Kubernetes clusters.

Service mesh federation (Istio multi-cluster, Linkerd multicluster) solves cross-cluster service discovery and traffic management: services in cluster A can call services in cluster B as if they were local, with mTLS, load balancing, and failover handled by the mesh's control plane spanning both clusters. This is the mechanism behind patterns like "active-active" multi-region deployments or gradual migration of workloads from an old cluster to a new one without a big-bang cutover.

Teams typically go multi-cluster when a single cluster's failure domain becomes unacceptable (one bad cluster-wide change taking down everything), when compliance mandates data/workload separation, or when the organization outgrows a single platform team's ability to safely share one cluster across many independent consumers — at which point the operational cost of running N clusters is repaid by the isolation and blast-radius containment it buys.

Internal Working / Interactions

  • Management cluster (Cluster API) runs controllers that watch Cluster/MachineDeployment custom resources and talk to cloud provider APIs (via provider-specific CAPI controllers) to create/scale/upgrade workload clusters.
  • GitOps fleet controller (Argo CD ApplicationSet/Flux) reads a Git repo, generates one Application per target cluster (often templated from a cluster list or label selector), and applies desired state to each cluster's own API server.
  • Service mesh control plane (Istio) in multi-cluster mode exchanges service/endpoint information between clusters (via a shared root CA for mTLS and an East-West gateway), letting sidecars route cross-cluster traffic transparently.
  • Centralized observability — VictoriaMetrics/Grafana/Loki (Topics 52-57) are often deployed as one central instance (or federated) so engineers get one dashboard/query surface across every cluster rather than N separate stacks.

Code Example

# Cluster API: declaratively describes a workload cluster's control plane and infra,
# reconciled by CAPI controllers running in the management cluster.
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: eu-west-prod
  namespace: capi-clusters
spec:
  clusterNetwork:
    pods: { cidrBlocks: ["10.244.0.0/16"] }
  controlPlaneRef:
    apiVersion: controlplane.cluster.x-k8s.io/v1beta1
    kind: KubeadmControlPlane
    name: eu-west-prod-control-plane
  infrastructureRef:
    apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
    kind: AWSCluster                     # provider-specific; created/managed by the AWS CAPI provider
    name: eu-west-prod
---
# Argo CD ApplicationSet — fans out one Application per registered cluster using a
# cluster generator, so the same Git path is deployed consistently everywhere.
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: checkout-service-fleet
  namespace: argocd
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            environment: production      # targets every cluster registered with this label in Argo CD
  template:
    metadata:
      name: 'checkout-service-{{name}}'
    spec:
      project: default
      source:
        repoURL: https://github.com/org/checkout-service
        targetRevision: main
        path: deploy/overlays/{{name}}    # per-cluster kustomize overlay for region-specific config
      destination:
        server: '{{server}}'
        namespace: checkout

Best Practices & Common Pitfalls

Best Practices

  • Only go multi-cluster when you have a concrete isolation, compliance, or latency requirement — not by default.
  • Standardize cluster provisioning via Cluster API or equivalent so every cluster is reproducible, not hand-built.
  • Centralize observability (one Grafana, federated or remote-write metrics) so engineers aren't context-switching per cluster during incidents.
  • Use GitOps fleet tooling (ApplicationSets/Flux) so per-cluster drift is structurally prevented, not manually audited.
  • Keep a clear naming/labeling convention (region, environment) consistent across every cluster for tooling to key off.

Pitfalls

  • Going multi-cluster purely for perceived scale before actually hitting single-cluster limits — multiplies operational overhead for no real benefit.
  • Running N independent, hand-configured clusters with no fleet tooling, leading to silent configuration drift.
  • Assuming service mesh federation gives free cross-cluster failover without testing actual latency/failure behavior.
  • Neglecting a single source of truth for cluster inventory, causing "which clusters do we even have" uncertainty during incidents.

50 Interview Questions

1. Why do teams split into multiple clusters well before hitting Kubernetes' node-count limits?
Short answer: Blast-radius isolation, regulatory/data-residency boundaries, multi-region latency/DR, and organizational boundaries — not raw capacity.
Detail: A single cluster is officially supported up to roughly 5,000 nodes, but a control-plane incident or bad CRD/webhook in one cluster shouldn't take down every team's workloads at once.
Production example: A company keeps EU customer data on EU-region clusters purely for data-residency compliance, unrelated to node count.
Common mistake: Going multi-cluster purely for perceived scale before actually hitting single-cluster limits — this multiplies operational overhead for no real benefit.
2. What new problems appear once an organization runs more than one cluster?
Short answer: Consistent deployment across clusters, cross-cluster service discovery/routing, and unified observability.
Detail: A single cluster's control plane never had to solve "deploy the same app everywhere" or "call a service in a different cluster" — these are the defining multi-cluster problems.
Production example: Without fleet tooling, engineers context-switch kubeconfigs all day trying to get one view of health across clusters.
Common mistake: Assuming these problems have one standard solution — different organizations weight them differently, producing different dominant patterns.
3. What problem does fleet management tooling (Argo CD ApplicationSets, Flux, Rancher Fleet) solve?
Short answer: Deploying the same application consistently across every registered cluster from one Git repository.
Detail: A controller applies desired state across every cluster, often with per-cluster overlays for environment-specific values — a direct extension of the single-cluster GitOps model to N clusters.
Production example: An ApplicationSet with a cluster generator fans out one Application per cluster labeled environment: production.
Common mistake: Running N independent, hand-configured clusters with no fleet tooling, leading to silent configuration drift.
4. What problem does Cluster API (CAPI) solve, and how does it solve it?
Short answer: Provisioning and lifecycle-managing the clusters themselves declaratively, using Kubernetes' own reconciliation model.
Detail: CAPI represents clusters, node pools, and their infrastructure as Kubernetes custom resources managed by controllers running in a small management cluster, rather than clicking through a cloud console or running ad-hoc Terraform per cluster.
Production example: A Cluster object referencing a KubeadmControlPlane and provider-specific AWSCluster resource, reconciled by CAPI controllers to create eu-west-prod.
Common mistake: Treating CAPI as just another provisioning script rather than recognizing it literally reuses the same watch/reconcile machinery as any in-cluster controller.
5. What does service mesh federation (Istio multi-cluster, Linkerd multicluster) solve?
Short answer: Cross-cluster service discovery and traffic management, so services in one cluster can call services in another as if local.
Detail: mTLS, load balancing, and failover are handled by the mesh's control plane spanning both clusters, via a shared root CA and an East-West gateway exchanging service/endpoint information.
Production example: This is the mechanism behind "active-active" multi-region deployments or gradually migrating workloads from an old cluster to a new one without a big-bang cutover.
Common mistake: Assuming service mesh federation gives free cross-cluster failover without testing actual latency/failure behavior.
6. What is the role of a "management cluster" in Cluster API?
Short answer: It runs the CAPI controllers that watch Cluster/MachineDeployment custom resources and talk to cloud provider APIs to create/scale/upgrade workload clusters.
Detail: The management cluster itself is a small Kubernetes cluster whose only job is reconciling the existence and shape of other clusters.
Production example: Provider-specific CAPI controllers (e.g. the AWS provider) reconcile an AWSCluster resource by creating the underlying VPC/instances.
Common mistake: Conflating the management cluster with a workload cluster — the management cluster typically runs no application workloads itself.
7. How does an Argo CD ApplicationSet fan out deployments across a fleet of clusters?
Short answer: A cluster generator (e.g. matching a label selector) produces one Application per registered target cluster from a single template.
Detail: Each generated Application can point at a per-cluster kustomize overlay path (like deploy/overlays/{{name}}) for region-specific configuration while sharing the same source repo.
Production example: A checkout-service-fleet ApplicationSet selecting clusters labeled environment: production deploys checkout-service identically to every production cluster.
Common mistake: Manually creating one Argo CD Application per cluster by hand instead of using a generator, which reintroduces the drift problem GitOps was meant to solve.
8. Why is centralized observability considered part of the multi-cluster architecture rather than an afterthought?
Short answer: Without it, engineers get N separate dashboards/query surfaces and must context-switch kubeconfigs during incidents.
Detail: VictoriaMetrics/Grafana/Loki are typically deployed as one central instance (or federated) so there's a single dashboard/query surface across every cluster.
Production example: During an incident spanning us-east and eu-west clusters, a single Grafana view avoids toggling between per-cluster dashboards.
Common mistake: Neglecting a single source of truth for cluster inventory and metrics, causing "which clusters do we even have" uncertainty during incidents.
9. What triggers a shared root CA and East-West gateway in Istio multi-cluster mode?
Short answer: The need for sidecars in one cluster to establish mTLS and route traffic to services in another cluster.
Detail: The mesh control planes in each cluster exchange service/endpoint information across this gateway, letting cross-cluster calls look local to the calling service.
Production example: kubectl --context=cluster-a get svc -n istio-system istio-eastwestgateway verifies the gateway that enables this federation.
Common mistake: Assuming mesh federation works automatically without validating the shared trust root (CA) and gateway connectivity between clusters.
10. When is the operational cost of running N clusters considered "repaid"?
Short answer: When a single cluster's failure domain becomes unacceptable, compliance mandates separation, or one platform team can no longer safely share a cluster across many consumers.
Detail: At that point the isolation and blast-radius containment N clusters provide outweighs the added operational overhead of fleet management, provisioning, and cross-cluster networking.
Production example: A platform team splits a shared cluster into per-business-unit clusters once RBAC/quota conflicts between teams become unmanageable.
Common mistake: Only going multi-cluster when there's a concrete isolation, compliance, or latency requirement — going multi-cluster by default inverts this tradeoff.
