DaemonSets

Guarantees exactly one copy of a Pod on every (or every matching) node — the pattern behind log/metrics agents and CNI.
← StatefulSets Course Index Jobs →

Explanation

A DaemonSet guarantees that a copy of a specific Pod runs on every node in the cluster (or every node matching a selector), and automatically adds that Pod to any new node as it joins, and removes it when a node leaves. Unlike a Deployment/ReplicaSet, a DaemonSet doesn't have a "desired replica count" you set — the desired count is implicitly derived from the number of eligible nodes: one Pod per node, no more, no less.

Kubernetes needs this pattern because some workloads are fundamentally node-scoped infrastructure rather than application replicas: a log shipper that must read every node's container logs, a metrics agent that must scrape every node's kubelet/cAdvisor, a CNI plugin that must configure every node's network, or a storage daemon that must expose every node's local disks. None of these make sense as "3 replicas somewhere in the cluster" — they only make sense as "exactly one, on every node, forever." Trying to model this with a Deployment would require constantly adjusting replica count to match node count and would offer no guarantee of node coverage (the Scheduler could legally pile multiple replicas onto one node and zero onto another).

Architecturally, DaemonSets bypass the normal Scheduler filtering in one important way: since Kubernetes 1.12, the DaemonSet controller itself decides placement (one Pod per matching node) and the default scheduler merely honors that binding, while still respecting node taints/tolerations. This is why DaemonSet Pods commonly carry tolerations for taints like node-role.kubernetes.io/control-plane:NoSchedule — you often want your logging or CNI agent to run even on nodes that reject ordinary workloads, because the control plane's own nodes still need to be monitored/networked.

A nodeSelector or node affinity rule on the DaemonSet template restricts it to a subset of nodes — e.g. a GPU-monitoring DaemonSet only needs to run on nodes labeled gpu=true. This "one-per-matching-node" model, combined with automatic Pod creation/removal as nodes join/leave the cluster, is what makes DaemonSets the standard mechanism for cluster-wide infrastructure that must track the node fleet exactly.

Internal Working / Interactions

  • DaemonSet controller (in kube-controller-manager) — watches Node objects and the DaemonSet spec; for every node matching the nodeSelector/affinity and tolerating its taints, ensures exactly one Pod exists.
  • API Server / etcd — DaemonSet spec and each node-bound Pod are ordinary objects, watched and persisted the normal way.
  • Node lifecycle — new node joins → controller detects it → creates a Pod pre-bound to that node; node removed/drained → its DaemonSet Pod is garbage collected.
  • Taints and tolerations — DaemonSet Pods typically declare tolerations for NoSchedule/NoExecute taints so infrastructure agents still run on tainted/control-plane nodes where regular workloads are excluded.
  • kubelet — starts/runs the Pod exactly as any other Pod once bound to its node; no special kubelet-side behavior is required for DaemonSets.

Code Example

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
spec:
  selector:
    matchLabels:
      app: fluent-bit          # Must match template labels; NO replicas field — one per eligible node.
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      tolerations:
        - key: node-role.kubernetes.io/control-plane   # Allow scheduling onto control-plane nodes too.
          effect: NoSchedule
        - key: node.kubernetes.io/not-ready             # Keep running briefly even if node is degrading.
          effect: NoExecute
          operator: Exists
      nodeSelector:
        kubernetes.io/os: linux     # Skip Windows nodes if any exist in the cluster.
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:2.2
          resources:
            requests: { cpu: "50m", memory: "64Mi" }   # Small footprint — runs on EVERY node, cost multiplies fast.
            limits:   { cpu: "150m", memory: "128Mi" }
          volumeMounts:
            - name: varlog
              mountPath: /var/log       # Read the HOST's log directory, not a container-local one.
              readOnly: true
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
      updateStrategy:
        type: RollingUpdate
        rollingUpdate:
          maxUnavailable: 1             # Update one node's agent at a time.

kubectl Commands

# List DaemonSets and their desired/current/ready node counts
kubectl get ds -n logging

# Confirm one Pod exists per node
kubectl get pods -n logging -l app=fluent-bit -o wide

# Show scheduling decisions and any nodes it's skipping
kubectl describe ds fluent-bit -n logging

# Trigger/monitor a rolling update of the DaemonSet's image
kubectl set image ds/fluent-bit fluent-bit=fluent/fluent-bit:2.2.1 -n logging
kubectl rollout status ds/fluent-bit -n logging

# Roll back a DaemonSet update
kubectl rollout undo ds/fluent-bit -n logging

# See which nodes are excluded and why (check taints)
kubectl get nodes -o json | jq '.items[].spec.taints'

Best Practices & Common Pitfalls

Best Practices

  • Set tight resource requests/limits — costs and risk multiply by node count for every DaemonSet.
  • Add tolerations deliberately for any taint your infra agent genuinely needs to survive (control-plane, not-ready).
  • Use nodeSelector/affinity to scope specialized agents (GPU, storage) instead of running everywhere.
  • Use RollingUpdate with a bounded maxUnavailable to avoid losing all node coverage at once during an update.

Pitfalls

  • Forgetting tolerations, so critical monitoring/logging silently never runs on control-plane or tainted nodes.
  • Not setting resource limits, causing a "small" agent to become a cluster-wide noisy-neighbor at scale.
  • Using hostPath volumes without readOnly where not needed, widening the security blast radius.
  • Trying to use a DaemonSet for horizontal application scaling — it scales with nodes, not with load.

50 Interview Questions

1. Why doesn't a DaemonSet have a spec.replicas field like a Deployment?
Short answer: Its desired count is implicitly derived from the number of eligible nodes — one Pod per matching node, automatically.
Detailed explanation: Setting an explicit count would require constantly reconciling it against a changing node count; instead the DaemonSet controller watches Node objects directly and creates/removes Pods as nodes join/leave.
Production example: A fluent-bit DaemonSet automatically gains a new Pod the instant a node is added to the cluster, with no manual scale action.
Common mistake: Looking for a replicas field to "scale" a DaemonSet — you scope it via nodeSelector/affinity instead, not a count.
2. Why can't a Deployment reliably model "one Pod per node" infrastructure like log or metrics agents?
Short answer: A Deployment's replica count has no concept of node coverage — the Scheduler could legally place multiple replicas on one node and zero on another.
Detailed explanation: DaemonSets solve this by having the controller itself decide placement (one Pod per matching node) rather than leaving distribution to normal Scheduler scoring, which optimizes for bin-packing, not coverage.
Production example: A cluster-wide log shipper needs a guarantee every node is covered — a Deployment with replicas=node-count could still leave a node with zero agents.
Common mistake: Trying to approximate DaemonSet behavior with a Deployment plus anti-affinity rules — fragile and doesn't auto-track node count changes.
3. Why do DaemonSet Pods commonly carry tolerations for control-plane taints?
Short answer: Because infrastructure agents like logging/monitoring/CNI often need to run even on nodes that reject ordinary workloads.
Detailed explanation: Control-plane nodes are normally tainted NoSchedule to keep regular application Pods off them, but those nodes still need to be monitored and networked — so DaemonSets explicitly tolerate that taint while regular Deployments don't.
Production example: The fluent-bit DaemonSet tolerates node-role.kubernetes.io/control-plane:NoSchedule so control-plane node logs are still shipped.
Common mistake: Forgetting this toleration, silently leaving control-plane nodes unmonitored/unlogged while assuming full cluster coverage.
4. How does the DaemonSet controller's scheduling differ from the normal Scheduler's filter/score model?
Short answer: Since Kubernetes 1.12, the DaemonSet controller itself decides placement and pre-binds the Pod; the default scheduler mostly just honors that binding.
Detailed explanation: This bypasses normal scoring entirely — there's no "best node" competition; every eligible node (matching nodeSelector/affinity and tolerating its taints) gets exactly one Pod, full stop.
Production example: A GPU-monitoring DaemonSet scoped with nodeSelector: gpu=true places a Pod on every GPU node without any scoring/ranking step.
Common mistake: Assuming DaemonSet Pods compete for "best fit" placement the way ReplicaSet Pods do.
5. Why is resource sizing especially critical for DaemonSets compared to other workload types?
Short answer: Because whatever you set is multiplied by the entire node count — a "small" agent becomes a large aggregate cost/risk at scale.
Detailed explanation: A DaemonSet with unset or generous limits can become a cluster-wide noisy neighbor, since every single node reserves/consumes that amount simultaneously, unlike a Deployment where only a few replicas exist regardless of cluster size.
Production example: A 50m/64Mi fluent-bit request seems trivial on one node but reserves 5 full CPU cores and 6.4Gi of memory across a 100-node cluster.
Common mistake: Copy-pasting Deployment-style generous resource requests onto a DaemonSet template without considering the node-count multiplier.
6. How does nodeSelector let a DaemonSet run on only a subset of nodes?
Short answer: It restricts "eligible nodes" to only those matching the selector, so the DaemonSet controller only creates Pods there instead of cluster-wide.
Detailed explanation: This is how specialized infrastructure agents (e.g. GPU drivers, storage daemons) avoid wasting resources on nodes that don't need them, while still getting the "exactly one, auto-tracked" guarantee on the nodes that do.
Production example: A GPU-monitoring DaemonSet with nodeSelector: gpu=true only ever runs on labeled GPU nodes, never on general-purpose worker nodes.
Common mistake: Omitting a needed nodeSelector and having a specialized agent uselessly attempt to run (and possibly crash-loop) on incompatible nodes.
7. What happens automatically when a new node joins the cluster, from the DaemonSet's perspective?
Short answer: The DaemonSet controller detects the new Node object, checks it against nodeSelector/taints, and creates a pre-bound Pod for every DaemonSet it applies to.
Detailed explanation: This happens without any manual intervention — the controller watches Node objects the same way other controllers watch their primary resource, reacting within its normal reconcile cycle.
Production example: Adding a node to an autoscaling node group automatically gets its own fluent-bit and CNI Pods moments after joining, before application workloads are even scheduled there.
Common mistake: Assuming new nodes need manual DaemonSet Pod creation — this is fully automatic and requires no action.
8. Why is a DaemonSet the wrong tool for horizontal application scaling?
Short answer: Its Pod count scales with the number of nodes, not with application load.
Detailed explanation: Adding more nodes to handle unrelated workloads would also add more instances of a DaemonSet-based "application," which has nothing to do with that application's actual traffic/load — the wrong scaling signal entirely.
Production example: Running a stateless API server as a DaemonSet would tie its replica count to cluster size rather than request volume, breaking any sensible capacity planning.
Common mistake: Choosing DaemonSet for "I want it always running" without realizing that's node-scoped, not load-scoped.
9. What's the danger of using hostPath volumes in a DaemonSet without readOnly?
Short answer: It widens the security blast radius — a compromised DaemonSet Pod with write access to the host filesystem can tamper with node-level files across every node in the cluster.
Detailed explanation: Because DaemonSets run on every node (often including control-plane nodes), a writable hostPath mount effectively gives that level of host access cluster-wide, not just on one machine.
Production example: A logging DaemonSet only needs read access to /var/log — mounting it writable unnecessarily would let a compromised container tamper with host logs everywhere.
Common mistake: Defaulting to a writable hostPath mount out of convenience instead of explicitly setting readOnly where write access isn't actually required.
10. How does a DaemonSet's RollingUpdate strategy differ in effect from a Deployment's?
Short answer: It updates node-by-node (bounded by maxUnavailable), rather than shifting traffic between two ReplicaSets — there's only ever one DaemonSet generation active per node at a time.
Detailed explanation: Since there's no "surge" concept for DaemonSets (you can't run two Pods on the same node under the same DaemonSet), the update proceeds by taking down and replacing the agent on a bounded number of nodes at once.
Production example: Setting maxUnavailable: 1 on the fluent-bit DaemonSet updates one node's logging agent at a time, so log coverage is never lost cluster-wide during the rollout.
Common mistake: Setting maxUnavailable too high for a critical infrastructure DaemonSet, briefly losing coverage (e.g. CNI or logging) across too many nodes simultaneously.
11–50. Additional rapid-fire questions
11. What API group/version is DaemonSet under? — apps/v1.
12. Does a DaemonSet Pod get an ownerReference like a ReplicaSet's Pods? — Yes, pointing back to the owning DaemonSet.
13. What happens to a DaemonSet's Pod when its node is drained? — It's evicted; the controller will recreate it once the node is uncordoned/rejoins (drain typically also uses --ignore-daemonsets since DaemonSet Pods otherwise block drain completion).
14. Why does kubectl drain need --ignore-daemonsets? — Because DaemonSet Pods are meant to run on every node including the one being drained, and would otherwise never "successfully" evict.
15. Can a DaemonSet use podAffinity/podAntiAffinity? — Rarely meaningful — placement is already one-per-node via the controller, not the normal scheduler affinity mechanism.
16. What update strategies does DaemonSet support? — RollingUpdate (default) and OnDelete (manual, Pod-by-Pod on deletion only).
17. What's the purpose of OnDelete for a DaemonSet? — Full manual control over exactly when each node's Pod picks up the new template, rather than an automatic bounded rollout.
18. Does a DaemonSet support HPA? — No — HPA scales replica counts, which DaemonSets don't have; scaling is inherently tied to node count.
19. How do you check which nodes a DaemonSet is skipping? — Compare kubectl get nodes against kubectl get pods -o wide for that DaemonSet, and inspect node taints.
20. What field shows desired vs current vs ready DaemonSet Pod counts? — kubectl get ds output columns: DESIRED, CURRENT, READY, UP-TO-DATE, AVAILABLE.
21. Can a DaemonSet run more than one Pod per node? — No — exactly one per eligible node is the entire model; you can't have two DaemonSet Pods from the same DaemonSet on one node.
22. How would you completely exclude one specific node from a DaemonSet? — Add a taint that Pod's tolerations don't cover, or use node affinity/anti-affinity rules to exclude it.
23. What's a real production use case beyond logging/monitoring? — CNI plugins (e.g. Calico's calico-node), CSI node plugins, and security agents (e.g. Falco) all run as DaemonSets.
24. Why is DaemonSet the natural fit for CNI plugins specifically? — Networking must be configured on literally every node for Pod-to-Pod connectivity to work — a coverage gap on any single node breaks that node's Pods entirely.
25. What happens if a DaemonSet's Pod repeatedly crashes on one specific node? — Normal Pod restart policy applies locally on that node; the DaemonSet controller doesn't reschedule it elsewhere since it's pinned to that node by design.
26. Is DaemonSet namespaced or cluster-scoped? — Namespaced, like Deployments and StatefulSets, even though its effect (one Pod per node) is cluster-wide.
27. What's the risk of a DaemonSet's readinessProbe misconfiguration? — Every single node's Pod could simultaneously fail readiness, causing the underlying capability (e.g. metrics scraping) to silently break cluster-wide.
28. Can DaemonSets be updated via kubectl set image like Deployments? — Yes — kubectl set image ds/<name> ... triggers a rolling update the same way.
29. How do you roll back a DaemonSet update? — kubectl rollout undo ds/<name>, same command family as Deployments.
30. What's a common cost-optimization concern specific to DaemonSets? — Aggregate resource reservation across all nodes — even modest per-Pod requests add up quickly at scale and affect bin-packing headroom.
31. Why might a DaemonSet need tolerations for node.kubernetes.io/not-ready? — So it keeps running briefly on a degrading node rather than immediately evicting, useful for agents that should keep reporting status as long as possible.
32. Does the DaemonSet controller retry failed Pod creation? — Yes — like any controller, it keeps reconciling toward "one Pod per eligible node" and retries on failure.
33. What's the effect of labeling a node to match a DaemonSet's nodeSelector after the DaemonSet already exists? — The controller detects the newly-matching node and creates a Pod there immediately.
34. What's the effect of removing a label so a node no longer matches? — The controller detects the node no longer matches and deletes that node's DaemonSet Pod.
35. How is DaemonSet Pod placement different from a Job's Pod placement? — Job Pods are scheduled normally (filter/score, any eligible node); DaemonSet Pods are pre-bound one-per-matching-node by the controller itself.
36. Why is monitoring DaemonSet rollout status important operationally? — A stuck DaemonSet rollout could leave some nodes running an old, possibly vulnerable, version of a security-critical agent indefinitely.
37. Can you scope a DaemonSet to just Windows nodes? — Yes, via nodeSelector: kubernetes.io/os: windows (or the inverse to exclude them, as shown for Linux-only agents).
38. What's a subtle failure mode if a DaemonSet's tolerations are too broad? — It could schedule onto nodes explicitly cordoned/tainted for maintenance or dedicated workloads, defeating the purpose of those taints.
39. How does a DaemonSet interact with cluster-autoscaler scale-down? — cluster-autoscaler generally ignores DaemonSet Pods when deciding if a node is "empty enough" to remove, since they're expected on every node.
40. What's the relationship between DaemonSet and node taints during cordon? — Cordoning (marking unschedulable) doesn't taint the node in a way that excludes existing DaemonSet Pods from continuing to run.
41. Why do storage CSI node plugins commonly run as DaemonSets? — Volume attach/mount operations must happen on the specific node where a Pod using that volume is scheduled — needs a local agent everywhere.
42. Is it possible for a DaemonSet Pod's image update to require a node reboot? — No inherently — but some low-level agents (e.g. certain CNI/CSI plugins) may require a brief node network/mount disruption during their own update.
43. What's the interview-relevant distinction between DaemonSet coverage and Deployment redundancy? — DaemonSet guarantees "everywhere," Deployment guarantees "somewhere, N times" — different guarantees entirely, not interchangeable.
44. How would you audit whether a DaemonSet is missing coverage on any node? — Compare node count to DaemonSet's DESIRED/CURRENT/READY columns from kubectl get ds, and cross-check taints on any excluded nodes.
45. What happens if two DaemonSets have overlapping nodeSelectors targeting the same infrastructure role? — Both run independently on the same matching nodes — no conflict, since each DaemonSet places its own distinct Pod per node.
46. Why is it risky to run heavyweight agents (e.g. full APM instrumentation) as a DaemonSet without careful sizing? — The per-node overhead directly competes with application workloads for CPU/memory on every single node, unlike a handful of Deployment replicas.
47. What's a good practice for versioning DaemonSet images in production? — Pin specific tags (not latest) and roll out via the same controlled RollingUpdate mechanism used for Deployments.
48. Can a DaemonSet's Pod template include init containers? — Yes — same Pod spec capabilities as any workload, including initContainers for pre-setup tasks (e.g. kernel module checks).
49. What's the operational risk of running privileged containers in a DaemonSet? — Since it runs on every node, a security flaw is maximally exploitable cluster-wide rather than confined to a handful of replicas.
50. What's the first thing to check when a DaemonSet shows fewer READY than DESIRED? — kubectl describe ds and per-Pod events for scheduling/toleration mismatches or readinessProbe failures on the missing nodes.
