StorageClass

The template for dynamic provisioning: provisioners, parameters, and volumeBindingMode.
← PersistentVolumeClaim Course Index CSI →

Explanation

A StorageClass is the template that tells Kubernetes how to create storage on demand. Without it, every PV needs a human to provision the backing disk out-of-band and write the PV YAML by hand — workable for a handful of volumes, unworkable for hundreds of PVCs created daily by CI pipelines, StatefulSets, and self-service developer namespaces. A StorageClass names a provisioner (the plugin responsible for actually creating storage — almost always a CSI driver identifier like ebs.csi.aws.com or pd.csi.storage.gke.io) plus a set of parameters specific to that provisioner (disk type, IOPS, filesystem, encryption key, replication factor).

The core idea: a PVC that sets storageClassName: fast-ssd is not requesting a specific disk — it's requesting "whatever fast-ssd is configured to produce." This lets platform teams define a small menu of storage tiers (e.g. standard, fast-ssd, archive) once, and every application team just picks a tier by name without needing storage-vendor knowledge.

reclaimPolicy on the StorageClass becomes the default reclaim policy stamped onto every PV it dynamically creates — this is where most teams first encounter the Delete-vs-Retain decision, since it's set once at the class level rather than per-PV.

volumeBindingMode controls when the actual disk gets created, and it matters a lot for multi-zone clusters. Immediate (the historical default) provisions the volume as soon as the PVC is created, before any Pod is scheduled — which can create a disk in zone A while the Scheduler later decides to place the Pod in zone B, resulting in a Pod that can never attach its volume. WaitForFirstConsumer delays provisioning until a Pod that uses the PVC is actually scheduled, so the provisioner can create the disk in the exact zone the Scheduler picked — the correct choice for essentially all zone-aware cloud storage today.

A cluster typically has one StorageClass marked as default (via the storageclass.kubernetes.io/is-default-class annotation) so PVCs that omit storageClassName still get dynamically provisioned rather than staying Pending forever.

Internal Working / Interactions

  • PVC — references a StorageClass by name (or uses the cluster default).
  • External provisioner sidecar — a controller running alongside the CSI driver that watches PVCs and calls CreateVolume when their StorageClass matches its driver name.
  • Scheduler — with WaitForFirstConsumer, the scheduler's node decision happens first and informs which topology the provisioner uses.
  • CSI driver — receives the provisioner's request and talks to the actual cloud/storage backend.

Code Example

# StorageClass for zone-aware SSD block storage on AWS
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"  # PVCs with no storageClassName use this
provisioner: ebs.csi.aws.com          # CSI driver that implements provisioning
parameters:
  type: gp3                            # driver-specific: EBS volume type
  fsType: ext4
  encrypted: "true"
reclaimPolicy: Delete                  # PVs created here default to Delete on PVC removal
allowVolumeExpansion: true             # lets PVCs referencing this class be resized later
volumeBindingMode: WaitForFirstConsumer  # provision only after a Pod is scheduled, in its zone

---
# StorageClass for a cheaper, retained "archive" tier
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: archive
provisioner: ebs.csi.aws.com
parameters:
  type: sc1                            # cold HDD-backed, cheaper, lower IOPS
reclaimPolicy: Retain                  # keep data even if the PVC is deleted — deliberate choice
volumeBindingMode: WaitForFirstConsumer

kubectl Commands

kubectl get storageclass                        # list all StorageClasses, note the (default) marker
kubectl describe storageclass fast-ssd          # provisioner, parameters, reclaimPolicy, bindingMode
kubectl get sc -o jsonpath='{.items[*].metadata.name}'  # quick list of class names
kubectl patch storageclass standard -p '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"false"}}}'  # unset old default
kubectl get pvc -A -o custom-columns=NAME:.metadata.name,CLASS:.spec.storageClassName  # audit which class each PVC uses
kubectl get events -A --field-selector reason=ProvisioningFailed  # find provisioning errors cluster-wide

Best Practices & Common Pitfalls

Do
  • Default to WaitForFirstConsumer for any zone-aware/multi-AZ cluster.
  • Keep exactly one StorageClass marked default to avoid ambiguity.
  • Expose a small, well-documented menu of classes (tiered by performance/cost) rather than one-off classes per team.
  • Set allowVolumeExpansion: true so growth doesn't require data migration.
Pitfalls
  • Immediate binding mode in a multi-zone cluster causing "volume node affinity conflict" scheduling failures.
  • Two StorageClasses both marked default, causing unpredictable PVC behavior.
  • Leaving reclaimPolicy: Delete on a class used for stateful production data by default.
  • Assuming StorageClass changes retroactively affect already-bound PVCs — they don't, only new PVCs.

50 Interview Questions

1. Why does a StorageClass separate the "provisioner" field from the "parameters" block instead of having one flat driver config?
Short answer: The provisioner names which plugin (CSI driver) handles the request; parameters are an opaque, driver-specific bag of settings that only that plugin understands.
Detailed explanation: Kubernetes core has no idea what "gp3" or "encrypted" mean — it just passes the parameters map straight through to whichever CSI driver the provisioner field identifies. This keeps the StorageClass API stable across every possible backend while still letting each driver expose its own knobs.
Production example: The same cluster can have a fast-ssd class using provisioner: ebs.csi.aws.com with type: gp3 and an on-prem vsan class using a completely different provisioner and parameter set, with no changes needed to Kubernetes itself.
Common mistake: Copy-pasting parameters from one cloud's StorageClass example into a different provisioner's class, where those keys are silently ignored or rejected.
2. What actually happens between a PVC being created and a disk existing, when it references a StorageClass?
Short answer: An external-provisioner sidecar running alongside the CSI driver watches for unbound PVCs, matches their StorageClass to its own driver name, and calls the CSI CreateVolume RPC.
Detailed explanation: The StorageClass itself does nothing passively — it's just a lookup target. The sidecar controller is the active component: it lists PVCs, filters to ones whose storageClassName resolves to a class with a matching provisioner, and drives the actual create-and-bind sequence, producing a new PV object once the disk exists.
Production example: If the external-provisioner Pod for a given CSI driver crashes, new PVCs referencing that driver's StorageClass sit indefinitely in Pending even though the StorageClass YAML looks perfectly correct.
Common mistake: Debugging a stuck PVC by only inspecting the StorageClass object, instead of checking the health/logs of the provisioner sidecar that's supposed to act on it.
3. Why does reclaimPolicy live on the StorageClass rather than being set per-PVC?
Short answer: reclaimPolicy governs what happens to the underlying disk, which is a property of the PV the class creates — so it's set once at the class level and stamped onto every PV it produces.
Detailed explanation: PVCs are a request for storage, not an owner of disk-lifecycle policy; letting every PVC author choose Delete vs Retain would make data-loss risk inconsistent across a namespace. Centralizing it on the class means a platform team makes that call once per storage tier.
Production example: A fast-ssd class defaults to reclaimPolicy: Delete for ephemeral CI workloads, while an archive class defaults to Retain so accidental PVC deletion never destroys long-term backup volumes.
Common mistake: Assuming you can change a PV's reclaim policy by editing the StorageClass after the PV already exists — the class's setting only applies at PV-creation time, not retroactively.
4. What's the practical difference between volumeBindingMode: Immediate and WaitForFirstConsumer, and why does it matter in a multi-zone cluster?
Short answer: Immediate provisions the disk the moment the PVC is created, before any Pod is scheduled; WaitForFirstConsumer delays provisioning until the Scheduler has picked a node, so the disk is created in the correct zone.
Detailed explanation: Cloud block storage is typically zone-locked — a disk created in zone A cannot attach to a node in zone B. With Immediate, the provisioner has no idea where the Pod will land, so it just picks a zone (often arbitrarily), creating a race condition against the Scheduler's later, independent decision.
Production example: A StatefulSet Pod gets scheduled to a node in zone B, but its PVC's disk was already provisioned in zone A under Immediate mode — the Pod's volume attach step fails and it never becomes Ready.
Common mistake: Leaving a multi-AZ cluster's StorageClass on the historical default of Immediate instead of explicitly setting WaitForFirstConsumer.
5. What error would you see, and why, if a PVC with Immediate binding gets its disk provisioned in the wrong zone?
Short answer: The Pod stays stuck in scheduling with an event describing a "volume node affinity conflict."
Detailed explanation: The PV created under Immediate mode carries a node affinity restricting it to the zone/nodes where the disk actually lives; if the Scheduler later tries to place the Pod anywhere outside that zone, the PV's affinity rules make every candidate node ineligible, and the Pod can never be scheduled without manual intervention.
Production example: A cluster autoscaler adds nodes preferentially in zone C for cost reasons, and Pods whose PVCs were provisioned under Immediate in zone A start failing to schedule cluster-wide.
Common mistake: Trying to "fix" the conflict by cordon/draining nodes rather than recognizing the root cause is the StorageClass's binding mode.
6. Why should a cluster have exactly one StorageClass marked as default, and what happens with the is-default-class annotation?
Short answer: The storageclass.kubernetes.io/is-default-class: "true" annotation tells Kubernetes which class to use when a PVC omits storageClassName; having more than one creates ambiguity.
Detailed explanation: Without a default class, any PVC that doesn't specify one stays unbound with no provisioner to satisfy it. If two classes are both annotated default, the admission behavior becomes implementation-dependent and unpredictable — some PVCs may bind to one, some to the other, with no clear rule from the application author's perspective.
Production example: A platform migration that adds a new default-marked class without first unsetting the old one's annotation causes newly created PVCs across the cluster to unpredictably split between the old and new storage tier.
Common mistake: Forgetting to patch the old class's annotation to "false" before marking a replacement class as the new default.
7. Why is allowVolumeExpansion: true on a StorageClass important, and what happens if it's left off?
Short answer: It permits PVCs referencing that class to be resized later by editing spec.resources.requests.storage; without it, a PVC's size is fixed for life.
Detailed explanation: Many CSI drivers support online volume expansion, but Kubernetes only allows the resize request through if the StorageClass that produced the PV explicitly opts in. Omitting it forces teams into disruptive migrate-to-a-bigger-PVC workflows instead of a simple in-place resize.
Production example: A database PVC nearing capacity can have its size bumped from 100Gi to 250Gi with a single kubectl edit pvc, with no Pod restart needed if the underlying filesystem supports online growth — but only because the class had allowVolumeExpansion: true set from the start.
Common mistake: Discovering the class doesn't allow expansion only after a production disk is nearly full, with no way to retroactively enable it on already-created PVs from a different class.
8. Why does editing an existing StorageClass's parameters not affect PVs that were already provisioned from it?
Short answer: A StorageClass is only consulted at the moment a PVC is provisioned; once a PV exists, it's a standalone object disconnected from any future edits to the class.
Detailed explanation: The provisioner reads the class's parameters exactly once, at CreateVolume time, and bakes the result into the PV/disk. There's no reconciliation loop that re-applies StorageClass changes to previously created volumes — the class is a template used at creation, not a live source of truth.
Production example: Changing a class's type parameter from gp3 to a faster tier has zero effect on the hundreds of PVs already provisioned under the old parameters — only PVCs created after the edit get the new disk type.
Common mistake: Assuming a StorageClass edit is a fleet-wide upgrade lever, when in reality existing volumes need to be manually recreated or migrated to pick up the change.
9. Why would a platform team expose a small menu of named StorageClasses (e.g. standard, fast-ssd, archive) instead of letting teams configure raw provisioner parameters directly?
Short answer: Named tiers hide storage-vendor complexity behind a simple label so application teams pick a tier by name without needing to know CSI parameter semantics.
Detailed explanation: This is the whole point of the StorageClass abstraction — a PVC author writes storageClassName: fast-ssd without knowing or caring that it maps to ebs.csi.aws.com with type: gp3 and encryption on. It also gives the platform team a single place to change encryption defaults, disk types, or reclaim policy for an entire tier at once.
Production example: A compliance requirement to encrypt all new disks is satisfied by adding encrypted: "true" to each StorageClass's parameters once, rather than auditing every application's raw storage config.
Common mistake: Letting every team define its own one-off StorageClass with slightly different parameters, defeating the purpose of a curated, auditable tier menu.
10. How would you troubleshoot a PVC that stays Pending and never gets a PV, when the StorageClass looks correctly configured?
Short answer: Check kubectl get events for ProvisioningFailed reasons, then confirm the named provisioner's driver Pods are healthy and actually watching that StorageClass.
Detailed explanation: A YAML-correct StorageClass can still fail silently if its provisioner string doesn't match any installed CSI driver, the driver's controller Pods are crashlooping, or backend quota/permissions reject the create call — none of which show up by reading the StorageClass object itself.
Production example: A typo like ebs.csi.aws.co (missing "m") in the provisioner field leaves the PVC permanently unbound with no driver ever claiming it, and no obvious YAML validation error at apply time.
Common mistake: Repeatedly re-applying the same PVC/StorageClass YAML expecting a different result, instead of checking events and the provisioner sidecar's logs for the actual failure reason.

16. Which volumeBindingMode is recommended for multi-zone/multi-AZ clusters? — WaitForFirstConsumer.

19. What field enables resizing PVCs created from a StorageClass? — allowVolumeExpansion: true.
20. Can allowVolumeExpansion be enabled after PVs from that class already exist? — Yes, editing the class affects future resize requests, but retroactive behavior depends on the driver supporting expansion.
21. What field on a StorageClass holds provisioner-specific settings like disk type or IOPS? — parameters.
22. Is the parameters block validated by Kubernetes core? — No, it's opaque to Kubernetes and interpreted entirely by the CSI driver.
23. What CSI provisioner string is commonly used for AWS EBS? — ebs.csi.aws.com.
24. What CSI provisioner string is commonly used for GCP Persistent Disk? — pd.csi.storage.gke.io.
25. What command lists all StorageClasses and shows which is marked default? — kubectl get storageclass (default shown with a (default) suffix).
26. What command shows a StorageClass's provisioner, parameters, and policies in detail? — kubectl describe storageclass .
27. Can a StorageClass specify mountOptions? — Yes, a mountOptions field applies default mount flags to PVs it creates.
28. What happens if two StorageClasses are both annotated as default? — Behavior is undefined/implementation-dependent; typically only one is effectively honored, causing inconsistent PVC binding.
29. Does deleting a StorageClass affect PVs already created from it? — No, existing PVs are unaffected; only future provisioning using that class name fails.
30. Why can't Immediate binding mode guarantee correct zone placement? — The disk is created before the Scheduler decides which node/zone the consuming Pod will run on.
31. What Kubernetes component actually invokes CreateVolume on the CSI driver? — The external-provisioner sidecar running alongside the CSI controller plugin.
32. What's a typical reclaimPolicy choice for a StorageClass backing StatefulSet databases? — Often Retain, to prevent accidental data loss on PVC deletion.
33. What's a typical reclaimPolicy choice for ephemeral CI/test PVCs? — Delete, so disks are cleaned up automatically.
34. How would you unset an existing default StorageClass before assigning a new one? — Patch its annotation to "false" via kubectl patch storageclass.
35. What event reason indicates a dynamic provisioning attempt failed? — ProvisioningFailed.
36. Why might a "fast-ssd" and an "archive" StorageClass share the same provisioner but differ in behavior? — Their parameters (disk type, IOPS) and reclaimPolicy differ even though the same CSI driver implements both.
37. Does changing volumeBindingMode on a StorageClass affect PVCs already bound under the old mode? — No, only PVCs provisioned after the change use the new binding behavior.
38. What's a reason a PVC referencing a valid StorageClass name still fails to provision? — The named provisioner isn't installed, is unhealthy, or the parameters are rejected by the backend (e.g. invalid disk type).
39. How can you audit which StorageClass each PVC in the cluster uses? — kubectl get pvc -A -o custom-columns selecting .spec.storageClassName.
40. Why is WaitForFirstConsumer especially important for StatefulSets with per-Pod PVCs? — Each replica's PVC must bind in the same zone its Pod is scheduled to, which only WaitForFirstConsumer guarantees.
41. What's a downside of Immediate binding mode in a single-zone cluster? — Little downside there — the zone-affinity race condition mainly matters when nodes span multiple zones.
42. Can a StorageClass restrict which nodes/zones it provisions into directly? — Yes, via allowedTopologies, independent of or alongside volumeBindingMode.
43. What's the relationship between a StorageClass and a VolumeSnapshotClass? — They're separate but analogous constructs — one names a provisioner for volumes, the other for snapshots of those volumes.
44. Why would encrypted: "true" appear in a StorageClass's parameters rather than as a top-level field? — Encryption is a provisioner-specific capability, so it's expressed through the opaque parameters map, not the core API schema.
45. What happens if a PVC specifies a storageClassName that doesn't exist? — It stays Pending; there's no provisioner to match against, and no PV is ever created.
46. Is it possible to have zero default StorageClasses in a cluster? — Yes; then every PVC must explicitly set storageClassName or remain unbound.
47. What's a common mistake when migrating workloads between StorageClasses with different reclaimPolicy defaults? — Assuming the new policy applies to old PVs, when in fact only newly provisioned PVs get it.
48. Why do some parameters (like IOPS) only make sense for certain disk types within the same provisioner? — Because the parameters map is passed through unchanged to the CSI driver, which may reject or ignore combinations that don't apply to the chosen underlying disk type.
49. How would you quickly find every namespace using a specific StorageClass before deleting it? — kubectl get pvc -A -o custom-columns=NS:.metadata.namespace,CLASS:.spec.storageClassName | grep .
50. Why is it good practice to document each StorageClass's parameters and reclaimPolicy alongside its name? — Because the name alone (e.g. "fast-ssd") tells consumers nothing about cost, durability, or data-loss risk on deletion — that context lives only in the opaque parameters and policy fields.
