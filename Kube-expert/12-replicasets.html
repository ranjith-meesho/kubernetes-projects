ReplicaSets

The controller that guarantees a stable number of identical Pod replicas are always running.
← Pods Course Index Deployments →

Explanation

A ReplicaSet (RS) is a controller whose only job is to guarantee that a specified number of identical Pod replicas are running at all times. You give it a Pod template and a desired replicas count; it continuously watches the cluster and creates or deletes Pods so that the observed count always converges to the desired count. It is the simplest possible expression of Kubernetes' core idea: declare desired state, let a control loop reconcile reality to match it.

Kubernetes needs this because Pods are inherently ephemeral and un-self-healing on their own — a bare Pod that crashes or is evicted simply disappears forever. Any real application needs redundancy (more than one replica for availability) and self-healing (automatic replacement on failure). A ReplicaSet supplies both: it spreads identical copies across the cluster, and the moment one is deleted, evicted, or its node fails, the ReplicaSet controller notices the gap between desired and actual count and creates a replacement.

Mechanically, a ReplicaSet does not care which Pods it manages by name — it identifies "its" Pods purely by a label selector. Every Pod created from the RS's template gets the selector's labels, and the RS controller continuously lists Pods matching that selector to count how many exist. This label-based ownership (rather than a fixed list of Pod names) is what makes Kubernetes controllers composable: multiple controllers can, in principle, watch overlapping label sets, and Pods can be "adopted" by a ReplicaSet if they already exist and match its selector when it's created.

In practice, you almost never create a ReplicaSet directly. It sits one layer below Deployments in the object hierarchy: a Deployment creates and manages ReplicaSets, and each ReplicaSet creates and manages Pods. This chain — Deployment → ReplicaSet → Pod — is how Kubernetes achieves rolling updates without sacrificing the self-healing guarantee: each ReplicaSet in the chain still independently guarantees its own replica count, while the Deployment orchestrates shifting traffic and replica counts between an old and a new ReplicaSet during a rollout.

Understanding ReplicaSets in isolation matters because when things go wrong in production — a Deployment stuck mid-rollout, orphaned Pods after a botched kubectl edit, or a scale event that doesn't seem to take effect — the root cause is almost always visible at the ReplicaSet layer: an owner reference pointing to the wrong RS, a selector that no longer matches the Pod template's labels, or two ReplicaSets fighting over the same Pods.

Internal Working / Interactions

  • API Server — the ReplicaSet controller (running inside kube-controller-manager) never talks to etcd directly; it watches ReplicaSet and Pod objects via the API Server's watch/informer mechanism.
  • etcd — stores the ReplicaSet object (desired replica count, template, selector) and every Pod it owns, as with all objects.
  • Pod owner references — every Pod created by a ReplicaSet gets an ownerReferences entry pointing back to that RS's UID. This is how kubectl delete rs can cascade-delete its Pods, and how the garbage collector knows a Pod is "owned."
  • Scheduler / kubelet — unchanged from a bare Pod: the RS controller only creates the Pod object; the Scheduler still binds it to a node and kubelet still starts the containers.
  • Deployment controller — sits above the ReplicaSet controller, creating/scaling ReplicaSets during rollouts; the RS controller has no awareness that a Deployment exists above it.

Reconciliation loop, every cycle: list Pods matching spec.selector → compare count to spec.replicas → if too few, create new Pods from spec.template; if too many (e.g. after a manual Pod creation with matching labels), delete the newest/least-ready extras first.

Code Example

apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: web-rs                    # Name of the ReplicaSet object itself.
  labels:
    app: web
spec:
  replicas: 3                     # Desired number of identical Pods to maintain at all times.
  selector:
    matchLabels:
      app: web                   # MUST be a subset of template.metadata.labels, or the API Server rejects it.
  template:                       # Pod template — stamped out verbatim for each replica.
    metadata:
      labels:
        app: web                 # These labels are what the selector above actually matches against.
    spec:
      containers:
        - name: web
          image: nginx:1.25
          ports:
            - containerPort: 80
          resources:
            requests: { cpu: "100m", memory: "128Mi" }
            limits:   { cpu: "250m", memory: "256Mi" }
Selector immutability: spec.selector is immutable after creation. If you need to change which Pods a ReplicaSet manages, you must delete and recreate it — this is one reason Deployments (which manage this for you) are preferred over hand-written ReplicaSets.

kubectl Commands

# List ReplicaSets in the current namespace
kubectl get rs

# Show desired vs current vs ready replica counts, and events
kubectl describe rs web-rs

# Scale a ReplicaSet directly (bypasses any owning Deployment's desired state — usually not recommended)
kubectl scale rs web-rs --replicas=5

# See which Pods a ReplicaSet actually owns
kubectl get pods -l app=web -o wide

# Inspect a Pod's ownerReferences to confirm which RS created it
kubectl get pod web-abc12 -o jsonpath='{.metadata.ownerReferences}'

# Delete a ReplicaSet but orphan its Pods instead of cascading the delete
kubectl delete rs web-rs --cascade=orphan

# Watch replica reconciliation live while deleting a Pod
kubectl get pods -l app=web -w

Best Practices & Common Pitfalls

Best Practices

  • Never create bare ReplicaSets in production — always go through a Deployment for rollout/rollback support.
  • Keep the label selector as specific as possible to avoid accidentally adopting unrelated Pods.
  • Set resource requests/limits on the Pod template so the Scheduler can place replacements reliably.
  • Use topology spread constraints in the template to avoid all replicas landing on one node.

Pitfalls

  • Editing a Pod's labels so it no longer matches the RS selector — the RS silently creates a replacement, leaving the edited Pod orphaned.
  • Two ReplicaSets with overlapping selectors fighting over the same Pods, causing flapping create/delete cycles.
  • Assuming kubectl scale on an RS persists through a Deployment rollout — the Deployment will overwrite it.
  • Forgetting selector immutability and trying to kubectl edit it, which the API Server rejects.

50 Interview Questions

1. What is the single responsibility of a ReplicaSet?
Short answer: Guarantee that a specified number of identical Pod replicas are running at all times, nothing more.
Detailed explanation: It has no concept of rollouts, versions, or history — it only compares the count of Pods matching its selector against spec.replicas and creates/deletes Pods to close that gap.
Production example: A Deployment creates a new ReplicaSet per rollout; each RS only ever cares about its own replica count, letting the Deployment layer own the rollout logic.
Common mistake: Expecting a ReplicaSet to handle image updates gracefully — editing its Pod template does nothing to already-running Pods.
2. How does a ReplicaSet decide which Pods belong to it?
Short answer: Via a label selector, not a fixed list of Pod names.
Detailed explanation: Every reconcile cycle it lists Pods matching spec.selector.matchLabels and counts them; any existing Pod with matching labels can be "adopted" even if the RS didn't create it.
Production example: Manually creating a bare Pod with app: web labels in a namespace that already has a web-rs ReplicaSet causes that Pod to be adopted and counted toward the replica total.
Common mistake: Using overly broad selectors that unintentionally adopt Pods belonging to a different, unrelated workload.
3. What is an ownerReference and how does a ReplicaSet use it?
Short answer: A back-pointer on every Pod the RS creates, referencing the RS's UID.
Detailed explanation: It's what lets kubectl delete rs cascade-delete its Pods by default, and lets the garbage collector recognize a Pod as "owned" versus orphaned.
Production example: kubectl get pod web-abc12 -o jsonpath='{.metadata.ownerReferences}' shows exactly which RS (and, transitively, which Deployment) created a given Pod.
Common mistake: Assuming deleting a ReplicaSet always deletes its Pods — --cascade=orphan explicitly disables this.
4. Why is spec.selector immutable on a ReplicaSet?
Short answer: To prevent a running RS from suddenly changing which Pods it considers "its own," which could orphan or double-adopt Pods mid-flight.
Detailed explanation: The API Server rejects any update to spec.selector after creation; the only way to change it is to delete and recreate the object.
Production example: This is one reason Deployments are preferred — the Deployment controller handles creating a brand-new ReplicaSet with a new selector/hash rather than trying to mutate an existing one.
Common mistake: Trying kubectl edit rs to change the selector and being surprised by a rejected update.
5. What happens if the ReplicaSet controller sees more matching Pods than spec.replicas?
Short answer: It deletes the excess Pods, generally preferring to remove the newest/least-ready ones first.
Detailed explanation: This can happen after a manual Pod creation with matching labels, or right after a scale-down, or when two overlapping-selector ReplicaSets both think they own the same Pods.
Production example: Scaling a ReplicaSet from 5 to 3 replicas via kubectl scale triggers exactly this deletion path.
Common mistake: Not realizing that manually created Pods matching an RS's selector will simply get deleted by the controller shortly after appearing.
6. Why should you almost never create a bare ReplicaSet directly in production?
Short answer: Because it has no rollout or rollback mechanism — updating its Pod template does not touch existing Pods.
Detailed explanation: A Deployment solves this by creating a brand-new ReplicaSet on every template change and shifting traffic between old and new RS objects; a bare RS gives you none of that orchestration.
Production example: Teams that hand-roll ReplicaSets for "simplicity" typically end up manually deleting Pods one at a time to force a rolling update — reinventing what Deployments already do safely.
Common mistake: Editing a bare ReplicaSet's image field and expecting existing Pods to update themselves.
7. How does the object hierarchy Deployment → ReplicaSet → Pod enable rolling updates without losing self-healing?
Short answer: Each layer keeps its own narrow guarantee — the RS still guarantees its own replica count independently, while the Deployment shifts counts between an old and new RS.
Detailed explanation: If a Pod crashes mid-rollout, the ReplicaSet it belongs to (old or new) notices the count drop and replaces it immediately, without the Deployment needing to intervene.
Production example: During a checkout-service rollout, if a brand-new Pod in the new ReplicaSet crashes, that RS replaces it on its own — the Deployment's surge/unavailable math is unaffected by which specific Pod failed.
Common mistake: Believing the Deployment directly manages every individual Pod rather than delegating that job to ReplicaSets.
8. What's the risk of two ReplicaSets with overlapping selectors?
Short answer: They fight over the same Pods, causing flapping create/delete cycles.
Detailed explanation: Each RS independently reconciles toward its own replica count using the same label-matched Pod pool, so one RS's "delete extra Pod" can trigger the other RS's "create missing Pod," repeating indefinitely.
Production example: A botched migration that leaves both an old hand-written RS and a Deployment-owned RS selecting app: web will visibly show Pods being created and terminated in a loop in kubectl get events.
Common mistake: Copy-pasting a ReplicaSet manifest without changing its selector/labels, accidentally creating a selector collision with an existing workload.
9. Why does editing a Pod's labels so it no longer matches its ReplicaSet's selector cause an orphan?
Short answer: The RS controller sees the count of matching Pods drop by one and immediately creates a replacement — the edited Pod is not deleted, just no longer tracked.
Detailed explanation: The RS never "notices" the Pod still exists; it only counts Pods matching its selector, so a relabeled Pod simply falls outside its view while continuing to run and consume resources.
Production example: A debugging kubectl label pod web-abc12 app- to isolate one Pod from load causes a brand-new replacement Pod to spin up, doubling capacity unless you clean up the orphan.
Common mistake: Using label edits as a way to "pause" one Pod without accounting for the replacement the RS will create.
10. Why doesn't kubectl scale on a ReplicaSet persist if it's owned by a Deployment?
Short answer: The Deployment controller is the authoritative source for the RS's desired replica count and overwrites manual changes on its next reconcile.
Detailed explanation: The Deployment continuously ensures its active ReplicaSet has the replica count derived from Deployment.spec.replicas; a direct kubectl scale rs is treated as configuration drift and corrected.
Production example: An on-call engineer scaling a ReplicaSet directly during an incident will see it silently reset back to the Deployment's configured value within seconds.
Common mistake: Scaling the RS instead of the owning Deployment during an emergency, then being confused when the change doesn't stick.
11–50. Additional rapid-fire questions
11. What API group/version is ReplicaSet under? — apps/v1.
12. What predecessor object did ReplicaSet replace? — ReplicationController; RS supports set-based selectors (matchExpressions), RC only equality-based.
13. Can a ReplicaSet's template labels be a superset of its selector? — Yes, template labels must be a superset (include) the selector's matchLabels, not identical.
14. What component actually runs the ReplicaSet controller? — kube-controller-manager, as one of its many control loops.
15. Does the ReplicaSet controller talk to etcd directly? — No, only through the API Server, like every controller.
16. What happens to Pods when you delete a ReplicaSet with default cascade behavior? — They are deleted too (cascade foreground/background delete).
17. How do you delete an RS but keep its Pods running? — kubectl delete rs <name> --cascade=orphan.
18. Why does the Scheduler still get involved even though the RS creates the Pod? — The RS only creates the Pod object with empty nodeName; the Scheduler still performs filtering/scoring and binds it.
19. Does a ReplicaSet perform any rolling update logic itself? — No — it has no update strategy concept at all, only replica-count reconciliation.
20. What field on a Pod tells you which controller last decided to replace it? — Its ownerReferences UID, cross-referenced against the RS's own UID.
21. Why is a specific label selector recommended over a broad one? — To avoid accidentally adopting unrelated Pods that happen to share a common label.
22. How does an RS decide which extra Pod to delete first when scaling down? — Kubernetes uses a deterministic ranking (not-ready, then younger Pods, then Pods on the same node as others) to pick deletion order.
23. Can you have a ReplicaSet with replicas: 0? — Yes — commonly used by Deployments to keep an old RS around at zero replicas for fast rollback.
24. What does kubectl describe rs show that kubectl get rs doesn't? — Full Events history (creations, deletions) and detailed condition/selector information.
25. Is a ReplicaSet namespaced or cluster-scoped? — Namespaced, like Pods and Deployments.
26. What happens if you set resources.requests on the template but the Scheduler can't find capacity? — New replacement Pods stay Pending until capacity frees up or cluster-autoscaler adds a node.
27. Does a ReplicaSet spread its Pods across nodes automatically? — No — that requires explicit topologySpreadConstraints or anti-affinity in the Pod template.
28. What's the relationship between a ReplicaSet's UID and pod-template-hash? — Deployments inject a pod-template-hash label derived from the template into both the RS's selector and its Pods, distinguishing RS generations.
29. If you kubectl edit a running Pod's image directly, does the RS notice? — No — the RS only tracks count, not content; the edited Pod keeps running until deleted/replaced for some other reason.
30. Why can't you rename a ReplicaSet in place? — Name is immutable metadata; you must delete and recreate under a new name.
31. What happens to a ReplicaSet's Pods if the RS's node is drained? — The Pods are evicted and the RS creates replacements on other schedulable nodes.
32. How would you find every Pod not currently accounted for by any ReplicaSet? — Compare cluster Pods against the union of ownerReferences from all RS objects — orphaned Pods have no matching owner.
33. Does deleting a Deployment also delete its ReplicaSets? — Yes, by default (cascading delete) — and each RS's Pods too, transitively.
34. Can a ReplicaSet use matchExpressions instead of matchLabels? — Yes — it supports set-based selectors like In, NotIn, Exists.
35. Why might kubectl get rs show a ReplicaSet with DESIRED=3, CURRENT=3, READY=1? — Pods exist but haven't passed their readiness probes yet — CURRENT counts existence, READY counts probe success.
36. What's the risk of setting extremely aggressive resource requests on the Pod template? — Replacement Pods may repeatedly fail to schedule, leaving the RS permanently under-replica'd.
37. Does the ReplicaSet controller use a watch or poll the API Server? — Watch, via an informer — reacts to changes in milliseconds rather than on a fixed interval.
38. What happens if two Deployments' ReplicaSets accidentally share the same selector? — Classic selector-collision flapping — both RSs create/delete Pods fighting over the same label-matched set.
39. Can you scale a ReplicaSet using an HPA directly? — Yes, technically, but it's uncommon in practice — HPAs almost always target the owning Deployment instead.
40. What field would you check to see how long a ReplicaSet has existed? — metadata.creationTimestamp.
41. Why do old ReplicaSets from past rollouts still exist at 0 replicas? — Deployments retain them (up to revisionHistoryLimit) to make instant rollback possible without rebuilding images.
42. Is there a "status" subresource on ReplicaSet? — Yes — tracks replicas, fullyLabeledReplicas, readyReplicas, and availableReplicas separately.
43. What's the difference between fullyLabeledReplicas and replicas in RS status? — fullyLabeledReplicas counts Pods whose labels fully match the template (detects drift from manual edits).
44. Can a ReplicaSet manage Pods across multiple namespaces? — No — Pods and their owning ReplicaSet must be in the same namespace.
45. What's a realistic scenario where you WOULD create a bare ReplicaSet? — Rare cases needing precise, manual control over rollout timing without any Deployment abstraction — uncommon outside specialized tooling.
46. How does kubectl rollout history relate to ReplicaSets? — Each entry corresponds to one ReplicaSet the owning Deployment created, identified by revision annotation.
47. What happens if you delete all Pods of a ReplicaSet manually? — The controller notices the count drop to zero and immediately recreates replicas up to spec.replicas.
48. Why is understanding ReplicaSets critical for debugging a stuck Deployment rollout? — Because rollout state (which Pods are old/new, why counts aren't converging) is actually implemented at the RS layer, not the Deployment layer.
49. Does a ReplicaSet enforce any ordering when creating multiple replacement Pods? — No — unlike a StatefulSet, ReplicaSet Pods are created in parallel with no ordering guarantee.
50. What's the first thing to check when a ReplicaSet is stuck below its desired replica count? — kubectl describe rs Events for scheduling failures (insufficient resources, taints) blocking new Pod creation.
