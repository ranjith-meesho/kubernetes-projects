Kubernetes Internals

Informers, watch, resourceVersion, and client-go — the machinery that every controller, including the ones you write, is built on.
← Troubleshooting Course Index Multi-cluster →

Explanation

Every controller in Kubernetes — the built-in ReplicaSet and Deployment controllers, cluster-autoscaler, and any custom operator you write — is built from the same underlying primitive: a watch connection to the API server plus a local cache, wrapped in a pattern called an informer. Understanding this machinery is what separates "I can use kubectl" from "I understand why Kubernetes scales the way it does and can build on it myself," which is exactly why this topic exists near the end of the course, after you've seen the effects (reconciliation, self-healing, fast reaction to change) without yet seeing the mechanism.

The naive way to build a controller would be to repeatedly poll the API server ("give me all Pods") on an interval. This doesn't scale: with thousands of controllers across a large cluster all polling, the API server would collapse under redundant, mostly-unchanged data. Instead, client-go's watch API lets a client open one long-lived HTTP connection and receive a stream of only the changes (Added/Modified/Deleted events) to a resource type, starting from a given resourceVersion — a monotonically increasing marker (backed by etcd's own revision numbering) that lets a client resume exactly where it left off after a disconnect, without re-fetching everything.

An informer wraps this raw watch mechanism into something usable: it does an initial List to populate a local, thread-safe cache (backed by a Store/Indexer), then switches to Watch to keep that cache incrementally up to date, and finally dispatches Add/Update/Delete events to registered handler functions. Controllers read from this local cache instead of hitting the API server on every reconcile — this is why Kubernetes controllers can react within milliseconds and yet impose almost no incremental load on the API server as cluster size grows. A SharedInformerFactory lets multiple controllers within one process share a single underlying watch/cache per resource type, avoiding redundant watches when a controller needs to read several resource kinds.

The typical custom controller pattern (used by every Kubernetes Operator, and the model client-go, controller-runtime, and kubebuilder all formalize) is: watch a resource via an informer, push changed object keys onto a workqueue (deduplicating rapid repeated changes to the same object), and run a reconcile loop that pulls a key off the queue, reads current desired+actual state (from the informer's cache and possibly live API calls), and takes whatever action closes the gap — retrying with exponential backoff on failure. This is the exact same level-triggered reconciliation pattern from Topic 1, just now visible as code you can write yourself.

This topic connects everything: the "watch, don't poll" principle from Topic 1's architecture, the troubleshooting Events you read in Topic 59 (themselves API objects that controllers watch), and every operator you'll ever install (Prometheus Operator, cert-manager, the OpenTelemetry Operator, ArgoCD) — all of them are, underneath, just another informer-driven reconcile loop talking to the same API server.

Internal Working / Interactions

  • API Server exposes List and Watch endpoints per resource type/namespace; every watch event carries the resource's new state plus its updated resourceVersion.
  • Reflector (inside an informer) does the initial List, then opens a Watch from that resourceVersion, feeding events into a local DeltaFIFO queue.
  • Indexer/Store is the thread-safe local cache the reflector populates; controllers query this instead of the API server for reads.
  • Workqueue receives object keys from informer event handlers, deduplicates rapid-fire updates to the same key, and supports rate-limited retry on reconcile failure.
  • Reconciler (your custom controller logic) pulls a key, fetches current state, compares to desired state (often from a CRD spec), and issues API calls (Create/Update/Delete) to converge — the same "compare and converge" loop used by every built-in controller.

Code Example

# Custom Resource Definition — the desired-state object a custom controller watches
# and reconciles, following the exact same List/Watch/reconcile pattern as built-ins.
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: backupschedules.ops.example.com
spec:
  group: ops.example.com
  names:
    kind: BackupSchedule
    plural: backupschedules
  scope: Namespaced
  versions:
    - name: v1
      served: true
      storage: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                schedule: { type: string }     # cron expression - the "desired state" a controller reconciles toward
                targetPVC: { type: string }
            status:
              type: object
              properties:
                lastBackupTime: { type: string }   # written only by the controller, never by users

// Simplified Go controller using client-go's SharedInformerFactory — illustrates the
// informer -> workqueue -> reconcile pattern described above.
factory := informers.NewSharedInformerFactory(clientset, 30*time.Second)
podInformer := factory.Core().V1().Pods().Informer()

queue := workqueue.NewRateLimitingQueue(workqueue.DefaultControllerRateLimiter())

podInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{
    AddFunc: func(obj interface{}) {
        key, _ := cache.MetaNamespaceKeyFunc(obj)
        queue.Add(key)                     // dedup happens automatically if key already queued
    },
    UpdateFunc: func(old, new interface{}) {
        key, _ := cache.MetaNamespaceKeyFunc(new)
        queue.Add(key)
    },
})

factory.Start(stopCh)
factory.WaitForCacheSync(stopCh)           // block until the initial List has fully populated the cache

for {
    key, shutdown := queue.Get()
    if shutdown { break }
    if err := reconcile(key.(string)); err != nil {
        queue.AddRateLimited(key)          // retry with exponential backoff on failure
    } else {
        queue.Forget(key)
    }
    queue.Done(key)
}

Best Practices & Common Pitfalls

Best Practices

  • Always read from the informer's local cache in reconcile logic, never issue a live API GET per event — that defeats the entire pattern.
  • Use a workqueue with rate limiting so transient reconcile failures back off instead of hot-looping against the API server.
  • Build custom controllers with controller-runtime/kubebuilder rather than raw client-go — it encodes these patterns correctly by default.
  • Make reconcile logic idempotent and level-triggered (compute full desired vs actual state) rather than edge-triggered.
  • Use leader election for any controller you run with multiple replicas to avoid duplicate/conflicting reconciles.

Pitfalls

  • Polling the API server on a timer inside a custom controller instead of using informers — reintroduces the exact scalability problem watch was built to solve.
  • Forgetting WaitForCacheSync before starting reconciliation, causing reconciles against an incomplete cache.
  • Not handling resourceVersion "too old" errors (etcd compaction) gracefully — the informer must re-List, not crash.
  • Writing non-idempotent reconcile logic (e.g. "always create") that fails or duplicates resources on a re-run.

50 Interview Questions

1. Why does client-go use a watch connection instead of polling the API server?
Short answer: Polling thousands of controllers on an interval would swamp the API server with mostly-redundant reads; watch streams only the changes.
Detail: A watch is one long-lived HTTP connection that delivers Added/Modified/Deleted events starting from a given resourceVersion, so a client never re-fetches unchanged data.
Production example: cluster-autoscaler and every built-in controller share this same watch-based approach, which is why adding more controllers doesn't linearly increase API server load.
Common mistake: Writing a custom controller that polls the API server on a timer — it reintroduces the exact scalability problem watch was built to solve.
2. What is an informer and what does it wrap?
Short answer: An informer wraps the raw List/Watch mechanism into a local cache plus event dispatch layer.
Detail: It does an initial List to populate a Store/Indexer, switches to Watch to keep it incrementally current, and dispatches Add/Update/Delete events to registered handlers.
Production example: The Deployment, ReplicaSet, and every custom Operator controller are all, underneath, informer-driven reconcile loops.
Common mistake: Assuming informers make a live API call on every reconcile — they don't; controllers read the local cache instead.
3. What is the Reflector's job inside an informer?
Short answer: It performs the initial List, then opens a Watch from that resourceVersion, feeding events into a local DeltaFIFO queue.
Detail: The Reflector is the only piece that actually talks to the API server for List/Watch; everything downstream (Store, handlers) consumes what it produces.
Production example: When a controller process restarts, its Reflector re-Lists once, then resumes Watch — that's the visible "initial sync" delay you see on startup.
Common mistake: Confusing the Reflector with the Reconciler — the Reflector only syncs the cache, it never calls your reconcile logic.
4. What is DeltaFIFO and why is it a queue of deltas rather than raw objects?
Short answer: DeltaFIFO is the ordered change queue the Reflector feeds, and it preserves both the type of change and the object so nothing about ordering is lost.
Detail: It sits between the Reflector and the Indexer/Store, ensuring Add/Update/Delete events are processed in order even if several changes to the same object arrive in a burst.
Production example: Rapid successive updates to a busy ConfigMap still get applied to the local cache in the correct sequence.
Common mistake: Assuming the informer cache updates in the same call that dispatches your handler — the two are coupled through DeltaFIFO, not direct writes.
5. What is the Indexer/Store and why do controllers read from it instead of the API server?
Short answer: It's the thread-safe local cache the Reflector populates; reading it avoids a network round trip and API server load on every reconcile.
Detail: Reconcile logic should always query this local cache for current state, never issue a live GET per event.
Production example: A controller managing thousands of Pods can look up any Pod's cached state in-memory instead of hitting the API server thousands of times per second.
Common mistake: Issuing a live API GET inside the reconcile loop "just to be safe" — this defeats the entire purpose of the informer pattern.
6. What problem does a SharedInformerFactory solve?
Short answer: It lets multiple controllers in one process share a single underlying watch/cache per resource type instead of each opening its own.
Detail: Without sharing, five controllers watching Pods in the same binary would open five redundant watches and five redundant caches.
Production example: A controller-runtime manager running several reconcilers for different CRDs shares informers for any resource type they have in common.
Common mistake: Manually constructing a separate informer per controller in the same process, wasting memory and API server connections.
7. Why does the reconcile pattern use a workqueue instead of acting directly inside the event handler?
Short answer: The workqueue deduplicates rapid repeated changes to the same object key and supports rate-limited retry, decoupling event arrival from processing.
Detail: If ten updates to the same object arrive in a second, the workqueue collapses them to one pending key rather than running reconcile ten times.
Production example: workqueue.NewRateLimitingQueue(workqueue.DefaultControllerRateLimiter()) backs off exponentially on repeated reconcile failures instead of hot-looping.
Common mistake: Doing the actual create/update/delete work inside AddFunc/UpdateFunc directly — this blocks the informer's event dispatch and loses retry/backoff semantics.
8. What is resourceVersion and why does watch depend on it?
Short answer: A monotonically increasing marker, backed by etcd's own revision numbering, that lets a client resume a watch exactly where it left off.
Detail: An informer's List returns objects plus a resourceVersion; the subsequent Watch call starts from that version so no changes are missed or duplicated across a reconnect.
Production example: kubectl get pods -o json | jq '.items[0].metadata.resourceVersion' shows this marker on any object.
Common mistake: Treating resourceVersion as a simple counter you can compare across different resource types or clusters — it's only meaningful for resuming a watch on the same collection.
9. Why is WaitForCacheSync required before starting reconciliation?
Short answer: It blocks until the informer's initial List has fully populated the local cache, so reconcile logic never runs against incomplete data.
Detail: Starting the reconcile loop before the cache is synced can cause a controller to see a partial world and take wrong or duplicate actions.
Production example: factory.Start(stopCh); factory.WaitForCacheSync(stopCh) is the standard bootstrap sequence in a client-go controller.
Common mistake: Forgetting to call WaitForCacheSync, causing reconciles against an incomplete cache right after controller startup.
10. What happens when a watch's resourceVersion becomes "too old" due to etcd compaction?
Short answer: The watch fails with an error, and the informer must fall back to a fresh List rather than crash.
Detail: etcd periodically compacts old revisions to bound storage growth, so a resourceVersion that's been held too long may no longer exist to resume from.
Production example: A controller offline for an extended period reconnects, hits a "too old resource version" error, and re-Lists to rebuild its cache before resuming Watch.
Common mistake: Not handling this error gracefully — letting the informer crash instead of re-Listing leaves the controller stuck.
11–50. Additional rapid-fire questions
11. What is the reconcile loop pattern shared by every controller? — Watch via informer, push changed keys to a workqueue, pull a key, compare desired vs actual state, take action to close the gap.
12. Why is the reconcile pattern level-triggered rather than edge-triggered? — Level-triggered (recompute full desired vs actual state) is resilient to missed events; edge-triggered would lose state on a dropped message.
13. What does a CustomResourceDefinition's spec vs status split represent? — spec is the desired state written by users; status (e.g. lastBackupTime) is written only by the controller reconciling toward that spec.
14. Why must reconcile logic be idempotent? — Because it can be called repeatedly for the same object (retries, resyncs); non-idempotent logic like "always create" fails or duplicates resources on a re-run.
15. What does queue.AddRateLimited(key) do versus queue.Forget(key)? — AddRateLimited requeues the key with exponential backoff after a failure; Forget clears the backoff state after a successful reconcile.
16. What is the purpose of cache.MetaNamespaceKeyFunc? — Converts an object into a namespace/name string key used to identify it in the workqueue and cache lookups.
17. Why do event handlers only enqueue a key rather than the full object? — By the time the key is processed, the object may have changed again; re-reading from the cache by key always gets the freshest known state.
18. What's the difference between client-go and controller-runtime? — client-go is the low-level library (informers, workqueue, clientset); controller-runtime is a higher-level framework built on it that encodes the informer/workqueue/reconcile pattern correctly by default.
19. Why is controller-runtime/kubebuilder recommended over raw client-go for new controllers? — It handles cache sync, leader election, and workqueue wiring correctly out of the box, reducing the chance of subtle bugs.
20. Why does every Kubernetes Operator follow the informer -> workqueue -> reconcile model? — It's the same underlying primitive Kubernetes itself uses for built-in controllers, formalized by client-go/controller-runtime/kubebuilder.
21. What role does leader election play for a controller run with multiple replicas? — Ensures only one replica actively reconciles at a time, avoiding duplicate or conflicting actions from concurrent reconcilers.
22. How would you observe leader election objects for controllers in a cluster? — kubectl get leases -n kube-system lists the lease objects controllers use to coordinate.
23. How can you watch built-in controller reconcile activity for troubleshooting? — kubectl logs -n kube-system deploy/kube-controller-manager | grep -i "sync" surfaces reconcile-related log lines.
24. Why does a CRD schema define both a cron-style schedule field and a lastBackupTime status field? — The former is desired state a controller reconciles toward; the latter is observed state the controller reports back after acting.
25. What does kubectl api-resources reveal in the context of informers? — The full list of resource types informers can watch, since any registered API resource (built-in or CRD) is watchable the same way.
26. How are Events relevant to the informer pattern? — Events are just another watched resource type; a controller can watch Events the same way it watches Pods or CRDs.
27. Why does a raw watch stream show only Added/Modified/Deleted, not a full object list? — Because watch is a delta stream, not a re-list; you already have the prior state from the initial List, so only diffs are sent.
28. How would you inspect a raw watch stream manually? — kubectl get --raw '/api/v1/pods?watch=1' --v=8 shows the underlying event stream.
29. Why is the initial List step necessary even though watch delivers all subsequent changes? — Watch only delivers changes going forward; without an initial List the cache would start empty and miss every object that existed before the watch began.
30. What happens if a controller's cache falls behind the actual cluster state? — Reconciles act on stale data until the next sync; this is why WaitForCacheSync and correct resourceVersion handling matter.
31. Why is the workqueue described as supporting "rate-limited retry"? — Failed reconciles are requeued with increasing backoff instead of being retried immediately, avoiding hot-looping against the API server.
32. What's the risk of not deduplicating rapid updates to the same object key? — The reconcile function would run redundantly many times for what is effectively one logical change, wasting API server calls.
33. Why does the reconciler issue Create/Update/Delete calls back to the API server rather than mutating the local cache directly? — The local cache is read-only from the controller's perspective; only the API server (and thus etcd) is the source of truth, and the informer's watch will reflect the change back into the cache.
34. What does the reconciler read from before deciding what action to take? — The informer's local cache (Indexer/Store) for current state, and the CRD/object spec for desired state, occasionally supplemented with live API calls when needed.
35. How does this topic connect to the "watch, don't poll" architecture principle from earlier in the course? — It's the same level-triggered reconciliation principle, now shown as the actual client-go code a custom controller runs.
36. Why are Prometheus Operator, cert-manager, and the OpenTelemetry Operator all described as "informer-driven reconcile loops"? — Because every Kubernetes Operator, regardless of vendor, is built from the same watch-cache-workqueue-reconcile primitive talking to the same API server.
37. Why does a CustomResourceDefinition need served: true and storage: true on a version? — served controls whether the API server exposes that version to clients; storage controls which version is persisted in etcd — exactly one version must be the storage version.
38. What's the danger of writing non-idempotent "always create" reconcile logic? — Re-running reconcile (which happens routinely on resync or retry) would attempt to create the same resource again and fail or duplicate it.
39. Why does the informer pattern reduce API server load "as cluster size grows," specifically? — Because reads happen against the local cache regardless of scale; only the initial List and the ongoing Watch touch the API server, not each reconcile.
40. What's the practical effect of forgetting to call WaitForCacheSync in a real controller? — Early reconciles may see zero or partial objects and take incorrect actions (e.g. thinking a resource doesn't exist yet).
41. Why is polling described as something that "collapses" the API server at scale? — With thousands of controllers each polling on a timer, the redundant, mostly-unchanged reads compound linearly with controller count, unlike watch's incremental model.
42. How does a BackupSchedule CRD's targetPVC field fit the reconcile model? — It's part of desired state the controller reads to know which PersistentVolumeClaim to act on when executing the reconcile.
43. What is the purpose of registering handler functions (AddFunc/UpdateFunc/DeleteFunc) on an informer? — They're the entry point that translates a cache change into a workqueue enqueue, bridging the informer's internal event stream to your controller logic.
44. Why does UpdateFunc receive both an old and new object? — So the handler can decide whether the change is meaningful enough to enqueue (e.g. ignore updates that only touch resourceVersion or status noise).
45. What's the relationship between the informer's resync period and correctness? — A periodic resync re-delivers all cached objects to handlers as a safety net, catching any reconcile that might have been missed due to a transient bug or dropped event.
46. Why does the diagram show the Reconciler both reading from the Store and writing to the API Server? — Reads are cheap and local (from cache); writes must go through the API server since it's the only path to persisting changes in etcd.
47. What would you check first if a custom controller seems to have stopped reacting to changes? — Whether its informer's cache is synced and its watch connection is alive — check controller logs for watch errors or a stuck leader election lease.
48. Why is the workqueue keyed by object identity rather than holding full event payloads? — Keying by identity naturally deduplicates repeated changes to the same object, since only the latest state matters at reconcile time, not the sequence of changes.
49. How does this internals topic explain why Kubernetes controllers can react within milliseconds at large scale? — Because reconciliation reads from an in-memory cache kept current by watch, not from a live API call, so reaction time is decoupled from API server load.
50. Why is understanding informers described as separating "I can use kubectl" from being able to build on Kubernetes yourself? — Because every Operator you'd write or extend is built from this exact List/Watch/cache/workqueue/reconcile machinery, not from ad-hoc polling scripts.
