Backup & Restore

Protecting cluster state (etcd) and application state (PVs, manifests) so recovery is a procedure, not a prayer.
← High Availability Course Index Disaster Recovery →

Explanation

Kubernetes backup has two entirely separate layers, and conflating them is the most common mistake teams make. The first layer is cluster state — everything the control plane knows about: every Deployment, Secret, ConfigMap, RBAC rule, CRD, and their metadata, all of which lives in etcd. The second layer is application state — the actual data your workloads produced: rows in a database backed by a PersistentVolume, files on disk, object storage contents. A full etcd snapshot will restore every Kubernetes object definition perfectly, but it will not back up the bytes sitting inside your PersistentVolumes — those live on the underlying storage system (EBS, PD, Azure Disk, Ceph, etc.) and need their own backup path.

etcd backup is done via etcdctl snapshot save, which takes a point-in-time consistent snapshot of the entire keyspace by streaming it over the client API — no need to stop etcd. Restoring is the more delicate half: etcdctl snapshot restore doesn't restore in place, it builds a brand-new data directory from the snapshot, which you then point a (re-initialized) etcd member at. In practice this means restoring etcd is effectively "stand up a fresh single-node etcd from the snapshot, then re-join or re-cluster it" — you cannot simply overwrite a running member's data directory.

For application-level backup, the de-facto open-source tool is Velero. Velero backs up Kubernetes object manifests (like a targeted, on-demand etcd export scoped to a namespace or label selector) AND triggers your cloud provider's native volume snapshot API (EBS snapshots, GCE PD snapshots, Azure Disk snapshots) for any PVs attached to the backed-up pods, storing everything in an object-storage bucket (S3/GCS/Azure Blob). Restore recreates the manifests and re-provisions volumes from the snapshots — critically, this works cross-cluster, which etcd snapshots do not (an etcd snapshot restore only makes sense onto the same cluster/topology it came from).

The operational reasoning that matters: etcd backups protect you against control-plane corruption or loss (a bad upgrade wipes etcd, a fat-fingered kubectl delete namespace kube-system, a failed etcd member replacement gone wrong). Velero-style backups protect you against application-level disasters (accidental kubectl delete pvc, a bad migration corrupting a database, needing to clone a namespace into a new cluster for testing, or actual disaster recovery to a different region/cluster). Most production incidents that actually happen are the latter — someone deletes the wrong namespace — which is exactly why Velero (namespace and label-selector-scoped, restorable to any cluster) tends to be exercised far more often than a full etcd restore.

A backup strategy without a tested restore is not a backup strategy. The single highest-value practice here is scheduling a recurring, automated restore-to-scratch-cluster drill and treating any failure as a production incident — because the worst time to discover your etcd snapshot is corrupt, or your Velero bucket credentials expired, is during an actual outage.

Internal Working / Interactions

  • etcdctl snapshot save streams a consistent point-in-time copy of the whole keyspace over the client gRPC API; it does not require quorum loss or downtime.
  • etcdctl snapshot restore writes a new on-disk data directory + cluster metadata; the resulting member starts as a fresh single-node cluster which the other control-plane nodes must then be re-pointed at.
  • Velero server runs as an in-cluster Deployment; on a scheduled or manual backup trigger it calls the Kubernetes API to export object manifests and calls the cloud provider's volume-snapshot API via a VolumeSnapshotLocation plugin.
  • Velero restore replays manifests (respecting namespace/label filters and remapping if needed) and creates new PVCs bound to volumes restored from the cloud snapshots.
  • Object storage (S3/GCS/Azure Blob) is the durable backing store for both the manifest data and the snapshot metadata Velero manages — treat this bucket itself as needing cross-region replication.

Code Example

#############################################
# 1. etcd snapshot backup and restore
#############################################
# Take a consistent snapshot (run on a control-plane / etcd host)
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot-$(date +%F).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify snapshot integrity before trusting it
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-snapshot-2026-07-19.db -w table

# Restore into a NEW data directory (never overwrite a live member's data-dir)
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot-2026-07-19.db \
  --data-dir /var/lib/etcd-restored \      # fresh directory for the restored member
  --name etcd-restored-1 \
  --initial-cluster "etcd-restored-1=https://10.0.0.11:2380" \
  --initial-advertise-peer-urls https://10.0.0.11:2380
# Then update the static pod manifest / etcd unit to point --data-dir at the restored path
# and restart kubelet/etcd so the API server picks up the recovered state.

---
# 2. Velero — install, back up, and restore a namespace (AWS example)
velero install \
  --provider aws \
  --plugins velero/velero-plugin-for-aws:v1.9.0 \
  --bucket my-cluster-velero-backups \             # S3 bucket holding manifests + snapshot metadata
  --backup-location-config region=us-east-1 \
  --snapshot-location-config region=us-east-1 \
  --secret-file ./credentials-velero

# On-demand backup of one namespace, including PV snapshots
velero backup create checkout-ns-backup \
  --include-namespaces checkout \
  --snapshot-volumes=true

# Schedule nightly backups (cron syntax) with 30-day retention
velero schedule create nightly-checkout \
  --schedule="0 2 * * *" \
  --include-namespaces checkout \
  --ttl 720h0m0s

# Restore into the same or a different cluster
velero restore create --from-backup checkout-ns-backup

kubectl / CLI Commands

# Confirm which etcd endpoint / certs are in use on a kubeadm cluster
cat /etc/kubernetes/manifests/etcd.yaml | grep -A5 command

# List and inspect Velero backups
velero backup get
velero backup describe checkout-ns-backup --details

# Check restore progress and any warnings/errors
velero restore describe 
velero restore logs 

# List all PVCs/PVs to confirm what needs volume-level backup coverage
kubectl get pvc -A
kubectl get pv

# Dry-run check of what a Velero backup would include
velero backup create --include-namespaces checkout --dry-run -o yaml

# Verify etcd cluster health right before/after a restore operation
etcdctl endpoint health --cluster

Best Practices & Common Pitfalls

Best Practices

  • Automate etcd snapshots on a schedule and ship them off-node immediately (S3/GCS, not local disk).
  • Run Velero for namespace/application-level backup in addition to etcd snapshots — they cover different failure modes.
  • Encrypt backups at rest and restrict bucket access via least-privilege IAM.
  • Regularly restore into a scratch cluster to prove backups actually work.
  • Version and retain multiple snapshot generations, not just the latest one.

Common Pitfalls

  • Assuming an etcd snapshot backs up PersistentVolume data — it does not.
  • Storing etcd snapshots on the same node/disk that could fail alongside the cluster.
  • Never testing restore, only backup — discovering a corrupt snapshot during a real outage.
  • Forgetting Secrets/CRDs are namespaced objects too and need explicit inclusion in Velero backup scope.
  • Restoring an etcd snapshot in place over a live member's data directory instead of a fresh one.

50 Interview Questions

1. What are the two entirely separate layers of Kubernetes backup, and why does conflating them cause incidents?
Short answer: Cluster state (everything in etcd — object definitions, Secrets, RBAC) and application state (the actual bytes in PersistentVolumes).
Detail: An etcd snapshot restores every object definition perfectly but never touches the data sitting on EBS/PD/Azure Disk — teams who only take etcd snapshots discover this gap during a real PVC-deletion incident.
Production example: Restoring etcd after a bad upgrade brings back a Deployment's spec instantly, but if the underlying PV was also deleted, the database rows inside it are gone unless backed up separately via Velero + volume snapshots.
Common mistake: Assuming "we back up etcd" means the whole cluster's data is protected.
2. How does etcdctl snapshot save take a consistent backup without stopping etcd?
Short answer: It streams a point-in-time consistent copy of the entire keyspace over the client gRPC API while etcd keeps serving reads/writes.
Detail: No quorum loss or downtime is required — it's a live, non-disruptive snapshot mechanism, which is why it can be safely automated on a schedule.
Production example: A CronJob runs etcdctl snapshot save nightly and immediately uploads the resulting .db file to S3.
Common mistake: Believing a snapshot requires a maintenance window — it doesn't, and delaying automation because of that assumption leaves gaps in RPO coverage.
3. Why can't you restore an etcd snapshot "in place" over a live member's data directory?
Short answer: etcdctl snapshot restore builds a brand-new data directory and cluster metadata from scratch — it doesn't merge into or overwrite an existing running member.
Detail: The practical restore flow is: stand up a fresh single-node etcd from the snapshot, point the static pod manifest at the new data directory, then re-cluster/re-join the other control-plane nodes to it.
Production example: After a corrupted etcd cluster, the restore procedure creates /var/lib/etcd-restored as a new directory rather than touching /var/lib/etcd directly.
Common mistake: Attempting to restore directly into a live member's existing data directory, which fails or corrupts cluster metadata.
4. What does Velero actually back up, and how is that different from an etcd snapshot?
Short answer: Velero backs up Kubernetes object manifests scoped to a namespace/label selector, plus triggers the cloud provider's native volume-snapshot API for attached PVs.
Detail: Unlike a full etcd snapshot (all-or-nothing, same-cluster restore only), Velero is targeted and cross-cluster-restorable, because it works through the API and cloud snapshot APIs rather than etcd's raw keyspace.
Production example: velero backup create checkout-ns-backup --include-namespaces checkout --snapshot-volumes=true captures both manifests and PV data for one namespace only.
Common mistake: Trying to use an etcd snapshot to migrate one namespace to a new cluster — Velero is the right tool for that, not etcd.
5. Why do most production incidents get resolved via Velero restore rather than a full etcd restore?
Short answer: Most real incidents are application-level (accidental kubectl delete pvc, a bad migration, needing to clone a namespace) rather than control-plane corruption.
Detail: Velero's namespace/label-selector scoping and cross-cluster restore make it the natural tool for these everyday disasters, while a full etcd restore is reserved for genuine control-plane loss.
Production example: An engineer deletes the wrong namespace in production; the fix is velero restore create --from-backup for that namespace, not a cluster-wide etcd restore.
Common mistake: Treating etcd snapshots as the primary recovery tool for application-level mistakes, when Velero is scoped and faster for that exact scenario.
6. What's the difference in how PV data is protected by Velero versus by an etcd snapshot?
Short answer: etcd never touches PV data at all; Velero calls the cloud provider's native volume-snapshot API (EBS/GCE PD/Azure Disk snapshots) via a VolumeSnapshotLocation plugin.
Detail: Velero restore recreates the PVC manifests and re-provisions volumes from those cloud snapshots — the actual snapshot mechanism is the cloud platform's, not Kubernetes'.
Production example: A Velero backup with --snapshot-volumes=true triggers an EBS snapshot for every PV attached to Pods in the included namespace.
Common mistake: Running Velero backups without confirming a VolumeSnapshotLocation is properly configured — manifests get backed up but volume data silently doesn't.
7. Why is "a backup strategy without a tested restore" considered not a backup strategy at all?
Short answer: An unverified backup could be corrupt, incomplete, or unrestorable, and you only find out during a real outage — the worst possible time.
Detail: The highest-value practice is a recurring, automated restore-to-scratch-cluster drill, treating any failure (expired bucket credentials, corrupt snapshot) as a production incident in its own right.
Production example: A monthly job restores the latest Velero backup and etcd snapshot into an isolated scratch cluster and alerts if either restore fails.
Common mistake: Verifying only that the backup job "completed successfully," never actually attempting a restore.
8. Why should etcd snapshots and Velero backups be stored off-node immediately?
Short answer: A backup stored on the same node/disk that could fail alongside the cluster provides no real protection.
Detail: Shipping snapshots to S3/GCS/Azure Blob (with its own durability and ideally cross-region replication) decouples backup survival from the failure you're trying to protect against.
Production example: The etcd snapshot CronJob writes to local disk only as a staging step, then immediately uploads to an S3 bucket before the job completes.
Common mistake: Leaving snapshots on the control-plane node's local disk "temporarily," which becomes permanent and defeats the point of backing up at all.
9–50. Additional rapid-fire questions
9. What command verifies an etcd snapshot's integrity before trusting it? — etcdctl snapshot status <file> -w table.
10. Does an etcd snapshot include Secrets? — Yes, in whatever form they're stored in etcd (plaintext unless encryption-at-rest is configured).
11. Does Velero back up CRDs and Secrets by default? — Only within the included namespace/label scope you specify — they must be explicitly covered, not assumed.
12. What's the risk of restoring an etcd snapshot from a different cluster? — Cluster-specific metadata (member IDs, peer URLs) won't match, unlike Velero which is designed to be cross-cluster-restorable.
13. Why does Velero require a VolumeSnapshotLocation plugin per cloud provider? — Because the actual snapshot mechanism is the cloud's native API (EBS, PD, Azure Disk), not something Kubernetes/Velero implements itself.
14. What's a `velero schedule` and why use it over manual `velero backup create`? — It automates recurring backups on a cron expression with retention (TTL), removing reliance on someone remembering to back up manually.
15. What does --ttl 720h0m0s control in a Velero schedule? — Retention period for each backup generation (30 days here) before it's eligible for garbage collection.
16. Why should you retain multiple etcd snapshot generations, not just the latest? — The latest snapshot could itself be a snapshot of an already-corrupted state; older generations give you a fallback.
17. What's the operational meaning of "etcd backup protects against control-plane loss, Velero protects against application-level disasters"? — They cover different failure domains and are complementary, not redundant.
18. How do you check what a Velero backup would include without actually running it? — velero backup create ... --dry-run -o yaml.
19. What's the first command to run after a Velero restore to check for problems? — velero restore describe <restore-name>, followed by velero restore logs.
20. Why must backup bucket credentials be tested regularly, not just configured once? — Expired/rotated credentials silently break backup jobs until someone needs to restore and discovers the failure.
21. What's a realistic RPO if etcd snapshots run every 4 hours? — Up to 4 hours of control-plane object changes could be lost in a worst-case restore.
22. Does restoring an etcd snapshot restart running Pods? — No — Pods are managed by kubelet/controllers reconciling against whatever state is now in etcd; existing running containers aren't directly touched by the restore itself.
23. Why encrypt backups at rest in the object storage bucket? — Backups contain the same sensitive data (Secrets, config) as the live cluster and are an equally valuable attack target.
24. What IAM principle should govern access to the backup bucket? — Least privilege — restrict who/what can read or delete backup objects, since deleting backups is itself a disaster vector.
25. What's the difference between backing up a StatefulSet's manifest and backing up its data? — The manifest (via etcd/Velero) restores the StatefulSet's desired state; the actual database files need PV-level snapshot coverage separately.
26. Why is namespace-scoped Velero backup preferable to whole-cluster backup in many workflows? — Faster, more targeted restores and lower blast radius when only one team's data needs recovery.
27. What happens if you restore a Velero backup into a cluster with different StorageClass names? — The restore can fail or misbind PVCs unless you configure Velero's resource remapping for storage classes.
28. Why is testing restore into a *different* cluster valuable, not just the same one? — It validates the cross-cluster DR use case and catches assumptions baked into cluster-specific configuration.
29. What's the risk of only ever testing restore in a low-traffic staging environment? — Restore time and volume snapshot provisioning time can scale with data size — production-scale restores may reveal much longer RTOs.
30. Why does etcdctl snapshot save not require etcd downtime, unlike some traditional DB backup tools? — It streams from the existing running process over the API rather than requiring a stop-the-world file copy.
31. What's a common cause of a corrupt etcd snapshot going unnoticed? — Never running etcdctl snapshot status or an actual restore test against it.
32. How does Velero decide which volumes to snapshot? — Based on PVs attached to Pods matched by the backup's namespace/label selector, when --snapshot-volumes=true.
33. Why should Secrets/CRDs be explicitly verified as included in Velero's backup scope? — They're namespaced objects like any other and can be silently excluded by an overly narrow label selector.
34. What's the operational cost of skipping automated snapshot upload to S3/GCS? — A local-only snapshot is vulnerable to the exact node/disk failure that might trigger the need to restore it.
35. Why is "point-in-time consistent" an important property of an etcd snapshot? — It guarantees the restored state reflects a single coherent moment, not a mix of before/after states from concurrent writes.
36. What's the relationship between Velero and GitOps for DR? — GitOps (Argo CD/Flux) can reapply manifests from source control quickly, but Velero is still needed to restore PV data and any state not tracked in Git.
37. Why does restoring etcd typically require re-clustering the other control-plane nodes? — The restored member starts as a fresh single-node cluster with new metadata; existing peers must be re-pointed or rejoined to it.
38. What's a good cadence for restore-drill exercises? — Regularly enough (e.g. monthly) that credential rotation, snapshot format changes, or tooling drift get caught before a real incident.
39. Why treat a failed restore drill as a production incident? — Because it reveals your actual recovery capability is broken right now, not hypothetically — the same urgency as a live outage.
40. What's the danger of relying solely on cloud-provider automatic disk snapshots without Velero? — You lose the coordinated manifest+volume restore workflow and cross-cluster restore capability Velero provides.
41. How do you confirm etcd cluster health immediately before/after a restore? — etcdctl endpoint health --cluster.
42. What's the purpose of versioning/retaining multiple snapshot generations rather than overwriting the latest? — Protects against restoring a snapshot that itself captured an already-corrupted or compromised state.
43. Why is an etcd snapshot restore considered "the more delicate half" of etcd backup/restore? — Because save is a simple live streaming operation, while restore requires careful data-directory and re-clustering steps that are easy to get wrong under pressure.
44. What's a practical reason to keep etcd snapshot and Velero backup schedules independent rather than coupled? — They protect different layers on different natural cadences — control-plane changes are often less frequent than application data changes.
45. How would you migrate a namespace to a new cluster using these tools? — Velero backup of the namespace, then velero restore create --from-backup targeting the new cluster — not an etcd snapshot.
46. What's a red flag when reviewing a team's backup setup? — Snapshots exist and run on schedule, but nobody can produce evidence of a successful restore test in the last few months.
47. Why does Velero's manifest export resemble "a targeted etcd export"? — Conceptually it captures the same kind of object definitions etcd stores, just scoped and via the API rather than the raw datastore.
48. What should you check before assuming a Velero restore is fully successful? — velero restore describe --details for partial failures/warnings, not just that the command exited without error.
49. Why is bucket replication (cross-region) itself part of a serious backup strategy? — The bucket holding your backups is a single point of failure too — losing it defeats the purpose of backing up at all.
50. What's the single highest-value practice this topic emphasizes above all others? — Scheduling and treating regular restore-to-scratch-cluster drills as mandatory, not optional.
